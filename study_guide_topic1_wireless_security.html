§1 — Concept Overview

📘 LECTURE PDF COVERAGE: pp. 435-464, 558-578 (also see pp. 362, 393, 513, 515 for supporting context)

1.1 The Problem This Solves 📄 p. 559

Wireless networks send every frame as a radio broadcast — anyone with a Wi-Fi card in range can listen, inject, or jam. On top of that, mobile devices leave the building, may be lost or stolen, and often have limited battery and CPU for heavy cryptography 📄 p. 559. Wireless / link-layer security exists to make the first hop (laptop ↔ access point) confidential, authenticated, and resistant to replay — before the packet ever reaches an IP router.

1.2 The Core Idea 📄 pp. 441, 560

Link-layer security wraps every Wi-Fi frame in two protections: encryption so nobody nearby can read it, and a Message Integrity / Authentication Code so nobody can forge or replay it 📄 p. 446. But encryption alone is not enough — the access point must also decide who is allowed on the network. That second job is called Network Access Control (NAC), and on Wi-Fi it is implemented by the 802.1X framework using EAP messages to talk to a separate authentication server 📄 pp. 451-456, 562-567.

1.3 Analogy — The Nightclub with a Back-Office Guest List 📄 pp. 454-456, 565-567

Think of WPA2-Enterprise as a nightclub door:

  • You (the laptop) = the Supplicant 📄 pp. 454, 565
  • The bouncer (the Wi-Fi access point) = the Authenticator — he does not know who is on tonight's guest list; he only relays messages 📄 pp. 455, 566
  • The guest-list manager in the back office (RADIUS / AS) = the Authentication Server — he is the one who actually checks your credentials 📄 pp. 454, 565
  • The front-door intercom = the uncontrolled port (always open, but only passes authentication messages) 📄 pp. 455, 566
  • The main entrance = the controlled port (stays locked until the back office says "approved") 📄 pp. 455, 566
  • The style of ID check (passport? fingerprint? invite code?) = an EAP method. The bouncer does not care which — he just forwards it to the manager 📄 pp. 458-459, 569-570

This mapping is why changing the authentication method (password → certificate → token) requires no hardware change at the access point — only the client software and the RADIUS server need updates 📄 pp. 456, 567.

1.4 Technical Definition 📄 pp. 441, 446, 448, 561

Wireless link-layer security is a family of protocols that operate at Layer 2 (Data Link) of the TCP/IP stack to provide confidentiality, integrity, origin authentication, and access control between a wireless station (STA) and an access point (AP) 📄 p. 437. The current standard, WPA2 (formally IEEE 802.11i — Robust Security Network, RSN, 2004) specifies AES in Counter Mode with CBC-MAC Protocol (AES-CCMP) for encryption + integrity 📄 p. 446. User authentication comes in two flavors: WPA2-Personal uses a shared Pre-Shared Key (PSK), and WPA2-Enterprise uses 802.1X with EAP to authenticate each user individually against a back-end RADIUS authentication server 📄 pp. 448, 561.

Access Point (AP) / Authenticator The Wi-Fi radio the station talks to. In 802.1X, called the Authenticator; it relays EAP messages but does not decide authentication by itself. Common exam trap: "Who makes the authentication decision?" — it is not the AP. 📄 pp. 439, 455, 566

Instructor used the eduroam example explicitly. 📄 pp. 454, 460, 565, 571
Uncontrolled Port Logical port on the AP that is always open but only carries authentication (EAP) traffic. Before login you can only talk to the AS via this port — expect a True/False on this. 📄 pp. 455, 566
Controlled Port Logical port on the AP that stays blocked until 802.1X authentication succeeds; then it opens and carries normal data traffic. Controlled = data. Unlocked only after the AS says "Accept." 📄 pp. 455, 566
WEP 🟡 Wired Equivalent Privacy (1999). First Wi-Fi encryption standard; uses RC4 + 24-bit IV + shared key. Fundamentally broken since 2001. Know the problems (short IV, no mutual auth, no replay protection, weak CRC-32 ICV). 📄 pp. 442-445
WPA Wi-Fi Protected Access (2003). A software patch on WEP hardware using TKIP (key mixing + counter + RC4) plus a 64-bit Message Integrity Check called "Michael." Key exam fact: WPA was deployable without hardware replacement; WPA2 was not. 📄 pp. 446, 448, 561
WPA2 / RSN / 802.11i 🔴 Final 802.11i standard (2004). Uses AES-CCMP (AES in Counter Mode for encryption + CBC-MAC for integrity). Requires hardware replacement. Comes in Personal (PSK) and Enterprise (802.1X/EAP) variants. This is the current-day standard and the heart of the topic — expect MCQ on modes and user-auth flavors. 📄 pp. 446, 448, 561
WPA3 2018 successor to WPA2. Replaces the PSK handshake with SAE (Simultaneous Authentication of Equals — a Diffie-Hellman-based password-authenticated key exchange), giving forward secrecy. Key exam fact per slide: WPA3-PSK fixes WPA2-PSK's "session key is fully derivable from PSK + nonce" weakness. 📄 pp. 441, 449, 560
PSK (Pre-Shared Key) A single password/passphrase shared by every user of the network. Used in WPA-Personal / WPA2-Personal / WPA3-Personal. Vulnerability: all users share the same key, so a malicious user on the same LAN can decrypt others' traffic if the session key is derivable. 📄 pp. 448, 449, 459, 561, 570
802.1X 🔴 IEEE standard for port-based Network Access Control. Defines how EAP messages flow between Supplicant, Authenticator, and Authentication Server — and gates the "controlled port" on the AP. This is the named topic in the final review script. Know its 3 components by heart. 📄 pp. 451, 455, 456, 562, 566, 567
EAP 🔴 Extensible Authentication Protocol — a framework (not a single protocol) that encapsulates any authentication method between client and AS. Know that EAP is an envelope, not a specific method. "EAP itself doesn't authenticate — the method inside does." 📄 pp. 458, 569
EAP-TLS EAP method that embeds a full TLS handshake in EAP. Requires both client and server X.509 certificates → mutual public-key authentication. Instructor said: "You need to know TLS requires both client and server public-key certificates." 📄 pp. 459, 570
EAP-TTLS Tunneled TLS. Only the server has a certificate; the client authenticates with username/password (or similar) inside the TLS tunnel. Instructor explicitly contrasted TTLS vs TLS: "that is one difference between TLS versus tunneled TLS." 📄 pp. 459, 570
RADIUS Remote Authentication Dial-In User Service. The protocol the AP uses to talk to the authentication server. EAP messages ride inside RADIUS Access-Request / Access-Accept packets. Know that the EAP payload crosses two legs: client↔AP (EAPOL) and AP↔AS (RADIUS). 📄 pp. 460, 571
AAA Authentication (who are you?), Authorization (what may you do?), Accounting (keep track of usage) — the three-part framework NAC implements. Short-answer goldmine. Memorize all three A's with their one-sentence meanings. 📄 pp. 454, 565
NAC Network Access Control — the umbrella concept of authenticating users, deciding what they may access, and even checking device health before admission. Instructor emphasized NAC is "umbrella" — 802.1X is one of several enforcement methods (plus Firewall, DHCP, VLAN). 📄 pp. 451, 453, 562, 564
VLAN Segmentation Logically slicing one physical LAN into many virtual LANs so that users/devices in one VLAN cannot see traffic in another. Instructor tied this to the smart-manufacturing example: IT VLAN must be isolated from OT VLAN. 📄 pp. 362, 393, 452, 563
AES-CCMP AES in Counter mode with CBC-MAC Protocol. Counter mode provides confidentiality; CBC-MAC provides integrity. Standard WPA2 link-layer cipher. Parallel to AES-GCM at higher layers. Know that counter mode alone ≠ integrity. 📄 p. 446

2.2 — Phase-by-Phase Annotation

Phase 1 — Discovery 📄 pp. 439-440
What happens: The AP periodically broadcasts a beacon containing its SSID and advertised security parameters (supported ciphers, auth methods). Stations passively listen, measure signal strength, and choose an AP.
Technical: If SSID is hidden, the station can send an active Probe Request; any AP with that SSID replies with a Probe Response 📄 pp. 513, 515. The instructor explicitly noted: "SSID broadcasting is optional — if no SSID broadcasting, the station can also send out probe requests and get the response."
Example: Your laptop sees "eduroam" in its Wi-Fi list because the campus AP broadcast a beacon every ~100 ms.
Phase 2 — Open ("Null") Authentication 📄 pp. 440, 442-443
What happens: A formality exchange of two frames (Auth Request, Auth Response) used for legacy reasons. In WPA2 this provides no real security — the genuine authentication occurs in Phase 4.
Technical: Historically, WEP performed Shared-Key authentication here using an RC4 challenge-response 📄 p. 443; WPA2 replaces that with an empty Open exchange because the real work moves to EAP.
Example: In Wireshark you will still see "Authentication" frames with status "Successful" — but no cryptographic check actually happened.
Phase 3 — Association 📄 pp. 439-440
What happens: The station sends an Association Request with its desired cipher suite and auth method; the AP confirms in an Association Response. The station is now "associated" but data is still blocked — the 802.1X controlled port is closed.
Technical: This is when security parameters are negotiated so both ends agree on WPA2-CCMP, WPA2-TKIP, etc. If parameters mismatch, the AP rejects the association.
Example: Your phone tells the AP "I support AES-CCMP"; the AP replies "agreed, AES-CCMP it is." Any application packet sent now would be dropped at the controlled port.
Phase 4 — 802.1X / EAP Authentication 📄 pp. 455-456, 460-461, 566-567, 571-572
What happens: The AP sends an EAP-Request/Identity over the uncontrolled port; the supplicant responds with its identity; the AP tunnels the EAP payload inside a RADIUS Access-Request to the authentication server. Many round-trips follow, driven by the chosen EAP method (EAP-TLS runs a whole TLS handshake here; EAP-TTLS / PEAP run TLS then inner password auth).
Technical: Two different encapsulations are used in sequence: EAPOL (EAP over LAN) from client to AP, and RADIUS from AP to AS 📄 p. 460. The AP never inspects the inner EAP payload — it just relays. On success, the AS sends an Access-Accept with keying material (the Master Session Key, MSK).
Example — eduroam: A UA student visiting Georgia Tech opens Wi-Fi. Georgia Tech's RADIUS does not have their myBama password, so it forwards the request over the eduroam federation to UA's RADIUS, which verifies and returns Accept. This is exactly what the instructor described: "it will be forwarded to myBama authentication server, so they verify your identity." 📄 pp. 571-572
Phase 5 — Key Management (4-way handshake) 📄 pp. 457, 568
What happens: Four EAPOL-Key messages are exchanged between the supplicant and the authenticator. They prove both sides possess the same master key (PMK, derived from the MSK) and derive a fresh per-session Pairwise Transient Key (PTK) plus a Group Temporal Key (GTK) for broadcast/multicast traffic.
Technical: After message 3 is confirmed, the authenticator opens the controlled port. No actual user data has flowed yet.
Example: This handshake is what the famous KRACK (Key Reinstallation) attack targeted in 2017 📄 p. 450.
Phase 6 — Protected Data Transfer 📄 p. 446
What happens: Every subsequent Wi-Fi frame is encrypted under the PTK using AES-CCMP (AES Counter mode + CBC-MAC). You now have web access.
Technical: CCMP provides confidentiality via counter mode and integrity via CBC-MAC — an analog of the AES-GCM mode used at higher layers. Instructor's note: "counter mode plus message authentication code gives us the AES GCM mode. Basically, the AES GCM mode is the most commonly used solution that provides encryption and message authentication for the link-layer communication."
Example: When you open HTTPS, you now have two layers of encryption: AES-CCMP at Layer 2 (covers the whole frame including IP/TCP headers within the Wi-Fi segment) and TLS at Layer 5 (covers the HTTP payload end-to-end).

2.3 — The Two Personalities of WPA2

The entire flow above describes WPA2-Enterprise. WPA2-Personal skips Phase 4 entirely — instead of contacting a RADIUS server, both sides simply derive the PMK from the pre-shared passphrase and go straight to the 4-way handshake 📄 pp. 448-449, 561. 

 WPA2-PERSONAL (PSK)                    WPA2-ENTERPRISE (802.1X / EAP)
 ──────────────────────                 ──────────────────────────────

   1 Discovery                             1 Discovery
   2 Open Auth                             2 Open Auth
   3 Association                           3 Association
   4 (skipped)                             4 EAP / 802.1X via RADIUS
     ↓ PMK = PBKDF2(passphrase)              ↓ PMK = derived from MSK
   5 4-way handshake                       5 4-way handshake
   6 AES-CCMP data                         6 AES-CCMP data
SOURCE: 📄 pp. 448-449, 451, 561-562

2.4 — How EAP Sits Inside the Flow

EAP is not one protocol — it is a message envelope that can carry many different authentication methods 📄 pp. 458, 569. The message format is fixed: 

 EAP Message Format (📄 pp. 460-461, 571-572):

   ┌──────┬────────────┬────────┬──────┬─────────────────────┐
   │ Code │ Identifier │ Length │ Type │        Data         │
   │  8b  │     8b     │  16b   │  8b  │       variable       │
   └──────┴────────────┴────────┴──────┴─────────────────────┘

   Code values:  01 = Request   02 = Response
                 03 = Success   04 = Failure

   Type values:  identifies the inner authentication method
                 (e.g. EAP-TLS, EAP-TTLS, EAP-GPSK, EAP-AKA)

The same framing travels over both legs — on the wireless leg (Supplicant ↔ AP) it is wrapped in EAPOL; on the wired leg (AP ↔ RADIUS server) it is wrapped in RADIUS attributes. The AP is a "pass-through" — it does not understand the inner Type 📄 pp. 460, 571.

3.2 — Common Misconceptions & Traps

❌ MYTH: "The Access Point authenticates the user."

✅ REALITY: In 802.1X, the AP (Authenticator) only relays EAP messages. The Authentication Server (RADIUS) actually validates credentials and issues Accept/Reject. 📄 pp. 454-456, 565-567

🎯 EXAM TRAP: MCQ with "Which device decides if the user is legitimate?" and one of the options is "Access Point." It is not the AP.

❌ MYTH: "WPA3 is just a rebrand of WPA2."

✅ REALITY: WPA3-Personal replaces the WPA2 4-way handshake with SAE (Dragonfly, a DH-like ephemeral key exchange), giving forward secrecy. In WPA2-PSK the session key is fully derivable from the PSK plus the handshake nonces, so a malicious co-tenant on the same LAN can decrypt your traffic. 📄 p. 449

🎯 EXAM TRAP: "What security property does WPA3-Personal add?" — answer: forward secrecy (session keys no longer derivable from the PSK alone).

❌ MYTH: "EAP is an authentication protocol."

✅ REALITY: EAP is a framework — a message envelope. The actual authentication is performed by the inner method named in the Type field (EAP-TLS, EAP-TTLS, EAP-GPSK, …). 📄 pp. 458, 569

🎯 EXAM TRAP: True/False "EAP specifies a single authentication algorithm." Answer: False — it is a generic container.

❌ MYTH: "Once 802.11 Association completes, the device can send data."

✅ REALITY: Association only negotiates capabilities. The controlled port remains blocked until Phase 4 (EAP) succeeds and Phase 5 (4-way handshake) derives the PTK. 📄 pp. 455-457, 566-568

🎯 EXAM TRAP: "After Association, the station can immediately send HTTP traffic." — False; EAP + key management must complete first.

❌ MYTH: "WPA-Personal and WPA-Enterprise differ in encryption algorithm."

✅ REALITY: They differ in user authentication, not encryption. Both use the same cipher within a given WPA version (e.g., WPA2-Personal and WPA2-Enterprise both use AES-CCMP). Personal = PSK; Enterprise = 802.1X/EAP. 📄 pp. 448, 561

🎯 EXAM TRAP: Matching-table questions that swap the "Encryption" and "User Auth" columns. Always identify which column you are filling in.

Mnemonic — "Six Phases of 802.11i" = D-O-A-E-K-D Discovery → Open auth → Association → EAP authentication → Key mgmt (4-way) → Data. Remember as "Dogs Often Ask Every Kind of Dinner." 📄 pp. 457, 568
Mnemonic — "SAA" for 802.1X components Supplicant (you) → Authenticator (AP, the dumb relay) → Authentication server (RADIUS, the brain). "Supplicant Asks, Authenticator Announces, AS Approves." 📄 pp. 454-456, 565-567
Mnemonic — "TTLS is 1 cert, TLS is 2 certs" The extra T in TTLS ("Tunneled") means the second cert is replaced by a tunnel — so only the server needs a cert. EAP-TLS = 2 certs; EAP-TTLS = 1 cert. 📄 pp. 459, 570
Mnemonic — "AAA = Authentication, Authorization, Accounting" Think "Who are you? What can you do? What did you do?" 📄 pp. 454, 565
Mnemonic — "PSK is derivable, SAE is ephemeral" WPA2-PSK: attacker who knows the PSK + records the handshake can decrypt everything. WPA3-SAE: ephemeral DH means past traffic stays private even if PSK leaks later. 📄 p. 449

INT MCQ 📄 pp. 455-456, 566-567

Q2. In a WPA2-Enterprise deployment, which entity makes the decision to accept or reject a user's credentials?

  1. The Supplicant (user's laptop).
  2. The Authenticator (the wireless access point).
  3. The Authentication Server (RADIUS).
  4. The DHCP server on the local subnet.
▶ Show Answer

✅ Correct Answer: C) The Authentication Server (RADIUS).

📝 Explanation:

  • Why C is correct: The AS is the entity that holds the user database and runs the inner EAP method to verify credentials; the AP only relays EAP messages in and out of EAPOL / RADIUS envelopes. 📄 pp. 455-456, 566-567
  • Why A is wrong: The Supplicant is the subject being authenticated, not the decider. 📄 pp. 454, 565
  • Why B is wrong: The AP is a pass-through "Authenticator" — it is explicitly designed not to understand the inner EAP method, so it cannot make the decision. 📄 pp. 456, 567
  • Why D is wrong: DHCP comes after authentication (it is a separate NAC enforcement method that assigns IPs once you are allowed on the network). 📄 pp. 452, 563
  • Exam Tip: Remember the nightclub analogy — the bouncer (AP) does not run the guest list, the back-office manager (RADIUS) does.

ADV · challenge MCQ 📄 pp. 459, 570

Q4. You deploy EAP-TTLS on a campus eduroam network. Which statement most accurately describes the certificate requirements for this deployment?

  1. Both every user device and the authentication server must have a valid X.509 certificate.
  2. Only the authentication server must have a valid X.509 certificate; user devices authenticate with username/password carried inside the TLS tunnel.
  3. Only user devices must have certificates; the AS relies on a pre-shared key.
  4. Certificates are not used at all; authentication is done via Diffie-Hellman SAE.
▶ Show Answer

✅ Correct Answer: B) Only the AS has a certificate; clients authenticate via inner password inside the TLS tunnel.

📝 Explanation:

  • Why B is correct: Slide 459/570 specifies EAP-TTLS as "only the server has a certificate to authenticate itself to the client. A secure connection (the 'tunnel') is established with secret keys." Inside that tunnel, the client presents username/password. This is why the instructor said it is the practical choice for consumer/guest/eduroam networks. 📄 pp. 459, 570
  • Why A is wrong: That is EAP-TLS (both sides have certs), not EAP-TTLS. 📄 pp. 459, 570
  • Why C is wrong: That is EAP-GPSK (Generalized Pre-Shared Key), not TTLS. 📄 pp. 459, 570
  • Why D is wrong: SAE is the WPA3-Personal handshake, not an EAP method. 📄 p. 449
  • Exam Tip: The extra T ("Tunneled") in TTLS literally means the second certificate is replaced by a tunnel. Count the T's = count the certs.

Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG TF 📄 pp. 455, 566

Q5. Statement: In an 802.1X deployment, the controlled port on the AP is open by default and becomes blocked only after authentication fails.

▶ Show Answer

✅ Answer: FALSE.

It is exactly the opposite. The controlled port starts blocked and only opens after the RADIUS server returns Access-Accept and the 4-way handshake completes. The uncontrolled port is the one that is always open — and only for EAP/auth traffic. To make the statement true it would need to read: "the controlled port is blocked by default and becomes open only after authentication succeeds." 📄 pp. 455, 566

INT TF 📄 pp. 448, 561

Q6. Statement: WPA2-Personal and WPA2-Enterprise use different encryption algorithms — AES-CCMP for Enterprise and TKIP for Personal.

▶ Show Answer

✅ Answer: FALSE.

Within a single WPA version, Personal and Enterprise use the same encryption algorithm. In WPA2, both use AES-CCMP (more specifically, AES in Counter mode with CBC-MAC). The difference between Personal and Enterprise lies entirely in user authentication: Personal uses PSK; Enterprise uses 802.1X/EAP. 📄 pp. 448, 561 TKIP is the WPA-1 (2003) cipher, not WPA2. 📄 p. 446

ADV · subtle error TF 📄 p. 449

Q7. Statement: In a WPA2-Personal (PSK) network used at a public hotel, an attacker sitting in the same Wi-Fi network can decrypt another victim's TLS-free HTTP traffic because the PTK is derivable from the PSK and the handshake nonces — even without breaking any cryptographic primitive.

▶ Show Answer

✅ Answer: TRUE.

Slide 449 states this weakness explicitly: "session key fully derivable — PSK + nonce from handshake. Vulnerable in public Wi-Fi: in hotel or conference site scenario, all users share the same PSK (password). If your application data is not encrypted (without TLS), attacker in the same LAN can extract sensitive data." This is precisely why WPA3-Personal's SAE handshake — which uses an ephemeral Diffie-Hellman exchange so the session key is no longer purely derivable from the password — provides forward secrecy. 📄 p. 449 The statement is subtle because it sounds like cryptography is broken, but actually no primitive is "broken" — the derivation itself is designed to be reversible given inputs the attacker can observe.

Part C — Short Answer (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG SA 📄 pp. 454, 565

Q8. Name the three components of the 802.1X framework and describe the role of each in one sentence.

▶ Show Answer

📝 Model Answer:

  • Supplicant — the client device (laptop, phone) that is requesting network access; it originates EAP Identity/Response messages.
  • Authenticator — the network edge device (Wi-Fi access point or wired switch port); it relays EAP messages between the Supplicant (over EAPOL) and the Authentication Server (over RADIUS), and it enforces the controlled/uncontrolled port split.
  • Authentication Server — the back-end service (typically RADIUS) that holds the user database, runs the actual authentication logic (EAP method), and returns Accept or Reject with keying material.

📄 pp. 454, 565

🔑 Key Points (Grading Checklist):

  • ☐ All three names correctly spelled and in correct role 📄 pp. 454, 565
  • ☐ States that the Authenticator relays only (does not decide) 📄 pp. 455-456, 566-567
  • ☐ States that the Authentication Server is the decision-maker 📄 pp. 454, 565
INT SA 📄 pp. 457, 568

Q9. List, in order, the six phases of 802.11i operation, and state in one clause what the main purpose of each phase is.

▶ Show Answer

📝 Model Answer:

  1. Discovery — the station finds nearby APs via beacons or probe requests and learns their security parameters. 📄 pp. 439-440
  2. Open authentication — a legacy null-authentication frame exchange; no real security check happens here in WPA2. 📄 p. 443
  3. Association — the station and AP agree on a cipher suite and auth method; station is now associated but the controlled port is still blocked. 📄 pp. 439-440
  4. 802.1X / EAP authentication — the supplicant proves its identity to the authentication server via the AP; on success the AS returns keying material (MSK). 📄 pp. 455-456, 460, 566-567, 571
  5. Key management (4-way handshake) — the supplicant and AP derive the PTK and install the GTK; the controlled port opens. 📄 pp. 457, 568
  6. Protected data transfer — all subsequent frames are encrypted and integrity-protected by AES-CCMP. 📄 p. 446

🔑 Key Points (Grading Checklist):

  • ☐ Six phases named in the correct order 📄 pp. 457, 568
  • ☐ States that controlled port opens between Phase 5 and Phase 6 📄 pp. 455, 566
  • ☐ Explicitly mentions AES-CCMP for Phase 6 📄 p. 446
  • ☐ Uses the bullet-layout the instructor requested in the final review script
ADV · challenge SA 📄 pp. 459, 570  +  pp. 227-230 (TLS)

Q10. Explain how EAP-TLS (used inside WPA2-Enterprise) reuses the TLS handshake mechanism from Topic 4. Specifically: what message in the TLS handshake authenticates the server to the client, and why does EAP-TLS require mutual authentication rather than just server authentication like HTTPS?

▶ Show Answer

📝 Model Answer:

EAP-TLS embeds a complete TLS handshake inside EAP messages. The server authenticates itself to the supplicant by sending its Server Certificate (step 5a of the TLS handshake 📄 pp. 230, 240), which the supplicant validates against a trusted CA just as in HTTPS 📄 pp. 104, 128, 145. However, unlike HTTPS — where the client typically authenticates via an application-layer password over the already- encrypted tunnel — in EAP-TLS the network itself must decide whether to admit this device to the LAN before any higher-layer protocol exists. Therefore the supplicant must present its own Client Certificate so the authentication server can verify the device's identity cryptographically during the same handshake. The final TLS master secret is then used to derive the Pairwise Master Key (PMK) that feeds into the 802.11i 4-way handshake. 📄 pp. 459, 570

🔑 Key Points (Grading Checklist):

  • ☐ Identifies Server Certificate as the TLS message that authenticates the server 📄 pp. 230, 240
  • ☐ Identifies Client Certificate (mutual auth) as the reason EAP-TLS differs from HTTPS 📄 pp. 459, 570
  • ☐ Explains why mutual auth is needed (network-layer admission before apps can run) 📄 pp. 451-456
  • ☐ Bonus: mentions the TLS master secret feeding into the 802.11i PMK / 4-way handshake 📄 pp. 457, 568

§5 — Topic Summary · Exam Cheat Sheet

📘 LECTURE PDF COVERAGE: pp. 435-464, 558-578 (highest-density review of the entire topic)

IN ONE SENTENCE, THIS TOPIC IS:

Wireless link-layer security uses WPA2's AES-CCMP to encrypt every Wi-Fi frame and uses 802.1X with an EAP method to decide who is allowed on the network via a back-end RADIUS authentication server. 📄 pp. 446, 448, 451-456, 561-567

THE THREE THINGS YOU MUST KNOW FOR THE EXAM:

  1. The three 802.1X components and the controlled vs uncontrolled port rule: Supplicant → Authenticator (AP, relay only) → Authentication Server (RADIUS, the decider). Uncontrolled port is always open (auth traffic); controlled port is blocked until Access-Accept + 4-way handshake complete. 📄 pp. 454-456, 565-567
  2. The WEP / WPA / WPA2 / WPA3 matrix of (Encryption × User Authentication), and the fact that Personal and Enterprise differ in user auth, not in encryption. WPA2-Personal = AES-CCMP + PSK; WPA2-Enterprise = AES-CCMP + 802.1X/EAP. 📄 pp. 448, 561
  3. EAP-TLS vs EAP-TTLS. EAP-TLS = both client and server certificates (mutual public-key auth). EAP-TTLS = only server certificate + a TLS tunnel inside which the client sends username/password. This is the challenge-question integration point with Topic 4 (TLS). 📄 pp. 459, 570

THE #1 MISTAKE STUDENTS MAKE:

Confusing the Authenticator (AP — a dumb relay) with the Authentication Server (RADIUS — the decider). The AP never inspects the inner EAP payload; the authentication method lives entirely between the supplicant and the AS. This is why you can upgrade from password to certificate authentication without touching the AP hardware. 📄 pp. 455-456, 566-567

IF YOU REMEMBER NOTHING ELSE, REMEMBER THIS:

"Supplicant asks, Authenticator announces, AS approves — and the controlled port stays locked until the AS says yes." 📄 pp. 454-456, 565-567
