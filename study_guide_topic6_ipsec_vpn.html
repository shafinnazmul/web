From scriptMidtermReview-cs548.txt: "You need to understand what security services provided by IPsec... the two different modes: transport mode and tunnel mode... what's the difference? Transport mode: IP addresses are NOT encrypted. Tunnel mode: inner IP is encrypted."

IPsec is the Layer-3 mirror of TLS — same cryptographic building blocks (AES, HMAC, (EC)DHE) but applied to entire IP packets instead of application-layer records. Two design choices distinguish it from TLS: (1) it lives in the kernel, transparent to applications, and (2) it offers a tunnel mode that hides the original IP header itself. Those two properties are exactly what makes IPsec the engine behind site-to-site VPNs and enterprise remote access. Expect MCQ/T-F density on the transport-vs-tunnel distinction, AH-vs-ESP comparison, and challenge questions linking IPsec's chain of trust to TLS's and DNSSEC's.

Her stated test philosophy: "the purpose of this class is to make sure you understand" — questions ask why each service exists (why AH when ESP already does MAC? why tunnel mode if transport is simpler? why traffic-flow confidentiality only in tunnel mode?) rather than memorise byte layouts.

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 355–383

1.1 The Problem This Solves 📄 pp. 356, 370

The IP protocol gives no authentication, integrity, or confidentiality. Every network-layer attack you studied in Topic 2 — IP spoofing, ICMP redirect, reset, hijacking — exploits that. TLS and SSH solve the problem per-application at upper layers, but require each application to be TLS-aware and cannot hide the who-talks-to-whom metadata. IPsec fills the Layer-3 gap: it adds cryptographic protection to IP packets themselves, transparently to every application, and can hide the source/destination addresses when deployed in tunnel mode. 📄 pp. 356, 370

1.2 The Core Idea — A Kernel-Level Crypto Shim Between IP and the Upper Stack 📄 pp. 370, 375

IPsec is a framework, not a single protocol. The framework consists of four cooperating components (slide 375):

  • Authentication Header (AH) — adds integrity + data-origin authentication + optional anti-replay. No encryption. 📄 pp. 375, 379
  • Encapsulating Security Payload (ESP) — adds encryption + integrity + anti-replay + optional limited traffic-flow confidentiality. The workhorse. 📄 pp. 375, 381
  • Security Association (SA) databases — the Security Association Database (SAD) holds the active cryptographic contexts; the Security Policy Database (SPD) holds the rules for which traffic gets which SA. 📄 pp. 375, 378
  • Internet Key Exchange (IKE / IKEv2) — negotiates the SAs (algorithms, keys, lifetimes), typically via an ISAKMP exchange over UDP 500/4500, using (EC)DHE + certificates or pre-shared keys. 📄 pp. 375, 386, 389

Because everything runs at Layer 3, applications see an ordinary IP socket; the kernel's IPsec stack applies AH/ESP transparently on send and strips it on receive. This is the "transparent to applications" benefit on slide 370. 📄 p. 370

1.3 Analogy — A Diplomatic Pouch System 📄 pp. 378, 381, 387

Think of the international diplomatic mail:

  • The SA is a bilateral agreement: "Embassy A ↔ Ministry B, pouch code 0xA1B2, these ciphers, these keys, valid until…"
  • The SAD is the thick binder in the mailroom listing every active bilateral agreement.
  • The SPD is the checklist at the dispatch desk: "If an envelope goes to Paris, use pouch 0xA1B2; if it's for the WHO, use 0xC9D4…"
  • AH is a tamper-evident wax seal: you can still read the address, but any tampering is obvious.
  • ESP is the locked diplomatic pouch itself: contents unreadable, seal intact.
  • IKE is the quarterly meeting where the two embassies rotate keys and agree on which ciphers/wax seals to use next.

1.4 Technical Definition 📄 pp. 370, 375, 378

IPsec is an IETF framework providing network-layer security via two protocols (AH and ESP) and a key-management component (IKE). A Security Association is a simplex (one-way) security relationship between two endpoints, uniquely identified by the triple (Security Parameters Index, destination IP address, security protocol identifier where identifier is either AH or ESP). Because it is one-way, bidirectional communication requires two SAs. Each SA entry in the SAD specifies the algorithms, keys, sequence counters, anti-replay window, lifetime, and mode (transport or tunnel). The SPD maps inbound/outbound traffic (selectors such as src/dst IP, protocol, ports) to one of three actions: BYPASS (send in the clear), DISCARD, or PROTECT-via-SA-X. 📄 pp. 370, 375, 378

Section 2 · How It Works — Architecture, SA Setup, AH & ESP on the Wire

LECTURE PDF COVERAGE: pp. 375–381, 395

2.1 End-to-End Life of an IPsec-Protected Packet

Phase 1 · IKE negotiates Security Associations (before any data flows)
Peers run IKE over UDP 500 (or 4500 when NAT is traversed): they authenticate each other via certificates or pre-shared keys, perform (EC)DHE to derive fresh key material, then negotiate crypto parameters (cipher, MAC, lifetime, mode).
Result: Two SAs are installed (one per direction) in both peers' SADs. Each has a unique SPI. 📄 pp. 375, 378, 389
Phase 2 · On packet send, SPD lookup decides policy
Outgoing IP packet hits the IPsec hook. SPD is searched with selectors (src IP, dst IP, protocol, ports). Three possible actions:
  • BYPASS — send as plain IP.
  • DISCARD — drop.
  • PROTECT — use the SA referenced by this SPD entry. 📄 pp. 375–378
Phase 3 · SA lookup in SAD; apply AH or ESP
Kernel reads the SA (algorithms, keys, counters, mode). Then either:
  • AH path — compute HMAC-ICV over immutable IP-header fields + payload; insert AH header with SPI, sequence number, ICV. 📄 pp. 379–380
  • ESP path — encrypt payload (and, in tunnel mode, the original IP header); append MAC; prepend ESP header with SPI and sequence number. 📄 p. 381
Increment the SA's sequence counter; if it would overflow, negotiate a new SA. 📄 pp. 378, 395
Phase 4 · Receiver: dispatch by SPI, verify, deliver
Receiver reads the AH/ESP header's SPI, looks up SA in its SAD. Checks:
  • Sequence number is inside the anti-replay window and has not been seen before (sliding window of width W, accept only if SN ≥ N − W + 1). 📄 p. 395
  • ICV / MAC verifies with the SA's key.
  • For ESP, decrypt the payload.
If any check fails → discard and audit. Otherwise, strip AH/ESP header and hand the inner packet to the normal IP stack. 📄 p. 378

Line-by-line annotation (from 📄 pp. 379–380):

  • Next Header — the protocol of the payload that follows AH. Common values: TCP = 6, UDP = 17, IP = 4 (this is tunnel-mode AH — inner IP packet follows), AH = 51 (nested AH). 📄 p. 379
  • Payload Len — length of AH in 32-bit words minus 2 (for IPv4) or 64-bit words minus 1 (for IPv6). Byte-level trivia the slide highlights. 📄 p. 379
  • SPI — 32-bit SA identifier; 0 = local use only, 1–255 reserved by IANA. 📄 p. 379

  • ICV — HMAC-SHA output. Computed over (a) immutable or predictable fields of the IP header (source address — immutable; destination address with source routing — mutable-but-predictable); (b) the AH header itself; (c) the upper-layer protocol data. Mutable fields like TTL and checksum are zeroed during ICV computation. 📄 p. 380

2.4 Encapsulating Security Payload (ESP) — Services & Structure 📄 p. 381

ESP adds encryption on top of everything AH does, and throws in limited traffic-flow confidentiality (in tunnel mode) as a bonus. Slide 381 lists exactly five services:

  1. Message content confidentiality — symmetric encryption of the payload (and inner IP header in tunnel mode).
  2. Data origin authentication — MAC over the ESP-protected fields proves the packet came from the SA peer.
  3. Connectionless integrity — any bit flip invalidates the MAC.
  4. Anti-replay service — via sliding sequence-number window (same as AH).
  5. Limited traffic-flow confidentiality — tunnel mode only; hides inner source/destination addresses.

3.1 What Exams Typically Test on IPsec Architecture + AH/ESP

  • "List the security services IPsec provides." — Instructor named this verbatim. The full list from p. 370: access control, data integrity, data-origin authentication, anti-replay, confidentiality, limited traffic-flow confidentiality. 📄 p. 370

  • "Difference between AH and ESP." — AH = integrity + auth only; ESP = integrity + auth + confidentiality + TFC. AH authenticates outer IP header; ESP does not. 📄 pp. 379, 381
  • "What identifies a Security Association?" — The triple (SPI, destination IP, AH/ESP). Memorise it. 📄 p. 378
  • "Why is an SA one-way?" — So each direction can use different keys / algorithms / lifetimes. Bidirectional comms needs two SAs. 📄 p. 378
  • "What does SPD vs SAD do?" — SPD holds policy (should this traffic be protected?); SAD holds state (how is it protected right now?). 📄 pp. 375, 378
  • "How does anti-replay work?" — Sliding sequence-number window of width W; accept SN ≥ N − W + 1 and not previously seen. 📄 p. 395

3.2 Common Misconceptions & Traps

MYTH: "AH encrypts the packet."

REALITY: AH provides authentication and integrity only — it never encrypts. If you see an option saying "AH encrypts the payload" on an MCQ, it's the distractor. ESP is the one that encrypts. 📄 p. 379

EXAM TRAP: Multi-answer question "Which of the following does AH provide?" with "confidentiality" as one of the options — do not select.

MYTH: "A Security Association is two-way."

REALITY: An SA is simplex (one-way). For bidirectional communication, each peer installs one SA for outbound traffic and one for inbound — so two SAs per pair of peers per protocol. This allows each direction to have independent keys and algorithms. 📄 p. 378

EXAM TRAP: T/F claiming one SA is enough for bidirectional traffic — FALSE.

MYTH: "The SPI alone uniquely identifies an SA."

REALITY: The SPI is unique per destination IP per protocol. The receiver uses the full triple (SPI, destination IP, AH/ESP) to look up the SA. Different peers can independently pick the same SPI. 📄 p. 378

EXAM TRAP: "An SA is uniquely identified by its SPI" — incomplete; mark false or choose the "triple" option.

MYTH: "ESP authenticates the outer IP header."

REALITY: ESP's ICV covers the ESP header and payload (and, in tunnel mode, the encrypted inner IP packet) — but not the outer IP header. AH does cover the outer header. That's the one and only reason AH still exists in the standard. 📄 pp. 380–381, 388

MYTH: "IPsec requires every application to be rewritten."

REALITY: IPsec is implemented in the kernel, below the socket API. Applications use ordinary TCP/UDP sockets; the kernel transparently encrypts/decrypts. This is the explicit "transparent to applications" benefit on slide 370 — a stark contrast with TLS. 📄 p. 370

MYTH: "Anti-replay works by remembering every sequence number ever seen."

REALITY: The receiver maintains only a sliding window of the most recent W sequence numbers (typically W = 64). Packets with SN < N − W + 1 are discarded; newer ones update the window. This gives O(1) memory and handles legitimate reordering up to W positions. 📄 p. 395

Section 5 · 6A IPsec Architecture + AH/ESP Cheat Sheet

IPsec is a four-component Layer-3 framework — IKE + SAD + SPD + (AH|ESP) — where the SPD chooses whether to protect a packet, the SAD holds the live one-way Security Association (keyed by SPI + dst IP + protocol), AH adds authentication over the IP header + payload without encrypting, and ESP adds encryption + auth + anti-replay over the payload (and in tunnel mode, the inner IP packet) to deliver the full six services: access control, integrity, origin auth, anti-replay, confidentiality, and limited traffic-flow confidentiality.

THE 3 THINGS YOU MUST KNOW

  1. Six services — access control · data integrity · data-origin auth · anti-replay · confidentiality · limited TFC. 📄 p. 370
  2. AH vs ESP — AH = auth only + covers outer IP header; ESP = auth + encryption + anti-replay + TFC, does not cover outer IP header. 📄 pp. 379, 381
  3. SA = (SPI, dst IP, AH/ESP) simplex; bidirectional comms = two SAs. 📄 p. 378

THE #1 MISTAKE STUDENTS MAKE

Saying "AH provides confidentiality." It never does — only ESP encrypts. Also: treating SA as two-way (it's not), or SPI as a global unique ID (it's unique only per dst+protocol). 📄 pp. 378–381

IF YOU REMEMBER NOTHING ELSE

"SPD decides whether. SAD decides how. AH signs. ESP seals."

6B Transport vs Tunnel Mode 📄 pp. 384–388  ·  🔴 EXPLICIT ⭐ Instructor named verbatim

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 384–388

1.1 The Problem This Solves 📄 pp. 386–387

IPsec's two authors faced a forked requirement: (a) two end hosts that both speak IPsec want to protect traffic between themselves without changing their visible IP addresses — they want end-to-end security at Layer 3; (b) two security gateways (firewalls, VPN concentrators) want to protect traffic from the hosts behind them, hiding those hosts' real IPs from the public Internet. One mode cannot serve both. The solution: two modes, same AH/ESP protocols. Transport mode addresses case (a); tunnel mode addresses case (b). 📄 pp. 386–387

1.2 The Core Idea — Who Wraps Whom 📄 pp. 386–387

  • Transport mode: the existing IP header stays visible and routable; AH/ESP is inserted between the IP header and the upper-layer payload. The payload (and, for ESP, any IPv6 extension headers after ESP) is protected. The original IP addresses are NOT encrypted — this is the instructor's exact wording. 📄 p. 386
  • Tunnel mode: the entire original IP packet (header + payload) becomes the inner packet, treated as a blob and encapsulated inside a new outer IP header. AH/ESP protects that whole inner packet. The inner IP addresses ARE encrypted by ESP in tunnel mode — observer sees only the outer (gateway-to-gateway) addresses. 📄 p. 387

1.3 Analogy — Locked Envelope vs Locked Pouch Inside a Bigger Envelope 📄 pp. 386–387

Imagine sending a confidential letter from Alice in San Francisco to Bob in Berlin:

  • Transport mode = you take your normal envelope (addressed "From: Alice, 123 Mission St · To: Bob, Unter den Linden"), lock just the letter inside it with a tamper-evident seal, and mail it. Postmen see who's talking to whom; they can't read the letter.
  • Tunnel mode = you put your addressed envelope inside a locked diplomatic pouch, then put that pouch inside a second envelope addressed "SF Consulate → Berlin Consulate." The public postmen see only consulate-to-consulate. Only the two consulates know who the letter is really for.

1.4 Technical Definition 📄 pp. 386–388

Transport mode inserts AH or ESP between the existing IP header and the upper-layer payload; the IP header is not encapsulated, only (optionally, in AH) authenticated. It's typically used for end-to-end host-to-host communication where both endpoints speak IPsec. Tunnel mode encapsulates the entire original IP packet as the payload of a new IP packet; AH or ESP is inserted between the new outer IP header and the encapsulated inner packet. ESP in tunnel mode encrypts the entire inner IP packet (thereby hiding inner source/destination addresses from on-path observers — "limited traffic-flow confidentiality"). Tunnel mode is used when at least one endpoint of the SA is a security gateway protecting hosts on a private network. 📄 pp. 386–387

The inner packet — including its own IP header — is encrypted wholesale. Observers see only the outer gateway-to-gateway IPs. This is the configuration that delivers limited traffic-flow confidentiality and is the basis of every site-to-site IPsec VPN. 📄 pp. 370, 387–388

Section 3 · Exam Intelligence

LECTURE PDF COVERAGE: pp. 384–388

3.1 What Exams Typically Test on Transport vs Tunnel

  • "What's the difference between transport mode and tunnel mode?" — Instructor asked this verbatim. Answer in one sentence: Transport mode leaves the IP header in the clear and protects only the payload; tunnel mode encapsulates the original packet inside a new outer IP header and (with ESP) encrypts the whole inner packet — hiding its addresses from observers. 📄 pp. 386–387
  • "Which mode gives traffic-flow confidentiality?" — ESP in tunnel mode only. Transport mode can never hide the addresses (the real header is visible); AH can never encrypt. 📄 pp. 370, 387
  • "Which mode is used for site-to-site VPN?" — Tunnel mode, because at least one SA endpoint is a security gateway (VPN concentrator) protecting hosts behind it. 📄 p. 387
  • "Which mode for end-to-end host-to-host?" — Transport mode is typical, because both hosts speak IPsec and there's no gateway. Tunnel mode is also allowed but adds unnecessary overhead. 📄 p. 386
  • "Does AH authenticate the outer IP header in tunnel mode?" — Yes, selected portions. This is one thing AH-in-tunnel uniquely provides that ESP-in-tunnel cannot. 📄 p. 388

3.2 Common Misconceptions & Traps

MYTH: "Transport mode encrypts the IP header if ESP is used."

REALITY: No — ESP in transport mode encrypts only the payload. The IP header is untouched. The instructor's verbatim quote: "Transport mode: IP addresses are NOT encrypted." If you want to hide the addresses, you need tunnel mode. 📄 p. 386

EXAM TRAP: T/F "ESP in transport mode encrypts the outer IP addresses" — FALSE.

MYTH: "AH in tunnel mode encrypts the inner IP header."

REALITY: AH never encrypts, in either mode. Tunnel mode AH authenticates the inner IP packet (and selected fields of the outer header), but the inner header is still readable to anyone with wire-level access to the packet. Only ESP in tunnel mode hides inner IPs. 📄 pp. 387–388

MYTH: "Tunnel mode is only for gateways."

REALITY: Tunnel mode is required when at least one SA endpoint is a gateway, but two hosts can also choose tunnel mode (e.g., a remote-access VPN client using tunnel mode to hide its real address). The slide's wording is "used when one or both ends are a security gateway." 📄 p. 387

MYTH: "Transport mode is more secure because it's simpler."

REALITY: Neither mode is strictly "more secure." Tunnel mode offers strictly more confidentiality (hides inner addresses); transport mode offers strictly less overhead (no outer header). The choice is driven by deployment topology, not by security strength. 📄 pp. 386–388

MYTH: "You can mix transport and tunnel mode in the same SA."

REALITY: The mode is a property of the SA, set during IKE negotiation. One SA = one mode. You can, however, chain SAs (e.g. transport-mode inside a tunnel-mode outer SA) — but that's two SAs. 📄 p. 378

Section 5 · 6B Transport vs Tunnel Mode Cheat Sheet

Transport mode inserts AH or ESP between the existing IP header and its payload, leaving the original IP addresses visible; tunnel mode wraps the entire original IP packet inside a new outer header, and — only when ESP is used — encrypts the inner header, delivering limited traffic-flow confidentiality. Tunnel mode is required whenever at least one SA endpoint is a security gateway, which is why every real-world site-to-site VPN uses ESP in tunnel mode.

THE 3 THINGS YOU MUST KNOW

  1. Instructor's verbatim rule — "Transport mode: IP addresses NOT encrypted. Tunnel mode: inner IP IS encrypted." 📄 pp. 386–387
  2. Only ESP-tunnel provides limited traffic-flow confidentiality — because only it encrypts an inner IP header. 📄 pp. 370, 387
  3. Gateway involved → tunnel required. End-to-end host-to-host defaults to transport, but may use tunnel for TFC. 📄 p. 387

THE #1 MISTAKE STUDENTS MAKE

Saying "AH in tunnel mode encrypts the inner IP header." AH never encrypts, in either mode. It authenticates the inner packet but the bytes are readable on the wire. 📄 pp. 387–388

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 356, 373–374, 389–391

1.1 The Problem This Solves 📄 pp. 373, 390

A company has a private network (say, 10.0.0.0/24) inside its office — file servers, printers, internal databases, all on non-routable RFC-1918 addresses. Three scenarios create pressure to extend access beyond the building: remote workers who need office resources from home, branch offices that must share resources, and mobile users who want privacy from untrusted Wi-Fi. Direct exposure of internal services to the Internet is a disaster waiting to happen. A Virtual Private Network (VPN) solves all three by tunnelling authenticated, encrypted traffic over the public Internet so that the effect is "logically connected to the private network" without the physical wiring. 📄 pp. 373, 390

1.2 The Core Idea — Encrypted Tunnel + Routing 📄 pp. 373–374, 389

A VPN is really just two mechanisms glued together:

  • An encrypted tunnel between the client (or branch router) and the VPN server, implemented by either IPsec (tunnel-mode ESP, typically with IKEv2) or a TLS-based protocol (OpenVPN, WireGuard, Cisco AnyConnect). 📄 p. 389
  • Routing / NAT that makes traffic sent through the tunnel appear to originate from the VPN server's private network, so internal services treat the remote user as "local." On the client, a virtual network interface (e.g. tun0) has an IP on the private subnet; the OS routing table sends traffic for the company out through it. 📄 pp. 374, 391

Slide 390 diagrams the "without VPN" case to show what leaks: a malicious tcpdump on the WAN sees the real source IP, destination IP, TCP ports, and HTTP payload of every connection. Slide 391 then shows how OpenVPN cures this — all the WAN sees is a UDP packet between the client's public IP and the VPN server's public IP; the real inner flow is hidden. 📄 pp. 390–391

1.3 Analogy — The Company Shuttle Bus 📄 pp. 373, 374

Picture a company with a private campus. Two options for off-site employees:

  • No VPN: they drive their own cars through public streets. Every turn, parking stop, and passenger is visible to onlookers.
  • VPN: the company runs a secure shuttle. The employee steps into a tinted-window bus at a public stop; the bus drives directly onto the private campus via a gated entrance. To outside observers, it's just "employee got on bus at stop X, bus went to the gate." Where they go inside campus — which building, which desk — is invisible from outside.

1.4 Technical Definition 📄 pp. 373, 389

A VPN is a virtual network overlay providing the appearance of a direct private-network connection over a shared public infrastructure (typically the Internet), using cryptographic tunnels for confidentiality, integrity, and authentication. Three archetypal deployments (slide 372/389): site-to-site (two branch-office gateways tunnel traffic between whole subnets); remote-access (individual client devices tunnel to a corporate concentrator); extranet (controlled cross-organisation site-to-site). The cryptographic backbone is either IPsec/IKEv2 (running natively at Layer 3, usually UDP 500/4500) or a TLS-based protocol such as OpenVPN (TLS over UDP/DTLS typically), Cisco AnyConnect, or WireGuard. 📄 pp. 373, 389

Step 1 · Client-side: application writes normal TCP/IP packet
The browser calls send() on a TCP socket aimed at the webserver. The kernel constructs an IP packet with source = tun0 address (10.0.10.5), destination = webserver, TCP header + HTTP payload. 📄 p. 391
Step 2 · Routing sends it out of tun0
The OS routing table was modified at VPN-connect time: default route now points to tun0. So the outbound packet is delivered to OpenVPN (user-space) reading from tun0 via read(). 📄 p. 391
Step 3 · OpenVPN wraps the inner packet in TLS + UDP
OpenVPN takes the entire inner IP packet as its application-layer payload, runs it through a TLS (OpenSSL) session with the VPN server for auth + encryption, then sends the result via a normal UDP socket to vpnserver:1194. Outer headers: source = client's public IP, dst = VPN server's public IP. 📄 p. 391
Step 4 · VPN server unwraps and NATs onward
At the server: UDP/IP arrives, OpenVPN user-space process decrypts via TLS, recovers the inner IP packet. Kernel routing delivers it out the server's external interface — but NAT rewrites the source to the server's own IP (so replies come back to the server). The NAT table records the mapping (VPN client → NAT src port) for de-multiplexing replies. 📄 p. 391
Step 5 · Reply path reverses
Webserver replies to the NATed source (the VPN server). The VPN server looks up the NAT table, re-rewrites the destination to the inner client IP, hands back to OpenVPN, which encrypts and sends UDP back over the WAN to the client. Client OpenVPN decrypts, writes to tun0, kernel delivers to the browser. Round trip complete. 📄 p. 391

2.3 IPsec-Based VPN — What Changes

For an IPsec/IKEv2 VPN, the architecture is similar but simpler and lives in the kernel:

  • No tun0 / user-space process. The kernel's IPsec module wraps the inner packet in ESP (tunnel mode), with the outer IP header addressing the VPN peer. 📄 pp. 387, 389
  • IKEv2 handshake first establishes two SAs (one per direction) over UDP 500/4500. 📄 p. 389
  • Everything below the socket API works the same. This is why IPsec is called "transparent to applications" — browsers see a normal socket. 📄 p. 370
  • AES-GCM is the typical AEAD cipher in modern ESP, same primitive as in TLS record layer. 📄 p. 392

3.2 Common Misconceptions & Traps

MYTH: "A VPN and TLS/HTTPS provide the same protection."

REALITY: They overlap on payload confidentiality but not on traffic-flow confidentiality. HTTPS hides what you send to bank.com, but an observer still sees you connecting to bank.com. A VPN hides the fact that you visit bank.com at all — the observer sees only client↔VPN-server. The two are complementary, not redundant. 📄 pp. 390–391

MYTH: "A VPN makes you anonymous on the Internet."

REALITY: A VPN hides your traffic from your ISP and shifts the "source" to the VPN server's IP. But the VPN operator itself can see everything. It is trust-shifting, not anonymisation. Tor, not VPN, is the anonymisation tool. 📄 p. 391

MYTH: "All VPNs use IPsec."

REALITY: Slide 389 lists two clear families: IPsec/IKEv2 and TLS-based (OpenVPN, AnyConnect, WireGuard). WireGuard in particular uses neither IPsec nor TLS — it uses the Noise protocol framework with modern crypto (Curve25519, ChaCha20-Poly1305). Don't assume "VPN" implies "IPsec." 📄 p. 389

MYTH: "The VPN encrypts my traffic end-to-end to the webserver."

REALITY: The VPN encrypts only the hop from client to VPN server. Beyond the VPN server, traffic re-emerges onto the public Internet (or into the private LAN) in whatever state it was — plain HTTP is plain HTTP once it leaves the VPN concentrator. For end-to-end encryption you still need TLS. 📄 p. 391

MYTH: "A VPN always uses tunnel mode."

REALITY: If it's IPsec-based, yes — VPNs are the canonical tunnel-mode use case. If it's TLS-based (OpenVPN), the notion of transport/tunnel mode doesn't apply at the IPsec level; the TLS tunnel itself plays the same role at a higher layer. 📄 pp. 387, 389

Sketch of a likely question: "You've learned five countermeasures for TCP attacks (randomised source ports, randomised ISNs, SYN cookies, TLS, IPsec). Which is the ultimate defence and why?" Answer: IPsec, because it encrypts the entire transport-layer payload so spoofed/injected packets lack the keys needed to pass the ICV.

Common threads across all three

  • All three use (EC)DHE for forward secrecy. Long-term keys sign or authenticate; ephemeral DH actually carries the secret. 📄 pp. 231, 252, 389
  • All three separate authentication from record protection. A handshake phase establishes identity + keys; a record/ESP phase uses those keys. 📄 pp. 215, 252, 375
  • All three tunnel something. TLS tunnels application records; SSH tunnels shell/SFTP/port-forwards; IPsec-tunnel-mode tunnels entire IP packets. 📄 pp. 233, 253, 387
  • Only IPsec hides Layer-3 metadata. That's the unique selling point — TLS and SSH can't hide destination IPs. 📄 pp. 370, 390
