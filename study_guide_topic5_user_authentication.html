§1 — Concept Overview

📘 LECTURE PDF COVERAGE: pp. 259-354 (three consecutive lectures dedicated to user authentication)

1.1 The Problem This Solves 📄 p. 261

Every system that enforces access control must first answer one question: "Is this person really who they claim to be?" That process is user authentication — verifying personal identity before granting access 📄 p. 261. The challenge is that the claimed identity is transmitted over an insecure network to a server who has never met the claimant in person, so the system must use cryptographic mechanisms — symmetric encryption, public-key encryption, or cryptographic hash functions — to bind a claim to proof. 📄 p. 261 The harder the cryptography, the stronger the binding.

1.2 The Core Idea — Scaling Authentication 📄 pp. 270, 294, 298

User authentication splits into two very different scales. Many-to-one is what happens when N users log into one particular service — e.g., SSH into a specific server, or TLS-connect to one website. Many-to-many is when N users need to authenticate to M different services without registering a separate credential at each one — this is Single Sign-On (SSO) territory. 📄 pp. 270, 294

The course teaches three fundamentally different architectural answers:

  • Many-to-one, asymmetric — X.509 certificates (HTTPS) and SSH public-key authentication. The server trusts a known CA or a pre-registered public key. 📄 pp. 263, 265
  • Many-to-many, web-style (asymmetric + PKI) — OpenID Connect for who you are and OAuth 2.0 for what you can do. Based on JSON Web Tokens and public-key signatures. 📄 pp. 271-275, 298
  • Many-to-many, enterprise-style (symmetric-only) — Kerberos with a centralized Key Distribution Center. No public-key math anywhere — only shared secrets and clever key-distribution choreography. Used by Active Directory and most corporate networks. 📄 pp. 298, 311

The naïve alternative — every server knows every user's password — fails on both security and manageability: one server compromise leaks all users, and changing your password means contacting every server 📄 p. 270. Everything in this topic exists to avoid that outcome.

1.3 Analogy — The Hotel Check-In Concierge 📄 pp. 300, 302, 306-308

Think of Kerberos as a hotel with an unusual security arrangement:

  • You (Joe the Guest) = the Client. You show up with a password that only the front desk knows. 📄 pp. 302, 306
  • Front desk (Authentication Server, AS) = validates your identity once in the morning and hands you a special "hotel concierge letter" (the TGS ticket). You never see your password on the wire again today. 📄 p. 306
  • Concierge (Ticket-Granting Service, TGS) = reads the concierge letter and issues individual "service vouchers" (service tickets) for each facility you want to use — pool, gym, spa. 📄 p. 307
  • Service tickets = sealed envelopes written in a language only the specific service (pool attendant) can read. You carry them but cannot forge or alter them. 📄 pp. 300, 307
  • Authenticator = a signed Post-it note you attach to each ticket saying "it's me, here's the exact time" — proves you didn't just steal someone else's ticket from the lobby trash. 📄 pp. 310, 321
  • Mutual authentication = the pool attendant echoes back "yes, this is the real pool" by sending you a note signed with the pool's key — defends against a fake pool set up by an imposter. 📄 p. 308

The elegance: you type your password once per day (to the front desk), and every service you use afterward is authenticated without typing a password again. This is the "single logon" property 📄 p. 306 and it's why Kerberos powers every enterprise network running Active Directory.

1.4 Technical Definition 📄 pp. 261, 311

User authentication is a protocol in which a claimant proves possession of a secret bound to an identity to a verifier, without revealing the secret itself to an eavesdropper or to the verifier beyond what is necessary 📄 p. 261. The secret can be a password (shared), a private key (asymmetric), a symmetric session key (via a third party like Kerberos), or a cryptographic token (JWT). Authentication answers "who are you?"; authorization (distinct concept) answers "what are you allowed to do?" 📄 pp. 262, 294. Kerberos is a centralized authentication service (MIT, RFC 4120 for v5) that uses only symmetric cryptography to provide mutual authentication between users and services on an untrusted network, via a trusted Key Distribution Center 📄 p. 311.

Kerberos ⭐⭐ Centralized, symmetric-only authentication service developed at MIT. Provides mutual authentication of users to services via a trusted Key Distribution Center. Heaviest exam emphasis in this topic. Instructor will give you the diagram; you must read it. 📄 pp. 298, 311
KDC (Key Distribution Center) The trusted third party holding every principal's long-term key. Hosts both AS and TGS. Single point of trust. Single point of compromise. 📄 pp. 302, 306
AS (Authentication Server) The Kerberos component that validates the user's initial credential (password → K_c) and issues a TGS ticket. Lives inside the KDC. User only interacts with AS once per day (typically at login). 📄 pp. 302, 306, 318
TGS (Ticket-Granting Service) The Kerberos component that issues per-service tickets on demand after the user has already obtained a TGS ticket from AS. Instructor: "Why do we have AS, TGS, and service-server separate?" — separation of concerns. 📄 pp. 302, 307, 319
V (Service Server) The actual application the user wants to access (file server, printer, email, etc.). Holds its own long-term key K_v shared with TGS. Think of V as "the pool, spa, gym" in the hotel analogy. 📄 pp. 303, 308
K_c (long-term key of client) ⭐ Derived from the user's password via a key-derivation function. Known only to the client and the KDC. "Who knows K_c?" — Client and AS/KDC. Not TGS alone, not V. 📄 pp. 303, 345
K_TGS (long-term key of TGS) Master secret of the Ticket-Granting Service. Known only to the KDC and TGS itself. Used to encrypt every TGS ticket so only TGS can decrypt it. 📄 pp. 303, 345
K_v (long-term key of service V) Master secret of service server V. Known to V and to TGS. Each service has its own K_v. Used to encrypt the service ticket. 📄 pp. 303, 345
K_c,TGS (short-term session key) ⭐ Fresh session key created by AS for secure communication between client and TGS. Known to client and TGS. Two-entity session key ⇒ both entities know it. Embedded in the TGS ticket. 📄 pp. 303, 345
K_c,v (short-term session key) ⭐ Fresh session key created by TGS for secure communication between client and service V. Known to client and V. Embedded in the service ticket. Used for the actual secure application session. 📄 pp. 303, 345
Ticket ⭐ A sealed envelope the client carries to a server as proof of authentication. Encrypted with a key the client cannot read. Instructor: "What's inside the ticket? Why is it encrypted with that key? Who can decrypt?" 📄 pp. 300, 320, 322
Authenticator ⭐⭐ A freshly generated token (ID_c, Addr_c, timestamp) encrypted under the session key, attached to each ticket use. Has a very short lifetime. Instructor asked directly: "What's the purpose of the authenticator?" — Answer: proves the ticket presenter is the rightful owner (the same client who obtained the ticket) + prevents replay. 📄 pp. 310, 319, 321, 322
Implicit authentication Authentication demonstrated by the ability to decrypt a message. If you can read it, you must know the key. Example: client's ability to decrypt K_c-encrypted AS reply proves they know the password. 📄 pp. 306, 310 📄 pp. 306, 310
Explicit authentication / Mutual auth Authentication demonstrated by sending a specific reply the other party can verify. In Kerberos: server returns E_Kc,v(timestamp + 1) to prove it knows K_c,v. This closes the loop so the client knows the server is genuine, not an imposter. 📄 pp. 308, 310
Realm A Kerberos administrative domain. Each realm has its own KDC and its own population of principals. "Principal = service/user name + instance + realm" 📄 p. 325
Principal A named entity in Kerberos (a user, service, or host). Identified by a principal name. A Kerberos environment has: one KDC, many clients (each with K_c), many services (each with K_v). 📄 p. 325
Active Directory Microsoft's enterprise directory service. Uses Kerberos as its default authentication protocol. The "where do we actually use Kerberos?" answer. 📄 pp. 298, 313
Single Sign-On (SSO) One login event grants access to many services. Kerberos calls it "single logon." Kerberos does it via the reusable TGS ticket; web does it via ID Tokens. 📄 pp. 272, 306

§2 — How Each Method Works (Step-by-Step)

📘 LECTURE PDF COVERAGE: pp. 262-354 (three lectures, with Kerberos dominating pp. 292-342)

Authentication methods in this topic vary wildly in architecture — certificates, shared secrets, JSON tokens, or a choreography of symmetric keys around a trusted third party. The common pattern: each method has a mechanism, what-you-need-to-authenticate, and where it's used. Focus especially on Kerberos (§2c) — the instructor was explicit that she will test your ability to read its message-exchange diagram.

2aMany-to-One Authentication Methods 📄 pp. 262-269 · Lecture 11 part 1

X.509 Certificate Authentication (HTTPS / Web) 📄 p. 263

The server presents an X.509 certificate signed by a Certificate Authority (CA). The client verifies the signature chain back to a pre-trusted Root CA in its store. The standard does not mandate a specific algorithm but recommends RSA 📄 p. 263. In HTTPS, this is combined with session cookies to maintain the authenticated session after the initial TLS handshake — the cookie-based approach is the baseline for "many-to-one" web authentication 📄 pp. 263, 294.

Public-Key Authentication (SSH / GitHub) 📄 pp. 265-268

 SSH / GitHub public-key authentication       📄 pp. 265-268

   Client (you)                              Server (SSH / GitHub)
   ─────────                                  ─────────────────────
   holds: private key (sk)                   holds: public key (pk) or
                                                    hash of pk pre-registered

   ─── connect, offer: my pk fingerprint ──►
                                              check: does this pk appear
                                              in authorized_keys (SSH)
                                              or in my registered keys (GitHub)?

   ◄── challenge: random nonce R ────────────

   ─── response: Sign_sk(R) ──────────────────►
                                              verify Sign_sk(R) with pk
                                              ─── authenticated ────────────►
SOURCE: 📄 pp. 265-268

The elegance of GitHub's SSH approach: you register only the public key hash in your GitHub account. Your private key never leaves your machine. Every push/pull triggers the sign-the-challenge dance above. The instructor called out: "You can register the secure hash value of your public key on GitHub, so you don't need to enter your password every time." 📄 p. 266

Password Authentication & Two-Factor 📄 pp. 269, 294, 344

Classic password auth is the weakest link — vulnerable to phishing, brute force, replay, and database breaches. The course's recommended strengthening is 2FA: combine "something you know" (password) with "something you have" (phone code, hardware key) or "something you are" (biometric) 📄 pp. 267, 294. Modern protocols (Kerberos, OpenID) deliberately avoid transmitting the password at all after the initial login.

2bOpenID Connect & OAuth 2.0 📄 pp. 271-291 · Lecture 11 part 2

The Core Distinction — ID Token vs Access Token 📄 pp. 272, 298

OpenID Connect and OAuth 2.0 are often confused because they share the same underlying HTTP redirect flow. The critical distinction the instructor demanded you memorize:

ProtocolQuestion it answersToken issuedUsed for
OpenID Connect "Who is this user?" ID Token (JWT with sub, email, name) Single Sign-On, user authentication
OAuth 2.0 "What can this app do on the user's behalf?" Access Token (granting scoped API permissions) Access control / authorization

Both are built on JSON Web Tokens (JWTs), a three-part structure:

 JWT structure:   base64url(Header) . base64url(Payload) . Signature    📄 p. 272

   Header    = { "alg": "RS256", "typ": "JWT" }
   Payload   = { "sub": "user123", "iss": "https://login.microsoft.com",
                 "aud": "outlook-client", "exp": 1740000000, ... }
   Signature = RSA_sign(private_key, Header + "." + Payload)
SOURCE: 📄 p. 272

2cKerberos — The Core Flow 📄 pp. 292-312 · Lecture 12 (instructor's most emphasized material)

The "Who Knows Which Key" Table 📄 pp. 303, 345

⚠️ Direct instructor exam pattern "Given a key, you will need to know: is this the long-term key or session key? If there is only one entity that is the long-term key, belonging to for example client or TGS or the service server. But if you see two entities, that is the short-term session key for the communication between C and TGS. Also, you need to know for example for K_c, the client and AS know K_c. That means TGS or V don't know K_c."

Key Type Held by Client Held by AS Held by TGS Held by Service V Purpose
K_c Long-term ✅ (derived from password) ✅ ❌ ❌ Encrypts AS reply to client
K_TGS Long-term ❌ ✅ ✅ ❌ Encrypts the TGS ticket (client can't read)
K_v Long-term ❌ ❌ ✅ ✅ Encrypts the service ticket (client can't read)

Rule of thumb the instructor gave: long-term keys are subscripted with one entity (K_c, K_v, K_TGS); session keys are subscripted with two entities separated by a comma (K_c,TGS or K_c,v). 📄 p. 303

Step-by-Step Walk-Through

Phase 1 — Login (Messages 1 + 2) 📄 pp. 306, 318
What happens: User types password ONCE at the start of the day. Client sends (ID_c, ID_TGS, timestamp) to AS. AS looks up K_c from its database, generates a fresh session key K_c,TGS, builds the opaque TGS ticket (encrypted with K_TGS so only TGS can read it), and sends it all back encrypted with K_c.
Implicit auth: Only the real client can decrypt the K_c-encrypted reply, so completing this step proves the user knew the password. 📄 p. 310
Result: Client now holds K_c,TGS and Ticket_TGS. K_c can be discarded from memory. Password is not re-used until next login.
Phase 2 — Obtaining Service Ticket (Messages 3 + 4) 📄 pp. 307, 319
What happens: Client wants to access service V (e.g., printer). Client presents Ticket_TGS + a freshly-minted Authenticator (encrypted with K_c,TGS, containing ID_c, AD_c, current timestamp). TGS decrypts Ticket_TGS with K_TGS, extracts K_c,TGS, then decrypts the Authenticator to verify the timestamp is fresh and the ID/address match the ticket.
Why the authenticator? Without it, anyone who stole Ticket_TGS from the wire could replay it. The authenticator proves the ticket presenter still knows K_c,TGS — which only the original client does. 📄 pp. 310, 321
Result: TGS issues a fresh K_c,v plus Ticket_v (encrypted with K_v so only V can read). Wrapped in K_c,TGS so only the client can unwrap.
Phase 3 — Accessing Service V (Messages 5 + 6) 📄 pp. 308, 322
What happens: Client presents Ticket_v + a new Authenticator (encrypted with K_c,v this time). V decrypts Ticket_v with K_v, extracts K_c,v, decrypts Authenticator, verifies freshness and identity. V then sends back E_K_c,v { TS_5 + 1 } — the incremented timestamp encrypted under K_c,v.
Mutual authentication: By returning TS+1 encrypted under K_c,v, the server proves it could decrypt Ticket_v — which requires knowing K_v — which means it must be the real V, not an imposter. This is explicit authentication of server to client. 📄 pp. 308, 310
Result: Both parties share K_c,v and can proceed with an authenticated, optionally encrypted application session.
The elegance of this design (instructor's framing) "The client only needs to obtain the TGS ticket once — say, every morning. Use the TGS to get tickets for any network service. Single logon — user enters password once per day." The entire protocol is engineered so that a user's password never crosses the wire after initial login and a compromised ticket expires within minutes, not days. 📄 pp. 302, 306

Remaining Threats (even with full Kerberos) 📄 p. 352

3.2 — Common Misconceptions & Traps

❌ MYTH: "OpenID Connect and OAuth do the same thing — they're just different brand names."

✅ REALITY: They solve different problems. OpenID Connect answers "who is this user?" and issues an ID Token (JWT identifying the user). OAuth 2.0 answers "what can this app do on behalf of the user?" and issues an Access Token (granting scoped API access). They're often used together — OpenID Connect is actually built on top of OAuth — but the tokens and their purposes are distinct. 📄 pp. 272, 298

🎯 EXAM TRAP: "OAuth 2.0 is primarily for user authentication" — FALSE. OAuth is for access delegation (authorization).

❌ MYTH: "Kerberos uses public-key cryptography like TLS does."

✅ REALITY: Kerberos uses symmetric cryptography only. The instructor called this out explicitly: "Relies exclusively on symmetric encryption, making no use of public-key encryption." This is a defining architectural choice — everything is based on shared secrets and a trusted KDC. The downside: scaling requires cross-realm trust relationships. 📄 pp. 298, 311

🎯 EXAM TRAP: "What kind of key exchange does Kerberos use?" Correct answer: Kerberos distributes keys via a trusted third party (KDC), NOT via Diffie-Hellman or RSA.

❌ MYTH: "The ticket alone is enough to authenticate the client."

✅ REALITY: A ticket only proves "someone who obtained a ticket is presenting it" — it does not prove you are the one who obtained it. Anyone who intercepts a ticket on the wire could replay it. The authenticator (freshly generated, encrypted under the session key, with current timestamp) is what proves the presenter actually knows the session key embedded in the ticket — i.e., is the original holder. 📄 pp. 310, 316, 321

🎯 EXAM TRAP: Given the simplified Kerberos without authenticator, "What's the security flaw?" → Ticket replay attack.

❌ MYTH: "In Kerberos, the client and the service server share a long-term key directly."

✅ REALITY: Client and service V share only a short-term session key (K_c,v) created fresh by TGS for each session. Client has no long-term key with V — that's the whole point of the KDC mediation. The long-term keys K_c and K_v are both held with KDC, not with each other. 📄 pp. 303, 307

🎯 EXAM TRAP: Fill-in-the-blank or MCQ: "Which entities hold the long-term K_v?" → Only V and TGS/KDC. Not the client.

❌ MYTH: "Mutual authentication happens at the start of the Kerberos flow."

✅ REALITY: Mutual authentication is only achieved at Message 6 — when the service V replies with E_K_c,v(TS+1), proving it could decrypt Ticket_v (thus knows K_v, thus is the real V). Messages 1-5 are one-directional proofs from client to server. 📄 pp. 308, 310

🎯 EXAM TRAP: "At which step is mutual authentication achieved?" → Message 6, not Message 1 or 2.

❌ MYTH: "The TGS ticket expires quickly to enforce security."

✅ REALITY: The TGS ticket is deliberately long-lived (hours, typically a full workday) — that's what enables single sign-on. It's the authenticator that has a very short lifetime (seconds to a minute) to prevent replay. The ticket defines "you're authorized to request services"; the authenticator proves "you're doing so right now." 📄 pp. 306, 321, 322

🎯 EXAM TRAP: Do not conflate ticket lifetime with authenticator lifetime — they serve different purposes and have very different time scales.

3.5 — Memory Aids

Mnemonic — Long-term vs Session keys = "1 vs 2 entities" If a key has one entity in its subscript (K_c, K_v, K_TGS) it's a long-term key belonging to that entity. If a key has two entities separated by a comma (K_c,TGS or K_c,v) it's a short-term session key shared between those two entities. This was the instructor's exact rule-of-thumb in the review. 📄 p. 303
Mnemonic — "Ticket vs Authenticator" = "Passport vs Fresh Selfie" The ticket is like a passport — issued in advance, has a picture and lifetime, carries your authorization. The authenticator is like a fresh selfie you take on the spot, proving you are the one holding the passport right now. Without the selfie, anyone who steals the passport can use it. 📄 pp. 310, 321
Mnemonic — "OpenID vs OAuth" = "IDentity vs Access" OpenID has "ID" right in the name — it's about who the user is. OAuth contains "auth"-orization — it's about what the app may do. Built together but distinguishable by their tokens: ID Token (OpenID) vs Access Token (OAuth). 📄 pp. 272, 298
Mnemonic — "Three-part Kerberos principal" = Name + Instance + Realm A Kerberos principal name has the format name/instance@REALM, e.g., joe/admin@UA.EDU. Memorize the three pieces — they're often asked in fill-in-the-blank. 📄 p. 325
Mnemonic — "Kerberos phases" = "AS → TGS → V" = "ID once, Ticket per service, Session per use" Three phases, three-part motto: ID yourself to the AS once per day → get tickets from TGS per service → establish sessions with each V per use. 📄 pp. 302, 306-308
Mnemonic — "Mutual auth = Message 6" In Kerberos, mutual authentication of server to client happens only at the last message (Msg 6), when V returns E_K_c,v(TS+1). Until then, only the client has been proven to the server. If an exam question says "when is mutual authentication achieved?", the answer is Message 6. 📄 pp. 308, 310

Q4. Kerberos uses symmetric-only cryptography, unlike TLS which uses a hybrid of asymmetric (for key exchange and signatures) plus symmetric (AES-GCM for bulk traffic). Which of the following is the primary practical reason Kerberos made this architectural choice?

  1. Symmetric cryptography is always more secure than asymmetric cryptography.
  2. Kerberos was developed in the 1980s at MIT before public-key cryptography was mature and widely deployable, and the centralized KDC architecture makes symmetric-key distribution tractable within a trust domain.
  3. The MIT creators wanted Kerberos to run on hardware without floating-point units.
  4. Symmetric ciphers are required by the IEEE 802.11 standard, which Kerberos uses as its transport.
▶ Show Answer

✅ Correct Answer: B) Historical context + KDC architecture makes it tractable.

📝 Explanation:

  • Why B is correct: Slide 300 explicitly notes: "Hash functions would've worked well, but this is 1980s design." Kerberos predates practical public-key deployment. The KDC sits at the center of a trust domain and already knows every principal's long-term key — this makes symmetric key distribution feasible in a way that it isn't on the open Internet (where no such central authority exists). 📄 pp. 298, 300, 311 TLS, in contrast, operates between parties that may have no pre-existing trust relationship, which is why it needs certificates + DH. 📄 pp. 150, 181
  • Why A is wrong: Symmetric ciphers are not inherently more secure than asymmetric at equivalent strength levels. They're faster, not stronger.
  • Why C is wrong: No such hardware constraint is relevant to Kerberos.
  • Why D is wrong: 802.11 is the Wi-Fi standard — Kerberos predates it by decades and operates over any TCP/IP transport.

Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG TF 📄 pp. 262, 294

Q5. Statement: Authentication and authorization are synonyms in modern identity systems — they both answer the question "Can this user perform this action?"

▶ Show Answer

✅ Answer: FALSE.

They are two distinct processes. Authentication answers "who are you?" — it verifies claimed identity. Authorization answers "what are you allowed to do?" — it checks whether the authenticated identity has permission for a specific action. Authentication must succeed first; only then can authorization apply its policies. Slide 294 states this verbatim: "Authentication: who are you. Authorization: what you are allowed to do." Conflating the two is one of the instructor's flagged traps. 📄 pp. 262, 294

INT TF 📄 pp. 306, 302

Q6. Statement: In Kerberos, the user must re-enter their password every time they request access to a new network service (printer, file server, email, etc.).

▶ Show Answer

✅ Answer: FALSE.

The defining feature of Kerberos is exactly the opposite — single logon. The user enters their password once per day (or per TGS-ticket lifetime) to the Authentication Server. After that, the client holds the TGS ticket and session key K_c,TGS; any subsequent service request (printer, email, file server) uses TGS to obtain a service ticket automatically, with no further password entry. Slide 306: "Client only needs to obtain TGS ticket once — say, every morning. Single logon — user enters password once per day." This separation of AS from TGS is precisely what enables single sign-on. 📄 pp. 302, 306

ADV · subtle TF 📄 pp. 308, 310

Q7. Statement: In Kerberos, mutual authentication between the client and the service server is established as soon as the client successfully presents its service ticket (Message 5), because the service can verify the ticket was signed by TGS.

▶ Show Answer

✅ Answer: FALSE.

Message 5 only authenticates the client to the service server. At that point, the service has verified the client knows K_c,v (because the client produced a valid authenticator), but the client has not yet verified it's talking to the real V. The client could be talking to an imposter who intercepted Message 5.
Mutual authentication is achieved only at Message 6, when V returns E_K_c,v(TS_5 + 1). By producing this reply, V proves it could decrypt Ticket_v — which requires K_v — which only the real V (and TGS) possesses. Slide 308 makes this explicit: "Authenticates server to client. Server can produce this message only if he knows key K_c,v. Server can learn K_c,v only if he can decrypt service ticket. Server can decrypt service ticket only if he knows correct key K_v. If server knows correct key K_v, then he is the right server." 📄 pp. 308, 310

Subtle trap: tickets are encrypted with K_v, not signed by TGS. There are no digital signatures anywhere in Kerberos — symmetric crypto only.

INT SA 📄 pp. 300, 310, 314-316, 321

Q9. Consider a simplified version of Kerberos that omits the authenticator — clients present only the encrypted ticket to service servers. Explain what specific attack becomes possible against this design, and describe how the authenticator in full Kerberos defeats it.

▶ Show Answer

📝 Model Answer:

Without the authenticator, a ticket replay attack becomes possible. An attacker who sniffs a legitimate client-to-service exchange captures the encrypted ticket (Ticket_v). Even though the attacker cannot decrypt it (they don't know K_v), they can simply replay the same ticket bytes at a later time — and the service server, seeing a well-formed ticket it can decrypt to a valid session key, would accept it without being able to distinguish the replay from the original request. 📄 pp. 310, 316

The authenticator closes this hole. It is a fresh data block — E_K_c,v(ID_c, AD_c, timestamp) — generated per use, encrypted under the session key K_c,v that is embedded in the ticket. To produce a valid authenticator, the presenter must actually know K_c,v, which only the original client does (because only the original client received the TGS reply containing K_c,v). The timestamp inside the authenticator, combined with its very short lifetime (seconds to a minute), prevents replay: the service verifies the timestamp is fresh and rejects stale authenticators. A replay attacker, holding only the ciphertext of the old authenticator, cannot mint a new one with a current timestamp. 📄 pp. 310, 321, 322

🔑 Key Points (Grading Checklist):

  • ☐ Names "ticket replay attack" as the specific vulnerability 📄 pp. 310, 316
  • ☐ Explains attacker sniffs the encrypted ticket but cannot read it 📄 p. 300
  • ☐ States authenticator is encrypted under session key K_c,v 📄 p. 321
  • ☐ Mentions timestamp + short lifetime for freshness 📄 p. 322
  • ☐ Notes only the original client can produce a valid authenticator (knows K_c,v)
ADV · challenge SA 📄 pp. 302, 306, 345  +  cross-topic

Q10. Explain why Kerberos separates the Authentication Server (AS) from the Ticket-Granting Service (TGS), even though both components live inside the same KDC. Your answer should address: (1) what would break if AS and TGS were merged into a single entity that directly authenticates the user every time, (2) how the separation enables the "single logon" property, and (3) how it affects exposure of the user's long-term key K_c.

▶ Show Answer

📝 Model Answer:

(1) Without separation. If AS and TGS were merged such that the KDC re-authenticated the user (via K_c and the password-derived key) every time a service ticket is requested, then the user's long-term key K_c would need to be used — and potentially re-derived from the password — on every service request. This means K_c (or operations involving K_c) happens many times per day, enlarging the window in which password-equivalent material is present in the client's memory and on the wire. It also breaks the clean architectural separation that lets KDC components scale independently. 📄 pp. 302, 306

(2) The "single logon" property. By separating AS from TGS, the user performs the password-based step only once per TGS-ticket lifetime (typically a workday). The AS hands out a TGS ticket + session key K_c,TGS. All subsequent service requests use K_c,TGS (not K_c) to authenticate to TGS. The password itself is used precisely once per day. This is exactly the "single logon" Kerberos is famous for — one authentication grants access to arbitrarily many services. 📄 p. 306

(3) K_c exposure reduction. The client's memory needs to hold K_c (or the password that derives it) only during the brief AS exchange at login. Afterward, K_c can be zeroed from memory; only K_c,TGS and per-service K_c,v are kept live. Compromise of client memory mid-day leaks only session keys — whose validity expires with the ticket — rather than the password-equivalent long-term key. This fundamentally limits the blast radius of a workstation compromise. 📄 pp. 303, 306

🔑 Key Points (Grading Checklist):

  • ☐ Identifies K_c exposure as the central concern 📄 pp. 303, 306
  • ☐ Explains TGS uses K_c,TGS (not K_c) for subsequent authentications 📄 p. 307
  • ☐ Links the separation to "single logon" explicitly 📄 p. 306
  • ☐ Notes ticket lifetime bounds blast radius of session-key compromise 📄 p. 306
  • ☐ Bonus: mentions architectural scalability (AS and TGS loads differ)

Q11. In web-based many-to-many authentication, OpenID Connect issues an ____________ to identify who the user is, while OAuth 2.0 issues an ____________ to grant an application scoped permissions on the user's behalf.

▶ Show Answer

✅ Answers: (1) ID Token (JWT carrying identity claims); (2) Access Token (JWT granting API access).

📝 Why: Slide 272 lists the three things Web-Auth 2.0 tokens can carry — Random Number, ID token, Access token — and slide 298 distinguishes OpenID Connect (ID token) from OAuth (access token). Both are JWTs but serve fundamentally different purposes. 📄 pp. 272, 298

Q12. In Kerberos, the three components that together constitute the full authentication architecture are the ____________ (which validates the user's password once per day), the ____________ (which issues per-service tickets after initial login), and the ____________ (the actual application the user ultimately wants to reach).

▶ Show Answer

✅ Answers: (1) Authentication Server (AS); (2) Ticket-Granting Service (TGS); (3) Service Server V.

📝 Why: Slide 302 introduces the two-step architecture: "Prove identity once to obtain a special TGS ticket. Use TGS to get tickets for any network service." The AS lives inside the KDC alongside TGS, but functionally they play distinct roles — AS does password-based authentication, TGS issues service tickets without re-consulting the password. Service servers V are the endpoints the user ultimately wants to use (file server, printer, email, etc.). 📄 pp. 302, 306-308

Q13. In Message 2 of the Kerberos exchange, the AS returns a reply encrypted with ____________, a key derived from the user's password. The ability to decrypt this message demonstrates ____________ authentication — a form of proof based on key possession rather than explicit challenge-response.

▶ Show Answer

✅ Answers: (1) K_c (the client's long-term key, derived from the user's password via a key-derivation function); (2) implicit (sometimes written as "implicit authentication").

📝 Why: Slide 306 walks through Message 2: "Encrypt_Kc(K_c,TGS, ID_TGS, time_KDC, lifetime, ticket_TGS) — Implicit authentication: only someone who knows K_c can decrypt." This is the purest form of implicit authentication in Kerberos. The user proves possession of the password-derived key K_c simply by successfully decrypting the AS reply — no separate challenge-response handshake is required. 📄 pp. 306, 310

§5 — Topic Summary · Exam Cheat Sheet

📘 LECTURE PDF COVERAGE: pp. 259-354 (condensed view of the entire user-authentication spectrum)

IN ONE SENTENCE, THIS TOPIC IS:

User authentication proves who someone is (distinct from authorization, which decides what they can do) — done at scale via three architectures: web-based SSO (OpenID Connect's ID Token + OAuth's Access Token, both JWTs over PKI) and enterprise SSO (Kerberos with a centralized KDC, using only symmetric crypto plus tickets, authenticators, and timestamps). 📄 pp. 262, 272, 298, 311

THE THREE THINGS YOU MUST KNOW FOR THE EXAM:

  1. The Kerberos key table — which key is long-term vs session, and which entities hold each one. One subscript = long-term; two subscripts = session. Long-term keys live in exactly two places (entity + KDC component). 📄 pp. 303, 345
  2. The 6-message Kerberos flow — the instructor said she'll give you the diagram and ask you to read it. Know which key encrypts each message, who can decrypt what, why each element (ticket, authenticator, timestamp, lifetime) exists. Mutual authentication happens only at Message 6. 📄 pp. 305-308
  3. OpenID Connect vs OAuth 2.0 — OpenID for identity (ID Token, "who"); OAuth for authorization (Access Token, "what"). Both are JWTs, both ride the Authorization Code Flow, but they answer different questions. 📄 pp. 272, 298

THE ONE-PAGE KERBEROS MESSAGE MATRIX:

MsgFrom → ToPayloadEncrypted underWhy📄 Page
1 Client → AS ID_c, ID_TGS, TS_1 (plaintext) Request TGS ticket 📄 pp. 306, 318
2 AS → Client K_c,TGS, ID_TGS, TS_2, Lifetime, Ticket_TGS K_c (outer); K_TGS (inner ticket) Deliver TGS session key + opaque TGS ticket 📄 pp. 306, 318
3 Client → TGS ID_v, Ticket_TGS, Authenticator_c K_TGS (ticket); K_c,TGS (auth) Request service ticket for V 📄 pp. 307, 319

5 Client → V Ticket_v, Authenticator_c K_v (ticket); K_c,v (auth) Authenticate to V 📄 pp. 308, 322
6 V → Client TS_5 + 1 K_c,v Mutual auth — V proves it knows K_v (thus K_c,v) 📄 pp. 308, 310

THE #1 MISTAKE STUDENTS MAKE:

Confusing the ticket with the authenticator. The ticket is long-lived (hours, a workday) and reusable — it's the "passport". The authenticator is ephemeral (seconds) and single-use — it's the "fresh selfie" proving the ticket-holder is actually present right now. Without the authenticator, a sniffed ticket can be replayed; the authenticator is what prevents that. They serve opposite temporal roles. 📄 pp. 310, 321, 322

IF YOU REMEMBER NOTHING ELSE, REMEMBER THIS:

"Kerberos = symmetric only + KDC-centralized + single logon + mutual auth at Msg 6. Authentication asks 'who', authorization asks 'what'. OpenID = ID Token; OAuth = Access Token." 📄 pp. 298, 306, 308
