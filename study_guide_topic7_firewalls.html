§1 — Concept Overview

📘 LECTURE PDF COVERAGE: pp. 397-434 (one concentrated lecture on firewalls as a whole)

1.1 The Problem This Solves 📄 pp. 400-401

Any organization with a network faces a fundamental asymmetry: inside the network, machines have higher trust — they reach sensitive file servers, printers, databases; outside the network, machines have unknown trust — they could be adversaries or benign Internet services. Firewalls are the traffic police at the border, deciding which packets cross and which are turned away 📄 p. 400. They are a choke point of control and monitoring that enforces a security policy on every packet entering or leaving, so that only authorized traffic is allowed and abnormal patterns can trigger alarms 📄 pp. 400-401.

1.2 The Core Idea — Classification by What Context They Inspect 📄 pp. 404-407

The instructor's three-way classification 📄 p. 404 hinges on how much context the firewall retains per packet:

  • Stateless (Packet Filter) — evaluates each packet in isolation, only examining network- and transport-layer headers (IP src/dst, port, protocol, TCP flags). No memory of prior packets. Fast, simple, but cannot tell whether a packet belongs to a legitimate established connection. 📄 pp. 405, 408
  • Stateful — maintains a connection state table that tracks every TCP/UDP flow in progress. Allows return packets only if they belong to an established outgoing session. Smarter than stateless but still operates at the transport layer. 📄 pp. 406, 420-421
  • Application / Proxy — terminates the client's connection at the firewall, opens a new connection to the destination, and inspects payload at the application layer (understands HTTP URLs, SMTP envelopes, FTP commands). Deepest inspection but requires a separate proxy per protocol and adds latency. 📄 pp. 407, 422-425

Choice among the three is a tradeoff: stateless wins on speed and simplicity; stateful balances speed against correctness (no bogus "reply" packets); proxy wins on inspection depth but costs latency, breaks end-to-end connectivity, and needs a separate implementation per application protocol.

1.3 Analogy — A Building Security Staff With Three Levels of Vetting 📄 pp. 405-407, 422

Think of three different security guards at the door of a corporate building:

  • The checklist guard (stateless) — holds a simple list of rules: "no visitors after 8 PM; no one with a red badge; no one carrying boxes." Looks at each arrival independently, applies the rules, doesn't remember who just left or who is expected back. Fast but gullible — if a visitor left at lunch and returns after 8 PM, the guard has no way to know they should be re-admitted. 📄 p. 405
  • The logbook guard (stateful) — keeps a log of every person who entered, their expected return, and what they're doing. When someone returns, the guard checks the logbook to verify they were originally signed in. Catches impersonators (someone trying to enter claiming to be returning) but doesn't know what's in their briefcase. 📄 pp. 406, 421
  • The front-desk concierge (proxy) — meets every visitor, asks their business, and then personally carries the message to the intended recipient. The visitor never actually enters the building; the concierge does. Can refuse inappropriate requests ("you can't ask the CEO that") but is slow, needs to be fluent in every topic guests might bring up, and imposes itself on every conversation. 📄 pp. 407, 422

In this analogy, iptables is the guard's rulebook — a prioritized sequence of rules arranged in chains (INPUT, OUTPUT, FORWARD), each with a default action if no rule matches 📄 p. 413. Netfilter is the set of "hooks" in the Linux kernel where the guard sits — fixed points in every packet's journey where the rules get checked 📄 p. 409. An evading visitor who sneaks past via a hidden passage corresponds to SSH tunneling or VPN — the firewall only sees encrypted tunnel traffic and cannot inspect what's inside 📄 pp. 427, 429.

1.4 Technical Definition 📄 pp. 400-401, 404

A firewall is a system — hardware, software, or a combination — placed at the boundary between two networks of different trust levels, enforcing a security policy by permitting or denying traffic flow based on source/destination addresses, transport-layer ports and flags, connection state, and/or application-layer content 📄 p. 400. It must itself be immune to penetration (hardened OS, minimal attack surface) and must be positioned so that all traffic between the protected zones is forced to pass through it — otherwise packets can bypass the policy 📄 p. 401. Firewalls are conventionally classified by operational mode: packet-filter / stateless (layer-3/4, per-packet), stateful (layer-3/4 plus connection-state table), and application-layer / proxy (layer-7 with full payload inspection via a protocol-aware intermediary) 📄 p. 404.

§2 — How Each Firewall Works (Step-by-Step)

2aStateless (Packet-Filter) Firewall 📄 pp. 405, 408, 410-412 · Lecture 16 part 2

① Mechanism — Per-Packet Header Inspection 📄 pp. 405, 408

A stateless firewall looks at each packet in complete isolation — it does not remember anything about past packets. For each packet, it extracts a small set of header fields and consults a rule list. If a rule matches, the associated action (ACCEPT, DROP, REJECT) is applied; otherwise a default policy applies. 

 Stateless Packet-Filter Firewall           📄 pp. 405, 408

   Packet arrives at interface
             │
             ▼
   ┌────────────────────────────────────┐
   │ Extract header fields:             │
   │   - IP source address              │
   │   - IP destination address         │
   │   - Protocol (TCP/UDP/ICMP/…)      │
   │   - TCP/UDP source port            │
   │   - TCP/UDP destination port       │
   │   - TCP flags (SYN/ACK/FIN/RST/…)  │
   │   - ICMP message type              │
   └────────────────┬───────────────────┘
                    │
                    ▼
   ┌────────────────────────────────────┐
   │ Match against rule list (sequential)│
   │   rule 1:  match fields → ACCEPT   │
   │   rule 2:  match fields → DROP     │
   │   rule 3:  match fields → REJECT   │
   │     ...                            │
   │   default policy (if no match)     │
   └────────────────┬───────────────────┘
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
      ACCEPT      DROP      REJECT
     (forward)  (silent)  (ICMP reply)

② Concrete Example — Port-Based Rules 📄 p. 408

The instructor gave two canonical examples on slide 408:

Rule: "No incoming port 53 packets except from known trusted servers" 📄 p. 408
Purpose: DNS uses UDP/TCP port 53. Restricting incoming port-53 traffic to a whitelist of trusted resolvers prevents attackers from injecting DNS replies (see Topic 2, DNS attacks).
iptables form:
iptables -A INPUT -p udp --dport 53 ! -s <trusted-IP> -j DROP
Rule: "Block all incoming TCP packets with port 23 (Telnet)" 📄 p. 408
Purpose: Telnet is cleartext — credentials travel in plaintext and are trivially sniffable. Firewalls routinely block port 23 unconditionally, forcing users toward SSH (port 22) instead.
iptables form:
iptables -A INPUT -p tcp --dport 23 -j DROP

③ What Stateless Firewalls Cannot Do 📄 p. 420

The central limitation: they cannot correlate a "reply" packet to an outgoing flow. If an internal host opens a TCP connection to an external web server, the server's reply comes back with source port 80 — but the stateless firewall has no record of the outbound request. To let replies through, administrators must open broad inbound port ranges, enlarging the attack surface. 📄 p. 420 The instructor: "Traditional packet filters do not examine transport-layer context — i.e., matching return packets with outgoing flow."

2bStateful Firewall 📄 pp. 406, 419-421 · Lecture 16 part 2

① Mechanism — Connection State Tracking 📄 pp. 406, 419

A stateful firewall maintains a connection state table: an in-memory record of every TCP/UDP flow that is currently "open" according to observed traffic. For each incoming packet, the firewall checks whether it belongs to an already-established connection; if yes, the packet is allowed through automatically without re-matching the rule list. New connection attempts still traverse the rule list. 📄 pp. 406, 419

                    │
                    ▼

   │  10.0.0.5   8.8.8.8    51234 443   NEW      │  ← added on outbound SYN
   │  10.0.0.5   8.8.8.8    51234 443   ESTAB.   │  ← after ACK completes 3-way hs
   │  ...                                        │

   Inbound SYN-ACK (external → internal)
   ◄──────────────────────────────────

② Concrete Example — The "Allow Return Traffic" Rule 📄 pp. 414, 421

On Linux, iptables becomes stateful via the conntrack extension: 

 iptables stateful rules  📄 p. 414

   # 1. Allow NEW outgoing connections
   iptables -A OUTPUT -m conntrack --ctstate NEW -j ACCEPT

   # 2. Allow incoming packets only if they belong to
   #    an ESTABLISHED or RELATED connection
   iptables -A INPUT -m conntrack \
            --ctstate ESTABLISHED,RELATED -j ACCEPT

   # 3. Everything else: DROP
   iptables -P INPUT DROP
SOURCE: 📄 p. 414

③ What Stateful Firewalls Add — and What They Still Miss

Adds: Can detect bogus "out-of-context" packets — e.g., an inbound SYN-ACK with no matching outbound SYN must be a spoofed or stale packet and is dropped 📄 p. 420. Can safely permit replies without opening broad inbound port ranges. Dramatically improves security-per-rule.

Still misses: Cannot see inside packet payloads. If an HTTP request contains a SQL injection attack, a stateful firewall happily forwards it because at layer-4 everything looks like a legitimate TCP segment. That's what application/proxy firewalls fix.

2cApplication / Proxy Firewall 📄 pp. 407, 422-425 · Lecture 16 part 2

① Mechanism — The Two-Connection Model 📄 pp. 407, 422

Unlike packet-filter firewalls (which forward the same packet through), an application/proxy firewall terminates the client's connection at the proxy, inspects the full application-layer message, and opens a new connection from the proxy to the destination server. The two connections are cryptographically and logically independent — the server never talks directly to the client. 📄 pp. 407, 422

③ Power & Tradeoffs 📄 pp. 423, 425, 426

PowerTradeoff
Full access to application-layer content — can block specific URL paths, reject malformed requests, enforce protocol correctness Requires a dedicated proxy per protocol; novel/custom services aren't supported
Can sanitize attacks at the payload level (SQL injection, XSS patterns in a WAF) Adds latency — every connection traverses two TCP handshakes
Hides internal server from direct contact — the proxy is the only reachable endpoint Breaks true end-to-end encryption: proxy must decrypt TLS to inspect payload (TLS termination)
Logs entire application-layer interactions for forensics Needs more compute and memory per connection; harder to scale

The slide 425 note about web proxies doubling as egress-evasion tools is a key exam hook: "If a firewall conducts packet filtering based on destination address, we can evade this firewall by browsing the Internet using web proxy. The destination address will be modified to the proxy server which defeats the packet filtering rules." This bridges application-layer firewall concepts with the evasion material in §2d. 📄 p. 425

2dNetfilter / iptables & Firewall Evasion 📄 pp. 409-418, 426-433 · Lecture 16 part 2

The Netfilter Framework 📄 p. 409

Netfilter is the Linux kernel's packet-processing framework. Every packet traversing the network stack passes through a series of named "hook points" (e.g., NF_IP_PRE_ROUTING, NF_IP_LOCAL_IN). Kernel modules — typically loaded as LKMs — register callback functions at one or more of these hooks. When a packet reaches a hook, each registered callback inspects or modifies it, and returns a verdict (NF_ACCEPT, NF_DROP, NF_STOLEN, etc.) 📄 p. 409.

Slide 410-412: Building a simple firewall with Netfilter 📄 pp. 410-412
What the LKM does:
  1. Registers a hook at NF_IP_LOCAL_IN (incoming traffic destined for this host)
  2. Callback receives the full packet (struct sk_buff) with all headers accessible
  3. Hardcoded logic: if the destination TCP port is 23 (Telnet), return NF_DROP; else return NF_ACCEPT
This is fundamentally the same pattern iptables uses under the hood, but hardcoded in C rather than configured via rule syntax. 📄 pp. 410-412

iptables — User-Space Configuration of Netfilter 📄 pp. 413-417

iptables is the Linux userspace tool that writes rules into the Netfilter-based Xtables kernel module. Rules are organized hierarchically:

  • Tables (filter, nat, mangle, raw) — categorize rules by purpose; filter is the firewall table
  • Chains (INPUT, OUTPUT, FORWARD, PREROUTING, POSTROUTING) — correspond to Netfilter hook points
  • Rules — match conditions + action target; evaluated top-to-bottom per chain
  • Default policy — action applied if no rule matches; typically DROP for secure configs

Worked Example — Building a Default-Deny Firewall 📄 pp. 415-418

 iptables worked example from lecture       📄 pp. 415-418

 # Step 1: Flush all existing rules (clean slate)              📄 p. 415
 iptables -F
 iptables -X

 # Step 2: Temporarily set default policy to ACCEPT
 #         (so we don't lock ourselves out while editing)
 iptables -P INPUT  ACCEPT
 iptables -P OUTPUT ACCEPT

 # Step 3: Allow TCP traffic to SSH (22) and HTTP (80) on INPUT   📄 p. 416
 iptables -A INPUT -p tcp --dport 22 -j ACCEPT
 iptables -A INPUT -p tcp --dport 80 -j ACCEPT

 # Step 4: Allow all outgoing TCP traffic
 iptables -A OUTPUT -p tcp -j ACCEPT

 # Step 5: Allow DNS queries (outgoing) and replies (incoming)    📄 p. 417
 iptables -A OUTPUT -p udp --dport 53 -j ACCEPT
 iptables -A INPUT  -p udp --sport 53 -j ACCEPT

 # Step 6: Switch default policy to DROP
 #         (now only explicitly-allowed traffic passes)
 iptables -P INPUT DROP
 iptables -P OUTPUT DROP

 # Test:  wget http://example.com   → succeeds (port 80 allowed)   📄 p. 418
 #        telnet example.com 23     → fails (port 23 not allowed)
SOURCE: 📄 pp. 415-418

iptables Extensions — conntrack & owner 📄 p. 414

conntrack: Makes iptables stateful 📄 p. 414
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
This single rule lets iptables behave statefully — return traffic for already-initiated outbound connections is allowed without opening inbound port ranges. The connection state table is maintained in the kernel by nf_conntrack.
owner: Per-user filtering (OUTPUT chain only) 📄 p. 414
Can match packets by the UID/GID of the process that generated them. Example: prevent user alice from sending Telnet packets:
iptables -A OUTPUT -p tcp --dport 23 -m owner --uid-owner alice -j DROP
Limitation: only works on OUTPUT (outgoing) — incoming packets have no notion of UID at the network layer. 📄 p. 414

Firewall Evasion — The Key Techniques 📄 pp. 426-433

Firewalls aren't perfect. 📄 p. 426 They are useless against insiders ("evildoer exists on inside; malicious code executes internally") and cannot inspect encrypted traffic in general. The main evasion techniques exploit encryption as a shield:

① SSH Tunneling 📄 pp. 427-431
Scenario A (slide 428): From home, you want to Telnet to work. Company firewall blocks incoming Telnet but allows SSH to apollo. Solution: ssh -L 8000:work:23 user@apollo. This opens an SSH tunnel from home's port 8000 through apollo to work's port 23. Then telnet localhost 8000 works — the firewall only sees SSH traffic between home and apollo.
Scenario B (slide 431): From work, you want to reach Facebook (blocked by company firewall). Solution: ssh -L 8000:facebook.com:443 user@home. Tunnel forwards your HTTPS requests through home to Facebook. The firewall only sees SSH traffic between work and home.
Why it works: SSH traffic is encrypted; the firewall cannot see the inner payload. It knows SSH is happening; it cannot know Telnet/Facebook is inside.
② Dynamic Port Forwarding / SOCKS 📄 pp. 432-433
ssh -D 9000 user@home opens a dynamic port-forwarding tunnel (a SOCKS proxy on localhost:9000). Any application configured to use this SOCKS proxy (browser, curl, etc.) has all its traffic forwarded through home. Unlike static SSH tunneling, this works for arbitrary destinations without pre-configuring a forward per destination. 📄 pp. 432-433
③ VPN as Firewall Evasion 📄 p. 427
A VPN tunnel between a computer inside the firewall and another outside encrypts everything passing through. The firewall cannot inspect the encrypted traffic and therefore cannot filter by destination or content. The exam connection here: this is the same tunneling principle IPsec provides in Topic 6. 📄 p. 427
Cross-topic integration hook Firewall evasion via encryption is the mirror image of why firewalls add deep packet inspection: defenders want to look inside; attackers use encryption to prevent it. This tension appears in the TLS-inspection debate for WAFs, in enterprise MITM proxies that deliberately break TLS, and in the fact that every one of Topic 6's VPNs (IPsec, TLS-based VPN) is also an egress-filter bypass mechanism. 📄 pp. 425, 427, 434

3.2 — Common Misconceptions & Traps

❌ MYTH: "Stateless means the firewall does not maintain any state — not even its rule list."

✅ REALITY: "Stateless" refers specifically to connection state — the firewall does not track which TCP/UDP flows are currently active. It still maintains its configured rule list (a static configuration, not per-connection state). The opposite of "stateless" is "stateful," which adds a per-connection tracking table. 📄 pp. 405-406

🎯 EXAM TRAP: MCQ option "stateless firewalls have no rules" — FALSE. They have rules but no per-flow memory.

❌ MYTH: "A stateful firewall inspects the application payload and can block SQL injection attacks."

✅ REALITY: Stateful firewalls operate at layers 3–4 only. They add connection-state tracking to packet filtering but do not look inside the TCP payload at application data. To block SQL injection or inspect HTTP URLs, you need an application/proxy firewall (specifically a WAF for HTTP). 📄 pp. 406, 423

🎯 EXAM TRAP: "Which firewall type can block malicious URL patterns?" → Only application/proxy, not stateful.

❌ MYTH: "Application/proxy firewalls work transparently — the client connects directly to the destination."

✅ REALITY: The defining feature of proxy firewalls is that the client's connection terminates at the proxy, and the proxy opens a new connection to the destination. There are always two separate TCP connections, and the proxy impersonates the destination to the client. 📄 pp. 407, 422

🎯 EXAM TRAP: "A proxy firewall forwards the original packets unchanged to the destination." — FALSE. It creates a fresh connection.

❌ MYTH: "Ingress and egress filtering are the same thing viewed from different sides."

✅ REALITY: They're structurally different defenses at different points. Ingress filtering inspects incoming traffic at your network edge to drop packets that don't belong (e.g., arriving with internal source IPs). Egress filtering inspects outgoing traffic to prevent spoofed packets from leaving your network or to block users from reaching disallowed destinations. The instructor specifically emphasized egress filtering at ISPs as more effective against IP spoofing because it stops spoofed packets at the source. 📄 pp. 403, 512

🎯 EXAM TRAP: "Which is more effective against IP spoofing: ingress or egress?" → Egress (per instructor's direct quote).

❌ MYTH: "A properly configured firewall prevents all attacks on the network."

✅ REALITY: Slide 426 explicitly names the three fundamental blind spots: "(1) Useless against attacks from the inside — evildoer exists on inside; malicious code is executed on an internal machine. (2) Organizations with greater insider threat — banks and military. (3) Cannot protect against transfer of all virus-infected programs or files — because of huge range of O/S and file types." Additionally, encrypted payloads (SSH tunnels, VPN, HTTPS) are opaque to layer-3/4 firewalls. 📄 pp. 426-427

🎯 EXAM TRAP: T/F: "A firewall alone is sufficient network security." — FALSE. Defense-in-depth is required.

❌ MYTH: "The iptables default policy should always be ACCEPT for usability."

✅ REALITY: Secure configurations use default DROP with explicit ACCEPT rules for the specific traffic you need to allow. This is a "whitelist" (default-deny) model — by far the safer stance. The instructor's own worked example switches from ACCEPT (during editing) to DROP (for production) specifically to illustrate this. 📄 pp. 415, 417

🎯 EXAM TRAP: "Which default policy reflects a least-privilege stance?" → DROP, not ACCEPT.

Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG TF 📄 p. 403

Q5. Statement: A firewall action of "Reject" is functionally identical to "Deny" — both silently discard the packet without any notification to the sender.

▶ Show Answer

✅ Answer: FALSE.

Slide 403 distinguishes them clearly: Deny is not permitted to enter the other side of firewall (silent drop), while Reject is similar to "Deny", but tells the source about this decision through ICMP packet. The observable difference matters for debugging (Reject tells the sender the packet was dropped) but also for information disclosure (Reject confirms the firewall is active, which a stealthy deployment might prefer to hide with silent Drop). 📄 p. 403

INT TF 📄 pp. 407, 422

Q6. Statement: In an application/proxy firewall deployment, a single TCP connection carries traffic from the client, through the proxy, to the destination server — the proxy simply forwards the packets without terminating the connection.

▶ Show Answer

✅ Answer: FALSE.

The defining feature of a proxy firewall is precisely the opposite. Slide 407 states: "The client's connection terminates at the proxy and a separate connection is initiated from the proxy to the destination host." There are two independent TCP connections: connection #1 between client and proxy, connection #2 between proxy and destination. The proxy acts as an intermediary impersonating the intended recipient to the client. This is what enables full application-layer inspection (the proxy fully reassembles and parses the message) and also what makes proxy firewalls incompatible with end-to-end TLS without deliberate TLS termination. 📄 pp. 407, 422

ADV · subtle TF 📄 pp. 427-431

Q7. Statement: SSH tunneling evades a firewall because the SSH protocol is not listed in any standard port-filtering rules, making SSH traffic invisible to firewalls.

▶ Show Answer

✅ Answer: FALSE.

SSH uses a well-known port (22) and is fully visible to the firewall — the firewall can see that SSH traffic is occurring and could block it outright if configured to do so. The evasion works because SSH traffic is encrypted: the firewall sees SSH packets but cannot inspect their contents. If an administrator has permitted SSH (as in the lecture's scenario A: "the company's firewall does allow SSH traffic to reach its internal machine 'apollo'" 📄 p. 428), then arbitrary TCP traffic tunneled inside SSH becomes opaque. The fix is to disallow SSH or to inspect traffic at the endpoints — not to try to "see" SSH at layers 3-4. 📄 pp. 427-431

Subtle trap: encryption is the mechanism, not "invisibility to port filtering." The firewall knows SSH is happening; it just can't see what's inside.

INT SA 📄 pp. 415-418

Q9. You need to configure an iptables firewall on a server that should only accept incoming SSH (port 22) and HTTP (port 80), plus let normal outbound TCP traffic flow. Outline the five key steps you would execute, in order, and explain why the default policy change is the final step.

▶ Show Answer

📝 Model Answer:

  1. Flush existing rules with iptables -F to start from a clean slate. 📄 p. 415
  2. Temporarily set default policy to ACCEPT on INPUT and OUTPUT chains so that editing doesn't accidentally lock you out of the server. 📄 p. 415
  3. Add explicit ACCEPT rules for the required traffic: iptables -A INPUT -p tcp --dport 22 -j ACCEPT and iptables -A INPUT -p tcp --dport 80 -j ACCEPT for SSH and HTTP; iptables -A OUTPUT -p tcp -j ACCEPT for outgoing. 📄 p. 416
  4. Add any additional allow rules for protocols like DNS that the server needs to function (outgoing UDP/53 and incoming replies). 📄 p. 417
  5. Finally, switch default policy to DROP with iptables -P INPUT DROP and iptables -P OUTPUT DROP — only now does the firewall become "default-deny" and only explicitly-allowed traffic flows. 📄 p. 417

The default policy change is last because if it were first, the administrator would be immediately disconnected (every SSH packet including the existing session's replies would be dropped before any ACCEPT rule was in place). The ACCEPT-then-DROP sequence is the safe configuration pattern. 📄 pp. 415-417

🔑 Key Points (Grading Checklist):

  • ☐ Steps in correct order: flush → set ACCEPT default → add rules → set DROP default 📄 pp. 415-417
  • ☐ Explicitly permits ports 22 (SSH) and 80 (HTTP) on INPUT 📄 p. 416
  • ☐ Permits outbound TCP 📄 p. 416
  • ☐ Explains that DROP-last prevents self-lockout 📄 p. 417
  • ☐ Bonus: identifies this as a "default-deny" / whitelist approach
ADV · challenge SA 📄 pp. 403, 425, 427  +  pp. 370-371 (Topic 6) · p. 512 (Topic 2)

Q10. Describe the fundamental tension between firewall effectiveness and encryption. Cover: (1) why firewalls are limited against encrypted traffic, (2) how users exploit this to evade firewalls (name at least two techniques), and (3) how this tension relates to the IP-spoofing defense stack the instructor discussed in the final review (ingress/egress filtering, IPsec).

▶ Show Answer

📝 Model Answer:

(1) Why firewalls are limited against encrypted traffic. Stateless and stateful firewalls inspect only headers (IP, ports, TCP flags), and even application/proxy firewalls need to parse application-layer content to make decisions. When traffic is end-to-end encrypted (TLS, SSH, IPsec tunnels, VPN), the firewall sees only an opaque encrypted stream. It can observe that SSH or HTTPS is happening and to whom, but not what is being transmitted. Application/proxy firewalls can address this only by terminating TLS at the proxy — breaking end-to-end encryption, which conflicts with confidentiality goals. 📄 pp. 407, 423, 427

(2) Firewall evasion via encryption. The lecture covered three main techniques: (a) SSH tunneling — forward arbitrary TCP traffic through an allowed SSH connection (e.g., telnet localhost:8000 → ssh tunnel → telnet real-destination:23), where the firewall sees only the SSH exterior. 📄 pp. 428-431 (b) Dynamic port forwarding — create a SOCKS proxy over SSH so any application can tunnel through. 📄 pp. 432-433 (c) VPN — establish an encrypted tunnel (IPsec or TLS-based) between an internal and external machine; firewalls cannot filter the opaque VPN payload. 📄 p. 427 A fourth related technique is using a web proxy as the apparent destination, which defeats destination-based egress filtering. 📄 p. 425

(3) Relation to IP-spoofing defense. Packet filtering at network edges — specifically ingress and especially egress filtering — is one of the instructor's named defenses against IP spoofing 📄 pp. 403, 512. These filters work by inspecting source/destination IP addresses, which are visible in the IP header even when the payload is encrypted. This is why layer-3 filtering remains effective against spoofing despite encryption: the attacker cannot encrypt the outer IP header or the packet would not route. IPsec in transport mode similarly leaves the outer IP addresses visible, allowing ingress/egress filtering to remain effective; tunnel mode encrypts the inner IP, providing limited traffic-flow confidentiality at the cost of making spoofing detection harder at inner endpoints. 📄 pp. 370-371, 403, 512

🔑 Key Points (Grading Checklist):

  • ☐ Identifies header-only inspection as firewall limit 📄 pp. 405, 406
  • ☐ Names SSH tunneling and VPN as evasion techniques 📄 pp. 427, 429
  • ☐ Notes ingress/egress filtering works on IP headers, which remain visible 📄 pp. 403, 512
  • ☐ Addresses the tension between end-to-end encryption and deep inspection 📄 pp. 423, 427
  • ☐ Bonus: mentions web proxy as destination-filter evasion 📄 p. 425

THE #1 MISTAKE STUDENTS MAKE:

Confusing stateful with application/proxy. Both retain per-connection memory, but stateful firewalls operate at layers 3-4 only (tracking TCP/UDP flows via their 5-tuple + state) and do not inspect payload. Application/proxy firewalls go further — they terminate the connection, parse the application-layer protocol, and can block content-based attacks (SQL injection, malicious URLs). If the question involves "inspecting payload" or "blocking URL patterns," the answer is always Application/Proxy, not Stateful. 📄 pp. 406-407, 423
