Discussion — Annotated in light of Kleinberg & Tardos + CLRS
WHY — The Purpose

Why study algorithms formally? Because not all algorithms are equally good. The gap between an O(n²) and an O(n log n) algorithm isn't a minor technical detail — it's the difference between a system that works for 10,000 inputs and one that works for 10 million inputs (assuming 10⁸ ops/sec). Kleinberg & Tardos open their book with this exact argument: as problems grow in scale — genomics, social networks, supply chains — the algorithm you pick determines whether a solution is possible at all. CS is not just about writing code that works; it's about writing code that scales.

HOW — The Process

The course is structured around three interlocking skills: (1) Design — picking the right paradigm (divide & conquer, greedy, DP) for the right problem; (2) Analysis — proving your algorithm is fast using asymptotic notation; (3) Correctness — proving your algorithm is right, typically by induction or exchange argument. CLRS (Chapter 1) calls this the foundation of algorithmic thinking. You cannot shortcut any of the three.

WHAT — The Deliverable

By the end of this course you should be able to look at a novel problem and immediately ask: Is this a shortest-path problem? A matching problem? An NP-hard problem? And then choose — and justify — the right approach. This is exactly what technical interviews, research, and real-world engineering require.

Mathematical Maturity — What You Need

The course assumes comfort with proof by induction. This is non-negotiable because virtually every algorithm correctness proof uses induction. If you're rusty, review: (1) Base case, (2) Inductive hypothesis, (3) Inductive step. CLRS Appendix A reviews summations; Appendix B reviews basic combinatorics. Kleinberg & Tardos Chapter 0 reviews asymptotic notation informally.

TOPIC 02

The Stable Matching Problem

Lectures 2 · Slides 23–31
Advisor Selection / Stable Marriages

The slides introduce the Stable Matching Problem through the lens of advisor selection: given a set of students and advisors, each with preference lists, find a matching that is stable — no student and advisor would both prefer each other over their current match.

  • Motivating question: Can this be solved algorithmically? Are all algorithmic solutions equally good?
  • This is the canonical problem from Kleinberg & Tardos, Chapter 1, Section 1.1
Discussion — Gale-Shapley Algorithm
WHY — The Purpose

The Stable Matching Problem is the perfect opening problem for an algorithms course because it illustrates the entire design-and-prove workflow on a problem you can fully understand without any prior theory. It's used in real life to match medical residents to hospitals (the NRMP program), students to schools, and even organ donors to recipients. The key insight: a "fair" or "stable" outcome is not obvious, and finding it efficiently requires a clever algorithm.

HOW — The Gale-Shapley Algorithm (K&T Ch. 1.1)

The Gale-Shapley algorithm (also called "deferred acceptance") works as follows. Kleinberg & Tardos prove it finds a stable matching in O(n²) time:

// n students, n advisors, each with complete preference lists
Initialize: all students and advisors free
while some student s is free and hasn't proposed to all advisors:
    a = first advisor on s's list s hasn't proposed to yet
    if a is free:
        tentatively match (s, a)
    else if a prefers s over current match s':
        free s', tentatively match (s, a)
    else:
        a rejects s
return the final matching

Key invariant: Once an advisor is matched, they only trade up. Hence the algorithm terminates after at most n² proposals.

WHAT — Correctness & Optimality

Claim 1 (Stability): The algorithm always produces a stable matching. Proof by contradiction: suppose student s and advisor a form an instability. Either s never proposed to a (impossible — s proposes in order and reaches a before any worse option) or a rejected s for someone better. Either way, a is not worse off — contradiction.

Claim 2 (Student-optimality): The G-S algorithm gives every student their best valid partner — the best advisor they could be matched with in any stable matching. Advisors get their worst valid partner.

Complexity

Each student proposes to each advisor at most once → at most n² proposals → O(n²) time. This is optimal for this problem.

K&T Connection: Chapter 1, Section 1.1 covers this in full, including the proof of stability and optimality. This is also Problem 1 in K&T's first problem set. The slides directly reference this section as required reading.
✦ Practice Questions — Stable Matching
Q1. (K&T 1.1 style) Prove that the Gale-Shapley algorithm always terminates and produces a perfect matching.
Source: K&T Chapter 1 · Difficulty: Medium
Solution

Termination: Each student proposes to each advisor at most once. With n students and n advisors, there are at most n² proposals. Each iteration of the while loop makes at least one proposal, so the algorithm terminates in at most n² steps.

Perfect matching: Suppose some student s is unmatched at termination. Then s proposed to all n advisors and was rejected by each. But an advisor only rejects a student to stay with (or upgrade to) someone better, so every advisor is matched at termination. But then there are n matched advisors and n students, so every student is matched — contradicting s being unmatched.

Q2. Is the stable matching produced by G-S unique? Provide an argument or counterexample.
Difficulty: Medium
Solution

No, stable matchings are not unique in general. Consider 2 students {s1, s2} and 2 advisors {a1, a2} with preferences: s1: a1 > a2; s2: a2 > a1; a1: s2 > s1; a2: s1 > s2. Both {(s1,a1),(s2,a2)} and {(s1,a2),(s2,a1)} are stable. The G-S algorithm, run with students proposing, will produce the student-optimal matching.

Algorithm Comparison & Asymptotic Analysis

Lectures 2–6 · Slides 32–77
Linear Search vs. Binary Search
  • Linear Search: Check each element sequentially. Works on sorted & unsorted. O(1) best, O(n) worst & average.
  • Binary Search: Repeatedly halve the search space. Requires sorted input. O(log n) all cases.
  • Motivates: we need a formal framework to compare algorithms.
Big-O, Big-Ω, Big-Θ Notation
  • O(g(n)): Upper bound — f(n) grows no faster than c·g(n) for large n
  • Ω(g(n)): Lower bound — f(n) grows at least as fast as c·g(n) for large n
  • Θ(g(n)): Tight bound — both O and Ω simultaneously
  • Exercises: prove/disprove asymptotic claims (Exercises 1–3 from Lecture 5–6)
Recurrences — Recursive Factorial
def factorial(n):
    if n == 0 or n == 1: return 1
    return n * factorial(n - 1)
  • Each call does O(1) work, makes 1 recursive call on (n−1)
  • Recurrence: T(n) = T(n−1) + 1, T(1) = 1 → T(n) = O(n)
Discussion — Asymptotic Notation Deeply
WHY — The Purpose

Asymptotic notation exists to let us reason about algorithm performance independent of hardware. Whether you run code on a 2010 laptop or a 2025 server cluster, the scaling behavior of the algorithm doesn't change. A Θ(n²) algorithm is always overtaken by a Θ(n log n) one as n grows. This allows us to make machine-agnostic performance claims — and it's why the notation appears everywhere in systems, research papers, and interviews.

HOW — Formal Definitions (CLRS Ch. 3)

Definition (Big-O): f(n) = O(g(n)) iff there exist positive constants c and n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀.

Definition (Big-Ω): f(n) = Ω(g(n)) iff there exist positive constants c and n₀ such that f(n) ≥ c·g(n) for all n ≥ n₀.

Definition (Big-Θ): f(n) = Θ(g(n)) iff f(n) = O(g(n)) AND f(n) = Ω(g(n)).

Common hierarchy: O(1) ⊂ O(log n) ⊂ O(n) ⊂ O(n log n) ⊂ O(n²) ⊂ O(n³) ⊂ O(2ⁿ)

WHAT — Key Proof Techniques

To prove f(n) = O(g(n)): Find explicit c and n₀. Example: 3n² + 5n = O(n²). Choose c = 4: for n ≥ 5, 3n² + 5n ≤ 3n² + n² = 4n². So c = 4, n₀ = 5 works.

To disprove f(n) = O(g(n)): Assume such c, n₀ exist and derive a contradiction.

Limit method: If lim[n→∞] f(n)/g(n) = 0: f = o(g). If constant c > 0: f = Θ(g). If ∞: f = ω(g).

Solving Recurrences — The Master Theorem (CLRS Ch. 4)

For recurrences of the form T(n) = aT(n/b) + f(n) where a ≥ 1, b > 1:

  • If f(n) = O(n^(log_b(a) − ε)): T(n) = Θ(n^(log_b a)) [Case 1: recursive work dominates]
  • If f(n) = Θ(n^(log_b a)): T(n) = Θ(n^(log_b a) · log n) [Case 2: tie]
  • If f(n) = Ω(n^(log_b(a) + ε)): T(n) = Θ(f(n)) [Case 3: combining dominates]
Merge Sort: T(n) = 2T(n/2) + n → Case 2 → T(n) = Θ(n log n)
✦ Practice Questions — Asymptotic Analysis
Q1. (CLRS Ch. 3 style) Is 2ⁿ⁺¹ = O(2ⁿ)? Is 2²ⁿ = O(2ⁿ)?
Solution

2ⁿ⁺¹ = O(2ⁿ): Yes. 2ⁿ⁺¹ = 2·2ⁿ ≤ c·2ⁿ for c = 2, any n₀ ≥ 1. ✓

2²ⁿ = O(2ⁿ): No. 2²ⁿ = (2ⁿ)². Assume 2²ⁿ ≤ c·2ⁿ for all large n. Then 2ⁿ ≤ c, which is false since 2ⁿ → ∞. Contradiction. ✗

Q2. (K&T Ch. 2 style) Rank the following in increasing order of growth rate: n!, 2ⁿ, n³, n log n, n^(3/2), log² n, √n.
Solution

log² n ⊂ √n ⊂ n log n ⊂ n^(3/2) ⊂ n³ ⊂ 2ⁿ ⊂ n!

Note: log² n means (log n)², not log(log n). Use L'Hôpital or limit comparisons to verify each step.

Q3. Solve the recurrence T(n) = 3T(n/3) + n using the Master Theorem.
Solution

Here a = 3, b = 3, f(n) = n. log_b(a) = log₃(3) = 1. f(n) = n = Θ(n¹) = Θ(n^(log_b a)). This is Case 2 of the Master Theorem. Therefore T(n) = Θ(n log n).

Code Complexity Analysis — Multiple Algorithms

Slides 60–67 present a sequence of Python/pseudocode algorithms — for each, the question is: (1) What does this algorithm do? (2) What is its time complexity?

  • Nested loops: typically O(n²) or O(n³)
  • Recursive algorithms: solve recurrence T(n) = aT(n/b) + f(n)
  • Slides 68–75 focus on recursive factorial: T(n) = T(n−1) + O(1) = O(n)
Discussion — Analyzing Code Complexity Step by Step
HOW — Systematic Complexity Analysis

Step 1: Identify the dominant operation. What operation is performed the most? (comparisons, assignments, function calls)

Step 2: Count how many times it runs relative to input size n.

Step 3: Express as a recurrence if recursive, then solve using substitution, recursion tree, or Master Theorem.

Common patterns to recognize:

  • Single loop from 1 to n: O(n)
  • Nested loop (both to n): O(n²)
  • Loop halving input each iteration: O(log n)
  • Recursive call on half + linear work: O(n log n) [Master Case 2]
  • Recursive call on half + constant work: O(log n) [binary search]
  • Two recursive calls on half + linear: O(n log n) [merge sort]
  • Two recursive calls on half + nothing: O(n) [tree traversal]
WHAT — Reading Exercises (from Slide 76)

The course recommends: K&T Chapter 1 Section 1.1 and K&T Chapter 2, Sections 2.1, 2.2, 2.4; CLRS Chapters 1–3. These cover the formal definition of asymptotic notation, growth of functions, and standard summation formulae.

Substitution Method for Recurrences (CLRS Ch. 4.3)

To solve T(n) = T(n−1) + 1 by substitution:

  1. Guess: T(n) = O(n)
  2. Assume T(k) ≤ c·k for all k < n
  3. Show T(n) ≤ c·n: T(n) = T(n−1) + 1 ≤ c(n−1) + 1 = cn − c + 1 ≤ cn for c ≥ 1.
  4. Base case: T(1) = 1 ≤ c·1 for c ≥ 1. ✓
TOPIC 05

Recurrences & Recursive Algorithm Analysis

Lectures 6–7 · Slides 68–84 · CLRS Ch. 4, K&T Ch. 5.1
Setting Up and Solving Recurrences
  • Recurrence captures the recursive structure of an algorithm's running time
  • T(n) = 2T(n/2) + n is the canonical merge-sort recurrence
  • Methods: substitution, recursion tree, Master Theorem
  • Reading: K&T Ch. 2 Sec. 2.4; CLRS Ch. 4
Discussion — Three Methods to Solve Recurrences
WHY — Why Do Recurrences Matter?

Recursive algorithms are everywhere: merge sort, quicksort, binary search, Karatsuba multiplication, Strassen matrix multiplication, FFT. Their running times all follow recurrences. Being able to solve T(n) = aT(n/b) + f(n) is not optional — it's the core of analyzing divide-and-conquer algorithms.

HOW — Recursion Tree Method

For T(n) = 2T(n/2) + n (merge sort):

  • Level 0: one node, cost n
  • Level 1: two nodes, each cost n/2, total n
  • Level k: 2^k nodes, each cost n/2^k, total n
  • Depth = log₂ n (until size 1)
  • Total cost = n × log n = O(n log n)
T(n) = 2T(n/2) + n → T(n) = Θ(n log n)
WHAT — Master Theorem Summary

For T(n) = aT(n/b) + Θ(n^d):

  • If d > log_b(a): T(n) = Θ(n^d)
  • If d = log_b(a): T(n) = Θ(n^d · log n)
  • If d < log_b(a): T(n) = Θ(n^(log_b a))
✦ Practice Questions — Recurrences
Q1. Solve T(n) = 4T(n/2) + n using the Master Theorem.
Solution

a=4, b=2, f(n)=n. log_b(a) = log₂(4) = 2. f(n) = n = O(n^(2−1)) → Case 1. T(n) = Θ(n²).

Q2. Solve T(n) = T(n−1) + n.
Solution

Expand: T(n) = n + (n−1) + (n−2) + … + 1 = n(n+1)/2 = Θ(n²). This is insertion sort's worst case.

TOPIC 06

Sorting Algorithms — Insertion Sort & Foundations

Lectures 8–10 · Slides 86–101 · CLRS Ch. 2, K&T Ch. 2.1
Insertion Sort
  • Build a sorted array one element at a time, inserting each new element in its correct position
  • Best case: O(n) (already sorted)
  • Worst case: O(n²) (reverse sorted)
  • Average case: O(n²)
  • In-place: Yes. Stable: Yes.
  • Useful in practice for small arrays (n < 16), or nearly-sorted data
Discussion — Why Study Insertion Sort?
WHY — The Purpose

Insertion sort is the canonical example of a loop invariant proof in CLRS (Chapter 2). Understanding it is less about using it in production and more about practicing the skill of algorithm correctness proof. The invariant: "after the i-th iteration, the subarray A[1..i] is sorted." Every algorithms course starts here for good reason.

HOW — Loop Invariant Proof (CLRS 2.1)

Invariant: At the start of iteration j, A[1..j-1] contains the first j-1 elements of the original array, in sorted order.

  • Initialization: j=2: A[1..1] is trivially sorted. ✓
  • Maintenance: We insert A[j] into its correct position, maintaining sortedness. ✓
  • Termination: j = n+1, so A[1..n] is sorted. ✓
WHAT — Practical Notes

Hybrid sorting (e.g., Timsort, Introsort) uses insertion sort for small subarrays. Python's built-in sort and Java's Arrays.sort both use this hybrid approach. So insertion sort is not "dead" — it's the fast path for tiny chunks in real-world O(n log n) sorts.

TOPIC 07

Merge Sort & Divide-and-Conquer

Lectures 10–11 · Slides 102–137 · K&T Ch. 5.1, CLRS Ch. 2.3, Ch. 4
Merge Sort — Algorithm & Analysis
function MergeSort(A, left, right, B):
    if (right - left == 1):
        B[0] = A[left]          // Base case
    if (right - left > 1):
        mid = (left + right) / 2
        MergeSort(A, left, mid, L)
        MergeSort(A, mid, right, R)
        Merge(L, mid-left, R, right-mid, B)

function Merge(A, m, B, n, C):
    i=0; j=0; k=0
    while (k < m+n):
        if (j==n or A[i] <= B[j]):
            C[k] = A[i]; i++; k++
        else:
            C[k] = B[j]; j++; k++
  • Recurrence: T(n) = 2T(n/2) + n → T(n) = O(n log n)
  • Stable: Yes (favor left when equal). In-place: No (needs O(n) extra space)
  • Variations: union/intersection/difference of sorted lists using merge!
Discussion — Divide and Conquer Deeply
WHY — The Purpose

Divide-and-conquer is not just a sorting trick — it's a fundamental algorithmic paradigm. The key insight: many problems have the property that a problem of size n can be broken into smaller versions of itself. When recombining solutions takes O(n) time, and we split into 2 halves, the master theorem gives O(n log n). This pattern appears in fast Fourier transforms, closest pair of points, matrix multiplication (Strassen), and beyond.

HOW — Full Analysis (K&T 5.1)

Let T(n) = time to sort n elements.

T(1) = 1    T(n) = 2T(n/2) + n

Expanding the recurrence step by step (as shown in Slide 114):

T(n) = 2T(n/2) + n
     = 2[2T(n/4) + n/2] + n = 4T(n/4) + 2n
     = 2^j · T(n/2^j) + j·n
At j = log n: T(n) = n·T(1) + n·log n = n + n log n = O(n log n)
WHAT — Practical Impact (Slide 115)

At 10⁸ operations/second: O(n²) is feasible up to n ≈ 10,000. O(n log n) is feasible up to n ≈ 10,000,000. That's a 1000× improvement. For real datasets (social networks: millions of nodes; genomics: billions of base pairs), the difference between O(n²) and O(n log n) determines whether a computation finishes in seconds or years.

Shortcomings of merge sort: requires O(n) extra space, inherently recursive (stack overhead). This motivates quicksort.

Merge Sort Variations — Practical Applications (Slide 116)

The merge subroutine is more general than just sorting:

  • Union of sorted lists: if A[i] == B[j], copy once and advance both
  • Intersection: only copy when A[i] == B[j]
  • Difference (A − B): copy A[i] only if A[i] < B[j] or B is exhausted

These run in O(m + n) time — far better than naive O(mn).

✦ Practice Questions — Merge Sort & Divide-and-Conquer
Q1. (K&T Ch. 5 style) Trace through MergeSort on the array [43, 32, 22, 78, 63, 57, 91, 13]. Show the recursive calls and merges.
Solution

Split: [43,32,22,78] | [63,57,91,13]

Recurse left: [43,32] | [22,78] → merge: [32,43] → [22,78] → merge: [22,32,43,78]

Recurse right: [63,57] | [91,13] → [57,63] | [13,91] → merge: [13,57,63,91]

Final merge: [13,22,32,43,57,63,78,91] ✓

Q2. Counting Inversions (K&T 5.3): An inversion in array A is a pair (i,j) with i < j but A[i] > A[j]. Design an O(n log n) algorithm to count inversions.
Source: K&T Chapter 5, Section 5.3 — Classic D&C Application
Solution

Modify MergeSort to count inversions during the merge step. When we copy B[j] into C (because B[j] < A[i]), every remaining element in A forms an inversion with B[j]. Count = (m − i) inversions per such step.

Correctness: every inversion is either (1) within the left half, (2) within the right half, or (3) across the two halves. Cases 1 and 2 are counted recursively. Case 3 is counted during merge.

Complexity: T(n) = 2T(n/2) + O(n) = O(n log n) — same as merge sort.

TOPIC 08

Quicksort — The Practical Champion

Lectures 11 · Slides 121–137 · CLRS Ch. 7, K&T Ch. 13.5
Quicksort Algorithm & Analysis
function Quicksort(A, l, r):
    if (r - l <= 1): return  // Base case
    // Partition with respect to pivot A[l]
    yellow = l + 1
    for green = l+1 to r-1:
        if A[green] <= A[l]:
            swap(A, yellow, green); yellow++
    swap(A, l, yellow-1)  // Move pivot into place
    Quicksort(A, l, yellow)
    Quicksort(A, yellow+1, r)
  • Best/Average case: O(n log n)
  • Worst case: O(n²) — sorted input with first-element pivot
  • In-place: Yes. Stable: No.
  • Randomization: Choose pivot randomly → expected O(n log n)
Discussion — Quicksort Deeply
WHY — Why Quicksort Dominates in Practice

Despite a worst-case of O(n²), quicksort is typically the default sort in practice (Python, C++, Java all use variants). The reason: its constants are small, it's in-place (no auxiliary array), has excellent cache performance, and its average case is O(n log n). Merge sort's O(n) extra memory allocation is often the bottleneck in real systems.

HOW — Average Case Analysis (CLRS 7.4)

Assume input is a random permutation of {1…n}. The expected number of comparisons is Θ(n log n). The key argument uses indicator random variables: let X_ij = 1 if elements i and j are ever compared. E[total comparisons] = Σ Pr[X_ij = 1] = Σ 2/(j−i+1) = O(n log n).

This is a classic probabilistic analysis — one of the rare cases where average-case running time is rigorously provable.

WHAT — Stability & Other Criteria (Slides 134–137)

Stable sort: preserves original order of equal elements. Merge sort is stable (favor left). Quicksort is NOT stable (swap operation can reorder equal elements).

Why does stability matter? Sorting a spreadsheet by column B without disturbing a previous sort on column A requires stability.

Hybrid algorithms: Timsort (Python's sort) and Introsort (C++ std::sort) use O(n log n) D&C for large n, insertion sort for small n (n < 16), and heap sort as a fallback if recursion depth exceeds O(log n).

✦ Practice Questions — Quicksort
Q1. What input causes quicksort (with first-element pivot) to run in O(n²)?
Solution

An already sorted (or reverse-sorted) array. In this case, every partition picks the minimum (or maximum) as pivot, creating partitions of size 0 and n−1. The recurrence becomes T(n) = T(n−1) + n = O(n²). To avoid this, use median-of-three or random pivot selection.

Q2. Explain why randomized quicksort has expected O(n log n) running time regardless of input.
Solution

With random pivot selection, for any fixed input, each element is equally likely to be chosen as pivot. The expected split is approximately n/2 : n/2. The recurrence in expectation is E[T(n)] ≤ E[T(k)] + E[T(n−k−1)] + O(n) where k is uniform on {0,…,n−1}. Solving this expectation gives E[T(n)] = O(n log n). No adversary can construct a worst-case input because the pivot choice is unpredictable.

TOPIC 09

Graph Theory Fundamentals

Lectures 12 · Slides 138–151 · K&T Ch. 3.1–3.3, CLRS Ch. 22
Graph Definitions & Representations
  • Graph: G = (V, E) where V = vertices, E ⊆ V × V = edges
  • Undirected: (u,v) = (v,u). Directed: (u,v) ≠ (v,u)
  • Adjacency matrix: A[i][j] = 1 if (i,j) ∈ E. Space O(n²). Query O(1). Neighbors: O(n).
  • Adjacency list: For each vertex, list of neighbors. Space O(n+m). Neighbors of v: O(deg(v)). Query: O(deg(v)).
  • Map coloring intro: minimum colors to color a graph with no adjacent same-color vertices
Discussion — Graphs as Universal Models
WHY — Why Graphs Are Everywhere

Graphs are the most versatile data structure in computer science. They model: social networks (vertices = people, edges = friendships), road maps (vertices = intersections, edges = roads with weights = distances), the web (vertices = pages, directed edges = links), dependency graphs (build systems, course prerequisites), and biological networks (protein interactions, neural connectivity). Understanding graph algorithms means you can solve an enormous class of real-world problems.

HOW — Choosing the Right Representation
OperationAdjacency MatrixAdjacency List
SpaceO(n²)O(n + m)
Is (u,v) an edge?O(1)O(deg(u))
All neighbors of uO(n)O(deg(u))
Add edgeO(1)O(1) amortized
Best forDense graphs (m ≈ n²)Sparse graphs (m ≪ n²)

In practice, most real-world graphs are sparse (social networks: avg degree ~100, not 1 million), so adjacency lists are almost always preferred. BFS and DFS run O(n+m) with lists, O(n²) with matrices — a huge difference for large sparse graphs.

WHAT — Key Terminology
  • Path: sequence of vertices v₁,v₂,…,vₖ where each consecutive pair is an edge
  • Cycle: path where v₁ = vₖ
  • Connected: there is a path between every pair of vertices (undirected)
  • Degree: number of edges incident to a vertex
  • Tree: connected acyclic undirected graph on n vertices has exactly n−1 edges

Breadth-First Search (BFS)

Lectures 12–14 · Slides 153–184 · K&T Ch. 3.2, CLRS Ch. 22.2
BFS Algorithm — Queue-Based
function BFS(i):
    for j=1..n: visited[j]=0; Q=[]
    visited[i]=1; append(Q,i)
    while Q is not empty:
        j = extract_head(Q)
        for each (j,k) in E:
            if visited[k]==0:
                visited[k]=1; append(Q,k)
                parent[k]=j
                level[k]=level[j]+1
  • Each vertex enters Q exactly once → O(n) queue operations
  • With adjacency matrix: O(n²). With adjacency list: O(n + m)
  • BFS computes shortest paths in terms of number of edges (not weights)
  • level[v] = shortest path length from source to v
  • parent[v] = predecessor on shortest path → reconstruct path by following parents
Discussion — BFS Deeply
WHY — Why BFS Finds Shortest Paths

BFS explores vertices in order of increasing distance from the source. At any moment, the queue contains vertices at distance d and d+1. When we dequeue a vertex at distance d and explore its neighbors, those neighbors are at distance d+1 — and we only mark them if they haven't been visited, guaranteeing we record the shortest path. This is a classic argument-by-induction on BFS level.

HOW — BFS Tree & Level Structure

BFS builds a BFS tree: when we first visit vertex v from vertex u, we set parent[v] = u and level[v] = level[u] + 1. The BFS tree has the property: for every edge (u,v) in the original graph, |level[u] − level[v]| ≤ 1 (they can differ by at most 1). This immediately proves that the levels represent exact shortest-path distances.

The path from source s to any vertex v is recovered by: v → parent[v] → parent[parent[v]] → … → s, then reversing.

WHAT — Applications Beyond Shortest Paths
  • Connectivity: Run BFS from any vertex. If not all vertices are visited, graph is disconnected.
  • Bipartite testing: Try to 2-color the graph during BFS. If any edge connects two vertices of the same color, graph is not bipartite.
  • Level-synchronous parallelism: BFS structure naturally parallelizes — all vertices at level k can be processed simultaneously.
  • Six degrees of separation: BFS on the Facebook/LinkedIn graph gives "distance" between any two people.
✦ Practice Questions — BFS
Q1. (K&T Ch. 3 style) On a graph with n vertices and m edges, what is the complexity of BFS with adjacency list? Justify your answer.
Solution

Each vertex is enqueued and dequeued exactly once: O(n) total queue operations, each O(1). When we dequeue vertex v, we scan all neighbors of v. Total scanning across all vertices = total sum of degrees = 2m (handshaking lemma). Hence total work = O(n + m).

Q2. Does BFS find shortest paths in weighted graphs? Explain.
Solution

No. BFS minimizes the number of edges, not the total weight. Example: a path of 1 edge with weight 100 vs. a path of 3 edges each with weight 1 (total weight 3). BFS returns the 1-edge path (shortest in hops) but the 3-edge path is shorter in weight. For weighted shortest paths, use Dijkstra's algorithm.

Depth-First Search (DFS) & Applications

Lectures 14–15 · Slides 185–221 · K&T Ch. 3.3, 3.5–3.6, CLRS Ch. 22.3–22.5
DFS Algorithm & DFS Numbering
for j=1..n: visited[j]=0; parent[j]=-1; count=0

function DFS(i):
    visited[i]=1; pre[i]=count; count++
    for each (i,j) in E:
        if visited[j]==0:
            parent[j]=i
            DFS(j)
    post[i]=count; count++
  • Complexity: O(n²) with matrix, O(n+m) with adjacency list
  • DFS numbering: pre[v] = enter time, post[v] = exit time
  • Edge types in directed graphs: Tree, Forward, Back, Cross edges
  • Back edge ↔ Cycle: A directed graph has a cycle iff DFS reveals a back edge
  • Interval nesting: [pre(u),post(u)] ⊃ [pre(v),post(v)] iff u is an ancestor of v in DFS tree
Discussion — DFS Numbering and Its Power
WHY — Why DFS Over BFS?

BFS gives shortest paths — but DFS gives something richer: a complete structural decomposition of the graph. The pre/post numbering reveals the depth structure. From this one run, you can extract: all cycles, topological orderings, strongly connected components, cut vertices, bridges, and more. These structural properties are needed for compiler design (dependency analysis), network analysis, and scheduling.

HOW — Edge Classification (CLRS 22.3)

For a directed graph, classify each non-tree edge (u,v):

  • Tree edge: v was first discovered via u in DFS. [pre(u),post(u)] ⊃ [pre(v),post(v)]
  • Forward edge: (u,v) where v is a descendant of u not via tree edge. pre(u) < pre(v) < post(v) < post(u)
  • Back edge: (u,v) where v is an ancestor of u. pre(v) < pre(u) < post(u) < post(v). Indicates cycle.
  • Cross edge: all other edges. Intervals [pre(u),post(u)] and [pre(v),post(v)] are disjoint.
WHAT — Strongly Connected Components (SCCs)

Kosaraju's Algorithm (O(n+m)): (1) Run DFS on G, record finish times. (2) Transpose graph G^T. (3) Run DFS on G^T in decreasing finish-time order. Each DFS tree in step 3 is a SCC. This is covered in Slides 215–220 and CLRS 22.5.

✦ Practice Questions — DFS
Q1. (K&T 3.5 style) Prove: a directed graph has a cycle if and only if DFS reveals a back edge.
Solution

(⇒) Suppose there's a cycle v₁→v₂→…→vₖ→v₁. Consider the first vertex in the cycle that DFS visits — say v₁. DFS will eventually reach vₖ (all vertices in cycle are reachable from v₁). When DFS processes vₖ, the edge vₖ→v₁ is a back edge (since v₁ is still on the DFS stack).

(⇐) A back edge (u,v) means v is an ancestor of u in the DFS tree, so there's a path v → … → u in the tree. Combined with the edge u → v, we have a cycle.

Q2. What are cut vertices (articulation points)? How does DFS numbering help find them?
Solution

A cut vertex is a vertex whose removal disconnects the graph. Using DFS: vertex u is a cut vertex if: (1) u is the root of the DFS tree and has ≥ 2 children, OR (2) u is not the root and has a child v such that no vertex in the subtree rooted at v has a back edge to an ancestor of u. This can be computed in O(n+m) using "low" values (lowest pre[] reachable from a subtree via back edges).

DAGs & Topological Ordering

Lectures 15–17 · Slides 222–263 · K&T Ch. 3.6, CLRS Ch. 22.4
DAGs, Topological Sort, Longest Path
function TopologicalOrder(G):  // O(n+m) with adjacency list
    compute indegree[i] for all i  // O(m)
    put all 0-indegree vertices in Queue
    while Queue not empty:
        j = remove_head(Queue)
        enumerate j
        for (j,k) in E:
            indegree[k] -= 1
            LPT[k] = max(LPT[k], 1 + LPT[j])
            if indegree[k]==0: add k to Queue
  • DAG = Directed Acyclic Graph (no directed cycles)
  • Topological sort lists vertices so that for every edge (u,v), u comes before v
  • Exists iff graph is a DAG. Computable in O(n+m).
  • Longest path in DAG = minimum number of "rounds" to complete all tasks
  • Contrast: Longest path in general graph is NP-hard!
Discussion — DAGs and Scheduling
WHY — DAGs Model Dependency

Whenever you have tasks with prerequisites, you have a DAG. Examples: course prerequisites (COMP201 before COMP570), build systems (Makefile: compile A before linking B), project scheduling (Critical Path Method in project management), spreadsheet recalculation (recompute cells in dependency order). The topological sort gives you a valid execution order.

HOW — Two Algorithms for Topological Sort

Kahn's Algorithm (used in slides): Repeatedly remove vertices with indegree 0. O(n+m) with adjacency list. Directly gives topological order and can detect cycles (if any vertex is never removed, a cycle exists).

DFS-based (CLRS 22.4): Run DFS; output vertices in decreasing order of finish time. Correctness: for any edge (u,v), post(u) > post(v) (v finishes before u). So decreasing finish time gives valid topological order.

WHAT — Critical Path / Longest Path

The "longest path in a DAG" problem models the critical path in project scheduling — the minimum time to complete all tasks given parallelism. Algorithm: compute LPT[v] = longest path ending at v. Process vertices in topological order; for each vertex v and each incoming edge (u,v): LPT[v] = max(LPT[v], 1 + LPT[u]). This is O(n+m) — a beautiful application of dynamic programming on DAGs.

Important contrast: Longest path in a DAG = O(n+m). Longest simple path in a general graph = NP-hard. The DAG structure is what makes it tractable.
✦ Practice Questions — DAGs
Q1. (K&T 3.6) Given a DAG with 8 vertices and the dependency structure from Slides 231–246, what is the minimum number of semesters to complete all courses if you can take any courses with satisfied prerequisites in the same semester?
Solution Approach

Compute the longest path in the DAG. Process vertices in topological order, setting LPT[v] = 1 + max{LPT[u] : (u,v) is an edge}. The answer is max{LPT[v]}. For the example in Slide 244, the longest path has length 4 (vertices with LPT values 0,1,2,3,4 along the critical path), so the answer is 5 semesters (length+1 to account for the starting semester).

Q2. If a directed graph has n vertices and you run Kahn's topological sort algorithm but only n−1 vertices get enumerated, what can you conclude?
Solution

The graph has a directed cycle. One vertex remains unenumerated because it always has positive indegree (it's part of a cycle — each vertex in the cycle waits for the previous one to be removed first, creating a deadlock). This gives an O(n+m) cycle detection algorithm as a bonus of Kahn's algorithm.

Dijkstra's Algorithm — Single Source Shortest Paths

Lectures 17–18 · Slides 264–291 · K&T Ch. 4.4, CLRS Ch. 24.3
Dijkstra's Algorithm
function Dijkstra(s):
    for i=1..n: Visited[i]=False; Distance[i]=∞
    Distance[s]=0
    for i=1..n:
        // Find unvisited vertex with minimum distance
        u = argmin{Distance[u] : Visited[u]==False}
        Visited[u]=True
        for each edge (u,v) with Visited[v]==False:
            if Distance[v] > Distance[u] + weight(u,v):
                Distance[v] = Distance[u] + weight(u,v)
  • Precondition: All edge weights must be non-negative
  • Complexity: O(n²) with array; O((n+m) log n) with priority queue (min-heap)
  • Greedy algorithm: always processes the closest unvisited vertex
  • Correctness proven by induction: shortest paths to "burned" vertices are final
Discussion — Dijkstra Deeply
WHY — The Burning Forest Intuition

Dijkstra's algorithm has an elegant physical interpretation: imagine setting fire to the source vertex. Fire spreads along edges at a rate proportional to edge weight. Vertices "burn" in order of their distance from the source. The first time a vertex burns, it burns via the shortest path — there's no shortcut through as-yet-unburned territory (since all edge weights are positive, a detour through an unburned vertex can only be longer).

HOW — Correctness Proof (K&T 4.4)

Claim: When vertex u is extracted (burned), Distance[u] = true shortest path distance from s.

Proof by induction on number of burned vertices:

  • Base case: s is burned first with Distance[s] = 0. Correct since s has no self-loop and all weights ≥ 0. ✓
  • Inductive step: Suppose the claim holds for all previously burned vertices. When we burn u, suppose for contradiction there's a shorter path P from s to u. P must leave the burned set at some edge (x,y). But then Distance[y] ≤ dist(s,y) ≤ dist(s,u) (since P passes through y before u). Since all remaining edge weights are ≥ 0, the path from y to u adds more, so Distance[y] ≤ Distance[u]. But then Dijkstra would have burned y before u — contradiction!

Why non-negative weights are essential: If a negative-weight edge exists, we could find a shorter path to an already-burned vertex by going through a later-burned vertex with a negative edge — invalidating the induction.

WHAT — Complexity with Priority Queue

Naive implementation: O(n) to find minimum each round, n rounds = O(n²) total.

With a binary min-heap (priority queue): each extract-min takes O(log n); each decrease-key (updating Distance[v]) takes O(log n); at most m updates total → O((n + m) log n) = O(m log n) for connected graphs.

With Fibonacci heap (theoretical): O(m + n log n) — optimal for dense graphs, but complex to implement.

✦ Practice Questions — Dijkstra's Algorithm

Solution (from Slide 272 example)

Q2. Why does Dijkstra's algorithm fail with negative edge weights? Give a concrete counterexample.
Solution

Consider: s → a (weight 3), s → b (weight 2), b → a (weight −2). True shortest path: s → b → a with total weight 0. Dijkstra burns b first (distance 2), then a (distance 3 via s→a), but the correct distance to a is 0. The algorithm misses the negative shortcut through b because b was burned after a was already "settled".

Bellman-Ford Algorithm — Handling Negative Weights

Lectures 18–21 · Slides 288–322 · K&T Ch. 6.8, CLRS Ch. 24.1
Bellman-Ford Algorithm
function BellmanFord(s):
    for i=1..n: Distance[i]=∞
    Distance[s]=0
    for i=1 to n-1:       // Repeat n-1 times
        for each edge (j,k) in E:
            Distance[k] = min(Distance[k],
                              Distance[j] + weight(j,k))
    // Optional: detect negative cycles
    for each edge (j,k) in E:
        if Distance[k] > Distance[j] + weight(j,k):
            report negative cycle
  • Handles negative weights (but not negative cycles)
  • Complexity: O(nm) with adjacency list; O(n³) with matrix
  • Key insight: after k iterations, correct shortest path distances for paths using ≤ k edges
  • After n−1 iterations: all shortest paths have at most n−1 edges (since they're simple)
Discussion — Bellman-Ford and Dynamic Programming
WHY — When Do Negative Weights Arise?

Negative weights aren't just a theoretical curiosity. In chemistry, reactions can release energy (negative weight = energy gain). In finance, some routes earn money (an airline segment where you earn more in cargo than you spend on fuel). In arbitrage detection, a negative cycle in a currency exchange graph means an infinite profit opportunity — which is why Bellman-Ford is used in financial systems to detect such cycles.

HOW — Correctness via Dynamic Programming (K&T 6.8)

Key insight: Any simple shortest path has at most n−1 edges. Define OPT(i, v) = minimum cost of a path from s to v using at most i edges.

Recurrence: OPT(i, v) = min(OPT(i−1, v), min over edges (u,v): OPT(i−1, u) + w(u,v))

This is exactly what Bellman-Ford computes. After n−1 iterations, OPT(n−1, v) = true shortest distance to v.

Negative cycle detection: If OPT(n, v) < OPT(n-1, v) for any v, then a negative cycle exists (the shortest "path" keeps shrinking beyond n−1 edges because it loops a negative cycle).

Solution Summary

Q2. Explain: "shortest paths will never loop" — why does this guarantee Bellman-Ford works in n−1 iterations?
Solution

If there are no negative cycles, any shortest path is a simple path (never visits a vertex twice — if it did, removing the loop can only decrease or maintain cost since there are no negative cycles). A simple path on n vertices has at most n−1 edges. Therefore, after n−1 rounds of relaxation, we've covered all possible simple shortest paths. If running a 0th round would still improve any distance, a negative cycle must exist.

Floyd-Warshall — All-Pairs Shortest Paths

Lectures 21–22 · Slides 323–342 · CLRS Ch. 25.2, K&T Ch. 6.10
Floyd-Warshall Algorithm
function FloydWarshall():
    for i=1..n, j=1..n:
        W[i][j][0] = weight(i,j) if edge exists, else ∞
    for k=1..n:
        for i=1..n:
            for j=1..n:
                W[i][j][k] = min(W[i][j][k-1],
                                  W[i][k][k-1] + W[k][j][k-1])
  • W[i][j][k] = shortest path from i to j using only intermediate vertices in {1,…,k}
  • Complexity: O(n³) time, O(n²) space (only need 2 slices at a time)
  • Handles negative weights (no negative cycles)
  • Warshall's algorithm: Same structure but for transitive closure (reachability)
Discussion — Floyd-Warshall as Dynamic Programming
WHY — All-Pairs vs Single-Source

Sometimes you need distances between ALL pairs of vertices (railway timetable, road atlas, "6 degrees" for all pairs). Running Dijkstra n times gives O(n · m log n). For dense graphs (m ≈ n²), Floyd-Warshall's O(n³) is competitive. For sparse graphs, repeated Dijkstra is better. The real advantage of Floyd-Warshall is its simplicity: just 3 nested loops.

HOW — The DP Recurrence (CLRS 25.2)

Define W_k(i,j) = length of shortest path from i to j using only {1,2,…,k} as intermediate vertices.

  • Case 1: Shortest path from i to j via {1,…,k} doesn't use vertex k → W_k(i,j) = W_{k-1}(i,j)
  • Case 2: Path goes through k → split as i→k→j, both using {1,…,k-1} as intermediates → W_k(i,j) = W_{k-1}(i,k) + W_{k-1}(k,j)
  • Combine: W_k(i,j) = min(W_{k-1}(i,j), W_{k-1}(i,k) + W_{k-1}(k,j))

Note k can appear at most once on any shortest path (since there are no negative cycles → no benefit from revisiting k). This is the key substructure property.

WHAT — Warshall's Transitive Closure

The exact same recurrence works for boolean reachability: P_k(i,j) = 1 iff there's a path from i to j through intermediates {1,…,k}. Replace min(a,b) with OR, and addition with AND. This was the original Warshall algorithm for computing the transitive closure (path matrix) of a graph.

✦ Practice Questions — Floyd-Warshall
Q1. (CLRS 25.2) What is the space complexity of Floyd-Warshall if we naively store W[i][j][k] for all k? How do we reduce it?
Solution

Naively: O(n³) — we store an n×n matrix for each of the n+1 values of k. Optimization: at iteration k, we only use W[...][...][k-1] to compute W[...][...][k]. So we only need two n×n slices at a time: O(n²) space. In practice, we can even update in-place (same matrix) — it can be shown this gives the same result due to the recurrence structure.

Q2. After running Floyd-Warshall, how do you check if a negative cycle exists?
Solution

Check the diagonal: if W[i][i] < 0 for any vertex i, there is a negative cycle (a path from i to i with negative total weight). Without negative cycles, W[i][i] = 0 for all i (the empty path from i to i has cost 0, and no shorter path exists).

Integer Multiplication & Karatsuba's Algorithm

Lectures 23 · Slides 348–360 · K&T Ch. 5.5, CLRS Ch. 30 (related)
Grade-School vs. Divide-and-Conquer Multiplication
  • Grade-school multiplication: O(n²) bit operations for two n-bit integers
  • Kolmogorov's conjecture (1956): O(n²) is optimal
  • Karatsuba's refutation (1960): Only 3 recursive calls needed, not 4!
// x = 2^m·a + b,  y = 2^m·c + d,  m = n/2
// xy = 2^2m·ac + 2^m·(bc+ad) + bd
// Karatsuba: bc+ad = ac + bd - (a-b)(c-d)
// Only 3 multiplications: ac, bd, (a-b)(c-d)
T(n) = 3T(n/2) + Θ(n)
     → T(n) = Θ(n^log₂3) ≈ Θ(n^1.585)
Discussion — Karatsuba's Breakthrough
WHY — Why Integer Multiplication Matters

Big integer multiplication is the core operation in cryptography (RSA, elliptic curves), scientific computing, and arbitrary-precision arithmetic. Modern cryptographic keys are 2048–4096 bits. Multiplying such numbers with O(n²) bit operations is costly; Karatsuba's O(n^1.585) and more modern algorithms (Schönhage-Strassen O(n log n log log n), Harvey-van der Hoeven 2019 O(n log n)) are essential. GNU's GMP library uses different algorithms depending on input size.

HOW — The Karatsuba Identity

Naive D&C: xy = 2^2m·(ac) + 2^m·(bc + ad) + bd requires 4 half-size multiplications. Karatsuba's insight:

bc + ad = ac + bd − (a−b)(c−d)

This requires only 3 multiplications: (ac), (bd), (a−b)(c−d). The trick: compute ac and bd for the leading/trailing terms anyway, then use the identity to get the middle term "for free". Additions/subtractions are O(n) — much cheaper than multiplication.

Recurrence: T(n) = 3T(n/2) + O(n). By Master Theorem Case 1 (log₂3 ≈ 1.585 > 1): T(n) = Θ(n^log₂3) ≈ O(n^1.585).

TOPIC 17

Computational Intractability, NP, & Polynomial Reductions

Lectures 25–27 · Slides 373–389 · K&T Ch. 8.1–8.4, CLRS Ch. 34
P, NP, Reductions — Key Definitions & Problems
  • P: problems solvable in polynomial time
  • NP: problems where solutions can be verified in polynomial time
  • NP-hard: at least as hard as any NP problem (via polynomial reductions)
  • NP-complete: NP-hard AND in NP
  • Polynomial reduction: X ≤_P Y means X can be solved using a polynomial-time algorithm for Y as a subroutine
  • Key problems: 3-SAT, Independent Set, Vertex Cover, Set Cover
  • 3-SAT ≤_P Independent Set (proven in Slide 389)
  • Independent Set ⟺_P Vertex Cover (S is independent set ↔ V−S is vertex cover)
  • Vertex Cover ≤_P Set Cover
Discussion — The Hardest Open Problem in CS
WHY — Why Intractability Theory Matters

The P vs. NP question is arguably the most important open problem in mathematics and computer science. Its resolution would affect cryptography (most of the internet's security), optimization (logistics, scheduling), AI (can machines learn efficiently?), and mathematics itself. In practice, recognizing that a problem is NP-hard tells you to stop looking for an exact polynomial algorithm and instead consider: approximation algorithms, heuristics, special-case algorithms, or integer programming.

HOW — The Reduction Chain

The reduction chain from Lecture 25–27 establishes relative hardness:

3-SAT ≤_P INDEPENDENT-SET ≤_P VERTEX-COVER ≤_P SET-COVER

Independent Set ↔ Vertex Cover (Slide 381): Complementarity theorem: S is an independent set iff V−S is a vertex cover. Proof: If S is independent, no edge has both endpoints in S, so every edge has at least one endpoint in V−S (vertex cover). Conversely, if V−S is a vertex cover, every edge has an endpoint in V−S, so no edge has both endpoints in S (independent set).

Vertex Cover ≤_P Set Cover (Slide 385): Given a graph G=(V,E) and integer k, construct Set Cover instance with universe U=E, one set S_v = {edges incident to v} for each vertex v. G has a vertex cover of size k iff the set cover instance has a cover of size k.

WHAT — Key NP-Complete Problems to Know
  • 3-SAT: satisfying a boolean formula in 3-CNF form. The "base" NP-complete problem (by Cook's Theorem).
  • Independent Set: find maximum set of vertices with no edges between them.
  • Vertex Cover: find minimum set of vertices covering all edges.
  • Set Cover: minimum number of subsets whose union = universe.
  • Clique: find a complete subgraph of size k.
  • Hamiltonian Path: visit every vertex exactly once.
  • TSP: shortest tour visiting every vertex (generalization of Hamiltonian Path).
Key Fact: All NP-complete problems are polynomial-time reducible to each other. Solving any one in polynomial time would solve all NP problems in polynomial time (P = NP).

SAT and Electronic Design (Slide 387)

3-SAT: A CNF formula with exactly 3 literals per clause. E.g., (x₁ ∨ x₂ ∨ ¬x₃) ∧ (¬x₁ ∨ x₂ ∨ x₄) ∧ …. SAT solvers are used heavily in chip verification and formal methods — despite the worst case being exponential, modern SAT solvers handle formulas with millions of variables in practice using sophisticated heuristics.

✦ Practice Questions — NP & Reductions
Q1. (K&T Ch. 8.2) Prove: if X ≤_P Y and Y can be solved in polynomial time, then X can be solved in polynomial time.
Solution

By definition of polynomial reduction, any instance of X can be solved using a polynomial number of steps plus a polynomial number of calls to a polynomial-time oracle for Y. Each call to the oracle takes polynomial time. A polynomial number of polynomial-time operations is still polynomial (composition of polynomials is polynomial). Therefore X can be solved in polynomial time overall.

Q2. (K&T Ch. 8.2) Prove the complement theorem: S is an independent set in G iff V−S is a vertex cover. Use this to show INDEPENDENT-SET ≡_P VERTEX-COVER.
Solution

Theorem: S is independent ↔ V−S is a vertex cover.

Proof (⇒): Let S be independent. For any edge (u,v) ∈ E, since S is independent, not both u and v are in S. So at least one of u,v is in V−S. Thus V−S covers every edge. ✓

Proof (⇐): Let V−S be a vertex cover. For any edge (u,v), at least one endpoint is in V−S. So it's impossible that both endpoints are in S. Thus no two vertices in S share an edge; S is independent. ✓

Reduction: To solve INDEPENDENT-SET(G, k): call VERTEX-COVER(G, n−k). Return "yes" iff VERTEX-COVER returns "yes". The reduction is O(1) (same graph, different k). Hence both directions reduce to each other in constant time: INDEPENDENT-SET ≡_P VERTEX-COVER.

Q3. What is the significance of the reduction 3-SAT ≤_P INDEPENDENT-SET for the theory of NP-completeness?
Solution

3-SAT is NP-complete (this follows from Cook's Theorem). The reduction 3-SAT ≤_P INDEPENDENT-SET means that if we could solve INDEPENDENT-SET in polynomial time, we could also solve 3-SAT (and therefore all NP problems) in polynomial time. This establishes that INDEPENDENT-SET is NP-hard. Since INDEPENDENT-SET is also in NP (a solution — the independent set itself — can be verified in polynomial time), it is NP-complete. The reduction thus "transfers" hardness from 3-SAT to INDEPENDENT-SET.

Q4. (Reading Exercise — Slide 390 / K&T Ch. 8) Is the decision version of VERTEX-COVER (does G have a vertex cover of size ≤ k?) in NP? Justify.
Solution

Yes. A certificate is the vertex cover itself (a subset S ⊆ V with |S| ≤ k). To verify: (1) check |S| ≤ k in O(n), (2) for every edge (u,v) ∈ E, check that at least one of u,v is in S in O(m) total. Total verification time: O(n + m) = polynomial. Since solutions can be verified in polynomial time, VERTEX-COVER ∈ NP.

Algorithm Design Paradigms — At a Glance
DIVIDE & CONQUER

Split into subproblems of size n/b, solve recursively, recombine. T(n) = aT(n/b) + f(n). Use when: subproblems are independent and recombination is efficient. Examples: Merge sort, Binary search, Karatsuba multiplication.

GREEDY

Make locally optimal choice at each step, never reconsider. Requires proof (exchange argument or inductive correctness). Examples: Dijkstra, Kahn's topological sort, Gale-Shapley. Key pitfall: greedy often fails — you must prove optimality.

DYNAMIC PROGRAMMING

Solve overlapping subproblems, store results (memoization/tabulation). Optimal substructure + overlapping subproblems. Examples: Bellman-Ford, Floyd-Warshall, Longest Path in DAG. Coming in later lectures: interval scheduling, sequence alignment, knapsack.

✦ Final Exam Style Questions
Q1. Design an O(n + m) algorithm to detect whether an undirected graph is bipartite (2-colorable).
Solution

Use BFS. Assign source to color 0. For each edge (u,v): if v is uncolored, assign it the opposite color of u. If v is already colored the same as u, the graph is not bipartite. If BFS completes without conflict, the graph is bipartite.

Correctness: A graph is bipartite iff it has no odd-length cycle. BFS finds conflicts iff an odd cycle exists.

Complexity: O(n + m) — standard BFS complexity.

Q2. (Design algorithms) You are given a DAG where each edge (u,v) has weight w(u,v). Design an O(n+m) algorithm to find the shortest path from a source s to all other vertices.
Solution

Use topological sort + dynamic programming. This works even with negative weights (no Dijkstra needed) because DAGs have no negative cycles.

  1. Compute topological order of vertices in O(n+m).
  2. Initialize dist[s] = 0, dist[v] = ∞ for all v ≠ s.
  3. Process vertices in topological order. For each vertex u, for each edge (u,v): dist[v] = min(dist[v], dist[u] + w(u,v)).

Correctness: When we process u in topological order, all vertices that can reach u have already been processed, so dist[u] is already correct (by induction). Hence updating v from u gives the correct shortest path.

Complexity: Topological sort O(n+m) + one pass over all edges O(m) = O(n+m) total.
