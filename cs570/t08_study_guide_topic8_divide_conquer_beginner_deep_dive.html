🧗 Warm-up Glossary — Vocabulary You Need Before §1

Divide & Conquer builds on recursion, logarithms, and array manipulation. If any of these terms feel shaky, read the entry carefully before moving to §1.

What is recursion? 📄 p. 79

Recursion is when a function calls itself on a smaller version of the problem. Think of a Russian nesting doll (Matryoshka): each doll contains a smaller identical doll inside it, until you reach the smallest one that contains nothing. In code, that "smallest doll that contains nothing" is the base case — the version of the problem you can answer directly without recursing. 

  factorial(4) calls factorial(3)
    factorial(3) calls factorial(2)
      factorial(2) calls factorial(1)
        factorial(1) = 1              ← BASE CASE (no more recursion)
      factorial(2) = 2 × 1 = 2
    factorial(3) = 3 × 2 = 6
  factorial(4) = 4 × 6 = 24
SOURCE: 📄 p. 79

What is a base case? 📄 p. 79

A base case is the simplest input to a recursive function that can be solved immediately without making another recursive call. Without a base case, recursion would continue forever (a "stack overflow"). In Merge Sort: the base case is an array of size 1 — a single element is already sorted.

What is a recurrence relation? 📄 p. 79

A recurrence relation is an equation that describes a function's value in terms of its own value on a smaller input. Merge Sort's running time T(n) satisfies: T(n) = 2T(n/2) + n. This reads: "To sort n elements, I sort two halves of size n/2 (costing 2T(n/2)) and then merge them (costing n extra work)."

What is logarithm base 2, and why does it appear everywhere? 📄 p. 39

log₂(n) answers the question: "How many times do I have to divide n by 2 before I get down to 1?" Example: log₂(8) = 3 because 8 → 4 → 2 → 1 (three halvings). log₂(16) = 4 because 16 → 8 → 4 → 2 → 1. In Divide & Conquer algorithms that halve the input at each step (like Merge Sort or Binary Search), the recursion has exactly log₂(n) levels — which is why O(log n) appears so frequently.

What is a recursion tree? 📄 p. 79

A recursion tree is a diagram that maps out all the recursive calls an algorithm makes. Each node in the tree represents one call; the root is the original call on size n; leaves are the base cases. The total work done by the algorithm is the sum of work done at every node in the tree. Visualizing the recursion tree is the most intuitive way to understand why Merge Sort runs in O(n log n).

What does "divide and conquer" mean structurally? 📄 p. 79

Every Divide & Conquer algorithm has three phases: (1) Divide: split the input into smaller sub-problems. (2) Conquer: solve each sub-problem recursively (or directly if small enough). (3) Combine: merge the sub-problem solutions into a solution for the original input. The key insight is that if the sub-problems are significantly smaller (e.g., half the size), and the combine step is cheap (e.g., O(n)), the total work can be much better than a naïve O(n²) approach.

What is a matrix, and how does multiplication work? 📄 p. 239

A matrix is a rectangular grid of numbers with rows and columns. An n×n matrix has n rows and n columns. To multiply two n×n matrices A and B to get C, the entry C[i][j] is the dot product of row i of A with column j of B: C[i][j] = A[i][1]·B[1][j] + A[i][2]·B[2][j] + ... + A[i][n]·B[n][j]. The naïve algorithm computes all n² entries, each taking O(n) work: total O(n³). 📄 p. 239

What is O(n log n) and why is it better than O(n²)? 📄 p. 91

O(n²) means the work roughly scales as the square of the input. For n = 1,000,000 elements, O(n²) = 1012 operations — infeasible. O(n log n) for n = 1,000,000 is roughly 20,000,000 operations — perfectly fast. The lecture slide states: "Assuming 108 operations per second, feasible input size goes from 10,000 (for O(n²)) to 10,000,000 (for O(n log n))." 📄 p. 91

What is a "bit" and what is bit-shifting? 📄 p. 227

A bit is a single binary digit: 0 or 1. An n-bit integer uses n bits to represent its value. Bit-shifting left by m positions multiplies a number by 2m; shifting right divides by 2m. In the Karatsuba and D&C multiplication algorithms, multiplying by 2m is done in O(1) time using bit-shifts — this is the "add and shift" step that costs only O(n) bit operations. 📄 p. 227

What is a "pivot" in sorting? 📄 p. 95

A pivot is a chosen element from the array used to split the array into two parts: elements smaller than the pivot and elements larger than it. Quicksort uses a pivot to partition the array. After partitioning, the pivot is in its final sorted position, and we recursively sort the left and right halves. 📄 p. 95

What is "stable sorting"? 📄 p. 103

A stable sorting algorithm preserves the relative order of equal elements. If students Alice and Bob have the same score, and Alice was listed before Bob before sorting, a stable sort will keep Alice before Bob after sorting. Merge Sort is stable (with careful implementation); Quicksort is NOT stable. 📄 p. 103

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 79–104 (Lec 10–11) and pp. 227–248 (Lec 23–25)

1.1 The fundamental problem — why does "split in half, solve each half" work so well? 📄 p. 79

Imagine you are a chef who needs to chop 16 kilograms of vegetables. You could do it all yourself, or you could split the pile in half, hand 8 kg to another chef, and have both of you chop simultaneously. Then split again: each of you hands half to a third helper. After log₂(16) = 4 rounds of splitting, every chef has just 1 kg — trivially easy. Then you collect and combine the results. The total work done (all vegetable-chopping, everywhere) is 16 kg × 4 rounds = 64 kg-steps, not 16² = 256 kg-steps. That's the essence of Divide & Conquer. 📄 p. 79

The same principle drives Merge Sort (split the array in half, sort each half, merge), Karatsuba multiplication (split n-bit numbers into halves, reduce 4 multiplications to 3), and Strassen's algorithm (split n×n matrices into n/2 × n/2 blocks, reduce 8 multiplications to 7). In each case, shaving just one recursive call off the "obvious" approach produces a dramatically faster algorithm. 📄 pp. 79, 227, 239

📌 Core Intuition The power of D&C comes from halving. Halving creates log₂(n) levels in the recursion tree. If each level does O(n) total work, the total is O(n log n). If each level does more work — as in Strassen where matrix entries must be processed — the total is O(n^c) for c slightly above 2. 📄 pp. 79–80

1.3 A second framing — the recursion tree as a cost ledger 📄 p. 79

Think of the recursion tree as a company ledger. The root (top level) is the CEO solving a size-n problem. The CEO splits the work and delegates to two VPs (each solving a size-n/2 problem). Each VP delegates to two managers (each solving n/4). This continues until every worker is handling a single element (base case).

The key question is: how much does each level of the tree cost? In Merge Sort: Level 0 (CEO): one problem of size n → costs n. Level 1 (2 VPs): two problems of size n/2 → costs 2 × (n/2) = n. Level 2 (4 managers): four problems of size n/4 → costs 4 × (n/4) = n. ...each level costs n total, and there are log₂(n) levels... Total cost = n × log₂(n) = O(n log n). 📄 pp. 79–90

1.4 Technical definition — recurrences and the Master Theorem connection 📄 pp. 79–80, 227

The running time of a D&C algorithm satisfies a recurrence of the form: T(n) = a · T(n/b) + f(n), where a = number of sub-problems, b = factor by which input shrinks, and f(n) = work done at this level (divide + combine steps).

AlgorithmRecurrencea (sub-probs)b (shrink factor)f(n) (combine cost)Result

⚠️ Common Point of Confusion — "Fewer sub-problems = always faster" Reducing the number of recursive calls (a) only helps if the combine step f(n) doesn't grow too fast. Karatsuba reduces a from 4 to 3 with the SAME combine cost O(n) → wins. But if reducing a required a combine step of O(n²), the savings might disappear. The Master Theorem precisely captures this trade-off. 📄 p. 227

2a Merge Sort — The Canonical D&C Algorithm, Fully Traced Lecture PDF: pp. 79–90 (Lecture 10)

Intuition: Sorting by repeatedly splitting in half 📄 p. 79

Consider sorting a shuffled deck of cards. One approach: split the deck in half, hand each half to a friend, have them sort their halves, then interleave the two sorted halves into one sorted deck. This interleaving step ("merging") is the key — if both halves are already sorted, merging takes only O(n) time by comparing the front cards of each half and moving the smaller one to the output.

The trick is recursion: each friend sorting a half does the same thing — splits their half into two quarters, hands to helpers, etc. The recursion bottoms out at single cards (already sorted). 📄 p. 79

Mechanism: Pseudocode for Merge Sort 📄 pp. 79–90

function MergeSort(A, left, right):
    # Sort A[left .. right-1]
    if right - left == 1:          # base case: single element
        return [A[left]]           # already sorted

    mid = (left + right) // 2      # divide: find midpoint
    L = MergeSort(A, left, mid)    # conquer: sort left half
    R = MergeSort(A, mid, right)   # conquer: sort right half
    return Merge(L, R)             # combine: merge sorted halves

function Merge(A, B):
    # Merge two sorted arrays A and B into one sorted array C
    C = []; i = 0; j = 0
    while i < len(A) and j < len(B):
        if A[i] <= B[j]:
            C.append(A[i]); i += 1
        else:
            C.append(B[j]); j += 1
    C.extend(A[i:])  # remaining elements of A (if any)
    C.extend(B[j:])  # remaining elements of B (if any)
    return C

Deriving the recurrence and solving it 📄 pp. 79–90

Step 1: Write the recurrence.
T(1) = 1 (base case: one element takes constant time). T(n) = 2T(n/2) + n (two recursive calls on halves, plus O(n) merge). 📄 p. 80
Step 2: Expand (unroll) the recurrence.
T(n) = 2T(n/2) + n
= 2[2T(n/4) + n/2] + n = 4T(n/4) + 2n
= 4[2T(n/8) + n/4] + 2n = 8T(n/8) + 3n
= 2jT(n/2j) + j·n (pattern after j levels) 📄 pp. 80–81
Step 3: Find when the base case is reached.
When n/2j = 1, i.e., j = log₂(n). At this point: T(n/2j) = T(1) = 1. 📄 p. 81
Step 4: Substitute j = log₂(n).
T(n) = 2log₂n · T(1) + log₂(n) · n = n · 1 + n log₂(n) = n + n log n. 📄 p. 81
Step 5: Conclusion.
T(n) = n + n log n = O(n log n) ✓ This confirms what the recursion tree showed: log n levels, each costing n total. 📄 p. 81
⚠️ Merge Sort Shortcomings Despite being O(n log n), Merge Sort has two drawbacks: (1) It requires O(n) extra space for the merged array — no obvious way to merge in-place efficiently. (2) It is inherently recursive — recursive calls and returns have overhead. This is why Quicksort, despite being O(n²) in the worst case, is often faster in practice for in-memory sorting. 📄 pp. 91–95
2b Quicksort — D&C Without Merging: Partition, Recurse, Done Lecture PDF: pp. 95–104 (Lecture 11)

Intuition: What if we divided so merging was free? 📄 p. 95

Merge Sort's weakness is the O(n) merge step. What if we could divide the array so that everything to the left is smaller than everything to the right? Then after sorting both halves recursively, the whole array is already sorted — no merge needed! This is Quicksort's insight. 📄 p. 95

The trick is the pivot: choose one element as the pivot, then rearrange ("partition") the array so all smaller elements come before the pivot and all larger elements come after. After partitioning, the pivot is in its final sorted position. Recursively sort the two sub-arrays on either side. 📄 p. 95

Mechanism: Quicksort pseudocode 📄 pp. 95–104

function Quicksort(A, l, r):
    # Sort A[l .. r-1]
    if r - l <= 1:
        return          # base case: 0 or 1 element

    # Partition: choose pivot = A[l], rearrange in-place
    yellow = l + 1      # boundary: elements A[l+1..yellow-1] <= pivot
    for green in range(l + 1, r):
        if A[green] <= A[l]:     # A[l] is the pivot
            swap(A, yellow, green)
            yellow += 1
    swap(A, l, yellow - 1)       # move pivot to its final position

    Quicksort(A, l, yellow)      # sort left part (all <= pivot)
    Quicksort(A, yellow + 1, r)  # sort right part (all > pivot)

2c Karatsuba Multiplication — One Less Recursive Call Changes Everything Lecture PDF: pp. 227–238 (Lecture 23)

Intuition: From 4 multiplications to 3 📄 pp. 227–238

To multiply two n-bit numbers x and y, the naïve D&C approach splits each into two n/2-bit halves: x = 2m·a + b and y = 2m·c + d (where m = n/2). Then xy = (2ma + b)(2mc + d) = 22mac + 2m(bc + ad) + bd.

This requires computing four sub-products: ac, ad, bc, bd. Each is a multiplication of two n/2-bit numbers. So T(n) = 4T(n/2) + O(n). Using the Master Theorem: log₂(4) = 2, so T(n) = O(n²) — same as the naïve grade-school algorithm! No improvement. 📄 p. 227

Karatsuba's insight (1960): We only need bc + ad (not bc and ad separately), and we can get bc + ad with just ONE extra multiplication using the identity: 📄 pp. 227–238 

  KEY IDENTITY:
  bc + ad = ac + bd − (a − b)(c − d)

  So instead of computing all 4 products, compute:
    p₁ = ac        (one recursive call)
    p₂ = bd        (one recursive call)
    p₃ = (a−b)(c−d) (one recursive call)

  Then: bc + ad = p₁ + p₂ − p₃

  xy = 2^(2m) · p₁ + 2^m · (p₁ + p₂ − p₃) + p₂

  TOTAL: 3 recursive multiplications (not 4)!
  T(n) = 3T(n/2) + O(n)
SOURCE: 📄 pp. 227–238

Mechanism: Karatsuba pseudocode 📄 pp. 227–238

function KaratsubaMultiply(x, y, n):
    # Multiply two n-bit integers x and y
    if n == 1:
        return x * y     # base case: single bit

    m = n // 2
    a = x >> m           # high-order half of x  (a = ⌊x / 2^m⌋)
    b = x & ((1 << m) - 1)  # low-order half of x  (b = x mod 2^m)
    c = y >> m           # high-order half of y
    d = y & ((1 << m) - 1)  # low-order half of y

    # Three recursive multiplications (Karatsuba's trick)
    p1 = KaratsubaMultiply(a, c, m)      # ac
    p2 = KaratsubaMultiply(b, d, m)      # bd
    p3 = KaratsubaMultiply(abs(a-b), abs(c-d), m)  # |a-b||c-d|

    # Fix sign of p3
    sign = 1 if (a >= b) == (c >= d) else -1

    # Combine: xy = 2^(2m)*p1 + 2^m*(p1+p2 - sign*p3) + p2
    return (p1 << (2*m)) + ((p1 + p2 - sign*p3) << m) + p2
SOURCE: 📄 pp. 227–238

Worked Example: Multiply x = 1101 (= 13) and y = 1010 (= 10) using 4-bit Karatsuba 📄 pp. 227–238

  n = 4 bits, m = 2.

  x = 1101₂ = 13:  a = high 2 bits = 11₂ = 3,  b = low 2 bits = 01₂ = 1
  y = 1010₂ = 10:  c = high 2 bits = 10₂ = 2,  d = low 2 bits = 10₂ = 2

  ─── THREE RECURSIVE CALLS ──────────────────────────────────────

  p1 = a × c = 3 × 2 = 6        (2-bit multiply)
  p2 = b × d = 1 × 2 = 2        (2-bit multiply)

     = 16 × 6      +  4 × 8                         +  2
     = 96           +  32                            +  2

  ─── VERIFICATION ───────────────────────────────────────────────
  13 × 10 = 130  ✓

  (a×c, a×d, b×c, b×d). Karatsuba uses only 3. ✓

2d Strassen's Algorithm + A6 Q1 Connection: How 7 Beats 8 Lecture PDF: pp. 239–248 (Lec 24–25) · 📋 A6 Q1

Intuition: Why 7 multiplications instead of 8? 📄 pp. 239–248

Naïve n×n matrix multiplication: C[i][j] = Σ A[i][k]·B[k][j]. For n=2×2 matrices, this requires 8 multiplications and 4 additions. If we split an n×n matrix into four n/2 × n/2 blocks and multiply naïvely, we get 8 recursive calls → T(n) = 8T(n/2) + O(n²). Master Theorem: log₂8 = 3, and n³ beats n² → T(n) = O(n³). No improvement over naïve. 📄 p. 239

Strassen (1969) showed that 2×2 block multiplication only requires 7 multiplications (with more additions). This gives T(n) = 7T(n/2) + O(n²). Master Theorem: log₂7 ≈ 2.807, and n2.807 beats n² → T(n) = O(n2.807). A genuine improvement over O(n³)! 📄 pp. 239–248

Mechanism: Strassen's 7 intermediate matrices 📄 pp. 239–248

For 2×2 block matrices A = [[a₁₁,a₁₂],[a₂₁,a₂₂]] and B = [[b₁₁,b₁₂],[b₂₁,b₂₂]], Strassen computes:

  m₁ = (a₁₁ + a₂₂)(b₁₁ + b₂₂)
  m₂ = (a₂₁ + a₂₂) b₁₁
  m₃ = a₁₁ (b₁₂ − b₂₂)
  m₄ = a₂₂ (b₂₁ − b₁₁)
  m₅ = (a₁₁ + a₁₂) b₂₂
  m₆ = (a₂₁ − a₁₁)(b₁₁ + b₁₂)
  m₇ = (a₁₂ − a₂₂)(b₂₁ + b₂₂)

  Result matrix C:
  c₁₁ = m₁ + m₄ − m₅ + m₇
  c₁₂ = m₃ + m₅
  c₂₁ = m₂ + m₄
  c₂₂ = m₁ − m₂ + m₃ + m₆

  Only 7 multiplications (m₁..m₇)!
  (Original naïve 2×2 needs 8 multiplications.)

Misconception Cards 📄 pp. 79–248

❌ Misconception: "Merge Sort is always the best sorting algorithm because it's always O(n log n)."

✅ Reality: Merge Sort requires O(n) extra space for the merged array and is slower in practice due to memory allocation overhead. Quicksort — despite having O(n²) worst case — is usually faster in practice because it sorts in-place and has better cache behavior. Real-world implementations use hybrid algorithms (like Timsort). 📄 pp. 91, 103

❌ Misconception: "Karatsuba needs 3 multiplications, so it computes bc and ad individually."

✅ Reality: Karatsuba computes ac, bd, and (a−b)(c−d). Neither bc nor ad is computed directly. The key identity bc + ad = ac + bd − (a−b)(c−d) lets us recover the middle term bc+ad from the three products we DID compute. 📄 pp. 227–238

❌ Misconception: "Strassen's algorithm is faster than O(n³), so it's always used in practice."

✅ Reality: Strassen has large constant factors and complex code that makes it impractical for small matrices. The GNU Multiple Precision library uses it only for very large inputs where the asymptotic advantage overcomes the constant-factor penalty. For typical matrix sizes in engineering (n ≤ 1000), naïve multiplication with BLAS optimizations is faster. 📄 p. 248

❌ Misconception: "T(n) = 4T(n/2) + n has the same complexity as T(n) = 4T(n/2) + n²."

✅ Reality: Very different! T(n)=4T(n/2)+n: log₂4=2, f(n)=n — since n = O(n^(2−ε)), Case 1 applies → O(n²). T(n)=4T(n/2)+n²: log₂4=2, f(n)=n² — since n² = Θ(n^2), Case 2 applies → O(n² log n). The combine cost f(n) completely changes the answer. 📄 p. 227

❌ Misconception: "Quicksort worst case O(n²) is only a theoretical concern."

✅ Reality: Quicksort on an already-sorted array with first-element pivot hits the worst case every time — each partition produces one element and n−1 remaining. This is a real concern for nearly-sorted data (common in practice). Solution: randomize the pivot. "For any fixed strategy, can work backwards to construct O(n²) worst case." 📄 p. 100

❌ Misconception: "log₂(7) and log₄(7) are the same thing."

✅ Reality: log₂(7) ≈ 2.807; log₄(7) = log₂(7)/log₂(4) = 2.807/2 ≈ 1.404. They differ by a factor of log₂(4) = 2. In the A6 Q1 problem, the recurrence uses b=4, so the critical exponent is log₄(a), NOT log₂(a). Confusing bases is the most common A6 Q1 error. 📄 pp. 239–248

❌ Misconception: "The Master Theorem applies to ALL recurrences of the form T(n) = aT(n/b) + f(n)."

✅ Reality: The MT requires a polynomial gap between f(n) and n^(log_b a). If f(n) = n² log n with n^(log_b a) = n², neither Case 1 (needs f = O(n^(2−ε))), Case 2 (needs f = Θ(n²)), nor Case 3 (needs f = Ω(n^(2+ε))) applies. The log n factor prevents any case from holding — the MT is not applicable. This is Assignment 6 Q3. 📄 p. 227

🧠 Mnemonic 1: "DCC" — Divide, Conquer, Combine Every D&C algorithm: Divide the input into smaller sub-problems, Conquer them recursively, Combine the results. On any exam question about a D&C algorithm, identifying these three phases earns structure points even if you forget details. 📄 p. 79
🧠 Mnemonic 2: Merge Sort's "Levels × n = n log n" Each level of the Merge Sort recursion tree does exactly n total work (regardless of how many nodes are on that level). There are log₂(n) + 1 levels. Total = n × (log n + 1) = O(n log n). If you forget the derivation, this level-by-level counting always works. 📄 pp. 79–81
🧠 Mnemonic 3: "Karatsuba saves the middle" — bc+ad from 3 products Karatsuba computes corner products (ac and bd) and one diagonal product ((a−b)(c−d)), then recovers the "middle cross-term" bc+ad = ac + bd − (a−b)(c−d). Think of it as computing the four corners of a rectangle, then computing the diagonal to infer the two off-diagonal entries. 📄 pp. 227–238
🧠 Mnemonic 4: Strassen threshold — "7² = 49, so a_max = 48" For any recurrence T(n) = aT(n/4) + Θ(n²) to beat Strassen's Θ(n^log₂7): a < 4^(log₂7) = (2²)^(log₂7) = 2^(2log₂7) = 7² = 49. The answer is always 48 for this specific form. Memorize the chain: "4-log₂7 = square of 7 = 49, max a = 48." 📄 pp. 239–248
🧠 Mnemonic 5: MT Case selector — "LEFT, TIE, RIGHT" For T(n) = aT(n/b) + f(n), compute c = log_b(a):
If f(n) = O(n^(c−ε)): LEFT (a dominates) → T = Θ(n^c).
If f(n) = Θ(n^c): TIE → T = Θ(n^c · log n).
If f(n) = Ω(n^(c+ε)) AND regularity: RIGHT (f dominates) → T = Θ(f(n)).

WARM-UP CONCEPTUAL 📄 pp. 79–90

WC-1. A classmate says: "Merge Sort and Selection Sort both eventually sort the whole array, so they must have the same complexity." What is wrong with this reasoning, and what is the key structural difference that makes one faster?

▶ Show Explanation

What is wrong: "Eventually producing the same output" says nothing about how long the algorithm takes. The argument ignores the number of comparisons performed. 📄 pp. 79–90

The key structural difference: Selection Sort finds the minimum of the unsorted portion by scanning ALL remaining elements → O(n) per pass → T(n) = n + T(n−1) → O(n²). Merge Sort divides the problem in HALF at each level → only log₂(n) levels of work → T(n) = O(n log n). The halvings create a shallow recursion tree; Selection Sort's linear decrease creates a deep one. 📄 pp. 79–80

Concretely: for n = 1,000,000 elements, Selection Sort takes ~1012 steps; Merge Sort takes ~20,000,000. That is a factor of 50,000 difference. 📄 p. 91

WC-2. For Merge Sort on n = 8 elements: (a) How many levels does the recursion tree have? (b) How many nodes are there at level 2? (c) What is the total merge work done at level 2? (d) What is the total work across all levels?

▶ Show Explanation

(a) Levels: log₂(8) = 3, so there are 4 levels (levels 0, 1, 2, 3). 📄 p. 80

Q11. Write pseudocode for the Maximum Product Integer Split problem: given a positive integer n ≥ 2, split it into integers (each ≥ 2) whose product is maximized. No part should equal 1. Provide a step-wise English explanation and complexity analysis.

▶ Show Answer 
function maxProductSplit(n):
    # Returns the maximum product from splitting n into parts ≥ 2

    # Base cases: small n cannot be split further profitably
    if n == 2: return 2            # only split: {2} → product 2
    if n == 3: return 3            # only split: {3} → product 3; {2,1}→2 (worse)
    if n == 4: return 4            # {2,2}→4 = 4; keep as is

    # Greedy recursive step: always carve off a 3
    # (3+3=6 → product 9 > 2+2+2=6 → product 8; so prefer 3s)
    return 3 * maxProductSplit(n - 3)
SOURCE: 📄 A0 p.11 · 📋 A1 Q1

Step-wise English Explanation:

  1. Base cases (n ≤ 4): Return n itself — further splitting doesn't increase the product. Note: {4} = {2,2} both give product 4, so either is valid; just return 4.
  2. Greedy insight (n > 4): We always split off a 3. Why? Compare using 3s vs 2s: three 2s (sum 6) give product 8, but two 3s (sum 6) give product 9. So for large n, favor 3s.
  3. Recursive call: After carving off 3, recursively split n−3 and multiply by 3.
  4. Trace for n = 10: 10 → 3 × maxProductSplit(7) → 3 × 3 × maxProductSplit(4) → 3 × 3 × 4 = 36. Compare to {5,5}: 25 or {2,2,2,2,2}: 32. Answer: 36 ✓

📝 Complexity Analysis:

Time: Each recursive call reduces n by 3. Depth = n/3. Each call does O(1) work. Total: O(n). 📄 A0 p.11

Space: Recursion stack depth = n/3 = O(n). Can be made O(1) with an iterative implementation.

Q12. Write pseudocode for an algorithm that merges two sorted arrays A (of size m) and B (of size n) into a single sorted array C. Also implement the three variations: (a) Union (discard duplicates), (b) Intersection, (c) Difference (elements in A but not in B). Analyze time and space complexity.

▶ Show Answer 
# STANDARD MERGE (preserves duplicates)
function Merge(A, B):
    C = []; i = 0; j = 0
    while i < len(A) and j < len(B):
        if A[i] <= B[j]:  C.append(A[i]); i += 1
        else:              C.append(B[j]); j += 1
    C.extend(A[i:]); C.extend(B[j:])
    return C

# (a) UNION — discard duplicates
function MergeUnion(A, B):
    C = []; i = 0; j = 0
    while i < len(A) and j < len(B):
        if A[i] == B[j]:
            C.append(A[i]); i += 1; j += 1   # add once, skip both
        elif A[i] < B[j]: C.append(A[i]); i += 1
        else:              C.append(B[j]); j += 1
    # append remaining unique elements
    while i < len(A):
        if not C or C[-1] != A[i]: C.append(A[i])
        i += 1
    while j < len(B):
        if not C or C[-1] != B[j]: C.append(B[j])
        j += 1
    return C

# (b) INTERSECTION — only elements in both
function MergeIntersection(A, B):
    C = []; i = 0; j = 0
    while i < len(A) and j < len(B):
        if A[i] == B[j]:   C.append(A[i]); i += 1; j += 1
        elif A[i] < B[j]:  i += 1    # A element not in B; skip
        else:               j += 1    # B element not in A; skip
    return C

# (c) DIFFERENCE — elements in A but not in B
function MergeDifference(A, B):
    C = []; i = 0; j = 0
    while i < len(A) and j < len(B):
        if A[i] == B[j]:   i += 1; j += 1   # A[i] in both; skip
        elif A[i] < B[j]:  C.append(A[i]); i += 1
        else:               j += 1           # B[j] not relevant to A\B
    C.extend(A[i:])     # remaining A elements not matched
    return C
SOURCE: 📄 pp. 91–95

📝 Complexity Analysis (all four variants):

Time: Each pointer (i or j) advances by 1 at every iteration, never retreating. Total iterations ≤ m + n. Each iteration is O(1). Total: O(m + n). 📄 p. 91

Space: Output array C has at most m + n elements for Merge; at most min(m,n) for Intersection. Extra: O(m + n).

Key insight for Union: When A[i] == B[j], copy ONCE and advance BOTH pointers. 📄 p. 91

Q13. (A6 Q5 variant) You have two sorted arrays A and B, each of size n, stored as external databases. You can query each database with a partition index to get the element at that position. Design a D&C algorithm to find the median of the combined 2n elements in O(log n) queries. Provide pseudocode, step-wise English explanation, and complexity analysis. Demonstrate on A = [1,3,5,7], B = [2,4,6,8].

▶ Show Answer 
function findMedian(A, B, n):
    # A, B: sorted arrays of size n each
    # Combined size = 2n → median is average of elements at positions n and n+1

    lo = 0; hi = n
    while lo <= hi:
        partA = (lo + hi) // 2        # partition A: A[0..partA-1] in left half
        partB = n - partA             # remaining in left half from B

        # "Infinity" sentinels for boundary conditions
        maxLeftA  = A[partA - 1] if partA > 0 else -INF
        minRightA = A[partA]     if partA < n else +INF
        maxLeftB  = B[partB - 1] if partB > 0 else -INF
        minRightB = B[partB]     if partB < n else +INF

        if maxLeftA <= minRightB and maxLeftB <= minRightA:
            # Correct partition found
            if (2 * n) % 2 == 0:
                return (max(maxLeftA, maxLeftB) + min(minRightA, minRightB)) / 2
            else:
                return max(maxLeftA, maxLeftB)

        elif maxLeftA > minRightB:
            hi = partA - 1        # too many elements from A in left half
        else:
            lo = partA + 1        # too few elements from A in left half
SOURCE: 📄 pp. 55–62 (binary search concept) · 📋 A6 Q5

Step-wise English Explanation:

  1. D&C setup: Binary search on the partition index of A (how many elements from A go in the "left half" of the merged array). As partA is determined, partB = n − partA automatically.
  2. Invariant: We want: max(left half of A) ≤ min(right half of B) AND max(left half of B) ≤ min(right half of A). When both hold, the left half is exactly the smaller n elements.
  3. Binary search: If maxLeftA > minRightB, partA is too large — move hi left. If maxLeftB > minRightA, partA is too small — move lo right.
  4. Termination: lo > hi → valid partition found → compute median from the boundary elements.

📝 Complexity Analysis:

Time: Binary search on partA from 0 to n → log₂(n) iterations. Each iteration does O(1) comparisons and 4 queries. Total: O(log n) queries and O(log n) time. 📄 pp. 55–62

Space: Only variables partA, partB, lo, hi: O(1) extra space.

THE CANONICAL ALGORITHM (MERGE SORT):

function MergeSort(A, left, right):
    if right - left == 1: return [A[left]]     # base case
    mid = (left + right) // 2
    L = MergeSort(A, left, mid)                # divide + conquer left
    R = MergeSort(A, mid, right)               # divide + conquer right
    return Merge(L, R)                         # combine: O(n)
SOURCE: 📄 pp. 79–90

THE #1 MISTAKE ON EXAM QUESTIONS ABOUT THIS TOPIC:

Applying the Master Theorem without verifying the polynomial-gap condition. Students compute c* = log_b(a) and compare f(n) to n^c*, but forget that f(n) must be polynomially smaller/larger — not just asymptotically different. The classic trap is f(n) = n^c* · log n (Assignment 6 Q3), where "almost Case 2" fails all three cases. Always ask: "is the ratio f(n) / n^c* either bounded by n^(−ε) (Case 1), bounded between constants (Case 2), or bounded below by n^ε (Case 3)?" If the ratio involves log n, the MT is likely not applicable. 📄 p. 227
