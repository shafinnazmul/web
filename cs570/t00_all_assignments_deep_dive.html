Q1.2SET OPERATIONS

Define set intersection S₁ ∩ S₂ in set-builder notation. Also define set minus, complementation, and Cartesian product.

▶ Full Answer & Explanation

Intersection: S₁ ∩ S₂ = {x : x ∈ S₁ AND x ∈ S₂} — elements appearing in BOTH sets. 📄 Lec 1, A0

Set minus: S₁ − S₂ = {x ∈ S₁ : x ∉ S₂} — elements in S₁ but NOT in S₂.

Complementation: S̄₁ = {x ∈ U : x ∉ S₁} — everything in the universe U that is NOT in S₁. Note you need a universal set U.

Cartesian product: S₁ × S₂ = {(x, y) : x ∈ S₁, y ∈ S₂} — all ordered pairs.

Example: If S₁ = {1,2,3} and S₂ = {a,b}, then S₁ × S₂ = {(1,a),(1,b),(2,a),(2,b),(3,a),(3,b)} — 3×2=6 pairs.

Powerset: If S = {1,2,3}, then 2^S = {∅, {1}, {2}, {3}, {1,2}, {1,3}, {2,3}, {1,2,3}}. If |S|=n, then |2^S| = 2^n. 📄 A0 Q1

📌 Key formulaThe powerset of a set with n elements has exactly 2^n elements. Memorize this.
Q1.4RELATIONS

Define reflexive, symmetric, and transitive relations.

▶ Full Answer & Explanation

Reflexive: R on set A is reflexive if every element is related to itself: ∀a ∈ A, (a,a) ∈ R. Example: "≤" on integers is reflexive (x ≤ x for all x). 📄 A0 Q1.4

Symmetric: ∀a,b ∈ A, (a,b) ∈ R ⟹ (b,a) ∈ R. Example: "is a sibling of" is symmetric. "is a parent of" is NOT symmetric.

Transitive: ∀a,b,c ∈ A, (a,b) ∈ R ∧ (b,c) ∈ R ⟹ (a,c) ∈ R. Example: "less than" (<) is transitive. 📄 A0 Q1.4

Q2.1PROOF BY INDUCTIONMIDTERM ⭐

A binary tree of height n has at most 2ⁿ leaves. Prove by induction.

▶ Full Answer with Every Step Explained
What is a binary tree of height n?Height = length of the longest root-to-leaf path. Height 0 = just a single node. Height k = root with subtrees of height ≤ k-1.
PROOF BY MATHEMATICAL INDUCTION
Statement P(n):
A binary tree of height n has at most 2ⁿ leaves. 📄 A0 Q2.1
Base Case: n = 0
A binary tree of height 0 is a single node. That node is itself the only leaf. Number of leaves = 1. 2⁰ = 1. Since 1 ≤ 1, P(0) is TRUE. ✓ 📄 A0 Q2.1
Inductive Hypothesis (IH):
Assume P(k) is true for some integer k ≥ 0. That is: any binary tree of height k has at most 2^k leaves. 📄 A0 Q2.1
Inductive Step: Prove P(k+1)
Consider any binary tree T of height k+1. T's root has a left subtree T_L and a right subtree T_R (either or both may be empty, or they may exist with height ≤ k).
Since T has height k+1, the subtrees T_L and T_R have height at most k.
By the IH: T_L has at most 2^k leaves. T_R has at most 2^k leaves.
Every leaf of T is a leaf of T_L or T_R. So the total number of leaves in T is at most: 2^k + 2^k = 2 · 2^k = 2^(k+1). ✓ 📄 A0 Q2.1
Conclusion:
By the principle of mathematical induction, P(n) holds for all n ≥ 0. ∎
  Concrete verification:
  n=0: 1 leaf  ≤ 2⁰=1  ✓
  n=1: up to 2 leaves ≤ 2¹=2  ✓   (root + 2 children)
  n=2: up to 4 leaves ≤ 2²=4  ✓   (fully filled tree)

       ●          height 2
      / \
     ●   ●        height 1
    / \ / \
   ● ● ● ●        4 leaves = 2² ✓

Q2.2PROOF BY CONSTRUCTIONMIDTERM ⭐

If a and b are consecutive integers, then a + b is odd. Prove by construction.

▶ Full Answer with Every Step Explained
What is proof by construction?We explicitly construct a mathematical object that satisfies the required property, rather than just asserting it exists.
PROOF BY CONSTRUCTION
Setup: Let a and b be consecutive integers. By definition, b comes immediately after a, so b = a + 1 for some integer a. 📄 A0 Q2.2
Construct the sum: a + b = a + (a + 1) = 2a + 1.
Show it's odd: Since a is an integer, 2a is an even integer (divisible by 2). Then 2a + 1 is the next integer after 2a. By definition, an integer is odd iff it equals 2k + 1 for some integer k. Taking k = a: 2a + 1 = 2(a) + 1 = odd. ✓ 📄 A0 Q2.2
Conclusion: For any consecutive integers a and b = a+1, their sum a + b = 2a + 1 is always odd. ∎

Concrete example: a=7, b=8. Sum = 15 = 2(7)+1 = odd ✓. a=100, b=101. Sum=201=2(100)+1=odd ✓.

Q2.3PROOF BY CONTRADICTIONMIDTERM ⭐

If n² is even, then n is even. Prove by contradiction.

▶ Full Answer with Every Step Explained
What is proof by contradiction?Assume the NEGATION of what you want to prove. Show this assumption leads to an impossible situation (contradiction). Therefore the original statement must be true.
PROOF BY CONTRADICTION
Original claim (P → Q): If n² is even, then n is even. 📄 A0 Q2.3
Assume the negation: Assume n² is even AND n is NOT even (i.e., n is odd).
Work with the assumption: Since n is odd, we can write n = 2k + 1 for some integer k.
Compute n²: n² = (2k+1)² = 4k² + 4k + 1 = 2(2k² + 2k) + 1. 📄 A0 Q2.3
Identify the contradiction: Let r = 2k² + 2k (this is an integer). Then n² = 2r + 1, which is ODD by definition.
Contradiction: We assumed n² was even, but derived that n² must be odd. These cannot both be true. ✗
Conclusion: Our assumption (n is odd) must be false. Therefore n is even. ∎ 📄 A0 Q2.3

Concrete verification of the contradiction: Take k=3, so n=7. n²=49. 49 = 2(24)+1 = odd. But we assumed n² is even. Contradiction. ✓

⚠️ MisconceptionProof by contradiction does NOT mean showing the claim is false. You assume the OPPOSITE of what you want to prove, then derive absurdity.

📘 Assignment 1 — Algorithm Design: Max Product Integer Split

A1 Q1 Pseudocode + English Explanation: Maximum Product by Integer Splitting ▶
Q1PSEUDOCODE + EXPLANATION

Given an integer n, break it into the sum of k ≥ 2 positive integers that maximizes the product. Return the maximum product.

Example: n=10 → 10=3+3+4 → product=36.   n=2 → 2=1+1 → product=1.
▶ Full Answer with Deep Explanation

The Core Mathematical Insight

Why do we want pieces of size 3? The submitted solution uses the inequality: (x+a)(x-a) = x² - a² ≤ x². This shows that for a fixed sum 2x, the product is maximized when a=0 (equal halves). So we always want to split as evenly as possible. 📄 A1 Q1

For 2: 2 = 1+1 → product = 1 < 2. So DON'T break 2 (but we must for k≥2).

For 3: 3 = 1+2 → product = 2 < 3. So DON'T break 3.

For 4: 4 = 2+2 → product = 4 = 4. Breaking doesn't help.

For 6: 6 = 3+3 → product = 9 > 6. Breaking DOES help here.

The greedy rule: keep splitting until all pieces are ≤ 3. The halving strategy (split into two ≈ equal halves) achieves this recursively.

Pseudocode (from submitted solution)

FUNCTION MaxProduct(n):
    FUNCTION Split(x, current_split_count):
        # Base cases: x is small enough (≤ 3)
        IF current_split_count < 2 AND x <= 3 THEN
            # Must split at least once — break off 1
            RETURN ( [x-1, 1], current_split_count + 1 )
        ELSE IF x <= 3 THEN
            # Already split enough — keep x as-is
            RETURN ( [x], current_split_count )
        END IF

        # Recursive split into two halves
        left_split  = CEILING(x / 2)      # round up
        right_split = FLOOR(x / 2)         # round down
        current_split_count = current_split_count + 1

        # Recursively split each half if still > 3
        IF left_split > 3 THEN
            (left_splits, current_split_count) = Split(left_split, current_split_count)
        ELSE
            left_splits = [left_split]
        IF right_split > 3 THEN
            (right_splits, current_split_count) = Split(right_split, current_split_count)
        ELSE
            right_splits = [right_split]

        RETURN (left_splits + right_splits, current_split_count)
    END FUNCTION

    (splits, count) = Split(n, 1)
    result = 1
    FOR EACH s IN splits: result = result * s
    RETURN result
END FUNCTION
SOURCE: 📄 A1 Q1 submitted solution

Step-by-step English Explanation

Step 1: Base cases for small numbers
If the number x is ≤ 3 and we haven't yet made 2 splits, we must split once (to satisfy k≥2). We return [x-1, 1]. If x≤3 and we've already split enough, keep x intact.
Step 2: Halving strategy
For x > 3, split into ⌈x/2⌉ and ⌊x/2⌋. This is the "most balanced" possible split, maximizing the product by the (x+a)(x-a) ≤ x² inequality.
Step 3: Recurse on large halves
If either half is still > 3, recurse on it. If ≤ 3, keep it. This continues until all pieces are ≤ 3.
Step 4: Multiply all pieces
Multiply all resulting pieces together for the maximum product.

Worked Example: n = 10

  Split(10, 1):
    left = 5, right = 5
    Split(5): left=3, right=2 → returns [3, 2]
    Split(5): left=3, right=2 → returns [3, 2]
    Final splits: [3, 2, 3, 2] (but assignment says answer is [3,3,4])

  Note: The instructor's canonical answer is 10 = 3+3+4 → 3×3×4 = 36.
  The halving approach yields [3,2,3,2] → 3×2×3×2 = 36. ✓ Same product!

Complexity: Each split divides the problem in half. The recursion depth is O(log n). Total work: O(n) for the multiplication step. Time = O(n). 📄 A1 Q1

A2 Q1 True/False with Formal Proof or Counterexample: Three Asymptotic Claims ⭐ MIDTERM ⭐ ▶
📌 Formal definition reminderf(n) = O(g(n)) means: ∃c > 0, ∃n₀ such that ∀n ≥ n₀: 0 ≤ f(n) ≤ c·g(n). The witnesses c and n₀ must be produced explicitly. 📄 Lec 5–6
Q1.1PROOF⭐

Claim: If f(n) = O(g(n)) and f(n) = Ω(h(n)), then g(n) = Ω(h(n)). TRUE or FALSE?

▶ Full Answer: TRUE — Chain Inequality Proof
DIRECT PROOF (chaining inequalities)
Given 1: f(n) = O(g(n)) → ∃c₁ > 0, n₁: ∀n ≥ n₁, f(n) ≤ c₁·g(n). 📄 Lec 5
Given 2: f(n) = Ω(h(n)) → ∃c₂ > 0, n₂: ∀n ≥ n₂, f(n) ≥ c₂·h(n). 📄 Lec 5
Combine: Let n₀ = max(n₁, n₂). For all n ≥ n₀, both inequalities hold: c₂·h(n) ≤ f(n) ≤ c₁·g(n).
Derive: c₂·h(n) ≤ c₁·g(n). Divide both sides by c₁ > 0: (c₂/c₁)·h(n) ≤ g(n).
Conclusion: Let c = c₂/c₁ > 0. Then g(n) ≥ c·h(n) for all n ≥ n₀. By definition, g(n) = Ω(h(n)). ✓ ∎ 📄 A2 Q1.1

Intuition: f is sandwiched between h (from below) and g (from above). So g must be above h too.

Q1.2PROOF

Claim: If f(n) = O(g(n)), then f(n)² = O(g(n)²). TRUE or FALSE?

▶ Full Answer: TRUE — Square Both Sides
DIRECT PROOF
Given: f(n) = O(g(n)) → ∃c₀ > 0, n₀: ∀n ≥ n₀, 0 ≤ f(n) ≤ c₀·g(n). 📄 A2 Q1.2
Square both sides: f(n)² ≤ (c₀·g(n))² = c₀²·g(n)².
Conclusion: Let c = c₀² > 0. Then f(n)² ≤ c·g(n)² for all n ≥ n₀. Therefore f(n)² = O(g(n)²). ✓ ∎ 📄 A2 Q1.2

Why squaring preserves the inequality: If 0 ≤ a ≤ b, then a² ≤ b² (both sides non-negative, squaring is monotone). The key assumption is f(n) ≥ 0, which holds for algorithmic complexity functions.

Q1.3COUNTEREXAMPLE⭐

Claim: If f(n) = Θ(g(n)), then log f(n) = Θ(log g(n)). TRUE or FALSE?

▶ Full Answer: FALSE — Concrete Counterexample
⚠️ This is the hard one — FALSE!Logarithms can destroy Theta relationships when one function approaches a constant.
COUNTEREXAMPLE
Define: f(n) = 2 (constant), g(n) = 1 + 1/n. 📄 A2 Q1.3
Step 1: Show f(n) = Θ(g(n)). For all n ≥ 1: 1 ≤ g(n) = 1+1/n ≤ 2 = f(n). Choose c₁=1, c₂=2, n₀=1: c₁·g(n) ≤ f(n) ≤ c₂·g(n) ✓. So f = Θ(g).
Step 2: Show log f(n) ≠ Θ(log g(n)). log f(n) = log 2 = constant. log g(n) = log(1 + 1/n) → 0 as n → ∞.
Ratio diverges: lim log f(n)/log g(n) = lim (log 2) / (log(1+1/n)) → ∞ as n→∞. The ratio is not bounded by any constant. So log f(n) ≠ Θ(log g(n)). ✗ ∎ 📄 A2 Q1.3

Intuition: Two functions can be within a constant factor of each other (Theta), but their logarithms can diverge wildly if one approaches a constant while the other approaches 0. Theta is NOT preserved under logarithms in general.

A3 Q1 Max Area of Square-Shaped Hole After Removing Bars ▶
Q1PYTHON CODING

Given n, m, hBars, vBars: find the maximum area of a square hole by removing some removable horizontal and vertical bars.

Input: n=2, m=1, hBars=[2,3], vBars=[2] → Output: 4
▶ Full Answer with Algorithm Explanation

Core Insight

The maximum gap in horizontal bars determines the max height of the hole. The maximum gap in vertical bars determines max width. The square side = min(max_h_gap, max_v_gap). Area = side². 📄 A3 Q1

To maximize consecutive removable bars: sort hBars and find the longest consecutive run (each consecutive pair a, a+1 can be removed together to create a gap of size = run_length + 1).

def maxRectangleArea(n, m, hBars, vBars):
    def maxGap(bars):
        bars.sort()
        max_gap = 1    # minimum gap without removing anything
        streak = 1
        for i in range(1, len(bars)):
            if bars[i] == bars[i-1] + 1:
                streak += 1    # consecutive bars can be removed together
            else:
                streak = 1
            max_gap = max(max_gap, streak + 1)
        return max_gap

    h_gap = maxGap(hBars)
    v_gap = maxGap(vBars)
    side = min(h_gap, v_gap)
    return side * side
SOURCE: 📄 A3 Q1

Complexity: Sorting hBars and vBars takes O(|hBars| log + |vBars| log). Overall: O((h+v) log(h+v)). 📄 A3

A4 Q1 Growth Rate Ordering of 7 Functions [5 pts] ⭐ GRADED ⭐ ▶
Q1ASYMPTOTIC ORDERING5 pts

Arrange in ascending order of growth rate: g₁=2^√(log n), g₂=2^n, g₃=n(log n)³, g₄=n^(4/3), g₆=2^(2^n), g₇=2^(n²)

▶ Full Answer with All 5 Limit Comparisons

Method: Use limit comparison. lim f/g as n→∞: → 0 means f=o(g) (f slower), → c>0 means f=Θ(g), → ∞ means f=ω(g) (f faster). 📄 Lec 5–6, A4 Q1

Comparison 1: g₁ vs g₃ (g₁ slower)
log g₁ = √(log n). For g₃ = n(log n)³, log g₃ = log n + 3 log log n ≈ log n. lim √(log n) / log n = lim 1/√(log n) → 0. So g₁ = o(g₃). 📄 A4 Q1
Comparison 2: g₃ vs g₄ (g₃ slower)
g₄/g₃ = n^(4/3) / (n(log n)³) = n^(1/3) / (log n)³. Let x = log n → ∞. Then n^(1/3) = e^(x/3). The ratio e^(x/3)/x³ → ∞. So g₃ = o(g₄). 📄 A4 Q1
Comparison 3: g₄ vs g₂ (g₄ slower)
Exponential grows faster than any polynomial. lim 2^n / n^(4/3) = ∞. So g₄ = o(g₂). 📄 A4 Q1
Comparison 4: g₂ vs g₇ (g₂ slower)
lim g₇/g₂ = lim 2^(n²) / 2^n = lim 2^(n²-n) = lim 2^(n(n-1)) → ∞. So g₂ = o(g₇). 📄 A4 Q1
Comparison 5: g₇ vs g₆ (g₇ slower)
lim g₆/g₇ = lim 2^(2^n) / 2^(n²) = lim 2^(2^n - n²). For large n, 2^n dominates n², so 2^n - n² → ∞. So g₇ = o(g₆). 📄 A4 Q1
✅ Final Answer — Ascending Order g₁ ≺ g₃ ≺ g₄ ≺ g₂ ≺ g₇ ≺ g₆
i.e., 2^√(log n) ≺ n(log n)³ ≺ n^(4/3) ≺ 2^n ≺ 2^(n²) ≺ 2^(2^n)

A4 Q2 (n+a)^b Asymptotic Relationship [3 pts] ⭐ GRADED ⭐ ▶
Q2LIMIT COMPARISON3 pts

For real constants a and b with b > 0, what is the correct relationship? (A) Θ(n^b) (B) O(n^(b+1)) but not Θ(n^b) (C) Θ(n^(b+1))

▶ Full Answer: (A) (n+a)^b = Θ(n^b)
PROOF via Limit Comparison
(n+a)^b = n^b · (1 + a/n)^b. As n → ∞, a/n → 0, so (1 + a/n)^b → 1^b = 1. 📄 A4 Q2
Therefore: lim (n+a)^b / n^b = 1. Since the limit is a positive finite constant (1), (n+a)^b = Θ(n^b).
Answer: (A) — The additive constant a in the base is asymptotically irrelevant. ∎ 📄 A4 Q2

Formal witnesses: Choose c₁ = (1-ε)^b and c₂ = (1+ε)^b for any ε ∈ (0,1). For large enough n, (1 + a/n)^b is within these bounds. More directly: for n ≥ 2|a|, we have n/2 ≤ n+a ≤ 2n, so (n/2)^b ≤ (n+a)^b ≤ (2n)^b, giving (1/2^b)n^b ≤ (n+a)^b ≤ 2^b·n^b.

📌 Key takeawayAdditive constants inside a polynomial base DO NOT change the asymptotic class. Only the exponent b matters: (n + anything)^b = Θ(n^b).

▶ Full Answer with Trace & Complexity

Core Idea: Sliding Window

Instead of recomputing the sum for each window from scratch (O(nk)), we maintain a running sum and slide it: add the new right element, subtract the dropped left element. O(1) per step. 📄 A4 Q7

def findMaxAverage(nums, k):
    # Step 1: compute initial window sum (first k elements)
    window_sum = sum(nums[0:k])   # O(k)
    max_sum = window_sum

    # Step 2: slide window from position k to end
    for i in range(k, len(nums)):
        window_sum = window_sum - nums[i - k] + nums[i]  # slide: remove left, add right
        max_sum = max(max_sum, window_sum)

    return max_sum / k   # return average

def findMinDifference(timePoints):
    # Step 1: Convert all times to total minutes from 00:00
    minutes = []
    for time in timePoints:
        h = int(time[0:2])    # hours part
        m = int(time[3:5])    # minutes part
        minutes.append(h * 60 + m)

    # Step 2: Sort to bring close times adjacent
    minutes.sort()    # O(m log m)

    # Step 3: Find min consecutive difference
    min_diff = 1440   # max possible is one full day
    for i in range(1, len(minutes)):
        diff = minutes[i] - minutes[i-1]
        min_diff = min(min_diff, diff)

    # Step 4: Check circular wrap (last time to first time through midnight)
    circular = minutes[0] + 1440 - minutes[-1]
    min_diff = min(min_diff, circular)

    return min_diff

A4 Q9 Total Hamming Distance Sum in O(n) [20 pts] ⭐ GRADED ⭐ ▶
Q9PSEUDOCODE20 pts

Given array nums, return sum of Hamming distances between all pairs. Hamming distance = number of bit positions where two numbers differ. Example: [4,14,2] → 6

▶ Full Answer with Bit-Position Trick

The Naive Approach (Too Slow)

Compute Hamming distance for all O(n²) pairs → O(n²) time. For n=10000, this is 10^8 operations — too slow. 📄 A4 Q9

The O(n) Bit-Position Trick

For each bit position i: count how many numbers have bit i set to 1 (call it p) and how many have bit i set to 0 (call it q = n-p). Every pair where one has 1 and the other has 0 contributes 1 to the total Hamming distance at this bit. Number of such pairs = p × q. Sum over all 32 bit positions.

def totalHammingDistance(nums):
    total = 0
    n = len(nums)

    for bit in range(32):       # for each bit position 0..31
        p = 0                   # count of 1s at this position
        for num in nums:
            if (num >> bit) & 1:    # check if bit 'bit' is set
                p += 1
        q = n - p               # count of 0s = total - count of 1s
        total += p * q          # pairs with different bits

    return total
SOURCE: 📄 A4 Q9

Trace: nums = [4, 14, 2]

  Binary: 4=0100, 14=1110, 2=0010  (showing bits 3..0)

  Bit 0: 4→0, 14→0, 2→0. p=0, q=3. p×q=0.
  Bit 1: 4→0, 14→1, 2→1. p=2, q=1. p×q=2.
  Bit 2: 4→1, 14→1, 2→0. p=2, q=1. p×q=2.
  Bit 3: 4→0, 14→1, 2→0. p=1, q=2. p×q=2.
  Bits 4-31: all zeros. p×q=0.

  Total = 0+2+2+2+0+...+0 = 6 ✓ (matches expected output)

Complexity Analysis

  • Time: Outer loop 32 times × inner loop n times × O(1) = 32n = O(n)
  • Space: Constant variables only → O(1)
📌 Why this worksThe total Hamming distance across all pairs at bit position i equals (# of 1s) × (# of 0s) because every 1-0 pair contributes exactly 1. We've decomposed the problem per bit, avoiding O(n²) pairs entirely.

The median of 2n values is the n-th smallest overall. We binary search on how many values come from database A in the "lower half." If A contributes i values and B contributes n-i, we check if this is the correct partition. 📄 A6 Q5

def findMedian(A, B, n):
    lo, hi = 0, n
    while lo <= hi:
        i = (lo + hi) // 2      # A contributes i elements to lower half
        j = n - i               # B contributes j = n-i elements

        A_i  = query(A, i)   if i > 0 else -∞   # i-th smallest in A
        A_i1 = query(A, i+1) if i < n else +∞
        B_j  = query(B, j)   if j > 0 else -∞
        B_j1 = query(B, j+1) if j < n else +∞

        if A_i <= B_j1 and B_j <= A_i1:
            return max(A_i, B_j)   # correct partition! median = larger of two max elements

        elif A_i > B_j1:
            hi = i - 1   # A has too many large values — go left
        else:
            lo = i + 1   # B has too many large values — go right

Why O(log n): Binary search on i ∈ [0,n] does O(log n) iterations. Each iteration makes ≤ 4 queries (A_i, A_{i+1}, B_j, B_{j+1}). Total = 4 log n = O(log n) queries. 📄 A6 Q5

▶ Full Answer: (A) Must be a tree — Proof by Contradiction
PROOF BY CONTRADICTION 📄 A6 Q6
Setup: Let H be a connected spanning subgraph of G with minimum total weight. All edge weights are positive.
Assume H has a cycle: Let C be a cycle in H. Pick any edge e=(u,v) in C.
Remove e: Form H' = H − {e}. H' is still connected! Why? Any path that used e can now route through the rest of the cycle C (going the long way around). The alternative path uses all other edges of C. 📄 A6 Q6
Weight decreases: weight(H') = weight(H) − weight(e) < weight(H), since weight(e) > 0.
Contradiction: H' is a connected spanning subgraph with strictly less total weight than H. But H was assumed minimum. ✗
Conclusion: H cannot contain a cycle. A connected acyclic graph with n vertices has exactly n−1 edges — by definition, it's a tree. ∎ 📄 A6 Q6
⚠️ Why (C) is falseWith zero-weight edges, removing the cycle edge might not decrease total weight (zero weight = free). With negative weights, removing an edge might INCREASE total weight. The argument breaks. Positive weights are essential.
✅ Answer: (A) Such a subset must be a tree
A6 Q7 Star Graph Center — O(1) Algorithm [20 pts] ⭐ GRADED ⭐ ▶
Q7PSEUDOCODE20 pts

Given a star graph's edge list, find the center node. Example: edges=[[1,2],[2,3],[4,2]] → center=2

▶ Full Answer with O(1) Trick

Key Observation

In a star graph, the center appears in EVERY edge. Each leaf appears in exactly ONE edge. Therefore the center must appear in BOTH the first and second edge. Just check which node from edges[0] also appears in edges[1]. 📄 A6 Q7

def findCenter(edges):
    # The center appears in both edges[0] and edges[1]
    u0, v0 = edges[0][0], edges[0][1]
    u1, v1 = edges[1][0], edges[1][1]

    # Whichever endpoint of edges[0] also appears in edges[1] is the center
    if u0 == u1 or u0 == v1:
        return u0
    else:
        return v0

Trace: edges=[[1,2],[2,3],[4,2]]: edges[0]=[1,2], edges[1]=[2,3]. u0=1, v0=2, u1=2, v1=3. u0==u1? 1==2? No. u0==v1? 1==3? No. Return v0=2. ✓

  • Time: O(1) — exactly 4 comparisons regardless of graph size
  • Space: O(1) — just 4 variables
A6 Q8 Arithmetic Progression Check via Sort [20 pts] ⭐ GRADED ⭐ ▶
Q8PSEUDOCODE20 pts

Given array arr, return true if it can be rearranged into an arithmetic progression (constant consecutive difference). Example: [3,5,1] → true (rearranges to [1,3,5]).

▶ Full Answer with Trace & Complexity

Core Idea: Sort then Verify Constant Difference

An arithmetic progression has a constant difference between consecutive terms. After sorting, the differences between adjacent elements must all be equal. 📄 A6 Q8

def canMakeArithmeticProgression(arr):
    n = len(arr)
    if n < 2:
        return True      # trivially AP

    arr.sort()           # O(n log n)

    d = arr[1] - arr[0]  # expected common difference

    for i in range(2, n):
        if arr[i] - arr[i-1] != d:   # check every consecutive pair
            return False

    return True

Trace: arr=[3,5,1]: Sort → [1,3,5]. d=3-1=2. Check: 5-3=2==d ✓. Return True.

Trace: arr=[1,2,4]: Sort → [1,2,4]. d=2-1=1. Check: 4-2=2 ≠ 1. Return False.

  • Time: Sort O(n log n) + verification loop O(n) → O(n log n)
  • Space: In-place sort → O(1) auxiliary (O(n) if using stable sort)
A6 Q9 Maximum Probability Path: Modified Dijkstra [20 pts] ⭐⭐ HARDEST GRADED ⭐ ▶
Q9PSEUDOCODE + ALGORITHM DESIGN20 pts

Undirected graph where each edge has a success probability. Find max-probability path from start to end. Example: n=3, edges=[[0,1],[1,2],[0,2]], probs=[0.5,0.5,0.2], start=0, end=2 → 0.25

▶ Full Answer: Modified Dijkstra with Max-Heap

The Key Insight: 6 Changes from Standard Dijkstra

AspectStandard Dijkstra (min dist)Modified (max prob)
GoalMinimize distanceMaximize probability
Init sourceDistance[s]=0Prob[s]=1.0
Init othersDistance[v]=∞Prob[v]=0.0
Heap typeMin-heap (smallest dist)Max-heap (negate: −prob)
Relaxationdist[u]+w(u,v)Prob[u]×p(u,v)
Update whennew < Dist[v]new > Prob[v]

Why multiplication works: Probabilities of independent events multiply. P(path A→B→C) = P(A→B) × P(B→C). We maximize this product. 📄 A6 Q9

def maxProbability(n, edges, succProb, start, end):
    # Build adjacency list
    graph = {i: [] for i in range(n)}
    for i, (u, v) in enumerate(edges):
        p = succProb[i]
        graph[u].append((v, p))
        graph[v].append((u, p))   # undirected

    # Initialize
    Prob = [0.0] * n
    Prob[start] = 1.0

    # Max-heap: store (-prob, vertex) to use Python's min-heap
    max_heap = [(-1.0, start)]

    while max_heap is not empty:
        neg_p, u = heappop(max_heap)     # extract max-prob vertex
        curr_prob = -neg_p               # actual probability

        if u == end:
            return curr_prob             # found destination!

        if curr_prob < Prob[u]:          # stale entry — skip
            continue

        for (v, edge_prob) in graph[u]:
            new_prob = curr_prob * edge_prob   # multiply probabilities
            if new_prob > Prob[v]:             # found better path?
                Prob[v] = new_prob
                heappush(max_heap, (-new_prob, v))

    return 0.0   # no path found
SOURCE: 📄 A6 Q9

Trace on Example

  n=3, edges=[[0,1],[1,2],[0,2]], probs=[0.5,0.5,0.2], start=0, end=2.
  Paths: 0→1→2 = 0.5×0.5 = 0.25; 0→2 = 0.2.
  Max = 0.25 ✓

Complexity

  • Time: Build adjacency list O(E). Each edge can trigger one heap operation O(log E). Total: O((V+E) log E)
  • Space: Prob array O(V) + adjacency list O(V+E) + heap O(E) → O(V+E)
📌 Why greedy works hereProbabilities p ∈ [0,1]. Multiplying by p ≤ 1 can only DECREASE the running probability. So once we extract the vertex with maximum Prob (greedily settling it), no future path through a lower-prob vertex can improve it. This mirrors Dijkstra's non-negative weights argument.
