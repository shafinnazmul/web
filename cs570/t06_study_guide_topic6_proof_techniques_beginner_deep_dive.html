🧗 Warm-up Glossary — Vocabulary You Need Before §1

Mathematical proofs use a small set of precisely-defined words. When a proof says "for all," "assume," "integer," or "height of a tree," it means something exact. This glossary pins down those meanings before we need them. If you already know all of these, skip to §1.

What is a "mathematical proof"? 📄 p. 1

A mathematical proof is a finite sequence of logical steps that starts from known facts (axioms, definitions, or previously proved results) and arrives at the claim you want to establish — with no gaps and no appeals to intuition. Every step must follow from previous steps by a valid rule of logic.

Why is this stricter than everyday argument? Because in mathematics, "it looks true for all the examples I tried" is NOT a proof. The claim might fail for some huge n you never checked. A proof covers ALL cases, including ones you can't directly compute. 📄 p. 1

What is mathematical induction? 📄 p. 1

Mathematical induction is a proof technique for statements of the form "P(n) is true for every natural number n ≥ n₀." The idea: imagine an infinite row of dominoes. If you can knock over the first one, AND you can prove that knocking over any domino k automatically knocks over domino k+1, then ALL dominoes fall — even the ones a billion positions away.

Formally, you do two things: (1) Base case — verify P(n₀) directly. (2) Inductive step — assume P(k) is true (the "inductive hypothesis") and prove P(k+1) follows from it. Once both are done, P(n) is proved for all n ≥ n₀. 📄 p. 1 

  Domino analogy:

    [Dom 1] → [Dom 2] → [Dom 3] → [Dom 4] → ... → [Dom n] → ...
       ↑
    Base case: knock the first one over.
    Inductive step: if domino k falls, it knocks over k+1.
    Conclusion: ALL dominoes fall.

What does "for all n ≥ 0" mean? 📄 p. 1

The phrase "for all n ≥ 0" (also written ∀n ≥ 0, using the upside-down "A" symbol ∀ which stands for "for all") means: the statement is claimed to be true for every single non-negative integer n — including n = 0, n = 1, n = 2, n = 1000000, n = 10^100, every one without exception.

This is the hardest kind of claim to prove by just testing examples, because you'd need to test infinitely many values. That's exactly why induction exists — it proves all cases at once.

What is the "inductive hypothesis"? 📄 p. 1

The inductive hypothesis (IH) is the temporary assumption you make in the inductive step: "Assume P(k) is true." You are NOT proving P(k) here — you are borrowing it as a given so that you can derive P(k+1).

This feels circular to many beginners: "Am I not assuming what I'm trying to prove?" The answer is no — you're proving P(k+1) given P(k), which is a separate statement. The base case handles P(n₀), and the inductive step handles the "if P(k) then P(k+1)" chain. Together they cover every n.

What is a "contradiction"? 📄 p. 1

A contradiction is a situation where you have derived both "X is true" and "X is false" simultaneously — which is logically impossible. In mathematics, a contradiction signals that one of your assumptions was wrong.

Example: Suppose you assume "n is odd" and later derive "n is even." That's a contradiction — a number cannot be both odd and even. So the assumption "n is odd" must have been false.

What is "proof by contradiction"? 📄 p. 1

Proof by contradiction works as follows: you want to prove claim P. You temporarily assume P is false (i.e., assume ¬P, the negation of P), and then you derive a logical impossibility (a contradiction). Since the assumption ¬P led to an impossible situation, ¬P must be false — which means P must be true.

Think of it as a courtroom argument: "Suppose the defendant was NOT at the scene. But then the security footage showing them at the scene must be fake. But the footage is verified genuine. Contradiction — the defendant WAS at the scene."

⚠️ Classic Misconception "Proof by contradiction disproves the claim." WRONG. Proof by contradiction PROVES the claim by assuming the opposite and deriving an absurdity. At the end, the original claim is proved TRUE — not disproved. 📄 p. 1

What is a "counterexample"? 📄 pp. 43–54

A counterexample is a single specific example that disproves a universal claim. If someone says "all swans are white," one black swan is a counterexample — it proves the claim false.

In CS 570, counterexamples appear when a claim like "f(n) = O(g(n)) always implies log f(n) = O(log g(n))" turns out to be false. You don't need to show it fails for all functions — just find two specific functions where f = O(g) but log f is NOT O(log g). That one example suffices to disprove the claim. 📄 pp. 43–54

What is a binary tree? 📄 p. 1

A binary tree is a hierarchical data structure where each node has at most two children: a left child and a right child. The root is the top node (no parent). A leaf is a node with no children. The height of a tree is the number of edges on the longest path from root to any leaf.

        /       \

  Key fact: a binary tree of height n has AT MOST 2^n leaves.

What does "parity" mean? 📄 p. 1

The parity of an integer is whether it is odd or even. Formally: an integer n is even if there exists an integer k such that n = 2k (divisible by 2 with no remainder). An integer n is odd if there exists an integer k such that n = 2k + 1.

Examples: 8 = 2×4 (even, k=4). 13 = 2×6+1 (odd, k=6). 0 = 2×0 (even, k=0). −7 = 2×(−4)+1 (odd, k=−4). Key property: Every integer is either even OR odd, never both. This is what makes parity arguments work in contradiction proofs.

What is "proof by construction"? 📄 p. 1

A proof by construction (also called a direct proof by example) proves an existence claim by actually building the object. If someone asks "prove there exists a positive even number greater than 100," you construct one: 102 = 2 × 51.

In CS 570, construction proofs often prove claims of the form "for any two consecutive integers a and b, a + b is odd." You prove this by constructing the explicit relationship: b = a + 1, so a + b = 2a + 1, which has the form 2k + 1 (odd). You don't guess — you build it. 📄 p. 1

What does it mean for a claim to be TRUE vs. FALSE in asymptotic proofs? 📄 pp. 39–54

When an assignment or exam asks you to determine if an asymptotic claim is TRUE or FALSE, the tasks are different:

  • TRUE: You must prove it — typically by finding the constants c, n₀ or by using the limit comparison method.
  • FALSE: You must produce a counterexample — two specific functions f and g that violate the claim.

Saying "it seems false" or "I can't find a proof" is NOT sufficient for showing falsity. You must exhibit the explicit counterexample. 📄 pp. 43–54

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 1–12 (Lecture 1) · A0 pp. 1–11 · pp. 39–54 (Lectures 5–6)

1.1 The fundamental problem — why do we need formal proofs at all? 📄 p. 1

Imagine you are designing a sorting algorithm for a hospital's patient-priority queue. You test it on 10,000 patients and it works perfectly every time. Does that mean it's correct? Not in mathematics. There might be a specific combination of inputs — perhaps the 10,001st case — that causes it to fail, and patients could be harmed. Testing confirms behavior on the cases you tried. Proof confirms behavior on all cases, including ones you will never see. 📄 p. 1

In CS 570, this is not just philosophy. Dr. Ghosh directly asks: "Have you formally proved correctness of algorithms?" and "Are you familiar with proofs using induction?" 📄 p. 1 The course is built on the premise that an algorithm you cannot prove correct is an algorithm you cannot trust — regardless of how many test cases pass.

The good news: formal proofs are not magic. They use a small toolkit of techniques. CS 570 focuses on exactly three, which appeared on the midterm and on every assignment: induction, construction, and contradiction. Master these three and you can handle any proof question the exam throws at you.

1.2 The three proof techniques — when to use each one 📄 p. 1

1.3 The staircase analogy — a second framing for induction 📄 p. 1

Imagine you are at the bottom of an infinite staircase. Someone claims you can reach every step. How would you prove this?

Step 1 (Base case): You prove you can climb from the ground to Step 1. This is possible — you just do it, directly and concretely.

Step 2 (Inductive step): You prove a general rule: IF you are on step k, THEN you can always reach step k+1. You don't need to physically climb all the stairs — you just need a universal rule that works from any step.

Conclusion: Once both are established, every step is reachable. Step 1 (base case). Step 2 follows from the rule applied to step 1. Step 3 follows from the rule applied to step 2. And so on, for every n. 📄 p. 1 

  The staircase — induction visualized:

  Step n    ────────────────────────── ← Want to reach here? ...
    ↑  (inductive step rule works here too!)
  Step k+1  ────────────────────────── ← Inductive step: if on k, reach k+1
    ↑  (inductive step)
  Step k    ────────────────────────── ← Inductive hypothesis: ASSUME we're here
    ↑
  Step 2    ────────────────────────── ← Rule works here
    ↑
  Step 1    ────────────────────────── ← Base case: proved directly
    ↑
  [Ground]                               ← Starting point
SOURCE: conceptual, 📄 p. 1

The power of induction is that the inductive step rule, once proved, works at every step — so combining base case + inductive step gives you all of them for free. 📄 p. 1

1.4 Technical definitions (lecture notation) 📄 pp. 1–12

Proof by Induction. To prove P(n) for all integers n ≥ n₀: 📄 p. 1

  • Base Case: Prove P(n₀) directly by computation or definition.
  • Inductive Hypothesis (IH): Assume P(k) is true for some arbitrary integer k ≥ n₀.
  • Inductive Step: Using the IH, prove P(k+1) is true. The proof of P(k+1) MUST explicitly use P(k).
  • Conclusion: By induction, P(n) holds for all n ≥ n₀.

Proof by Construction. To prove "For all integers a and b with b = a+1, a + b is odd": 📄 p. 1 Construct the explicit expression for b (here, b = a+1), then compute a + b = a + (a+1) = 2a+1. Since 2a+1 = 2(a) + 1, it has the form 2k+1 with k = a — which is the definition of an odd integer. No "for all k" quantifier needed; the construction works for any a directly.

Proof by Contradiction. To prove claim P: 📄 p. 1

  • Assume ¬P (the negation of P) is true.
  • Derive a statement that contradicts a known fact, a definition, or an earlier step in the proof.
  • Conclude ¬P must be false, so P must be true. ∎

Counterexample. To disprove a universal claim "For all f, g: C(f,g)": 📄 pp. 43–54 Find specific functions f₀ and g₀ such that C(f₀, g₀) is false. One such pair suffices to disprove the claim.

2a Proof by Induction — "Binary tree of height n has at most 2ⁿ leaves" Lecture PDF: p. 1 (Lec 1) · Assignment 0 Q2.1 📋 A0 Q2.1 · Midterm [a] ⭐

Intuition: Why is induction the right tool here? 📄 p. 1

We want to prove something about ALL binary trees, regardless of height — height 0, height 1, height 50, height 10^6. We can't draw all of them. But we notice that a tree of height k+1 is made of a root plus two subtrees, each of height at most k. If we already know the bound holds for height-k trees (that's the IH), we can compute the bound for height-(k+1) trees directly. Induction lets us leverage the smaller case to prove the larger one.

Mechanism: The 5-step induction template 📄 p. 1

INDUCTION TEMPLATE (fill in the blanks for any induction proof)
1. State P(n): "For all integers n ≥ [n₀], [claim]."
2. Base Case: "When n = [n₀], [verify the claim holds by direct calculation]."
3. Inductive Hypothesis (IH): "Assume P(k) is true for some arbitrary k ≥ [n₀]. That is, assume [restate claim with n=k]."
4. Inductive Step: "We must show P(k+1). That is, we must show [restate claim with n=k+1]. [Use the IH explicitly here.]"
5. Conclusion: "By the principle of mathematical induction, P(n) holds for all integers n ≥ [n₀]. ∎"

Worked Example — traced step by step 📄 p. 1 📋 A0 Q2.1

Claim: A binary tree of height n has at most 2ⁿ leaves.

Step 1 — State P(n) precisely

P(n): "Every binary tree of height n has at most 2ⁿ leaf nodes."

We will prove this for all integers n ≥ 0. 📄 p. 1

Step 2 — Base Case (n = 0)

When n = 0, the tree has only one node (the root), and since it has no children, the root is itself a leaf. So the number of leaves is 1.

Upper bound: 2⁰ = 1. Since 1 ≤ 1, the base case holds. ✓ 📄 p. 1

  Height-0 tree:

       [Root / Leaf]

  Number of leaves = 1 = 2⁰ ✓
SOURCE: conceptual, 📄 p. 1

Step 3 — Inductive Hypothesis

"Assume P(k) is true for some arbitrary integer k ≥ 0. That is, assume that every binary tree of height k has at most 2^k leaves." 📄 p. 1

Notice: we are NOT saying "fix one specific tree." We are assuming the claim holds for every tree of height k — this generality is what makes the inductive step work.

Step 4 — Inductive Step (prove P(k+1) using IH)

Goal: Show that any binary tree T of height k+1 has at most 2^(k+1) leaves. 📄 p. 1

Sub-step 4a: Decompose T into its parts.
Any binary tree T of height k+1 has a root with at most two subtrees: left subtree T_L and right subtree T_R. Since T has height k+1, each subtree has height at most k. 📄 p. 1
Sub-step 4b: Apply the IH to each subtree.
T_L has height ≤ k, so by the IH: T_L has at most 2^k leaves.
T_R has height ≤ k, so by the IH: T_R has at most 2^k leaves. 📄 p. 1
Sub-step 4c: Count total leaves in T.
The leaves of T are exactly the leaves of T_L plus the leaves of T_R (the root is NOT a leaf because it has children). Total leaves ≤ 2^k + 2^k = 2 × 2^k = 2^(k+1). 📄 p. 1
Sub-step 4d: Conclude P(k+1).
We have shown: any binary tree of height k+1 has at most 2^(k+1) leaves. This is exactly P(k+1). ✓ 📄 p. 1

Step 5 — Conclusion

By the principle of mathematical induction, P(n) holds for all integers n ≥ 0: every binary tree of height n has at most 2ⁿ leaves. ∎ 📄 p. 1

  Height 0: 1 node, 1 leaf.   2⁰ = 1.  ✓  (leaf count ≤ 1)
  Height 1: root + ≤ 2 leaves.          2¹ = 2.  ✓  (leaf count ≤ 2)
  Height 2: at most 4 leaves.            2² = 4.  ✓  (leaf count ≤ 4)

         /   \
       [A]   [B]              Height 1 (internal)

  One-sided tree of height 2:

          [R]
         /

2b Proof by Construction — "Sum of two consecutive integers is always odd" Lecture PDF: p. 1 (Lec 1) · Assignment 0 Q2.2 📋 A0 Q2.2 · Midterm [b] ⭐

Intuition: Building the answer directly 📄 p. 1

Some claims can be proved by simply constructing an explicit algebraic expression. If the claim says "for any two consecutive integers a and b, their sum a + b is odd," we don't need induction or contradiction — we just express b in terms of a and compute a + b, then show the result is odd.

Construction proofs are often the shortest and most elegant. The key is to start from the DEFINITION of the objects involved (what does "consecutive" mean? what does "odd" mean?) and compute directly. 📄 p. 1

Mechanism: The construction template 📄 p. 1

CONSTRUCTION TEMPLATE
1. State the claim: "For all [objects with property X], [conclusion C] holds."
2. Express the objects explicitly: "Let [objects]. By definition of X, we can write [explicit formula for the objects]."
3. Compute directly: "Calculate [the thing in C] using the explicit formula. [Show the algebraic steps.]"
4. Verify C: "The result has the form [required form], which by definition means [C]. ✓"
5. Conclude: "Since a and b were arbitrary [objects with X], C holds for all such pairs. ∎"

Worked Example — traced step by step 📄 p. 1 📋 A0 Q2.2

Claim: For any two consecutive integers a and b, the sum a + b is an odd integer.

Step 1: What does "consecutive integers" mean?
Two integers are consecutive if they differ by exactly 1. So if a is the smaller one, then b = a + 1. We may write b in terms of a: b = a + 1. 📄 p. 1
Step 2: Compute the sum a + b.
Substitute b = a + 1:
a + b = a + (a + 1) = 2a + 1.
The sum is 2a + 1, where a is any integer. 📄 p. 1
Step 3: Is 2a + 1 odd?
By definition, an integer n is odd if n = 2k + 1 for some integer k. Here, 2a + 1 = 2k + 1 with k = a. Since a is an integer, k = a is also an integer. So yes, 2a + 1 is odd. ✓ 📄 p. 1
Step 4: Conclusion.
For any two consecutive integers a and b (with b = a + 1), we have a + b = 2a + 1, which is an odd integer by definition. Since a was an arbitrary integer, this holds for ALL pairs of consecutive integers. ∎ 📄 p. 1

Verify on concrete values 📄 p. 1

  Let's check a few specific cases:

  a = 3,  b = 4:   3 + 4 = 7 = 2(3) + 1  ✓  (odd)
  a = 7,  b = 8:   7 + 8 = 15 = 2(7) + 1  ✓  (odd)
  a = 0,  b = 1:   0 + 1 = 1 = 2(0) + 1  ✓  (odd)
  a = -5, b = -4: -5 + (-4) = -9 = 2(-5) + 1  ✓  (odd)
  a = 100, b=101:  100 + 101 = 201 = 2(100)+1  ✓  (odd)

  All cases satisfy 2a + 1 — always odd! ✓

📌 Why Construction Is Often Shorter Than Induction Notice there was no base case, no inductive hypothesis, and no inductive step — just direct algebra. When the claim is about "for all a" but involves only arithmetic on a (not recursive structure), construction is the cleaner tool. Induction is needed when the structure is recursive (like trees or sequences defined in terms of their smaller instances). 📄 p. 1
2c Proof by Contradiction — "If n² is even, then n is even" Lecture PDF: p. 1 (Lec 1) · Assignment 0 Q2.3 📋 A0 Q2.3 · Midterm [c] ⭐

Intuition: Sometimes the back door is easier 📄 p. 1

Sometimes, proving a claim directly is hard because you don't have much to work with. But the NEGATION of the claim gives you an extra assumption — and that extra assumption, combined with what you already know, quickly creates a contradiction. In those situations, proof by contradiction is the smart choice.

For the claim "n² even → n even," the direct approach says: "Given n² is even, I must show n is even." But we don't know how to work backwards from n² to n easily. The contradiction approach says: "Assume n is ODD (the opposite of what I want to prove). Now I have both 'n is odd' AND 'n² is even' to work with. Let me compute n² using the odd formula and see if a contradiction appears." 📄 p. 1

Mechanism: The contradiction template 📄 p. 1

CONTRADICTION TEMPLATE
1. State the claim P.
2. Negate: "Assume for contradiction that ¬P is true." Write out what ¬P means concretely.
3. Derive: Using the assumption ¬P (and other known facts), compute or reason until you reach an impossible statement.
4. Name the contradiction: "This contradicts [specific fact or assumption]."
5. Conclude: "Therefore our assumption that ¬P was wrong. Hence P must be true. ∎"

Worked Example — traced step by step 📄 p. 1 📋 A0 Q2.3

Claim (P): If n² is even, then n is even. [Equivalently: n² is even implies n is even.]

Step 1: State the claim clearly.
P: "For any integer n, if n² is even, then n is even." We want to prove this. 📄 p. 1
Step 2: Negate P.
¬P: "There exists an integer n such that n² is even BUT n is ODD."
We assume ¬P and try to reach a contradiction. So: assume n is odd. (We keep the given: n² is even.) 📄 p. 1
Step 3: Use the definition of "odd."
Since n is odd, by definition there exists an integer k such that n = 2k + 1. 📄 p. 1
Step 4: Compute n².
n² = (2k + 1)² = 4k² + 4k + 1 = 2(2k² + 2k) + 1.
Let m = 2k² + 2k (which is an integer). Then n² = 2m + 1. 📄 p. 1
Step 5: Identify the contradiction.
n² = 2m + 1 has the form 2(integer) + 1, which by definition means n² is ODD.
But we assumed n² is EVEN (given in the claim's hypothesis).
An integer cannot be both odd and even. This is a contradiction. ✗ 📄 p. 1
Step 6: Conclusion.
Our assumption that n is odd led to the contradiction "n² is both even and odd." Therefore n cannot be odd — n must be even. ∎ 📄 p. 1

Verify on concrete values with k = 3 📄 p. 1

  Let k = 3 in the assumption: n = 2(3)+1 = 7.

  If n is odd (n = 7), compute n²:
    n² = 7² = 49

  Check parity of 49: 49 = 2(24) + 1 → ODD.

  But the claim's hypothesis says n² is EVEN.
  49 is ODD, not even. → CONTRADICTION ✗

  This specific case confirms that an odd n gives an odd n²,
  contradicting the assumption that n² is even.
  Therefore n cannot be odd — n must be even. ✓
SOURCE: conceptual, 📄 p. 1
⚠️ The Critical Algebra Step The key move is: (2k+1)² = 4k² + 4k + 1. Many students expand this incorrectly as 4k² + 1 (forgetting the cross-term 2·2k·1 = 4k). Always FOIL carefully: (2k+1)² = (2k)(2k) + 2(2k)(1) + (1)(1) = 4k² + 4k + 1. 📄 p. 1
📌 Contrapositive Connection This proof is equivalent to the contrapositive argument: "if n is odd, then n² is odd." The contrapositive of "P → Q" is "¬Q → ¬P," and both are logically equivalent. In fact, the derivation we did in steps 3–5 IS a direct proof of the contrapositive. The contradiction framing is just one way to present it — the substance is identical. 📄 p. 1
2d Counterexample — Disproving a FALSE Asymptotic Claim Lecture PDF: pp. 43–54 (Lecs 5–6) · Assignment 2 Q1.3 📋 A2 Q1.3

Intuition: One black swan is enough 📄 pp. 43–54

When a claim says "for ALL functions f and g, [property] holds," you can disprove it by showing a single pair where the property fails. You don't need to analyze all pairs — just one specific example suffices.

The art of counterexamples is finding the simplest, smallest, most obvious pair. For asymptotic claims, the trick is often to use constant functions or functions that approach different limits. 📄 pp. 43–54

Mechanism: The counterexample template 📄 pp. 43–54

COUNTEREXAMPLE TEMPLATE
1. State the claim to disprove.
2. Exhibit specific objects: "Let f(n) = [specific function] and g(n) = [specific function]."
3. Verify the hypothesis holds: "We check that f(n) = O(g(n)) [or whatever the if-part requires]. [Show this.] ✓"
4. Show the conclusion fails: "We show that [log f(n) ≠ Θ(log g(n))] for these specific functions. [Show this.] ✗"
5. Conclude: "Since f and g satisfy the hypothesis but not the conclusion, the claim is FALSE. ∎"

Worked Example: Claim — "f = Θ(g) implies log f = Θ(log g)" — FALSE 📄 pp. 43–54 📋 A2 Q1.3

Claim to disprove: "For all positive functions f and g, if f(n) = Θ(g(n)), then log f(n) = Θ(log g(n))."

Step 1: Choose the counterexample functions.
Let f(n) = 2 (constant function, always equals 2) and g(n) = 1 + 1/n (converges to 1 from above). 📄 pp. 43–54
Step 2: Verify that f(n) = Θ(g(n)).
We need constants c₁, c₂ and n₀ such that c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀.
g(n) = 1 + 1/n. For n ≥ 1: g(n) ranges in (1, 2].
f(n) = 2 always.
Choose c₁ = 1 and c₂ = 2 and n₀ = 1:
Lower: 1·(1 + 1/n) = 1 + 1/n ≤ 2 = f(n) ✓ (since 1 + 1/n ≤ 2 for all n ≥ 1).
Upper: 2·(1 + 1/n) = 2 + 2/n ≥ 2 = f(n) ✓ (since 2 + 2/n ≥ 2 always).
So f(n) = Θ(g(n)). The if-part of the claim is satisfied. ✓ 📄 pp. 39–44
Step 3: Compute log f(n) and log g(n).
log f(n) = log 2 = 1 (constant, since f(n) = 2 for all n).
log g(n) = log(1 + 1/n). As n → ∞: log(1 + 1/n) → log(1) = 0. 📄 pp. 43–54
Step 4: Show log f(n) ≠ Θ(log g(n)) using the limit test.
Examine the ratio: (log f(n)) / (log g(n)) = 1 / log(1 + 1/n).
As n → ∞: log(1 + 1/n) ≈ 1/n → 0 (for any logarithm base). So the ratio 1 / (1/n) = n → ∞.
Since lim (log f(n)) / (log g(n)) = ∞, we have log f = ω(log g) — log f grows MUCH FASTER than log g (in the asymptotic sense of ratios).
In particular, there is NO constant c such that log f(n) ≤ c · log g(n) for all large n, because log g(n) → 0 while log f(n) = 1 remains fixed.
Therefore log f(n) ≠ O(log g(n)), which means log f(n) ≠ Θ(log g(n)). ✗ 📄 pp. 43–54
Step 5: Conclude.
With f(n) = 2 and g(n) = 1 + 1/n: f = Θ(g) holds, but log f ≠ Θ(log g). The claim is FALSE. ∎ 📄 pp. 43–54

Verifying two TRUE claims from Exercises 2–3 📄 pp. 43–54 📋 A2 Q1.1 📋 A2 Q1.2

Not all asymptotic claims are false. Let's quickly trace the proofs for the two TRUE claims from the lecture exercises and Assignment 2:

TRUE Claim 1 (A2 Q1.1): If f(n) = O(g(n)) and f(n) = Ω(h(n)), then g(n) = Ω(h(n)).
Given: f = O(g): ∃ c₁, n₁ such that f(n) ≤ c₁·g(n) for n ≥ n₁. 📄 pp. 39–44
f = Ω(h): ∃ c₂, n₂ such that f(n) ≥ c₂·h(n) for n ≥ n₂. 📄 pp. 39–44
Chain the inequalities: For n ≥ max(n₁, n₂):
c₂·h(n) ≤ f(n) ≤ c₁·g(n).
So h(n) ≤ (c₁/c₂)·g(n). 📄 pp. 43–54
Conclude: There exists constant c = c₁/c₂ > 0 such that h(n) ≤ c·g(n) for all n ≥ max(n₁,n₂). Wait — we need the other direction for Ω: g(n) ≥ (1/c)·h(n). Yes: g(n) ≥ (c₂/c₁)·h(n). So g = Ω(h). ✓ 📄 pp. 43–54
TRUE Claim 2 (A2 Q1.2): If f(n) = O(g(n)), then f(n)² = O(g(n)²).
Given: f = O(g): ∃ c₀, n₀ such that f(n) ≤ c₀·g(n) for n ≥ n₀. 📄 pp. 39–44
Since both f and g are positive: we can square both sides (squaring preserves inequality for positive values):
f(n)² ≤ (c₀·g(n))² = c₀²·g(n)². 📄 pp. 43–54
Conclude: Let c' = c₀² > 0. Then f(n)² ≤ c'·g(n)² for all n ≥ n₀. So f² = O(g²). ✓ 📄 pp. 43–54

📋 A0 Q2.1 Prove by induction: a binary tree of height n has at most 2ⁿ leaf nodes.

▶ Key Points in the Solution
  • Statement P(n): Every binary tree of height n has at most 2^n leaves. Domain: all integers n ≥ 0. 📄 p. 1
  • Base case (n=0): A tree of height 0 is just the root (a single leaf). Leaf count = 1 = 2⁰. ✓ 📄 p. 1
  • IH: Assume any binary tree of height k has at most 2^k leaves. 📄 p. 1
  • Inductive step: Take any tree T of height k+1. T has a root plus two subtrees T_L, T_R, each of height ≤ k. By IH: |leaves(T_L)| ≤ 2^k and |leaves(T_R)| ≤ 2^k. Total leaves = |leaves(T_L)| + |leaves(T_R)| ≤ 2^k + 2^k = 2^(k+1). ✓ 📄 p. 1
  • Grading criterion (100%): All 5 parts present (P(n) statement, base case with explicit value, IH explicitly stated, inductive step using IH, conclusion). Missing any part = partial credit. 📄 p. 1
  • Exam variant: "A binary tree of height n has at least 1 leaf." — Also provable by induction; base case n=0: 1 leaf ≥ 1 ✓; IH: assume height-k tree has ≥ 1 leaf; IS: height-(k+1) tree has ≥ 1 subtree which has ≥ 1 leaf by IH. 📄 p. 1

📋 A0 Q2.2 Prove by construction: the sum of any two consecutive integers is odd.

▶ Key Points in the Solution
  • Construction: Let a be any integer. Its consecutive pair is b = a+1. Then a + b = a + (a+1) = 2a+1. 📄 p. 1
  • Verification: 2a+1 = 2k+1 with k = a (an integer). By definition, 2k+1 is odd. So a+b is odd. 📄 p. 1
  • Why NOT induction: The claim is about pairs of integers, not about "all n ≥ n₀" with recursive structure. The construction is direct and complete in 2 lines. Induction would be circular overkill. 📄 p. 1
  • Grading criterion (100%): Explicit substitution b = a+1; algebraic computation 2a+1; identification as odd via 2k+1 form; any valid integer a; conclusion stated. 📄 p. 1
  • Exam variant: "Prove the product of two consecutive integers is even." — Let b = a+1; a·b = a(a+1). One of a, a+1 must be even (consecutive integers have opposite parity); so a(a+1) = 2m for some m. ✓ 📄 p. 1

📋 A0 Q2.3 Prove by contradiction: if n² is even, then n is even.

▶ Key Points in the Solution
  • Negation: Assume n is ODD (this is the negation of the conclusion "n is even"). 📄 p. 1
  • Write n = 2k+1: By definition of odd integers. 📄 p. 1

Misconception Cards 📄 pp. 1–54

❌ Misconception: "In a proof by contradiction, I disprove the claim."

✅ Reality: You PROVE the claim. You temporarily assume the claim is false, derive an absurdity, and conclude the claim is TRUE. The contradiction shows the negation is impossible — leaving only the original claim standing. 📄 p. 1

❌ Misconception: "The base case proves the whole claim."

✅ Reality: The base case only proves P(n₀) — one single value. The inductive step is what extends the proof to infinitely many n. Without the inductive step, you've proved nothing beyond the base case. 📄 p. 1

❌ Misconception: "I can use any value of k in the inductive step — I'll just pick k=5."

✅ Reality: The inductive step must work for an ARBITRARY k ≥ n₀ — you cannot fix a specific value. Picking k=5 would only prove P(6), not all P(k+1). The step must work for every possible k. 📄 p. 1

❌ Misconception: "To show a claim is FALSE, I just say 'I couldn't find a proof.'"

✅ Reality: Inability to find a proof does NOT mean the claim is false — you might just be missing the right technique. To show a claim is FALSE, you must produce a specific counterexample that visibly violates it. 📄 pp. 43–54

❌ Misconception: "(2k+1)² = 4k² + 1."

✅ Reality: (2k+1)² = 4k² + 4k + 1. The cross term 2·(2k)·1 = 4k is essential and is always forgotten under exam pressure. Write FOIL explicitly: (2k+1)(2k+1) = 4k² + 2k + 2k + 1 = 4k² + 4k + 1. 📄 p. 1

❌ Misconception: "For asymptotic proofs, the limit method is optional — I can just 'say' it's O(n²)."

✅ Reality: On graded assignments and the final exam, you must SHOW the work: either exhibit explicit c and n₀ satisfying the formal definition, or compute the limit explicitly. Stating the conclusion without proof receives no credit. 📄 pp. 39–54

❌ Misconception: "Proof by induction can prove any statement about natural numbers."

✅ Reality: Induction proves statements that have the property "if n works, so does n+1." Some number-theoretic statements (like Goldbach's conjecture) have not been proved by any method, including induction. Induction is powerful but not omnipotent. 📄 p. 1

Memory Aids & Mnemonics 📄 pp. 1–54

🧠 Mnemonic 1: Induction steps — "B-I-IS-C" (Base, IH, Inductive Step, Conclude) Write these four words at the top of every induction proof before you write a single formula: Base case → Inductive Hypothesis → Inductive Step → Conclusion. If any of the four is missing, the proof is incomplete. Check by covering each with your finger. 📄 p. 1
🧠 Mnemonic 2: Contradiction structure — "ANDC" (Assume Negation, Derive, Contradict) Assume ¬P. Name what ¬P says explicitly (write n = 2k+1 etc.). Derive consequences using definitions. Contradict a known fact. Then conclude P. "ANDC" in contradiction proofs = full credit. 📄 p. 1
🧠 Mnemonic 3: Counterexample recipe — "EHF" (Exhibit, Hypothesis, Fail) To disprove a claim: Exhibit specific f₀, g₀. Hypothesis: show f₀=Θ(g₀) (the if-part holds). Fail: show log f₀ ≠ Θ(log g₀) (the conclusion fails). Without all three, the counterexample is incomplete. 📄 pp. 43–54
🧠 Mnemonic 4: Even/Odd algebra — "2k is even, 2k+1 is odd" Any expression of the form 2·(integer) is even. Any expression of the form 2·(integer)+1 is odd. When you need to show something is odd: manipulate it into the form 2m+1. When you need even: into 2m. This pattern appears in EVERY parity proof — internalize it. 📄 p. 1
🧠 Mnemonic 5: Asymptotic formal proof — "c, n₀, and the inequality" Every Big-O proof needs THREE things: a constant c > 0, a threshold n₀, and the inequality f(n) ≤ c·g(n) verified for ALL n ≥ n₀. Missing any one of the three = incomplete proof = partial credit. 📄 pp. 39–44

WARM-UP PROOF LABELING 📄 p. 1

WC-2. Below is an induction proof skeleton with the structural labels removed. Identify which part is the base case, which is the IH, and which is the inductive step.

  Claim: For all integers n ≥ 1, 1 + 2 + 3 + ... + n = n(n+1)/2.

  (A) When n = 1: LHS = 1. RHS = 1(1+1)/2 = 1. LHS = RHS. ✓

  (B) Assume that for some integer k ≥ 1:
      1 + 2 + ... + k = k(k+1)/2.

  (C) We want to show: 1 + 2 + ... + k + (k+1) = (k+1)(k+2)/2.
      LHS = (1+2+...+k) + (k+1)
           = k(k+1)/2 + (k+1)     [using part B]
           = (k+1)(k/2 + 1)
           = (k+1)(k+2)/2         ✓
▶ Show Explanation

(A) is the BASE CASE: It verifies the claim for the smallest value n = 1, directly by computation. No inductive hypothesis is used here. 📄 p. 1

(B) is the INDUCTIVE HYPOTHESIS (IH): It states the assumption "assume P(k) is true" — i.e., the claim holds for some arbitrary k. Note the word "Assume" — this is being borrowed, not proved. 📄 p. 1

(C) is the INDUCTIVE STEP: It proves P(k+1) — the claim for k+1. Crucially, it uses part (B) (the IH) on the third line: "using part B" is the explicit citation of the inductive hypothesis. This is the required connection. 📄 p. 1

After (C) is established, the conclusion would follow: "By induction, the claim holds for all n ≥ 1. ∎" 📄 p. 1

▶ Show Answer 
function classifyAsymptoticRelation(f_values, g_values, N):
    # f_values[i] = f(i), g_values[i] = g(i), for i = 1 to N
    # Determine the asymptotic relationship by looking at the limit ratio

    ratios = array of size N

    # Step 1: Compute f(n)/g(n) for each n
    for i = 1 to N:
        if g_values[i] == 0:
            ratios[i] = infinity   # avoid division by zero
        else:
            ratios[i] = f_values[i] / g_values[i]

    # Step 2: Look at the last few values as a proxy for the limit
    WINDOW = min(10, N)   # use last 10 values
    sum_last = 0
    for i = (N - WINDOW + 1) to N:
        sum_last = sum_last + ratios[i]
    avg_last = sum_last / WINDOW

    # Step 3: Classify based on average ratio
    EPSILON = 0.01   # tolerance for "converging to constant"
    if avg_last < EPSILON:
        return "f = o(g): ratio → 0, g grows faster"
    elif avg_last > 1 / EPSILON:   # very large
        return "g = o(f): ratio → ∞, f grows faster"
    else:
        return "f = Θ(g): ratio → constant ≈ " + avg_last

  1. For each n from 1 to N, compute the ratio f(n)/g(n). This is the empirical analog of the limit test.
  2. Examine the last WINDOW = 10 ratios as an approximation of the limit behavior.
  3. If the average ratio is nearly 0 → f is growing slower than g (f = o(g)). If very large → f is growing faster (g = o(f)). If a moderate constant → f = Θ(g).

    while n0 <= 10000:   # search up to 10000
        fn = a * n0 * n0 + b * n0 + c
        g = n0 * n0       # g(n) = n^2
        if fn >= c1 * g and fn <= c2 * g:
            # Check if bounds hold for next 100 values (confidence check)
            valid = True
            for n = n0 + 1 to n0 + 100:
                fn_check = a*n*n + b*n + c
                gn_check = n*n
                if fn_check < c1 * gn_check or fn_check > c2 * gn_check:
                    valid = False
                    break
            if valid:
                return (c1, c2, n0)
        n0 = n0 + 1

    return "Could not find witnesses (function may not be Theta(n^2))"
SOURCE: conceptual implementation, 📄 pp. 39–44

English Explanation:

  1. Upper bound c₂: every lower-degree term (b·n and c) is dominated by n² for large n. We use c₂ = a + |b| + |c| as a safe overestimate.
  2. Lower bound c₁: use half the leading coefficient. For large enough n, the lower-order terms become negligible and f(n) ≈ a·n².
  3. Search for n₀: scan increasing n until both bounds hold stably for 100 consecutive values.

📝 Complexity Analysis:

Time: Steps 1–2 are O(1). Step 3 scans up to 10,000 values, and for each candidate n₀, runs 100 verification steps: O(10,000 × 100) = O(1,000,000) = O(1) (constant, since the bound is fixed regardless of input). 📄 pp. 39–44

Space: O(1) — only a fixed number of scalar variables. 📄 pp. 39–44

Q13. Consider the recursive algorithm for computing the sum of an array: Sum(A, n) returns A[0] if n=1, else returns A[n-1] + Sum(A, n-1). (a) Write this pseudocode cleanly. (b) Prove by induction that Sum(A, n) correctly computes A[0]+A[1]+...+A[n-1] for all n ≥ 1. (c) State and solve its recurrence relation T(n) for time complexity.

▶ Show Answer

(a) Pseudocode:

function Sum(A, n):
    # Returns sum of A[0] + A[1] + ... + A[n-1]
    # Base case: single element
    if n == 1:
        return A[0]
    # Recursive case: last element + sum of first n-1 elements
    else:
        return A[n-1] + Sum(A, n-1)
SOURCE: conceptual, 📄 pp. 55–62

(b) Correctness Proof by Induction: 📄 p. 1

Claim P(n): Sum(A, n) = A[0] + A[1] + ... + A[n-1] for all n ≥ 1.
Base Case (n=1): Sum(A, 1) returns A[0]. The sum A[0] is exactly A[0]+...+A[0] (one term). P(1) holds. ✓ 📄 p. 1
Inductive Hypothesis: Assume P(k): Sum(A, k) = A[0]+A[1]+...+A[k-1] for some k ≥ 1. 📄 p. 1
Inductive Step (n = k+1): Sum(A, k+1) = A[k] + Sum(A, k) [by the algorithm's recursive case]
= A[k] + (A[0]+A[1]+...+A[k-1]) [by the IH applied to Sum(A,k)]
= A[0]+A[1]+...+A[k-1]+A[k]
= A[0]+...+A[(k+1)-1]. ✓ This is P(k+1). 📄 p. 1
Conclusion: By induction, Sum(A, n) correctly computes the sum for all n ≥ 1. ∎ 📄 p. 1

(c) Recurrence and Time Complexity: 📄 pp. 79–90

The recurrence: T(1) = 1 (base case does O(1) work). T(n) = T(n-1) + 1 (recursive call on n-1 plus O(1) addition).

Solve by unrolling: T(n) = T(n-1)+1 = T(n-2)+2 = ... = T(1)+(n-1) = 1+(n-1) = n.

Therefore: T(n) = O(n) time, O(n) stack space due to n-deep recursion. 📄 pp. 79–90

THE #1 MISTAKE ON EXAM QUESTIONS ABOUT THIS TOPIC:

In induction proofs: forgetting to explicitly cite the inductive hypothesis in the inductive step. Writing "by induction" or "by assumption" is too vague. You must write: "By the inductive hypothesis, [quote exactly what the IH says for k]." The grader needs to see the IH activated and applied — not implied. 📄 p. 1

In contradiction proofs: stopping at "n² is odd" without connecting it back to the stated hypothesis ("n² is even"). Always name the contradiction explicitly: "This contradicts our assumption that n² is even." 📄 p. 1
