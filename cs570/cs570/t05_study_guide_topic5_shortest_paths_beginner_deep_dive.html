🧗 Warm-up Glossary — Vocabulary You Need Before §1

Shortest-path algorithms build on several concepts from earlier lectures. If you feel shaky on any of these, read the explanation before continuing. None of them are hard — they just need to be introduced before the algorithms use them.

What is a weighted graph? 📄 p. 169

A weighted graph G = (V, E) is a graph where each edge (u, v) has an associated number called its weight (also called cost or length). The weight might represent distance in kilometers, time in minutes, ticket price in dollars, or probability of a connection. 

  Example weighted graph:

       A ——10—— B
       |         |
       6        15
       |         |
       C ——4——  D ——8—— E

  Edge (A,B) has weight 10. Edge (A,C) has weight 6.
  "The path A → C → D has total weight 6 + 4 = 10."
SOURCE: 📄 p. 169

What is a "path" in a graph? 📄 p. 169

A path from vertex s to vertex t is a sequence of vertices s = v₀, v₁, v₂, ..., v_k = t where each consecutive pair (v_i, v_{i+1}) is an edge in E. The cost (or length) of the path is the sum of all edge weights along it. The shortest path is the path with minimum total cost.

Example: In the graph above, from A to E there are two paths: A→B→D→E (cost 10+15+8=33) and A→C→D→E (cost 6+4+8=18). The shortest path is A→C→D→E with cost 18.

What does "greedy" mean as an algorithmic strategy? 📄 p. 179

A greedy algorithm makes a sequence of locally optimal choices — at each step it picks the best-looking option available right now — without ever going back and reconsidering earlier choices. The hope is that a sequence of locally best choices produces a globally best result. Sometimes this works (Dijkstra's algorithm, for example). Sometimes it doesn't.

Analogy: Imagine navigating a maze by always taking the door that looks closest to the exit. That's greedy. You commit to each door without backtracking. If the maze has no tricks, you'll reach the exit quickly. If there are dead ends or penalties for certain paths, you might get stuck.

What is a priority queue (min-heap)? 📄 p. 179

A priority queue is a data structure that always lets you quickly access and remove the item with the smallest value (in a min-heap). The three key operations are:

  • Insert(item, priority): Add an item with a priority value — O(log n)
  • Extract-Min(): Remove and return the item with the smallest priority — O(log n)
  • Decrease-Key(item, new_priority): Update an item's priority to a smaller value — O(log n)

Dijkstra's algorithm uses a min-heap to always find the "next closest unvisited vertex" efficiently. Without a heap, this search takes O(n) per step; with a heap, it takes O(log n).

What is "edge relaxation"? 📄 p. 207

Relaxation is the core operation of all shortest-path algorithms. For an edge (u, v) with weight w(u,v):

📌 The Relaxation Formula If Distance[u] + weight(u,v) < Distance[v], then update: Distance[v] = Distance[u] + weight(u,v)

In plain English: "We found a shorter path to v by going through u. Update our best known distance." Relaxation is safe — it can only make distances smaller, never accidentally enlarge them. 📄 p. 207

What is a "settled" (or "burnt") vertex in Dijkstra's? 📄 p. 173

When Dijkstra's algorithm "settles" a vertex v (also called "burning" it in the lecture's fire analogy), it means: we have found the definitive shortest path from the source s to v, and we will never update Distance[v] again. The set of settled vertices grows one by one until all vertices are settled.

Why the fire analogy? Imagine pouring kerosene on vertex s and lighting it on fire. The fire spreads along edges, and the time it reaches each vertex is proportional to the edge weight. Dijkstra's algorithm simulates this spreading fire. 📄 p. 173

What is a spanning tree? 📄 p. 227

A spanning tree of a connected graph G = (V, E) is a subgraph that: (1) is connected, (2) contains all n vertices of G, and (3) has exactly n−1 edges. A spanning tree has no cycles — adding any edge to it would create a cycle.

Why n−1 edges? A tree with n nodes always has exactly n−1 edges — one fewer than the number of vertices. This is a theorem from graph theory.

What is a negative cycle? 📄 p. 207

A negative cycle is a cycle (a path that returns to its starting vertex) whose total edge weight is negative. Negative cycles make the shortest-path problem undefined: you could keep traversing the cycle to make the path cost arbitrarily negative. Example: a cycle A→B→C→A with weights 3, −5, 1 has total weight −1.

What is the difference between single-source and all-pairs shortest paths? 📄 pp. 169, 217

Single-source: Fix one source vertex s. Find the shortest path from s to every other vertex. Result: n−1 distances. Used when you have one starting point (e.g., one factory shipping to all stores). 📄 p. 169

All-pairs: Find the shortest path between every pair of vertices (u, v). Result: n² distances stored in an n×n matrix. Used when you need flexible routing (e.g., railway connections between all cities). 📄 p. 217

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 169–226 (Lectures 17–22)

1.1 The fundamental problem — navigating a weighted network 📄 p. 169

Imagine you're a GPS system. A driver is at point A and wants to reach point F. The road network has dozens of intersections (vertices) and roads (edges), and each road has a travel time (weight). Your job is to find the fastest route — not necessarily the route with the fewest turns.

This is the shortest path problem. It appears in shipping logistics (cheapest route from warehouse to store), network routing (fastest path for a data packet), social networks (degrees of separation), and even chemistry (minimum energy reaction pathways).

The crucial word is weighted. Without weights, BFS finds the shortest path (fewest edges) in O(n+m) time — you already know this from Topic 4. But with weights, BFS fails: the path with fewest edges might cost far more than a longer path with cheap edges. We need smarter algorithms. 📄 pp. 169–171 

  WHY BFS FAILS ON WEIGHTED GRAPHS:

  Source S ——1—— A ——1—— T         (path S→A→T: 2 edges, cost 2)
  S ——50—————————————— T           (path S→T:   1 edge,  cost 50)

  BFS finds S→T first (1 hop). But the TRUE shortest path is S→A→T (cost 2).
  BFS counts hops, not costs. We need a cost-aware algorithm.
SOURCE: 📄 pp. 169–171

1.3 A second framing — the spreading fire analogy 📄 p. 173

Dijkstra's algorithm is most naturally understood through the lecture's fire analogy. Imagine the graph is a network of fuses. Each edge is a fuse, and its weight is how long the fuse burns (in seconds). You light vertex s on fire at time 0.

The fire spreads: if vertex u burns at time T, then the fuse to vertex v (edge weight w) will reach v at time T + w. Vertex v burns at the earliest time any fuse reaches it. Dijkstra's algorithm simulates this process — at each step, it commits to the vertex that will burn next (the one with the current minimum "expected burn time"). 

  FIRE ANALOGY for Dijkstra's:

  Step 0: Light vertex 1 at time 0. Fuses start burning.
           1 ——(6)—— 2
           |
          (5)        The fuse to 2 will finish at time 0+6=6.
           |         The fuse to 3 will finish at time 0+5=5.
           3

  Step 1: Vertex 3 burns first (time 5). Burn time committed.
           New fuses start from 3 to its neighbors.

  Step 2: Next earliest burn is vertex 2 at time 6. Committed.
           ...and so on.

  Key insight: once a vertex burns, its burn time is FINAL.
  A later fire via a longer path can never arrive earlier.
  This is why the greedy choice is safe — for non-negative weights!
SOURCE: 📄 p. 173
⚠️ Common Point of Confusion — Read This Before §2 Dijkstra's algorithm does NOT work with negative edge weights. The fire analogy explains why: if some fuses could "un-burn" (negative weights = going back in time), a vertex you already committed as "done" might later be reachable via a cheaper path. With negative edges, the greedy commitment becomes invalid. This is exactly why Bellman-Ford was invented. 📄 pp. 184–190

1.4 Technical definitions (lecture notation) 📄 pp. 169–217

Weighted graph: G = (V, E) together with weight function w: E → ℝ (the reals). Each edge e ∈ E has a weight w(e). 📄 p. 169

Path cost: If the path is v₀→v₁→...→v_k, the cost = w(v₀,v₁) + w(v₁,v₂) + ... + w(v_{k-1},v_k). 📄 p. 169

Shortest path from s to t: The path with minimum cost among all paths from s to t. 📄 p. 169

Relaxation: update(j,k): Distance[k] = min(Distance[k], Distance[j] + weight(j,k)). 📄 p. 207

MST (Minimum Spanning Tree): A spanning tree (n vertices, n−1 edges, connected, no cycles) of minimum total edge weight. Key property: any minimum-weight connected spanning subgraph with positive weights IS a tree. 📄 p. 227

2a Dijkstra's Algorithm — Greedy Shortest Paths (Non-Negative Weights Only) Lecture PDF: pp. 169–184 · Midterm trace question · 📋 A6 Q9 variant

Intuition: Commit to the closest unvisited vertex 📄 pp. 172–175

Dijkstra's algorithm is a greedy algorithm. It maintains an array Distance[] — your best current guess for how far each vertex is from the source. Initially, Distance[source] = 0 and all others are infinity (we don't know yet).

At each step, it commits (settles) the unvisited vertex with the smallest Distance value — this is the greedy choice. Then it looks at all that vertex's neighbors and tries to improve their Distance values through the settled vertex.

Why is the greedy choice safe? Because all edge weights are non-negative! If Distance[u] is the smallest among all unsettled vertices, there is NO path to u that could be shorter — any alternative path must pass through at least one unsettled vertex with an even higher Distance value. So Distance[u] must already be correct. 📄 pp. 179–184

Mechanism: Dijkstra's pseudocode 📄 pp. 172–178

function Dijkstra(G, s):
    # Initialize all distances to infinity, source to 0
    for each vertex i in V:
        Visited[i] = False
        Distance[i] = infinity      # we know nothing yet

    Distance[s] = 0                 # source is 0 hops from itself

    # Main loop: settle one vertex per iteration
    repeat n times:
        # Greedy choice: pick unvisited vertex with smallest Distance
        u = argmin{ Distance[i] : Visited[i] == False }

        Visited[u] = True           # COMMIT: Distance[u] is now final

        # Relax all edges out of u
        for each neighbor k of u:
            if Visited[k] == False:
                # Can we reach k more cheaply through u?
                if Distance[u] + weight(u, k) < Distance[k]:
                    Distance[k] = Distance[u] + weight(u, k)
                    # (In heap version: also decrease key in priority queue)
SOURCE: 📄 pp. 172–178

Each line in plain English:
Line 3-4: We start by saying "I don't know how to get anywhere from s yet." ∞ means "unknown."
Line 6: The source costs 0 to reach itself — trivially true.
Line 9: We're about to commit the most promising vertex. Who's closest?
Line 11: Once committed, u's distance is permanently correct.
Line 14-15: For every road leading out of u, check: "Is going through u a shortcut to k?" If yes, update. 📄 pp. 172–178

2b Bellman-Ford Algorithm — Handling Negative Edge Weights Lecture PDF: pp. 207–216 (Lecture 21)

Intuition: Why Dijkstra's fails and what Bellman-Ford does differently 📄 pp. 184–190

With negative edge weights, Dijkstra's greedy strategy breaks. Here is a concrete example showing why: 

  DIJKSTRA'S FAILURE WITH NEGATIVE WEIGHTS:

       Source S                Target T
           \                  /
            1               -5
             \              /
              A ——(2)——— B

  Edges: (S,A)=1, (S,T)=4, (A,B)=2, (B,T)=-5

  Dijkstra's would settle S first, then T (Distance=4), then A (Distance=1).
  But the correct shortest path is S→A→B→T = 1+2+(−5) = −2!
  Dijkstra committed T at distance 4 too early — T was later reachable at −2.

  The problem: a later vertex (B) had a negative edge that "un-did" an earlier
  commitment. Greedy commitment becomes invalid with negative edges.
SOURCE: 📄 pp. 184–190

Bellman-Ford's key insight: Instead of committing one vertex per step (greedy), just do a "sweep" — relax ALL edges in the graph. Repeat this sweep n−1 times. After k sweeps, we have the correct shortest paths that use AT MOST k edges. Since any shortest path in an n-vertex graph uses at most n−1 edges (it never revisits a vertex), n−1 sweeps guarantee correctness. 📄 pp. 207–216

Mechanism: Bellman-Ford pseudocode 📄 pp. 210–216

function BellmanFord(G, s):
    # Initialize all distances to infinity, source to 0
    for each vertex i in V:
        Distance[i] = infinity
    Distance[s] = 0

    # Repeat n-1 times: relax ALL edges
    for iteration = 1 to n-1:
        for each edge (j, k) in E:            # scan ALL edges
            # Relaxation step
            if Distance[j] + weight(j, k) < Distance[k]:
                Distance[k] = Distance[j] + weight(j, k)

    # Optional: detect negative cycles
    # If we can still relax after n-1 iterations, a neg. cycle exists
    for each edge (j, k) in E:
        if Distance[j] + weight(j, k) < Distance[k]:
            return "NEGATIVE CYCLE DETECTED"

    return Distance
SOURCE: 📄 pp. 210–216

Why n−1 iterations? A shortest path from s to any vertex t visits at most n−1 edges (it never visits the same vertex twice in a simple path — n vertices means at most n−1 hops). After i iterations, Bellman-Ford has correctly computed all shortest paths using at most i edges. After n−1 iterations, all paths are covered. 📄 pp. 207–210

2c Floyd-Warshall — All-Pairs Shortest Paths via Dynamic Programming Lecture PDF: pp. 217–226 (Lecture 22)

Intuition: Build up solutions by allowing more and more intermediate vertices 📄 pp. 217–220

Floyd-Warshall is a dynamic programming algorithm. The key idea is to progressively allow more vertices to serve as "intermediate stops" on shortest paths. We build a series of distance matrices W⁰, W¹, W², ..., Wⁿ where W^k[i][j] = the shortest path from i to j using only vertices {1, 2, ..., k} as possible intermediate stops (with i and j being any vertices).

W⁰[i][j]: Shortest path using NO intermediate vertices = direct edge weight (or ∞ if no edge).
W^k[i][j]: Shortest path allowed to pass through vertices {1,...,k}.
Wⁿ[i][j]: Shortest path using ANY vertex as intermediate = TRUE shortest path.

Mechanism: The recurrence 📄 pp. 220–226

Going from W^{k-1} to W^k: when we allow vertex k as a new intermediate, for each pair (i,j) there are exactly two cases:

Floyd-Warshall Recurrence
Case 1: The shortest path from i to j (via {1,...,k}) does NOT use k as intermediate.
Then W^k[i][j] = W^{k-1}[i][j]. (same as before, k doesn't help)
Case 2: The shortest path from i to j (via {1,...,k}) DOES go through k.
Since a simple path visits k at most once: path = (i → ... → k → ... → j)
Both sub-paths only use {1,...,k-1} as intermediates.
So: W^k[i][j] = W^{k-1}[i][k] + W^{k-1}[k][j]
Final formula: W^k[i][j] = min( W^{k-1}[i][j], W^{k-1}[i][k] + W^{k-1}[k][j] ) 📄 pp. 220–226 
function FloydWarshall(G):
    # Initialize: W[i][j] = direct edge weight, or infinity
    for i = 1 to n:
        for j = 1 to n:
            if i == j: W[i][j] = 0
            elif edge (i,j) exists: W[i][j] = weight(i,j)
            else: W[i][j] = infinity

    # Main DP: add vertices one by one as potential intermediates
    for k = 1 to n:                          # k is the new intermediate
        for i = 1 to n:
            for j = 1 to n:
                # Can we do better going through k?
                W[i][j] = min(W[i][j], W[i][k] + W[k][j])

    return W    # W[i][j] = shortest path from i to j
SOURCE: 📄 pp. 217–226

Space optimization: We don't need to keep all k-levels in memory. At each step of the k-loop, we only need W^{k-1} to compute W^k. In practice we use a single 2D array W and update it in-place — this works correctly because any path from i to k or k to j that uses k as intermediate is not a simple path and won't be selected. 📄 p. 226

MST Property (A6 Q6): Why minimum connected = tree 📄 p. 227

Claim (Assignment 6 Q6): In a graph where all edge weights are positive, consider any subset of edges that connects all vertices and has minimum total weight. Such a subset must be a tree.

Plain English: If you want the cheapest "skeleton" that keeps all cities connected, it will always be a tree (no cycles). This makes intuitive sense: a cycle is redundant — remove any edge from the cycle and you save weight while staying connected.

Proof: Minimum Connected Spanning Subgraph with Positive Weights = Tree
Step 1 [It's connected and spans all vertices]. Given by assumption — it connects all n vertices. 📄 p. 227
Step 2 [Suppose it has a cycle]. Assume for contradiction that our subgraph contains a cycle C = (v₁, v₂, ..., v_k, v₁).
Step 3 [Remove any edge from the cycle]. Pick any edge e = (v_i, v_{i+1}) in C and remove it. The graph stays connected! Why? Because any path that previously used edge e can now reroute through the rest of the cycle C (going the long way around). 📄 p. 227
Step 4 [Total weight strictly decreases]. Since weight(e) > 0 (all weights are positive), removing e strictly reduces the total weight.
Step 5 [Contradiction]. We found a connected spanning subgraph with strictly less weight. But we assumed our original was minimum. Contradiction. ✗
Step 6 [Conclusion]. No cycles exist. A connected graph with n vertices and no cycles has exactly n−1 edges — by definition, it's a tree. ∎ 📄 p. 227

function MaxProbabilityPath(edges, probs, n, src, tgt):
    # Build adjacency list with probabilities
    adj = {} (default empty list)
    for each (u,v), prob in zip(edges, probs):
        adj[u].append((v, prob))
        adj[v].append((u, prob))    # undirected graph

    # Modified Dijkstra: maximize probability
    Prob = [0.0] * n
    Prob[src] = 1.0

    # Max-heap (negate probability to use Python's min-heap)
    maxHeap = [(-1.0, src)]        # (-prob, vertex)

    while maxHeap is not empty:
        (-prob, u) = heappop(maxHeap)   # extract max-probability vertex
        prob = -prob

        if u == tgt:
            return prob                  # found target!

        for (v, edge_prob) in adj[u]:
            new_prob = prob * edge_prob
            if new_prob > Prob[v]:
                Prob[v] = new_prob
                heappush(maxHeap, (-new_prob, v))

    return 0.0    # target not reachable

Misconception Cards 📄 pp. 169–226

❌ Misconception: "Dijkstra's algorithm works with any edge weights — it just finds the minimum."

✅ Reality: Dijkstra's algorithm is ONLY correct for non-negative edge weights. With negative weights, a greedy commitment can be wrong — a later path through negative edges might be shorter than a committed distance. This is why Bellman-Ford was invented. 📄 pp. 184–190

❌ Misconception: "Bellman-Ford does n iterations."

✅ Reality: Bellman-Ford does n−1 iterations. Because the shortest simple path in an n-vertex graph uses at most n−1 edges (it never revisits a vertex). The n-th iteration is only used to DETECT negative cycles (if anything still updates, a negative cycle exists). 📄 pp. 207–210

❌ Misconception: "Floyd-Warshall stores n separate k-indexed matrices, using O(n³) space."

✅ Reality: Floyd-Warshall can be implemented with just ONE n×n matrix, updated in-place. The key insight: W[i][k] and W[k][j] are not changed when updating W[i][j] in iteration k (because those involve k as an endpoint, not just intermediate). So updating in-place is safe. Space = O(n²). 📄 p. 226

❌ Misconception: "Dijkstra's finds the shortest path tree (the tree of shortest paths from s to all vertices) in O(m) time."

✅ Reality: Dijkstra's runs in O(n²) naively, or O((n+m) log n) with a heap. It's not O(m) — the heap operations and the n Extract-Min calls add O(n log n). Only BFS finds shortest paths in O(n+m) — but only for unweighted graphs. 📄 pp. 179–184

❌ Misconception: "The MST property proof works even with zero-weight edges."

✅ Reality: The proof requires ALL POSITIVE weights. Step 3 of the proof says "removing a cycle edge strictly reduces total weight." If some edge has weight 0, removing it doesn't reduce the total — so the contradiction fails, and the minimum connected subgraph might have cycles (if cycle edges have weight 0). The claim explicitly needs positive weights. 📄 p. 227

❌ Misconception: "For max-probability path, just replace + with × in standard Dijkstra's code and change 'min' to 'max'."

✅ Reality: Almost right, but you also need to change the INITIALIZATION. Standard Dijkstra starts with Distance[s]=0, others=∞. Modified starts with Prob[s]=1.0 (100% certain we're at source), others=0 (0% probability of being there). These initial values reflect the multiplicative identity (1) vs additive identity (0). 📄 p. 227

❌ Misconception: "Negative cycles make Bellman-Ford return wrong answers."

✅ Reality: Negative cycles make the shortest-path PROBLEM undefined — not just Bellman-Ford. If a negative cycle is reachable from source, the "shortest" path to any vertex beyond it can be made arbitrarily negative (just loop the cycle more times). Bellman-Ford detects negative cycles by checking if any edge can still be relaxed after n−1 iterations. 📄 p. 207

WARM-UP ALGORITHM STRUCTURE 📄 pp. 172–178

WC-2. In Dijkstra's algorithm, what exactly happens at each "iteration"? Describe in 3 steps without looking at the pseudocode. Then explain why we mark a vertex as "visited" (settled) before we relax its neighbors, rather than after.

▶ Show Explanation

Three steps per iteration:

  1. Find the minimum: Among all unvisited vertices, pick the one with the smallest Distance value (the "closest" unvisited vertex). 📄 p. 172
  2. Settle it: Mark it as Visited = True. Its distance is now permanently correct and will never change. 📄 p. 172
  3. Relax neighbors: For each unvisited neighbor k of the settled vertex u, check if going through u is shorter. If so, update Distance[k]. 📄 pp. 172–173

Why settle BEFORE relaxing neighbors? Once we commit to vertex u (mark it Visited), we guarantee we won't revisit it. This allows the relaxation step to correctly skip already-settled neighbors — we don't want to update their distances again (they're already correct). Settling before relaxing also prevents double-counting: if u's distance were not committed first, a neighbor could be relaxed using an intermediate (not final) distance for u. 📄 pp. 179–184

WC-4. Floyd-Warshall uses the notation W^k[i][j]. Explain in plain English what this means. If W^2[3][5] = 7 in Floyd-Warshall, what does that tell us? What would W^0[3][5] mean?

▶ Show Explanation

W^k[i][j] means: the length of the shortest path from vertex i to vertex j, where all intermediate vertices on the path belong to {1, 2, ..., k}. In other words: the cheapest way to get from i to j if the only "relay stations" you're allowed to use are vertices numbered 1 through k. 📄 p. 217

W^2[3][5] = 7 means: the shortest path from vertex 3 to vertex 5, when only vertices 1 and 2 are allowed as intermediate stops, has cost 7. 📄 p. 217

W^0[3][5] means: the shortest path from 3 to 5 with NO intermediate stops — i.e., the direct edge weight w(3,5). If there's no direct edge from 3 to 5, then W^0[3][5] = ∞. 📄 p. 217

WARM-UP MISCONCEPTION PUNCTURE 📄 pp. 184–210

WC-5. Your classmate says: "Since Bellman-Ford handles negative weights and Dijkstra's doesn't, we should always use Bellman-Ford — it's more general." Explain what they got right and why you'd often still prefer Dijkstra's.

▶ Show Explanation

What they got right: Bellman-Ford is more general — it handles negative edge weights (as long as there are no negative cycles). If your graph has negative edges, Bellman-Ford is the right choice. 📄 pp. 207–210

Why Dijkstra's is often preferred: Bellman-Ford is O(nm) — for a dense graph with m = O(n²) edges, this is O(n³), which is very slow. Dijkstra's with a heap runs in O((n+m) log n), which is much faster. For road networks, power grids, and most real applications, all weights are non-negative, so Dijkstra's is both correct AND faster. Choosing Bellman-Ford unnecessarily wastes computation. 📄 pp. 179–184

The principle: Always use the most efficient correct algorithm. If weights are non-negative, Dijkstra's. If weights can be negative (and no negative cycles), Bellman-Ford. 📄 pp. 179–216

Q11. Write pseudocode for Dijkstra's algorithm. Include initialization, the main loop structure, and the relaxation step. Analyze time and space complexity for the naive (no heap) implementation.

▶ Show Answer 
function Dijkstra(G, s):
    # INITIALIZATION
    n = number of vertices in G
    for each vertex i in V:
        Distance[i] = infinity
        Visited[i] = False

    Distance[s] = 0    # source is free to reach

    # MAIN LOOP: n iterations, one vertex settled per iteration
    for iteration = 1 to n:
        # Step 1: Find unvisited vertex with minimum Distance (greedy choice)
        u = None
        for i in V:
            if not Visited[i]:
                if u is None or Distance[i] < Distance[u]:
                    u = i

        if Distance[u] == infinity:
            break    # remaining vertices unreachable

        # Step 2: Settle vertex u
        Visited[u] = True

        # Step 3: Relax all edges from u
        for each neighbor k of u:
            if not Visited[k]:
                new_dist = Distance[u] + weight(u, k)
                if new_dist < Distance[k]:
                    Distance[k] = new_dist   # found shorter path to k!

    return Distance   # Distance[v] = shortest path from s to v
SOURCE: 📄 pp. 172–178

English Explanation:

  1. Initialize all distances to ∞ (unknown). Set Distance[s]=0 (source is free).
  2. Repeat n times: find the closest unvisited vertex u (greedily). Commit its distance.
  3. For each unvisited neighbor k of u, check if going through u is cheaper. If so, update.
  4. After n iterations, all reachable vertices have their correct shortest distances.

📝 Complexity Analysis:

Time: Outer loop runs n times. Finding minimum (Step 1) scans all n vertices = O(n). Relaxing neighbors (Step 3) scans all edges of u; across all iterations total = O(m). Dominant term: n × O(n) = O(n²). 📄 pp. 179–184

Space: Distance[] and Visited[] arrays = O(n). Adjacency list or matrix = O(n+m) or O(n²). Total extra space: O(n). 📄 pp. 179–184

Q12. Write pseudocode for Bellman-Ford. Include the initialization, main relaxation loop, and the optional negative-cycle detection step. Analyze time complexity. Explain in plain English why scanning all edges in each iteration (rather than just edges adjacent to recently updated vertices) is necessary for correctness.

▶ Show Answer 
function BellmanFord(G, s):
    # INITIALIZATION
    n = number of vertices in G
    for each vertex i in V:
        Distance[i] = infinity
    Distance[s] = 0

    # MAIN LOOP: n-1 iterations, relax ALL edges each time
    for iteration = 1 to n-1:
        updated = False              # track if anything changed
        for each edge (j, k) in E:  # scan EVERY edge
            if Distance[j] + weight(j, k) < Distance[k]:
                Distance[k] = Distance[j] + weight(j, k)
                updated = True
        if not updated:
            break                    # early termination: already converged

    # OPTIONAL: Detect negative cycles
    for each edge (j, k) in E:
        if Distance[j] + weight(j, k) < Distance[k]:
            return "NEGATIVE CYCLE DETECTED"

    return Distance
SOURCE: 📄 pp. 210–216

English Explanation:

  1. Initialize: all distances ∞, source = 0.
  2. Repeat n−1 times: go through every edge (j,k) and check if going through j gives a shorter path to k. If yes, update Distance[k].
  3. After n−1 iterations, all shortest simple paths have been found.
  4. Optionally: if any edge can still be relaxed, a negative cycle exists (paths can be made arbitrarily short).

Why scan ALL edges, not just recently updated? Consider a chain s→v₁→v₂→...→v_k where each edge has a large weight. In iteration 1, only v₁'s distance is updated (from ∞ to correct). In iteration 2, only v₂'s distance is updated. If we only relaxed edges adjacent to recently updated vertices, we'd need k iterations just for this chain. By relaxing ALL edges every iteration, we guarantee that after i iterations, all paths of length ≤ i are correctly computed — regardless of which vertices were updated. 📄 pp. 207–216

📝 Complexity Analysis:

Time: Outer loop runs n−1 times. Inner loop scans all m edges each time. Total: O(nm). 📄 pp. 210–216

Space: Distance[] array = O(n). Adjacency list = O(n+m). Total: O(n+m). 📄 pp. 210–216

▶ Show Answer 
function maxProbPath(n, edges, succProb, src, dst):
    # BUILD adjacency list with probabilities
    adj = [[] for each vertex 0..n-1]
    for i, (u, v) in enumerate(edges):
        adj[u].append((v, succProb[i]))
        adj[v].append((u, succProb[i]))   # undirected

    # INITIALIZE probabilities
    Prob = [0.0] * n
    Prob[src] = 1.0    # 100% certain we start at src

    # MAX-HEAP (negate probabilities to use min-heap)
    heap = [(-1.0, src)]   # (-prob, vertex)

    while heap is not empty:
        (neg_prob, u) = heappop(heap)   # extract MAX probability vertex
        prob = -neg_prob

        if u == dst:
            return prob    # found destination!

        # RELAX (multiply probabilities, not add distances)
        for (v, edge_prob) in adj[u]:
            new_prob = prob * edge_prob
            if new_prob > Prob[v]:      # is this path more reliable?
                Prob[v] = new_prob
                heappush(heap, (-new_prob, v))

    return 0.0    # dst not reachable

  • Path 0→2: 0.2
  • Max = 0.25 (via path 0→1→2)

📝 Complexity Analysis:

Time: Each edge processed at most twice (undirected). Each heap push/pop = O(log E). Total: O((V + E) log E). 📄 p. 227

Space: Prob[] array = O(V). Heap = O(E) in worst case. Adjacency list = O(V+E). Total: O(V + E). 📄 p. 227

Why the greedy approach works here: Probabilities p ∈ [0,1]. Multiplying by p ≤ 1 can only DECREASE probability. So once we extract the vertex with maximum probability (the greedy choice), no later path can improve upon it — exactly analogous to Dijkstra's argument with non-negative weights. 📄 p. 227

IN ONE SENTENCE, THIS TOPIC IS:

Shortest-path algorithms find minimum-cost routes in weighted graphs: Dijkstra's is the greedy fast choice for non-negative weights, Bellman-Ford handles negative edges by relaxing all edges n−1 times, and Floyd-Warshall computes all-pairs shortest paths via a DP recurrence. 📄 pp. 169–226

THE THREE THINGS YOU MUST KNOW FOR THE EXAM:

THE CANONICAL ALGORITHM (Dijkstra's):

function Dijkstra(G, s):
    Distance[i] = ∞ for all i; Distance[s] = 0; Visited[i] = F for all i
    repeat n times:
        u = unvisited vertex with minimum Distance     # greedy choice
        Visited[u] = True                              # commit
        for each unvisited neighbor k of u:
            if Distance[u] + weight(u,k) < Distance[k]:
                Distance[k] = Distance[u] + weight(u,k)   # relax
    return Distance
SOURCE: 📄 pp. 172–178

Time: O(n²) naive. Space: O(n). Use heap for O((n+m) log n). Requires non-negative weights.
