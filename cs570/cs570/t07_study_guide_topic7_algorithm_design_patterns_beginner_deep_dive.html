🧗 Warm-up Glossary — Vocabulary You Need Before §1

Algorithm Design Patterns assumes you know some vocabulary that CS 570 uses without always re-defining it. None of these are hard — they just need to be stated plainly before we use them. If you know all of these already, skip to §1.

What is a "contiguous subarray"? 📄 p. 23

An array is a list of values at numbered positions. A contiguous subarray is a slice of consecutive elements — you pick a start index and an end index, and take everything in between, without skipping.

Example: Array = [1, 12, -5, -6, 50, 3]. A contiguous subarray of length 4 starting at index 1 is [12, -5, -6, 50]. You cannot take [1, 50] — those are not consecutive. 📄 p. 23

What does "O(n)" or "Big-O" mean, really? 📄 pp. 39–41

Big-O notation describes how the running time of an algorithm grows as the input size n grows. O(n) means: "if the input doubles in size, the running time roughly doubles too." O(n²) means: "if the input doubles, the running time quadruples." The key insight: we ignore constants and lower-order terms.

Example: An algorithm that does 3n + 100 operations is still O(n) — for large n, the 3n term dominates and the 100 is negligible. 📄 p. 39

What is a "bit"? What does "bit position i" mean? 📄 p. 35

A bit is a single binary digit — either 0 or 1. Every integer stored on a computer is represented as a sequence of bits. For example, the number 4 in binary is 0100 (four bits: position 3 is 0, position 2 is 1, position 1 is 0, position 0 is 0).

Bit position i refers to the i-th digit from the right (0-indexed). The expression (num >> i) & 1 in Python extracts the bit at position i: shift the number right by i places, then check if the last bit is 1.

What is the "Hamming distance" between two integers? 📄 p. 35

The Hamming distance between two integers is the number of bit positions where they differ.

Example: 4 = 0100, 14 = 1110. Compare bit by bit: position 3: 0 vs 1 (different); position 2: 1 vs 1 (same); position 1: 0 vs 1 (different); position 0: 0 vs 0 (same). Total differences = 2. So HammingDistance(4, 14) = 2. 📄 p. 35

What is a "priority queue" (or "heap")? 📄 p. 169

A priority queue is a data structure that always gives you the highest-priority element when you ask for one. In a min-heap, the smallest element comes out first. In a max-heap, the largest comes out first.

Why it matters: Inserting or extracting from a heap takes O(log n) time — much faster than scanning through a list (O(n)). Dijkstra's algorithm uses a min-heap to always process the node with the smallest current distance next. 📄 p. 169

What does "sorting" give you, exactly? 📄 p. 63

After sorting an array, the elements are in order from smallest to largest (ascending) or largest to smallest (descending). The key benefit: once sorted, you can scan once from left to right and be guaranteed you're seeing elements in order. Standard sorting (like Python's sort()) runs in O(n log n) time. 📄 pp. 63–90

What is a "greedy algorithm"? 📄 p. 169

A greedy algorithm makes the locally best choice at each step — it picks the option that looks best right now, without reconsidering past choices. It works only when the locally best choice leads to a globally optimal solution. Dijkstra's algorithm is greedy: at each step, it commits to the node with the smallest tentative distance. 📄 p. 169

What does "modular arithmetic" mean for time problems? 📄 p. ?? verify

When working with circular quantities like times on a clock, we use modular arithmetic. A 24-hour clock has 24 × 60 = 1440 total minutes. After 23:59 comes 00:00, so the circular distance from 23:59 to 00:01 is 2 minutes, not 1438 minutes.

Formula for circular difference: After sorting all times, compare consecutive differences AND the "wrap-around" difference: (first_time + 1440 - last_time). The minimum of all these is the answer. 📄 p. ?? verify

What is an "adjacency list" representation of a graph? 📄 pp. 105–110

An adjacency list stores for each vertex a list of its neighbors and (for weighted graphs) the edge weights. For example, if vertex 0 connects to vertex 1 with probability 0.5, and to vertex 2 with probability 0.2, the adjacency list for vertex 0 is: [(1, 0.5), (2, 0.2)]. 📄 pp. 105–110

What is "path probability" vs "path length"? 📄 p. 169

In a standard shortest-path problem, the cost of a path is the sum of edge weights. In a maximum-probability path problem, the "value" of a path is the product of edge probabilities. For example, if edges have probabilities 0.5 and 0.5, the path probability is 0.5 × 0.5 = 0.25. We want to maximize this product. 📄 p. 169

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 23–35 (Lectures 4–9) + Assignments 4 & 6

1.1 The fundamental problem — why design patterns matter 📄 pp. 23–35

You have been given a programming problem. The brute-force approach (try everything, compare all pairs, use nested loops) gives you a working answer but runs in O(n²) or O(n³) — too slow for large inputs. The challenge is: can you do it in O(n) or O(n log n)?

Algorithm design patterns are reusable strategic templates that transform slow brute-force solutions into fast ones. Think of them like cooking techniques: once you know how to "sauté" or "poach," you don't reinvent the technique for every new ingredient. Once you know the sliding window pattern, you can solve dozens of "find the best contiguous subarray" problems instantly. 📄 pp. 23–35

This topic matters enormously on the CS 570 exam because three of the six 20-point questions in Assignment 4 and three of the six 20-point questions in Assignment 6 are pseudocode design problems. Every question asks for: (1) pseudocode, (2) step-wise English explanation, and (3) time and space complexity. Mastering the core patterns lets you answer all six. 📄 pp. 23–35

1.3 A second framing — the "efficiency multiplier" lens 📄 pp. 23–35

Here's another way to think about design patterns: they are efficiency multipliers on brute force.

Sliding window: The naive approach to "maximum average of subarray of length k" computes the sum of every window from scratch — O(n×k) total. The sliding window avoids recomputing: instead of summing k elements fresh each time, we slide the window by subtracting the element leaving and adding the element entering. That's 2 operations instead of k — an efficiency multiplier of k. The algorithm goes from O(nk) to O(n). 📄 pp. 23–35

Bit manipulation: The naive approach to "sum of Hamming distances between all pairs" checks every pair — O(n²) pairs, each taking O(32) bit comparisons = O(32n²). The bit-counting trick flips the order of thinking: instead of "per pair, count differing bits," we think "per bit position, count how many pairs differ." This collapses the O(n²) into O(32n) = O(n). An efficiency multiplier of n. 📄 pp. 23–35

2a Sliding Window — Maximum Average Subarray Assignment 4 Q7 (20 pts, graded) · 📋 A4 Q7 · Lecture PDF: pp. 23–35

Intuition: The window slides, not restarts 📄 pp. 23–35

Imagine you're looking at an array through a window that shows k elements at once. The brute force approach: each time you move the window one step right, you compute the sum of all k elements inside — k operations per step, n-k+1 steps total → O(nk). That's slow.

The sliding window insight: the new window shares k-1 elements with the old window. So the new sum is just: old sum - (element leaving the left) + (element entering the right). That's 2 operations instead of k — hence O(n) total. 📄 pp. 23–35

Mechanism: The sliding window template 📄 pp. 23–35

Step 1: Initialize the window on the first k elements
Compute the sum of nums[0..k-1]. Save this as both window_sum and max_sum. This is O(k).
Step 2: Slide the window from index k to n-1
For each index i from k to n-1: new window_sum = old window_sum - nums[i-k] + nums[i]. Update max_sum if window_sum > max_sum. Each step is O(1).
Step 3: Return max_sum / k
The maximum average is the maximum sum divided by k. O(1).
def findMaxAverage(nums, k):
    # Step 1: Initialize window on first k elements
    window_sum = 0
    for i in range(k):
        window_sum += nums[i]      # sum nums[0..k-1]
    max_sum = window_sum           # best sum seen so far

    # Step 2: Slide window from index k to n-1
    for i in range(k, len(nums)):
        window_sum = window_sum - nums[i - k] + nums[i]
        #           ^remove left              ^add right
        max_sum = max(max_sum, window_sum)

    # Step 3: Return average
    return max_sum / k

Why didn't we recompute the sum each iteration? The key insight is that the window [12, -5, -6, 50] is just the previous window [1, 12, -5, -6] with 1 removed and 50 added. We used this to avoid recomputing the sum from scratch (saving k-2 = 2 additions per step). For large k, this savings becomes enormous.

2b Sort + Linear Scan — Minimum Time Difference & Arithmetic Progression A4 Q8 (min time diff) + A6 Q8 (AP check) · 📋 A4 Q8 📋 A6 Q8

Intuition: Sorting makes patterns visible 📄 p. 63

The brute-force approach to "find minimum difference between any two times" compares all pairs — O(n²) work. The sorting insight: the minimum difference always occurs between adjacent elements in the sorted order. Once sorted, a single pass from left to right finds the minimum in O(n). So sort first (O(n log n)), then scan (O(n)), total O(n log n). 📄 p. 63

The circular twist for time problems: after sorting, also check the "wrap-around" difference between the last element and the first element (adding 1440 minutes to handle the midnight boundary). 📄 p. 63

Mechanism: Minimum Time Difference (A4 Q8) 📄 p. ?? verify

Step 1: Convert "HH:MM" strings to total minutes
"23:59" → 23×60 + 59 = 1439 minutes. "00:01" → 1 minute. Now we have a list of integers.
Step 2: Sort the minutes list in ascending order
Python's sort() uses Timsort: O(n log n).
Step 3: Compare adjacent pairs
For i from 1 to n-1: diff = minutes[i] - minutes[i-1]. Track the minimum.
Step 4: Handle the circular wrap-around
circular_diff = (minutes[0] + 1440) - minutes[-1]. Compare with current minimum.
Step 5: Return the minimum found

Worked Example: timePoints = ["23:59", "00:00"] 📄 p. ?? verify 📋 A4 Q8

  Input: ["23:59", "00:00"]

  Step 1: Convert to minutes
    "23:59" → 23×60 + 59 = 1439
    "00:00" → 0×60 + 0  = 0
    minutes = [1439, 0]

  Step 2: Sort
    minutes = [0, 1439]

  Step 3: Compare adjacent pairs
    diff = 1439 - 0 = 1439       ← way too big

  Step 4: Circular wrap-around
    circular_diff = (0 + 1440) - 1439 = 1   ← 1 minute from 23:59 to 00:00!

  min_diff = min(1439, 1) = 1

  Output: 1 ✓ (matches expected output)

Worked Example: Arithmetic Progression Check (A6 Q8) — arr = [3, 5, 1] 📄 p. ?? verify 📋 A6 Q8

  Input: arr = [3, 5, 1]

  Step 1: Sort
    arr = [1, 3, 5]

  Step 2: Compute expected common difference
    d = arr[1] - arr[0] = 3 - 1 = 2

  Step 3: Verify all consecutive differences equal d
    arr[2] - arr[1] = 5 - 3 = 2 = d ✓

  Step 4: All differences match d → return True

  Output: True ✓ (elements can be reordered as [1, 3, 5] with diff 2)

def canMakeArithmeticProgression(arr):
    n = len(arr)
    if n < 2:
        return True               # trivially an AP

    arr.sort()                    # O(n log n)
    d = arr[1] - arr[0]           # expected common difference

    for i in range(2, n):         # verify all consecutive diffs
        if arr[i] - arr[i - 1] != d:
            return False          # found a mismatch

    return True                   # all diffs matched

2c Bit Manipulation — Total Hamming Distance Sum Assignment 4 Q9 (20 pts, graded) · 📋 A4 Q9 · Lecture PDF: pp. 35

Intuition: Stop thinking about pairs; think about positions 📄 p. 35

The brute-force approach: for each pair (i, j), compute HammingDistance(nums[i], nums[j]) by comparing all 32 bits. That's O(n²) pairs × O(32) per pair = O(32n²) = O(n²). Too slow for large n.

The bit-position insight: instead of thinking "how many bits differ between these two specific numbers?", think "for bit position 2, how many pairs of numbers differ at that position?"

Key formula: If p numbers have a 1 at bit position i, and q = n - p numbers have a 0 at bit position i, then exactly p × q pairs differ at that position (one number from the 1-group, one from the 0-group). Sum p × q over all 32 bit positions to get the total Hamming distance sum. This is O(32n) = O(n). 📄 p. 35

Mechanism: Bit-position counting 📄 p. 35

Step 1: Initialize total = 0
Step 2: For each bit position i from 0 to 31
Count p = how many numbers in nums have a 1 at bit position i. Compute q = n - p (numbers with a 0 at position i). Add p × q to total.
Step 3: Return total
def totalHammingDistance(nums):
    total = 0
    n = len(nums)

    for bit in range(32):         # for each of the 32 bit positions
        p = 0
        for num in nums:
            if (num >> bit) & 1:  # is bit 'bit' set to 1 in num?
                p += 1
        q = n - p                 # numbers with 0 at this position
        total += p * q            # pairs that differ at this bit position

    return total

2d Modified Dijkstra — Maximum Probability Path Assignment 6 Q9 (20 pts, graded) · 📋 A6 Q9 · Lecture PDF: pp. 169–178

Intuition: Same structure, different objective 📄 pp. 169–178

Standard Dijkstra's algorithm finds the shortest path — it minimizes the sum of edge weights. But what if edge weights are probabilities, and we want to maximize the probability of reaching the destination?

Path probability works differently from path length: it's a product (not a sum). If edge (0→1) has probability 0.5 and edge (1→2) has probability 0.5, the path 0→1→2 has probability 0.5 × 0.5 = 0.25. 📄 p. 169

The modified Dijkstra makes two key changes from the standard version: (1) Replace "minimize sum" with "maximize product" (update rule: prob[v] = max(prob[v], prob[u] × edge_prob)). (2) Use a max-heap (priority queue sorted by highest probability) instead of a min-heap. Everything else is identical to standard Dijkstra! 📄 pp. 169–178

Mechanism: Modified Dijkstra for max probability 📄 pp. 169–178

Step 1: Build the adjacency list with edge probabilities
For each edge [u, v] with probability p: add (v, p) to graph[u] and (u, p) to graph[v] (undirected graph).
Step 2: Initialize prob[] array to 0 for all nodes; set prob[start] = 1.0
prob[i] represents the maximum probability of reaching node i from start. We start with probability 1.0 at the source (we're definitely there).
Step 3: Push (1.0, start) onto the max-heap
Python's heapq is a min-heap, so we negate probabilities: push (-1.0, start) and treat the most negative as the largest.
Step 4: While heap is not empty: pop the node with maximum probability
Extract (neg_p, u). curr_prob = -neg_p. If curr_prob < prob[u], this is a stale entry — skip it.
Step 5: Relax all neighbors
For each (v, edge_prob) in graph[u]: new_prob = curr_prob × edge_prob. If new_prob > prob[v]: update prob[v] = new_prob, push (-new_prob, v) to heap.
Step 6: Return prob[end]
If end was never reached, prob[end] remains 0.0.
def maxProbability(n, edges, succProb, start, end):
    # Build adjacency list
    graph = defaultdict(list)
    for i in range(len(edges)):
        u, v = edges[i][0], edges[i][1]
        p = succProb[i]
        graph[u].append((v, p))
        graph[v].append((u, p))        # undirected

    # Initialize probabilities
    prob = [0.0] * n
    prob[start] = 1.0

    # Max-heap via negation (Python heapq is min-heap)
    max_heap = [(-1.0, start)]

    while max_heap:
        neg_p, u = heappop(max_heap)
        curr_prob = -neg_p

        if curr_prob < prob[u]:        # stale entry: skip
            continue

        if u == end:
            return curr_prob           # found best path to end!

        for (v, edge_prob) in graph[u]:
            new_prob = curr_prob * edge_prob
            if new_prob > prob[v]:
                prob[v] = new_prob
                heappush(max_heap, (-new_prob, v))

    return prob[end]                   # 0.0 if end unreachable

Misconception Cards 📄 pp. 23–35, 169–178

❌ Misconception: "Sliding window only works for subarrays with positive numbers."

✅ Reality: Sliding window works for any fixed-size contiguous subarray regardless of sign. The example in A4 Q7 contains -5 and -6, and the algorithm still correctly finds the window [12,-5,-6,50] with sum 51. 📄 pp. 23–35

❌ Misconception: "For Hamming distance, I need to compare all pairs."

✅ Reality: The bit-position counting technique avoids all pairwise comparison. For each of the 32 bit positions, count ones (p) and zeros (q=n-p); add p×q to the total. This is O(n), not O(n²). 📄 p. 35

❌ Misconception: "Modified Dijkstra adds probabilities just like standard Dijkstra adds distances."

✅ Reality: Probabilities are multiplied, not added. The path probability is the PRODUCT of edge probabilities along the path. This changes the update rule from dist[v] = min(dist[v], dist[u]+w) to prob[v] = max(prob[v], prob[u]*p). 📄 pp. 169–178

❌ Misconception: "Space complexity O(1) means the algorithm uses no memory."

✅ Reality: O(1) space means a constant amount of extra memory, regardless of input size. The input itself (the array) doesn't count. Variables like window_sum, max_sum, and loop counters are O(1) extra space. 📄 pp. 23–35

❌ Misconception: "The star graph center is always the first node in the edge list."

✅ Reality: The center is the node that appears in BOTH edges[0] and edges[1]. Check both endpoints of edges[0] against both endpoints of edges[1]. The common node is the center. It may be at position [0] or [1] in either edge. 📄 p. ?? verify

❌ Misconception: "O(n log n) is always slower than O(n), so sorting should be avoided."

✅ Reality: O(n log n) is slower than O(n) asymptotically, but for many problems (like min time difference or AP check), sorting is the KEY that enables the O(n) scan afterward. Without sorting, the scan alone would be O(n²). Sorting + O(n) scan = O(n log n) overall, which is MUCH better than O(n²). 📄 p. 63

🧠 Mnemonic 1: Sliding Window — "Subtract Left, Add Right" Whenever you slide the window one step right: you subtract the element that just left the window on the left (nums[i-k]) and add the element that just entered on the right (nums[i]). WC-5. Your classmate says: "Modified Dijkstra for max probability doesn't need a max-heap — I can use a min-heap with regular (positive) probabilities and just look for the node with the largest prob array value instead of popping from the heap." Explain why this approach fails.

▶ Show Explanation

Why it fails: The efficiency of Dijkstra's algorithm depends on the priority queue always giving you the next node to process in the correct order. If you use a min-heap with positive probabilities, the smallest probability comes out first — but we want the LARGEST probability (most likely path so far). Processing nodes in wrong order breaks the greedy correctness argument. 📄 pp. 169–178

Q3. In the modified Dijkstra algorithm for maximum probability path, what is the correct update condition for relaxing edge (u, v) with probability p?

  1. if prob[u] + p > prob[v]: update prob[v] = prob[u] + p
  2. if prob[u] × p > prob[v]: update prob[v] = prob[u] × p
  3. if prob[u] + p < prob[v]: update prob[v] = prob[u] + p
  4. if prob[u] × p < prob[v]: update prob[v] = prob[u] × p
▶ Show Answer

✅ Correct Answer: B

Probabilities multiply along a path (prob = product of edge probs). We want to maximize the path probability, so we update when the new path is BETTER (higher probability). The update: if prob[u] × p > prob[v]: prob[v] = prob[u] × p. 📄 pp. 169–178

Why A/C are wrong: addition is for standard Dijkstra shortest paths — probabilities multiply, they don't add.

Why D is wrong: we want to maximize (update when new is LARGER), not minimize (update when new is smaller).

Exam Tip: The two changes from standard Dijkstra: × instead of +, and > instead of < in the update condition. 📄 pp. 169–178

Q4. Which algorithm achieves the best asymptotic time complexity for finding the minimum difference between any two times in a list of m time points in "HH:MM" format?

Q5. Claim: "The sliding window algorithm for finding the maximum average subarray of length k is correct — it never misses the optimal window." Prove this claim.

▶ Show Answer (Model Proof)

✅ Answer: TRUE — Prove by exhaustive case analysis.

Direct Proof: Sliding Window Correctness
Step 1 [What we're proving]. The algorithm examines every contiguous subarray of length k exactly once and tracks the maximum sum seen. 📄 pp. 23–35
Step 2 [Windows examined]. The initialization computes the sum of window [0..k-1]. The loop at i=k computes [1..k], at i=k+1 computes [2..k+1], ..., at i=n-1 computes [n-k..n-1]. This is every window of length k starting at indices 0, 1, 2, ..., n-k. Together these are ALL n-k+1 windows of length k. 📄 pp. 23–35
Step 3 [Sum correctness]. We prove by induction that window_sum at each step i equals sum(nums[i-k..i-1]). Base case: after initialization, window_sum = sum(nums[0..k-1]) ✓. Inductive step: window_sum at step i = (window_sum at step i-1) - nums[i-k] + nums[i] = sum(nums[i-k-1..i-2]) - nums[i-k-1] + ... (the algebra confirms the telescoping update maintains the invariant). ✓ 📄 pp. 23–35
Step 4 [Conclusion]. Since we examine every window and correctly compute its sum, max_sum correctly holds the maximum sum over all windows. Dividing by k gives the maximum average. ∎ 📄 pp. 23–35

Q6. Claim: "For p numbers having a 1 at bit position i, and q = n-p numbers having a 0, exactly p×q pairs differ at bit position i." Prove this claim.

▶ Show Answer (Model Proof)

✅ Answer: TRUE — Direct counting proof.

Direct Proof: p × q Pairs Formula
Step 1 [Setup]. Partition the n numbers into two groups: Group A = {numbers with a 1 at bit position i}, |A| = p. Group B = {numbers with a 0 at bit position i}, |B| = q = n-p. 📄 p. 35
Step 2 [Which pairs differ?] A pair (x, y) differs at bit position i if and only if one is in Group A and the other is in Group B. Proof: if both are in A (both have bit 1) → they agree at position i; if both are in B (both have bit 0) → they agree; if one is in A and one in B → they disagree. 📄 p. 35
Step 3 [Count such pairs]. The number of pairs (x, y) where x ∈ A and y ∈ B is |A| × |B| = p × q. (For each of the p numbers in A, we can pair it with each of the q numbers in B — giving p × q distinct pairs.) 📄 p. 35
Step 4 [Conclusion]. Therefore, exactly p × q pairs of numbers differ at bit position i. Summing over all 32 bit positions gives the total Hamming distance sum. ∎ 📄 p. 35

🔑 Grading Checklist:

  • ☐ Defined Group A (1s) and Group B (0s) and their sizes p and q. 📄 p. 35
  • ☐ Showed pairs in the same group agree; pairs across groups disagree. 📄 p. 35
  • ☐ Counted cross-group pairs as p × q. 📄 p. 35

Q7. Claim: "The modified Dijkstra algorithm (for maximum probability path) is correct: when a node u is popped from the max-heap with probability curr_prob, curr_prob equals the true maximum probability path from start to u." Prove this claim by induction on the order of extraction from the max-heap.

▶ Show Answer (Model Proof)

✅ Answer: TRUE — Proof by induction on extraction order.

Inductive Proof: Modified Dijkstra Correctness
Base case [First extraction]. The first node extracted is start with probability 1.0. Since prob[start] = 1.0 (we're definitely there), this is the correct maximum probability to start. ✓ 📄 pp. 169–178
Inductive hypothesis (IH). Assume that for all nodes extracted from the max-heap so far, their extracted probability equals the true maximum path probability from start. 📄 pp. 169–178
Inductive step. Let u be the next node extracted with probability curr_prob = prob[u]. We claim curr_prob is the true maximum path probability to u. Suppose not — suppose there's a better path via some unextracted node v with probability prob_better > prob[u]. But prob_better ≤ prob[v] (v hasn't been extracted, so its probability could only increase when we extract it). Since we extract in max order, prob[u] ≥ prob[v] (u was extracted before v). So prob_better ≤ prob[v] ≤ prob[u]. Contradiction. ✓ 📄 pp. 169–178
Key ingredient [Edge probabilities ≤ 1]. This proof holds because probabilities are in (0,1]. Since edge probs ≤ 1, multiplying along any path never increases the probability. Hence any path through an unextracted node v (with prob[v] ≤ prob[u]) cannot yield a better path to u than prob[u]. ✓ 📄 pp. 169–178
Conclusion. By induction, every extracted node has the correct maximum probability. ∎ 📄 pp. 169–178

🔑 Grading Checklist:

  • ☐ Base case: start node extracted with probability 1.0 (correct). 📄 p. 169
  • ☐ IH stated for all previously extracted nodes. 📄 p. 169
  • ☐ Key argument: any better path must go through an unextracted node, but extraction order ensures prob[u] ≥ prob[v] for unextracted v. 📄 pp. 169–178
  • ☐ Critical note: requires edge probabilities ≤ 1 (so paths cannot improve by adding more edges). 📄 pp. 169–178

Q11. Write pseudocode to find the center node of a star graph. Input: edges (a 2D array where edges[i] = [u, v] is an undirected edge). The center appears in every edge. Provide English explanation and complexity analysis.

▶ Show Answer

✅ Pseudocode:

def findCenter(edges):
    # Key observation: the center appears in BOTH edges[0] and edges[1].
    # Leaves appear in only one edge.
    u0, v0 = edges[0][0], edges[0][1]    # endpoints of first edge
    u1, v1 = edges[1][0], edges[1][1]    # endpoints of second edge

    if u0 == u1 or u0 == v1:
        return u0      # u0 is common to both edges
    else:
        return v0      # v0 must be the common node
SOURCE: 📄 p. ?? verify (A6 Q7 solution)

English Explanation:

  1. Key observation: In a star graph, every edge connects the center to a leaf. The center appears in every edge; a leaf appears in exactly one edge.
  2. Check first two edges: The center must appear in both edges[0] = [u0, v0] and edges[1] = [u1, v1]. So either u0 or v0 is the center — whichever also appears in edges[1].
  3. Two comparisons: If u0 == u1 or u0 == v1, then u0 is the center. Otherwise v0 is the center.
  4. Return the center.

📝 Complexity: Time = O(1) — only 2 comparisons regardless of n. Space = O(1) — only 4 variables. 📄 p. ?? verify

Q12. Write pseudocode to check if an array can be rearranged to form an arithmetic progression. Return True or False. Input: arr = [1, 5, 3, 7]. Provide English explanation and complexity analysis.

▶ Show Answer

✅ Pseudocode:

def canMakeArithmeticProgression(arr):
    n = len(arr)
    if n < 2:
        return True             # trivially an AP with 0 or 1 elements

    arr.sort()                  # sort ascending: O(n log n)

    # Compute expected common difference
    d = arr[1] - arr[0]

    # Verify all consecutive differences equal d
    for i in range(2, n):       # n-2 iterations
        if arr[i] - arr[i - 1] != d:
            return False        # mismatch found

    return True                 # all differences match
SOURCE: 📄 p. ?? verify (A6 Q8 solution)

English Explanation:

  1. Handle trivial case: Arrays of size 0 or 1 are trivially arithmetic progressions.
  2. Sort: In an AP, elements must appear in order with a constant gap. Sorting places them in the right order to check this.
  3. Compute d: The expected common difference is d = arr[1] - arr[0] after sorting.
  4. Verify: For each consecutive pair from index 1 to n-1, check if the difference equals d. If any pair differs, return False.
  5. Return True if all pairs have the same difference d.

Trace on [1, 5, 3, 7]: Sort → [1, 3, 5, 7]. d = 3-1 = 2. Check: 5-3=2 ✓, 7-5=2 ✓. Return True. 📄 p. ?? verify

📝 Complexity: Time = O(n log n) (sort dominates; scan is O(n)). Space = O(1) auxiliary (in-place sort). 📄 p. 63

Q13. A "maximum reliability window" problem: you have n network nodes. Each node i has reliability r[i] ∈ [0,1]. You want to find the contiguous window of k nodes that maximizes the PRODUCT of reliabilities (not the sum). Write pseudocode, provide English explanation, and analyze complexity.

▶ Show Answer

✅ Key Insight: We cannot use the standard sliding window (which uses additive increment: new = old - left + right). Products don't support this simple increment — dividing by the departing element works only if all values are non-zero. For general r[i] ∈ [0, 1], we use log-transform: since log(a×b) = log(a) + log(b), transform to logs and then sliding window works additively! Or, since values are in [0,1] and can be zero, we explicitly compute the product of each window.

def maxReliabilityWindow(r, k):
    n = len(r)
    if n == 0 or k > n:
        return 0.0

    # Option A: Direct product (works even with zeros)
    # Initialize product of first window
    window_prod = 1.0
    for i in range(k):
        window_prod *= r[i]         # O(k)
    max_prod = window_prod

    # Slide window: divide by departing, multiply by entering
    # (Only safe if no zeros; else use Option B)
    for i in range(k, n):
        if r[i - k] != 0:
            window_prod = window_prod / r[i - k] * r[i]   # O(1)
        else:
            # Recompute from scratch if zero leaves
            window_prod = 1.0
            for j in range(i - k + 1, i + 1):
                window_prod *= r[j]                        # O(k) — rare
        max_prod = max(max_prod, window_prod)

    return max_prod

# Option B (simpler, always correct): log-transform if no zeros allowed
def maxReliabilityWindowLog(r, k):
    import math
    logs = [math.log(ri) for ri in r]    # transform to log space
    # Now use standard sliding window on logs (sum = log of product)
    window_log_sum = sum(logs[:k])        # O(k)
    max_log_sum = window_log_sum
    for i in range(k, len(r)):
        window_log_sum = window_log_sum - logs[i-k] + logs[i]   # O(1)
        max_log_sum = max(max_log_sum, window_log_sum)
    return math.exp(max_log_sum)          # convert back
SOURCE: Integrates 📄 pp. 23–35 (sliding window) + 📄 p. 169 (product along path)

English Explanation (Option B — log transform):

  1. Transform: compute log of each reliability. log(product) = sum of logs — so maximizing the product is equivalent to maximizing the sum of logs.
  2. Sliding window on log values: initialize on first k logs, slide by subtracting departing log and adding arriving log. O(n).
  3. Transform back: exp(max_log_sum) gives the maximum product. O(1).

📝 Complexity (Option B): Time = O(n) (log transform O(n) + sliding window O(n) + exp O(1)). Space = O(n) for the logs array. 📄 pp. 23–35

Cross-topic insight: The log trick appears in the modified Dijkstra too — maximizing probability products can be converted to maximizing sum of log-probabilities, allowing standard min-cost Dijkstra (negated logs = min-sum = max probability). 📄 pp. 169–178
