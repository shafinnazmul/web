🧗 Warm-up Glossary — Vocabulary You Need Before §1

Dynamic programming and DAG algorithms build on several foundational ideas that lectures often use without re-introducing. If every term below is already crystal-clear, skip to §1. Otherwise, read each entry carefully — these are the prerequisites.

What is a directed graph? 📄 p. 105

A directed graph (also called a digraph) G = (V, E) has vertices (circles) connected by directed edges (arrows). An arrow from A to B means "you can travel A → B" but NOT automatically B → A. This is different from an undirected graph where edges have no direction. 

  Directed graph example:

     A ──▶ B ──▶ C
     │           │
     ▼           ▼
     D ──────▶  E

  Edge (A,B) exists — travel from A to B is allowed.
  Edge (B,A) does NOT exist — you cannot go backward.
  This directionality is what makes DAGs useful for modeling dependencies.
SOURCE: 📄 p. 105

What is a cycle in a graph? 📄 p. 141

A cycle is a path that starts and ends at the same vertex. In a directed graph: a directed cycle means you can follow arrows and return to where you started. 

  Cycle:      A ──▶ B ──▶ C ──▶ A    (you can go round and round forever)

  No cycle:   A ──▶ B ──▶ C          (no arrow leads back to A or B)
SOURCE: 📄 p. 141

What is a DAG? 📄 pp. 141–156

A DAG (Directed Acyclic Graph) is a directed graph with no directed cycles. The key idea: you can always move "forward" in a DAG — you can never loop back to where you were. This makes DAGs perfect for representing dependencies (if A must happen before B, draw an arrow A → B).

Concrete example: University course prerequisites. Algebra must come before Calculus. Calculus before Differential Equations. You can never take Algebra "after" Differential Equations — there's no cycle. 📄 p. 141

What is "indegree" and "outdegree"? 📄 pp. 141–156

The indegree of a vertex is the number of arrows pointing INTO it (incoming edges). The outdegree is the number of arrows pointing OUT OF it. 

  In the graph:  A ──▶ B ──▶ C
                        │
                        ▼
                        D

  Indegree of A = 0 (no arrows come into A)
  Indegree of B = 1 (only A points to B)
  Indegree of C = 1 (only B points to C)
  Indegree of D = 1 (only B points to D)
  Outdegree of B = 2 (B points to C and D)
SOURCE: 📄 pp. 141–156

A vertex with indegree 0 has no dependencies — it can be done first. This is the key to topological sorting. 📄 pp. 141–156

What is recursion? 📄 p. 63

Recursion is when a function calls itself with a smaller input. The function gets simpler and simpler until it hits a base case — the simplest possible input where the answer is known directly (no further recursion needed).

Example: factorial(4) = 4 × factorial(3) = 4 × 3 × factorial(2) = 4 × 3 × 2 × factorial(1) = 4 × 3 × 2 × 1 = 24. The base case is factorial(1) = 1. 📄 p. 63

What are "overlapping subproblems"? 📄 pp. 207–226

When solving a problem recursively, you sometimes compute the same smaller sub-problem multiple times. This redundancy is wasteful. For example, when computing Fibonacci(5): Fibonacci(3) gets computed twice — once for Fibonacci(5) and once for Fibonacci(4). 

  fib(5)
  ├─ fib(4)
  │   ├─ fib(3)          ← computed here
  │   │   ├─ fib(2)
  │   │   └─ fib(1)
  │   └─ fib(2)
  └─ fib(3)              ← computed AGAIN — redundant!
      ├─ fib(2)
      └─ fib(1)
SOURCE: 📄 pp. 207–226

What is memoization? 📄 pp. 207–226

Memoization (note: memo, not memori-z-ation) is the technique of storing the result of a sub-problem the first time you compute it, so you never re-compute it. It turns exponential recursion into polynomial-time computation.

Analogy: Imagine you solve a crossword clue ("Capital of France") and write "Paris" in the margin. Next time you see the same clue, you look at your margin note instead of thinking again. Memoization is your "margin note" system for recursive functions.

What is "optimal substructure"? 📄 pp. 207–226

A problem has optimal substructure if: the optimal solution to the whole problem contains optimal solutions to its sub-problems.

Example: The shortest path from city A to city C via city B is the shortest path A→B plus the shortest path B→C. You can't have a "shortcut" that ignores the optimal sub-paths. 📄 pp. 207–226

What is "path relaxation" in shortest path algorithms? 📄 pp. 207–216

Relaxation is the operation: "If I can reach vertex k through vertex j and the total distance would be less than what I currently know, update my estimate." 

  Current estimate:  dist[C] = 10
  We discover:       dist[A] = 3,  weight(A → C) = 4
  Relaxation:        3 + 4 = 7 < 10, so dist[C] = 7  ← UPDATE!

  This is the "relax(A, C)" operation. 📄 pp. 207–210
SOURCE: 📄 pp. 207–210

What is a weight matrix W[i][j]? 📄 pp. 217–226

A weight matrix W is a 2D array (n × n for n vertices) where W[i][j] is the weight of the edge from vertex i to vertex j. If no edge exists: W[i][j] = ∞ (or a very large number). If i = j: W[i][j] = 0 (no cost to stay at the same vertex). 

  Graph:  1 ──3──▶ 2
          │         │
          5         2
          ▼         ▼
          3 ──1──▶ 4

  Weight matrix W:
       1    2    3    4
  1  [ 0    3    5    ∞ ]
  2  [ ∞    0    ∞    2 ]
  3  [ ∞    ∞    0    1 ]
  4  [ ∞    ∞    ∞    0 ]
SOURCE: 📄 pp. 217–226

What does "bottom-up" vs "top-down" mean in DP? 📄 pp. 207–226

Top-down DP: Start with the big problem, recurse down to smaller problems, and use memoization to avoid recomputation. You solve sub-problems on-demand.

Bottom-up DP: Start with the smallest base cases and iteratively build up to the big problem. You fill in a table from left-to-right (or bottom-to-top), ensuring sub-problem answers are available before they're needed.

Both produce the same answer. Bottom-up is often faster in practice because it avoids function-call overhead. 📄 pp. 207–226

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 141–168, 207–226 (Lectures 15–16, 21–22)

1.1 The fundamental problem — how do we handle dependencies efficiently? 📄 pp. 141–156

Imagine you're planning a trip abroad. You need to: get a passport, buy a ticket, apply for a visa (requires ticket), get travel insurance (requires passport), buy foreign exchange (requires visa). You can't buy a ticket before getting a passport (some airlines require it). You can't get a visa without a ticket. These are dependencies.

The question is: what order do you do tasks in? And a harder question: what is the minimum number of days if some tasks take one day each, but independent tasks can be done in parallel?

This is exactly what Directed Acyclic Graphs (DAGs) model, and topological sorting + longest path answer. 📄 pp. 141–156 

  Trip planning as a DAG:

         [passport] ────▶ [ticket] ────▶ [visa] ────▶ [forex]
               │                │
               └──▶ [insurance]  └──▶ [insurance]

  Arrow A → B means: "A must come before B"
  No cycles (you can't need a visa to get a passport to get a visa)

  Topological sort gives one valid ordering:
    passport, insurance, ticket, visa, forex

  Longest path (in terms of tasks) tells you: how many STEPS minimum?
    passport(1) → ticket(2) → visa(3) → forex(4)  = 4 steps minimum
SOURCE: 📄 pp. 141–156

Now for a completely different-looking problem: we want the shortest path between every pair of cities in a country's road network. Bellman-Ford and Floyd-Warshall use dynamic programming — the same "build up from sub-problems" idea — to solve this efficiently. 📄 pp. 207–226

1.3 Dynamic programming — the "smart recursion" metaphor 📄 pp. 207–226

Imagine you're a delivery driver who must find the fastest route between every pair of cities in a country. The naive approach: for each pair (A, B), search exhaustively — but there are n² pairs and exponentially many paths. The smart approach: build up systematically.

The DP insight for Floyd-Warshall: Start by allowing only city 1 as an intermediate stop. Then expand to allow city 1 or 2. Then city 1, 2, or 3. At each step, the new answer is either: (a) the old answer (city k isn't useful on this path), or (b) go through city k (combine path i→k with path k→j, both already computed). 

  Floyd-Warshall's expanding "allowed intermediate" idea:

  Step k=0:  Only direct edges.  W⁰[i][j] = direct edge weight (or ∞)
  Step k=1:  Can pass through city 1.  W¹[i][j] = min(W⁰[i][j], W⁰[i][1] + W⁰[1][j])
  Step k=2:  Can pass through cities 1 or 2.  W²[i][j] = min(W¹[i][j], W¹[i][2] + W¹[2][j])
  ...
  Step k=n:  All cities allowed as intermediates. Final answer!

  "Each step, we ask: does going THROUGH city k improve any path i→j?"
SOURCE: 📄 pp. 217–226

The DP insight for Bellman-Ford: Shortest paths have at most n−1 edges (no loops in a shortest path). So iterate: after iteration k, we know the shortest path using at most k edges. After n−1 iterations, we have all shortest paths. 📄 pp. 207–210

📌 Unifying Principle Every algorithm in Topic 9 follows the same DP recipe: (1) Define what a "sub-problem" is (smaller graph, fewer edges allowed, fewer vertices allowed). (2) Write a recurrence: "the answer to this sub-problem in terms of slightly smaller sub-problems." (3) Solve in the RIGHT ORDER — so that when you compute a sub-problem, all the sub-problems it depends on are already done. 📄 pp. 141–226

1.4 Technical definitions (using lecture notation) 📄 pp. 141–226

DAG: A directed graph G = (V, E) with no directed cycles. 📄 pp. 141–156

Topological ordering: An enumeration of the vertices of a DAG as {i₁, i₂, ..., iₙ} such that for any edge (j, k) ∈ E, j appears before k in the enumeration. Equivalently: all dependencies come before their dependents. 📄 pp. 141–156

Longest path to vertex v (LPT[v]): If indegree(v) = 0, then LPT[v] = 0. Otherwise, LPT[v] = 1 + max{ LPT[j] : (j,v) ∈ E }. Compute in topological order so all predecessors j are computed before v. 📄 pp. 157–168

Bellman-Ford update rule: For each edge (j,k): Distance[k] = min(Distance[k], Distance[j] + weight(j,k)). Repeat for all edges n−1 times. Initialize Distance[source] = 0, Distance[others] = ∞. 📄 pp. 207–216

Floyd-Warshall recurrence: W⁰[i][j] = direct edge weight (∞ if no edge). For k = 1 to n: W^k[i][j] = min(W^(k-1)[i][j], W^(k-1)[i][k] + W^(k-1)[k][j]). Final answer in W^n. 📄 pp. 217–226

⚠️ Common Point of Confusion — Know This Before §2 Longest path in a DAG (Topic 9) is NOT the same as shortest path. And it is NOT computationally hard in a DAG — it can be computed in O(n+m) using topological sort. But longest path in a general graph (with cycles allowed) IS NP-hard (the Hamiltonian path problem). The key distinction: DAG = tractable; general graph = intractable. 📄 pp. 157–168

2a Topological Sort — Ordering Tasks with Dependencies Lecture PDF: pp. 141–156 (Lectures 15) · Lec 16 Python session

Intuition: Find who has no prerequisites, do them first 📄 pp. 141–156

Think about making breakfast: you can't eat toast before you toast the bread; you can't toast bread before you put it in the toaster; you can't put it in the toaster before you plug in the toaster. But you CAN make coffee in parallel with all of this, because coffee has no dependency on toast.

Topological sort finds a valid ordering where all dependencies come before their dependents. The algorithm is elegant: repeatedly find a vertex with no remaining prerequisites (indegree 0), output it, then remove it (decrementing the indegrees of its successors). 📄 pp. 141–156

Mechanism: Two versions of topological sort 📄 pp. 141–156

# VERSION 1: O(n²) — simple, uses adjacency matrix
function TopologicalSort_Simple(G):
    compute indegree[i] for all i        # scan adjacency matrix, O(n²)
    for each step from 1 to n:
        choose j with indegree[j] == 0   # find next vertex, O(n)
        output j
        indegree[j] = -1                 # mark as done
        for each neighbor k of j:
            indegree[k] -= 1             # update successors, O(n)
    # Total: O(n²)

# VERSION 2: O(n+m) — efficient, uses queue + adjacency list
function TopologicalSort_Fast(G):
    compute indegree[v] for all v        # scan adjacency lists, O(n+m)
    Queue = all vertices with indegree 0 # initialize queue
    result = []
    while Queue is not empty:
        v = remove_front(Queue)          # dequeue a ready vertex
        result.append(v)                 # output it
        for each neighbor w of v:        # update successors
            indegree[w] -= 1
            if indegree[w] == 0:
                Queue.add(w)             # w is now ready
    return result                        # total: O(n+m)
SOURCE: 📄 pp. 141–156

⚠️ Key Point: Topological Sort is NOT Unique Multiple valid topological orderings can exist for the same DAG. In our example, A and F can be swapped. Any ordering where all edges (u,v) have u appearing before v is valid. The exam may ask you to verify an ordering is valid or find ONE valid ordering — both are fine. 📄 pp. 141–156
⚠️ What if the DAG has a cycle? If the graph has a cycle, topological sort will fail — the queue will become empty before all vertices are output. This is the detection method! If |output| < |V|, a cycle exists. 📄 pp. 141–156
2b Longest Path in a DAG — Minimum Semesters to Graduate Lecture PDF: pp. 157–168 (Lecture 16)

Intuition: Group courses into semesters — how many minimum? 📄 pp. 157–168

Imagine you're planning which semester to take each course. CS101 must come before CS201, which must come before CS301. This is a chain of length 3 — you need at least 3 semesters. The longest chain of dependencies determines the minimum number of sequential steps.

The longest path in the DAG IS the answer to "minimum time to completion assuming parallel tasks." Every vertex on the longest path is a bottleneck — it cannot be parallelized. 📄 pp. 157–168

Mechanism: DP in topological order 📄 pp. 157–168

The recurrence is beautifully simple:

function LongestPath(G):
    # Assumes G is a DAG; compute topological order first
    topo_order = TopologicalSort_Fast(G)

    # Initialize: all vertices start with LPT = 0
    for each vertex v:
        LPT[v] = 0

    # Process vertices in topological order (left to right)
    for each vertex v in topo_order:
        # LPT[v] is already set from when predecessors were processed
        for each successor w of v:              # for each outgoing edge (v,w)
            if LPT[v] + 1 > LPT[w]:
                LPT[w] = LPT[v] + 1            # relax: can we reach w in more steps via v?

    # The answer is the maximum LPT across all vertices
    return max(LPT[v] for all v)
SOURCE: 📄 pp. 157–168

Why must we process in topological order? Because when we compute LPT[v], we need LPT[j] for ALL predecessors j of v to already be correct. Topological order guarantees that ALL edges (j,v) have j appearing before v — so all predecessors are processed before v. 📄 pp. 157–168

2c Floyd-Warshall — All-Pairs Shortest Paths via DP Lecture PDF: pp. 217–226 (Lecture 22)

Intuition: Progressively allow more intermediate cities 📄 pp. 217–226

Imagine you're planning road trips between 4 cities {1, 2, 3, 4}. Initially, you only know direct roads. Then you realize: "If I'm allowed to stop in city 1, does that create any shortcuts?" Then: "If I'm allowed to stop in city 1 OR 2, any more shortcuts?" At each step k, you check: "Does going THROUGH city k improve any path from i to j?" 

  Floyd-Warshall's core question at each step k:

  "Is  i ──▶ k ──▶ j  shorter than the current known i ──▶ j?"

  If W^(k-1)[i][k] + W^(k-1)[k][j] < W^(k-1)[i][j]:
      W^k[i][j] = W^(k-1)[i][k] + W^(k-1)[k][j]    ← use city k as intermediate
  Else:
      W^k[i][j] = W^(k-1)[i][j]                     ← skip city k

  This is the DP recurrence. Two cases. That's it!
SOURCE: 📄 pp. 217–226

Mechanism: Floyd-Warshall pseudocode 📄 pp. 217–226

function FloydWarshall(W, n):
    # W is the n×n weight matrix: W[i][j] = edge weight (∞ if no edge), W[i][i] = 0
    # We work in-place using a rolling update (only need previous W^(k-1))

    D = copy(W)          # D[i][j] will hold the current shortest path estimate

    for k = 1 to n:                          # try each vertex as an intermediate
        for i = 1 to n:                      # from each source
            for j = 1 to n:                  # to each destination
                if D[i][k] + D[k][j] < D[i][j]:
                    D[i][j] = D[i][k] + D[k][j]   # found a shorter path via k

    return D             # D[i][j] = shortest path from i to j

# Complexity: O(n³) — three nested loops of size n
# Space: O(n²) — just the n×n matrix D (rolling update, no extra k-dimension)

2d Bellman-Ford as DP — Single-Source Shortest Paths with Negative Weights Lecture PDF: pp. 207–216 (Lecture 21)

Intuition: Build shortest paths edge by edge 📄 pp. 207–216

Dijkstra's algorithm "burns" vertices — once settled, never revisited. This works only with non-negative weights. With negative edges, a "settled" vertex might have a shorter path discovered later via a negative edge. Bellman-Ford takes a different approach: don't commit to any vertex being final. Instead, repeat the relaxation process.

The key observation: In a graph with n vertices, any shortest path (without negative cycles) uses at most n−1 edges (visiting each vertex at most once). So iterate n−1 times, each time relaxing all edges. After k iterations, we know all shortest paths of length ≤ k. 📄 pp. 207–210 

  Why n−1 iterations suffice:

  Any shortest path in a graph with n vertices visits at most n different vertices.
  That means it has at most n−1 edges (edges = vertices − 1 for a simple path).

  After iteration 1: we know all 1-edge shortest paths.
  After iteration 2: we know all 2-edge shortest paths.
  ...
  After iteration n−1: we know all (n−1)-edge shortest paths.
  = ALL shortest paths (since no shortest path needs more than n−1 edges).
SOURCE: 📄 pp. 207–210

Mechanism: Bellman-Ford pseudocode 📄 pp. 207–216

function BellmanFord(G, source s):
    # Initialize distances
    Distance[s] = 0
    for each vertex v ≠ s:
        Distance[v] = ∞

    # Relax all edges n−1 times
    for iteration = 1 to n−1:                    # ← outer loop: n−1 times
        for each edge (j, k) with weight w(j,k):  # ← inner loop: m edges
            if Distance[j] + w(j,k) < Distance[k]:
                Distance[k] = Distance[j] + w(j,k)   # relax!

    # OPTIONAL: detect negative cycles
    for each edge (j, k) with weight w(j,k):
        if Distance[j] + w(j,k) < Distance[k]:
            print("Negative cycle detected!")   # n-th iteration still improves → cycle

    return Distance

# Complexity: O(n × m) = O(nm)
# Works with: negative edge weights (but not negative cycles)
SOURCE: 📄 pp. 207–216

Misconception Cards 📄 pp. 141–226

❌ Misconception: "Longest path in a general graph can be computed in O(n+m) using the DAG algorithm."

✅ Reality: The O(n+m) longest path algorithm ONLY works for DAGs. For general graphs with cycles, computing the longest simple path is NP-hard (it's equivalent to the Hamiltonian Path problem). The DAG structure is essential — it guarantees topological order exists and prevents infinite loops. 📄 pp. 157–168

❌ Misconception: "Floyd-Warshall can only be used for directed graphs."

✅ Reality: Floyd-Warshall works for both directed and undirected graphs. For undirected graphs, represent each undirected edge (u,v,w) as TWO directed edges: (u,v,w) and (v,u,w). The algorithm then proceeds identically. 📄 pp. 217–226

❌ Misconception: "Bellman-Ford runs until convergence — if it converges early, we can stop."

✅ Reality: Early convergence IS allowed as an optimization, but the guaranteed correctness comes from running exactly n−1 iterations. For exam purposes, always show n−1 full iterations (or explicitly state "converged after k iterations, which is ≤ n−1, so the answer is correct"). 📄 pp. 207–216

❌ Misconception: "Topological sort uniquely determines an ordering of vertices."

✅ Reality: A DAG may have many valid topological orderings. Any ordering where all edges (u,v) have u before v is valid. The algorithm using a queue produces ONE valid ordering, but which one depends on the implementation (choice of initial vertex when multiple have indegree 0). 📄 pp. 141–156

❌ Misconception: "max(LPT) gives the minimum number of semesters."

✅ Reality: max(LPT) + 1 gives the minimum number of semesters. LPT[v] = 0 means v is in Semester 1 (group 0). LPT[v] = k means v is in Semester k+1. So if max LPT = 3, you need semesters 0, 1, 2, 3 = 4 total semesters. Watch the off-by-one! 📄 pp. 157–168

❌ Misconception: "Floyd-Warshall's recurrence uses i and j from the previous k-layer, but you must update k in the same pass."

✅ Reality: Floyd-Warshall's correctness DEPENDS on using values from W^(k-1) when computing W^k. In-place update actually still works (this is a known subtlety), but conceptually each W^k[i][j] depends on W^(k-1)[i][k] and W^(k-1)[k][j], which are NOT changed during iteration k (since W^k[i][k] = W^(k-1)[i][k] and W^k[k][j] = W^(k-1)[k][j]). 📄 pp. 217–226

❌ Misconception: "The update rule for Floyd-Warshall is W[i][j] = W[i][k] × W[k][j]."

✅ Reality: The update uses ADDITION (for weights) and MIN (to find the shortest), NOT multiplication. W^k[i][j] = min(W^(k-1)[i][j], W^(k-1)[i][k] + W^(k-1)[k][j]). The multiplication version would be Warshall's transitive closure algorithm (path existence, not shortest path). 📄 pp. 217–226

Memory Aids & Mnemonics 📄 pp. 141–226

🧠 Mnemonic 1: Topological Sort — "Zero First" Always pick the vertex with indegree ZERO. Remove it, decrement its successors. Repeat. If you run out of zero-indegree vertices before processing all: CYCLE DETECTED. Memory hook: "Zero before hero" — vertices with zero prerequisites go first, enabling all others. 📄 pp. 141–156
🧠 Mnemonic 2: Longest Path (Topo DP) — "Process left, update right" In topological order, each vertex v: look LEFT (its predecessors' LPT values, already computed) to set LPT[v]. Then look RIGHT (update v's successors). Never go backward in topological order. Formula: LPT[v] = 1 + max{LPT[u] : (u,v) ∈ E}. Source vertices get LPT = 0. 📄 pp. 157–168
🧠 Mnemonic 3: Bellman-Ford — "Relax n-1 Times" n vertices → shortest paths have ≤ n−1 edges → relax ALL edges EXACTLY n−1 times. Memory hook: "n-1 Laps Around the Track" — the track is all edges; you run n−1 laps. If a (n-th) lap still improves anything: negative cycle exists. 📄 pp. 207–216
🧠 Mnemonic 4: Floyd-Warshall — "Through k, or Not Through k" For each intermediate vertex k, ask: "Does going through k improve the path from i to j?" Two-case recurrence: W^k[i][j] = min(W^(k-1)[i][j], W^(k-1)[i][k] + W^(k-1)[k][j]) "skip k" "go through k" Run this for k=1 to n. Final W^n is the answer. Space: just two n×n matrices (rolling). 📄 pp. 217–226
🧠 Mnemonic 5: Algorithm Selection DAG + dependencies → Topological sort → Longest path (DATO-LO)
One source + negative weights → Bellman-Ford (ONE-NEG-BELL)
All pairs + any weights → Floyd-Warshall (ALL-FLOYD)
One source + positive weights → Dijkstra (ONE-POS-DI) 📄 pp. 141–226

WARM-UP CONCEPTUAL 📄 pp. 141–156

WC-1. A classmate says: "Topological sort is just sorting vertices alphabetically or by number — you always output them in order 1, 2, 3, 4, ..." Is this correct? Explain what topological sort actually does.

▶ Show Explanation

Incorrect. Topological sort is NOT numeric/alphabetic ordering of vertices. 📄 pp. 141–156

Topological sort produces an ordering where every directed edge (u,v) has u appearing BEFORE v. The ordering respects the dependency structure of the DAG. 📄 pp. 141–156

Counter-example: If vertex 3 must come before vertex 1 (edge 3→1 exists), a valid topological sort must output ...3...1... — with 3 before 1. Numeric order (1,2,3,4...) would be wrong here.

What makes it "sorting" at all: The algorithm uses indegrees to determine which vertex has no remaining unfulfilled dependencies and can be output next. This might happen to produce numeric order on some specific graphs, but not in general.

WARM-UP PROOF STRUCTURE 📄 pp. 157–168

WC-2. For the Longest Path in a DAG computation, why is it ESSENTIAL to process vertices in topological order? What goes wrong if you process them in a different order?

▶ Show Explanation

The core issue: dependencies. When we compute LPT[v] = 1 + max{LPT[u] : (u,v) ∈ E}, we need all LPT[u] values for predecessors u to be ALREADY CORRECT. 📄 pp. 157–168

What goes wrong with wrong order: Suppose we process vertex v before its predecessor u. At that moment, LPT[u] = 0 (not yet computed). We compute LPT[v] = 1 + 0 = 1. Later, we compute LPT[u] = 5. Now LPT[v] should be 6, but we already set it to 1 — and we don't go back to fix it.

Why topological order guarantees correctness: In topological order, ALL predecessors u of v appear BEFORE v. So when we reach v, all LPT[u] values for predecessors are already correctly computed. We never need to backtrack. 📄 pp. 157–168

See §2b for the full traced example where this plays out on the course prerequisite DAG.

WARM-UP SMALL TRACE 📄 pp. 217–226

WC-3. In Floyd-Warshall, when we process intermediate vertex k=2 and compute W²[3][1], what exactly are we computing? Explain in plain English what W²[3][1] represents.

▶ Show Explanation

W²[3][1] = the length of the shortest path from vertex 3 to vertex 1, where the ONLY allowed intermediate vertices are {1, 2}. 📄 pp. 217–226

In plain English: "What is the cheapest way to travel from city 3 to city 1, if you are only allowed to stop at cities 1 and 2 along the way (and not cities 3, 4, 5, ...)?"

How it's computed: W²[3][1] = min(W¹[3][1], W¹[3][2] + W¹[2][1]). This asks: is it cheaper to go 3→1 WITHOUT using city 2 (= W¹[3][1]), or to go 3→2→1 (= W¹[3][2] + W¹[2][1])? 📄 pp. 217–226

After all n iterations (k=1 to n), W^n[i][j] = shortest path from i to j allowing ALL vertices as intermediates = the true shortest path.

WARM-UP ALGORITHM COMPARISON 📄 pp. 207–216

WC-4. Bellman-Ford and Dijkstra both compute single-source shortest paths. Your friend says "Bellman-Ford is just a slower Dijkstra — you should always use Dijkstra." When is your friend wrong? Give a specific type of example.

▶ Show Explanation

Your friend is wrong when the graph has negative edge weights. 📄 pp. 207–216

Why Dijkstra fails with negative edges: Dijkstra "settles" a vertex — marks it as finalized — when it has the minimum current distance. With negative edges, a later negative edge can create a shorter path to an already-settled vertex. Dijkstra doesn't go back and re-update settled vertices, so it misses these shortcuts.

Concrete counter-example: Graph: 1→2 (weight 3), 1→3 (weight 5), 2→4 (weight -2). Dijkstra settles vertex 2 first. But later, using edge 2→4 (weight -2), the actual shortest 1→4 path is 1→2→4 = 1. If Dijkstra settled 4 before processing the negative edge from 2, it might miss this. Bellman-Ford handles this correctly by iterating over ALL edges multiple times. 📄 pp. 179–216

When the friend IS right: If all edge weights are non-negative, Dijkstra is both correct AND faster (O((n+m)log n) vs Bellman-Ford's O(nm)). Use Dijkstra when no negative edges exist.

WARM-UP MISCONCEPTION PUNCTURE 📄 pp. 157–168

WC-5. Your classmate says: "The minimum number of semesters to complete 7 courses where the longest dependency chain has 3 courses is 3 semesters." What did they get wrong about the off-by-one?

▶ Show Explanation

The answer should be 4 semesters, not 3. Your classmate confused the longest chain length with the number of semesters. 📄 pp. 157–168

The off-by-one explained: If the longest chain is A → B → C (length 3 courses), the LPT values are: LPT[A] = 0 (first semester), LPT[B] = 1 (second semester), LPT[C] = 2 (third semester). But max(LPT) = 2 means 3 distinct "levels" (0, 1, 2), which corresponds to 3 semesters.

Wait — is the answer 3 or 4? Let's recount: A chain of 3 courses where no two can be taken simultaneously requires 3 semesters (one course per semester). The longest PATH value (LPT) of the last node = 2. Number of semesters = max(LPT) + 1 = 3. 📄 pp. 157–168

So if your classmate said "3 semesters," they might actually be right! The off-by-one trips up students who say "max(LPT) = answer" (getting 2) instead of "max(LPT) + 1 = answer" (getting 3). Check the specific wording — LPT uses 0-indexing for semesters.

Q1. A directed graph G has a valid topological ordering if and only if:

  1. G is connected
  2. G has no directed cycles (G is a DAG)
  3. Every vertex has the same indegree
  4. G has at least one vertex with indegree 0
▶ Show Answer

✅ Correct Answer: B — G has no directed cycles (G is a DAG)

A topological ordering exists if and only if the directed graph is a DAG (Directed Acyclic Graph). If there is a cycle A→B→C→A, then A must come before B, B before C, and C before A — an impossibility. 📄 pp. 141–156

Q7. Prove the Floyd-Warshall recurrence is correct: W^k[i][j] = min(W^(k-1)[i][j], W^(k-1)[i][k] + W^(k-1)[k][j]). Then analyze its time and space complexity, explaining why the "rolling array" optimization reduces space from O(n³) to O(n²).

▶ Show Answer (Full Model Proof)

✅ Part 1: Correctness of the recurrence 📄 pp. 217–226

Definition: W^k[i][j] = shortest path from i to j using only {1,...,k} as intermediate vertices.
Base case. W⁰[i][j]: no intermediate vertices allowed. Only direct edges count. So W⁰[i][j] = direct edge weight (∞ if none), and W⁰[i][i] = 0. ✓ 📄 p. 217
Inductive step. Assume W^(k-1) correctly gives shortest paths using {1,...,k-1}. We derive W^k. 📄 pp. 217–226
Two cases for any shortest i→j path using {1,...,k}:
Case 1: The shortest path does NOT use vertex k as an intermediate. Then it uses only {1,...,k-1} as intermediates → W^k[i][j] = W^(k-1)[i][j]. 📄 pp. 217–226
Case 2: The shortest path DOES use vertex k as an intermediate. Vertex k appears exactly once (otherwise we could remove a k-cycle to shorten the path, since no negative cycles assumed). So the path splits as i→...→k→...→j, where the subpaths i→k and k→j use only {1,...,k-1} as intermediates. Hence: W^k[i][j] = W^(k-1)[i][k] + W^(k-1)[k][j]. 📄 pp. 217–226
Combining: W^k[i][j] = min(Case 1, Case 2) = min(W^(k-1)[i][j], W^(k-1)[i][k] + W^(k-1)[k][j]). ✓ ∎ 📄 pp. 217–226

Part 2: Complexity analysis 📄 pp. 217–226

Time: Three nested loops: k from 1 to n, i from 1 to n, j from 1 to n. Each iteration does O(1) work (one min and one addition). Total: O(n³).

Space (naive): If we store W[i][j][k] for all three dimensions, we need n × n × n = n³ entries → O(n³) space.

Space (rolling array optimization): When computing W^k, we only use values from W^(k-1). Specifically, W^k[i][j] depends on W^(k-1)[i][k] and W^(k-1)[k][j]. Crucially, W^k[i][k] = W^(k-1)[i][k] (vertex k as destination, using k as intermediate, is the same as not using k since you'd have to go i→k and then stay — no improvement). This means W^(k-1)[i][k] = W^k[i][k], so we can safely update the matrix IN PLACE. Only ONE n×n matrix is needed, not n of them. Space: O(n²). 📄 pp. 217–226

Q11. Write pseudocode for the O(n+m) topological sort algorithm using a queue. Provide a step-wise English explanation and analyze time and space complexity.

▶ Show Answer

✅ Pseudocode:

function TopologicalSort(G, n):
    # G is given as adjacency list. n = number of vertices.

    # Step 1: Compute indegree of every vertex
    indegree = array of size n, initialized to 0
    for each vertex u in G:
        for each neighbor v of u:       # edge (u,v)
            indegree[v] += 1

    # Step 2: Initialize queue with all zero-indegree vertices
    Queue = empty queue
    for each vertex v:
        if indegree[v] == 0:
            Queue.enqueue(v)

    # Step 3: Process vertices
    result = empty list
    while Queue is not empty:
        v = Queue.dequeue()             # pick a ready vertex
        result.append(v)               # output it
        for each neighbor w of v:      # for each edge (v,w)
            indegree[w] -= 1           # remove v's dependency
            if indegree[w] == 0:
                Queue.enqueue(w)       # w is now ready

    # Step 4: Check for cycles
    if len(result) < n:
        print("CYCLE DETECTED — no topological ordering exists!")
        return None

    return result
SOURCE: 📄 pp. 141–156

English Explanation:

  1. Count incoming edges for each vertex (indegree). O(n+m) time.
  2. All vertices with indegree 0 (no prerequisites) go into the queue first.
  3. Repeatedly: dequeue a vertex v (it's safe to process now), add to result, then reduce indegrees of v's neighbors. If any neighbor now has indegree 0, add it to the queue.
  4. If we couldn't process all n vertices: a cycle prevented the algorithm from finishing.

📝 Complexity Analysis:

Time: Step 1 (indegree computation): each edge processed once = O(m). Step 2 (init queue): each vertex checked once = O(n). Step 3 (main loop): each vertex dequeued once (O(n)) and each edge processed once (O(m)) = O(n+m). Total: O(n+m). 📄 pp. 141–156

Space: Indegree array = O(n). Queue = O(n) (at most n vertices). Result = O(n). Total: O(n). 📄 pp. 141–156

Q12. Write pseudocode for computing the Longest Path To each vertex in a DAG (the "minimum semesters" computation). Combine it with topological sort in one pass. Analyze time and space complexity.

▶ Show Answer

✅ Pseudocode (combined topological sort + LPT in one pass):

function LongestPathToAll(G, n):
    # Computes LPT[v] = longest path from any source to vertex v.
    # Uses topological sort (queue-based) and updates LPT during sort.

    # Step 1: Compute indegrees and initialize LPT
    indegree = array of size n+1, initialized to 0
    LPT = array of size n+1, initialized to 0      # base case: all start at 0

    for each vertex u:
        for each neighbor v of u:                   # edge (u,v)
            indegree[v] += 1

    # Step 2: Initialize queue with indegree-0 vertices
    Queue = empty queue
    for each vertex v:
        if indegree[v] == 0:
            Queue.enqueue(v)
            # LPT[v] = 0 already — these are the "source" vertices

    # Step 3: Process in topological order, computing LPT on the fly
    result_order = []
    while Queue is not empty:
        v = Queue.dequeue()
        result_order.append(v)

        for each neighbor w of v:                   # edge (v,w)
            # Can we reach w via v in longer steps?
            if LPT[v] + 1 > LPT[w]:
                LPT[w] = LPT[v] + 1               # relax!

            indegree[w] -= 1
            if indegree[w] == 0:
                Queue.enqueue(w)                    # w is ready; LPT[w] is now final

    # Step 4: Answer
    max_LPT = max(LPT)
    min_rounds = max_LPT + 1                       # 0-indexed, so add 1

    return LPT, min_rounds, result_order
SOURCE: 📄 pp. 157–168

English Explanation:

  1. Initialize LPT[v]=0 for all v. Source vertices (indegree 0) stay at 0 — they're in "semester 1."
  2. Process vertices in topological order. When we dequeue v, LPT[v] is guaranteed final (all predecessors were processed earlier in topo order).
  3. For each successor w of v: if going through v gives a longer path (LPT[v]+1 > LPT[w]), update LPT[w]. This is the "relaxation" step — same as Bellman-Ford, but on a DAG.
  4. When a vertex w enters the queue (indegree becomes 0), its LPT is finalized.

📝 Complexity Analysis:

Time: All steps are O(n+m) — each vertex and edge processed exactly once. O(n+m). 📄 pp. 157–168

Space: Indegree array O(n) + LPT array O(n) + Queue O(n) = O(n). 📄 pp. 157–168

Key insight: When w enters the queue (all its predecessors processed), LPT[w] is final because ALL predecessors' contributions have been incorporated. This is why topological order is critical. 📄 pp. 157–168

# TOPOLOGICAL SORT + LONGEST PATH — O(n+m)
function TopoSort_LPT(G, n):
    compute indegree[v] for all v       # O(n+m)
    LPT[v] = 0 for all v
    Q = {v : indegree[v] == 0}
    result = []
    while Q not empty:
        v = dequeue(Q)
        result.append(v)
        for each neighbor w of v:
            LPT[w] = max(LPT[w], LPT[v] + 1)   # relax LPT
            indegree[w] -= 1
            if indegree[w] == 0: Q.enqueue(w)
    return result, max(LPT) + 1         # topo order, min rounds

# BELLMAN-FORD — O(nm)
function BellmanFord(G, s, n, m):
    D[s]=0; D[v]=∞ for v≠s
    for i = 1 to n-1:                  # n-1 iterations
        for each edge (j,k,w):
            if D[j]+w < D[k]: D[k]=D[j]+w   # relax
    return D

# FLOYD-WARSHALL — O(n³), O(n²) space
function FloydWarshall(W, n):
    D = copy(W)                         # D[i][i]=0, D[i][j]=w or ∞
    for k=1 to n:
        for i=1 to n:
            for j=1 to n:
                if D[i][k]+D[k][j] < D[i][j]:
                    D[i][j] = D[i][k]+D[k][j]
    return D

THE #1 MISTAKE ON EXAM QUESTIONS ABOUT THIS TOPIC:

For Floyd-Warshall: Confusing the sub-problem definition with Bellman-Ford's. Floyd-Warshall's k is the set of allowed intermediate vertices ({1,...,k}), NOT the number of edges allowed. Writing the wrong recurrence (e.g., "shortest path using exactly k edges") will get zero points even if the code looks similar. Always state "using only vertices {1,...,k} as intermediates." 📄 pp. 217–226

For Longest Path in DAG: Returning max(LPT) instead of max(LPT)+1. LPT is 0-indexed — source vertices have LPT=0, which means "first group." Always add 1 to convert from 0-indexed group number to group count. 📄 pp. 157–168

IF YOU REMEMBER NOTHING ELSE, REMEMBER THIS:

All four algorithms — Topological Sort, Longest Path, Bellman-Ford, Floyd-Warshall — follow the same DP recipe: define a sub-problem, write a two-case recurrence, solve in the right order (topological or k=1 to n). The "right order" guarantee is what makes DP efficient. Without it, you'd need exponential recursion. 📄 pp. 141–226
