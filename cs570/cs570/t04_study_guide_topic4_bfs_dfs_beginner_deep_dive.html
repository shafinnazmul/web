🧗 Warm-up Glossary — Vocabulary You Need Before §1

Graph algorithms use many terms casually ("visit", "neighbor", "path") that have precise meanings in this context. Misunderstanding even one can cascade into confusion about BFS vs DFS behavior. Clear these up now.

What is a graph? 📄 p. 105

A graph G = (V, E) has two ingredients: V is a set of vertices (nodes — draw them as circles), and E is a set of edges (draw them as lines connecting circles). An edge (u, v) connects vertex u to vertex v. 

  Example: 6-vertex graph (the one we'll use throughout this guide)

       1 ——— 2 ——— 5
       |     |
       3 ——— 4 ——— 6

  V = {1, 2, 3, 4, 5, 6}
  E = {(1,2), (1,3), (2,4), (2,5), (3,4), (4,6)}

  Vertex 2 is "adjacent to" (a neighbor of): 1, 4, 5
  Vertex 4 is "adjacent to": 2, 3, 6
  Vertex 5 and 6 are "leaf" nodes — they connect to only one other vertex.
SOURCE: 📄 pp. 105–110

Directed vs Undirected graphs 📄 p. 105

In an undirected graph, edges have no direction: (u,v) and (v,u) are the same edge. If you can travel from u to v, you can also travel from v to u.

In a directed graph (digraph), edges have arrows: (u,v) means there is an arrow FROM u TO v. This does NOT automatically mean there is an arrow from v to u. Most BFS/DFS introductions assume undirected; the lectures cover both. 📄 p. 105

What is the "degree" of a vertex? 📄 p. 227

The degree of vertex v (written deg(v)) is the number of edges touching v — i.e., the number of its neighbors.

Example: In our 6-vertex graph above, deg(2) = 3 (neighbors: 1, 4, 5), deg(5) = 1 (neighbor: 2 only), deg(4) = 3 (neighbors: 2, 3, 6).

In directed graphs: out-degree = number of arrows leaving v; in-degree = number of arrows arriving at v. 📄 p. 227

What is a "path" in a graph? 📄 p. 105

A path from vertex s to vertex t is a sequence of vertices s = v₀, v₁, v₂, ..., v_k = t where each consecutive pair (v_i, v_{i+1}) is an edge. The length of the path is the number of edges, k.

Example: In our graph, a path from 1 to 6 is: 1 → 2 → 4 → 6 (length 3). Another path: 1 → 3 → 4 → 6 (also length 3). There is NO path from 5 to 6 without going through vertex 2 or 1.

What does "connected" mean? 📄 p. 110

A graph is connected if there exists a path between every pair of vertices. Our 6-vertex example above IS connected — you can get from any vertex to any other.

If some vertices cannot be reached from others, the graph is disconnected and splits into connected components — groups of mutually reachable vertices.

What is a queue (FIFO)? 📄 p. 110

A queue is a data structure that follows First In, First Out (FIFO): the first element added is the first element removed. Think of a line at a coffee shop — the person who arrived first is served first.

Operations: enqueue (add to the back), dequeue (remove from the front). BFS uses a queue to process vertices in the order they are discovered. 

  Queue state visualization:

  FRONT [  1  |  2  |  3  |  4  ] BACK

  dequeue() → removes 1 from FRONT
  enqueue(5) → adds 5 to BACK
  Result: FRONT [  2  |  3  |  4  |  5  ] BACK

What is a stack (LIFO)? 📄 p. 129

A stack is a data structure that follows Last In, First Out (LIFO): the most recently added element is the first to be removed. Think of a stack of plates — you always take from the top.

Operations: push (add to top), pop (remove from top). DFS uses a stack (either explicitly or implicitly through recursion).

What does "visited" mean in graph traversal? 📄 p. 110

Algorithms use a boolean array visited[] (sometimes called discovered[] or marked[]) to track which vertices have already been seen. This prevents infinite loops: without this bookkeeping, BFS/DFS on a cycle would run forever.

Example: visited = [F, F, F, F, F, F, F] at start (all False for vertices 0–6). When we visit vertex 2, we set visited[2] = True. We never re-process vertex 2.

What is an adjacency matrix? 📄 pp. 106–108

An adjacency matrix A is an n × n matrix where A[i][j] = 1 if there is an edge from i to j, and A[i][j] = 0 otherwise. (n = number of vertices)

Key property: You can check "is j a neighbor of i?" in O(1) time — just look up A[i][j]. But you must always allocate n² space regardless of how many edges exist.

What is an adjacency list? 📄 pp. 108–110

An adjacency list stores, for each vertex v, a list of its neighbors. It uses space proportional to the number of edges (O(V + E)) rather than V². To find all neighbors of v, just iterate over Adj[v].

What is a "parent" pointer in BFS/DFS? 📄 p. 118

When BFS/DFS visits vertex k by traversing an edge from vertex j, we record parent[k] = j. Following parent pointers from any vertex back to the source reconstructs the shortest path (in BFS) or the exploration tree (in DFS).

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 105–156 (Lectures 12–15)

1.1 The fundamental problem — How does a computer "look at" a graph? 📄 p. 105

Imagine you're a city planner with a map of roads (edges) and intersections (vertices). A human can glance at the map and immediately see if two cities are connected. But a computer cannot "glance" — it must be given the map in a precise format, and then it must follow a systematic rule to explore it.

This is the core challenge of this topic: How do you store a graph in memory? and How do you systematically explore it? The answers to these two questions shape everything — from how fast your search runs to whether you can even find a path at all. 📄 p. 105

The two storage methods (adjacency matrix and adjacency list) trade off space efficiency vs access speed. The two exploration strategies (BFS and DFS) trade off finding the shortest path vs finding any path quickly. Neither choice is universally "better" — the right answer depends on what your graph looks like and what you need to find. 📄 pp. 105–110

1.3 A vivid analogy — exploring a cave system 📄 pp. 110–140

Imagine a network of underground caves connected by tunnels. You start at the entrance (the source vertex). Your goal: find all caves.

BFS is like exploring with a flood. You fill the cave system with water from the entrance. Water spreads outward uniformly — the caves one tunnel away flood first, then two tunnels away, and so on. Every cave that CAN be reached will be flooded, and the water reaches each cave by the shortest possible route. 📄 pp. 110–118

DFS is like exploring with a spelunker following string. You walk into the first tunnel, then take the first fork, then the first fork again — diving as deep as possible. When you hit a dead end, you backtrack on your string to the last unexplored fork. This finds all caves efficiently but does NOT guarantee the shortest route to any cave. 📄 pp. 129–140

📌 Key Intuition BFS guarantees the shortest path (measured in edges) to every reachable vertex. DFS does NOT guarantee shortest paths — but it reveals structural properties like cycles, strongly connected components, and topological orderings that BFS cannot. 📄 pp. 118, 141

1.4 Technical definitions 📄 pp. 105–140

Adjacency matrix: For a graph G = (V, E) with n = |V| vertices, the adjacency matrix A is an n × n matrix where A[i][j] = 1 if (i,j) ∈ E and A[i][j] = 0 otherwise. For undirected graphs, A is symmetric: A[i][j] = A[j][i]. 📄 pp. 106–108

Adjacency list: An array Adj of n lists, where Adj[v] contains all vertices u such that (v,u) ∈ E. Total space = O(|V| + |E|) = O(n + m). 📄 pp. 108–110

BFS(s): Starting from source vertex s, BFS uses a queue to visit all vertices reachable from s, in order of increasing distance (edge count) from s. Sets visited[v] = True when v is first discovered, and records parent[v] and level[v]. 📄 pp. 110–118

DFS(s): Starting from source vertex s, DFS recursively explores as far as possible before backtracking. Records pre[v] (time when v is first entered) and post[v] (time when v's exploration finishes). These timestamps reveal cycle structure and component membership. 📄 pp. 129–148

G² (graph square): The square of G = (V, E) is a new graph G² = (V, E') where (u, w) ∈ E' if and only if there exists a vertex v such that both (u,v) ∈ E and (v,w) ∈ E — i.e., u and w are connected by a path of length ≤ 2 in G. 📄 pp. 108–110

2a BFS (Breadth-First Search) — Complete Step-by-Step Trace Lecture PDF: pp. 110–128 · Assignment 4 Q3 📋 A4 Q3

Intuition: Level-by-level flooding 📄 p. 110

BFS answers the question: "What is the shortest path (fewest edges) from source s to every other vertex?" It works by processing vertices in waves — first all vertices at distance 1, then all at distance 2, etc.

The queue is the secret: when you discover a new vertex, you add it to the BACK of the queue. When you process a vertex, you take it from the FRONT. This FIFO order guarantees that vertices at distance d are all processed before any vertex at distance d+1. 📄 pp. 110–118

Mechanism: BFS pseudocode 📄 pp. 110–114

function BFS(G, s):
    # Initialize: all vertices unvisited, infinite distance
    for each vertex i in G:
        visited[i] = False
        parent[i]  = -1
        level[i]   = infinity

    # Seed the source
    visited[s] = True
    level[s]   = 0
    Queue = [s]             # enqueue the source

    while Queue is not empty:
        j = dequeue(Queue)  # take from FRONT
        for each neighbor k of j (from adjacency list):
            if visited[k] == False:
                visited[k] = True
                parent[k]  = j
                level[k]   = level[j] + 1
                enqueue(Queue, k)   # add to BACK
SOURCE: 📄 pp. 110–114

BFS Complexity Analysis 📄 pp. 114–118

Why O(n²) with adjacency matrix
Each vertex j is dequeued exactly once. When processing j, we scan its entire row in the matrix — n entries. With n vertices, total work = n × n = O(n²). 📄 p. 114
Why O(n + m) with adjacency list
Each vertex is dequeued once — O(n) total. For each vertex j, we examine Adj[j] — that's deg(j) operations. Summing over all vertices: Σ deg(v) = 2m (each edge counted twice in undirected graphs). Total: O(n + 2m) = O(n + m). 📄 p. 114

2b DFS (Depth-First Search) — Complete Step-by-Step Trace with Pre/Post Numbers Lecture PDF: pp. 129–156 · Assignment 4 Q3 📋 A4 Q3

Intuition: Dive deep, then backtrack 📄 p. 129

DFS explores by committing fully to one direction before trying others. When vertex j is being explored, DFS immediately recursively explores each unvisited neighbor of j — going as deep as possible before returning. 📄 pp. 129–140

Two counters — pre[v] and post[v] — record when DFS enters and exits each vertex. These timestamps are surprisingly powerful: they reveal whether the graph has cycles, which edges are "back edges," and whether one vertex is an ancestor of another in the DFS tree. 📄 pp. 141–148

Mechanism: DFS pseudocode (recursive) 📄 pp. 129–140

# Global counter for pre/post timestamps
count = 0

function DFS_all(G):
    for each vertex i in G:
        visited[i] = False
        parent[i]  = -1

    for each vertex i in G:
        if visited[i] == False:
            DFS(i)           # start a new DFS tree

function DFS(i):
    visited[i] = True
    pre[i] = count;  count += 1    # ENTER vertex i

    for each neighbor j of i (in adjacency list order):
        if visited[j] == False:
            parent[j] = i
            DFS(j)                  # RECURSE into j

    post[i] = count;  count += 1   # EXIT vertex i (done exploring from i)

Intuition: Degree computation 📄 p. 227

The degree of a vertex is the count of its neighbors. In an undirected graph, every edge (u,v) contributes 1 to deg(u) and 1 to deg(v). In a directed graph, we distinguish out-degree (arrows leaving) from in-degree (arrows arriving).

Computing degrees seems trivial — just count edges. The interesting question for the exam is: what is the time complexity, and how does the graph representation change it? 📄 p. 227

Mechanism: Degree computation algorithms 📄 p. 227

# ── OUT-DEGREE using adjacency list ──────────────────────────────
# out_degree[v] = number of arrows leaving v = length of Adj[v]
function computeOutDegrees(G):
    for each vertex v in G.vertices:                # O(|V|) iterations
        out_degree[v] = len(G.adjacency[v])         # O(1) per vertex
    return out_degree
# Total: O(|V|)

# ── IN-DEGREE using adjacency list ───────────────────────────────
# Count how many times each vertex appears as a neighbor
function computeInDegrees(G):
    for each vertex v in G.vertices:
        in_degree[v] = 0                            # initialize

    for each vertex u in G.vertices:                # iterate all vertices
        for each v in G.adjacency[u]:               # iterate u's out-edges
            in_degree[v] += 1                       # v has one more incoming

    return in_degree
# Total: O(|V| + |E|) — each edge traversed once

Graph Squaring (G²) — Intuition 📄 pp. 108–110

The square of a graph G, written G², adds a new edge between every pair of vertices that are at most 2 hops apart in G. Two vertices u and w are "2-hop connected" if they share a common neighbor v.

Real-world analogy: G is your direct LinkedIn connections. G² adds everyone who is a "friend of a friend" — your 2nd-degree connections. 📄 pp. 108–110

Mechanism: Computing G² 📄 pp. 108–110

# ── G² using adjacency list ─────────────────────────────────────
# Add edge (u, w) to G² if u and w share a common neighbor
function computeGSquared_adjList(G):
    G2 = empty graph with same vertices as G

    for each vertex u in G.vertices:           # O(|V|)
        for each neighbor v in G.adjacency[u]: # O(deg(u))
            for each neighbor w in G.adjacency[v]:  # O(deg(v))
                if w != u and (u, w) not in G2:
                    G2.addEdge(u, w)

    return G2
# Total: O(|V| × max_deg²)  ≈  O(|E| × |V|)  in worst case
# Using adjacency matrix: O(|V|³) via matrix multiplication A² = A×A

2d Graph Connectivity Proof: deg(v) ≥ n/2 → G is Connected Lecture PDF: p. 110 · Assignment 4 Q5 📋 A4 Q5

Intuition: Why high degree forces connectivity 📄 p. 110

Imagine n people at a party. If every person knows at least n/2 other people, can two strangers always find a mutual acquaintance? Yes! And stronger: the entire party is one connected group — there are no isolated cliques.

The key insight: if the graph were disconnected, some component would be "small" (fewer than n/2 vertices). But any vertex in that small component can only know others in the same component — giving it degree less than n/2. This contradicts our assumption. 📄 p. 110

Mechanism: Proof by contradiction 📄 p. 110

THEOREM: If every vertex v has deg(v) ≥ n/2, then G is connected.
Proof technique: contradiction. We assume the graph is disconnected and derive an impossibility.

Worked Example: Full proof with concrete n = 6 📄 p. 110 📋 A4 Q5

  Setup: n = 6 vertices. Every vertex must have degree ≥ n/2 = 3.

  Assume (for contradiction) that G is DISCONNECTED.
  Then G splits into at least 2 connected components.
  Let C be the SMALLEST component.
  Since there are at least 2 components and |V| = 6:
    |C| ≤ ⌊6/2⌋ = 3  (at most half the vertices — otherwise the other component would be smaller)

  Now take any vertex v in C.
  v can ONLY have edges to other vertices IN C (edges cannot cross component boundaries — that would connect C to another component, contradicting the definition of a component).

  Therefore: deg(v) ≤ |C| - 1  (at most connected to all OTHER vertices in C)
             deg(v) ≤ 3 - 1 = 2

  But our assumption says deg(v) ≥ n/2 = 3 for ALL vertices.
  CONTRADICTION: deg(v) ≤ 2 AND deg(v) ≥ 3 is impossible.

  Therefore our assumption (G is disconnected) was FALSE.
  G must be connected. ✓ □
SOURCE: 📄 p. 110

Full structured proof (for exam writing) 📄 p. 110

Step 1 [Proof Type]: Proof by contradiction.
We assume the negation: G is NOT connected. We derive an impossibility from this assumption. 📄 p. 110
Step 2 [Assumption]: Assume G is disconnected.
Then G has at least two connected components. Let C be the smallest connected component, with |C| = k vertices. Since there are at least two components: k ≤ ⌊n/2⌋ (the smallest component has at most half the vertices). So k < n/2 or k = n/2.
Step 3 [Key observation]: Edges cannot cross component boundaries.
By definition of a connected component, no edge connects a vertex inside C to a vertex outside C. If such an edge existed, those vertices would be in the same component — contradicting C being a separate component. 📄 p. 110
Step 4 [Degree bound]: Any vertex v ∈ C has degree at most k − 1.
Since v can only connect to other vertices inside C (by Step 3), and there are only k − 1 other vertices in C: deg(v) ≤ k − 1 < k ≤ n/2. So deg(v) < n/2. 📄 p. 110
Step 5 [Contradiction].
We assumed deg(v) ≥ n/2 for ALL vertices (given condition). But Step 4 shows deg(v) < n/2 for any v ∈ C. This is a contradiction. 📄 p. 110
Step 6 [Conclusion]: Our assumption in Step 2 was false.
Therefore G must be connected. ∎ 📄 p. 110

📋 A4 Q6 Graph-squaring G²: construction and complexity.

▶ Key Points in the Solution

📋 A6 Q4 Compute out-degree and in-degree of all vertices using an adjacency list.

Misconception Cards 📄 pp. 105–156

❌ Misconception: "BFS always visits FEWER nodes than DFS."

✅ Reality: BFS visits fewer nodes when the goal is near the source (short path). But if the goal is deep in the graph and BFS must explore many wide levels, it may visit MORE nodes than a DFS that happens to take the right deep branch first. BFS is only guaranteed to be OPTIMAL in path length, not in nodes visited. 📄 pp. 110–140

❌ Misconception: "DFS always visits the same vertices in the same order as BFS, just differently timed."

✅ Reality: BFS and DFS can discover vertices in completely different orders. BFS levels are determined by hop count from source. DFS order is determined by the recursive dive — vertices far from source can be visited early if they're reached via a deep chain. 📄 pp. 110–140

❌ Misconception: "Adjacency list is always better than adjacency matrix because it uses less space."

✅ Reality: Adjacency list is space-efficient for sparse graphs (few edges). But for dense graphs (many edges), O(n+m) ≈ O(n²) anyway, so the space advantage disappears. And adjacency matrix gives O(1) edge existence checks vs O(deg) for lists. The right choice depends on the graph density and the operations needed. 📄 pp. 108–110

❌ Misconception: "BFS and DFS both run in O(n) time."

✅ Reality: Both run in O(n + m) with adjacency lists, where m = |E| is the number of edges. If the graph has m = O(n²) edges (dense graph), then complexity is O(n²). If sparse (m = O(n)), complexity is O(n). Never just say "O(n)" without qualifying. 📄 pp. 114–140

❌ Misconception: "pre[v] is the order in which vertices are added to the queue in BFS."

✅ Reality: pre[] and post[] are exclusively DFS concepts. BFS uses level[] (hop count from source) and the order of dequeuing. pre[v] records when DFS ENTERS vertex v for the first time. post[v] records when DFS FINISHES exploring v (all recursive calls have returned). These two timestamps together uniquely characterize the DFS tree structure. 📄 pp. 141–148

❌ Misconception: "A back edge means there's a back-and-forth (bidirectional) edge in the graph."

✅ Reality: A back edge in DFS is an edge (u,v) where v is an ANCESTOR of u in the DFS tree — meaning DFS visited v before u, and now u has an edge pointing back to v. This creates a directed cycle. It has nothing to do with the original graph being bidirectional. 📄 pp. 141–148

❌ Misconception: "If G is disconnected, BFS will still visit all vertices."

✅ Reality: BFS(s) starting from vertex s only visits all vertices in s's connected component. To visit all vertices in a disconnected graph, you must run BFS (or DFS) repeatedly from each unvisited vertex, just as the outer loop in DFS_all() does. 📄 p. 110

Memory Aids & Mnemonics 📄 pp. 105–156

🧠 Mnemonic 1: BFS vs DFS — "BFS Breadth → Belly of the cave; DFS Depth → Dive first" BFS = Belly-of-cave (explore all nearby, THEN go deeper). Uses a Queue. DFS = Dive first (go as deep as you can, then backtrack). Uses a Stack (or recursion). Queue → Breadth; Stack → Depth. 📄 pp. 110–140
🧠 Mnemonic 2: Adjacency representation complexity — "Matrix = n-squared; List = n plus m" Every operation on adjacency matrix costs at least O(n) (scan a row) → algorithms become O(n²). Every operation on adjacency list costs O(deg(v)) → algorithms become O(n+m). Dense graph (m = n²): both representations give O(n²). Sparse graph (m = n): lists give O(n), matrix gives O(n²). 📄 pp. 106–118
🧠 Mnemonic 3: Degree calculation — "Out = O(1) per vertex; In = O(1) per EDGE" Out-degree = just count the list = O(1) per vertex = O(V) total. In-degree = count appearances = O(1) per edge traversed = O(V+E) total. Sum: "out is quick, in is thorough." 📄 p. 227
🧠 Mnemonic 4: Pre/Post intervals and edge types — "Contains = ancestor; Disjoint = cross" If [pre(v), post(v)] CONTAINS [pre(u), post(u)]: v is an ancestor of u → edge (u,v) is a BACK edge (u points to ancestor v) or forward edge. If the intervals are DISJOINT: the vertices are in unrelated subtrees → edge is a CROSS edge. Back edges = cycles. 📄 pp. 141–148
🧠 Mnemonic 5: Connectivity proof skeleton — "ASSUME SMALLEST CONTRADICTION" To prove "property P implies connected": (1) ASSUME disconnected, (2) find the SMALLEST component (k vertices), (3) show any vertex there has degree < n/2 — CONTRADICTION. Three words = three steps. 📄 p. 110

WARM-UP PRE/POST INTERPRETATION 📄 pp. 141–148

WC-4. After running DFS on a directed graph, you find: pre[u]=3, post[u]=8, pre[v]=5, post[v]=6. Is there a directed edge (u,v), a directed edge (v,u), or can you not determine this? Classify the edge if it exists.

▶ Show Explanation

Interval analysis: [pre(u), post(u)] = [3, 8] and [pre(v), post(v)] = [5, 6]. 📄 pp. 141–148

Does [3,8] contain [5,6]? Yes — 3 < 5 < 6 < 8. So u's interval CONTAINS v's interval. This means: u was entered before v, and u was exited after v. v is a descendant of u in the DFS tree.

Edge (u,v): If this edge exists and u's interval contains v's interval, the edge goes from ancestor to descendant → this is a tree edge (if DFS used this edge to discover v) or forward edge (if v was already discovered another way).

Edge (v,u): If this edge exists and u's interval contains v's interval, then v points to its ancestor u → this is a back edge, indicating a cycle! 📄 pp. 141–148

Conclusion: From intervals alone, you cannot determine which edge exists — you need to know the actual graph edges. BUT you CAN classify any given edge: (u,v) would be a tree/forward edge; (v,u) would be a back edge (cycle indicator). 📄 pp. 141–148

WARM-UP MISCONCEPTION PUNCTURE 📄 p. 110

WC-5. Your classmate tries to prove "G is connected" on a graph where deg(v) ≥ 2 for all v (with n = 10 vertices). They say: "Every vertex has at least 2 neighbors, so you can always walk from any vertex to any other vertex." Identify the flaw in their reasoning and give a counterexample.

▶ Show Explanation

The flaw: "Can walk from any vertex to any other" does not follow just from each vertex having ≥2 neighbors. Having many local neighbors does not prevent the graph from being split into isolated clusters where each cluster is internally well-connected but disconnected from others. 📄 p. 110

Counterexample: Take two separate 5-cycles (pentagons): C₅ = {A,B,C,D,E} with edges (A,B),(B,C),(C,D),(D,E),(E,A) and a second C₅ = {F,G,H,I,J} with edges (F,G),(G,H),(H,I),(I,J),(J,F). Every vertex has degree exactly 2. But the graph is DISCONNECTED — there's no edge between the two cycles. You cannot walk from A to F. 📄 p. 110

Why does deg ≥ n/2 work but deg ≥ 2 doesn't? The threshold n/2 is the critical point where a small component can no longer sustain high-degree vertices. With n=10 and n/2=5: any component with fewer than 5 vertices cannot have vertices with degree ≥ 5. This forces the whole graph to be one component. deg ≥ 2 is far too weak a condition. 📄 p. 110

  1. Back edge — because u's interval [4,9] contains v's interval [6,7]

Q4. Let G be a directed graph with n vertices and m edges, stored as an adjacency list. You want to compute G² (add edges for all pairs reachable within 2 hops). Which is the TIGHTEST asymptotic bound for this using the adjacency list approach?

  1. O(n + m)
  2. O(n² + m)
  3. O(n · m) or equivalently O(|V| · |E|)
  4. O(n³)
▶ Show Answer

✅ Correct Answer: C — O(n · m)

The algorithm: for each vertex u (n vertices), for each neighbor v of u (O(m) total across all u), for each neighbor w of v (O(deg(v)) per v). The total work is Σ_{u} Σ_{v ∈ Adj[u]} deg(v). In the worst case (a star graph), this is O(n · m). More precisely: for each of the m edges (u,v), we scan v's adjacency list of size deg(v). Total = Σ_{(u,v) ∈ E} deg(v) ≤ m · max_deg ≤ m · n = O(nm). 📄 pp. 108–110

Why A is wrong: O(n+m) would mean we only look at each vertex and edge once — but we need to look at neighbors-of-neighbors.

Why B is wrong: O(n²+m) would be correct only if deg(v) were always O(n/m), which isn't guaranteed.

Why D is wrong: O(n³) is the complexity for the ADJACENCY MATRIX approach (matrix multiplication A×A). The adjacency list approach is faster for sparse graphs.

Exam Tip: G² via adjacency list = O(|E| × |V|) = O(nm). G² via adjacency matrix = O(n³). 📄 p. 108

Q5. Prove: If every vertex v in an undirected graph G with n vertices has degree deg(v) ≥ n/2, then G is connected.

▶ Show Answer (Model Proof)

✅ Answer: TRUE — Prove by contradiction.

Proof by Contradiction
Step 1 [Assumption]. Assume for contradiction that G is NOT connected. 📄 p. 110
Step 2 [Component structure]. Since G is disconnected, it has at least two connected components. Let C be the smallest connected component, with |C| = k vertices. Since there are at least two components sharing n vertices, we have k ≤ ⌊n/2⌋, so k ≤ n/2. 📄 p. 110
Step 3 [No cross-component edges]. By definition of a connected component, no edge in G connects a vertex inside C to a vertex outside C. If such an edge existed, both endpoints would be in the same component — contradicting C being a separate component. 📄 p. 110
Step 4 [Degree upper bound]. Let v be any vertex in C. Every edge incident to v must connect v to another vertex inside C (by Step 3). There are at most k − 1 other vertices in C. Therefore: deg(v) ≤ k − 1 ≤ n/2 − 1 < n/2. 📄 p. 110
Step 5 [Contradiction]. Step 4 gives deg(v) < n/2. But our given condition states deg(v) ≥ n/2 for every vertex v. This is a contradiction. 📄 p. 110
Step 6 [Conclusion]. Our assumption (G is disconnected) leads to a contradiction. Therefore G must be connected. ∎ 📄 p. 110

Q6. Prove: A directed graph G has a directed cycle if and only if DFS on G reveals at least one back edge (an edge (u,v) where v is an ancestor of u in the DFS tree).

▶ Show Answer (Model Proof)

✅ Answer: TRUE — Prove both directions of the iff.

Direction (→): If G has a directed cycle, then DFS reveals a back edge.
Let v₀ → v₁ → v₂ → ... → v_k → v₀ be a directed cycle in G. Consider DFS starting from v₀ (or any vertex from which v₀ is reachable). When DFS first visits v₀ (pre[v₀] is set), it will eventually explore the cycle. At some point, DFS will be at v_k and attempt to traverse edge (v_k, v₀). At this point, v₀ was already visited (it was visited first in the cycle) and v₀ is still active (post[v₀] has not been set — DFS is still within v₀'s subtree, exploring through the cycle). So pre[v₀] < pre[v_k] < post[v_k] < post[v₀], meaning v₀'s interval contains v_k's interval. The edge (v_k, v₀) goes from descendant (v_k) to ancestor (v₀) — a BACK EDGE. 📄 pp. 141–148
Direction (←): If DFS reveals a back edge (u,v), then G has a directed cycle.
Suppose DFS finds a back edge (u,v) where v is an ancestor of u in the DFS tree. Since v is an ancestor of u, there exists a directed path from v to u in the DFS tree: v → ... → u (following tree edges). Combined with the back edge u → v, this forms a directed cycle: v → ... → u → v. 📄 pp. 141–148

  • ☐ Proved both (→) and (←) directions. 📄 pp. 141–148
  • ☐ For (→): Identified the back edge from the cycle; used interval containment argument. 📄 pp. 141–148
  • ☐ For (←): Identified the tree path from v to u; combined with back edge to get cycle. 📄 pp. 141–148
  • ☐ Used pre/post interval notation correctly. 📄 pp. 141–148

Q7. Let G be an undirected connected graph with n vertices and m edges. Prove that if G is stored as an adjacency list, then BFS runs in O(n + m) time. Specifically prove (a) each vertex is enqueued at most once, and (b) the total work across all vertices is proportional to n + m.

▶ Show Answer (Full Model Proof)

✅ Full complexity proof for BFS with adjacency list.

Part (a): Each vertex is enqueued at most once.
Proof: When vertex v is enqueued, we immediately set visited[v] = True. Subsequently, whenever any vertex u is dequeued and we consider v as a neighbor, the check "if visited[v] == False" will fail, so we never enqueue v again. Thus every vertex is enqueued at most once and dequeued at most once. The total enqueue/dequeue operations are O(n). 📄 pp. 110–114
Part (b): Total work is O(n + m).
Initialization: Setting visited[], parent[], level[] for all vertices = O(n). 📄 p. 110
Outer while loop: From Part (a), each vertex is dequeued at most once → the loop runs at most n times. Each dequeue is O(1). Total dequeue cost: O(n). 📄 p. 110
Inner for loop (processing neighbors): When vertex j is dequeued, we iterate over Adj[j] — exactly deg(j) neighbors. Across ALL vertex dequeue operations: Σ_{j=1}^{n} deg(j) = 2m (each undirected edge (j,k) is counted once in Adj[j] and once in Adj[k], so each edge contributes exactly 2 to the total). Total neighbor-processing cost: O(2m) = O(m). 📄 pp. 114–118
Conclusion: Total = O(n) [init] + O(n) [dequeues] + O(m) [neighbor processing] = O(n + m). ∎ 📄 pp. 114–118

  • ☐ Proved each vertex enqueued ≤ once (via visited[] flag). 📄 p. 110
  • ☐ Stated outer loop runs O(n) times. 📄 p. 110
  • ☐ Applied handshaking lemma: Σ deg(j) = 2m. 📄 pp. 114–118

Q11. Write pseudocode for BFS starting from source vertex s in an undirected graph G stored as an adjacency list. Your pseudocode should track visited[], parent[], and level[]. Then analyze time and space complexity.

▶ Show Answer

✅ Pseudocode:

function BFS(G, s):
    # --- INITIALIZATION ---
    for each vertex v in G.vertices:   # O(|V|) setup
        visited[v] = False
        parent[v]  = -1                # -1 = no parent (root)
        level[v]   = infinity          # unknown distance

    # --- SEED THE SOURCE ---
    visited[s] = True
    level[s]   = 0
    Queue      = new Queue()
    Queue.enqueue(s)

    # --- MAIN LOOP ---
    while Queue is not empty:
        j = Queue.dequeue()           # take from FRONT — FIFO
        for each neighbor k in G.adjacency[j]:   # O(deg(j)) per vertex
            if visited[k] == False:
                visited[k] = True     # mark BEFORE enqueuing
                parent[k]  = j        # j discovered k
                level[k]   = level[j] + 1
                Queue.enqueue(k)      # add to BACK

    return (visited, parent, level)
SOURCE: 📄 pp. 110–114

English Explanation:

  1. Initialize all vertices as unvisited with no parent and infinite distance.
  2. Mark the source as visited (distance 0) and add it to the queue.
  3. While the queue is not empty: dequeue a vertex j, examine all its neighbors. For each unvisited neighbor k: mark it visited, record j as its parent and set its level to j's level + 1, then enqueue k.
  4. When the queue empties, all reachable vertices have been visited with their shortest distances recorded.

📝 Complexity Analysis:

Time: Initialization = O(|V|). Each vertex enqueued/dequeued at most once = O(|V|). Total neighbor scans = Σ deg(v) = 2|E| = O(|E|). Total: O(|V| + |E|). 📄 pp. 114–118

Space: visited[], parent[], level[] arrays = O(|V|) each. Queue can hold at most all vertices at once = O(|V|). Total: O(|V|) extra space (beyond the graph). 📄 p. 110

Q12. Write pseudocode for DFS that computes pre[] and post[] timestamps for every vertex. Then describe how to use these timestamps to detect a directed cycle (without needing to explicitly classify every edge type). Analyze time and space complexity.

▶ Show Answer

✅ Pseudocode:

# Global variables
count = 0
has_cycle = False

function DFS_all(G):
    for each vertex v in G.vertices:
        visited[v] = False
        parent[v]  = -1

    for each vertex v in G.vertices:
        if visited[v] == False:
            DFS(v)               # start new DFS tree from unvisited v

    return has_cycle, pre, post

function DFS(i):
    global count, has_cycle

    visited[i] = True
    pre[i]     = count;  count += 1       # ENTER vertex i

    for each neighbor j in G.adjacency[i]:
        if visited[j] == False:
            parent[j] = i
            DFS(j)                        # RECURSE — tree edge
        else:
            # j is already visited — check if j is still active (an ancestor)
            if post[j] not yet set:       # j's interval is still open
                has_cycle = True          # j is ancestor of i → BACK EDGE → CYCLE!

    post[i] = count;  count += 1          # EXIT vertex i (all neighbors explored)
SOURCE: 📄 pp. 129–148

English Explanation:

  1. Initialize all vertices as unvisited. Start DFS from each unvisited vertex to handle disconnected graphs.
  2. When entering vertex i: record pre[i] = current count, increment count.
  3. For each neighbor j of i: if unvisited, recurse. If already visited and still active (post[j] not set yet — j is an ancestor on the current path), record a cycle.
  4. When exiting vertex i: record post[i] = current count, increment count.

Cycle detection insight: A back edge (i,j) exists when j is already visited AND j is still active (its post[] hasn't been set). This means j is still on the current DFS call stack — j is an ancestor of i. The path from j to i via tree edges, plus the edge back to j, forms a cycle. 📄 pp. 141–148

📝 Complexity Analysis:

Time: Each vertex entered/exited once = O(|V|). Total neighbor scans = Σ deg(v) = 2|E| = O(|E|). Total: O(|V| + |E|). 📄 pp. 129–140

Space: pre[], post[], visited[], parent[] = O(|V|) each. Recursion call stack depth = O(|V|) worst case (linear graph). Total: O(|V|). 📄 p. 129

Q13. Design an algorithm to compute the SECOND-SHORTEST path (by edge count) from source s to every vertex t, where the second-shortest path must be strictly longer than the shortest path. Write pseudocode, explain your approach, and analyze time and space complexity. Hint: can you extend BFS?

▶ Show Answer

✅ Approach: Two-pass BFS with relaxed visited tracking.

Key insight: Standard BFS computes the first-shortest distance (dist₁[v]). For the second-shortest, we track an additional distance array dist₂[v] = second-shortest distance. A vertex can be "visited" twice: once for its shortest path, once more for its second-shortest. 📄 pp. 110–128

function secondShortestPaths(G, s):
    # dist1[v] = shortest distance from s to v
    # dist2[v] = second-shortest (must be > dist1[v])
    for each vertex v:
        dist1[v] = infinity
        dist2[v] = infinity

    dist1[s] = 0
    Queue = new PriorityQueue()  # or regular queue since BFS = unweighted
    Queue.enqueue( (s, 0) )     # (vertex, distance)

    while Queue is not empty:
        (u, d) = Queue.dequeue()

        # If d exceeds even dist2[u], this path is useless — skip
        if d > dist2[u]:
            continue

        for each neighbor v in G.adjacency[u]:
            new_dist = d + 1              # unit edge weights

            if new_dist < dist1[v]:
                # Found a new shortest path to v
                dist2[v] = dist1[v]       # demote old shortest to second
                dist1[v] = new_dist
                Queue.enqueue( (v, new_dist) )

            elif new_dist > dist1[v] and new_dist < dist2[v]:
                # Found a new second-shortest path to v
                dist2[v] = new_dist
                Queue.enqueue( (v, new_dist) )
            # If new_dist == dist1[v]: same length as shortest, skip
            # If new_dist >= dist2[v]: already have a better second, skip

    return (dist1, dist2)
SOURCE: Extended from 📄 pp. 110–128

English Explanation:

  1. Initialize dist1[s]=0, all others infinity.
  2. BFS-style: dequeue (u, d). If d exceeds even the second-shortest known distance to u, skip — this path is useless.
  3. For each neighbor v: compute new_dist = d+1. Update dist1[v] if new_dist is shorter (push old shortest to dist2). Update dist2[v] if new_dist is strictly between dist1[v] and dist2[v].
  4. Each vertex can be enqueued at most twice (once for shortest, once for second-shortest).

📝 Complexity Analysis:

Time: Each vertex processed at most twice → 2|V| dequeue operations. Each dequeue scans the neighbor list → Σ 2·deg(v) = 2·2|E| = O(|E|). Total: O(|V| + |E|). 📄 pp. 114–118

Space: dist1[], dist2[] = O(|V|) each. Queue holds at most 2|V| entries = O(|V|). Total: O(|V|). 📄 p. 110

IN ONE SENTENCE, THIS TOPIC IS:

Graph algorithms need the right representation (matrix vs list) and the right traversal strategy (BFS for shortest paths, DFS for structure discovery) — both ultimately run in O(n+m) with adjacency lists, but serve fundamentally different purposes. 📄 pp. 105–156

THE THREE THINGS YOU MUST KNOW FOR THE EXAM:

  1. Representations: Matrix = O(1) edge check, O(n²) space, O(n) neighbor-list. List = O(deg) edge check, O(n+m) space, O(deg) neighbor-list. BFS/DFS on list = O(n+m); on matrix = O(n²). 📄 pp. 106–118

THE #1 MISTAKE ON EXAM QUESTIONS ABOUT THIS TOPIC:

Confusing when to mark a vertex as "visited." In BFS, you MUST set visited[v] = True when you ENQUEUE v (not when you dequeue it). If you mark as visited only upon dequeuing, the same vertex can be enqueued multiple times from different neighbors, causing incorrect behavior and overcounting. In DFS, mark visited upon ENTERING the function. 📄 pp. 110–140
