🧗 Warm-up Glossary — Vocabulary You Need Before §1

Asymptotic analysis relies on concepts from calculus and discrete math that are often assumed. Here they are, introduced from scratch. Skip to §1 if you already know all of these.

What does "grows faster" mean — formally? 📄 pp. 39–41

We say function f(n) grows faster than g(n) if, as n gets very large, the ratio f(n)/g(n) increases without bound. In other words, f eventually dwarfs g — no matter how big a constant head-start g has.

Concrete example: Does n² grow faster than n? Compute the ratio: n²/n = n. As n → ∞, this ratio → ∞. Yes — n² grows strictly faster than n. Now check: does n grow faster than log₂ n? Ratio: n / log₂ n → ∞ as n → ∞ (use L'Hôpital or plug in n = 1,000,000: 1,000,000 / 20 = 50,000). Yes — n grows faster than log n. 📄 pp. 39–41

What is a "limit" and why do we use it? 📄 pp. 39–41

A limit describes what value a function approaches as its input goes to infinity. We write: limn→∞ f(n)/g(n) = L. This means: for very large n, the ratio f(n)/g(n) gets arbitrarily close to L.

Three useful outcomes when comparing f(n) to g(n):
— If L = 0: f grows slower than g (f is "little-oh" of g).
— If 0 < L < ∞ (finite, nonzero): f grows at the same rate as g (f is "Theta" of g).
— If L = ∞: f grows faster than g.

Concrete example: limn→∞ (3n²+5n) / n² = lim (3 + 5/n) = 3. A finite nonzero constant → Θ(n²). 📄 pp. 39–41

What is log base b? 📄 p. 39

logb(x) is the exponent you must raise b to in order to get x. So log2(8) = 3 because 2³ = 8. In CS 570, "log n" means log2 n unless stated otherwise. 📄 p. 39

Key facts: log2(n²) = 2·log₂(n). log2(2ⁿ) = n. Logarithm growth is extremely slow: log₂(1,000,000) ≈ 20. This is why O(log n) algorithms are nearly instantaneous even for massive inputs.

What is a "constant factor" and why do we ignore it? 📄 pp. 39–41

A constant factor is a multiplier that doesn't depend on n. For example, in 5n², the "5" is a constant factor. In asymptotic analysis, we ignore constant factors because:

(1) On modern hardware, constant factors are absorbed by instruction-level parallelism, cache effects, and compiler optimizations. (2) The growth rate (the exponent of n) dominates for large inputs. For n = 1,000,000: 5n² = 5 × 10¹² while n³ = 10¹⁸ — the cubic term is 200,000× larger regardless of the constant 5. 📄 pp. 39–41

What is a "witness" (c and n₀)? 📄 pp. 39–41

To prove f(n) = O(g(n)) formally, you must provide two constants: c > 0 and n₀ ≥ 0 such that f(n) ≤ c·g(n) for all n ≥ n₀. The pair (c, n₀) is called the witness for the O-claim.

How to find witnesses: Start with the inequality f(n) ≤ c·g(n). Solve for c or n₀ algebraically. Then verify. Example: prove 3n² + 5n = O(n²). We need 3n² + 5n ≤ c·n². Divide by n²: 3 + 5/n ≤ c. For n ≥ 1, 5/n ≤ 5, so 3 + 5/n ≤ 8. Choose c = 8, n₀ = 1. Done. 📄 pp. 39–41

What is a polynomial vs an exponential function? 📄 pp. 39–42

A polynomial in n has n raised to a fixed constant exponent: n, n², n³, n⁰·⁵ = √n, n^1.5, etc. An exponential has n as the exponent: 2ⁿ, 3ⁿ, 10ⁿ. Exponentials grow incomparably faster than any polynomial — for large enough n, 2ⁿ exceeds n^1000. This is why polynomial-time algorithms are "tractable" and exponential-time ones are not. 📄 pp. 39–42

What is a "tight bound"? 📄 pp. 39–41

A bound is tight when it is both an upper and a lower bound simultaneously — i.e., when f(n) = Θ(g(n)). This means g(n) captures the growth rate of f(n) exactly (up to constant factors). Example: n(n-1)/2 is a tight bound of Θ(n²) — it is both O(n²) and Ω(n²). 📄 pp. 39–41

What does ∀ and ∃ mean? 📄 pp. 39–41

∀ means "for all" — a statement must hold for every value. ∃ means "there exists" — at least one value satisfies the statement. The Big-O definition says: ∃ c > 0, ∃ n₀ ≥ 0 such that ∀ n ≥ n₀, f(n) ≤ c·g(n). In plain English: "We can find some c and n₀ such that, from n₀ onward, f(n) never exceeds c·g(n)." 📄 pp. 39–41

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 23–54 (Lectures 3–6)

1.1 The fundamental problem — comparing algorithms without running them 📄 pp. 23–30

Imagine two taxi companies quote you a price for a trip. Company A charges $10 + $1 per mile. Company B charges $0.50 per mile, but $50 upfront. For a 5-mile trip, A is cheaper ($15 vs $52.50). For a 1,000-mile trip, B is much cheaper ($550 vs $1,010). The crossover point — the trip length at which B becomes better — is roughly 80 miles.

Comparing algorithms is exactly like this. Two algorithms solve the same problem but with different "pricing structures." Algorithm A might do 100n + 5 operations; Algorithm B might do 2n² + 1 operations. For small inputs (n < 50), A might use more operations. For large inputs (n > 50), B is catastrophically slower. Asymptotic analysis formalizes the notion of "which algorithm is better for large inputs." 📄 pp. 23–30 

  Concrete operation counts for two algorithms:

  Input size n  |  Algorithm A: 100n + 5  |  Algorithm B: 2n²  | Winner
  ─────────────────────────────────────────────────────────────────────
  n = 10        |  1,005                  |  200               | B
  n = 50        |  5,005                  |  5,000             | B (≈tie)
  n = 100       |  10,005                 |  20,000            | A
  n = 1,000     |  100,005                |  2,000,000         | A (20×)
  n = 1,000,000 |  100,000,005            |  2,000,000,000,000 | A (20,000×)

  Conclusion: Algorithm A is O(n), Algorithm B is O(n²).
  For large n, the O(n) algorithm wins by an enormous margin.

1.2 O / Ω / Θ — three lenses for the same algorithm 📄 pp. 39–41

We use three different "lenses" to describe an algorithm's growth, each answering a different question:

NotationWhat it saysBound typeReal-world analogyPage
f(n) = O(g(n)) f grows at most as fast as g (up to a constant) Upper bound "Your bill will NEVER exceed c·g(n)" — a maximum guarantee 📄 p. 39
f(n) = Ω(g(n)) f grows at least as fast as g (up to a constant) Lower bound "Your bill will ALWAYS cost at least c·g(n)" — a minimum guarantee 📄 p. 40
f(n) = Θ(g(n)) f grows at exactly the same rate as g (up to constants) Tight bound "Your bill is sandwiched between c₁·g(n) and c₂·g(n)" — precise rate 📄 p. 41
f(n) = o(g(n)) f grows strictly slower than g (ratio → 0) Strict upper "Your bill becomes negligible compared to g(n) for large n" 📄 p. 41
f(n) = ω(g(n)) f grows strictly faster than g (ratio → ∞) Strict lower "Your bill eventually dwarfs g(n) by any factor" 📄 p. 41
📌 Critical Relationship f(n) = Θ(g(n)) if and only if f(n) = O(g(n)) AND f(n) = Ω(g(n)). Think of it as a "sandwich": Θ requires both the upper roof (O) and the lower floor (Ω) to be made of the same material. 📄 p. 41

1.3 The rental contract analogy — a second framing 📄 pp. 39–41

Think of the asymptotic notation as different types of rental contracts for a storage unit:

O (upper bound) = a price CAP. "I guarantee you'll pay no more than c × g(n) dollars from month n₀ onward." The actual price could be much lower — you just know it won't exceed the cap.

Ω (lower bound) = a price FLOOR. "The price will never drop below c × g(n) from month n₀ onward." Prices could be higher — but they won't fall below this floor.

Θ (tight) = a FIXED RATE bracket. "From month n₀ onward, you'll always pay between c₁ × g(n) and c₂ × g(n)." Both the floor and ceiling are set by the same function g(n) — only the constants differ.

⚠️ Most Common Confusion — O is NOT "equals" "O(n)" does NOT mean the algorithm performs exactly n operations. It means the algorithm performs at most c·n operations for large n. An O(n) algorithm might actually run in 0.001·n operations — still O(n). The notation is a bound, not a precise formula. 📄 pp. 39–41

1.4 Technical definitions (using lecture notation) 📄 pp. 39–41

Big-O Definition: f(n) = O(g(n)) if there exist constants c > 0 and n₀ ≥ 0 such that for all n ≥ n₀: f(n) ≤ c · g(n). 📄 p. 39

Big-Ω Definition: f(n) = Ω(g(n)) if there exist constants c > 0 and n₀ ≥ 0 such that for all n ≥ n₀: f(n) ≥ c · g(n). 📄 p. 40

Big-Θ Definition: f(n) = Θ(g(n)) if there exist constants c₁ > 0, c₂ > 0, and n₀ ≥ 0 such that for all n ≥ n₀: c₁ · g(n) ≤ f(n) ≤ c₂ · g(n). Equivalently: f(n) = O(g(n)) AND f(n) = Ω(g(n)). 📄 p. 41

2a Formal O / Ω / Θ Proofs — Finding Witnesses c and n₀ Lecture PDF: pp. 39–41 · Assignments 2 & 4 📋 A2 Q1 📋 A4 Q2

Intuition: Pinning down a growth rate with two constants 📄 pp. 39–41

When we say f(n) = O(g(n)), we're making a bet: "for every input large enough, f(n) fits under the ceiling c·g(n)." To prove it, we need to produce the ceiling height (c) and the input size from which the ceiling holds (n₀). These two numbers are the witnesses. Finding them is the core skill of asymptotic proofs.

The strategy: algebraically manipulate f(n) ≤ c·g(n) to solve for c given a concrete n₀ (or vice versa). Then verify the inequality holds. 📄 pp. 39–41

Mechanism: Big-O formal proof template 📄 pp. 39–41

BIG-O PROOF TEMPLATE: f(n) = O(g(n))
Goal: Find c > 0 and n₀ ≥ 0 such that f(n) ≤ c · g(n) for all n ≥ n₀.
Step 1 [Write the inequality]: f(n) ≤ c · g(n). Substitute the explicit formula for f(n).
Step 2 [Upper bound each term]: Replace each term in f(n) with something ≤ c · g(n). Use the fact that lower-order terms (like 5n) are ≤ the dominant term (n²) for large n.
Step 3 [Choose n₀ so lower-order terms are manageable]: If your upper bound requires n ≥ some value, that value becomes n₀.
Step 4 [State the witnesses explicitly]: "Choose c = [value] and n₀ = [value]." Then verify the inequality holds for all n ≥ n₀.

Worked Example 1: Prove 3n² + 5n = O(n²) 📄 pp. 39–41 📋 A4 Q2

  Goal: find c, n₀ such that 3n² + 5n ≤ c · n² for all n ≥ n₀.

  Step 1: Divide both sides by n² (valid since n² > 0 for n > 0):
          3 + 5/n ≤ c

  Step 2: When n ≥ 1, we have 5/n ≤ 5, so:
          3 + 5/n ≤ 3 + 5 = 8

  Step 3: Therefore, choose c = 8 and n₀ = 1.

  Verification: For all n ≥ 1:
          3n² + 5n ≤ 3n² + 5n²   (since n ≤ n² for n ≥ 1)
                   = 8n²
                   = c · n²  ✓

  Witnesses: c = 8, n₀ = 1.
  Conclusion: 3n² + 5n = O(n²). ✓
SOURCE: 📄 pp. 39–41

Worked Example 2: Prove n(n-1)/2 = Θ(n²) (tight bound) 📄 p. 39

  We need BOTH an upper bound (O) AND a lower bound (Ω).

  ─── Upper bound: n(n-1)/2 = O(n²) ────────────────────────────

  n(n-1)/2 = n²/2 - n/2 ≤ n²/2  (since -n/2 ≤ 0 for n ≥ 0)

  So: n(n-1)/2 ≤ (1/2)·n² for all n ≥ 0.
  Witnesses: c₂ = 1/2, n₀ = 0. ✓

  ─── Lower bound: n(n-1)/2 = Ω(n²) ───────────────────────────

  n(n-1)/2 = n²/2 - n/2
  We need: n²/2 - n/2 ≥ c₁·n²

  Rearrange: n²/2 - n/2 = (n/2)·(n - 1) ≥ (n/2)·(n/2)  when n ≥ 2
             (since n - 1 ≥ n/2 when n ≥ 2)
           = n²/4

  So: n(n-1)/2 ≥ (1/4)·n² for all n ≥ 2.
  Witnesses: c₁ = 1/4, n₀ = 2. ✓

  ─── Combined ──────────────────────────────────────────────────

  (1/4)·n² ≤ n(n-1)/2 ≤ (1/2)·n² for all n ≥ 2.
  Choose c₁ = 1/4, c₂ = 1/2, n₀ = max(0, 2) = 2.
  Conclusion: n(n-1)/2 = Θ(n²). ✓
SOURCE: 📄 p. 39

Worked Example 3: Prove 5n log n + 20n = O(n log n) and = Ω(n log n) → Θ(n log n) 📄 pp. 39–41

  f(n) = 5n log n + 20n.  Claim: f(n) = Θ(n log n).

  ─── Upper bound (O) ──────────────────────────────────────────

  5n log n + 20n ≤ 5n log n + 20n log n   (since n ≤ n log n for n ≥ 2)
                = 25n log n

  Witnesses: c₂ = 25, n₀ = 2. ✓

  ─── Lower bound (Ω) ──────────────────────────────────────────

  5n log n + 20n ≥ 5n log n               (since 20n ≥ 0)

  Witnesses: c₁ = 5, n₀ = 1. ✓

  Conclusion: 5n log n + 20n = Θ(n log n). ✓

  Verification at n = 64 (log₂ 64 = 6):
  f(64) = 5·64·6 + 20·64 = 1920 + 1280 = 3200
  c₁·g(64) = 5·64·6 = 1920 ≤ 3200 ✓
  c₂·g(64) = 25·64·6 = 9600 ≥ 3200 ✓
SOURCE: 📄 pp. 39–41
⚠️ Trap: Tightening Ω proofs For the lower bound (Ω), you need f(n) ≥ c·g(n). A common mistake is dropping terms that are POSITIVE — which makes the left side smaller and potentially breaks the inequality. You can drop negative terms, but never drop positive ones. If you drop a positive term, it should be to achieve a valid lower bound (e.g., "f(n) = A + B ≥ A" only works if you've already shown A ≥ c·g(n)). 📄 pp. 39–41
2b The Limit Comparison Method — A Faster Tool for Growth Rate Analysis Lecture PDF: pp. 39–42 · Assignments 2 & 4 📋 A4 Q1

Intuition: Asking the ratio "who wins the race?" 📄 pp. 39–42

The witness-based proof is rigorous but slow. The limit comparison method gives the same information much faster: compute limn→∞ f(n)/g(n) and read off the relationship from the result. Think of it as asking: "As n → ∞, who wins the race between f and g? By how much?"

Mechanism: The three cases 📄 pp. 39–42

  LIMIT COMPARISON DECISION TABLE:

  Compute L = lim  f(n) / g(n)
               n→∞

  ┌──────────────┬──────────────────────────────────────────────────┐
  │  L = 0       │  f(n) = o(g(n)) — f grows STRICTLY SLOWER        │
  │              │  Also: f(n) = O(g(n)) is true                    │
  │              │  But: f(n) ≠ Θ(g(n)) (no tight bound)            │
  ├──────────────┼──────────────────────────────────────────────────┤
  │  0 < L < ∞  │  f(n) = Θ(g(n)) — SAME growth rate               │
  │  (L finite,  │  Also: f(n) = O(g(n)) AND f(n) = Ω(g(n))        │
  │   nonzero)   │  c₁ = L/2, c₂ = 2L are valid witnesses           │
  ├──────────────┼──────────────────────────────────────────────────┤

2c True/False Asymptotic Claims — Proving and Disproving with Counterexamples Lecture PDF: pp. 43–45 · Assignments 2 & 4 📋 A2 Q1 📋 A4 Q4

Intuition: The three exercise types and what they test 📄 pp. 43–45

Asymptotic claims come in three kinds: (1) claims that are TRUE — requiring a formal proof via witnesses or limits. (2) Claims that are FALSE — requiring a counterexample (one specific f and g that violate the claim). (3) Claims that follow from transitivity — chain inequalities. The Assignment 2 Q1 exercises and the Quiz 1 questions follow this exact structure. 📄 pp. 43–45

Claim 1 (TRUE): If f=O(g) and f=Ω(h), then g=Ω(h) 📄 pp. 43–45 📋 A2 Q1-1

PROOF (direct, using witnesses)
Given: f(n) = O(g(n)) → ∃ c₁, n₁ such that f(n) ≤ c₁·g(n) for all n ≥ n₁. 📄 p. 43
Given: f(n) = Ω(h(n)) → ∃ c₂, n₂ such that f(n) ≥ c₂·h(n) for all n ≥ n₂. 📄 p. 43
Combining: For all n ≥ max(n₁, n₂): c₂·h(n) ≤ f(n) ≤ c₁·g(n). So c₂·h(n) ≤ c₁·g(n). 📄 p. 43
Rearrange: h(n) ≤ (c₁/c₂)·g(n). This shows g(n) ≥ (c₂/c₁)·h(n). 📄 p. 44
Conclusion: Set c = c₂/c₁ and n₀ = max(n₁, n₂). Then g(n) ≥ c·h(n) for all n ≥ n₀. So g(n) = Ω(h(n)). ✓ 📄 p. 44
  Concrete verification (n=100, using f=n², g=n³, h=n):
    f=O(g): n² ≤ n³ for n ≥ 1 ✓ (c₁=1)
    f=Ω(h): n² ≥ n for n ≥ 1  ✓ (c₂=1)
    Claim: g=Ω(h): n³ ≥ n for n ≥ 1 ✓ (c=1)  ← theorem holds

  The key insight: f is sandwiched between h (below) and g (above),
  so g must be above h as well — g ≥ f ≥ c·h.
SOURCE: 📄 pp. 43–44

Claim 2 (TRUE): If f=O(g), then f²=O(g²) 📄 pp. 43–45 📋 A2 Q1-2

PROOF (square both sides)
Given: f(n) = O(g(n)) → ∃ c, n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀. 📄 p. 43
Square both sides (both sides are non-negative for large n): f(n)² ≤ (c·g(n))² = c²·g(n)². 📄 p. 44
Conclusion: Set c' = c². Then f(n)² ≤ c'·g(n)² for all n ≥ n₀. So f² = O(g²). ✓ 📄 p. 44
  Concrete example: f = 3n² + 5n = O(n²) with c=8, n₀=1.
  Then f² = (3n² + 5n)² = O((n²)²) = O(n⁴), with c' = 64, n₀ = 1.

  Check at n=10: f²=(300+50)²=350²=122,500. c'·(n²)²=64·10,000=640,000. ✓
  Check at n=100: f²=(30,000+500)²=915,250,000. c'·n⁴=64·100,000,000=6,400,000,000. ✓

Claim 3 (FALSE): If f=Θ(g), then log f = Θ(log g) 📄 pp. 43–45 📋 A2 Q1-3

⚠️ HIGH EXAM PRIORITY — This appeared on the midterm and is the most-missed claim This claim looks intuitively true (logs are monotone, so they should preserve order). But it is FALSE. The counterexample is subtle and specific: you need f and g that are Θ of each other but whose log values are NOT in a constant ratio. The key is that logs are not homogeneous — they collapse multiplicative constants to additive ones. 📄 pp. 43–45
COUNTEREXAMPLE CONSTRUCTION
Choose: f(n) = 2 (constant), g(n) = 1 + 1/n. 📄 p. 45
Step 1 — Show f = Θ(g):
Since 1 ≤ g(n) = 1 + 1/n ≤ 2 for all n ≥ 1, we have: g(n)/2 ≤ 1 ≤ g(n), which gives 1 = Θ(g(n)).
Also f(n) = 2 = Θ(1) = Θ(g(n)). So f = Θ(g). ✓ 📄 p. 45
Step 2 — Compute log f(n): log f(n) = log 2 = 1 (a constant). 📄 p. 45
Step 3 — Compute log g(n): log g(n) = log(1 + 1/n) → 0 as n → ∞. 📄 p. 45
Step 4 — Compare: lim (log f(n)) / (log g(n)) = lim 1 / log(1 + 1/n) = ∞. So log f ≠ O(log g). 📄 p. 45
Conclusion: f = Θ(g) but log f ≠ Θ(log g). The claim is FALSE. ✗ 📄 p. 45
  Why does this happen? Because:
  log(c · g(n)) = log c + log g(n) ≠ c' · log g(n)

  The log of a constant multiple is a SHIFTED version, not a SCALED version.
  When log g(n) → 0, even a tiny additive shift (log c = log 2 = 1)
  makes the ratio (log f)/(log g) = 1/(log g) blow up to infinity.

  Concrete table for n = 1, 10, 100, 1000:
  ┌────────┬─────────┬──────────────┬──────────────┬───────────────────────┐

2d Function Growth Ordering — Iterative Complexity Analysis of Code Lecture PDF: pp. 23–35, 39–54 · Assignment 4 📋 A4 Q4 📋 A4 Q7–Q9

Intuition: Reading algorithm complexity from code structure 📄 pp. 23–35

Every loop, nested loop, and recursive call contributes to the overall running time. The key insight: to find the complexity of iterative code, identify the innermost operation and count how many times it runs. The total is determined by the nesting structure.

Mechanism: The loop-counting rules 📄 pp. 23–35

  RULE 1: Single loop from 0 to n-1 → O(n)
  RULE 2: Nested loops, each from 0 to n-1 → O(n^k) for k nested loops
  RULE 3: Loop that halves the problem each iteration → O(log n)
  RULE 4: Sequential code blocks → add their complexities
  RULE 5: Multiplicative nesting → multiply complexities
  RULE 6: Ignore additive lower-order terms and constant multipliers
SOURCE: 📄 pp. 23–35

Worked Example: Analyze 3 iterative examples from Assignment 4 📄 pp. 23–35

  ─── Example A: Sliding Window Maximum Average (A4 Q7) ────────
  function maxAverage(nums, k):
    window_sum = sum(nums[0:k])         # O(k) = O(n) in worst case
    max_avg = window_sum / k
    for i in range(k, len(nums)):        # runs n-k times ≈ O(n)
      window_sum += nums[i]              # O(1)
      window_sum -= nums[i-k]            # O(1)
      max_avg = max(max_avg, window_sum/k)  # O(1)
    return max_avg

  Analysis:
    Init sum: O(k) ≤ O(n)
    Loop body: O(1) per iteration × (n-k) iterations = O(n)
    Total: O(n) + O(n) = O(n).  Space: O(1) extra.

  Why NOT O(nk)? Because we SLIDE the window — we remove one element
  and add one element instead of recomputing the sum from scratch.

  ─── Example B: Hamming Distance Sum (A4 Q9) ────────────────
  function totalHammingDistance(nums):
    total = 0
    for bit in range(32):               # 32 iterations (constant)
      ones = sum(1 for x in nums if x>>bit & 1)  # O(n)
      zeros = len(nums) - ones          # O(1)
      total += ones * zeros             # O(1)
    return total

  Analysis:
    Outer loop: 32 iterations (constant = O(1))
    Inner operation: O(n) per bit
    Total: 32 × O(n) = O(n).  Space: O(1).
    Key insight: instead of comparing all O(n²) pairs,
    we count 1-bits per position and use combinatorics.

  ─── Example C: Minimum Time Difference (A4 Q8) ──────────────
  function findMinDifference(timePoints):
    minutes = sorted([h*60+m for t in timePoints for h,m in [map(int,t.split(':'))]])
    diffs = [minutes[i+1]-minutes[i] for i in range(len(minutes)-1)]
    diffs.append(1440 - minutes[-1] + minutes[0])  # circular diff
    return min(diffs)

  Analysis:
    Convert: O(n)
    Sort: O(n log n)   ← DOMINANT TERM
    Compute diffs: O(n)
    Find min: O(n)
    Total: O(n log n).  Space: O(n).

Misconception Cards 📄 pp. 39–54

❌ Misconception: "O(n) means the algorithm always takes exactly n steps."

✅ Reality: O(n) means the runtime is bounded ABOVE by c·n for some constant c and large n. The actual runtime could be 0.001n or 50n or even O(1) on a particular input. O is an upper bound, not an exact count. 📄 pp. 39–41

❌ Misconception: "O is always worst-case, Ω is always best-case."

✅ Reality: O, Ω, Θ describe the growth of a specific function, not input cases. You can have an O bound on the best case, or an Ω bound on the worst case. They are properties of FUNCTIONS, not cases. However, O is commonly used for worst-case analysis in practice. 📄 pp. 39–41

❌ Misconception: "If f=O(g), then f is always smaller than g."

✅ Reality: f can be much larger than g for small n. The O-bound only kicks in from n₀ onward. Before n₀, anything can happen. That's why we say "for sufficiently large n." 📄 pp. 39–41

❌ Misconception: "n² = O(n³) means n² and n³ have the same growth rate."

✅ Reality: n² = O(n³) is TRUE but it's not a tight bound. It says "n² grows at most as fast as n³," which is true but weak. The tight bound would be n² = Θ(n²), NOT Θ(n³). O allows the bound to be loose. 📄 pp. 39–41

❌ Misconception: "Θ(g(n)) means the algorithm uses exactly g(n) steps."

✅ Reality: Θ(g(n)) means the runtime is SANDWICHED between c₁·g(n) and c₂·g(n) for large n. There can be constant-factor differences. n²/4 and 100n² are both Θ(n²). 📄 p. 41

❌ Misconception: "log is preserved under Θ — if f=Θ(g), then log f = Θ(log g)."

✅ Reality: FALSE — see the counterexample f=2, g=1+1/n in §2c. Log turns multiplicative constants into additive shifts, which breaks the Θ relationship when log g → 0. This is the most-tested FALSE claim in CS 570. 📄 pp. 43–45

❌ Misconception: "To prove a claim FALSE, I just need to say 'this doesn't look right.'"

✅ Reality: A FALSE proof requires a COMPLETE counterexample: (1) specific f and g, (2) verification that the hypothesis holds (e.g., f=Θ(g) must be verified), and (3) proof that the conclusion fails (e.g., log f ≠ Θ(log g) must be shown via limits). All three parts are required for full credit. 📄 pp. 43–45

WARM-UP CONCEPTUAL 📄 pp. 39–41

WC-1. A classmate says: "O(n²) means the algorithm performs exactly n² operations." Explain what they got right and what they got wrong. Give a concrete function that is O(n²) but is NOT Θ(n²).

▶ Show Explanation

What they got right: O(n²) does communicate that n² is the relevant benchmark — it's the "ballpark" of the runtime. The algorithm's runtime won't exceed some constant multiple of n². 📄 pp. 39–41

What they got wrong: O(n²) is an UPPER BOUND, not an exact count. It means f(n) ≤ c·n² for some c and large n. The actual runtime could be n log n (much less than n²) and still be O(n²). 📄 pp. 39–41

Concrete example: f(n) = n is O(n²) because n ≤ 1·n² for all n ≥ 1 (choose c=1, n₀=1). But f(n) = n is NOT Θ(n²) because n/n² = 1/n → 0 as n → ∞, so n grows strictly slower than n². 📄 pp. 39–42

WARM-UP WITNESS FINDING 📄 pp. 39–41

WC-2. Find witnesses c and n₀ to formally prove that 7n + 100 = O(n). Show every algebraic step.

▶ Show Explanation

Goal: Find c > 0 and n₀ ≥ 0 such that 7n + 100 ≤ c·n for all n ≥ n₀. 📄 pp. 39–41

Step 1 — Divide both sides by n (valid for n > 0): 7 + 100/n ≤ c.

Step 2 — Bound 100/n: If n ≥ 100, then 100/n ≤ 1. So 7 + 100/n ≤ 7 + 1 = 8.

Step 3 — Choose witnesses: c = 8, n₀ = 100.

Verify: For all n ≥ 100: 7n + 100 ≤ 7n + n = 8n. ✓

Conclusion: 7n + 100 = O(n) with witnesses c = 8, n₀ = 100. 📄 pp. 39–41

WARM-UP LIMIT COMPUTATION 📄 pp. 39–42

WC-3. Use the limit comparison method to determine the relationship between f(n) = n² and g(n) = n log n. Is f = O(g), f = Ω(g), f = Θ(g), or none of the above?

▶ Show Explanation

Compute the limit: limn→∞ f(n)/g(n) = lim n² / (n log n) = lim n / (log n). 📄 pp. 39–42

Evaluate: As n → ∞, n grows without bound while log n grows much more slowly. The ratio n / log n → ∞. (Formally: L'Hôpital: lim n/log n = lim 1/(1/n) = lim n = ∞.)

L = ∞, so: f(n) = ω(g(n)) — f grows STRICTLY FASTER than g. This also means f = Ω(g). But f ≠ O(g) and f ≠ Θ(g). 📄 pp. 39–42

Conclusion: n² = Ω(n log n) — quadratic is strictly larger than linearithmic. 📄 p. 40

WARM-UP MISCONCEPTION PUNCTURE 📄 pp. 43–45

WC-5. Your classmate says: "Obviously if f=Θ(g), then log f = Θ(log g), because log is a monotone function so it preserves order." What is wrong with this reasoning? Give the specific counterexample.

▶ Show Explanation

Why the reasoning fails: "Preserving order" means log f ≤ log g when f ≤ g. But Θ doesn't say f ≤ g — it says c₁·g ≤ f ≤ c₂·g. Taking logs: log c₁ + log g ≤ log f ≤ log c₂ + log g. This gives ADDITIVE shifts (log c₁ and log c₂), not MULTIPLICATIVE ones. Θ requires c₁·(log g) ≤ log f ≤ c₂·(log g), which is a multiplicative bound. An additive shift ≠ multiplicative scaling when log g → 0. 📄 pp. 43–45

The specific counterexample: f(n) = 2, g(n) = 1 + 1/n. 📄 p. 45

  • f = Θ(g): since 1 ≤ g(n) ≤ 2 for all n ≥ 1, we have g/2 ≤ 1 ≤ g, so 1 = Θ(g). And f = 2 = Θ(1) = Θ(g). ✓
  • log f = log 2 = 1 (constant). log g = log(1 + 1/n) → 0 as n → ∞.
  • lim (log f)/(log g) = lim 1/log(1+1/n) = ∞. So log f ≠ O(log g). Thus log f ≠ Θ(log g). ✗

The claim is FALSE because log collapses multiplicative constants (which define Θ) into additive ones. 📄 pp. 43–45

Q1. Which of the following is the correct formal definition of f(n) = O(g(n))?

  1. f(n) < g(n) for all n ≥ 0
  2. ∃ c > 0, n₀ ≥ 0 such that f(n) ≤ c · g(n) for all n ≥ n₀
  3. ∀ c > 0, ∃ n₀ ≥ 0 such that f(n) ≤ c · g(n) for all n ≥ n₀
  4. limn→∞ f(n)/g(n) = 0
▶ Show Answer

✅ Correct Answer: B

The formal Big-O definition requires finding constants c and n₀ that EXIST (∃) — not that the bound holds for ALL constants. This is the standard lecture-slide definition. 📄 pp. 39–41

Why A is wrong: f(n) < g(n) for all n is too strong (and doesn't match asymptotic behavior — f can exceed g for small n).

Why C is wrong: swapping ∃ and ∀ gives the little-o definition (o), not Big-O. Little-o requires the bound to hold for EVERY c (even very small ones).

Why D is wrong: lim f/g = 0 gives the little-o (o) definition, which is strictly stronger than O. O includes Θ cases where the limit is a nonzero constant.

Exam Tip: The quantifier order ∃c, n₀ followed by ∀n ≥ n₀ is fixed. Memorize: "There exist constants such that for all large n, the bound holds." 📄 p. 39

Q3. Using the limit comparison method, what is the relationship between f(n) = n(log n)² and g(n) = n^(4/3)?

  1. f(n) = Θ(g(n)) — they grow at the same rate
  2. f(n) = O(g(n)) but f(n) ≠ Θ(g(n)) — f grows strictly slower

Q4. Consider the function h(n) = (log n)^(log n). Which of the following correctly classifies h(n)?

  1. h(n) = O(n) — h grows at most linearly
  2. h(n) = Θ(n^(log log n)) — tight superpolynomial bound
  3. h(n) = Ω(2ⁿ) — h grows at least exponentially
  4. h(n) = Θ(n^k) for some constant k — h is polynomial
▶ Show Answer

✅ Correct Answer: B

Rewrite h(n): h(n) = (log n)^(log n). Take log: log h(n) = (log n)·(log log n). Now compare log h to log n (the "exponent of n"): log h / log n = log log n → ∞. So h grows faster than any fixed polynomial n^k (since log h/log n → ∞ means h = n^(something growing)). 📄 pp. 39–42

Concretely: h(n) = (log n)^(log n) = n^(log log n), since (log n)^(log n) = 2^((log n)(log log n)) = (2^(log n))^(log log n) = n^(log log n). 📄 pp. 39–42

So h(n) = Θ(n^(log log n)). The exponent log log n grows very slowly (for n=2^64, log log n = 6), making this superpolynomial but far below exponential.

Why A is wrong: h is superpolynomial — much larger than n for large n.

Why C is wrong: 2ⁿ grows far faster than n^(log log n); log(2ⁿ)/log n = n/log n → ∞ while log h / log n = log log n → ∞ but far more slowly.

Why D is wrong: log log n is not a constant, so h is not polynomial.

Exam Tip: The "log trick" — rewrite (f(n))^(g(n)) = n^(g(n)·log f(n)/log n) — is a powerful way to classify unusual exponential forms. 📄 pp. 39–42

Q11. Write Python-like pseudocode for finding the maximum average subarray of length k (the sliding window problem from A4 Q7). Explain your approach in plain English, then analyze time and space complexity.

▶ Show Answer

✅ Pseudocode:

function maxAverage(nums, k):
    # Initialize window sum for the first k elements
    window_sum = sum(nums[0:k])       # O(k) — compute first window
    max_sum = window_sum

    # Slide the window from position k to n-1
    for i from k to len(nums)-1:     # runs n-k times
        window_sum += nums[i]        # add new element entering window
        window_sum -= nums[i-k]      # remove element leaving window
        max_sum = max(max_sum, window_sum)   # update max

    return max_sum / k               # return maximum average
SOURCE: 📄 pp. 23–35 📋 A4 Q7

English Explanation:

  1. Initialize: Compute the sum of the first k elements. This is our starting window.
  2. Slide: For each subsequent position, add the new element entering the window from the right and subtract the element leaving from the left. This maintains the current window sum in O(1) per step.
  3. Track maximum: After each slide, compare to the running maximum.
  4. Return: Divide the maximum sum by k to get the maximum average.

📝 Complexity Analysis:

Time: Initial sum = O(k) ≤ O(n). Loop runs n-k times, each iteration O(1). Total: O(n). 📄 pp. 23–35

Space: Only window_sum, max_sum, i are stored. O(1) extra space. 📄 pp. 23–35

Why not O(nk)? A naïve approach would recompute the sum of each k-window from scratch (O(k) per window × O(n) windows = O(nk)). Sliding avoids this by maintaining a running sum. 📄 pp. 23–35

Q12. The total Hamming distance between all pairs of n integers can be computed in O(n) rather than the naïve O(n²). Write pseudocode for the O(n) approach (the bit-position counting method from A4 Q9). Analyze time and space complexity and explain why it avoids the O(n²) pairwise comparison.

▶ Show Answer

✅ Pseudocode:

function totalHammingDistance(nums):
    total = 0
    n = len(nums)

    for bit in range(32):            # 32 bit positions (constant)
        ones = 0
        for x in nums:               # count 1-bits at this position
            if (x >> bit) & 1 == 1:
                ones += 1
        zeros = n - ones             # remaining are 0-bits
        total += ones * zeros        # each 1-bit pairs with each 0-bit

    return total
SOURCE: 📄 pp. 23–35 📋 A4 Q9

English Explanation:

  1. For each of the 32 bit positions (bit 0 through bit 31), count how many numbers have a 1-bit at that position. Call this count `ones`.
  2. The remaining `zeros = n - ones` numbers have a 0-bit at that position.
  3. Each pair (a 1-bit number, a 0-bit number) contributes 1 to the Hamming distance at this bit. There are ones × zeros such pairs.
  4. Sum across all 32 bit positions for the total Hamming distance.

Concrete example (nums = [4, 14, 2], i.e., [0100, 1110, 0010]):

📝 Complexity Analysis:

Time: Outer loop: 32 iterations (constant). Inner loop: O(n) per bit position. Total: 32 × O(n) = O(n). 📄 pp. 23–35

Space: Only `total`, `bit`, `ones`, `zeros`, `x` are stored. O(1) extra. 📄 pp. 23–35

Why not O(n²)? Instead of comparing every pair (O(n²) pairs, each comparison O(1) = O(n²) total), we use the combinatorial insight: the number of pairs differing at bit k is exactly (# with 1 at k) × (# with 0 at k). One pass over n numbers per bit position → O(n) per bit → O(32n) = O(n). 📄 pp. 23–35

Q13. The minimum time difference problem (A4 Q8) requires finding the smallest circular difference between any two times in a list. Write pseudocode for an O(n log n) solution. Then answer: could you solve this problem in O(n)? Justify your answer using asymptotic reasoning about comparison-based sorting.

▶ Show Answer

✅ Pseudocode:

function findMinDifference(timePoints):
    # Convert all times to minutes since midnight
    minutes = []
    for t in timePoints:              # O(n)
        h, m = parse(t)               # e.g., "14:30" → h=14, m=30
        minutes.append(h * 60 + m)

    # Sort the minute values
    minutes.sort()                    # O(n log n) — DOMINANT TERM

    # Compute minimum of consecutive differences + circular diff
    min_diff = 1440                   # 1440 = 24 * 60 (max possible)
    n = len(minutes)
    for i in range(n - 1):           # O(n)
        diff = minutes[i+1] - minutes[i]
        min_diff = min(min_diff, diff)

    # Handle circular: distance from last back to first
    circular = 1440 - minutes[-1] + minutes[0]
    min_diff = min(min_diff, circular)

    return min_diff
SOURCE: 📄 pp. 23–35 📋 A4 Q8

English Explanation:

  1. Convert each time string to minutes (e.g., "14:30" → 870 minutes). O(n).
  2. Sort the minute values. O(n log n). Sorting is necessary to put consecutive times next to each other.
  3. Scan adjacent pairs in sorted order. The minimum difference must be between consecutive sorted values OR the circular wrap-around (last to first via midnight). O(n).
  4. Return the overall minimum. The circular difference handles "23:58" vs "00:02" (difference = 4 min, not 1,436 min).

📝 Complexity Analysis:

Time: Convert O(n) + Sort O(n log n) + Scan O(n) = O(n log n) dominated by sort. 📄 pp. 23–35

Space: The minutes array = O(n). O(n). 📄 pp. 23–35

Could this be solved in O(n)? Not using comparison-based sorting, no. Here's the asymptotic argument:

  • If we need to find the minimum difference between any two elements, we need to know their relative order. 📄 pp. 39–42
  • Any comparison-based sorting algorithm requires Ω(n log n) comparisons (a known lower bound for comparison sorting). 📄 pp. 79–90
  • Finding minimum difference requires effectively sorting the data (if you know the sorted order, minimum difference is trivial in O(n); if you don't sort, you need O(n²) comparisons).
  • Exception: If the value range is bounded (e.g., times are always 0–1439 minutes), we could use counting sort in O(n + 1440) = O(n). Pseudocode: create a boolean array of size 1440; mark each time; scan for minimum gap. This is O(n) time and O(1440) = O(1) space. 📄 pp. 23–35

Exam Tip: When the input range is bounded by a constant (like 1440 minutes in a day or 32 bits in an integer), O(n) solutions often exist using bucket/counting sort techniques. When input range is unbounded, the Ω(n log n) comparison-sort lower bound applies. 📄 pp. 39–42

§5 — Topic Cheat Sheet

📘 SOURCE: Lectures 3–6, pp. 23–54 · Assignments 2 & 4

IN ONE SENTENCE, THIS TOPIC IS:

Asymptotic analysis provides a mathematical vocabulary — O, Ω, Θ — for comparing how fast algorithms scale as input size grows, ignoring constant factors and focusing on growth rates. 📄 pp. 39–41

THE THREE THINGS YOU MUST KNOW FOR THE EXAM:

  1. Formal definitions with witnesses: f = O(g) iff ∃ c > 0, n₀ ≥ 0 such that f(n) ≤ c·g(n) for all n ≥ n₀. For Θ, need both O and Ω. Always state c and n₀ explicitly. 📄 pp. 39–41
  2. Limit comparison method: Compute lim f/g. L=0 → f=O(g); L=finite nonzero → f=Θ(g); L=∞ → f=Ω(g). The fastest tool for comparing growth rates. 📄 pp. 39–42
  3. The FALSE claim: f=Θ(g) does NOT imply log f = Θ(log g). Counterexample: f=2, g=1+1/n. Verify f=Θ(g), then show log f=1 while log g→0. 📄 pp. 43–45
