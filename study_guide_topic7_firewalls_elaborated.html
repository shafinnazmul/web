🧗 Warm-up Glossary — The Networking Vocabulary You Need First

Firewall lectures throw around networking terms that earlier lectures introduced but you might not have fully internalized. None are conceptually hard — they just need a patient definition. If you're already comfortable with packets, ports, OSI layers, and TCP flags, skip ahead to §1.

What is a "packet"?

When your laptop sends data over the Internet, the data isn't sent as one continuous stream — it's chopped into small chunks called packets, typically 1.5 KB or less each. Every packet carries two things: a header with metadata (where it's going, where it came from, which protocol, etc.) and a payload with the actual data. 

  A typical TCP/IP packet (simplified):

  ┌──────────────────────────────────────────────┐
  │  IP Header  │  TCP Header  │  Payload (data) │
  │  ─────────  │  ──────────  │  ─────────────  │
  │  src IP     │  src port    │  HTTP request,  │
  │  dst IP     │  dst port    │  email content, │
  │  protocol   │  flags(SYN…) │  file bytes,    │
  │  TTL        │  seq num     │  whatever       │
  │  ...        │  ack num     │                 │
  └──────────────────────────────────────────────┘

  The HEADERS are what firewalls inspect. The PAYLOAD is what
  application/proxy firewalls also inspect.

What are "ports"?

An IP address identifies a machine; a port identifies a specific service running on that machine. Port numbers range from 0 to 65535. Some are standardized:

  • Port 22 — SSH (secure shell, remote login)
  • Port 23 — Telnet (insecure remote login, mostly historical)
  • Port 25 — SMTP (email sending)
  • Port 53 — DNS (domain name resolution)
  • Port 80 — HTTP (regular web traffic)
  • Port 443 — HTTPS (secure web traffic)

When the lecture says "block port 23," it means: drop any packet whose destination TCP port number equals 23. That's exactly how a firewall stops Telnet traffic.

What are "TCP flags"? 📄 p. 408

TCP packets carry flags — single-bit control signals embedded in the TCP header. The most important ones for firewall rules:

  • SYN — "Start a new connection." First packet of every TCP handshake has SYN=1.
  • ACK — "I acknowledge what you sent." Set on most packets after the initial SYN.
  • FIN — "I'm done sending; close connection gracefully."
  • RST — "Reset! Tear down this connection immediately."
  • PSH — "Push this data to the application now, don't buffer."

Stateful firewalls watch these flags to track conversation state: SYN means "new connection starting," ACK means "ongoing conversation," FIN/RST means "conversation ending."

What is the "OSI model" — and what are L3, L4, L7?

Networking is organized into layers, each handling a different level of abstraction. The lecture refers to these by number:

LayerNameWhat it deals withExamples
L1PhysicalWires, radio waves, voltageEthernet cables, Wi-Fi
L2Data LinkFrames within one local networkMAC addresses, ARP
L3NetworkRouting packets between networksIP addresses
L4TransportConnections between processesTCP, UDP, ports, flags
L5/6Session/Presentation(usually merged into L7 nowadays)TLS handshake
L7ApplicationWhat the user actually seesHTTP, SMTP, DNS, FTP

When the lecture says "a stateless firewall operates at L3+L4," it means: it inspects IP addresses (L3) and TCP/UDP ports + flags (L4), but never looks higher. "Application firewall operates at L7" means: it parses the actual HTTP request, the SMTP envelope, the DNS query — full application content.

What is "stateless" vs "stateful"?

In computing, state means "remembered information from previous events." Stateless means each event is treated independently — no memory; stateful means the system remembers what's been happening.

  • Stateless firewall — looks at each packet in isolation, no memory of past packets. Decision = pure function of the current packet's headers.
  • Stateful firewall — keeps a "connection state table" tracking every active TCP/UDP flow. Decision can depend on whether this packet belongs to an existing conversation.

Both have rules. The difference is whether the rules can refer to "what happened before." Stateful is more powerful but uses more memory and slightly more processing time.

What is a "proxy"?

A proxy is a server that sits in the middle of a connection, breaking what would have been one direct connection into two connections — one between the client and the proxy, another between the proxy and the destination. The client never directly talks to the destination; everything goes through the proxy. 

  Without a proxy:                With a proxy:

   Client ───► Server              Client ───► Proxy ───► Server
   (one TCP connection)            (two separate TCP connections)

Proxies can do useful things in the middle: log everything, filter content, cache popular responses, anonymize the source. Application/proxy firewalls are exactly this — proxies that enforce a security policy at the application layer.

What is "encryption" in firewall context?

Encryption scrambles data so it looks like noise to anyone without the right key. Once data is encrypted, a firewall in the middle cannot inspect its content — only its outer wrapper (IP and TCP headers, which stay unencrypted because routers need to read them).

This is why HTTPS, SSH, and VPN traffic defeats most firewall content inspection: the firewall can see that SSH or HTTPS is happening and where, but not what is being said. This becomes important in the §2d evasion section.

What does "loadable kernel module" (LKM) mean? 📄 p. 409

Linux's kernel — the core of the operating system — supports loading code at runtime via Loadable Kernel Modules (LKMs). You write C code that defines hooks, compile it into a .ko file, and load it with insmod. This is how custom packet filters get installed without rebooting. Netfilter uses LKMs as the building blocks of every iptables rule.

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 397-407 (the conceptual foundation before getting into mechanism)

1.1 Why does a network need a firewall? 📄 pp. 400-401

Imagine you run a small office with 50 employees. Inside the office, computers can freely share files, talk to each other, and access the printer. That's intentional — they're all trusted members of your network.

Now connect the office to the Internet. Suddenly there are billions of computers out there, almost none of which you trust. Some are benign (legitimate websites your employees visit), some are neutral (random servers somewhere), and some are actively hostile (attackers scanning for vulnerable services to exploit).

Without any protection, every computer on the Internet can reach every computer in your office. Your printer is reachable. Your file server is reachable. Your employees' laptops are reachable. An attacker who finds one vulnerable service inside has free run of your network.

A firewall is the policy enforcement point at the boundary between your trusted network and the untrusted Internet. Slide 400 puts it precisely: "a choke point of control and monitoring." Every packet trying to enter or leave must pass through the firewall, and the firewall decides — based on a security policy you configured — whether to let it through.

1.2 The four requirements every firewall must meet 📄 pp. 400-401

Slide 401 lists the conditions that make a firewall actually work. Memorize these — they appear on the exam:

RequirementWhat it meansWhat goes wrong without it
① All traffic must pass through the firewall There's no "back door" — every packet between trusted and untrusted networks goes through this single choke point. If there's a bypass route (a forgotten modem line, a Wi-Fi access point, a USB-tethered phone), packets escape the firewall and the policy doesn't apply.
② Only authorized traffic is allowed The security policy explicitly defines what can pass; everything else is dropped (or rejected). If the default is "allow everything not explicitly denied," any oversight in your rules creates a hole. Modern best practice: default DENY, explicit ALLOW.
③ The firewall itself is immune to penetration Hardened OS, minimal services running, kept up to date. The firewall is a high-value target — if it's compromised, your entire network is compromised. If an attacker breaks into the firewall machine itself, they can disable the policy from the inside.
④ Provides perimeter defense Sits at the boundary between trust zones, not in the middle of one zone. A firewall placed inside your trusted network protects nothing — packets can already be there.

1.3 What does a firewall actually do with each packet? 📄 p. 403

For every packet that arrives, the firewall consults its rule list and applies one of three actions. These are the only three options:

  • Accept — let the packet through. It continues to its destination as normal.
  • Deny (also called Drop) — silently discard the packet. The sender hears nothing back; from their perspective the packet just vanished into the void. Stealth: the attacker can't easily tell if the firewall is even there or if the destination is just unreachable.
  • Reject — discard the packet AND send back an ICMP error message saying "destination unreachable." More user-friendly: a legitimate user knows their connection failed instead of timing out. But it leaks information: the attacker now knows a firewall is filtering at this address.
Deny vs Reject — the tradeoff Both stop the packet. The difference is whether the sender finds out:
  • Deny (Drop) = better for security (stealth, attackers see nothing useful)
  • Reject = better for usability (legitimate users get fast clear failures)
Most public-facing firewalls use Deny by default; internal firewalls (where users need feedback) often use Reject. 📄 p. 403

1.4 Ingress vs Egress — direction matters 📄 p. 403

This is one of the most-tested distinctions in Topic 7. The terms come from Latin ingredi (to enter) and egredi (to exit):

  • Ingress filtering — inspects incoming traffic. Goal: stop external attacks from reaching internal services. Example: drop incoming Telnet, drop port-scan probes, drop packets claiming to come from internal IP ranges (which would be a spoof).
  • Egress filtering — inspects outgoing traffic. Two purposes: (a) prevent internal users from reaching disallowed destinations (e.g., schools blocking social media); (b) stop spoofed packets from leaving your network (the IP-spoofing defense from Topic 2).

The instructor was emphatic in the final review that egress filtering at ISPs is the most effective defense against IP spoofing — because if every ISP refused to forward packets with source addresses outside their customer's IP range, IP-spoofing attacks would be impossible. (Sadly not all ISPs do this; that's why DDoS amplification attacks still work.) 📄 pp. 403, 512

1.5 The three-way classification — choosing how much context to inspect 📄 p. 404

Slide 404 introduces the central taxonomy: firewalls are classified by how much context they retain when deciding whether to allow a packet. More context = smarter decisions = more resources used. The three types:

TypeSees what?SpeedSmart?Best for
Stateless (Packet Filter) Headers of the current packet only (L3 + L4). Nothing else. Fastest Limited High-throughput edge routing, simple block lists
Stateful Headers + connection state table tracking active flows Fast Better Standard enterprise perimeter
Application / Proxy Everything — headers + full payload (L7) Slowest Smartest Web application firewalls, email gateways, deep inspection

In practice, real enterprise networks use multiple firewalls in series, each at a different inspection depth. A stateless edge router quickly drops obvious garbage; a stateful perimeter firewall handles connection tracking; an application firewall in front of the web servers does deep inspection of HTTP traffic. Defense in depth.

1.6 The hotel security analogy — three guards with three levels of vetting

Hold this picture in your head whenever firewall types feel abstract:

Hotel guardFirewall typeHow they decide
Checklist guard at the front gate Stateless Holds a card with simple rules: "no visitors after 8 PM, no one with a red badge, no boxes." Looks at each arriving person independently, applies the rules, doesn't remember who just left or who's expected back. Fast but gullible.
Logbook guard at the lobby Stateful Keeps a logbook: who entered, when, what their stated business is, when they're expected to leave. When someone returns, checks the logbook to verify they were originally signed in. Can detect impersonators ("I'm the same plumber from earlier") but doesn't know what's in their toolbox.
Concierge meeting visitors at the desk Application / Proxy Meets every visitor, asks their business in detail, then personally carries the message to the intended recipient. The visitor never enters the building; the concierge does. Can refuse inappropriate requests ("you can't ask the CEO that") but is slow and needs to speak every topic visitors might bring up.

In this analogy: iptables is the rulebook each guard carries. Netfilter is the set of "checkpoint stations" placed throughout the hotel where the guards sit. An SSH-tunneling visitor who hides their real intent inside an encrypted briefcase corresponds to firewall-evasion via VPN — the guard sees the briefcase but can't open it. 📄 pp. 409, 413, 427

2aStateless (Packet-Filter) Firewall — with worked examples 📄 pp. 405, 408 · Lecture 16 part 2

① Intuition — the simplest firewall design

A stateless firewall is the most basic kind. For every packet that arrives, it asks one question: "Does this packet's header match any rule on my list?" If yes, apply the matching rule's action. If no, apply the default policy. No memory of past packets, no awareness of conversations — just header inspection one packet at a time.

② Mechanism — what the firewall actually inspects 📄 p. 408

Slide 408 enumerates the fields available to a stateless firewall. This is the entire set; nothing else can be used in a rule:

Header fieldWhat it tells youExample rule using it
IP source address Which IP claims to be sending "Block packets from 198.51.100.5" (a known bad IP)
IP destination address Which IP is the intended recipient "Drop incoming packets to 10.0.0.5" (a sensitive internal server)
Protocol / Next Header Which transport-layer protocol is in use (TCP, UDP, ICMP, …) "Drop all ICMP" (block ping floods)
TCP/UDP source port Which port the packet appears to be coming from "Drop packets from source port 0" (clearly invalid)
TCP/UDP destination port Which service the packet is trying to reach "Drop incoming TCP port 23" (block Telnet)
TCP flags SYN, ACK, FIN, RST, PSH — control bits in the TCP header "Block incoming SYN to port 22 from outside our network"
ICMP type Which ICMP message type (echo request, destination unreachable, …) "Allow only ICMP type 0 (echo reply) inbound"

Critically: everything in this list is at L3 or L4. Nothing about the packet's payload is accessible. A stateless firewall cannot tell whether an HTTP request contains a SQL injection — that's L7 information.

③ Worked example — building rules around port 23 (Telnet) 📄 p. 408

Let's walk through the example the lecture gives. We want to block Telnet (a notoriously insecure cleartext remote-shell protocol that uses TCP port 23). Here's a concrete rule and what happens with three different packets: 

  STATELESS FIREWALL EXAMPLE — block port 23     📄 p. 408

  Rule list:
  ┌───┬──────────┬───────────┬─────────┬──────────┬──────────┬─────────┐
  │ # │ Direction│ Protocol  │ Src IP  │ Dst Port │ Flags    │ Action  │
  ├───┼──────────┼───────────┼─────────┼──────────┼──────────┼─────────┤
  │ 1 │ INPUT    │ TCP       │ ANY     │ 23       │ ANY      │ DROP    │
  │ 2 │ INPUT    │ TCP       │ ANY     │ 22       │ ANY      │ ACCEPT  │
  │ 3 │ INPUT    │ TCP       │ ANY     │ 80       │ ANY      │ ACCEPT  │
  │ * │ INPUT    │ ANY       │ ANY     │ ANY      │ ANY      │ DROP    │   ← default
  └───┴──────────┴───────────┴─────────┴──────────┴──────────┴─────────┘

  Now three packets arrive:

  ──── PACKET A ──────────────────────────────────────────────────
  IP src: 192.0.2.50 (somewhere on the internet)
  IP dst: 10.0.0.5   (our server)
  Protocol: TCP
  Src port: 51234 (some random ephemeral port)
  Dst port: 23  ← Telnet
  Flags: SYN
  ─────────────────────────────────────────────────────────────────
  Firewall checks rule 1: dst port 23 + TCP + INPUT direction. MATCH.
  Action: DROP.
  → Packet vanishes. The Telnet attempt fails.

  ──── PACKET B ──────────────────────────────────────────────────
  IP src: 192.0.2.50
  IP dst: 10.0.0.5
  Protocol: TCP
  Src port: 51234
  Dst port: 22  ← SSH
  Flags: SYN
  ─────────────────────────────────────────────────────────────────
  Firewall checks rule 1: dst port 22 ≠ 23. NO MATCH.
  Firewall checks rule 2: dst port 22 + TCP + INPUT. MATCH.
  Action: ACCEPT.
  → Packet passes through. SSH connection can proceed.

  ──── PACKET C ──────────────────────────────────────────────────
  IP src: 192.0.2.50
  IP dst: 10.0.0.5
  Protocol: UDP
  Src port: 51234
  Dst port: 9999
  ─────────────────────────────────────────────────────────────────
  Firewall checks rule 1: protocol TCP ≠ UDP. NO MATCH.
  Firewall checks rule 2: protocol TCP ≠ UDP. NO MATCH.
  Firewall checks rule 3: protocol TCP ≠ UDP. NO MATCH.
  Default policy: DROP.
  → Packet vanishes.

Three things to notice from this example:

  1. Rules are evaluated top-to-bottom, and the first match determines the action. Once a rule matches, no later rules are consulted.
  2. Order matters. If rule 1 (DROP port 23) came after rule 2 (ACCEPT port 22), it wouldn't make a difference — but if a generic ACCEPT rule came before specific DROP rules, the DROPs would never fire.
  3. The default policy is the safety net. If no rule matches, the default applies. Best practice: default DROP. Make every allowed traffic explicit.

④ The fundamental limitation — return-traffic problem 📄 p. 420

The hard problem stateless firewalls cannot solve Suppose Alice (inside) wants to browse example.com (outside). She sends a packet out: src=Alice, dst=example.com, dst port=443. The web server replies: src=example.com, dst=Alice, src port=443. How does the stateless firewall know to allow this reply?

It can't tell that the inbound packet is a "reply" — it has no memory of Alice's outgoing request. The only way to allow replies is to add a rule like "allow all inbound TCP from source port 443." But this rule is a huge attack surface — it lets any external attacker who picks source port 443 send arbitrary packets to internal hosts.

This is the central reason stateful firewalls were invented. 📄 p. 420
2bStateful Firewall — with state-table walkthrough 📄 pp. 406, 419-421 · Lecture 16 part 2

① Intuition — adding memory to the firewall

A stateful firewall solves the return-traffic problem by remembering. As packets pass through, the firewall builds and maintains a connection state table: a record of every active TCP/UDP conversation. When a new packet arrives, the firewall first checks: "Does this packet belong to a conversation I'm already tracking?" If yes, allow it automatically. If no, fall back to the regular rule list.

② What goes into the connection state table 📄 pp. 406, 419

Each row in the table represents one active TCP connection. The standard fields:

FieldExampleWhy it's tracked
Source IP 10.0.0.5 (Alice) Who started the conversation
Source port 51234 Distinguishes multiple connections from the same machine
Destination IP 93.184.216.34 (example.com) Who's being talked to
Destination port 443 (HTTPS) Which service
Protocol TCP How the conversation works
State NEW / ESTABLISHED / RELATED / CLOSED Where in the conversation lifecycle we are
Last activity 2026-04-25 10:23:45 For timing out idle conversations

The state field is the most important one. It evolves as packets are seen:

StateWhat it meansHow a packet enters this state
NEW A connection is just being initiated First SYN packet from client to server
ESTABLISHED Three-way handshake complete; conversation is fully open After client's final ACK
RELATED A new connection that is logically tied to an existing one FTP data connection during an active FTP control session
CLOSING / CLOSED Tear-down in progress; will soon be removed from table FIN or RST packet seen

③ Worked example — Alice browses example.com 📄 pp. 419-421

Let's walk through the same return-traffic problem from §2a, this time with a stateful firewall. We trace exactly how the state table evolves as the TCP three-way handshake unfolds. 

  STATEFUL FIREWALL EXAMPLE — Alice browses example.com  📄 pp. 419-421

  ├───┼───────────────────────────────┼─────────┤

  Looking for inbound packet matching outbound conversation:
    inbound src=93.184.216.34, dst=10.0.0.5, src port=443, dst port=51234
    ↔ outbound dst=93.184.216.34, src=10.0.0.5, dst port=443, src port=51234
  YES — these mirror each other. The conversation is recognized.
  Rule 1 fires (state ESTABLISHED-in-progress): ACCEPT.
  State table updated:
       ┌──────────┬──────────────┬──────┬──────┬───────┬──────────────┐
       │ ...      │ ...          │ ...  │ ...  │ ...   │ SYN_RECV     │
       └──────────┴──────────────┴──────┴──────┴───────┴──────────────┘

  ────────────────── PACKET 3: Alice's final ACK ────────────
  src=10.0.0.5 dst=93.184.216.34, flags: ACK
  ─────────────────────────────────────────────────────────────
  Firewall: matches the state-table entry. Rule 2 (OUTBOUND): ACCEPT.
  State table updated to ESTABLISHED.

  ────────────────── PACKET 4..N: HTTPS data ────────────────
  Each packet matches the ESTABLISHED entry → rule 1: ACCEPT.

  ────────────────── ATTACK ATTEMPT: spoofed reply ───────────
  Suppose an attacker on the Internet now sends:
  src=8.8.8.8     dst=10.0.0.5
  src port=443    dst port=51234
  TCP flags: SYN-ACK
  ─────────────────────────────────────────────────────────────
  Firewall: "Is there a matching state-table entry?"
  Looking for: src=8.8.8.8 (NOT 93.184.216.34) — NO MATCH.
  Rule 2 (OUTBOUND): NO (this is inbound).
  Default policy: DROP.
  → Spoofed packet rejected, even though it claimed to be from "port 443."
  → A stateless firewall would have wrongly allowed this if it had a
     "src port 443 → ACCEPT" rule for return traffic.

④ Why this matters — the security improvement 📄 p. 420

The stateful firewall solved two problems at once:

  1. Return traffic is automatic. Alice's outgoing connection automatically allows the matching reply, with no need for a broad "allow inbound port 443" rule.
  2. Bogus inbound packets are rejected. An attacker can no longer fool the firewall by setting their source port to 443. The state-table lookup requires full 5-tuple matching, including the source IP. Spoofing the source IP is hard if the attacker isn't on the path between you and the real server.
Slide 420's exact framing "Traditional packet filters do not examine transport layer context — i.e. matching return packets with outgoing flow. Stateful packet filters address this need. They examine each IP packet in context, keep track of client-server sessions, check each packet validly belongs to one. Hence are better able to detect bogus packets out of context." 📄 p. 420

⑤ The cost — what stateful firewalls trade away

CostWhy it matters
Memory Each active connection consumes a row in the state table. A busy firewall with millions of connections needs gigabytes of state-table memory.
State-table exhaustion attacks An attacker who opens 100,000 SYN packets but never completes the handshake can fill the state table, forcing the firewall to either reject all new connections or drop old half-open ones.
Slightly higher per-packet processing Every packet now triggers a hash-table lookup before rule evaluation.

In modern enterprise networks these costs are easily absorbed; stateful is the default. But for very high throughput at the network edge (e.g., ISP backbones), stateless filtering is still used because of its raw speed.

2cApplication / Proxy Firewall — full two-connection model 📄 pp. 407, 422-425 · Lecture 16 part 2

① Intuition — going up to the application layer

Stateful firewalls are smart about conversations, but they still only see L3 + L4 headers. They cannot tell that an HTTP request contains a SQL injection, that an email contains malware, that a DNS query is asking for a known-malicious domain. To make decisions based on what the data says, the firewall needs to actually parse the application protocol — which means going up to L7.

Application/proxy firewalls do this by being radically more interventionist than packet filters: instead of forwarding packets, they terminate the client's TCP connection entirely, then open a new connection to the destination. Two separate connections, with the proxy in the middle reading and forwarding application-layer messages.

② The two-connection model walked end-to-end 📄 pp. 407, 422

This is the key mental model. Without a proxy, there is one TCP connection from client to server. With a proxy, there are two completely independent TCP connections, glued together at the application layer by the proxy: 

  WITHOUT A PROXY (regular routing)              WITH AN APPLICATION/PROXY FIREWALL

  Client            Server                       Client          Proxy            Server
  ──────            ──────                       ──────          ─────            ──────

  ── SYN ──────────────►                         ── SYN ────►
  ◄────────── SYN-ACK ──                         ◄── SYN-ACK ──
  ── ACK ──────────────►                         ── ACK ────►
  (TCP handshake done)                           (Connection #1: client ↔ proxy)

  ── HTTP GET / ──────►                          ── HTTP GET / ──►
  ◄──── HTTP 200 OK ───                                          ── SYN ────────────►
                                                                  ◄────── SYN-ACK ───
                                                                  ── ACK ────────────►
                                                                  (Connection #2: proxy ↔ server)

                                                                  ── HTTP GET / ────►
                                                                  ◄──── HTTP 200 OK ─

                                                 ◄── HTTP 200 OK ─

  Single TCP connection.                         Client never directly speaks to Server.
  Server sees client's IP.                       Server sees PROXY's IP, not client's.
                                                 Proxy sees full HTTP request and response.
                                                 Proxy can REJECT or MODIFY anything.

③ What this enables — why proxies are powerful 📄 p. 423

Because the proxy fully reassembles and parses the application-layer message, it can do things no packet filter could:

  • Inspect URLs — block requests to example.com/admin while allowing the rest of example.com
  • Sanitize content — detect and reject SQL injection patterns, XSS payloads, malformed protocol messages
  • Rate-limit by user — count requests per session/user, not just per IP
  • Cache responses — serve repeated requests from a local cache (faster + reduces load on the real server)
  • Log everything — full visibility into every interaction, including request bodies
  • Hide internal topology — the server sees only the proxy's IP, not the real internal client. (And vice versa for outbound proxies.)

④ The cost — separate proxy per protocol 📄 pp. 423-424

Because each application protocol has its own syntax, a proxy firewall needs a different implementation for each protocol it supports. There's no universal "application proxy" — there's a Web Application Firewall (WAF) for HTTP, an SMTP proxy for email, a DNS proxy for DNS, an FTP proxy for FTP. 📄 p. 423 

  APPLICATION-FIREWALL ARCHITECTURE              📄 p. 424

   Network connection arrives at the firewall machine
                       │
                       ▼
            ┌──────────────────────┐
            │  Dispatcher daemon   │  ← detects protocol by port + handshake pattern
            └──────────┬───────────┘
                       │
        ┌──────────────┼──────────────┬──────────────┐
        ▼              ▼              ▼              ▼
    Web daemon     SMTP daemon     FTP daemon     DNS daemon
        │              │              │              │
        ▼              ▼              ▼              ▼
    Web proxy      SMTP proxy     FTP proxy      DNS proxy   ← spawned on detection
   (parses HTTP) (parses RFC 5321)(parses FTP)  (parses DNS)

  Custom or non-standard protocols typically aren't supported —
  there's no proxy for them, so they get dropped or pass through unchanged.

⑤ Worked example — WAF blocks SQL injection

Imagine a Web Application Firewall sitting in front of an e-commerce site. A request comes in: 

  WAF EXAMPLE — SQL injection attempt blocked

  Attacker sends to the WAF:
       GET /search?q=laptop' OR '1'='1 HTTP/1.1
       Host: shop.example.com
       User-Agent: Mozilla/5.0 ...

  WAF action sequence:
    1. Accept TCP connection on port 443 from attacker's IP.
    2. Decrypt TLS (using the shop's certificate that's been installed on the WAF).
    3. Parse HTTP request: method=GET, path=/search, query=q=laptop' OR '1'='1.
    4. Run the query string through SQL-injection pattern matcher:
         - "OR '1'='1" matches a known SQLi pattern.
    5. ABORT — close connection #1 to attacker. Return:
         HTTP/1.1 403 Forbidden
    6. Connection #2 (WAF ↔ shop server) is never opened.
       The shop never sees the malicious request.

  Compare to packet-filter or stateful firewall:
    - They would see only TCP traffic on port 443 with TLS encryption.
    - They cannot read inside the encrypted HTTP request.
    - The malicious query reaches the shop server and exploits the database.

⑥ The TLS-termination tradeoff

A subtle but important point about HTTPS HTTPS encrypts the HTTP request between client and server. If the WAF is in the middle, it can only inspect the request if it terminates the TLS connection itself — meaning the WAF has its own server certificate and the client's TLS connection actually ends at the WAF. Then the WAF opens a new TLS connection to the real server.

This is exactly the two-connection model, but with TLS layered on top. It's how every cloud-based WAF (Cloudflare, AWS WAF, etc.) operates. The downside: the WAF has access to all plaintext traffic, which is a significant trust placement. The benefit: deep inspection of encrypted traffic that would otherwise be opaque.

⑦ Web proxy as evasion tool 📄 p. 425

Slide 425 makes a fascinating cross-cut: the same proxy mechanism that powers application firewalls can also be used to defeat egress firewalls. Here's how: 

  WEB PROXY EVADING EGRESS FILTERING             📄 p. 425

  Suppose your school blocks Facebook by destination IP filtering:
       firewall rule: "drop any packet to facebook.com IPs"

  You set up (or find) a public web proxy at proxy.example.com.
  In your browser, you configure all traffic to go through proxy.example.com.

  Browser → Firewall:
       "I want to talk to proxy.example.com" (allowed!)

  Browser → Proxy (over the now-permitted connection):
       "Please fetch facebook.com for me"

  Proxy → Facebook:
       (request goes from proxy.example.com to Facebook — the firewall
        doesn't see this leg, since the proxy is outside the school)

  Proxy → Browser:
       (returns Facebook's response)

  The firewall sees only "you're talking to proxy.example.com." It never
  sees Facebook's IP appearing in your traffic. The destination-based block
  is defeated.

  Slide 425 also notes ANONYMIZING PROXIES — the dual case where a user
  hides their real IP from the destination. The destination sees the proxy's
  IP, not the user's.

① The Netfilter framework 📄 p. 409

Before we look at iptables (the user-facing tool), we need to understand Netfilter — the kernel-level infrastructure underneath. The lecture's slide 409 description is dense; here it is unpacked: 

  NETFILTER — how the Linux kernel filters packets    📄 p. 409

  Every IP packet that traverses the Linux kernel passes through a series of
  named "hook points" along its journey:

   Incoming packet arrives at NIC
              │
              ▼
   ┌────────────────────────────┐
   │  NF_IP_PRE_ROUTING hook    │  ← packet just arrived, no decision yet
   └────────┬───────────────────┘
            │
       ┌────┴─────┐
       │ routing  │  Is this packet for me, or for forwarding?
       │ decision │
       └────┬─────┘
            │
    ┌───────┴───────┐
    ▼ (for me)      ▼ (forwarding)
  ┌──────────────────┐  ┌────────────────────────┐
  │ NF_IP_LOCAL_IN   │  │ NF_IP_FORWARD          │
  └──────────────────┘  └────────────────────────┘
            │                    │
            ▼                    ▼
       (delivered)          ┌─────────────────────┐
                            │ NF_IP_POST_ROUTING  │
                            └─────────────────────┘
                                       │
                                       ▼
                                   (sent out)

  Locally generated outgoing packet:
       app → NF_IP_LOCAL_OUT → NF_IP_POST_ROUTING → out to NIC

  At each hook, kernel modules can register callback functions.
  When a packet hits the hook, the callback inspects it and returns a verdict:
       NF_ACCEPT  → continue processing
       NF_DROP    → discard silently
       NF_QUEUE   → send to userspace for inspection
       NF_STOLEN  → kernel module took ownership; don't continue

Slides 410-412 show the lecture's example: a custom LKM written in C that registers a callback on NF_IP_LOCAL_IN and drops every TCP packet with destination port 23. This demonstrates the principle, but writing kernel modules in C for every firewall rule would be insane — which is why iptables exists.

② iptables — the user-friendly layer 📄 p. 413

iptables is two things working together:

  • Xtables — a kernel module that registers Netfilter hooks at all the standard points and provides a generic rule-evaluation engine.
  • The iptables command — a userspace program that sends rules into Xtables via a special interface.

Together, they let a sysadmin configure firewall behavior without writing C code. Rules are organized in a hierarchy: 

  IPTABLES STRUCTURE                            📄 p. 413

  ┌── TABLE: filter ────────────────────────────────┐
  │  (the default table — for actual filtering)     │
  │                                                  │
  │  ┌── CHAIN: INPUT ─────────────────────────┐    │
  │  │  Rules that apply to packets DESTINED   │    │
  │  │  for this machine.                       │    │
  │  │                                          │    │
  │  │  Rule 1: -p tcp --dport 22 -j ACCEPT     │    │
  │  │  Rule 2: -p tcp --dport 80 -j ACCEPT     │    │
  │  │  Default: DROP                           │    │
  │  └──────────────────────────────────────────┘    │
  │                                                  │
  │  ┌── CHAIN: OUTPUT ────────────────────────┐    │
  │  │  Rules that apply to packets ORIGINATING│    │
  │  │  on this machine.                        │    │
  │  └──────────────────────────────────────────┘    │
  │                                                  │
  │  ┌── CHAIN: FORWARD ───────────────────────┐    │
  │  │  Rules that apply to packets PASSING    │    │
  │  │  THROUGH this machine (acting as router)│    │
  │  └──────────────────────────────────────────┘    │
  │                                                  │
  └──────────────────────────────────────────────────┘

  ┌── TABLE: nat ────────────────────────────────────┐
  │  (Network Address Translation — port forwarding) │
  └──────────────────────────────────────────────────┘

  ┌── TABLE: mangle ─────────────────────────────────┐
  │  (Modifying packet header fields)                │
  └──────────────────────────────────────────────────┘

  Within each chain, rules are evaluated TOP TO BOTTOM, first match wins.
  If no rule matches, the chain's DEFAULT POLICY is applied.

④ Worked example — building a firewall in 8 steps 📄 pp. 415-417

Now we walk through the lecture's worked example carefully. The goal: a server that accepts only SSH (22) and HTTP (80) inbound, allows all outbound TCP, allows DNS lookups, and drops everything else. 

  IPTABLES WORKED EXAMPLE — full walkthrough     📄 pp. 415-417

  ─── STEP 1: Flush existing rules ──────────────────────────────
  $ iptables -F
  $ iptables -X

  Why first?
    -F flushes all rules from all chains.
    -X removes user-defined chains.
    Together they give you a clean slate. Important if old rules
    are lingering from previous experiments.

  ─── STEP 2: Set default policies to ACCEPT (TEMPORARILY) ──────
  $ iptables -P INPUT  ACCEPT
  $ iptables -P OUTPUT ACCEPT
  $ iptables -P FORWARD ACCEPT

  Why TEMPORARILY?
    Critical safety step. If you SSHed into the server to set up
    the firewall and you set the default to DROP first, your own
    SSH session would be killed and you'd be locked out.
    So: ACCEPT first, add allow rules, THEN switch to DROP.

  ─── STEP 3: Allow SSH and HTTP inbound ────────────────────────
  $ iptables -A INPUT -p tcp --dport 22 -j ACCEPT
  $ iptables -A INPUT -p tcp --dport 80 -j ACCEPT

  Reading these aloud:
    Append (-A) to the INPUT chain a rule that says:
      "if protocol (-p) is tcp and destination port (--dport) is 22,
       then jump (-j) to action ACCEPT."

  ─── STEP 4: Allow all outbound TCP ────────────────────────────
  $ iptables -A OUTPUT -p tcp -j ACCEPT

  Why permissive on outbound?
    For a server, outbound TCP is usually OK — replies to clients,
    package updates, monitoring, etc. (More restrictive setups
    would only allow specific outbound destinations.)

  ─── STEP 5: Allow DNS queries (outbound) and replies (inbound) ──
  $ iptables -A OUTPUT -p udp --dport 53 -j ACCEPT
  $ iptables -A INPUT  -p udp --sport 53 -j ACCEPT

  Why two rules?
    DNS uses UDP, not TCP. The query goes OUT (dst port 53), the
    reply comes IN (src port 53). Both legs need explicit ACCEPTs
    because the firewall is stateless without conntrack.

  ─── STEP 6: List rules to verify (good hygiene) ───────────────
  $ iptables -L -v -n

  ─── STEP 7: Switch default policy to DROP ─────────────────────
  $ iptables -P INPUT  DROP
  $ iptables -P OUTPUT DROP

  This is the BIG MOMENT. Now anything not explicitly ACCEPTed
  by the rules above gets dropped.

  ─── STEP 8: Test ──────────────────────────────────────────────
  From a different machine:
    $ wget http://server-ip/                ← succeeds (port 80 allowed)
    $ telnet server-ip 23                   ← fails (no rule allows port 23)
    $ ssh server-ip                         ← succeeds (port 22 allowed)
Why the order matters — a real-world disaster scenario Imagine you SSH into a remote server, run iptables -P INPUT DROP as your first command. The kernel immediately drops all incoming packets, including the SSH packets sustaining your current session. Your terminal hangs. You've locked yourself out of a remote machine you might not have physical access to. This is why ACCEPT-first then ADD-rules then DROP-default is the canonical safe sequence. 📄 p. 415

⑤ iptables extensions — turning iptables stateful 📄 p. 414

Out of the box, iptables is stateless. The conntrack extension adds the connection state table we discussed in §2b. With it, you can write stateful rules: 

  STATEFUL IPTABLES WITH CONNTRACK              📄 p. 414

  $ iptables -A INPUT -m conntrack \
             --ctstate ESTABLISHED,RELATED -j ACCEPT

  Reading this aloud:
    "On the INPUT chain, match (-m) the conntrack module's state field,
     where the state is ESTABLISHED or RELATED, and ACCEPT."

  This single rule replaces the explicit "allow port 53 reply" rule
  from step 5 above. Any return traffic for any outgoing connection
  is automatically allowed — much cleaner.

  ─── The owner extension ─────────────────────────────────────
  $ iptables -A OUTPUT -p tcp --dport 23 \
             -m owner --uid-owner alice -j DROP

  This blocks Telnet specifically when run by user alice.
  Note: only works on OUTPUT chain — inbound packets have no
  notion of user ID at the network layer.

⑥ Why firewalls aren't perfect 📄 p. 426

Slide 426 names three things firewalls fundamentally cannot defend against:

⑦ Firewall evasion techniques 📄 pp. 427-433

The lecture covers three ways users (whether benign or malicious) bypass firewalls. All three exploit encryption to hide content from the firewall.

Technique 1: SSH tunneling — Scenario A (incoming Telnet) 📄 pp. 428-430

  SSH TUNNELING — incoming evasion              📄 pp. 428-430

  Setup:
    - We work at "company" and have a machine called "work."
    - "work" runs Telnet but the company firewall blocks all incoming
      TCP from outside, including Telnet.
    - There's a machine "apollo" inside the company that we have an SSH
      account on. The firewall DOES allow SSH to apollo (port 22).
    - From "home" we want to telnet to "work."

  Solution: ssh -L tunnel through apollo

  At home, we run:
    $ ssh -L 8000:work:23 user@apollo

  This means:
    "Open an SSH connection to apollo. On home's localhost,
     bind port 8000. Anything sent to home:8000 should be
     forwarded — through the SSH tunnel — to work:23."

  Then we run:
    $ telnet localhost 8000

  Trace of what flows over each segment:
       home:telnet  ───► home:8000        (local loopback, not on network)
       home:ssh     ───► apollo:ssh       (encrypted SSH tunnel — this is what
                                            the company firewall sees)
       apollo:any   ───► work:23          (inside the corporate network,
                                            not subject to the firewall)

  What does the company firewall see?
       Just SSH traffic between home and apollo. Encrypted, opaque.
       It cannot tell the inner payload is actually Telnet to work.
       The Telnet block is defeated.

Technique 2: SSH tunneling — Scenario B (outbound Facebook) 📄 p. 431

  SSH TUNNELING — outbound evasion              📄 p. 431

  Setup:
    - We're at work on machine "work."
    - We want to visit Facebook. The company firewall blocks Facebook
      to keep employees focused.
    - We have an SSH-accessible machine "home" outside.

  Solution: ssh -L tunnel out through home

  At work, we run:
    $ ssh -L 8000:facebook.com:443 user@home

  Then in our browser at work:
    https://localhost:8000/

  Trace:
       work:browser ───► work:8000        (local loopback)
       work:ssh     ───► home:ssh         (encrypted SSH — what firewall sees)
       home         ───► facebook.com:443 (from home, outside the firewall)

  What does the company firewall see?
       Just SSH traffic between work and home. Encrypted.
       Cannot tell that Facebook is being accessed through the tunnel.

Technique 3: Dynamic port forwarding (SOCKS) 📄 pp. 432-433

The previous SSH tunnels were static — each tunnel was hardcoded for ONE destination (work:23 in scenario A, facebook.com:443 in scenario B). Setting up a separate tunnel for every destination is tedious. Dynamic port forwarding solves this: 

  DYNAMIC PORT FORWARDING (SOCKS)                📄 pp. 432-433

  At work, we run:
    $ ssh -D 9000 user@home

  The -D flag (instead of -L) creates a "dynamic" forwarding tunnel.
  This sets up a SOCKS proxy on localhost:9000 that knows how to
  route TCP connections to ANY destination through the SSH tunnel.

  Then we configure our browser/application to use a SOCKS proxy:
       SOCKS server: localhost
       SOCKS port:   9000

  What happens when we visit reddit.com:
       browser asks SOCKS proxy: "connect me to reddit.com:443"
       SOCKS proxy (running locally) tells SSH: "tunnel TCP to reddit.com:443"
       SSH forwards over the encrypted tunnel to home
       home opens the connection to reddit.com:443
       data flows back the same way

  Same for facebook.com, news.com, anything.
  ONE SSH tunnel handles them ALL.

  Slide 433 notes: "The client software must have native SOCKS support
  to use SOCKS proxies." Modern browsers do; some older tools don't.
Why all three techniques work — the common principle All three exploit the same property: SSH traffic is encrypted, so the firewall sees only opaque ciphertext between the SSH endpoints. The firewall knows that machine X is talking SSH to machine Y; it has no way to know what's inside.

The defense, when administrators want to prevent this: either disable SSH outbound entirely (drastic but effective), or use a TLS-terminating proxy that intercepts and decrypts SSH (very hard — SSH uses certificate pinning and detects MITM). In practice, most enterprises accept that their employees can tunnel out, and rely on endpoint security and HR policy rather than firewall blocks. 📄 p. 427

3.2 — Common Misconceptions & Traps

❌ MYTH: "Stateless means the firewall does not maintain any state — not even its rule list."

✅ REALITY: "Stateless" refers specifically to connection state — the firewall does not track which TCP/UDP flows are currently active. It still maintains its configured rule list (a static configuration, not per-connection state). The opposite of "stateless" is "stateful," which adds a per-connection tracking table. 📄 pp. 405-406

🎯 EXAM TRAP: MCQ option "stateless firewalls have no rules" — FALSE. They have rules but no per-flow memory.

❌ MYTH: "A stateful firewall inspects the application payload and can block SQL injection attacks."

✅ REALITY: Stateful firewalls operate at layers 3–4 only. They add connection-state tracking to packet filtering but do not look inside the TCP payload at application data. To block SQL injection or inspect HTTP URLs, you need an application/proxy firewall (specifically a WAF for HTTP). 📄 pp. 406, 423

🎯 EXAM TRAP: "Which firewall type can block malicious URL patterns?" → Only application/proxy, not stateful.

❌ MYTH: "Application/proxy firewalls work transparently — the client connects directly to the destination."

✅ REALITY: The defining feature of proxy firewalls is that the client's connection terminates at the proxy, and the proxy opens a new connection to the destination. There are always two separate TCP connections, and the proxy impersonates the destination to the client. 📄 pp. 407, 422

🎯 EXAM TRAP: "A proxy firewall forwards the original packets unchanged to the destination." — FALSE. It creates a fresh connection.

❌ MYTH: "Ingress and egress filtering are the same thing viewed from different sides."

✅ REALITY: They're structurally different defenses at different points. Ingress filtering inspects incoming traffic at your network edge to drop packets that don't belong (e.g., arriving with internal source IPs). Egress filtering inspects outgoing traffic to prevent spoofed packets from leaving your network or to block users from reaching disallowed destinations. The instructor specifically emphasized egress filtering at ISPs as more effective against IP spoofing because it stops spoofed packets at the source. 📄 pp. 403, 512

🎯 EXAM TRAP: "Which is more effective against IP spoofing: ingress or egress?" → Egress (per instructor's direct quote).

❌ MYTH: "A properly configured firewall prevents all attacks on the network."

✅ REALITY: Slide 426 explicitly names the three fundamental blind spots: "(1) Useless against attacks from the inside — evildoer exists on inside; malicious code is executed on an internal machine. (2) Organizations with greater insider threat — banks and military. (3) Cannot protect against transfer of all virus-infected programs or files — because of huge range of O/S and file types." Additionally, encrypted payloads (SSH tunnels, VPN, HTTPS) are opaque to layer-3/4 firewalls. 📄 pp. 426-427

🎯 EXAM TRAP: T/F: "A firewall alone is sufficient network security." — FALSE. Defense-in-depth is required.

❌ MYTH: "The iptables default policy should always be ACCEPT for usability."

✅ REALITY: Secure configurations use default DROP with explicit ACCEPT rules for the specific traffic you need to allow. This is a "whitelist" (default-deny) model — by far the safer stance. The instructor's own worked example switches from ACCEPT (during editing) to DROP (for production) specifically to illustrate this. 📄 pp. 415, 417

🎯 EXAM TRAP: "Which default policy reflects a least-privilege stance?" → DROP, not ACCEPT.

WARM-UP Conceptual 📄 p. 403

WC-1. A friend says: "Deny and Reject are two different words for the same firewall action — both block the packet." Pick apart what your friend got right and what they got wrong.

▶ Show Answer

📝 Answer: Your friend is half right. Both Deny and Reject do block the packet — neither allows it through. But they differ in what the sender learns about the block:

  • Deny (Drop) — silently discard. Sender hears nothing back. From their perspective, the packet just vanished. The destination might be down, the firewall might not exist, who knows.
  • Reject — discard AND send back an ICMP "destination unreachable" message. Sender immediately knows their connection failed.

The tradeoff:

  • Deny is more secure (stealth — attackers can't easily map your firewall).
  • Reject is more user-friendly (legitimate users get fast clear failures instead of waiting for timeouts).

Most public-facing firewalls use Deny by default; internal firewalls often use Reject. 📄 p. 403

WARM-UP Step-tracing 📄 pp. 419-421

WC-3. Alice (10.0.0.5) sends a SYN packet to example.com (93.184.216.34) on TCP port 443, source port 51234. The packet passes through a stateful firewall configured to allow outbound TCP. Trace exactly what the firewall does: (1) what state-table row gets created, (2) what happens when the SYN-ACK arrives back, and (3) what would happen if a totally unrelated packet from 8.8.8.8:443 → 10.0.0.5:51234 arrived afterward.

▶ Show Answer

📝 Answer:

(1) Outgoing SYN — state table updated:

   src IP    │ dst IP        │ s.pt  │ d.pt │ proto │ state
   10.0.0.5  │ 93.184.216.34 │ 51234 │ 443  │ TCP   │ NEW

The firewall checks rule list, sees outbound TCP is allowed, ACCEPTs and adds a NEW state-table row. Critical: the row is keyed on the FULL 5-tuple including destination IP 93.184.216.34.

(2) SYN-ACK arrives back from 93.184.216.34:

The firewall checks: is there a state-table row for (src=93.184.216.34, dst=10.0.0.5, src port=443, dst port=51234)? It looks for the mirror of an outbound row. YES — Alice's row matches when the IPs and ports are swapped. The firewall recognizes this as a return packet of an established conversation. ACCEPT, advance state to ESTABLISHED.

(3) Unrelated packet from 8.8.8.8:

The firewall checks: is there a state-table row for (src=8.8.8.8, dst=10.0.0.5, src port=443, dst port=51234)? NO — Alice's row has dst IP 93.184.216.34, not 8.8.8.8. The packet doesn't match any entry. Falls through to rule list: no inbound rule matches. Default policy: DROP.

This is exactly why stateful is more secure than stateless: a stateless firewall with a rule "allow inbound TCP from src port 443" would have wrongly allowed the 8.8.8.8 packet because it has src port 443. Stateful's full 5-tuple matching catches the spoof. 📄 pp. 419-421

WARM-UP Command reading 📄 pp. 415-417

WC-4. Read this sequence of iptables commands and explain (1) what the sequence as a whole accomplishes, and (2) what would go wrong if you moved the last command to the very top.

$ iptables -F
$ iptables -P INPUT  ACCEPT
$ iptables -A INPUT -p tcp --dport 22 -j ACCEPT
$ iptables -A INPUT -p tcp --dport 443 -j ACCEPT
$ iptables -P INPUT  DROP
▶ Show Answer

📝 Answer:

(1) What the sequence accomplishes:

  1. -F: flush all existing rules. Clean slate.
  2. -P INPUT ACCEPT: temporarily make the default policy ACCEPT so we can edit safely.
  3. -A INPUT -p tcp --dport 22 -j ACCEPT: append a rule allowing inbound SSH (port 22).
  4. -A INPUT -p tcp --dport 443 -j ACCEPT: append a rule allowing inbound HTTPS (port 443).
  5. -P INPUT DROP: switch the default policy to DROP. Now only port 22 and 443 inbound are allowed; everything else is dropped.

End result: a firewall that accepts only SSH and HTTPS inbound, drops everything else.

(2) What goes wrong if the last command moves to the top:

If iptables -P INPUT DROP is the FIRST command, the moment it runs the kernel starts dropping all incoming packets — including any SSH session you're using to configure the server. Your terminal hangs and you're locked out. If the server is remote (no physical access), you have to wait for an out-of-band recovery mechanism — a console, a friend with on-site access, or a scheduled rule timeout — to fix it.

This is why every iptables tutorial teaches the canonical safe order: flush → ACCEPT default → add rules → DROP default. The DROP comes LAST, only after every legitimate flow is explicitly allowed. 📄 pp. 415-417

WARM-UP Concept synthesis 📄 pp. 427-431

WC-5. Your friend says: "SSH tunneling defeats firewalls because the firewall can't see SSH traffic at all — SSH uses some special hidden protocol." Explain why your friend is wrong and describe what's actually happening that makes SSH tunneling effective as a firewall-bypass technique.

▶ Show Answer

📝 Answer: Your friend is wrong on two counts.

First: SSH is fully visible to the firewall. SSH uses a well-known TCP port (22), and SSH packets have ordinary IP and TCP headers like any other traffic. The firewall can absolutely see "machine X is sending SSH-flagged traffic to machine Y" — and could block SSH outright if configured to do so.

Second: SSH does not use any "hidden protocol." It's a standard well-documented protocol with public RFCs.

What actually makes SSH tunneling work is encryption of the payload. The firewall sees:

  • The IP and TCP headers (visible — these have to be unencrypted for routing to work)
  • The encrypted SSH payload (opaque ciphertext — content invisible to firewall)

So the firewall knows SSH is happening between the two machines, but cannot inspect what is inside the SSH session. If the firewall has been configured to ALLOW SSH (a common default — administrators need it for remote management), then anything tunneled inside SSH gets a free ride.

The defense, when administrators want to prevent SSH tunneling: either disable outbound SSH entirely (drastic but effective), or — more commonly — accept the risk and rely on endpoint security and HR policy. The fundamental point: encryption defeats deep inspection. Anything encrypted that the firewall is willing to forward becomes a potential evasion channel. 📄 pp. 427-431

Q10. Describe the fundamental tension between firewall effectiveness and encryption. THE #1 MISTAKE STUDENTS MAKE:

Confusing stateful with application/proxy. Both retain per-connection memory, but stateful firewalls operate at layers 3-4 only (tracking TCP/UDP flows via their 5-tuple + state) and do not inspect payload. Application/proxy firewalls go further — they terminate the connection, parse the application-layer protocol, and can block content-based attacks (SQL injection, malicious URLs). If the question involves "inspecting payload" or "blocking URL patterns," the answer is always Application/Proxy, not Stateful. 📄 pp. 406-407, 423
