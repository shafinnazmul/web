§1 — Concept Overview

📘 LECTURE PDF COVERAGE: pp. 30-31, 54-181 (five consecutive lectures make up the cryptography foundation)

1.1 The Problem This Solves 📄 pp. 30, 40, 56

Networks carry data across channels you do not control — coffee-shop Wi-Fi, the public Internet, shared storage. Cryptography is the toolbox that lets two parties achieve the four classical security objectives (slide 30) across such a hostile channel: Confidentiality (only the intended reader can read), Integrity (bits have not been altered in flight), Authenticity (the message genuinely came from the claimed sender), and Non-repudiation (the sender cannot later deny having sent it). 📄 pp. 30, 56

1.2 The Core Idea — Four Primitives, Four Jobs 📄 pp. 55, 115, 137, 152

The entire course-taxonomy tree on slides 55/115/137/152 reduces to four primitive families:

  • Symmetric ciphers (AES, DES, 3DES; also stream ciphers like RC4) — fast bulk encryption using a shared secret key. Appears in every practical system because it is the only primitive fast enough for large payloads. 📄 pp. 55, 150
  • Asymmetric ciphers (RSA, Diffie-Hellman, ECC) — slow but magical: they let two parties who have never met agree on a secret, sign messages nobody else can forge, and authenticate each other. Almost never used for bulk data — mostly for key exchange and digital signatures. 📄 pp. 55, 94, 150
  • Cryptographic hashes (SHA-256, SHA-512) — one-way fingerprints. Used for integrity checks, password storage, Merkle trees, and as the input to MACs and signatures. Unlike asymmetric primitives, not invertible. 📄 pp. 152, 157, 160
  • Message Authentication Codes (HMAC, AES-GCM, CBC-MAC) — keyed constructions that prove both integrity and authenticity simultaneously. The bridge between encryption and authentication. 📄 pp. 172, 181

The unifying pattern of every secure protocol on this course: use asymmetric crypto to agree on a secret, then switch to symmetric crypto (AES-GCM in particular) for the actual traffic. TLS does this. IPsec does this. WPA2 does this. SSH does this. 📄 pp. 150, 181

1.3 Analogy — A Building's Security System 📄 pp. 85, 87, 145, 155

Think of a corporate building where four different mechanisms each solve a different problem:

  • Shared door key (each employee has the same physical key that opens the back door) → Symmetric cipher. Fast, works at scale, but how do you distribute the key the first time? 📄 p. 85
  • The mailbox with a slot (anyone can drop a letter in, only the owner's key opens it) → Asymmetric cipher: public slot, private key. 📄 p. 87
  • A tamper-evident seal (any stranger can verify the seal is intact, but the seal itself proves nothing about identity) → Hash. Integrity without authenticity. 📄 p. 155
  • A wax seal imprinted with a signet ring (only the owner of the ring can produce it; anyone can verify by the imprint) → Digital signature + MAC family. Integrity with authenticity. 📄 pp. 103, 172

In this analogy, the Certificate Authority is the government registry that vouches for who owns which signet ring, preventing a "fake BOA" pretender from fooling Alice (slide 145). 📄 pp. 104, 130, 145

1.4 Technical Definition 📄 pp. 55, 87, 152, 172

Cryptography is the study of mathematical techniques for information security. A symmetric cipher is a pair of efficient algorithms (Encrypt, Decrypt) such that Ek(m) produces ciphertext and Dk(c) = m, using the same secret key k 📄 p. 55. An asymmetric cipher uses a pair (public-key pk, private-key sk) derived from a trap-door one-way function: computing f(x) is easy but inverting it is infeasible without the trap-door 📄 p. 87. A cryptographic hash H(m) is a deterministic function producing a fixed-length output with the properties of pre-image resistance, second-pre-image resistance, and collision resistance 📄 pp. 155-157. A Message Authentication Code MAC(k, m) is a keyed function such that only someone with k can compute a valid tag, and any change to m or k invalidates the tag 📄 pp. 172, 174.

CTR mode ⭐ Counter mode: encrypt a counter to produce keystream, XOR with plaintext. Parallelizable, random-access, no padding. Instructor's "why counter mode" favorite; also the basis for AES-GCM. 📄 pp. 77, 80, 82, 170

Instructor emphasized: "you will need to understand what is nonce, what is the purpose... can we reuse nonces? What's the problem of reusing?" 📄 pp. 76, 158, 165
Nonce "Number used Once" — a unique value per message. In CTR mode, the nonce forms the high half of the counter; the incremental counter forms the low half. Reusing a nonce with the same key under CTR mode → catastrophic (keystream reuse). 📄 pp. 80, 181
RSA ⭐ Rivest-Shamir-Adleman (1978). Key = (e, n) and (d, n) where n = pq, factoring n is hard. Supports encryption AND signatures AND key exchange. Instructor: "you don't need to remember the five steps, but you need to know how to use it." 📄 pp. 90-95
Diffie-Hellman (DH) ⭐ Two parties each pick a secret x, exchange g^x mod p publicly, and both compute g^(xy) mod p as the shared secret. Key exchange only — no encryption, no signatures. The "how two strangers agree on a secret" workhorse. 📄 pp. 94, 97-98, 122-123, 140
ECC / ECDHE Elliptic Curve Cryptography. Same operations as DH but over an elliptic-curve group — far shorter keys for equivalent security (256-bit ECC ≈ 3072-bit RSA). ECDHE = Ephemeral version → Perfect Forward Secrecy. Instructor said "most of our practice now uses ECDHE" for key exchange. 📄 pp. 94, 101, 115, 122-123, 195
Perfect Forward Secrecy (PFS) If a long-term private key is later compromised, past session traffic stays confidential because the session used an ephemeral DH key that was discarded. This is why we prefer DH over RSA for key exchange. 📄 pp. 101, 122
Digital signature ⭐ Sign: s = Encprivate(hash(m)). Verify: Decpublic(s) == hash(m). Provides integrity + authenticity + non-repudiation. Non-repudiation is uniquely provided by digital signatures (not MAC) — exam trap. 📄 pp. 94, 103
X.509 certificate A document binding a public key to an identity (server name, user, email), digitally signed by a trusted Certificate Authority (CA). The "how do I know this public key belongs to BOA?" solution. See the "Fake BOA" example slide 145. 📄 pp. 104, 128, 145
Certificate Authority (CA) Trusted third party that issues X.509 certificates. Root CAs are pre-installed in browsers/OSes. Forms the chain of trust for TLS (contrast: DNSSEC uses zone hierarchy). 📄 pp. 106, 110, 130
Hash function A deterministic function H: {0,1}* → {0,1}^n producing a fixed-length fingerprint. Properties: avalanche (bit change → total output change), pre-image resistance, collision resistance. Distinguish from asymmetric: hash is NOT invertible, even with a key. 📄 pp. 152, 155-157
SHA-256 / SHA-512 Members of the SHA-2 family. 256 / 512-bit outputs respectively. Based on Merkle-Damgård with Davies-Meyer compression. Know "birthday attack → half-the-digest-bits security" (SHA-256 → 128-bit collision resistance). 📄 pp. 160, 162, 200
Merkle-Damgård Iterative construction: pad message, split into blocks, chain through a compression function F with an IV. Structural diagram on slides 158/165 — "what does F do?" is a likely question. 📄 pp. 158, 165
MAC (Message Authentication Code) ⭐ A keyed, deterministic function MAC(k, m) attached to a message so the receiver can verify integrity + origin. Only parties with k can produce or verify. Contrast with plain hash (no key) and digital signature (asymmetric + non-repudiation). 📄 pp. 172, 174, 177
HMAC Hash-based MAC: HMAC(k,m) = H((k⊕opad) ‖ H((k⊕ipad) ‖ m)). Uses a hash (like SHA-256) plus two padding constants. The most common MAC in practice. 📄 pp. 178-180
AES-GCM ⭐ AES in Galois/Counter Mode. Combines AES-CTR for encryption + GHASH for authentication in a single pass. Authenticated encryption with associated data (AEAD). Instructor: "the most commonly used solution that provides encryption and message authentication" — used in TLS, IPsec, WPA2, HTTPS. 📄 pp. 181, 189, 202
Non-repudiation Property that the signer cannot later deny having created the message. Provided uniquely by digital signatures (asymmetric), not by MACs. Instructor: "if someone signed a piece of information using its private key, that means the signer cannot deny of this piece of information." 📄 pp. 30, 103

① Mechanism
AES operates on a 128-bit block viewed as a 4×4 matrix of bytes. Each round alternates confusion operations (SubBytes, AddRoundKey) with diffusion operations (ShiftRows, MixColumns) — Shannon's two principles. The key schedule derives a unique round key for each of the 10/12/14 rounds. 📄 pp. 63-68, 77
② Security Level & Why It Beat DES
AES-128 provides 128-bit security against brute-force key search. DES's 56-bit key falls to exhaustive search in ~1 hour at 10¹³ decryptions/s; AES-128 requires 5.3 × 10²¹ years at the same rate 📄 p. 190. AES also replaced DES's Feistel structure with a Substitution-Permutation Network that processes the whole 128-bit block per round rather than half. 📄 p. 67

The Nonce — What, Why, and the Cardinal Rule 📄 pp. 76, 80, 181

A nonce ("Number used Once") is a unique value prepended to each message's counter so that identical plaintexts encrypted under the same key produce different ciphertexts. Without a nonce, CTR-mode encryption of the same plaintext under the same key would yield identical ciphertext streams — and an eavesdropper who sees two such ciphertexts can XOR them to recover (P₁ ⊕ P₂), reducing the problem to known-plaintext cryptanalysis.

⚠️ The cardinal rule Never reuse a (key, nonce) pair in CTR mode. Reusing them leaks the XOR of plaintexts and is exactly what broke WEP in 2001. Instructor emphasized: "can we reuse nonces? What's the problem of reusing?" Answer: catastrophic confidentiality loss. 📄 pp. 80, 181
2bAsymmetric Ciphers 📄 pp. 83-151 · Lectures 4 & 5

The Core Insight — Trap-Door One-Way Functions 📄 p. 87

Symmetric ciphers solve confidentiality after you already share a key. But how do you share a key over an insecure network in the first place? Asymmetric crypto answers this with a mathematical object called a trap-door one-way function: a family {fk} such that

  • fk(x) = easy, given k and x
  • fk-1(y) = easy, given k-1 and y
  • fk-1(y) = infeasible, given y and k (but not k-1)

Publish k (the "public key") so anyone can encrypt; keep k-1 (the "private key") so only you can decrypt. The candidate trap-doors in this course are integer factoring (RSA), discrete logarithm (DH), and elliptic-curve discrete log (ECC). 📄 pp. 87, 94, 115

RSA — Encryption and Signatures 📄 pp. 90-95, 103

 RSA Key Generation                            📄 p. 95
  1. Pick primes p, q      (≥ 1024 bits each)
  2. n  = p · q            (modulus, public)
  3. φ(n) = (p-1)(q-1)     (Euler totient, private)
  4. Pick e with gcd(e, φ(n)) = 1   (public exponent)
  5. d  = e⁻¹ mod φ(n)     (private exponent)
  Public key = (e, n)     Private key = (d, n)

 Encryption / Decryption           Digital Signature
 ──────────────────────            ─────────────────────────
   C = M^e mod n   (encrypt)          s = hash(M)^d mod n    (sign with private)
   M = C^d mod n   (decrypt)          hash(M) =? s^e mod n    (verify with public)

 Notice: encryption and signing are the SAME mathematical op —
         they just use different keys.  📄 p. 103
SOURCE: 📄 pp. 95, 103
① RSA Encryption Use-Case
Alice wants to send Bob a confidential message. She fetches Bob's public key (e, n). She computes C = M^e mod n and sends C. Only Bob can decrypt, because only he holds d. Note: real systems don't encrypt user data directly with RSA — it's too slow. They encrypt a freshly-chosen AES session key with RSA, then use the AES key for everything else (hybrid encryption). 📄 pp. 95, 150
② RSA Signature Use-Case
Alice wants to prove she wrote message M. She computes s = hash(M)^d mod n using her private key and sends (M, s). Anyone can verify: compute hash(M) themselves, decrypt s with Alice's public key, check equality. The hash is essential — signing raw M is insecure because an attacker can choose M to make the math convenient. 📄 p. 103 Because Alice is the only one who could have produced s, this provides non-repudiation — she cannot later deny signing. 📄 pp. 30, 103

Diffie-Hellman — Agreeing on a Secret Without Pre-Sharing 📄 pp. 97-98, 122-123, 140

 Diffie-Hellman Key Exchange                   📄 pp. 97-98, 140

   Public parameters: large prime p, generator g

   Alice                                     Bob
   ─────                                     ─────
   pick secret x_A                           pick secret x_B
                    g^x_A mod p  ──────────►
                             ◄──────────  g^x_B mod p

   compute (g^x_B)^x_A = g^(x_A·x_B) mod p   compute (g^x_A)^x_B = g^(x_A·x_B) mod p
          └────────────────────┘                   └─────────────────────┘
           identical shared secret K            identical shared secret K

   Eavesdropper sees only g^x_A, g^x_B ─── but deriving x_A or x_B

SOURCE: 📄 pp. 97-98, 140

Ephemeral DH (DHE / ECDHE). If Alice and Bob pick fresh x_A, x_B for every session and discard them afterward, then even if an attacker later compromises their long-term keys, they cannot retroactively decrypt old traffic — this is Perfect Forward Secrecy (PFS). ECDHE (Elliptic-Curve Diffie-Hellman Ephemeral) is what TLS 1.3, modern IPsec, SSH, and WPA3 all use for this reason. 📄 pp. 101, 122-123

2cSecure Hash Functions 📄 pp. 152-162 · Lecture 5 part 2

The Three Required Properties 📄 pp. 155-157

  1. Pre-image resistance — given h, infeasible to find any m with H(m) = h. (Hence "one-way.")
  2. Second-pre-image resistance — given m₁, infeasible to find m₂ ≠ m₁ with H(m₂) = H(m₁).
  3. Collision resistance — infeasible to find any pair m₁, m₂ with H(m₁) = H(m₂). The birthday attack sets the bar here at 2^(n/2) work for an n-bit digest. 📄 pp. 162, 200

Plus one emergent property: avalanche — a single-bit change in the input flips roughly half the output bits. Slide 155 gives the canonical example: hashing "over" vs "once" gives two totally-unrelated digests. 📄 p. 155

2dMessage Authentication — MAC, HMAC & AES-GCM 📄 pp. 163-181 · Lecture 6

Why We Need MACs — AES Alone Is Not Enough 📄 pp. 170, 172

Encryption hides content but does not detect tampering. An attacker who flips bits in a CTR-mode ciphertext flips the corresponding bits in the decrypted plaintext — the receiver has no way to know. Encryption gives confidentiality; we separately need integrity + authenticity, which is what MACs provide. 📄 pp. 170, 172

Integrity vs Authenticity — A Critical Distinction 📄 p. 172

HMAC — The Standard MAC Construction 📄 pp. 178-180

HMAC is a specific recipe for building a MAC from any cryptographic hash: 

 HMAC(k, m) = H( (k ⊕ opad) ‖ H( (k ⊕ ipad) ‖ m ) )    📄 pp. 178-180

   k     = secret key
   m     = message
   H     = hash (typically SHA-256 → "HMAC-SHA-256")
   opad  = 0x5C repeated (outer padding constant)
   ipad  = 0x36 repeated (inner padding constant)
   ‖     = concatenation

The double-hash construction exists to immunize against length-extension attacks on Merkle-Damgård hashes. HMAC-SHA-256 is the default MAC for every major protocol (TLS, IPsec, IKE, JWT, etc.).

Encrypt-then-MAC vs MAC-then-Encrypt 📄 p. 180

Combining encryption + MAC — the right order matters Cryptographers have converged on Encrypt-then-MAC: encrypt the plaintext first, then compute the MAC over the ciphertext. This lets the receiver reject bad MACs before decrypting, avoids a class of chosen-ciphertext attacks, and is what modern AEAD modes (AES-GCM, ChaCha20-Poly1305) effectively do. 📄 p. 180

3.2 — Common Misconceptions & Traps

❌ MYTH: "ECB mode is secure as long as the plaintext is short."

✅ REALITY: ECB is insecure whenever two plaintext blocks are identical, because they encrypt to identical ciphertext blocks. For structured data (images, documents), this pattern leaks across blocks — the "Tux penguin" image example on slide 72 is canonical. ECB is only "safe" for a single block of true random data, which essentially never happens in practice. 📄 pp. 70-72

🎯 EXAM TRAP: T/F: "ECB mode is secure for messages ≤ 128 bits." — technically defensible but the instructor treats ECB as broadly insecure.

❌ MYTH: "A hash function can be 'decrypted' if you have the key."

✅ REALITY: Hashes have no key, and they are not invertible under any circumstance. This is the key distinction from asymmetric encryption. Instructor verbatim: "the difference between asymmetric cipher and secure hash is asymmetric cipher is invertible based on the correct keys... but the secure hash is not invertible because there is no way to recover the plaintext from the hash value — it is a long compression operation." 📄 p. 157

🎯 EXAM TRAP: MCQ offering "hash decryption" as an option.

❌ MYTH: "MAC and digital signature are the same thing."

✅ REALITY: Both provide integrity + authenticity, but only digital signatures provide non-repudiation. With a MAC, both sender and receiver share the key, so the receiver could forge any message and claim the sender wrote it. With a signature, only the sender holds the private key, so authorship is undeniable. 📄 pp. 103, 172

🎯 EXAM TRAP: "Which primitive is required for legal non-repudiation?" — the answer is digital signature, not MAC.

❌ MYTH: "RSA and ECC provide the same security level at the same key size."

✅ REALITY: ECC is dramatically more efficient. For 128-bit security: RSA needs 3072-bit keys, ECC needs only 256-bit keys. This is why modern protocols (TLS 1.3, SSH, WPA3) use ECDHE rather than DHE or RSA. 📄 p. 195

🎯 EXAM TRAP: Key-size matching tables that swap the RSA and ECC numbers.

❌ MYTH: "Counter mode adds integrity because it uses sequence counters."

✅ REALITY: CTR mode provides confidentiality only. The counter is for making each block's keystream unique, not for integrity. An attacker can flip any bit of CTR-mode ciphertext and the corresponding plaintext bit flips after decryption — with no detection. To add integrity you must either (a) run a separate MAC (Encrypt-then-MAC) or (b) use an AEAD mode like AES-GCM that bundles both. 📄 pp. 80, 170, 181

🎯 EXAM TRAP: Multi-answer: "Which AES modes provide integrity?" — CTR should NOT be selected.

❌ MYTH: "It's fine to reuse a nonce as long as you keep the key secret."

✅ REALITY: Reusing a (key, nonce) pair in CTR or GCM mode is catastrophic. Two ciphertexts C₁ = P₁ ⊕ KS and C₂ = P₂ ⊕ KS with the same keystream KS allow the attacker to compute C₁ ⊕ C₂ = P₁ ⊕ P₂, reducing the problem to known-plaintext cryptanalysis. GCM in particular also loses its authentication guarantees on nonce reuse. This is exactly how WEP was broken in 2001. 📄 pp. 80, 181

🎯 EXAM TRAP: Instructor said she will directly ask: "Can we reuse nonces? What's the problem?"

Q2. What goes wrong if the same (key, nonce) pair is reused for two different plaintexts P₁ and P₂ in AES-CTR mode?

  2. The server can no longer decrypt the messages because the counter becomes ambiguous.
  3. The attacker can compute C₁ ⊕ C₂ = P₁ ⊕ P₂, effectively reducing the problem to known-plaintext cryptanalysis and enabling content recovery.

✅ Correct Answer: C) Leaks P₁ ⊕ P₂ via ciphertext XOR.

📝 Explanation:

  • Why C is correct: In CTR mode, C_i = P_i ⊕ keystream(k, nonce, counter). If (key, nonce) pair is the same for two messages, both produce the same keystream KS. Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

Q5. Statement: Given a SHA-256 hash value h, someone with enough computing power and the right secret key can "decrypt" h back to the original message.

▶ Show Answer

✅ Answer: FALSE.

Hash functions are not invertible under any circumstance — there is no key, and there is no inverse algorithm. A hash compresses an arbitrary-length input to a fixed-length digest, destroying information. The instructor drew this contrast explicitly: "the difference between asymmetric cipher and secure hash is, asymmetric cipher is invertible based on the correct keys, but the secure hash is not invertible — there is no way to recover the plaintext from the hash value, because it is a long compression operation." The best an attacker can do is brute-force guess inputs (2²⁵⁶ for pre-image resistance on SHA-256). 📄 pp. 155-157, 162

INT TF 📄 p. 80

Q6. Statement: Because AES-CTR mode uses an incrementing counter that never repeats within a message, it provides built-in integrity protection against bit-flip attacks on the ciphertext.

▶ Show Answer

✅ Answer: FALSE.

The counter in CTR mode exists to vary the keystream across blocks and prevent keystream reuse. It provides confidentiality only, not integrity. An attacker who flips any bit of a CTR-mode ciphertext causes the corresponding bit of the decrypted plaintext to flip — with no detection. The counter never "checks" anything on decryption. For integrity you need a separate MAC (Encrypt-then-MAC) or an AEAD mode like AES-GCM, which explicitly adds GHASH authentication on top of CTR-mode encryption. 📄 pp. 80, 170, 181

ADV · subtle TF 📄 pp. 162, 200

Q7. Statement: A 256-bit hash function (such as SHA-256) provides 256 bits of collision resistance, meaning an attacker needs roughly 2²⁵⁶ operations to find two inputs with the same hash value.

▶ Show Answer

✅ Answer: FALSE.

The birthday attack finds collisions in approximately 2^(n/2) operations for an n-bit digest, not 2ⁿ. For SHA-256, that is roughly 2¹²⁸ operations — strong, but half the naive expectation. Slide 162 states: "Brute-force inversion requires 2²⁵⁶ ops, birthday attack on collision resistance requires 2²⁵⁶ ops" — wait, actually that slide shows SHA-512, where 2²⁵⁶ is the correct collision bound because n = 512. For SHA-256, the collision bound is 2¹²⁸. General rule: collision resistance = half the digest bits; pre-image resistance = full digest bits. The instructor's exam mnemonic: SHA-256 → 128-bit collision resistance = matches AES-128. 📄 pp. 162, 200

Q9. Describe the Diffie-Hellman key-exchange protocol in 5 steps or fewer. What secret does an eavesdropper need to compute in order to break it, and why is that considered hard?

▶ Show Answer

📝 Model Answer:

  1. Alice and Bob publicly agree on a large prime p and a generator g.
  2. Alice picks a secret integer x_A; computes A = gx_A mod p; sends A to Bob.
  3. Bob picks a secret integer x_B; computes B = gx_B mod p; sends B to Alice.
  4. Alice computes shared secret K = Bx_A = gx_A · x_B mod p.
  5. Bob computes shared secret K = Ax_B = gx_A · x_B mod p — same value.

An eavesdropper sees p, g, A, B on the wire but needs to recover either x_A from A or x_B from B to derive K. This is the Discrete Logarithm Problem: given g and gx mod p, compute x. For sufficiently large p (≥ 2048 bits), no efficient classical algorithm is known. Elliptic-curve variants (ECDH) use the analogous Elliptic-Curve Discrete Logarithm Problem, which appears even harder for equivalent key sizes — letting 256-bit ECC match 3072-bit classical DH. 📄 pp. 97-98, 140, 195

🔑 Key Points (Grading Checklist):

  • ☐ Correct math: gx_A · x_B is the shared secret 📄 p. 98
  • ☐ Both sides compute the same K via commutativity of exponents 📄 p. 140
  • ☐ Names the Discrete Logarithm Problem as the security assumption 📄 p. 150
  • ☐ Notes that the attacker sees A and B but cannot derive x_A or x_B
  • ☐ Bonus: mentions ECDH as the elliptic-curve variant used in practice 📄 p. 101
ADV · challenge SA 📄 pp. 94, 103, 172  +  p. 30

Q10. Compare MAC and Digital Signature across three dimensions: (1) what cryptographic primitive they're built on, (2) what security services they provide, (3) why only one of them provides non-repudiation. Conclude with one sentence explaining when you would choose each in practice.

▶ Show Answer

📝 Model Answer:

(1) Primitive. A MAC is a symmetric-key construction — both sender and receiver share the same secret key k, used to compute MAC(k, m). HMAC builds it from a hash function; AES-GCM's tag is an AEAD MAC. A digital signature is an asymmetric construction — the sender holds a private key sk, the receiver (or any verifier) uses the corresponding public key pk. Classic signing formula: s = hash(m)d mod n in RSA. 📄 pp. 94, 103, 172, 177

(2) Security services. Both provide integrity (message not altered) and authenticity (message came from a key-holder). The difference is in which key-holder: MAC verifies that some party with the shared key made the tag — either sender or receiver, indistinguishably. Signature verifies specifically that the private-key holder made it, which is unambiguously one party. 📄 pp. 103, 172

(3) Non-repudiation. With a MAC, Bob cannot convince a third party that Alice authored a message, because Bob also holds the key and could have forged the tag himself. With a signature, only Alice has her private key, so any valid signature demonstrably binds Alice to the message — she cannot later deny it. 📄 pp. 30, 103

Choosing in practice. Use MAC (especially AES-GCM or HMAC) for fast per-packet integrity on established sessions where both parties already trust each other; use a digital signature when you need legal non-repudiation or when the verifier is not the same party as the sender (e.g., code signing, certificates, software updates).

🔑 Key Points (Grading Checklist):

  • ☐ MAC uses symmetric / shared key; signature uses asymmetric / private key 📄 pp. 172, 103
  • ☐ Both give integrity + authenticity; only signature gives non-repudiation 📄 pp. 30, 103
  • ☐ Correctly explains why MAC lacks non-repudiation (shared key) 📄 p. 172
  • ☐ Concrete example of each use-case in the closing sentence

INT FIB 📄 pp. 162, 200

Q12. For a hash function with an n-bit digest, the expected cost to find a collision via the birthday attack is 2^(____________) operations, whereas the cost of a brute-force pre-image attack is 2^(____________) operations. Concretely, SHA-256 therefore provides ____________ bits of collision resistance.

▶ Show Answer

✅ Answers: (1) n / 2; (2) n; (3) 128 bits.

📝 Why: The birthday bound states that among 2^(n/2) random hashes there's a ≥50% chance of collision. Slide 162 matches this for SHA-512: "birthday attack on collision resistance requires 2²⁵⁶ ops" (= 2^(512/2)). Applying the same formula to SHA-256 gives 2^128 collision cost. Pre-image attacks on a 256-bit digest require ~2²⁵⁶ ops. 📄 pp. 162, 200

ADV FIB 📄 pp. 101, 122-123

Q13. When a TLS handshake uses ____________ — the ephemeral variant of Elliptic-Curve Diffie-Hellman — the resulting session key has the property of ____________, meaning that even if the server's long-term private key is later compromised, an adversary with recorded past traffic ____________ retroactively decrypt those past sessions.

▶ Show Answer

✅ Answers: (1) ECDHE (Elliptic-Curve Diffie-Hellman Ephemeral); (2) Perfect Forward Secrecy (PFS); (3) cannot.

📝 Why: Slide 101 defines ECDHE as "the last E represents Ephemeral mode" and names the property as "Perfect Forward Secrecy (PFS)." Because the session uses ephemeral keys that are discarded immediately after the handshake, no stored long-term key is sufficient to recover the session secret post-hoc. This is the architectural reason TLS 1.3 removed RSA key transport in favor of (EC)DHE-only key exchange. 📄 pp. 101, 122-123

§5 — Topic Summary · Exam Cheat Sheet

📘 LECTURE PDF COVERAGE: pp. 54-206 (condensed view of the entire foundations layer)

IN ONE SENTENCE, THIS TOPIC IS:

Four cryptographic primitives — symmetric ciphers (fast bulk encryption), asymmetric ciphers (key exchange + signatures), hashes (one-way fingerprints), and MACs (keyed integrity+authenticity) — composed in a hybrid pattern (asymmetric-for-setup + symmetric-for-bulk + MAC-for-integrity) that underlies every secure protocol on this course. 📄 pp. 55, 115, 137, 150, 181

THE THREE THINGS YOU MUST KNOW FOR EVERY PRIMITIVE:

  1. What service it provides — confidentiality (sym/asym), integrity (hash/MAC), authenticity (MAC/signature), non-repudiation (signature only). 📄 pp. 30, 103, 172
  2. Its key model — one shared key (sym/MAC), keypair (asym/signature), no key (hash). This is what determines whether non-repudiation is achievable.
  3. Its representative algorithm & security level — AES-128 → 128-bit; AES-256 → 256-bit; RSA-3072 → 128-bit; ECC-256 → 128-bit; SHA-256 → 128-bit collision resistance. 📄 pp. 162, 190, 195, 200

THE #1 MISTAKE STUDENTS MAKE:

Confusing MACs with digital signatures. They share a shape (tag appended to message, verifier checks) but the key model is fundamentally different — shared-key vs keypair — and that difference determines the single security property that courts care about: non-repudiation. A MAC never proves authorship to a third party because either party could have made it. Only a private-key signature does. 📄 pp. 103, 172
