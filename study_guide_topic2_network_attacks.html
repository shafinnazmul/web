§1 — Concept Overview

📘 LECTURE PDF COVERAGE: pp. 505-538, 541-557, 579-601 (three lectures: L19 IP/ARP, L20 pt.1 TCP, L22 DNS)

1.1 The Problem This Solves 📄 pp. 507, 528, 531, 588

The Internet protocol stack was designed in the 1970s-80s with almost no built-in authentication of source addresses or messages. IP packets carry whatever source address the sender writes into them 📄 p. 507, ICMP messages have "no authentication" at all 📄 p. 528, ARP replies are accepted even without a prior request 📄 p. 531, and DNS replies are trusted if they merely look like they came from the right server 📄 p. 588. This topic catalogues the resulting attacks — and, critically, the defenses we can stack on top at each layer of the TCP/IP model.

1.2 The Core Idea — A Top-to-Bottom Attack Tour 📄 pp. 505, 542, 580

The instructor explicitly organized this material "from top to bottom" of the TCP/IP stack, and so does this study guide:

  • Transport layer (TCP) — exploit the 3-way handshake and per-byte sequence number 📄 pp. 541-557
  • Network layer (IP + ICMP) — forge source addresses; abuse broadcast and error messages 📄 pp. 505-528
  • Link layer (ARP) — poison the IP-to-MAC mapping cache to sit between two hosts 📄 pp. 529-538
  • Application / infrastructure layer (DNS) — poison the name-to-IP mapping cache to redirect entire domains 📄 pp. 579-601

The unifying thread is that every attack exploits some form of missing authentication — and every defense either (a) adds cryptographic authentication, (b) makes the attacker's guesses harder by randomizing values, or (c) filters traffic at the network edge.

1.3 Analogy — The Postal Service With No Return-Address Verification 📄 pp. 507-508

Imagine a postal system with these properties:

  • Anyone can write any return address on an envelope and drop it in a mailbox → IP spoofing 📄 pp. 507-508
  • You can send a letter to a broadcast address ("everyone on Maple Street") and ask for replies — the amplification drowns the target → Smurf attack 📄 p. 521
  • A helpful neighbor keeps a book of "which house has which phone number" and trusts anyone who hands them an update card → ARP poisoning 📄 pp. 531-535
  • The phone directory operator answers "give me the number for ACME Co." by asking the first stranger who whispers an answer through the window → DNS cache poisoning 📄 pp. 589-590
  • You can whisper "Hang up! Call ended!" into a phone conversation you overhear, and if your voice is loud enough both parties drop the line → TCP Reset attack 📄 pp. 552-553

1.4 Technical Definition 📄 pp. 505-506, 542, 580

A network attack in this context is a packet-level abuse of a network protocol that violates one or more of confidentiality, integrity, availability, or authenticity of the service. The attacks in this topic fall into four protocol categories: TCP attacks exploit the transport-layer handshake and connection state (SYN flooding, RST injection, session hijacking) 📄 p. 542; IP / ICMP attacks exploit the lack of source-address authentication and the error-reporting channel (IP spoofing, Smurf, Ping of Death, Destination Unreachable) 📄 pp. 505-506; ARP attacks exploit the unsolicited-reply acceptance of the ARP cache (ARP spoofing → MITM) 📄 pp. 531-538; and DNS attacks exploit the unauthenticated reply mechanism of name resolution (cache poisoning, Kaminsky attack, DoS on nameservers) 📄 p. 588.

1.5 Key Terms & Definitions

Term Plain-English Definition Why It Matters for the Exam 📄 Page
TCP 3-way handshake SYN → SYN-ACK → ACK exchange that establishes a TCP connection. Each side picks a random initial sequence number (ISN). Foundation for SYN flood, RST, and hijacking. Know which party sends which packet. 📄 pp. 547-548
TCB (Transmission Control Block) In-memory structure the server allocates when it receives a SYN to track a half-open connection. The TCB queue is what SYN flooding exhausts. 📄 p. 548
SYN flooding 🔴 Attacker sends many SYNs (often from spoofed IPs) without completing the handshake, filling the server's half-open-connection queue. Classic DoS. The instructor explicitly named it in the final review. 📄 pp. 542, 549
SYN cookie Server encodes a keyed hash of connection info as the initial SYN-ACK sequence number, so it does not need to allocate a TCB until the ACK returns a valid cookie-based number. Canonical defense against SYN flooding. Know how it relaxes server state requirements. 📄 pp. 550-551
TCP RST attack 🔴 Attacker injects a spoofed RST packet that matches an active connection's (src IP, src port, dst IP, dst port, seq # within window) tuple to tear it down. Know all five fields the attacker must guess, and why local attackers succeed where remote ones struggle. 📄 pp. 552-554
TCP Session Hijacking Injecting spoofed TCP segments with valid sequence numbers to take over or corrupt an in-progress session. Similar preconditions to RST but the payload is malicious data, not just a RST flag. 📄 pp. 555-556
IP Spoofing 🔴 Sending IP packets with a forged source address. Trivial because IP has no source authentication. Underlies DDoS, Smurf, reflection attacks. Know its defenses (ingress/egress, IPsec). 📄 pp. 507-508, 511
Ingress / Egress filtering Ingress = incoming; Egress = outgoing. ISPs/edge routers drop packets whose source IP cannot legitimately belong to the incoming/outgoing interface. Instructor said egress filtering at ISPs is more effective. Expect a "which filter type" question. 📄 pp. 403, 425, 512
ICMP Internet Control Message Protocol — carries error and diagnostic messages (echo, destination unreachable, TTL exceeded, redirect). "No authentication" is the key exam point. 📄 pp. 518-519, 528
Smurf attack ICMP echo request with victim's IP as source, broadcast destination — every host on the broadcast network replies, drowning the victim. Classic amplification DDoS example. Know it uses broadcast. 📄 p. 521
Destination Unreachable attack Attacker sends forged ICMP "destination unreachable" to trick a host into dropping a legitimate connection. The "isolate a node" DoS variant. 📄 pp. 522-523, 528
Ping of Death Fragments that together reassemble to > 65 535 bytes overflow a kernel buffer and crash the host. Historical but named in the slide list. 📄 p. 527
ARP Address Resolution Protocol — maps IP addresses to link-layer MAC addresses on a LAN. Stateless: replies are cached even without a matching request. 📄 p. 529
ARP Spoofing 🔴 Attacker sends unsolicited ARP replies that map the victim's IP to the attacker's MAC, redirecting traffic through the attacker. The link-layer MITM enabler. Must explain the poisoning step by step. 📄 pp. 531-536
MITM (Man-in-the-Middle) Attacker sits between two parties, forwarding (and optionally modifying) their traffic. ARP poisoning is one way to achieve it on a LAN. Know ARP spoofing → MITM as the typical path on a local network. 📄 p. 537
DNS Cache Poisoning 🔴 Attacker sends a fake DNS reply (before the real one) to a resolver; the bogus mapping gets cached and used for all subsequent queries. The defining DNS attack. Local vs remote difficulty is a classic question. 📄 pp. 589-590
Transaction ID 16-bit random number the client picks for each DNS query; reply must match to be accepted. Analogous to TCP sequence number. Key defense variable. Along with random source port, these are the two numbers a remote attacker must guess. 📄 pp. 590-591
Kaminsky attack 🔴 Variant of DNS cache poisoning: attacker asks for random subdomains, so failed attempts don't get cached — removes the "wait for cache to expire" delay, enabling rapid brute-forcing. Instructor walked through this specifically. Know why random subdomains defeat the cache effect. 📄 pp. 592-593
DNSSEC DNS Security Extensions: adds a digital signature chain rooted at the DNS zone hierarchy, so clients can verify record authenticity. Contrast with TLS's CA-based chain of trust — explicit exam comparison. 📄 pp. 595, 597
Recursive vs Iterative query Recursive: local DNS does all the work on your behalf. Iterative: local DNS walks Root → TLD → authoritative, following referrals. Slide 584-585 walks through an iterative example (ua.edu). 📄 pp. 583-586
/etc/hosts Local file consulted before DNS. Attacker who can write here redirects the machine silently. Named as a DNS-spoofing vector: "modify /etc/hosts". 📄 pp. 587-588
/etc/resolv.conf Tells the machine which DNS resolver to use (typically set by DHCP). Attacker who can write here redirects all DNS to a malicious server. The other explicit DNS-spoofing vector on slide 588. 📄 pp. 587-588

The key structural fact: the server allocates a TCB at step 1 (when it gets the SYN) and only releases/moves it when the ACK arrives at step 3. That TCB-allocation-on-SYN is precisely what SYN flooding targets. 📄 p. 548

TCP SYN Flooding 📄 pp. 549-551

① Mechanism 📄 p. 549
The attacker sends a continuous stream of SYN packets to the victim server (often with spoofed source IPs so the real address cannot be back-traced). The server allocates a TCB for each SYN into its half-open queue. The attacker never sends the third ACK, so each TCB lingers until it times out. With enough pending SYNs, the queue fills and the server refuses all new legitimate connections.
② Preconditions
Needed: Ability to send raw SYN packets at high rate. That's it — no authentication is required because TCP has none at this stage.
Not needed: Knowledge of sequence numbers (you aren't trying to impersonate a connection, you are trying to exhaust state).
③ Defense — SYN Cookies 📄 pp. 550-551
Rather than allocate a TCB on receiving a SYN, the server computes H = keyed-hash(src IP, src port, dst IP, dst port, secret) and returns H as the server's ISN in the SYN-ACK. The server stores nothing. When a genuine ACK arrives with ack = H + 1, the server recomputes H and verifies. Attackers who spoofed source IPs never see the SYN-ACK, so they cannot send a valid ACK. This offloads server state onto the client's ACK, neutralizing the flood. 📄 pp. 550-551

TCP Reset Attack 📄 pp. 552-554

① Mechanism 📄 pp. 552-553
The attacker injects a forged TCP packet with the RST flag set, causing both endpoints to tear down the connection immediately. Unlike the graceful FIN/ACK/FIN/ACK close, a single RST ends the session with no round-trip.
② Preconditions — the five mandatory fields 📄 p. 553
The forged RST must carry a correct value in all of:
  1. Source IP address
  2. Source port
  3. Destination IP address
  4. Destination port
  5. Sequence number within the receiver's current window
A local attacker (on the same LAN) can sniff the live connection with Wireshark/tcpdump and observe all five fields directly. A remote attacker, who cannot sniff, must guess — and the sequence number alone is a 32-bit space, cut down to the ~receive-window-sized valid range. That is why the instructor said remote reset attacks are difficult. 📄 pp. 553-554
③ Defense 📄 pp. 554, 556
Randomize source port (beyond the OS default range) and randomize the initial sequence number heavily 📄 pp. 551, 554. These make remote guessing infeasible but do not help against a local attacker who can sniff. The ultimate defense is to encrypt the transport payload with IPsec, which hides the TCP header from any observer (local or remote) inside the encrypted payload. 📄 pp. 370-371, 556

TCP Session Hijacking 📄 pp. 555-556

① Mechanism 📄 p. 555
Same five-field precondition as RST, but instead of a RST flag, the attacker injects a segment with data payload and a sequence number the receiver will accept. If the receiver already has data up to byte x, the attacker must use sequence number x + 1; otherwise the bytes land out-of-order (gap left at positions x+1 … x+δ) and may fall outside the buffer.
② Preconditions
Identical to RST plus tighter sequence-number accuracy. Local attackers succeed easily; remote attackers are essentially blocked without sniffing. 📄 pp. 553-556
③ Defense 📄 p. 556
Randomize ISN and randomize source port (helps against remote); encrypt with IPsec (helps against both local and remote). The instructor noted the randomization approaches are not effective against local attackers — only encryption is.
2bIP / Network-Layer Attacks 📄 pp. 505-528 · Lecture 19 pt. 2

IP Address Spoofing 📄 pp. 507-512

① Mechanism 📄 pp. 507-508
An IP packet's source address is a header field filled in by the originating host. There is no check on whether it is truthful. Any sender can put any source IP in the field — using tools like Scapy (Python) or libnet (C) 📄 pp. 509-510 — and the packet routes normally to its destination. The destination replies to the forged source address.
② What it enables 📄 pp. 511-512
  • Unauthorized access — if a server uses source IP for weak "authentication" (r-utilities like rlogin/rsh relied on this historically) 📄 p. 507
  • DDoS / reflection — spoof the victim's IP as source, send requests to amplifiers, replies drown the victim 📄 pp. 511, 521
  • Enables downstream attacks — RST injection and session hijacking all start with a spoofed packet
③ Defenses 📄 p. 512
Slide 512 lists the full defense stack:
  1. Ingress filtering — at the edge of your network, drop incoming packets whose source IP cannot legitimately arrive on that interface
  2. Egress filtering — ISPs drop outgoing packets from their customers whose source IP is not in that customer's assigned range. The instructor said this is more effective because it stops spoofed packets at the source.
  3. Avoid trust relationships based on IP address — never use source IP as authentication
  4. TTL-value check (see pp. 513-517 for the probe-based detection scheme)
  5. IPsec — cryptographically authenticate source at layer 3
  6. Randomized Initial Sequence Number in TCP — indirectly makes the spoof less useful
  7. Unicast Reverse-Path Forwarding (uRPF)

ICMP Echo scan (ping scan / nmap): Send ICMP Echo Requests to every IP in a subnet; collect replies to map live hosts. This is the reconnaissance step that often precedes a more targeted attack. Defense: network segmentation (VLANs) so that an attacker landing in an IT subnet cannot ping the OT / manufacturing subnet. 📄 pp. 520, 362, 393

ICMP Destination Unreachable: Attacker forges ICMP "destination unreachable" messages to one host to trick it into dropping a legitimate connection — effectively isolating the victim. ICMP has no authentication, so the forgery is trivial on the wire. 📄 pp. 522-523, 528

Ping of Death: Multiple IP fragments that reassemble to a datagram > 65 535 bytes (the IP maximum), overflowing a kernel buffer in old TCP/IP stacks. Historical but named. 📄 p. 527

Fragmentation attack for firewall evasion: Since some firewalls inspect only the first fragment, an attacker can place benign headers in fragment 1 and malicious content (e.g., TCP flags) in later fragments. 📄 p. 526

2cARP (Link-Layer) Attacks 📄 pp. 529-538 · Lecture 19 pt. 2

How ARP Normally Works 📄 pp. 529-530

When Host A wants to send an IP packet to Host B on the same LAN, it needs B's MAC address. It broadcasts an ARP Request: "Who has IP 192.168.1.10? Tell 192.168.1.100." B replies unicast: "192.168.1.10 is at 00:01:03:1D:98:B8." A caches this mapping for subsequent traffic. 📄 p. 530

Two design choices set up the attack:

  • Stateless — a host accepts an ARP reply even if it never sent a request. 📄 p. 531
  • Replies overwrite the cache — the latest reply wins, so repeated spoofed replies keep the cache poisoned. 📄 p. 538

ARP Spoofing → Man-in-the-Middle 📄 pp. 531-537

 Initial state:
   Host A (192.168.1.100, MAC 08:00:46:07:04:A3)
   Host B (192.168.1.10,  MAC 00:01:03:1D:98:B8)
   Host C (192.168.1.137, MAC BA:DB:AD:BA:DB:AD)  ◄── attacker

 ┌─ Attack step 1 ────────────────────────────────────────────────┐
 │  C → A:   "192.168.1.10  is at  BA:DB:AD:BA:DB:AD"    (spoof) │  📄 p. 533
 │  C → B:   "192.168.1.100 is at  BA:DB:AD:BA:DB:AD"    (spoof) │  📄 p. 535
 └────────────────────────────────────────────────────────────────┘

 Poisoned state — both A's and B's ARP caches now point
 their peer's IP to C's MAC:                                       📄 p. 536
   A: 192.168.1.10  → BA:DB:AD:BA:DB:AD   (should be 00:01:03:…)
   B: 192.168.1.100 → BA:DB:AD:BA:DB:AD   (should be 08:00:46:…)

 ┌─ Data exchange post-poisoning ─────────────────────────────────┐
 │                                                                │
 │   A ──── "to 192.168.1.10" ────▶  C  ──── forward ──▶  B       │  📄 p. 537
 │                                   │                             │
 │                                   └── can read / modify every  │
 │                                       packet (MITM)            │

The attacker acts as a silent router between A and B. To keep the poisoning active — because a legitimate ARP reply would correct the cache — most ARP-spoofing tools send their forged replies repeatedly (gratuitous ARP). 📄 p. 538

Defenses:

  • Static ARP entries for critical gateways (operational overhead)
  • DHCP snooping + Dynamic ARP Inspection on managed switches
  • End-to-end encryption (TLS, SSH, IPsec) — if traffic is encrypted, a MITM who cannot produce the right certificate cannot read or modify the payload. Instructor's advice on slide 538: "Avoid sending data in cleartext — HTTP, FTP, Telnet." 📄 p. 538

Two additional local-machine files participate before the network lookup: /etc/hosts (a static name → IP map, consulted first) and /etc/resolv.conf (tells the machine which DNS server to use). Modifying either is an explicit DNS-spoofing vector on slide 588. 📄 pp. 587-588

Attack Category 1 — DNS Spoofing (LAN / Local) 📄 p. 589

① Mechanism
The attacker is on the same LAN as either the client or the local DNS server. When the client sends a DNS query out, the attacker (who can sniff) immediately sends a spoofed reply using the local DNS server's IP as the source. Because the attacker sees the real query, they copy the transaction ID, source port, and question section exactly — the victim accepts the reply.
② Variant: Cache Poisoning on the Local DNS Server
When the local DNS itself sends an iterative query outward, a LAN attacker can spoof the reply "to" the local DNS, using a remote-nameserver-looking source. If the spoof arrives before the real reply, the poisoned answer is cached and served to every downstream client until the TTL expires.

Attack Category 2 — Remote DNS Cache Poisoning 📄 pp. 590-591

① Mechanism
Same goal (poison the cache) but the attacker cannot sniff the outbound query. They must guess two 16-bit random numbers:
  1. Source port of the query (chosen randomly by the resolver) — 16 bits
  2. Transaction ID (chosen randomly by the client for each query) — 16 bits
Total guess space: 2³² ≈ 4 billion combinations per attempt.
② The "Cache Effect" Obstacle 📄 p. 590
After a failed guess, the real reply arrives and gets cached. The attacker must wait for the cache entry to time out before trying again — dramatically slowing the attack.

Attack Category 3 — Kaminsky Attack 📄 pp. 592-593

① The Trick 📄 p. 592
Kaminsky's insight was to ask a different question every time. Instead of querying www.example.com repeatedly (where one failed guess caches the real answer), the attacker queries random subdomains: aaaa.example.com, aaab.example.com, … Each is a cache miss that forces a fresh outward query → a fresh attempt.
② The Payload 📄 p. 593
The spoofed reply does not need to provide a useful answer for the random subdomain. What matters is the Authority / Additional section, which tells the local DNS: "Here is the authoritative nameserver for the whole example.com domain — and its IP is <attacker's IP>." That authoritative pointer gets cached and used for every future *.example.com lookup.
③ Why it was devastating
Before Kaminsky, DNS resolvers often used a fixed source port, cutting the guess space to 16 bits. The patch that followed randomized source ports at every resolver, restoring the 32-bit guess space. But the Kaminsky technique still works against any deployment that does not enforce full port randomization.

Defenses Against DNS Attacks 📄 pp. 594-597

① DNSSEC — Digital signatures on DNS records 📄 pp. 595, 597
Each DNS zone signs its records with a private key. The parent zone vouches for the child's public key. Chain of trust rooted at the DNS zone hierarchy: Root → TLD → authoritative → your record.
② DNS over TLS (or HTTPS) 📄 p. 596
Even if a poisoned reply gives the victim an attacker's IP, the subsequent TLS handshake to that IP fails because the attacker cannot produce a valid certificate for www.example.com. So HTTPS effectively defeats DNS cache poisoning at the application layer — even though it doesn't fix DNS itself.
Cross-topic hook "DNSSEC relies on the DNS zone hierarchy for its chain of trust, while TLS/SSL relies on the CA hierarchy (Public Key Infrastructure). Both use public-key signatures — what differs is where the chain is rooted." 📄 p. 597 — this exact comparison is the instructor's #1 integration question between Topic 2 and Topic 3/4.

DoS Attacks on DNS Infrastructure 📄 pp. 598-601

  • Root servers (pp. 599) — 13 logical servers (A-M) implemented as anycast clusters. Also, the 48-hour TTL on TLD entries means most resolvers rarely query root directly. Attacks must be sustained and massive to have impact.
  • TLD servers (pp. 600) — Large TLDs (.com, .net) are resilient; obscure country-code TLDs are softer targets and can effectively knock a whole country offline.
  • Authoritative nameservers (pp. 601) — DNS providers like UltraDNS host many companies' records; the 2004 UltraDNS DoS took Amazon/Walmart/Expedia offline for an hour.

3.2 — Common Misconceptions & Traps

❌ MYTH: "SYN cookies prevent SYN floods by filtering malicious SYNs at the firewall."

✅ REALITY: SYN cookies do not filter or drop SYNs — the server still accepts every SYN. The fix is that the server allocates no state (no TCB) until the ACK returns a valid keyed-hash cookie. State is offloaded onto the client's ACK value. 📄 pp. 550-551

🎯 EXAM TRAP: Answer-option phrased as "SYN cookies block incoming SYN packets" — this is wrong.

❌ MYTH: "Ingress filtering is what ISPs do to stop IP spoofing."

✅ REALITY: Ingress and egress filtering are applied at different points and catch different things. Egress filtering at the ISP drops customer packets with forged source IPs leaving the customer network — this is the most effective because it stops spoofing at the origin. Ingress filtering at your own network edge drops incoming packets whose source IP could not legitimately come from that interface (e.g., a packet from the outside claiming to have an internal source). 📄 pp. 403, 425, 512

🎯 EXAM TRAP: Multi-answer MCQ mixes up which side does which.

❌ MYTH: "A remote TCP RST attack is roughly as easy as a local one."

✅ REALITY: Local attackers can sniff the connection and read all five required fields — source IP, source port, destination IP, destination port, and the current sequence-number window — directly. Remote attackers, who cannot sniff, must guess the sequence number within the receive window (a 32-bit space narrowed by window size). The instructor explicitly said remote RST is "difficult." Sequence-number randomization defeats remote attacks; only encryption (IPsec) defeats local ones. 📄 pp. 553-556

🎯 EXAM TRAP: T/F: "Randomizing the initial sequence number protects against local TCP RST attacks." — FALSE.

❌ MYTH: "ARP spoofing can be done over the Internet from anywhere."

✅ REALITY: ARP is a link-layer protocol that operates only within a single broadcast domain (LAN). Routers do not forward ARP messages. So ARP spoofing is strictly a local-network attack. 📄 pp. 529-531

🎯 EXAM TRAP: Multi-answer: "Which of these attacks can a remote attacker perform from anywhere on the Internet?" — ARP spoofing should NOT be selected.

❌ MYTH: "DNSSEC and TLS both use the same chain of trust."

✅ REALITY: Both use public-key signatures, but where the chain is rooted is different. DNSSEC roots its chain in the DNS zone hierarchy — Root signs TLD, TLD signs second-level, second-level signs your record. TLS/SSL roots its chain in the Public Key Infrastructure (PKI) — a small number of Certificate Authorities (CAs) that the browser trusts by default. 📄 p. 597

🎯 EXAM TRAP: T/F: "DNSSEC uses Certificate Authorities just like TLS." — FALSE.

❌ MYTH: "The Kaminsky attack works because DNSSEC is broken."

✅ REALITY: The Kaminsky attack targets plain (non-DNSSEC) DNS. It bypasses the cache effect that previously slowed DNS cache poisoning by asking random subdomains. DNSSEC, once deployed, actually defeats Kaminsky because the spoofed reply has no valid signature. 📄 pp. 592-593, 595

🎯 EXAM TRAP: "Kaminsky attack defeats DNSSEC" — this is the reverse; DNSSEC defeats Kaminsky.

3.5 — Memory Aids

Mnemonic — "Four Attack Families" = T-I-A-D TCP (transport) → IP (network) → ARP (link) → DNS (app). Top-to-bottom-and-out the stack. "Top Internet Attack Directions." 📄 pp. 541, 505, 529, 580
Mnemonic — "Five Fields for a TCP RST" src IP · src port · dst IP · dst port · seq# (in window). Think of it as an address on a letter: from street, from apartment, to street, to apartment, page number. Miss any one and the receiver discards the packet. 📄 p. 553
Mnemonic — "Two 16's for remote DNS poisoning" Remote attacker must guess source port (16 bits) + transaction ID (16 bits) = 2³² ≈ 4 billion attempts per query. Kaminsky's trick asks a different question every time to defeat the cache effect. 📄 pp. 590-593
Mnemonic — "SYN-cookie = no TCB" The server computes a keyed-hash H and sends H as its ISN. No state is stored. A genuine client echoes H back (well, H+1 in ACK); the server just recomputes to verify. No TCB = no queue = no flood. 📄 pp. 550-551
Mnemonic — "DNSSEC zones, TLS CAs" Same math (public-key signatures), different tree root. DNSSEC's chain of trust is the zone hierarchy: Root → TLD → authoritative. TLS's chain of trust is the CA hierarchy: Root CAs → intermediates → server cert. 📄 p. 597
Mnemonic — "ARP is stateless, so it's spoof-able" Unsolicited ARP replies are accepted and cached. The attacker just has to keep sending them so that any correcting legitimate reply gets overwritten again. 📄 pp. 531, 538

Part A — Multiple Choice (4 questions: 1 BEG, 1 INT, 2 ADV)

BEG MCQ 📄 pp. 547-549

Q1. In a TCP SYN flooding attack, which server-side resource is primarily exhausted?

  1. The server's CPU cycles spent computing TCP checksums.
  2. The server's half-open connection queue, which stores a TCB for every SYN received.
  3. The server's cryptographic key-storage buffer used for TLS.
  4. The server's application-layer thread pool.
▶ Show Answer

✅ Correct Answer: B) The half-open connection queue / TCB storage.

📝 Explanation:

  • Why B is correct: Slide 548 states: "the server stores the TCB in a queue that is only for the half-open connection... if ACK doesn't arrive, the server will resend SYN+ACK packet. The TCB will eventually be discarded after a certain time period." The attacker fills this queue with incomplete connections. 📄 pp. 548-549
  • Why A is wrong: TCP checksum computation is trivial and is not the bottleneck.
  • Why C is wrong: TLS/crypto is a higher layer; SYN flooding occurs before any TLS handshake.
  • Why D is wrong: Application threads are not allocated until the 3-way handshake completes; a half-open connection has no application thread yet.
  • Exam Tip: If "half-open connection queue" appears in an answer option, it's almost always the SYN-flood answer.
INT MCQ 📄 p. 553

Q2. To successfully inject a TCP RST packet that tears down an existing connection between Alice and Bob, which fields must the attacker set correctly in the forged packet?

  1. Only the source and destination IP addresses.
  2. Source IP, source port, destination IP, and destination port — the sequence number is optional.
  3. Source IP, source port, destination IP, destination port, and a sequence number within the receiver's current window.
  4. Only the destination IP and destination port, along with the RST flag.
▶ Show Answer

✅ Correct Answer: C) All five fields — the classic "5-tuple + seq#".

📝 Explanation:

  • Why C is correct: Slide 553 explicitly lists: "Source IP address, Source Port, Destination IP address, Destination Port, Sequence number (within the receiver's window)." All five are mandatory. 📄 p. 553
  • Why A is wrong: TCP connections are identified by the full 4-tuple plus the sequence number state; IPs alone cannot match a specific connection.
  • Why B is wrong: The receiver rejects any RST whose sequence number falls outside the current receive window — it is not optional.
  • Why D is wrong: Without the source fields, the packet is not attributable to the connection and the receiver discards it.
  • Exam Tip: The "5-tuple + seq# within window" requirement is also why remote RST attacks are hard — the attacker cannot sniff to discover the current seq#.
ADV MCQ · multi-answer 📄 p. 512

Q3. Select all defenses against IP address spoofing that slide 512 explicitly lists.

  1. Ingress packet filtering at the network edge.
  2. Egress packet filtering at the ISP.
  3. Avoiding trust relationships based on IP address.
  4. TTL-value checks (with TCP probe verification).
  5. IPsec with cryptographic source authentication.
  6. Randomized Initial Sequence Number in TCP.
  7. Unicast Reverse-Path Forwarding (uRPF).
  8. Disabling all ICMP Echo responses on the host.
▶ Show Answer

✅ Correct Answers: A, B, C, D, E, F, G. (Everything except H.)

📝 Explanation:

  • Why A-G are correct: Slide 512 lists all seven verbatim: "Packet filtering (Ingress filtering, Egress filtering); Avoiding trust relationship based on IP address; TTL Value; IPSec; Randomized Initial Sequence Number in TCP; Unicast Reverse Path Forwarding." 📄 p. 512
  • Why H is wrong: Disabling ICMP Echo responses is a defense against ping scans and Smurf amplification — it does not prevent an attacker from forging IP source addresses. 📄 pp. 520-521
  • Exam Tip: On multi-answer MCQs where seven of eight options are correct, always look for a related but mis-categorized distractor — that's usually the wrong one.
ADV · challenge MCQ 📄 pp. 595-597

Q4. Suppose an attacker successfully poisons your local DNS cache so that www.bank.com resolves to the attacker's IP. The victim then opens HTTPS to www.bank.com. What happens?

  1. The attacker can read all traffic in cleartext — poisoning DNS also defeats TLS.
  2. The TLS handshake fails because the attacker cannot produce a valid certificate for www.bank.com signed by a trusted Certificate Authority.
  3. The TLS handshake succeeds but the browser shows a warning only if DNSSEC is enabled.
  4. TLS is bypassed entirely because DNS poisoning changes the routing layer.
▶ Show Answer

✅ Correct Answer: B) The TLS handshake fails; the attacker cannot forge the certificate.

📝 Explanation:

  • Why B is correct: Slide 596 states: "HTTPS is built on top of TLS/SSL. It defeats DNS cache poisoning attacks." After DNS resolution, the client still demands the server prove it owns the name via a CA-signed certificate. The attacker at the wrong IP cannot produce a certificate for www.bank.com that chains to a trusted CA, so the handshake aborts with a certificate error. 📄 p. 596
  • Why A is wrong: TLS and DNS are independent trust anchors; poisoning one does not compromise the other.
  • Why C is wrong: The certificate warning fires regardless of DNSSEC status — TLS has its own PKI chain-of-trust mechanism. 📄 p. 597
  • Why D is wrong: DNS returns an IP; routing to that IP happens normally. TLS runs on top of the resulting TCP connection, independent of how the IP was obtained.
  • Exam Tip: This is exactly the Topic 2 ⊕ Topic 4 integration the instructor flagged. The slide 597 DNSSEC-vs-TLS comparison is the cleanest short-answer version of the same idea.

Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG TF 📄 pp. 529-531

Q5. Statement: A remote attacker located on a different continent from the victim can perform an ARP spoofing attack against that victim's laptop as long as both machines are connected to the Internet.

▶ Show Answer

✅ Answer: FALSE.

ARP is a link-layer protocol that operates only within a single broadcast domain (the local LAN). Routers do not forward ARP frames across network boundaries. A remote attacker on a different network cannot send ARP messages to the victim's LAN. ARP spoofing is strictly a local attack — the attacker must be on the same physical/logical LAN as the victim. 📄 pp. 529-531

INT TF 📄 pp. 550-551

Q6. Statement: When SYN cookies are enabled, the server drops every incoming SYN packet until the client has completed a cryptographic handshake proving it is not a flood source.

▶ Show Answer

✅ Answer: FALSE.

SYN cookies do not drop SYN packets. The server still replies to every SYN with a SYN-ACK — but crucially, it sends back a keyed hash H as the server's ISN without storing any TCB. Only when (and if) the client returns a valid ACK with H + 1 does the server recompute the cookie, verify, and then allocate connection state. The key insight is that state allocation is deferred from SYN to ACK, so a flood of SYNs without matching ACKs costs the server essentially nothing. 📄 pp. 550-551

ADV · subtle TF 📄 pp. 553-556

Q7. Statement: Randomizing the TCP Initial Sequence Number (ISN) and source port successfully defeats both local and remote TCP Reset / Session Hijacking attacks.

▶ Show Answer

✅ Answer: FALSE.

Randomization helps against remote attackers who must guess these values, because the guess space explodes combinatorially. But a local attacker on the same LAN can sniff the ongoing connection with Wireshark/tcpdump and observe the current sequence number and ports directly — no guessing required. Slide 556 makes this explicit: "Randomize source port number; Randomize initial sequence number — Not effective against local attacks." The only defense that works against local attackers is encrypting the transport payload (IPsec), which hides the TCP header from the sniffer. 📄 p. 556

Part C — Short Answer (3 questions: 1 BEG, 1 INT, 1 ADV)

BEG SA 📄 pp. 531-537

Q8. Describe, in 3-5 bullet points, how ARP spoofing enables a Man-in-the-Middle attack between two hosts A and B on the same LAN.

▶ Show Answer

📝 Model Answer:

  • The attacker C sends an unsolicited ARP reply to A claiming "B's IP is at C's MAC address." Because ARP is stateless, A accepts and caches this mapping. 📄 p. 533
  • C sends a matching unsolicited ARP reply to B claiming "A's IP is at C's MAC address." B similarly caches it. 📄 p. 535
  • Now when A sends a frame destined for B's IP, the link-layer framing uses C's MAC as the destination — so the frame is physically delivered to C, not B. Same for B → A. 📄 p. 537
  • C acts as a silent router between A and B, reading and optionally modifying every frame, then forwarding the packet on to the real destination using the correct MAC it still knows. 📄 p. 537
  • To keep the poisoning active against any legitimate ARP traffic that would correct the caches, C repeats its gratuitous ARP replies periodically. 📄 p. 538

🔑 Key Points (Grading Checklist):

  • ☐ Identifies that two separate poisonings happen (A's cache AND B's cache) 📄 pp. 533-535
  • ☐ States that ARP is stateless / unsolicited replies are accepted 📄 p. 531
  • ☐ Explains how traffic flows through the attacker (MITM) 📄 p. 537
  • ☐ Mentions the need to keep replaying (gratuitous ARP) 📄 p. 538
INT SA 📄 pp. 590-593

Q9. Explain the Kaminsky DNS attack. Specifically: what two random numbers must the attacker match in a spoofed reply, and how does asking for "random subdomains" help the attacker?

▶ Show Answer

📝 Model Answer:

For a remote attacker, a spoofed DNS reply is accepted only if it matches the source port (16-bit random, chosen by the resolver) and the transaction ID (16-bit random, chosen by the client for each query) of the outstanding legitimate query. Before Kaminsky, attackers faced a "cache effect" problem: once a failed guess arrives, the resolver accepts the real reply that follows and caches it, so the attacker must wait for the TTL to expire before retrying. 📄 p. 590

Kaminsky's insight was to change the question itself every attempt. Instead of querying www.example.com repeatedly, the attacker induces queries for random subdomains — aaaa.example.com, aaab.example.com, etc. Each is a fresh cache miss, so the resolver issues a new outward query every time. The spoofed reply carries a valid-looking answer for the random subdomain but also includes an Authority / Additional section that reassigns the authoritative nameserver for the whole example.com domain to the attacker's IP. That delegation gets cached and is used for every subsequent *.example.com lookup. 📄 pp. 592-593

🔑 Key Points (Grading Checklist):

  • ☐ Names the two 16-bit numbers: source port + transaction ID 📄 p. 590
  • ☐ Identifies the "cache effect" as the obstacle Kaminsky bypasses 📄 p. 590
  • ☐ Explains that random subdomains cause cache misses every attempt 📄 p. 592
  • ☐ Mentions that the payload is the delegation (Authority section), not the direct answer 📄 p. 593
ADV · challenge SA 📄 p. 597  +  pp. 240-244 (TLS PKI)

Q10. Both DNSSEC and TLS/SSL use public-key signatures, but their chains of trust are structurally different. Contrast them: where is the root of trust in each system, and what infrastructure is responsible for vouching for subordinate entities?

▶ Show Answer

📝 Model Answer:

DNSSEC roots its chain of trust in the DNS zone hierarchy. The Root zone signs and publishes a key for each TLD (e.g., .com, .edu). Each TLD in turn signs keys for its second-level domains (e.g., ua.edu), which sign keys for their own sub-zones. A client validating a record walks upward through the zone hierarchy until reaching the Root key that is hard-coded in the resolver. The key property: the parent zone vouches for the child zone, mirroring the DNS delegation structure. 📄 pp. 595, 597

TLS/SSL roots its chain of trust in the Public Key Infrastructure (PKI). A small number of Certificate Authorities are pre-installed in every browser / OS as trusted roots. A CA signs a server's X.509 certificate, possibly via intermediate CAs. The client walks from the server certificate up through the intermediates until it reaches a pre-trusted Root CA. The key property: a separate body (the CA) vouches for the server's identity — there is no inherent relationship between the CA and the domain hierarchy. 📄 p. 597, pp. 240-244

Practical consequence: DNSSEC protects DNS answers themselves. TLS protects the TCP/IP connection after DNS has handed you an IP. They address different threats with the same cryptographic building block, and HTTPS effectively compensates for non-DNSSEC DNS because even a poisoned IP cannot produce a valid CA-signed certificate. 📄 p. 596

🔑 Key Points (Grading Checklist):

  • ☐ DNSSEC: chain rooted at DNS Root zone, parent-vouches-for-child structure 📄 p. 597
  • ☐ TLS: chain rooted at trusted CAs (PKI), CAs vouch for server certificates 📄 p. 597
  • ☐ Both use public-key signatures (same underlying technology) 📄 p. 597
  • ☐ Bonus: notes that HTTPS defeats DNS cache poisoning independently of DNSSEC 📄 p. 596

A top-to-bottom catalog of how the core Internet protocols — TCP, IP/ICMP, ARP, and DNS — can be abused because they lack built-in authentication, and what defenses (randomization, filtering, encryption, digital signatures) we deploy to restore that authentication at each layer. 📄 pp. 505-601
