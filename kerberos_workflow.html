Phase 0 — Pre-Setup

Before any login · One-time admin configuration
# Phase & Action Actor (Sender) Recipient Object / Key Involved What happens (procedure) Why this design (rationale) Security guarantee
0.1
Setup

KDC registers client		 KDC Admin KDC Database Kc
(long-term client key)		 Admin creates an account for Alice. The KDC computes Kc = hash(Alice's password) and stores Kc in its database. Kc is the cryptographic key that represents Alice's identity — it is never the raw password, always the derived key. The same hash function will run on Alice's own machine at login time to reproduce Kc locally without the KDC sending it to her. Kc is stored instead of the raw password so that even if the KDC database is stolen, the attacker gets only a derived key, not a reusable plaintext password. The derivation is deterministic — Alice can always re-derive Kc from her password without the KDC sending anything to her. If the KDC database is compromised, passwords are not directly exposed — only their derived keys. Alice's password never leaves her machine in any form.
0.2
Setup

KDC registers each service		 KDC Admin KDC Database + Server V Kv
(long-term server key — one per service)		 Admin generates a random secret key Kv specifically for service V (e.g., the file server). Kv is stored in two places: (1) the KDC database alongside V's service record; (2) installed directly on Server V's machine by the sysadmin out-of-band. Kv is not derived from a password — it is a purely random key. Each service (file server, mail server, print server) gets its own completely independent Kv. A random key per service means no shared secret between services. If the mail server's Kv is stolen, the file server's Kv is entirely unaffected. The out-of-band installation (sysadmin physically installs it on the server) avoids transmitting Kv over the network at all. Compromise of one service's Kv does not expose any other service or any client. Blast radius is limited to a single service.
0.3
Setup

TGS key established		 KDC Admin KDC (AS component) + TGS KTGS
(long-term TGS key)		 Admin generates a random secret key KTGS shared between the KDC's Authentication Server (AS) component and the Ticket Granting Service (TGS). KTGS is what allows the AS to seal TGT tickets that only the TGS can open. In most real deployments, AS and TGS run on the same physical KDC machine, so KTGS is an internal key on that machine. KTGS is what makes the TGT mechanism possible. The AS encrypts TGTs with KTGS so that the client — who carries the TGT — cannot read or modify its contents. Only the TGS, the intended recipient, can decrypt it using KTGS. The client is a ticket courier, not a ticket creator. The TGT's integrity and confidentiality are guaranteed by KTGS encryption. The client cannot forge or tamper with a TGT.

Phase 1 — Initial Authentication

Client ↔ Authentication Server (AS) · Password verification · TGT issuance
#Phase & ActionActor (Sender)Recipient Object / Key InvolvedWhat happens (procedure) Why this design (rationale)Security guarantee
1.1
AS

Client sends login request		 Client (Alice) AS (KDC) ID_client — Alice's username
ID_TGS — TGS identifier
TS1 — Alice's current timestamp
All sent in plaintext — no encryption 		Alice's machine sends three fields: (1) ID_client = Alice's username (e.g., "alice@REALM.COM") — tells the AS which account is requesting authentication; (2) ID_TGS = the name of the Ticket Granting Service (e.g., "krbtgt/REALM.COM") — tells the AS Alice wants a TGT specifically, not a service ticket; (3) TS1 = Alice's current clock time — included so the AS can correlate this request and so the TGT validity window has a clear start point. No password, no key, no credential of any kind is sent. Sending only ID_client — not the password or Kc — is the foundational security decision of Kerberos. The AS does not need the password to verify Alice. Instead, it encrypts its response with Kc, and only the real Alice (who can re-derive Kc from her password) can decrypt it. The verification is implicit. ID_TGS is explicit so the AS knows the intended scope. TS1 prevents replay of this request itself. The password cannot be intercepted, sniffed, or stolen from the network — it never appears on the wire in any form, not even encrypted.
1.2
AS

AS looks up client record		 AS (KDC) KDC Database Kc
(retrieved from DB using ID_client as lookup key)		 The AS receives ID_client = "alice@REALM.COM" and queries the KDC database using this as the primary key. The database record returns Kc — the stored key derived from Alice's password during account setup (Phase 0.1). The AS now holds Kc in memory and uses it in steps 1.4 and 1.5. The KDC is the single authoritative store of all long-term keys. Every client's Kc and every server's Kv lives here. This centralisation is Kerberos's core design trade-off: strong security guarantees in exchange for a single point of failure. If the KDC is unavailable, no authentication is possible. AS can now prepare a response that only the real Alice can decrypt, because only Alice (via her password) and the KDC database know Kc.
1.3
AS

AS generates fresh session key		 AS (KDC) Internal (not yet sent) Kc,TGS
(brand-new random session key — generated fresh for this login session)		 The AS calls a cryptographically secure random number generator (CSPRNG) to produce a new Kc,TGS. This key has never existed before and will never be reused. Kc,TGS is the session key that Alice and the TGS will use to communicate securely for the duration of this login session (until the TGT expires). It is not derived from any password — it is purely random. Generating a fresh Kc,TGS per session means: (1) if this session's key is ever compromised, it reveals nothing about Alice's password (Kc) or any other session's key; (2) an attacker recording encrypted traffic today cannot decrypt it even if they later steal Kc. The session key is the workhorse; Kc is the master key used only to bootstrap. Session key compromise is time-bounded (expires with TGT) and session-isolated — one session's key reveals nothing about any other session's key or the underlying password.
1.4
AS

AS constructs the TGT		 AS (KDC) Internal (will become Part B of response) Encrypted with: KTGS
Contents:
Kc,TGS, ID_client, AD_client, ID_TGS, TS1, Lifetime 		AS assembles the TGT plaintext with: (1) Kc,TGS — the session key just generated; the TGS will need this to later verify Alice's Authenticators; (2) ID_client = "alice@REALM.COM" — so the TGS knows whose identity this TGT represents; (3) AD_client = Alice's network address (IP) — optionally binds the ticket to Alice's machine; (4) ID_TGS — confirms this TGT is intended for the TGS; (5) TS1 — start of validity window; (6) Lifetime — typically 8–10 hours. The AS then encrypts this entire structure with KTGS:
TGT = E(KTGS, [Kc,TGS ‖ ID_client ‖ AD_client ‖ ID_TGS ‖ TS1 ‖ Lifetime])		 Encrypting with KTGS achieves two things: (1) the client, who carries this TGT, cannot read what is inside — specifically cannot read Kc,TGS embedded inside it; (2) the client cannot modify the TGT — any tampering corrupts the encryption and TGS will reject it. Lifetime is inside the encrypted content so the client cannot extend it. AD_client binds the ticket to Alice's IP. TGT is simultaneously opaque (client cannot read Kc,TGS inside) and tamper-proof (KTGS encryption detects any modification). A stolen TGT without Kc,TGS is useless.
1.5
AS

AS encrypts session key for client		 AS (KDC) Internal (will become Part A of response) Encrypted with: Kc
Contents:
Kc,TGS, ID_TGS, TS1 		The AS takes Kc,TGS (the same session key embedded in the TGT) and encrypts it separately with Kc (Alice's key from the database):
Part A = E(Kc, [Kc,TGS ‖ ID_TGS ‖ TS1])
Fields inside: (1) Kc,TGS — the session key Alice needs to communicate with TGS; (2) ID_TGS — tells Alice which service this key is for; (3) TS1 — echoed back so Alice can verify the AS responded to her specific request and not a replayed old response.		 This is the secure key delivery mechanism. Encrypting with Kc means only Alice (who can derive Kc from her password) can unwrap it. This is also the implicit password verification step — if Alice types the wrong password, she derives the wrong Kc, decryption of Part A produces garbage, and she cannot extract a valid Kc,TGS. The AS never explicitly says "wrong password." Kc,TGS is delivered securely: a network attacker who intercepts Part A cannot extract Kc,TGS without knowing Kc, which requires knowing Alice's password.
1.6
AS

AS sends complete response to client		 AS (KDC) Client (Alice) Part A: E(Kc, [Kc,TGS ‖ ID_TGS ‖ TS1])

Part B (TGT): E(KTGS, [Kc,TGS ‖ ID_client ‖ AD_client ‖ ID_TGS ‖ TS1 ‖ Lifetime]) 		The AS bundles Part A and Part B (TGT) into a single response message to Alice. Part A is addressed to Alice (encrypted with Kc — only Alice can open it). Part B is addressed to the TGS (encrypted with KTGS — only TGS can open it). Alice receives both simultaneously but can only read Part A. Bundling both in one message reduces round trips. Alice needs both pieces immediately: Kc,TGS from Part A to authenticate to the TGS, and the TGT from Part B to present to the TGS as proof of KDC endorsement. Neither piece is useful without the other. Single round trip completes the AS phase. Part A and Part B each have different intended recipients, so interception of either part by a third party yields no useful information.
1.7
AS

Client derives Kc locally		 Client (Alice) Internal (no network) Input: password (typed by Alice)
Output: Kc = hash(password)
Computed entirely on Alice's machine 		Alice types her password into the Kerberos client software. The software immediately runs Kc = hash(password) using the same algorithm the KDC used during account setup (Phase 0.1). This computation happens entirely on Alice's machine — no network packet is sent. Kc exists only in Alice's local memory for the duration of the decryption operation in step 1.8. The password is the one thing that never leaves Alice's machine. By using Kc = hash(password) rather than the password itself, Kerberos cleanly separates the human-memorable password from the cryptographic key. Even if Alice's machine memory is dumped, Kc is a hash — an attacker cannot reverse it to recover the original password (one-way hash property). The password cannot be stolen from the network because it is never placed on the network. Kc is also discarded after step 1.8 — its in-memory lifetime is seconds, not hours.
1.8
AS

Client decrypts Part A		 Client (Alice) Internal Key used: Kc
Input: E(Kc, [Kc,TGS ‖ ID_TGS ‖ TS1])
Output: Kc,TGS, ID_TGS, TS1 		Alice's software uses the locally-derived Kc to decrypt Part A from the AS response. If the password was correct, decryption succeeds and Alice extracts: (1) Kc,TGS — the session key she will use to communicate with the TGS, now known to Alice for the first time; (2) ID_TGS — Alice verifies this is the TGS she intended to contact; (3) TS1 — Alice verifies this matches the timestamp she sent in step 1.1. If the password was wrong → Kc is wrong → decryption produces garbage → authentication failure. This step is simultaneously: (a) key delivery — Alice learns Kc,TGS; (b) password verification — correct decryption proves the correct password was entered; (c) AS authentication — only the real AS (holding Kc from the KDC database) could have produced a message that decrypts correctly with Alice's Kc. The TS1 echo-check prevents a replay of an old AS response. Alice now holds Kc,TGS. An eavesdropper who captured the AS response cannot extract Kc,TGS without Kc, which requires Alice's password.
1.9
AS

Client stores credentials; discards Kc		 Client (Alice) Credentials cache (local memory) Stored: Kc,TGS, TGT (opaque blob)
Discarded: Kc 		Alice's Kerberos software stores two items in the local credentials cache (e.g., /tmp/krb5cc_1000 on Linux, LSASS process memory on Windows): (1) Kc,TGS — stored because Alice will need it repeatedly to request service tickets without re-entering her password; (2) TGT — stored exactly as received from AS, untouched and opaque, ready to be forwarded to TGS. Simultaneously, Kc (derived from the password) is erased from memory — it is no longer needed until the TGT expires. Kc is discarded immediately after serving its one purpose (decrypting Part A). The longer a key remains in memory, the larger the window for a memory-dump attack (e.g., Mimikatz on Windows). Kc is the highest-value key — it represents Alice's permanent identity — so minimising its in-memory lifetime is critical. The password-derived key Kc has the shortest possible in-memory lifetime: derive → decrypt → discard. The stored credentials (TGT + Kc,TGS) are sufficient for all service access until TGT expiry without ever needing the password again.

Phase 2 — Service Ticket Request

Client ↔ TGS · TGT presented · Service ticket (Ticket_v) issued
#Phase & ActionActor (Sender)Recipient Object / Key InvolvedWhat happens (procedure) Why this design (rationale)Security guarantee
2.1
TGS

Client constructs Authenticator for TGS		 Client (Alice) Internal (will be sent to TGS) Encrypted with: Kc,TGS
Contents:
ID_client, AD_client, TS2 (fresh timestamp) 		Alice creates Authenticator1 = E(Kc,TGS, [ID_client ‖ AD_client ‖ TS2]). Fields: (1) ID_client = "alice@REALM.COM" — Alice's identity stated inside the Authenticator; (2) AD_client = Alice's current network address — TGS can compare with what the TGT recorded; (3) TS2 = the current timestamp at the exact moment of this TGS request — a freshly read clock value, distinct from TS1. The Authenticator is created moments before sending so TS2 is as current as possible. The Authenticator is a proof of live presence. The TGT alone proves the KDC endorsed Alice, but the TGT was issued hours ago — the TGS cannot know if Alice is still present or if the TGT was stolen. The Authenticator, encrypted with Kc,TGS, proves: (1) the sender knows Kc,TGS right now; (2) the request is happening right now (fresh TS2). A new Authenticator is created for every TGS request — never reused. Replay attacks are prevented: a captured Authenticator with an old TS2 is rejected by TGS. Ticket theft without Kc,TGS is prevented: the thief cannot create a valid Authenticator without the session key.
2.2
TGS

Client sends TGS request		 Client (Alice) TGS ID_v (desired service)
TGT = E(KTGS, [...]) — opaque, forwarded untouched
Authenticator1 = E(Kc,TGS, [ID_client ‖ AD_client ‖ TS2]) 		Alice sends three items to the TGS: (1) ID_v = the identifier of the service Alice wants to access (e.g., "cifs/fileserver.realm.com") — tells the TGS which service ticket to issue; (2) TGT = the opaque blob received in Phase 1, forwarded byte-for-byte without modification — Alice cannot read it but TGS can; (3) Authenticator1 = the freshly constructed Authenticator from step 2.1 — proves Alice is present and knows Kc,TGS. The three-part message covers: what Alice wants (ID_v), KDC's endorsement of Alice (TGT), and proof Alice is genuinely present right now (Authenticator1). Removing any one piece breaks the security model. Alice forwards the TGT without reading it because she cannot — it is encrypted with KTGS, which she does not possess. TGS receives a complete, unforgeable request. Alice cannot fabricate the TGT (doesn't know KTGS) and cannot fake an Authenticator for a different identity (doesn't know Kc,TGS for any other user).
2.3
TGS

TGS decrypts TGT		 TGS Internal Key used: KTGS
Input: TGT = E(KTGS, [...])
Extracts: Kc,TGS, ID_client, AD_client, ID_TGS, TS1, Lifetime 		TGS uses its own long-term secret key KTGS to decrypt the TGT. Upon successful decryption, TGS extracts: (1) Kc,TGS — the session key; TGS now knows the key needed to decrypt Alice's Authenticator in step 2.5; (2) ID_client = "alice@REALM.COM" — the identity the KDC vouches for; (3) AD_client = Alice's IP as recorded by KDC; (4) ID_TGS — confirms this TGT was intended for this TGS; (5) TS1 and Lifetime — used to calculate remaining validity. Only the TGS knows KTGS — this decryption step is computationally impossible for any other party. Successful decryption is itself a proof that this TGT was created by the legitimate KDC/AS. An attacker who forged a TGT would not know KTGS, so their forgery would decrypt to garbage and be rejected. Kc,TGS extracted here is the critical piece for step 2.5. TGS verifies the KDC's endorsement of Alice's identity without any live network call to the KDC — the TGT is self-contained proof. The protocol scales to any number of TGS requests without burdening the KDC.
2.4
TGS

TGS checks TGT validity		 TGS Internal TS1 and Lifetime from decrypted TGT
Compared against TGS's current clock 		TGS performs temporal validation: computes TS1 + Lifetime and checks whether current time falls within the validity window: TS1 ≤ now ≤ TS1 + Lifetime. If now > TS1 + Lifetime → TGT is expired → TGS rejects with KRB_AP_ERR_TKT_EXPIRED error, and Alice must re-enter her password to get a new TGT from the AS. TGS also optionally checks AD_client (the IP from the TGT) matches the actual source IP of the incoming request. The Lifetime field is embedded inside the TGT's encrypted content — Alice cannot extend it because she cannot modify the TGT (it is encrypted with KTGS which she doesn't have). This is deliberate: validity controls are entirely under the KDC's authority, not the client's. The typical 8–10 hour Lifetime corresponds to a workday — long enough for convenience, short enough to limit damage from a stolen TGT. A stolen TGT automatically becomes worthless after Lifetime expires. The KDC controls validity; the client cannot extend it under any circumstances.
2.5
TGS

TGS decrypts Authenticator		 TGS Internal Key used: Kc,TGS (just extracted from TGT in 2.3)
Input: Authenticator1 = E(Kc,TGS, [ID_client ‖ AD_client ‖ TS2])
Extracts: ID_client, AD_client, TS2 		TGS uses Kc,TGS (just extracted from the TGT in step 2.3) to decrypt Authenticator1. Extracts: (1) ID_client = "alice@REALM.COM" — Alice's identity as stated inside the Authenticator; (2) AD_client = Alice's network address as stated in the Authenticator; (3) TS2 = the timestamp Alice embedded moments ago — the freshness proof. If decryption succeeds, it proves the sender knows Kc,TGS. This is the two-layer verification that is the heart of Kerberos. Layer 1 (TGT): "The KDC says this session belongs to alice and the session key is Kc,TGS." Layer 2 (Authenticator): "The sender of this request knows Kc,TGS right now." Both layers together prove: (1) KDC endorsed Alice, AND (2) the person making this request is Alice — not someone who merely stole her TGT. Stolen TGT without Kc,TGS is useless — the thief cannot create Authenticator1. Forged Authenticator without a valid TGT is also rejected. Both pieces must come from the same legitimate session.
2.6
TGS

TGS cross-validates identities		 TGS Internal ID_client from TGT (KDC-endorsed, trusted)
vs.
ID_client from Authenticator1 (claimed by sender) 		TGS compares the ID_client extracted from the TGT (step 2.3, trusted — the KDC put it there) with the ID_client extracted from Authenticator1 (step 2.5, claimed by the sender). They must match exactly: ID_client(TGT) == ID_client(Authenticator1). If they differ → TGS rejects immediately. Example: Mallory steals Alice's TGT but not Kc,TGS. She uses her own session key to make an Authenticator saying "mallory@REALM.COM". The TGT says "alice" — mismatch → rejected. Without this cross-check, an attacker could combine Alice's TGT (for KDC endorsement) with their own Authenticator (for key possession proof) — effectively splicing two pieces from different sessions into one seemingly valid request. The cross-check requires both pieces to name the same identity, closing this combination attack. A TGT stolen from one user cannot be combined with a different user's Authenticator. Both pieces must be from the same legitimate session.
2.7
TGS

TGS checks timestamp and replay cache		 TGS Internal TS2 from Authenticator1
TGS's current clock
TGS's replay cache (recent Authenticators seen) 		TGS performs two checks: (1) Freshness check: verifies |now − TS2| ≤ 5 minutes. If TS2 is older than 5 minutes → reject with KRB_AP_ERR_SKEW. (2) Replay check: queries the replay cache — a record of all (ID_client, TS2) pairs seen within the last 5-minute window. If this exact pair already exists → reject with KRB_AP_ERR_REPEAT. If not in cache → add it → accept. This is why Kerberos environments require NTP clock synchronisation across all machines. Without the freshness check: an attacker who captures Authenticator1 from the network could replay it hours or days later. Without the replay cache: even within the 5-minute window, an attacker could send the same Authenticator1 multiple times simultaneously. The combination gives a precise guarantee: each Authenticator is accepted exactly once. Each Authenticator is accepted at most once. Replay attacks are bounded to the 5-minute window, and even within that window, duplicates are rejected by the replay cache.
2.8
TGS

TGS generates new service session key		 TGS Internal (not yet sent) Kc,v
(brand-new random key — fresh for this specific service session)		 TGS calls a CSPRNG to produce a new Kc,v — the session key that Alice and Service V will use for their communication. This key has never existed before. It is entirely independent of Kc,TGS (which was for Alice↔TGS communication) and of any other service session key Alice may have (e.g., her concurrent session with the mail server uses a completely different Kc,v). A fresh Kc,v per service per session means Alice's communication with the file server and her communication with the mail server each use completely separate keys. This is the principle of key isolation — each security relationship gets its own key, minimising the blast radius of any individual key compromise. Session key compromise is scoped to exactly one user↔service relationship. Compromise of Kc,v for the file server has no impact on the mail server session or any other session.
2.9
TGS

TGS constructs Service Ticket (Ticket_v)		 TGS Internal (will become Part B of TGS response) Encrypted with: Kv
Contents: Kc,v, ID_client, AD_client, ID_v, TS2, Lifetime_v 		TGS assembles the plaintext body of Ticket_v with: (1) Kc,v — the service session key just generated; Server V needs this to decrypt Alice's Authenticators; (2) ID_client = "alice@REALM.COM" — Server V needs to know whose request this represents; (3) AD_client = Alice's IP; (4) ID_v = "cifs/fileserver" — Server V checks this matches its own service name; (5) TS2 — issuance time; (6) Lifetime_v — validity (often shorter than TGT lifetime, e.g., 1–2 hours). TGS encrypts the whole structure with Kv:
Ticket_v = E(Kv, [Kc,v ‖ ID_client ‖ AD_client ‖ ID_v ‖ TS2 ‖ Lifetime_v])		 Encrypting with Kv (known only to TGS/KDC and Server V) means: (1) Alice carries Ticket_v but cannot read Kc,v inside it; (2) Alice cannot modify Ticket_v; (3) Server V trusts the ticket contents because only TGS could have produced valid Kv-encryption. ID_v inside the encrypted content prevents ticket misdirection — V checks that ID_v matches its own identity. Ticket_v is opaque to Alice (cannot read Kc,v inside) and tamper-proof (Kv-encryption detects modification). A stolen Ticket_v without Kc,v cannot be used — the holder cannot create valid Authenticators for Server V.
2.10
TGS

TGS encrypts Kc,v for client		 TGS Internal (will become Part A of TGS response) Encrypted with: Kc,TGS
Contents: Kc,v, ID_v, TS2 		TGS creates Part A of its response: Part A = E(Kc,TGS, [Kc,v ‖ ID_v ‖ TS2]). Fields: (1) Kc,v — the service session key Alice needs to communicate with Server V — the main payload of Part A; (2) ID_v = "cifs/fileserver" — tells Alice which service this key is for, binding Kc,v to a specific service; (3) TS2 — allows Alice to verify this response corresponds to her current request, not a replayed old TGS response. Kc,TGS is the established secure channel between Alice and TGS — created specifically to protect exactly this kind of key delivery. Encrypting with Kc,TGS ensures only Alice can extract Kc,v. The ID_v field binds Kc,v to a specific service — Alice cannot accidentally or maliciously use Kc,v for a different service. Kc,v is delivered without ever appearing in plaintext on the network. A network attacker cannot extract Kc,v from Part A without knowing Kc,TGS.
2.11
TGS

TGS sends response to client		 TGS Client (Alice) Part A: E(Kc,TGS, [Kc,v ‖ ID_v ‖ TS2])

Part B (Ticket_v): E(Kv, [Kc,v ‖ ID_client ‖ AD_client ‖ ID_v ‖ TS2 ‖ Lifetime_v]) 		TGS bundles Part A and Part B (Ticket_v) into a single response to Alice. Part A is for Alice (encrypted with Kc,TGS — only Alice can open it). Part B is for Server V (encrypted with Kv — only Server V can open it). Alice receives both but can only read Part A. The structure mirrors exactly what the AS sent in Phase 1 — one part for the immediate recipient, one sealed part for forwarding to the next party. The same structural pattern from Phase 1 repeats here: one part for Alice, one opaque part for forwarding. This consistency means Alice's Kerberos client software handles Phase 1 and Phase 2 responses with identical logic — same code path. Alice now has everything needed for Phase 3. Alice now has both Kc,v (from Part A) and Ticket_v (from Part B) — all that is needed to contact Server V.
2.12
TGS

Client decrypts Part A from TGS		 Client (Alice) Internal Key used: Kc,TGS
Input: E(Kc,TGS, [Kc,v ‖ ID_v ‖ TS2])
Extracts: Kc,v, ID_v, TS2 		Alice uses Kc,TGS (stored in the credentials cache since Phase 1) to decrypt Part A. Extracts: (1) Kc,v — the service session key; Alice now knows this key for the first time and stores it in the credentials cache alongside Ticket_v as a matched pair; (2) ID_v — Alice verifies this is the service she requested; (3) TS2 — Alice verifies freshness of the TGS response. Alice stores Kc,v and Ticket_v together in the credentials cache. Alice now knows Kc,v but does not know the contents of Ticket_v (which also contains Kc,v, but encrypted with Kv which Alice doesn't have). Both Alice and Server V will independently arrive at Kc,v: Alice from this decryption, Server V from opening Ticket_v. This is how a shared secret is established between two parties who have never directly communicated. Alice is ready to contact Server V with a TGS-endorsed proof of identity (Ticket_v) and a shared secret (Kc,v) — without any direct exchange between Alice and Server V.

Phase 3 — Service Access

Client ↔ Server V · Ticket presented · Mutual authentication · Secure session
#Phase & ActionActor (Sender)Recipient Object / Key InvolvedWhat happens (procedure) Why this design (rationale)Security guarantee
3.1
Service

Client constructs Authenticator for Server V		 Client (Alice) Internal (will be sent to Server V) Encrypted with: Kc,v
Contents: ID_client, AD_client, TS3 (new timestamp — distinct from TS1 and TS2) 		Alice creates Authenticator2 = E(Kc,v, [ID_client ‖ AD_client ‖ TS3]). Fields: (1) ID_client = "alice@REALM.COM" — Alice's identity inside this Authenticator; (2) AD_client = Alice's current network address; (3) TS3 = current timestamp at the exact moment of this service request — a third fresh timestamp reading, distinct from TS1 and TS2. This Authenticator uses Kc,v, not Kc,TGS, because the intended recipient is Server V. A new Authenticator using the service-specific Kc,v means: the TGS-phase Authenticator1 (encrypted with Kc,TGS) cannot be replayed against Server V — wrong key. Each phase uses its own session key, so Authenticators from different phases are cryptographically incompatible. Replay of Phase 2 Authenticator1 against Server V is impossible — it was encrypted with Kc,TGS, not Kc,v. Replay of Authenticator2 itself is rejected by Server V's timestamp check.
3.2
Service

Client sends service access request		 Client (Alice) Server V Ticket_v = E(Kv, [...]) — opaque, forwarded untouched
Authenticator2 = E(Kc,v, [ID_client ‖ AD_client ‖ TS3]) 		Alice sends two items to Server V: (1) Ticket_v = the opaque blob forwarded exactly as received from TGS in step 2.11 — Alice cannot read it, but Server V can decrypt it with Kv to learn Kc,v and Alice's identity; (2) Authenticator2 = the freshly created Authenticator using Kc,v — proves Alice knows Kc,v right now. Note: unlike the TGS request (3 items), here Alice sends only 2 — the ID_v inside the encrypted Ticket_v is sufficient for Server V to verify the ticket was intended for it. Server V needs exactly these two pieces: (1) the TGS-endorsed proof of Alice's identity (Ticket_v); (2) proof that Alice is present right now and knows the session key (Authenticator2). The design mirrors what Alice sent to the TGS — consistent two-piece authentication pattern throughout the entire protocol. Server V receives an unforgeable, TGS-endorsed identity claim paired with a live-presence proof. Alice cannot fabricate Ticket_v (doesn't know Kv) and cannot fake Authenticator2 for a different identity (doesn't know Kc,v for any other user).
3.3
Service

Server V decrypts Ticket_v		 Server V Internal Key used: Kv
Input: Ticket_v = E(Kv, [...])
Extracts: Kc,v, ID_client, AD_client, ID_v, TS2, Lifetime_v 		Server V uses its own long-term secret key Kv (installed by the sysadmin during Phase 0.2) to decrypt Ticket_v. Extracts: (1) Kc,v — the service session key; Server V now knows this key for the first time and can use it to decrypt Authenticator2; (2) ID_client = "alice@REALM.COM" — the identity the TGS vouches for; (3) AD_client = Alice's IP; (4) ID_v = "cifs/fileserver" — Server V checks this matches its own name; (5) TS2 and Lifetime_v — used for validity checking in step 3.4. Only Server V knows Kv — this decryption step is the moment Server V independently verifies the TGS's endorsement without contacting the TGS or KDC. This is fully offline verification — no network call to any central authority. The ID_v inside the ticket prevents a ticket issued for the file server from being used at the mail server. Server V verifies KDC's endorsement of Alice without any live connection to KDC or TGS. The protocol scales to thousands of service accesses per second without creating a bottleneck at the KDC.
3.4
Service

Server V checks Ticket_v validity		 Server V Internal TS2 and Lifetime_v from decrypted Ticket_v
AD_client from Ticket_v vs. actual source IP 		Server V performs: (1) Expiry check: verifies TS2 ≤ now ≤ TS2 + Lifetime_v. If expired → reject; Alice must go back to TGS for a new Ticket_v (no password re-entry needed as long as TGT is valid); (2) Address check (optional): compares AD_client from Ticket_v (recorded by TGS at ticket issuance) with the actual source IP of the incoming TCP connection. If they differ → reject. In modern deployments this check is often disabled due to NAT and VPN complications. Lifetime_v in the service ticket is deliberately shorter than the TGT lifetime. A stolen Ticket_v for the file server will expire within hours, limiting the attacker's window. The address check provides an extra binding — even if both Ticket_v and Kc,v are stolen, they can only be used from Alice's specific IP address. Time-bounded validity limits the usefulness of a stolen service ticket. Optional IP binding adds a second factor tying the ticket to a specific machine.
3.5
Service

Server V decrypts Authenticator2		 Server V Internal Key used: Kc,v (just extracted from Ticket_v in 3.3)
Input: Authenticator2 = E(Kc,v, [ID_client ‖ AD_client ‖ TS3])
Extracts: ID_client, AD_client, TS3 		Server V uses Kc,v (just extracted from Ticket_v in step 3.3) to decrypt Authenticator2. Extracts: (1) ID_client = "alice@REALM.COM" — Alice's stated identity; (2) AD_client = Alice's address; (3) TS3 = the timestamp Alice embedded moments ago — the freshness proof for this service request. Successful decryption proves the sender knows Kc,v. Same two-layer verification as Phase 2 step 2.5, now applied by Server V: Layer 1 (Ticket_v): "TGS says this session belongs to alice and Kc,v is the session key." Layer 2 (Authenticator2): "The sender of this request knows Kc,v right now." The identical two-layer pattern appears at every trust boundary in the protocol — it is the structural core of Kerberos. Stolen Ticket_v without Kc,v cannot produce a valid Authenticator2. The two-layer check is enforced identically at the TGS and at every service server.
3.6
Service

Server V cross-validates and checks timestamp		 Server V Internal ID_client: Ticket_v vs Authenticator2
TS3: freshness check
Server V's own replay cache 		Server V performs three checks: (1) Identity cross-check: ID_client(Ticket_v) == ID_client(Authenticator2) — same ticket-theft-combination prevention as TGS step 2.6; (2) Freshness check: |now − TS3| ≤ 5 minutes; (3) Replay cache check: Server V maintains its own independent replay cache — checks that (alice@REALM.COM, TS3) has not been seen before within the 5-minute window. All three checks must pass. Each service runs all these checks independently — there is no central coordinator at access time. This is by design: the protocol is decentralised at the access point. Server V's replay cache is separate from TGS's cache because Authenticator2 has never been presented to TGS — it is a new Authenticator for a new relationship. Full authentication without any live connection to KDC or TGS. Each service independently enforces anti-replay protection using its own replay cache.
3.7
Service

Server V grants access (authorisation)		 Server V Alice's session Access control policy applied to ID_client = "alice@REALM.COM" Server V has completed all authentication checks. It now performs authorisation: looks up what alice@REALM.COM is permitted to do (read files? write files? admin access?). This is typically an ACL (Access Control List) or a directory lookup (LDAP/Active Directory query). If Alice is authorised → access granted. If not → reject with an authorisation error — distinct from the earlier authentication errors, which indicate identity failure. Authentication (the entire Phase 1–3 protocol) answers: "Is this really Alice?" Authorisation answers: "Is Alice allowed to do this?" These are deliberately separated in Kerberos. Kerberos handles authentication only — each service implements its own authorisation policy. This allows different services to have different permission models while sharing the same authentication infrastructure. Kerberos guarantees the identity; the service enforces its own permissions. Neither step is skipped. Authenticating as alice@REALM.COM does not automatically grant access — permissions are checked separately.
3.8
Service

Server V sends mutual auth response (optional)		 Server V Client (Alice) Encrypted with: Kc,v
Content: TS3 + 1 		Server V takes TS3 from Authenticator2, increments it by exactly 1, and sends: E(Kc,v, [TS3 + 1]). To produce this message, Server V had to: (1) decrypt Ticket_v with Kv → required knowing Kv; (2) extract Kc,v from the ticket; (3) decrypt Authenticator2 → extracted TS3; (4) encrypt TS3+1 with Kc,v. Only the real Server V (knowing Kv) can execute all four steps. An impostor server cannot produce this response — it doesn't know Kv, cannot decrypt Ticket_v, cannot extract Kc,v, and therefore cannot produce the encrypted TS3+1. The +1 convention (not TS3 itself, not TS3+2) proves the server processed Alice's specific Authenticator and is not replaying a captured old mutual-auth message from a previous session. Alice is now assured she is communicating with the genuine Server V, not an impostor or man-in-the-middle. This step completes the loop: Server V authenticated Alice (3.3–3.6); now Alice authenticates Server V (3.8–3.9).
3.9
Service

Client verifies mutual authentication		 Client (Alice) Internal Key used: Kc,v
Input: E(Kc,v, [TS3 + 1])
Verifies: decrypted value == TS3 + 1 		Alice decrypts Server V's mutual-auth response using Kc,v. Verifies three things: (1) Decryption succeeds — confirms Server V knows Kc,v, which required knowing Kv to extract from Ticket_v; (2) Decrypted value == TS3 + 1 — TS3 was the timestamp Alice put in Authenticator2, so TS3+1 proves Server V received and processed her specific Authenticator; (3) TS3+1 is appropriately recent — not a replay of a mutual-auth response from a previous session. All three checks must pass for Alice to trust Server V. Without mutual authentication, Alice could be deceived by a rogue server that pretended to accept her Ticket_v (couldn't actually decrypt it, but could pretend) and sent fake data. The mutual auth response, requiring knowledge of Kv to produce, is the cryptographic proof that this server is the genuine, KDC-registered Server V. Full mutual authentication achieved: Server V has verified Alice (steps 3.3–3.6); Alice has verified Server V (steps 3.8–3.9). Both parties have authenticated each other using only symmetric cryptography and the KDC as trusted third party.
3.10
Service

Secure session established		 Alice ↔ Server V Both Kc,v
(shared session key — never transmitted in plaintext)		 All subsequent application messages between Alice and Server V are encrypted and integrity-protected using Kc,v: E(Kc,v, [request]) from Alice; E(Kc,v, [response]) from Server V. Kc,v was never transmitted in plaintext: Alice learned it from TGS (step 2.12 via Kc,TGS encryption); Server V learned it from Ticket_v (step 3.3 via Kv encryption). Both arrived at the same key through independent, secure paths. The KDC is completely uninvolved from this point onward. Kc,v expires with Ticket_v. When it expires, Alice requests a new Ticket_v from TGS (Phase 2 repeats, no password re-entry). When TGT expires (typically after 8–10 hours), Alice must re-enter her password (Phase 1 repeats). This two-tier expiry is the key lifecycle design: short service key lifetime for security, longer TGT lifetime for usability. End-to-end confidentiality and integrity for all application communication. No further KDC involvement until Ticket_v expires. The KDC is entirely out of the critical path for all data communication.

4.1
Expiry

Ticket_v expires		 Server V (rejects) Client (Alice) Lifetime_v field in Ticket_v
Checked against current time		 Server V receives a request from Alice with a Ticket_v whose TS2 + Lifetime_v < now. Server V rejects with KRB_AP_ERR_TKT_EXPIRED. Alice's Kerberos client detects this error and automatically returns to Phase 2 (TGS exchange) — presents her still-valid TGT to get a fresh Ticket_v. Alice does not need to re-enter her password as long as her TGT is still valid. The ticket refresh is transparent to Alice and to the application she is using. Short service ticket lifetimes minimise the impact of a stolen Ticket_v. The automatic renewal process is transparent to Alice. Separating TGT lifetime (8–10 hours) from service ticket lifetime (1–2 hours) gives a two-tier system: service access refreshes quietly, full re-authentication happens once per workday. A stolen Ticket_v has a maximum usable lifetime of Lifetime_v — typically 1–2 hours — before it expires automatically with no manual revocation needed.
4.2
Expiry

TGT expires		 TGS (rejects) Client (Alice — must re-login) Lifetime field in TGT
Checked against current time		 Alice's TGT has expired (TS1 + Lifetime < now). When Alice next tries to access any service, the TGS rejects the TGT with KRB_AP_ERR_TKT_EXPIRED. Alice must return to Phase 1 — re-enter her password to get a new TGT. The old TGT and all cached Ticket_v's and session keys are discarded from the credentials cache. Kerberos environments typically set TGT Lifetime to 8–10 hours, aligning with a standard workday. TGT expiry forces periodic re-authentication. If Alice walked away from her desk and someone else sat down, they cannot continue her Kerberos session past the TGT lifetime — all service access requires going through Phase 1 again, which requires typing Alice's password. Expiry is automatic — it requires no manual revocation action from an administrator. A stolen TGT has a maximum lifetime of Lifetime — typically 8–10 hours. After that the attacker is locked out automatically. No revocation infrastructure (CRL, OCSP) is needed for this guarantee.
4.3
Threat

Pass-the-Ticket attack		 Attacker Server V (impersonating Alice) Stolen from LSASS memory:
TGT + Kc,TGS
or
Ticket_v + Kc,v 		An attacker compromises Alice's machine and uses a memory-dump tool (e.g., Mimikatz on Windows) to dump the LSASS process memory. Extracts from the credentials cache: TGT (opaque blob) + Kc,TGS (session key stored alongside it). With these two items, the attacker has everything needed to execute Phase 2 from any machine — they can request service tickets and access all services Alice could access, until the TGT expires. Alice's password is never recovered — only the temporary tickets are stolen. This attack succeeds because the TGT + Kc,TGS pair is equivalent to a temporary password. Kerberos itself is not broken — the attacker exploited OS-level credential storage. Countermeasures: (1) Credential Guard (Windows) — stores credentials in a hardware-isolated VM that Mimikatz cannot access; (2) Protected Users group — TGTs not cached in LSASS for high-privilege accounts; (3) Smart card / PKINIT — TGT cannot be used without the physical card; (4) Short TGT lifetimes — reduce the attacker's window. The Kerberos protocol design is sound — this attack exploits OS-level credential storage, not the cryptographic protocol. The primary defence must protect the host machine, not the protocol itself. Short ticket lifetimes and Credential Guard are the two most impactful mitigations.
