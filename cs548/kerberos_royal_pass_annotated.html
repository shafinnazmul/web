Phase 0 — Pre-Setup: Before Alice Arrives
0.1
KDC Registers Alice's Account
Actor: KDC Admin → KDC Database
Pre-Setup
Palace Story
Palace Archivist (KDC Admin):
"Alice of House Pemberton has joined the palace guild. I have taken the impression of her personal seal and recorded it in the Grand Ledger. She need not carry her seal everywhere — only its impression is kept here."
The Archivist does NOT store Alice's actual seal (her plaintext password). He stores only the wax impression — a derived copy — so that even if the Ledger is stolen, no one can impersonate Alice without her real seal.
Real Kerberos Protocol
Admin action (offline, not a network message): Kc = hash(Alice's password) Store Kc in KDC database under IDc
Kc ≡ Client's long-term secret key, derived from password via one-way hash = Alice's seal impression in the Ledger
IDc ≡ "alice@REALM.COM" — the database lookup key = Alice's name on the Ledger page
Why hash, not plaintext? If an attacker steals the KDC database, they get only Kc — a hash — not Alice's reusable password. Alice can re-derive Kc locally at any time: hash(password). One-way function: trivial to compute forward, computationally infeasible to reverse.
Mappings: Archivist → KDC Admin Grand Ledger → KDC Database Seal impression → Kc (hashed password key) Alice's actual seal → Alice's plaintext password (never stored)
0.2
KDC Registers Service V (Treasury)
Actor: KDC Admin → KDC Database + Service Server V
Pre-Setup
Palace Story
Palace Archivist:
"The Treasury has its own master key. I have made two copies: one stays in the Grand Ledger, one is given directly to the Treasury Head Guard. No one else receives a copy — not Alice, not the Permit Office."
Each room in the palace has a completely independent master key. Stealing the Treasury key does not open the Library. The key is handed to the Guard in person — never sent through the postal system.
Real Kerberos Protocol
Admin action (out-of-band, offline): Kv = random( ) Store Kv in KDC database under IDv Install Kv on Server V directly (sysadmin)
Kv ≡ Service server's long-term key — purely random (not password-derived) = Treasury's master key — a unique random physical key
IDv ≡ "cifs/fileserver.realm.com" — the service's identity string = "The Treasury Room" — the room's official name in the Ledger
Why one key per service? Principle of key isolation. If the file server is compromised and Kv is stolen, the mail server's Kv is entirely unaffected — each service has its own independent key. Out-of-band installation avoids exposing Kv on the network at any point.
Mappings: Treasury Head Guard → Service Server V Treasury master key → Kv Handing key in person → Out-of-band sysadmin installation Each room has its own key → One Kv per service, independent
Phase 1 — Initial Authentication: Client ↔ Authentication Server
1.1
Client Sends Login Request to AS
C → AS  |  Plaintext message, no credentials sent
AS Exchange
Palace Story
Alice (at Palace Front Desk):
"Good morning. I am Alice of House Pemberton. I wish to access palace services today. I am presenting myself now, at this precise moment."
Alice does not hand over her personal seal. She only states her name, what she wants (a pass to the Permit Office), and the time of her arrival. Her seal never leaves her possession at this step.
Real Kerberos Protocol
C → AS: IDc ‖ IDtgs ‖ TS1 All fields sent in PLAINTEXT — no encryption.
IDc ≡ "alice@REALM.COM" — tells AS which account is requesting = "I am Alice of House Pemberton"
IDtgs ≡ "krbtgt/REALM.COM" — tells AS the scope: I want a TGT, not a service ticket = "I want a Permit Office pass" (not a specific room pass)
TS1 ≡ Current timestamp — AS echoes this in reply so Alice can verify freshness = "The time of my arrival: 12:00 noon"
Critical Design Decision: No password, no key, no credential of any kind is transmitted here. The password never appears on the network in any form — not even encrypted. Verification happens implicitly in Step 1.4: only the real Alice can decrypt the reply.
Mappings: Alice stating her name → IDc in plaintext Requesting a Permit Office pass → IDtgs field Time of arrival → TS1 timestamp Seal NOT handed over → password/Kc NOT transmitted
1.2
AS Generates Session Key & Constructs TGT
AS internal — lookup Kc, generate Kc,tgs, seal TGT with Ktgs
AS Exchange
Palace Story
Front Desk Clerk (AS) — internal process:
"Let me check the Ledger for 'Alice of House Pemberton'... found. Now I will create a one-time code word for Alice and the Permit Office to share: 'BlueTiger'. I will seal a letter for the Permit Office containing this code word, Alice's name, and the pass expiration. Only the Permit Office can break this seal."
The clerk creates two separate packages: Package A — a letter for Alice, sealed with Alice's own impression, containing the code word "BlueTiger". Package B — the permit-pass, sealed with the Permit Office's master seal, also containing "BlueTiger" inside (so the Permit Office learns it). Alice cannot open Package B.
Real Kerberos Protocol
AS internal steps: 1. Lookup: retrieve Kc from database (IDc is the key) 2. Generate: Kc,tgs = CSPRNG( ) ← brand new, never reused 3. Build TGT (Package B — for TGS): Ticket_tgs = E(Ktgs, Kc,tgs ‖ IDc ‖ ADc ‖ IDtgs ‖ TS1 ‖ Lifetime) 4. Build Part A (Package A — for Alice): Part_A = E(Kc, Kc,tgs ‖ IDtgs ‖ TS1)
Kc,tgs ≡ Fresh random session key. Never existed before. Will expire with TGT. = "BlueTiger" — the one-time code word for this palace visit
Ticket_tgs ≡ TGT sealed with Ktgs. Alice carries this but cannot read it. = Sealed permit-pass envelope — Alice is a courier, not the recipient
Lifetime ≡ Ticket validity period — typically 8–10 hours (one workday). Inside encrypted content so Alice cannot extend it. = "Valid until sunset today" — written inside the sealed envelope
ADc ≡ Alice's IP address — optionally binds the ticket to her machine = Alice's house address — binds the pass to her specific residence
Why put Kc,tgs in BOTH packages? Alice needs it to prove herself to TGS (Part A → Alice decrypts and learns it). TGS needs it to verify Alice's Authenticator (TGT → TGS decrypts and learns it). Both parties arrive at the same session key through independent, secure channels — without ever transmitting it in plaintext.
Mappings: "BlueTiger" code word → Kc,tgs Sealed permit-pass → Ticket_tgs (sealed with Ktgs) Permit Office's wax seal → encryption with Ktgs Alice's sealed package A → E(Kc, [Kc,tgs ‖ ...]) Alice cannot open the permit-pass → she doesn't have Ktgs
1.3
AS Sends Reply → Alice Decrypts & Discards Kc
AS → C  |  Client decrypts Part A, stores Kc,tgs + TGT, discards Kc
AS Exchange
Palace Story
Front Desk Clerk → Alice:
"Here are two items: your personal envelope (sealed with your impression — only you can open it), and the sealed permit-pass to deliver to the Permit Office."
Alice (privately):
"I open my personal envelope with my seal. Inside: the code word is 'BlueTiger', and here is the sealed permit-pass. I memorise 'BlueTiger' and put the permit-pass in my satchel. I return my physical seal to its lockbox — I won't need it again today unless my pass expires."
Real Kerberos Protocol
AS → C: Part_A = E(Kc, [Kc,tgs ‖ IDtgs ‖ TS1]) Ticket_tgs = E(Ktgs, [...]) ← opaque to Alice Alice's client software: 1. Derive Kc = hash(typed_password) 2. Decrypt Part_A with Kc → extract Kc,tgs, IDtgs, TS1 3. Verify: IDtgs == expected TGS name TS1 == timestamp sent in Step 1.1 4. Store in credentials cache: Kc,tgs ← extracted session key Ticket_tgs ← opaque blob, forwarded later 5. DISCARD Kc from memory immediately
Decryption ≡ If Alice typed the wrong password → wrong Kc → Part_A decrypts to garbage → authentication fails silently = Wrong seal impression → envelope won't open cleanly
TS1 check ≡ Verifies this reply is for Alice's current request, not a replayed old AS response = "The clerk echoed my 12:00 arrival time — this is a fresh reply"
Discard Kc ≡ Kc represents Alice's permanent identity. In-memory time is minimized: derive → decrypt → discard. Survives seconds, not hours. = "Return the physical seal to the lockbox" — don't carry it around all day
Implicit password verification: The AS never explicitly says "right password" or "wrong password." The test is: can Alice decrypt Part_A and extract a coherent Kc,tgs? Only the real Alice (correct password → correct Kc) can pass this test.
Mappings: Opening personal envelope → Decrypt Part_A with Kc Memorising "BlueTiger" → Storing Kc,tgs in credentials cache Carrying sealed permit-pass in satchel → Storing opaque TGT blob Returning seal to lockbox → Erasing Kc from memory
Phase 2 — Service Ticket Request: Client ↔ TGS
2.1
Client Sends TGS Request (Ticket + Authenticator)
C → TGS  |  IDv + Ticket_tgs (opaque) + Authenticator_c
TGS Exchange
Palace Story
Alice (at Permit Office):
"I wish to access the Treasury Room. Here is my sealed permit-pass from the Front Desk. And here is my signed declaration: 'I am Alice. The time is precisely 12:05 PM.' — I wrote and sealed it just now with our shared code word 'BlueTiger'."
Alice presents three things: (1) what she wants — Treasury access, (2) the sealed pass she cannot open — the TGT, and (3) a fresh live declaration sealed with "BlueTiger" to prove she is Alice and that this is happening right now.
Real Kerberos Protocol
C → TGS: IDv ← service wanted (in plaintext) Ticket_tgs ← opaque blob, forwarded unchanged Authenticator_c = E(Kc,tgs, IDc ‖ ADc ‖ TS2) ← freshly created right now
IDv ≡ The target service identifier — tells TGS which Ticket_v to issue = "I want Treasury Room access" — stated openly
Ticket_tgs ≡ Forwarded byte-for-byte unchanged. Alice is a courier delivering a sealed envelope. = Sealed permit-pass — Alice delivers without reading
Auth_c ≡ E(Kc,tgs, IDc ‖ ADc ‖ TS2) — fresh per request. TS2 ≠ TS1. New timestamp reading. = "I am Alice, time is 12:05 PM" — written and sealed just now, not reused
Why is the Authenticator necessary? The TGT alone proves the KDC endorsed Alice — but it was issued hours ago. TGT theft is possible. The Authenticator proves Alice is physically present right now and knows Kc,tgs. A thief who stole only the TGT (without Kc,tgs) cannot forge this Authenticator.
Mappings: "I want Treasury Room" → IDv Sealed permit-pass (delivered unopened) → Ticket_tgs (opaque) Declaration "I am Alice, time 12:05" → Authenticator_c = E(Kc,tgs, IDc ‖ TS2) "BlueTiger" sealing the declaration → encrypting Authenticator with Kc,tgs
2.2
TGS Verifies TGT, Authenticator & Issues Ticket_v
TGS internal: decrypt TGT → extract Kc,tgs → verify Authenticator → cross-check → issue Ticket_v
TGS Exchange
Palace Story
Permit Office Clerk (TGS):
"I break the Front Desk's seal on this permit-pass with my master key. Inside I find: code word 'BlueTiger', Alice's name, and expiration: sunset. The pass is not expired. Now I open her declaration using 'BlueTiger' — it reads 'Alice, 12:05 PM'. The time is now 12:06 PM — fresh enough. The name on the pass and the name on the declaration both say Alice. She is verified."
"Now I create a new code word 'GoldenKey' for Alice and the Treasury. I seal a Treasury pass with the Treasury's master key — only the Treasury guard can open it. I also give Alice a separate envelope containing 'GoldenKey', sealed with 'BlueTiger' so only she can open it."
Real Kerberos Protocol
TGS internal steps: 1. Decrypt TGT with Ktgs: → extract: Kc,tgs, IDc, ADc, IDtgs, TS1, Lifetime → check: TS1 + Lifetime > now (not expired) 2. Decrypt Authenticator_c with Kc,tgs: → extract: IDc_auth, ADc_auth, TS2 → check A (freshness): |now − TS2| ≤ 5 min → check B (replay cache): (IDc, TS2) not seen before → add to cache 3. Cross-validate: IDc(TGT) == IDc(Authenticator) ← CRITICAL 4. Generate: Kc,v = CSPRNG( ) ← new session key 5. Issue Ticket_v (sealed for Server V): Ticket_v = E(Kv, Kc,v ‖ IDc ‖ ADc ‖ IDv ‖ TS2 ‖ Lifetime_v) 6. Reply to Alice: Part_A = E(Kc,tgs, [Kc,v ‖ IDv ‖ TS2]) Ticket_v (opaque)
Cross-validate Step 3 ≡ Prevents combining a stolen TGT from Alice with a Mallory-owned Authenticator. Both pieces must name the same identity. = "The pass says Alice; the declaration says Alice — both match."
Replay cache ≡ TGS records every (IDc, TS2) seen in the last 5 minutes. Duplicate = replay attack. = "Has this exact declaration been presented to me already today?"
Kc,v ≡ Fresh random key for Alice↔Treasury session. Independent of Kc,tgs. = "GoldenKey" — a new code word for Alice's Treasury visit
Ticket_v ≡ Sealed with Kv. Alice carries it opaquely to Server V. Cannot be used at a different service. = Treasury pass sealed with Treasury's master key — only the Treasury guard can open it
Mappings: Breaking seal with master key → Decrypt TGT with Ktgs Opening declaration with "BlueTiger" → Decrypt Auth_c with Kc,tgs Both items name "Alice" → Cross-validate IDc(TGT) == IDc(Auth) "GoldenKey" → Kc,v — new session key for Alice↔Treasury Treasury pass sealed with Treasury key → Ticket_v = E(Kv, [...])
Phase 3 — Service Access: Client ↔ Server V (Mutual Authentication)
3.1
Client Sends Service Request (Ticket_v + New Authenticator)
C → V  |  Ticket_v (opaque) + Authenticator_c' with Kc,v
Service Access
Palace Story
Alice (at Treasury door):
"Treasury Guard, here is my sealed Treasury pass from the Permit Office. And here is my fresh declaration: 'I am Alice. The time is precisely 12:10 PM.' — sealed now with our shared code word 'GoldenKey'."
This is a different sealed declaration than before. The old "BlueTiger" declaration cannot be reused here — it was sealed with "BlueTiger", but this door requires "GoldenKey." Alice must create a new declaration sealed specifically with "GoldenKey".
Real Kerberos Protocol
C → V: Ticket_v ← opaque, forwarded unchanged Authenticator_c' = E(Kc,v, IDc ‖ ADc ‖ TS3) ← new timestamp TS3, sealed with Kc,v
Ticket_v ≡ The sealed service ticket from TGS. Server V decrypts it with Kv to learn Kc,v and IDc. = Treasury pass — Alice hands it over; the Guard opens it with the Treasury key
Auth_c' ≡ New Authenticator using Kc,v (not Kc,tgs). TS3 is a third fresh timestamp. = "I am Alice, time 12:10" — sealed with "GoldenKey" (not "BlueTiger")
Why is Auth_c' (Phase 3) cryptographically incompatible with Auth_c (Phase 2)? Phase 2 Authenticator was sealed with Kc,tgs. Phase 3 Authenticator is sealed with Kc,v. Server V uses Kc,v to verify — it cannot be fooled by replaying Auth_c from Phase 2. Each phase uses its own dedicated session key, making inter-phase replay impossible.
Mappings: Handing Treasury pass to guard → Forwarding Ticket_v "GoldenKey" sealing new declaration → Auth_c' = E(Kc,v, IDc ‖ TS3) TS3 ≠ TS2 ≠ TS1 → Each Authenticator uses a brand-new timestamp reading
3.2
Server V Verifies Ticket_v & Authenticator → Grants Access
V internal: decrypt Ticket_v → verify Auth_c' → cross-check → grant access
Service Access
Palace Story
Treasury Head Guard (Server V):
"I open the Treasury pass with my master key. Inside: code word 'GoldenKey', Alice's name, valid until 2 PM today. Not expired. Now I open her declaration using 'GoldenKey' — it reads 'Alice, 12:10 PM'. The time is 12:11 — fresh. Both say Alice. She is who she claims."
"Alice is permitted entry. But wait — has Alice seen a real Treasury Guard? Or an impostor? Let me prove I am genuine."
Real Kerberos Protocol
V internal steps: 1. Decrypt Ticket_v with Kv: → extract: Kc,v, IDc, ADc, IDv, TS2, Lifetime_v → check: IDv == own service name (prevents ticket misdirection) → check: TS2 + Lifetime_v > now (not expired) → check: ADc == actual source IP (if enabled) 2. Decrypt Authenticator_c' with Kc,v: → extract: IDc_auth, ADc_auth, TS3 → freshness: |now − TS3| ≤ 5 min → replay cache: (IDc, TS3) not seen before 3. Cross-validate: IDc(Ticket_v) == IDc(Auth_c') 4. Grant access — Alice is authenticated. Perform authorization check (ACL lookup). 5. [Optional mutual auth reply]: V → C: E(Kc,v, [TS3 + 1])
IDv check (Step 1) ≡ Ticket_v contains IDv inside the Kv-encrypted content. Server V checks this matches its own name. Prevents using a file-server ticket at the mail server. = "This pass says Treasury Room — yes, that's where I am."
Server's own replay cache ≡ Server V maintains its own independent replay cache. Auth_c' has never been shown to TGS — it's a new Authenticator for a new relationship. = The Treasury guard's own logbook of recent declarations seen
No KDC contact ≡ Server V verifies everything offline using Kv. Zero network calls to KDC or TGS at this step. Fully decentralized. = The Treasury guard doesn't call the Front Desk — he verifies with his own key
Mappings: Opening Treasury pass with Treasury key → Decrypt Ticket_v with Kv Opening declaration with "GoldenKey" → Decrypt Auth_c' with Kc,v "This pass is for the Treasury" check → IDv == own service name Guard's own logbook → Server V's independent replay cache
3.3
Mutual Authentication — Server Proves Its Identity to Alice
V → C  |  E(Kc,v, [TS3 + 1]) — Alice verifies the server is genuine
Mutual Auth
Palace Story
Treasury Head Guard → Alice:
"You declared the time as 12:10 PM. Here is my reply: 12:11 PM — one minute later — sealed with 'GoldenKey'. Only a guard who actually opened your Treasury pass and read your declaration could know your exact time. This proves I am the real Treasury."
Alice:
"Correct. I declared 12:10. He echoed 12:11 — exactly one more than mine, sealed with 'GoldenKey'. He must have opened the Treasury pass (only he can do that) to learn 'GoldenKey'. This is the genuine Treasury. I trust this room."
Real Kerberos Protocol
V → C (optional mutual auth): Authenticator_v = E(Kc,v, [TS3 + 1]) Alice verifies: 1. Decrypt Authenticator_v with Kc,v 2. Check: decrypted value == TS3 + 1 3. Check: TS3 was her OWN timestamp from Auth_c' → Mutual authentication complete. → Both parties have now authenticated each other.
TS3 + 1 ≡ The +1 convention proves Server V read Alice's specific Auth_c' (extracting TS3) and is not replaying a captured old mutual-auth response from a different session. = "I saw YOUR 12:10 and added one minute" — proves the Guard read Alice's specific declaration
Why it proves identity ≡ An impostor server cannot produce this. It would need to: (a) know Kv to decrypt Ticket_v, (b) extract Kc,v, (c) decrypt Auth_c' to learn TS3, (d) encrypt TS3+1 with Kc,v. Missing Kv breaks the entire chain at step (a). = Only the real Treasury Guard has the Treasury master key to open the pass and learn "GoldenKey"
Full Mutual Authentication Achieved: Server V authenticated Alice in Steps 3.1–3.2. Alice authenticated Server V in Step 3.3. Both parties have now verified each other's identity using only symmetric cryptography and the KDC as a trusted third party — no certificates, no PKI, no asymmetric operations.
Mappings: Guard echoes "12:11" sealed with "GoldenKey" → E(Kc,v, [TS3+1]) Only real Guard knows "GoldenKey" → only real Server V has Kv → can extract Kc,v Alice recognises the correct echo → Alice decrypts and checks TS3+1 == her own TS3 incremented
Phase 4 — Attack Scenario: Why Replay Fails
4.1
Replay Attack — Why an Old Ticket + Authenticator Fails
Attacker role-tag: Mallory
Threat Model
Palace Story
Mallory (Attacker), the next morning:
"Yesterday I copied Alice's Treasury pass and her declaration from when she visited. Today I will try to use them to enter the Treasury myself."
Treasury Head Guard (reading the copied declaration):
"This declaration says 12:10 PM yesterday. The current time is 9:00 AM today. That is nearly 21 hours old. Rejected — stale declaration. Also, this Treasury pass expired at sunset yesterday. Rejected — expired pass. Guards! Remove this impostor."
Why Replay Fails — Multiple Independent Defenses
Mallory presents (stolen yesterday): Ticket_v_old ← from Alice's session yesterday Authenticator_old = E(Kc,v, IDc ‖ ADc ‖ TS_old) Server V checks: Defense 1 — Ticket expiry: TS2 + Lifetime_v < now → Ticket_v_old is EXPIRED → Rejected immediately Defense 2 — Authenticator freshness: |now − TS_old| >> 5 minutes → Rejected (stale timestamp) Defense 3 — Replay cache: (IDc, TS_old) already in cache from yesterday → Rejected (duplicate) Defense 4 — Without Kc,v: Mallory can't create a NEW Authenticator either (doesn't have Kc,v — expired with the ticket)
Defense 1: Lifetime ≡ Ticket_v expires automatically (typically 1–2 hours). Stolen ticket becomes worthless without revocation infrastructure. = The Treasury pass expired at sunset — it says so on the pass (inside the sealed content Alice cannot alter)
Defense 2: TS freshness ≡ Any Authenticator older than ±5 minutes is automatically rejected. No window for slow-replay attacks. = Declaration timestamped 21 hours ago — the guard rejects it as stale
Defense 3: Replay cache ≡ Even within the 5-minute window, the exact (IDc, TS) pair can only be used once. Simultaneous replay also blocked. = Guard's logbook: "This exact declaration was already used yesterday"
Defense 4: No Kc,v ≡ Even if Ticket_v were still valid, Mallory cannot create a new Authenticator without knowing Kc,v. The session key expired along with the ticket. = Mallory doesn't know "GoldenKey" — can't write a valid fresh declaration
Pass-the-Ticket (real threat): If Mallory steals BOTH Ticket_v AND Kc,v from Alice's machine memory (e.g., Mimikatz on Windows) WHILE the ticket is still valid, she can impersonate Alice until expiry. Defense: Credential Guard (hardware-isolated memory), short ticket lifetimes, Protected Users group.
Mappings: Expired Treasury pass → Lifetime_v exceeded 21-hour-old declaration → TS_old outside ±5 min freshness window Declaration already in guard's logbook → (IDc, TS) pair in replay cache Mallory doesn't know "GoldenKey" → Cannot forge new Authenticator without Kc,v

One-Line Memory Flow — From Login to Service Access

0
Setup (offline): KDC stores Kc = hash(Alice's password). KDC stores Kv per service. KDC/TGS holds Ktgs. All out-of-band — no network messages.
↓
1
AS Exchange: Alice sends IDc + IDtgs + TS1 in plaintext. AS replies with E(Kc,[Kc,tgs]) + TGT=E(Ktgs,[Kc,tgs ‖ IDc ‖ ...]). Alice decrypts Part_A → learns Kc,tgs → discards Kc.
↓
2
TGS Exchange: Alice sends IDv + TGT (opaque) + Auth_c=E(Kc,tgs,[IDc ‖ TS2]). TGS decrypts TGT → extracts Kc,tgs → verifies Auth_c → cross-checks IDc → generates Kc,v → replies with E(Kc,tgs,[Kc,v]) + Ticket_v=E(Kv,[Kc,v ‖ IDc ‖ ...]).
↓
3
Service Access: Alice sends Ticket_v (opaque) + Auth_c'=E(Kc,v,[IDc ‖ TS3]). Server V decrypts Ticket_v → extracts Kc,v → verifies Auth_c' → grants access → optionally replies with E(Kc,v,[TS3+1]) for mutual auth.
↓
✓
Mutual trust established. Alice knows Server V is genuine (TS3+1 proof). Server V knows Alice is genuine (two-layer ticket+authenticator check). All communication encrypted with Kc,v. KDC is entirely out of the critical path.
🔐
Password Never Transmitted
Alice's password never leaves her machine in any form — not even encrypted. Verification is implicit: only Alice can derive the correct Kc to decrypt the AS reply. The password is the "key to the key."
🎟️
Tickets Are Opaque Couriers
Alice carries TGT and Ticket_v but cannot read either. She is a courier, not a creator. This prevents her from modifying lifetimes, identities, or session keys embedded inside — all content is under KDC authority.
⏱️
Automatic Expiry — No Revocation Needed
Tickets expire by design. A stolen TGT is useless after ~8 hours; a stolen Ticket_v after ~1 hour. No CRL, no OCSP, no manual revocation infrastructure required for this time-bounded guarantee.
🔄
Anti-Replay: Three Independent Defenses
Ticket lifetime expiry + Authenticator ±5 minute freshness window + per-server replay cache. All three must fail simultaneously for a replay attack to succeed — defense in depth at the protocol level.
🤝
Mutual Authentication
Server V proves to Alice it holds Kv (by producing E(Kc,v,[TS3+1])), which required opening Ticket_v. Alice proves to Server V she holds Kc,v (by producing Auth_c'). Neither can be impersonated by a third party.
🏝️
Key Isolation Per Service
Each service has its own independent Kv. Each session has its own independent Kc,v. Compromise of the file server's keys has zero impact on the mail server's session — blast radius is a single service.
📴
Fully Offline Service Verification
Server V makes zero network calls to KDC or TGS at access time. Everything is verified locally using Kv. The KDC is only involved at login (AS) and at ticket refresh (TGS) — not in every service request.
🔑
Symmetric Cryptography Only
Kerberos achieves all goals — authentication, key distribution, mutual auth — using only symmetric encryption (AES in modern deployments). No asymmetric operations, no certificates, no PKI required.
