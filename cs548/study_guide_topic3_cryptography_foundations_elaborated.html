🧗 Warm-up Glossary — The Math You Need Before You Start

Cryptography looks scary mostly because it leans on a few math operations and notations that most courses assume you already know. None of them are actually hard. This short glossary unpacks each one. If you already know all of these, skip ahead to §1.

What is a "bit"?

A bit is a single binary digit — either 0 or 1. Computers store everything as long sequences of bits. Eight bits make a byte; one byte can hold a single ASCII character ('A' is byte 01000001) or a number from 0 to 255. When we talk about a "128-bit key," we mean a sequence of 128 zeros and ones — that's 16 bytes long.

What is XOR (⊕)?

XOR stands for "exclusive-or." It is the simplest possible cryptographic operation: a bit-by-bit comparison of two binary strings that asks "are these two bits different?" If yes → output 1; if same → output 0. The truth table: 

   A   B   A ⊕ B    Reading the table:
   0   0     0       same  → 0
   0   1     1       diff. → 1
   1   0     1       diff. → 1
   1   1     0       same  → 0

XOR has a magical property that makes it cryptography's favourite operation: applying it twice undoes it. That is, (P ⊕ K) ⊕ K = P. Concretely: 

  Plaintext  P:    1 0 1 1 1 1 0 1
  Key        K:    0 0 1 1 0 0 1 0
  ─────────────────────────────────  ⊕
  Ciphertext C:    1 0 0 0 1 1 1 1     (encrypt: C = P ⊕ K)

  Ciphertext C:    1 0 0 0 1 1 1 1
  Key        K:    0 0 1 1 0 0 1 0
  ─────────────────────────────────  ⊕
  Plaintext  P:    1 0 1 1 1 1 0 1     (decrypt: P = C ⊕ K, the original!)

This is the entire mathematical mechanism behind the One-Time Pad and the keystream-XOR step of every stream cipher and CTR-mode block cipher. 📄 p. 60

What is "mod" (modular arithmetic)?

"a mod n" is the remainder when you divide a by n. So 13 mod 5 = 3 (because 13 ÷ 5 = 2 remainder 3), and 17 mod 12 = 5 (think of a clock: 17 hours past midnight is 5 o'clock the next day). Modular arithmetic is "clock arithmetic" — once you reach n, you wrap back to 0.

In RSA we'll do operations like 12317 mod 3233. That looks intimidating but it just means "compute 123 to the 17th power, then take the remainder when divided by 3233." The number 12317 is enormous (about 1035), but the remainder modulo 3233 is always between 0 and 3232 — a tiny manageable number. That's why crypto can use huge exponents without producing impossibly large outputs. 📄 p. 89

What is "modular exponentiation"?

Just doing the mod operation after exponentiation, written as xe mod n. The exponent e can be very large in real RSA (e.g. 65,537), but algorithms like square-and-multiply compute the result quickly — about 30 multiplications even for a 30-bit exponent. 📄 pp. 92-93

What is "GCD" (greatest common divisor)?

The biggest integer that divides two numbers without remainder. GCD(12, 18) = 6; GCD(7, 11) = 1 (they share no common factor — they are coprime). RSA requires that GCD(e, φ(n)) = 1, which just means we pick the public exponent e to be coprime with φ(n). 📄 p. 89

What is the "Euler totient" φ(n)?

Euler's totient function φ(n) counts how many integers between 1 and n are coprime to n (share no common factor with n other than 1). For RSA we always set n = pq where p and q are primes, and there's a magic shortcut: φ(pq) = (p-1)(q-1). So if p=61 and q=53, then n=3233 and φ(n) = 60×52 = 3120. 📄 p. 89

What is a "block"?

A fixed-size chunk of bits that a block cipher processes at one time. AES uses 128-bit blocks — that's 16 bytes, or about 16 ASCII characters. If your message is longer than one block, the cipher processes it block-by-block; if it's shorter, you pad it out to a full block. 📄 p. 50

What is a "key"?

A secret value that controls how an encryption algorithm scrambles plaintext into ciphertext. In symmetric ciphers, sender and receiver share the same key. In asymmetric ciphers, you have a pair of keys: a public one (anyone can have it) and a private one (only you have it). Key sizes are measured in bits — common sizes: 128/192/256 bits for AES, 2048/3072 bits for RSA, 256 bits for ECC. The bigger the key, the more guesses an attacker would need to brute-force it. 📄 pp. 50, 88, 195

What does "infeasible" mean here?

Cryptographers say something is computationally infeasible when no known algorithm can do it in any reasonable amount of time on any realistic computer. "Infeasible" doesn't mean "literally impossible" — it means "would take longer than the age of the universe." 2128 operations is the typical bar; even a billion computers each doing a billion operations per second would take 1022 years. 📄 p. 50, 190

§1 — What & Why, in Plain English

📘 LECTURE PDF COVERAGE: pp. 30-31, 40-41, 54-55, 60, 85-87, 152-157

1.1 Why does cryptography exist at all? 📄 pp. 30, 40

Imagine you want to send a love letter to a friend across town, but you have to give it to a stranger to deliver. You don't know the stranger. They might read your letter, change a word in it, or replace it entirely with a fake letter pretending to be from you. The stranger can do all of this and you'd never know.

Now replace "letter" with "your bank password" and "stranger" with "the public Internet." That's the situation cryptography exists to fix. The lecture frames it as four security goals 📄 p. 30:

GoalPlain-English meaning"My banking app needs..."
Confidentiality Only the intended reader can read the message. To everyone else, it looks like random noise. ...nobody on coffee-shop Wi-Fi to see my password.
Integrity If someone tampered with the message in transit, the recipient can detect it. ...nobody to silently change "$10" to "$10,000".
Authenticity The recipient can verify the message really came from the claimed sender. ...to be sure the message claiming to be from my bank actually is from my bank.
Non-repudiation The sender can't later deny they sent it (legally provable). ...if I sign a contract online, I can't claim later "that wasn't me."

Each cryptographic primitive you'll learn — symmetric ciphers, asymmetric ciphers, hashes, MACs, digital signatures — exists to provide one or more of these goals. Memorize the table above. Every exam question about "which primitive provides which property?" reduces to this table.

1.2 The four primitive families (gentle introduction) 📄 pp. 55, 137, 152

Cryptography is not one tool. It's a small toolbox of four fundamentally different tools. Understanding what each one is for is more important than memorizing how each one works. Here's the soft-pitch version; we'll go deeper in §2.

① Symmetric ciphers — the workhorse

What it does: takes a message and a secret key, and produces ciphertext that looks like noise. With the same key, you can run it backwards to recover the message.
Real-world example: AES-128, the cipher protecting almost every HTTPS connection. Catch: sender and receiver must already agree on the secret key. Symmetric crypto is fast (gigabytes per second on modern CPUs) but can't solve the bootstrapping problem of how to agree on a key in the first place. 📄 pp. 55, 85

② Asymmetric ciphers — the magic trick

What it does: uses a pair of keys instead of one. You publish your public key; anyone in the world can use it to encrypt a message to you. Only you, holding the matching private key, can decrypt.
Real-world example: RSA, Diffie-Hellman, Elliptic-Curve cryptography.
Catch: 100-1000× slower than symmetric crypto, so we don't use it for bulk data. We use it to solve the symmetric-key-bootstrapping problem, then switch to symmetric for the actual traffic. 📄 pp. 55, 87, 150

③ Hashes — the fingerprint

What it does: takes any input (text, image, binary file) and produces a fixed-length "fingerprint" — typically 256 bits. The fingerprint is deterministic (same input → same fingerprint) but irreversible (you can't go back from fingerprint to original).
Real-world example: SHA-256.
Catch: A hash isn't keyed — anyone can compute it. So a hash by itself only gives you integrity (did this match what I expected?), not authenticity (was this from who I think?). 📄 pp. 152, 155

④ MACs — fingerprint with a key

What it does: exactly like a hash, but mixed with a secret key. Now to verify the fingerprint, you also need the key — which means anyone who can verify must be a trusted party. This gives you both integrity AND authenticity.
Real-world example: HMAC-SHA-256, AES-GCM.
Catch: Both sender and receiver hold the same key, so a MAC can't prove authorship in court (either party could have made it). For that you need a digital signature (asymmetric). 📄 pp. 172, 174

1.3 The "hybrid pattern" — how every secure protocol actually works 📄 pp. 150, 181

Now here's the punchline that makes all four primitives feel like they belong together. Real protocols (TLS, IPsec, SSH, WPA2) use them in a specific recipe: 

  THE HYBRID CRYPTO PATTERN  📄 pp. 150, 181

  Step 1.  Client and server agree on a fresh shared secret using
           ASYMMETRIC crypto (Diffie-Hellman or RSA).
           ─────► Solves the "we never met before" problem.

  Step 2.  Server proves its identity to client using a DIGITAL SIGNATURE
           (asymmetric, with a public-key certificate from a trusted CA).
           ─────► Solves the "are you really my bank?" problem.

  Step 3.  All bulk traffic is encrypted with SYMMETRIC crypto using the
           shared secret from step 1 — typically AES-GCM.
           ─────► Fast confidentiality + integrity for the data stream.

  Step 4.  Each packet's integrity is verified by AES-GCM's built-in MAC
           or by HMAC over the ciphertext (Encrypt-then-MAC).
           ─────► Catches tampering attempts.

Every TLS handshake, every SSH session-setup, every IPsec IKE exchange follows this exact recipe. Remembering the pattern is more useful than memorizing any single algorithm, because it gives you a mental schema for understanding all higher-level protocols.

1.4 An extended analogy — A corporate building's security 📄 pp. 85, 87, 145, 155

Each primitive maps cleanly to a different physical-security mechanism. Hold this picture in your head and crypto stops feeling abstract:

PrimitivePhysical analogProperty it provides
Symmetric cipher (AES) A lock with a single key. Both you and your partner have copies. You can both open or close it. Confidentiality, fast.
Asymmetric — encryption (RSA) A locked mailbox with a slot. Anyone can drop a letter in (using the public-facing slot = public key). Only you can unlock and retrieve letters (using your private key). Confidentiality without pre-sharing a key.
Asymmetric — key exchange (Diffie-Hellman) A locked supply room two strangers each hold a different key to. They open the lock together; afterwards both walk out with the contents. They never had to physically meet beforehand. Agreeing on a secret without ever transmitting it.
Hash function (SHA-256) A tamper-evident seal on a package. You can't tell who sealed it, but you can tell if it's been opened. Integrity (not authenticity).
MAC / HMAC A wax seal imprinted with a special signet ring that only you and your trusted partner share. Integrity + authenticity (both held secret).
Digital signature (RSA-sign) A wax seal with a unique imprint only you can create, and that anyone can verify by looking at it (because they have your public-key seal-pattern). Integrity + authenticity + non-repudiation (you can't deny it).
Public-key certificate The government registry that vouches for which signet-ring belongs to which person — so a stranger can trust the seal even if they've never met you. Anchors trust in a public-key infrastructure.

This building-security analogy will keep returning throughout §2. When something feels too abstract, ask yourself: "What is the physical thing this is replacing?"

Ciphertext The encrypted version of the plaintext. Looks like random noise to anyone without the key. "x9!$#%@kP&..." 📄 p. 50

Confusion Shannon's term for "make the relationship between key and ciphertext as complicated as possible." Achieved via substitution (S-boxes). AES SubBytes step 📄 pp. 62, 77
Diffusion Shannon's term for "spread the influence of every plaintext bit over many ciphertext bits." Achieved via permutation (mixing). AES ShiftRows + MixColumns 📄 pp. 62, 77
Substitution Replacing each input symbol with a (key-determined) output symbol via a lookup table. S-box: byte → byte mapping 📄 pp. 58, 65

Trap-door one-way function A function easy to compute forward, but impossible to invert UNLESS you know a secret "trap-door" (the private key). The mathematical heart of asymmetric cryptography. RSA: easy forward (raise to power), hard backward (without d) 📄 p. 87

Pre-image resistance Hash property: given hash h, infeasible to find any input m with H(m) = h. Cost: 2^256 for SHA-256 📄 p. 157
Collision resistance Hash property: infeasible to find ANY two inputs m1, m2 with same hash. Cost: 2^128 for SHA-256 (birthday) 📄 p. 162
Avalanche Property: a single-bit change in input flips ~half the output bits. Required of every good hash and cipher. SHA("over") vs SHA("once") look totally different 📄 p. 155
HMAC "Hash-based MAC." A specific recipe that takes any hash function and a key, and produces a keyed MAC. Used everywhere. HMAC-SHA-256 📄 p. 178
AES-GCM AES in Galois/Counter Mode. Combines AES-CTR encryption + a built-in MAC tag, all in one pass. The default for TLS 1.3, IPsec, and WPA2. Output: ciphertext + 128-bit tag 📄 p. 181
Common point of confusion (clear this up now) Students often think "encrypted = secure." It's not. Encryption (confidentiality) is only one of the four security goals. A message that's encrypted but not authenticated can still be tampered with — the attacker can flip bits in the ciphertext, which flips corresponding bits in the plaintext after decryption, and the receiver has no way to detect this. That's why modern protocols always combine encryption with a MAC (or use AEAD modes like AES-GCM that bundle both). 📄 pp. 170, 181

Building Block #1: The One-Time Pad (the only "perfect" cipher) 📄 pp. 60-61

Intuition

The simplest cipher in the world is the One-Time Pad. The recipe is one line: "XOR your plaintext with a random key as long as the plaintext." If the key is truly random and used only once, this cipher is provably unbreakable — Claude Shannon proved this in 1949. 📄 p. 60

Worked example, bit by bit

Suppose Alice wants to send Bob the 8-bit message 10111101 ("hello" in some toy encoding). They've previously met in person and shared a one-time random key 00110010. Encryption is just XOR:

  ENCRYPTION                              DECRYPTION

  Plaintext  P:  1 0 1 1 1 1 0 1          Ciphertext C:  1 0 0 0 1 1 1 1
  Key        K:  0 0 1 1 0 0 1 0          Key        K:  0 0 1 1 0 0 1 0
  ─────────────────────────────  ⊕        ─────────────────────────────  ⊕
  Ciphertext C:  1 0 0 0 1 1 1 1          Plaintext  P:  1 0 1 1 1 1 0 1

  Bob receives C, XORs with the same K, recovers P. Done.

Let's verify the magic: encryption took P, XORed it with K, and produced C = P ⊕ K. Decryption takes C and XORs again with K. Substituting: 

  C ⊕ K  =  (P ⊕ K) ⊕ K
         =  P ⊕ (K ⊕ K)        ← XOR is associative
         =  P ⊕ 0               ← anything XORed with itself = 0
         =  P                   ← anything XORed with 0 = itself

Why isn't OTP used for everything?

Three killer practical problems 📄 p. 61:

  1. The key is as long as the message. If you can securely share a random key of size N bytes, you might as well securely share the N-byte message itself. OTP doesn't solve the bootstrapping problem; it just shifts it earlier.
  2. The key can never be reused. If the same K encrypts two messages P₁ and P₂, then C₁ ⊕ C₂ = P₁ ⊕ P₂ — the attacker recovers the XOR of the plaintexts and can mostly crack them with statistical analysis.
  3. No integrity protection. An attacker who flips bits in the ciphertext flips the corresponding bits in the plaintext after decryption. OTP gives confidentiality, not integrity.

The takeaway: OTP is a benchmark, not a tool. Real ciphers (AES, RC4, etc.) are designed to approximate OTP's secrecy with much smaller, reusable keys.

Building Block #2: Block Ciphers (AES) — the workhorse 📄 pp. 50, 63-67

Intuition

AES takes a 128-bit chunk of plaintext and a 128/192/256-bit key, and outputs a 128-bit chunk of ciphertext that looks like noise. Internally, it scrambles the input through 10/12/14 rounds, where each round applies four small operations designed to maximize confusion and diffusion.

The four operations of one AES round

AES treats each 128-bit block as a 4×4 matrix of bytes (16 bytes total). Here's what one round does, with a small concrete example. 

  ONE AES ROUND — the four operations              📄 pp. 63-66

  Starting state (after previous step):
       ┌──┬──┬──┬──┐
       │57│83│5C│F0│        Each cell = one byte
       │82│21│7D│03│        (shown in hexadecimal — base 16,
       │1A│B6│C5│89│         where 0-9 then A=10, B=11, ..., F=15)
       │EE│45│22│71│
       └──┴──┴──┴──┘

  ─── ① SubBytes ─── (substitution)                📄 p. 65
  Each byte X is replaced by the corresponding byte in a fixed table called the S-box.
  The S-box is just a 256-entry lookup table that nobody could guess from inspection.
  E.g., the byte 0x57 becomes the byte at row 5, column 7 of the S-box.

  Purpose: introduces NONLINEARITY (Shannon's "confusion"). This is the only nonlinear
  step — without it, AES would be vulnerable to algebraic attacks.

  ─── ② ShiftRows ─── (permutation)                📄 p. 66
  Row 0: no shift
  Row 1: shift left by 1 byte                                    Before        After
  Row 2: shift left by 2 bytes                                  ┌──┬──┬──┬──┐  ┌──┬──┬──┬──┐
  Row 3: shift left by 3 bytes                                  │a │b │c │d │  │a │b │c │d │
                                                                │e │f │g │h │  │f │g │h │e │
  Purpose: spreads each byte across columns. (This step alone   │i │j │k │l │  │k │l │i │j │
  doesn't change byte values — only positions.)                 │m │n │o │p │  │p │m │n │o │
                                                                └──┴──┴──┴──┘  └──┴──┴──┴──┘

  ─── ③ MixColumns ─── (permutation + linear transformation)    📄 p. 66
  Each column is replaced by a new column where every output byte depends on
  every input byte in that column. (Implemented as a matrix multiplication
  in a finite field.)

  Purpose: spreads each byte's influence across all four bytes in its column —
  this is the bulk of Shannon's "diffusion."

  ─── ④ AddRoundKey ─── (XOR with subkey)          📄 p. 65
  XOR the entire 16-byte state with the round's subkey (a 16-byte value
  derived from the master key by the KEY SCHEDULE).

  Purpose: this is the ONLY step where the secret key affects the state.
  The other three are public, fixed operations.

  ⇒ Now the state is fed into the NEXT round, repeating ① ② ③ ④.
  ⇒ AES-128 does this 10 times (the last round skips MixColumns).
SOURCE: 📄 pp. 63-66
Why four operations? Why this exact sequence? Each round needs to be invertible (so we can decrypt) AND each round needs to mix the input thoroughly. SubBytes provides confusion; ShiftRows + MixColumns together provide diffusion; AddRoundKey injects the secret. Doing them in this order with 10+ rounds means a single bit of plaintext influences every bit of ciphertext through "avalanche." 📄 pp. 64-66

Building Block #3: Modes of Operation — making AES handle long messages 📄 pp. 70-82

Why we need modes

AES encrypts only one 128-bit block at a time. But your messages are usually longer than 16 bytes — a single tweet has more. So we need a recipe for using AES to encrypt arbitrary- length messages. That recipe is called a mode of operation. The three classic modes are ECB, CBC, and CTR. 📄 p. 70

ECB Mode — and why it's broken for real data 📄 pp. 71-72

ECB (Electronic Code Book) is the obvious approach: split the message into 16-byte blocks, encrypt each one independently with AES. 

  ECB ENCRYPTION                                  📄 p. 71

  P₁ ──► [AES_k] ──► C₁
  P₂ ──► [AES_k] ──► C₂          (independent encryptions)
  P₃ ──► [AES_k] ──► C₃
  ...

Sounds fine, right? Wrong, and the reason is profound: identical plaintext blocks produce identical ciphertext blocks. So if your message has structure (e.g., an image with regions of solid color, or an XML file with repeated tags), that structure leaks through. The famous "Tux penguin" image example on slide 72 shows this dramatically: an image encrypted with ECB mode is still recognizably the original image, just with weird colors. 📄 p. 72

Why ECB leaks structure (concretely) Suppose the plaintext is the text "YESYESYESYESYESY" (16 chars = 16 bytes = 1 AES block) followed by "ESYESYESYESYESYE" (another 16 bytes). The two blocks are different, fine. But if your plaintext is "AAAAAAAAAAAAAAAA" repeated three times, ECB outputs the same ciphertext block three times. An attacker who has no key but sees three identical ciphertext blocks knows: "those three regions had identical plaintext." For an image with sky regions, the sky regions all collapse to the same ciphertext color, leaking the image shape. 📄 p. 72

CBC Mode — chaining blocks together 📄 pp. 73-74

CBC (Cipher Block Chaining) fixes ECB's identical-blocks-leak-structure problem by XOR-ing each plaintext block with the previous ciphertext block before encrypting. The very first block has no previous ciphertext, so we use a fresh random IV (Initialization Vector) instead. 

  CBC ENCRYPTION                                  📄 p. 73

           IV ──┐
                │
                ▼
   P₁ ────────► XOR ──► [AES_k] ──► C₁
                                     │
                                     ▼
   P₂ ────────► XOR ──► [AES_k] ──► C₂
                                     │
                                     ▼
   P₃ ────────► XOR ──► [AES_k] ──► C₃
                                     │
                                     ▼
                                    ...

  CBC DECRYPTION                                  📄 p. 74

   C₁ ──► [AES_k⁻¹] ──► XOR ──► P₁
                          │
                          IV ──┘ (XOR with IV recovers P₁)

   C₂ ──► [AES_k⁻¹] ──► XOR ──► P₂
                          │
                          C₁ ──┘ (XOR with previous ciphertext recovers P₂)

   C₃ ──► [AES_k⁻¹] ──► XOR ──► P₃
                          │
                          C₂ ──┘

Now identical plaintext blocks produce different ciphertext (as long as the previous ciphertext differs). Encryption is strictly sequential — you can't compute C₂ until C₁ is done. Decryption can be parallelized (each block needs only its own ciphertext + the previous one). 📄 pp. 73-74

CTR Mode — turning AES into a stream cipher 📄 pp. 77, 80

CTR (Counter mode) does something clever: it doesn't encrypt the plaintext at all directly. Instead, it encrypts a counter, then XORs the resulting "keystream" with the plaintext. The plaintext is treated as if it were a stream — there's no notion of blocks at the data level. 

  CTR MODE ENCRYPTION                             📄 pp. 77, 80

  Step 1: Build a sequence of counters by combining a NONCE with an incremental counter:

           ┌─ 96-bit nonce ─┬─ 32-bit counter ─┐
           │   N (random)   │       0          │   ← counter input #1
           │   N            │       1          │   ← counter input #2
           │   N            │       2          │   ← counter input #3
           │   ...          │      ...         │

  Step 2: Encrypt each counter with AES under key k to get keystream blocks:

           AES_k(N‖0) ──► KS₀          ┐
           AES_k(N‖1) ──► KS₁          │  These look like random 16-byte blocks.
           AES_k(N‖2) ──► KS₂          │  Together they form a "keystream."
           ...                          ┘

  Step 3: XOR the plaintext bytes against the keystream:

           P₀ ⊕ KS₀ ──► C₀
           P₁ ⊕ KS₁ ──► C₁
           P₂ ⊕ KS₂ ──► C₂

  Decryption is identical: re-encrypt the same counters, get the same keystream,
  XOR with ciphertext to recover plaintext. (Notice we never use AES-decrypt!)
SOURCE: 📄 pp. 77, 80
Why CTR mode is the modern favourite Five concrete advantages over CBC, all from the lecture:
  1. Parallelizable in BOTH directions. Each keystream block is independent of the others, so you can compute KS₀, KS₁, KS₂... in parallel. CBC encryption is strictly sequential.
  2. Pre-generatable. If you know the nonce in advance, you can compute the entire keystream before the plaintext arrives. Then encryption is just XOR — extremely fast.
  3. Random access. To decrypt block i, you only need to compute KS_i — no need to process the previous blocks. Useful for encrypted disk storage where you might read block 1000 without reading blocks 0-999.
  4. No padding required. Since plaintext is XORed bit-by-bit (or byte-by-byte) against keystream, you can stop at any length. CBC requires the message length to be a multiple of the block size; you have to add padding bytes.
  5. Only AES-encrypt is needed. Decryption uses the same AES-encrypt operation (to regenerate the keystream), so hardware/firmware can omit the AES-decrypt circuit.
📄 p. 80
⚠️ The cardinal CTR-mode rule — never reuse a (key, nonce) pair If you encrypt P₁ and P₂ with the same key AND the same nonce, both produce the same keystream KS. Then C₁ ⊕ C₂ = (P₁ ⊕ KS) ⊕ (P₂ ⊕ KS) = P₁ ⊕ P₂. The attacker has recovered the XOR of two plaintexts — same problem as OTP key reuse — and can usually crack it with statistical analysis. This is exactly how WEP was broken in 2001 (24-bit IVs guaranteed reuse within hours). 📄 pp. 80, 181
2bAsymmetric Ciphers — with numerical examples 📄 pp. 83-151 · Lectures 4 & 5

The big idea: trap-door one-way functions 📄 p. 87

Symmetric ciphers solve confidentiality after two parties have a shared secret. But how do they get that secret in the first place — over an insecure network, with no prior meeting? You can't send the key in plaintext (eavesdroppers would read it), and you can't encrypt it (you'd need a key to encrypt the key, which is the same problem!).

The breakthrough is a mathematical object called a trap-door one-way function:

  • One-way means computing f(x) is easy, but going backward — finding x given f(x) — is computationally infeasible.
  • Trap-door means there's a special secret (the "trap-door") that, if you have it, makes the inverse easy.

The public key is the description of f. The private key is the trap-door. Anyone in the world can encrypt (compute f(message)); only you, holding the trap-door, can decrypt.

RSA — fully worked numerical example 📄 pp. 88-95

RSA is the simplest asymmetric scheme to actually compute by hand. The slide deck even gives the example on slide 89: p=61, q=53, n=3233. We'll walk through every step.

Step 1: Key generation

  RSA Key Generation — toy example                📄 p. 89

  ① Pick two primes p and q.
       p = 61
       q = 53
     (real RSA uses primes that are 1024+ bits each, but the math is identical)

  ② Compute n = p × q.
       n = 61 × 53 = 3233       ← This will be public.

  ③ Compute φ(n) = (p-1)(q-1).
       φ(n) = 60 × 52 = 3120    ← This stays private.

  ④ Pick a public exponent e.
     Requirement: 1 < e < φ(n) AND gcd(e, φ(n)) = 1.
       Pick e = 17.
       Check gcd(17, 3120)?  17 is prime, 3120 = 17 × 183 + 9, so gcd = gcd(17,9)
       = gcd(9, 17 mod 9) = gcd(9, 8) = gcd(8, 1) = 1. ✓

  ⑤ Compute private exponent d such that (d × e) mod φ(n) = 1.
     This is the "modular inverse" of e modulo φ(n) — found via the
     Extended Euclidean Algorithm.
       d = 17⁻¹ mod 3120 = 2753
       Check: 17 × 2753 = 46801 = 15×3120 + 1. ✓ So 17×2753 mod 3120 = 1. ✓

  ⇒ Public key:  (e, n) = (17, 3233)
  ⇒ Private key: (d, n) = (2753, 3233)
  ⇒ p, q, φ(n) are all secret and discarded after key generation.

Step 2: Encryption

  RSA Encryption — Alice → Bob                    📄 p. 89

  Bob's public key is (e=17, n=3233). It's posted on his website.

  Alice wants to send the message M = 123 to Bob.
  (Real RSA encrypts whole bytes; "123" is just our toy plaintext.)

  Compute:  C = M^e mod n = 123^17 mod 3233

  Let's verify by computing 123^17 step-by-step using square-and-multiply:

      123^1  =        123       mod 3233 = 123
      123^2  =      15129       mod 3233 = 855  (since 15129 = 4×3233 + 2197... actually let me recheck)
      123^2  =     15129        mod 3233 = 15129 - 4*3233 = 15129 - 12932 = 2197
      123^4  =   2197^2 = 4826809  mod 3233 = compute by hand or take it on faith
                              ⋯
      ...
      123^17 mod 3233 = 855          ← Alice sends this to Bob.

  (The key insight: 123^17 is about 10^35, but its remainder mod 3233 is just
  the small number 855. This is why RSA works with huge exponents — outputs
  always stay below n.)

Step 3: Decryption

  RSA Decryption — Bob receives 855               📄 p. 89

  Bob has private key (d=2753, n=3233).

  Compute:  M = C^d mod n = 855^2753 mod 3233

  This number 855^2753 is astronomically large (about 10^8000), but again
  the remainder modulo 3233 is small. Using square-and-multiply:

      855^2753 mod 3233 = 123        ← Bob recovers the original message!

  Why does this work? It's a mathematical identity (Fermat's Little Theorem):
  for any M coprime to n,
        M^(e·d) mod n = M^1 mod n = M
  because e·d ≡ 1 (mod φ(n)), and exponents reduce modulo φ(n).
  See slide 89 for the full proof sketch.
Where does RSA's security come from? An attacker sees the public key (e, n) and the ciphertext C. To recover M, they would need to know d. To compute d from e, they would need φ(n) = (p-1)(q-1). To compute φ(n) from n, they would need to factor n into p × q — which for 2048-bit n has no known efficient algorithm. So RSA's security rests on the conjectured difficulty of integer factoring. 📄 p. 91

Same operation, two uses: encryption and signature 📄 pp. 94, 103

Here's a beautifully economical design: the same mathematical operation (modular exponentiation) can be used for both encryption and digital signatures, just by swapping which key you use. 

  RSA — same math, two purposes                   📄 p. 103

  ENCRYPT (Alice → Bob, confidentiality)
      C = M^e mod n         using Bob's PUBLIC key (e,n)
      M = C^d mod n         using Bob's PRIVATE key (d,n)

  SIGN (Alice proves she wrote M, integrity + authenticity + non-repudiation)
      s = hash(M)^d mod n   using Alice's PRIVATE key (d,n)
      verify by checking:   s^e mod n  ==  hash(M)        using Alice's PUBLIC key

  Why hash first? Because real M is too long to use directly, and signing the
  raw M without hashing has security pitfalls (an attacker can pick a
  convenient M to make the signature math collapse).

Diffie-Hellman key exchange — agreeing on a secret without ever meeting 📄 pp. 97-98

Intuition

Diffie-Hellman lets two strangers, communicating over an insecure channel, agree on a shared secret that an eavesdropper cannot derive — even though everything they exchanged on the wire was visible to that eavesdropper.

Worked example with small numbers

  DH Key Exchange — Alice ↔ Bob, with toy numbers      📄 pp. 97-98

  Public parameters (everyone knows these):
       prime p = 23
       generator g = 5

  ─────────────────────────────────────────────────────────────
  Alice                                       Bob
  ─────────────────────────────────────────────────────────────
  Pick secret a = 6                           Pick secret b = 15
  (private to Alice)                          (private to Bob)

  Compute A = g^a mod p                       Compute B = g^b mod p
        = 5^6 mod 23
        = 15625 mod 23                              = 5^15 mod 23
        = 8                                         = 19

  ─── Send A=8 to Bob ────────►
                              ◄──── Send B=19 to Alice ────────

  Compute K = B^a mod p                       Compute K = A^b mod p
        = 19^6 mod 23                               = 8^15 mod 23
        = 47045881 mod 23                           = 35184372088832 mod 23
        = 2                                         = 2

  Both now share K = 2 !
  ─────────────────────────────────────────────────────────────

  An eavesdropper saw on the wire:  p=23, g=5, A=8, B=19.
  Could they compute K?  They'd need either a (Alice's secret) or b (Bob's secret).
  To find a from A: solve  5^a ≡ 8 (mod 23)  for a — this is the
  DISCRETE LOGARITHM problem, believed to be infeasible for large p.

In real-world DH, p is at least 2048 bits (a number with 600+ digits). Computing 5^a mod p is fast (the square-and-multiply trick); but solving 5^a ≡ A (mod p) for a is essentially trying every possible a — infeasible for large p.

Why DH instead of RSA for key exchange? 📄 pp. 98, 122-123

RSA can also do key exchange — Alice picks a fresh symmetric key, encrypts it with Bob's RSA public key, sends it. Why isn't that enough? The answer is Perfect Forward Secrecy (PFS):

ScenarioRSA key transportEphemeral DH (DHE/ECDHE)
An attacker records all encrypted traffic between Alice and Bob today. Recorded. Recorded.
Five years later, Bob's long-term private key is stolen. The attacker can retroactively decrypt all old recorded traffic. (Just decrypt the symmetric keys with the now-known private key.) The session keys were derived from ephemeral a, b that Alice and Bob discarded immediately. Bob's long-term key is unrelated. The traffic stays secret. ✅

TLS 1.3 removed RSA key transport for exactly this reason — only ephemeral (EC)DHE is permitted now. 📄 p. 101

Public-key certificates — closing the trust loop 📄 pp. 104, 145

One more piece. When Alice connects to www.bankofamerica.com and gets back a public key, how does she know it really is the bank's public key, and not an imposter's? The bank's website could have been replaced by a phishing site that hands out the attacker's public key.

The slide-145 example makes this concrete: a "Fake BOA" site can reply to Alice's request with its own public key. If Alice naively trusts it, Fake BOA can decrypt her messages. RSA alone doesn't solve this. 📄 p. 145

The fix: X.509 certificates. The real BOA's public key is digitally signed by a trusted Certificate Authority (CA — e.g., DigiCert, Let's Encrypt). Alice's browser comes with a list of trusted root CAs preinstalled. When the bank presents its certificate, Alice's browser verifies the CA's signature on the certificate; if valid, she trusts the public key inside. Fake BOA can't forge such a signature without the CA's private key. 📄 pp. 104, 130, 145

Why public-key crypto is slow — and what we do about it 📄 p. 150

The catch with asymmetric crypto: it is 100-1000× slower than symmetric crypto. Modular exponentiation with 2048-bit numbers is much more expensive than a single AES round. Encrypting a 1 GB file directly with RSA would take all day.

Solution — the hybrid pattern you already saw in §1.3: use RSA or DH only to agree on a fresh symmetric key, then use AES (with that key) to encrypt the actual data. You pay the asymmetric cost once per session, and the symmetric cost per byte. This is what every secure protocol does. 📄 p. 150

2cSecure Hash Functions — what they really do 📄 pp. 152-162 · Lecture 5 part 2

Intuition — what is a hash, really? 📄 pp. 152, 155

A cryptographic hash function is a black box that takes any input — a single byte, a 1MB document, or a 1TB hard-drive image — and produces a fixed-length output (called the digest, fingerprint, or hash value). For SHA-256, the output is always 256 bits (32 bytes, displayed as 64 hex characters). For SHA-512, always 512 bits. 

  HASH FUNCTION — input vs output                 📄 pp. 152, 155

  ┌────────────────────────────────┐                    ┌─────────────────┐
  │ Input: anything, any length    │  ──► [SHA-256] ──► │ Output: 256 bits│
  │  - "Hello"  (5 bytes)          │                    │  (always 256!)  │
  │  - 1 MB PDF                    │                    └─────────────────┘
  │  - your entire hard drive      │
  └────────────────────────────────┘

  Examples (SHA-256, hex form):

   SHA256("Hello")
     = 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969

   SHA256("hello")        ← only one byte different from above!
     = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824

The two outputs above look completely unrelated, even though their inputs differ by one bit (capital H vs lowercase h). This is the avalanche property: any single-bit change in the input flips ~50% of the output bits, in unpredictable positions. 📄 p. 155

The four required properties 📄 p. 157

To call a function "cryptographically secure," it must satisfy four properties:

PropertyPlain EnglishWhy we need it
① Easy to compute You can hash any input quickly (microseconds for typical inputs). Otherwise it would be too slow to use everywhere.
② Pre-image resistance Given a hash h, infeasible to find any input m with H(m) = h. Otherwise an attacker who sees a password hash could find the password.
③ Second-pre-image resistance Given a specific m₁, infeasible to find a different m₂ with H(m₂) = H(m₁). Otherwise an attacker could substitute a malicious file for a legitimate one and the hash check would still pass.
④ Collision resistance Infeasible to find ANY pair m₁ ≠ m₂ with H(m₁) = H(m₂). Stronger property required for digital signatures — without it, an attacker could craft two documents with the same hash and trick a signer.

Plus the avalanche property: a single-bit input change flips ~half the output bits. This emerges naturally from properties ② ③ ④ but is worth stating separately because it's so visible.

Crucial distinction: hash is NOT invertible "with a key" 📄 p. 157

Common point of confusion Many beginners think: "If asymmetric crypto is invertible with the right key, can hashes also be inverted with some special key?" No. Hashes have no key, and they are not invertible under any circumstances. The output is shorter than the input — for SHA-256, a 1MB file is compressed into 32 bytes — so by pigeonhole there must be MANY inputs producing the same output. The hash destroys information. The instructor was emphatic on this point: "the difference between asymmetric cipher and secure hash is asymmetric cipher is invertible based on the correct keys, but the secure hash is not invertible." 📄 p. 157

How hashes are built — Merkle-Damgård construction 📄 p. 158

Most hash functions (MD5, SHA-1, SHA-2 family) use the Merkle-Damgård construction. The recipe: 

  MERKLE-DAMGÅRD CONSTRUCTION                     📄 p. 158

  Step 1: Pad the message m so its length is a multiple of the block size
          (typically 512 bits for SHA-256).

  Step 2: Split into blocks m₁, m₂, ..., mₙ.

  Step 3: Iteratively apply a COMPRESSION FUNCTION F:

         m₁          m₂          m₃             mₙ
         │           │           │              │
         ▼           ▼           ▼              ▼
   IV ─►[F]──H₁────►[F]──H₂────►[F]──...──Hₙ₋₁─►[F]──► H(m)

  Each F takes:
       - the previous "chaining variable" Hᵢ₋₁ (a fixed-size state, 256 bits for SHA-256)
       - the current message block mᵢ
  And outputs a new chaining variable Hᵢ.

  The IV is a standardized constant defined by the algorithm spec.
  The final Hₙ IS the hash output.

  Why iterate? Lets us hash ARBITRARILY LONG messages with a fixed-size state.

Hash security levels — collision via birthday attack 📄 pp. 162, 200

Naively you might expect collision-finding to take 2n tries for an n-bit hash. But it actually takes only 2n/2 tries — the famous birthday attack.

Why "birthday"? In a room of 23 people, there's >50% chance two share a birthday. That's much smaller than 365 because we don't need a specific collision (one matching MY birthday) — we just need ANY collision (any pair sharing). Same idea applies to hashes: in a set of 2n/2 random hash outputs, we expect a collision somewhere. Concrete numbers:
  • SHA-256: 256-bit output → 2128 work to find a collision.
  • SHA-512: 512-bit output → 2256 work to find a collision.
  • SHA-1 (deprecated): 160-bit output → 280 work — Google demonstrated a real collision in 2017.
Mnemonic: "Collision resistance = HALF the digest bits." So a 256-bit hash provides 128-bit security, matching AES-128. 📄 p. 162

2dMessage Authentication — MAC, HMAC & AES-GCM 📄 pp. 163-181 · Lecture 6

Why we need MACs in addition to encryption 📄 pp. 170, 172

Here's a trap many students fall into: "I encrypted my message, so I'm secure." Wrong. Encryption only gives you confidentiality — secrecy. It says nothing about whether the message was tampered with in transit.

Take a concrete example. Suppose Alice sends Bob "Pay $100" encrypted with AES-CTR mode. The ciphertext is plaintext XORed with the keystream. An attacker can't read it (no key)... but they CAN flip bits. 

  THE TAMPERING-WITHOUT-DECRYPTION ATTACK         📄 p. 170

  Alice's plaintext:     "Pay $100"  =  50 61 79 20 24 31 30 30  (in ASCII hex)
                                          (just for example)

  Keystream (KS):                       95 32 6A 7B B7 12 5C 3D
                                        (whatever AES-CTR produced)

  Ciphertext (Alice→Bob): 50 ⊕ 95 = C5
                          61 ⊕ 32 = 53
                          79 ⊕ 6A = 13
                          20 ⊕ 7B = 5B
                          24 ⊕ B7 = 93
                          31 ⊕ 12 = 23
                          30 ⊕ 5C = 6C
                          30 ⊕ 3D = 0D

  Attacker intercepts ciphertext.  Cannot decrypt, but can FLIP BITS.

  Suppose the attacker wants to change the amount from $100 to $900:
        '1' is ASCII 0x31  =  00110001
        '9' is ASCII 0x39  =  00111001
        diff:                 00001000   (just flip bit 3)

  Attacker XORs ciphertext byte 5 with 00001000:
        original C[5] = 0x23 = 00100011
        modified C[5] = 0x2B = 00101011

  Bob receives modified ciphertext. Decrypts:
        modified C[5] ⊕ KS[5] = 0x2B ⊕ 0x12 = 0x39 = '9'
                                                       ↑
                                                The attacker just changed
                                                "Pay $100" to "Pay $900",
                                                and Bob has no way to know.

  → Encryption alone does NOT prevent tampering!  We need a MAC.

Integrity vs Authenticity — a critical distinction 📄 p. 172

PropertyProvided byWhat it guaranteesWhat it does NOT guarantee

The MAC protocol in action 📄 p. 174

  MAC PROTOCOL                                    📄 p. 174

  Alice                                       Bob
  (knows shared key K)                        (knows shared key K)
  ──────────────────────                      ──────────────────────
  message m
  Compute t = MAC(K, m)
                       ──── send (m, t) ────►
                                              Compute t' = MAC(K, m)
                                              Compare t' with t ?
                                              ─── if equal: ACCEPT
                                              ─── else: REJECT

  Properties:
    - Anyone without K cannot compute a valid MAC for any message.
    - Anyone with K can verify by recomputing.
    - The MAC is deterministic — the same (K, m) always gives the same t.

HMAC — the standard MAC, built from any hash 📄 p. 178

HMAC is a specific recipe for turning any hash function into a MAC. It uses two padding constants (opad and ipad) and the secret key: 

  HMAC RECIPE                                     📄 p. 178

  HMAC(k, m) = H( (k ⊕ opad) ‖ H( (k ⊕ ipad) ‖ m ) )

  Where:
    k     = the secret key
    m     = the message
    H     = any hash function (typically SHA-256 → "HMAC-SHA-256")
    opad  = the byte 0x5C repeated to fill a hash block
    ipad  = the byte 0x36 repeated to fill a hash block
    ‖     = concatenation

  In words: pad the key two different ways, hash one of them with the message,
  then hash that result with the other padded key.

  The double-hash structure prevents "length-extension attacks" that would
  affect a naive "just hash key‖message" scheme.

HMAC-SHA-256 is the default MAC in TLS, IPsec, JWT, and dozens of other protocols. When you see "HMAC" mentioned in any standards document, it's almost always HMAC-SHA-256.

Encrypt-then-MAC — the right way to combine encryption with integrity 📄 p. 180

When combining encryption and a MAC, the order matters surprisingly much. The cryptographic community has converged on:

Encrypt-then-MAC (ETM) Step 1. Encrypt plaintext: c = AES_K1(m).
Step 2. Compute MAC over the ciphertext: t = HMAC_K2(c).
Step 3. Send (c, t).

Receiver verifies the MAC before attempting decryption. If the MAC is bad, decrypt isn't even attempted — defending against chosen-ciphertext attacks. Use two separate keys for encryption and MAC. 📄 p. 180

The alternative orderings — MAC-then-Encrypt and Encrypt-and-MAC — have known weaknesses in some scenarios. Modern protocols always use Encrypt-then-MAC (or use AEAD modes like AES-GCM that bundle both into one operation).

AES-GCM — encryption and authentication in one pass 📄 p. 181

AES-GCM is the modern industry default. It's an AEAD mode — Authenticated Encryption with Associated Data — meaning it provides confidentiality + integrity + authenticity in a single algorithm pass. 

  AES-GCM (Galois/Counter Mode)                   📄 p. 181

  Inputs:
    K       = AES key
    IV      = nonce (must be unique per (K, message); typically 96 bits)
    P       = plaintext
    AAD     = "associated data" — header info that should be authenticated
              but NOT encrypted (e.g., the packet's TCP header)

  Two parallel branches:

  ┌───── BRANCH 1: AES-CTR encryption ─────┐    ┌───── BRANCH 2: GHASH authentication ─────┐
  │                                          │    │                                            │
  │  Generate keystream by encrypting:       │    │  Compute auth tag T over AAD + ciphertext  │
  │     KS_i = AES_K(IV ‖ counter_i)         │    │  using a finite-field hash function GHASH  │
  │                                          │    │  combined with another AES encryption.     │
  │  Encrypt: C_i = P_i ⊕ KS_i               │    │                                            │
  │                                          │    │  Output: 128-bit auth tag T                │
  └──────────────────────────────────────────┘    └────────────────────────────────────────────┘

3.2 — Common Misconceptions & Traps

❌ MYTH: "ECB mode is secure as long as the plaintext is short."

✅ REALITY: ECB is insecure whenever two plaintext blocks are identical, because they encrypt to identical ciphertext blocks. For structured data (images, documents), this pattern leaks across blocks — the "Tux penguin" image example on slide 72 is canonical. ECB is only "safe" for a single block of true random data, which essentially never happens in practice. 📄 pp. 70-72

🎯 EXAM TRAP: T/F: "ECB mode is secure for messages ≤ 128 bits." — technically defensible but the instructor treats ECB as broadly insecure.

❌ MYTH: "A hash function can be 'decrypted' if you have the key."

✅ REALITY: Hashes have no key, and they are not invertible under any circumstance. This is the key distinction from asymmetric encryption. Instructor verbatim: "the difference between asymmetric cipher and secure hash is asymmetric cipher is invertible based on the correct keys... but the secure hash is not invertible because there is no way to recover the plaintext from the hash value — it is a long compression operation." 📄 p. 157

🎯 EXAM TRAP: MCQ offering "hash decryption" as an option.

❌ MYTH: "MAC and digital signature are the same thing."

✅ REALITY: Both provide integrity + authenticity, but only digital signatures provide non-repudiation. With a MAC, both sender and receiver share the key, so the receiver could forge any message and claim the sender wrote it. With a signature, only the sender holds the private key, so authorship is undeniable. 📄 pp. 103, 172

🎯 EXAM TRAP: "Which primitive is required for legal non-repudiation?" — the answer is digital signature, not MAC.

❌ MYTH: "RSA and ECC provide the same security level at the same key size."

✅ REALITY: ECC is dramatically more efficient. For 128-bit security: RSA needs 3072-bit keys, ECC needs only 256-bit keys. This is why modern protocols (TLS 1.3, SSH, WPA3) use ECDHE rather than DHE or RSA. 📄 p. 195

🎯 EXAM TRAP: Key-size matching tables that swap the RSA and ECC numbers.

❌ MYTH: "Counter mode adds integrity because it uses sequence counters."

✅ REALITY: CTR mode provides confidentiality only. The counter is for making each block's keystream unique, not for integrity. An attacker can flip any bit of CTR-mode ciphertext and the corresponding plaintext bit flips after decryption — with no detection. To add integrity you must either (a) run a separate MAC (Encrypt-then-MAC) or (b) use an AEAD mode like AES-GCM that bundles both. 📄 pp. 80, 170, 181

🎯 EXAM TRAP: Multi-answer: "Which AES modes provide integrity?" — CTR should NOT be selected.

❌ MYTH: "It's fine to reuse a nonce as long as you keep the key secret."

✅ REALITY: Reusing a (key, nonce) pair in CTR or GCM mode is catastrophic. Two ciphertexts C₁ = P₁ ⊕ KS and C₂ = P₂ ⊕ KS with the same keystream KS allow the attacker to compute C₁ ⊕ C₂ = P₁ ⊕ P₂, reducing the problem to known-plaintext cryptanalysis. GCM in particular also loses its authentication guarantees on nonce reuse. This is exactly how WEP was broken in 2001. 📄 pp. 80, 181

🎯 EXAM TRAP: Instructor said she will directly ask: "Can we reuse nonces? What's the problem?"

WC-1. The XOR operation has the property that (A ⊕ B) ⊕ B = A. Why is this property essential for the One-Time Pad to work?

▶ Show Answer

📝 Answer: Because encryption and decryption use the SAME operation — XOR with the key. To encrypt, Alice computes C = P ⊕ K. To decrypt, Bob computes C ⊕ K. By the property above, C ⊕ K = (P ⊕ K) ⊕ K = P, recovering the original. If XOR didn't have this self-inverting property, sender and receiver would need different operations, and the simplicity of the OTP would disappear. 📄 p. 60

WARM-UP Conceptual 📄 p. 71-72

WC-2. Suppose you encrypt the plaintext message "AAAAAAAAAAAAAAAA" (16 A's, exactly one AES block) three times in a row using AES-ECB mode with the same key. What does the ciphertext look like? Why is this a problem?

▶ Show Answer

📝 Answer: The ciphertext is the same 16-byte block repeated three times. Because ECB encrypts each block independently with the same key, identical plaintext blocks always produce identical ciphertext blocks. An eavesdropper who sees the ciphertext — even with no key — can immediately tell that those three regions had the same plaintext. For images and structured data, this leaks the structure dramatically (the famous Tux penguin example on slide 72). The fix: use CBC or CTR, which mix in a per-block IV/counter so identical plaintext maps to different ciphertext. 📄 pp. 71-72

WARM-UP Numerical 📄 p. 89

WC-3. In the toy RSA example with p=61, q=53, n=3233, e=17, d=2753: if Alice encrypts the message M = 123, what ciphertext does she compute and send? And what does Bob compute to recover the original?

▶ Show Answer

📝 Answer:

  • Alice computes C = M^e mod n = 123^17 mod 3233 = 855. She sends 855.
  • Bob computes M = C^d mod n = 855^2753 mod 3233 = 123. He recovers the original.

Notice: the same mathematical operation (modular exponentiation) is used both ways — the only difference is which exponent (e or d) is applied. 📄 p. 89

WARM-UP Conceptual 📄 pp. 157, 87

WC-4. Bob hands you a SHA-256 hash: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824. He says "I have the secret key — can you give me back the original message that produced this hash?" What's wrong with his question?

▶ Show Answer

📝 Answer: Hash functions don't have keys and they are not invertible under any circumstances. There is no "decrypt" operation for SHA-256 even with a master key. The output is shorter than typical inputs (a 1MB file compresses to 32 bytes), so by pigeonhole many inputs produce the same hash — information is fundamentally destroyed. Bob is confusing a hash with an asymmetric cipher (which IS invertible with the right key). The instructor explicitly emphasized this point: "asymmetric cipher is invertible based on the correct keys, but the secure hash is not invertible." 📄 p. 157

The best Bob could do is brute-force: try inputs one-by-one and hash each, hoping to match. For SHA-256 that's 2256 tries — infeasible.

WARM-UP Conceptual 📄 pp. 80, 181

WC-5. In CTR mode, suppose Alice encrypts the messages P₁ = "Pay $100" and P₂ = "Pay $999" with the same key K and accidentally the SAME nonce N. The attacker captures both ciphertexts C₁ and C₂. What can they compute, and why is this a catastrophic security failure?

▶ Show Answer

📝 Answer: Both encryptions used the same keystream KS = AES_K(N‖counter). So:

  • C₁ = P₁ ⊕ KS
  • C₂ = P₂ ⊕ KS

The attacker XORs the two ciphertexts together:

C₁ ⊕ C₂ = (P₁ ⊕ KS) ⊕ (P₂ ⊕ KS) = P₁ ⊕ P₂

The keystream cancels out (KS ⊕ KS = 0). The attacker now has the XOR of the two plaintexts directly — no key needed. With even a small amount of known structure (English text, common headers, etc.), they can recover both plaintexts via statistical analysis. This is exactly the failure mode that broke WEP in 2001: 24-bit IVs guaranteed nonce reuse within hours on a busy network. The cardinal CTR-mode rule: NEVER reuse a (key, nonce) pair. 📄 pp. 80, 181

Part B — True/False (3 questions: 1 BEG, 1 INT, 1 ADV)

Q5. Statement: Given a SHA-256 hash value h, someone with enough computing power and the right secret key can "decrypt" h back to the original message.

▶ Show Answer

✅ Answer: FALSE.

Hash functions are not invertible under any circumstance — there is no key, and there is no inverse algorithm. A hash compresses an arbitrary-length input to a fixed-length digest, destroying information. The instructor drew this contrast explicitly: "the difference between asymmetric cipher and secure hash is, asymmetric cipher is invertible based on the correct keys, but the secure hash is not invertible — there is no way to recover the plaintext from the hash value, because it is a long compression operation." INT TF 📄 p. 80

Q6. Statement: Because AES-CTR mode uses an incrementing counter that never repeats within a message, it provides built-in integrity protection against bit-flip attacks on the ciphertext.

▶ Show Answer

✅ Answer: FALSE.

The counter in CTR mode exists to vary the keystream across blocks and prevent keystream reuse. It provides confidentiality only, not integrity. An attacker who flips any bit of a CTR-mode ciphertext causes the corresponding bit of the decrypted plaintext to flip — with no detection. The counter never "checks" anything on decryption. For integrity you need a separate MAC (Encrypt-then-MAC) or an AEAD mode like AES-GCM, which explicitly adds GHASH authentication on top of CTR-mode encryption. 📄 pp. 80, 170, 181

ADV · subtle TF 📄 pp. 162, 200

Q7. Statement: A 256-bit hash function (such as SHA-256) provides 256 bits of collision resistance, meaning an attacker needs roughly 2²⁵⁶ operations to find two inputs with the same hash value.

▶ Show Answer

✅ Answer: FALSE.

The birthday attack finds collisions in approximately 2^(n/2) operations for an n-bit digest, not 2ⁿ. For SHA-256, that is roughly 2¹²⁸ operations — strong, but half the naive expectation. Slide 162 states: "Brute-force inversion requires 2²⁵⁶ ops, birthday attack on collision resistance requires 2²⁵⁶ ops" — wait, actually that slide shows SHA-512, where 2²⁵⁶ is the correct collision bound because n = 512. For SHA-256, the collision bound is 2¹²⁸. General rule: collision resistance = half the digest bits; pre-image resistance = full digest bits. The instructor's exam mnemonic: SHA-256 → 128-bit collision resistance = matches AES-128. 📄 pp. 162, 200

Q9. Describe the Diffie-Hellman key-exchange protocol in 5 steps or fewer. What secret does an eavesdropper need to compute in order to break it, and why is that considered hard?

▶ Show Answer

📝 Model Answer:

  1. Alice and Bob publicly agree on a large prime p and a generator g.
  2. Alice picks a secret integer x_A; computes A = gx_A mod p; sends A to Bob.
  3. Bob picks a secret integer x_B; computes B = gx_B mod p; sends B to Alice.
  4. Alice computes shared secret K = Bx_A = gx_A · x_B mod p.
  5. Bob computes shared secret K = Ax_B = gx_A · x_B mod p — same value.

An eavesdropper sees p, g, A, B on the wire but needs to recover either x_A from A or x_B from B to derive K. This is the Discrete Logarithm Problem: given g and gx mod p, compute x. For sufficiently large p (≥ 2048 bits), no efficient classical algorithm is known. Elliptic-curve variants (ECDH) use the analogous Elliptic-Curve Discrete Logarithm Problem, which appears even harder for equivalent key sizes — letting 256-bit ECC match 3072-bit classical DH. 📄 pp. 97-98, 140, 195

🔑 Key Points (Grading Checklist):

  • ☐ Correct math: gx_A · x_B is the shared secret 📄 p. 98
  • ☐ Both sides compute the same K via commutativity of exponents 📄 p. 140
  • ☐ Names the Discrete Logarithm Problem as the security assumption 📄 p. 150
  • ☐ Notes that the attacker sees A and B but cannot derive x_A or x_B
  • ☐ Bonus: mentions ECDH as the elliptic-curve variant used in practice 📄 p. 101
ADV · challenge SA 📄 pp. 94, 103, 172  +  p. 30

Q10. Compare MAC and Digital Signature across three dimensions: (1) what cryptographic primitive they're built on, (2) what security services they provide, (3) why only one of them provides non-repudiation. Conclude with one sentence explaining when you would choose each in practice.

▶ Show Answer

📝 Model Answer:

(1) Primitive. A MAC is a symmetric-key construction — both sender and receiver share the same secret key k, used to compute MAC(k, m). HMAC builds it from a hash function; AES-GCM's tag is an AEAD MAC. A digital signature is an asymmetric construction — the sender holds a private key sk, the receiver (or any verifier) uses the corresponding public key pk. Classic signing formula: s = hash(m)d mod n in RSA. 📄 pp. 94, 103, 172, 177

(2) Security services. Both provide integrity (message not altered) and authenticity (message came from a key-holder). The difference is in which key-holder: MAC verifies that some party with the shared key made the tag — either sender or receiver, indistinguishably. Signature verifies specifically that the private-key holder made it, which is unambiguously one party. 📄 pp. 103, 172

(3) Non-repudiation. With a MAC, Bob cannot convince a third party that Alice authored a message, because Bob also holds the key and could have forged the tag himself. With a signature, only Alice has her private key, so any valid signature demonstrably binds Alice to the message — she cannot later deny it. 📄 pp. 30, 103

Choosing in practice. Use MAC (especially AES-GCM or HMAC) for fast per-packet integrity on established sessions where both parties already trust each other; use a digital signature when you need legal non-repudiation or when the verifier is not the same party as the sender (e.g., code signing, certificates, software updates).

🔑 Key Points (Grading Checklist):

  • ☐ MAC uses symmetric / shared key; signature uses asymmetric / private key 📄 pp. 172, 103
  • ☐ Both give integrity + authenticity; only signature gives non-repudiation 📄 pp. 30, 103
  • ☐ Correctly explains why MAC lacks non-repudiation (shared key) 📄 p. 172
  • ☐ Concrete example of each use-case in the closing sentence

INT FIB 📄 pp. 162, 200

Q12. For a hash function with an n-bit digest, the expected cost to find a collision via the birthday attack is 2^(____________) operations, whereas the cost of a brute-force pre-image attack is 2^(____________) operations. Concretely, SHA-256 therefore provides ____________ bits of collision resistance.

▶ Show Answer

✅ Answers: (1) n / 2; (2) n; (3) 128 bits.

📝 Why: The birthday bound states that among 2^(n/2) random hashes there's a ≥50% chance of collision. Slide 162 matches this for SHA-512: "birthday attack on collision resistance requires 2²⁵⁶ ops" (= 2^(512/2)). Applying the same formula to SHA-256 gives 2^128 collision cost. Pre-image attacks on a 256-bit digest require ~2²⁵⁶ ops. 📄 pp. 162, 200

ADV FIB 📄 pp. 101, 122-123

Q13. When a TLS handshake uses ____________ — the ephemeral variant of Elliptic-Curve Diffie-Hellman — the resulting session key has the property of ____________, meaning that even if the server's long-term private key is later compromised, an adversary with recorded past traffic ____________ retroactively decrypt those past sessions.

▶ Show Answer

✅ Answers: (1) ECDHE (Elliptic-Curve Diffie-Hellman Ephemeral); (2) Perfect Forward Secrecy (PFS); (3) cannot.

📝 Why: Slide 101 defines ECDHE as "the last E represents Ephemeral mode" and names the property as "Perfect Forward Secrecy (PFS)." Because the session uses ephemeral keys that are discarded immediately after the handshake, no stored long-term key is sufficient to recover the session secret post-hoc. This is the architectural reason TLS 1.3 removed RSA key transport in favor of (EC)DHE-only key exchange. 📄 pp. 101, 122-123

§5 — Topic Summary · Exam Cheat Sheet

📘 LECTURE PDF COVERAGE: pp. 54-206 (condensed view of the entire foundations layer)

IN ONE SENTENCE, THIS TOPIC IS:

Four cryptographic primitives — symmetric ciphers (fast bulk encryption), asymmetric ciphers (key exchange + signatures), hashes (one-way fingerprints), and MACs (keyed integrity+authenticity) — composed in a hybrid pattern (asymmetric-for-setup + symmetric-for-bulk + MAC-for-integrity) that underlies every secure protocol on this course. 📄 pp. 55, 115, 137, 150, 181

THE THREE THINGS YOU MUST KNOW FOR EVERY PRIMITIVE:

  1. What service it provides — confidentiality (sym/asym), integrity (hash/MAC), authenticity (MAC/signature), non-repudiation (signature only). 📄 pp. 30, 103, 172
  2. Its key model — one shared key (sym/MAC), keypair (asym/signature), no key (hash). This is what determines whether non-repudiation is achievable.
  3. Its representative algorithm & security level — AES-128 → 128-bit; AES-256 → 256-bit; RSA-3072 → 128-bit; ECC-256 → 128-bit; SHA-256 → 128-bit collision resistance. 📄 pp. 162, 190, 195, 200

THE #1 MISTAKE STUDENTS MAKE:

Confusing MACs with digital signatures. They share a shape (tag appended to message, verifier checks) but the key model is fundamentally different — shared-key vs keypair — and that difference determines the single security property that courts care about: non-repudiation. A MAC never proves authorship to a third party because either party could have made it. Only a private-key signature does. 📄 pp. 103, 172
