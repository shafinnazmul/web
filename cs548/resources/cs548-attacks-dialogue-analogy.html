               "HELLO EVERYONE! This is 42 Elm St speaking!
                Please — ALL of you — reply to me immediately!"
              [REAL: ICMP Echo Request (Type=8, Code=0)

Neighbor 2  : Me too! [sends reply to 42 Elm St]
...
Neighbor 500: [sends reply to 42 Elm St]

Neighbor 1  : [genuinely confused]

Ping of Death — The Oversized Package (Annotated)

Attacker    : [at a shipping counter, grinning]
               I'd like to send a package to Server please.
              [REAL: preparing to send a malformed IP datagram
                     exploiting the fragmentation/reassembly
                     mechanism defined in IP RFC 791]

Courier     : Sure! Standard package size limit is 65,535 grams.
              [REAL: maximum legal IP datagram size = 65,535 bytes
                     (16-bit Total Length field in IP header
                     can represent values 0–65535).
                     ICMP adds 8 bytes of header,
                     so max ICMP payload = 65,535 - 20(IP) - 8(ICMP)
                     = 65,507 bytes]

Attacker    : Great. I'd like to send a 65,538 gram package.
              [REAL: attacker crafts fragments whose
                     Fragment Offset + Fragment Size
                     adds up to MORE than 65,535 bytes.
                     e.g., last fragment has:
                     Offset = 65,120 bytes
                     Size   =    398 bytes
                     Total  = 65,518 + 20 (IP header) = 65,538 bytes
                     → EXCEEDS the 65,535 maximum!]
               But here's the thing —
               I'll split it into small parcels for shipping.
              [REAL: IP fragmentation splits large datagrams
                     into smaller fragments for transmission.
                     Each fragment has:
                     - Fragment Offset field (in 8-byte units)
                     - More Fragments (MF) flag
                     - Same Identification field (to reassemble)
                     Individual fragments look completely normal —
                     the problem only appears at reassembly]
               The label on the last parcel will say
               "total weight when reassembled: 65,538 grams."
              [REAL: the Fragment Offset on the last fragment
                     is set so that:
                     Offset × 8 + fragment_size > 65,535
                     Each individual fragment passes inspection —
                     only the reassembled total is illegal]

Courier     : Uh... okay. Rules say I just forward parcels.
               Reassembly is the recipient's problem.
               [forwards all parcels to Server]
              [REAL: routers forward fragments individually
                     without reassembling them.
                     Reassembly happens ONLY at the destination.
                     This is by IP design — routers don't
                     have resources to reassemble every datagram]

Server      : [starts reassembling parcels]
              [REAL: OS kernel allocates a static buffer of
                     65,535 bytes to hold the reassembled datagram.
                     It uses the Fragment Offset field to place
                     each fragment at the correct position]
               Parcel 1... Parcel 2... Parcel 3...
               [gets to the last parcel]
               Wait. The label says total weight is 65,538?
               But my storage room only fits 65,535!
               I don't have a plan for this situation!
              [REAL: the last fragment's data writes BEYOND
                     the end of the allocated 65,535 byte buffer.
                     This is a classic buffer overflow.
                     On unpatched systems (pre-1998):
                     → kernel panic (crash)
                     → system freeze
                     → arbitrary code execution in worst case]
               [CRASHES — storage room overflows]
              [REAL: this vulnerability affected Windows 95/NT,
                     Linux, macOS, and others before 1998.
                     Patched by adding a check:
                     if (offset + size > 65535) → drop fragment]

Attacker    : [to himself] The courier checked nothing.
               Each individual parcel looked perfectly normal.
               The bomb was hidden in the reassembly.
              [REAL: this is the key insight of fragmentation attacks —
                     individual fragments are valid and pass filters.
                     The exploit lives in their COMBINATION,
                     which only the destination sees]

Courier     : [shrugs] Each parcel was under the limit!
               Not my problem what happens when they stack up.
              [REAL: stateless firewalls had same blind spot —
                     they checked individual fragments, not totals.
                     Stateful firewalls that track and reassemble
                     fragments can catch this attack]

                      Code 1 = Host Unreachable

Restaurant  : [this is the Server]
              [REAL: server maintains a "backlog queue" —
                     a fixed-size data structure storing
                     TCB (Transmission Control Block) records
                     for each half-open connection.
                     Default size: typically 128-1024 entries
                     depending on OS configuration]

Attacker    : [calls the restaurant]
              [REAL: sends TCP SYN packet to server:
                     IP(src="9.9.9.1",        ← random spoofed IP
                        dst="SERVER_IP")
                     /TCP(sport=random,
                          dport=80,
                          flags="S")          ← SYN flag set

                     or unresponsive IP address]

               [gives fake callback number: 9.9.9.1]
              [REAL: spoofed source IP = 9.9.9.1

                     but that machine doesn't exist or doesn't respond]

Restaurant  : "Wonderful! I've noted your reservation.
               I'll call you back to confirm!"
              [REAL: server receives SYN, allocates TCB entry,
                     stores in backlog queue (half-open state),

               [writes on waiting list — slot 1 used]
               [calls 9.9.9.1 — no answer, keeps slot open]

Attacker    : [keeps calling with fake numbers]
               9.9.9.3... 9.9.9.4... 9.9.9.5...
              [REAL: attacker sends thousands of SYN packets

                     Rate can reach millions of packets/second
                     using tools like hping3:
                     hping3 -S --flood -V -p 80 SERVER_IP]

Restaurant  : [waiting list now completely full — all 10 slots]
              [REAL: backlog queue is completely full of
                     half-open TCB entries waiting for ACKs
                     that will never arrive.
                     New legitimate SYN packets are DROPPED —
                     server cannot accept new connections at all]

Real Customer: [calls to make a reservation]
               "Hi! Table for two tonight please?"
              [REAL: legitimate client sends TCP SYN to server]

Restaurant  : I'm so sorry — our waiting list is completely full!
               We cannot take any new reservations right now!
              [REAL: server's backlog queue is full,
                     legitimate SYN is silently dropped.
                     Client sees "Connection timed out" error.
                     Website appears completely down — DoS achieved]

Real Customer: But I can see the restaurant is empty!
              [REAL: server's CPU, bandwidth, and application
                     layer are all fine — only the TCP backlog
                     queue is exhausted. A very efficient DoS:

                     bandwidth by exploiting a small data structure]

SYN Cookie  : [the smart maitre d' who solves this]
               New policy! I write the reservation details
               on the callback number itself.
              [REAL: SYN Cookie algorithm (RFC 4987):

                     server encodes connection info INTO the

               No waiting list entry until the customer
               actually calls back to confirm!
              [REAL: if legitimate client sends ACK with
                     ACK number = cookie + 1,
                     server verifies the math, reconstructs TCB,

                     If no ACK comes (phantom SYN) — 
                     no memory was used at all!
                     Backlog queue never fills up.
                     Enabled in Linux:

               exact position so Server can reassemble them.
              [REAL: IP fragmentation uses a 13-bit

Normal mail :
Attacker    : [labels pieces correctly]
               Piece 1: covers positions 0-9
               Piece 2: covers positions 10-19
              [REAL: normal fragmentation:
                     Fragment 1: Offset=0,   Length=1480 bytes (MF=1)
                     Fragment 2: Offset=1480, Length=1480 bytes (MF=1)
                     Fragment 3: Offset=2960, Length=500  bytes (MF=0)
                     Each fragment starts exactly where the
                     previous one ended. No gaps. No overlaps.]

Server      : [receives all pieces, assembles perfectly]
               "Beautiful puzzle. No problems."

Attacker    : [deliberately mislabels pieces]
              [REAL: crafts fragments with invalid offsets:
                     send(IP(dst="SERVER_IP", id=1234,
                              flags="MF")         ← More Fragments

                     send(IP(dst="SERVER_IP", id=1234,
                              frag=100,            ← Offset=100*8=800
                              flags="MF")          ← starts at byte 800!
                          /Raw(b"B"*1480))         ← Fragment 2: bytes 800-2279
                                                   ← OVERLAP: bytes 800-1479
                                                      already covered by Frag 1!

                                                      doesn't make sense!]

               Piece 1: covers positions 0-99   ✓ normal
              [REAL: Fragment 1 covers bytes 0–1479. Looks fine.]

                     → kernel panic (Blue Screen of Death on Windows)
                     → system hang or reboot
                     Fixed by: checking offset+length <= 65535
                     before writing each fragment to buffer]

Attacker    : [to himself]
               Overlapping offsets + gaps = Server meltdown.
              [REAL: this was a ZERO-COST attack — tiny packets,
                     no bandwidth needed, instant crash.
                     Patch: validate fragment offsets on arrival.
                     Modern OS kernels all have this fix.
                     Stateful firewalls can also catch it by
                     tracking and validating fragment streams]

The Attack:

                     send(ARP(op=2,                    ← ARP Reply

                              psrc="2.2.2.2",          ← "I am Server"
                              hwsrc="EE:FF:00:11"))    ← Attacker's MAC

                     No request needed. No verification. By design.]

               [writes] "House 1.1.1.1 — doorbell is EE:FF:00:11"
              [REAL: Server's ARP cache now says:
                     1.1.1.1 → EE:FF:00:11 (Attacker's MAC)
                     So ALL frames Server sends to "Client"
                     physically arrive at Attacker's NIC]
               [also writes] "House 2.2.2.2 — doorbell is EE:FF:00:11"
              [REAL: Client's ARP cache now says:
                     2.2.2.2 → EE:FF:00:11 (Attacker's MAC)
                     So ALL frames Client sends to "Server"
