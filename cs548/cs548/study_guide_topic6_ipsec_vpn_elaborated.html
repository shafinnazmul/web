The core question of Topic 6: How do you protect IP packets — at the network layer, transparent to applications — when the Internet they cross is full of eavesdroppers, spoofers, and hijackers? IPsec is the IETF's answer. VPNs are how IPsec (and similar tools) get used in practice. By the end of this document, the words "AH," "ESP," "tunnel mode," "transport mode," and "VPN" will all click into a coherent picture, not feel like random acronyms.

🧠 What you should already know (a 5-minute refresh)

This section reminds you of four ideas you'll lean on heavily. If any of them feel unfamiliar, pause and review the relevant earlier topic before continuing.

1. The TCP/IP layering, and where each protocol lives

The Internet uses a layered protocol stack. Bottom to top: physical wires → link (Ethernet, Wi-Fi) → network (IP) → transport (TCP, UDP) → application (HTTP, SSH, DNS). Each layer wraps the layer above it in a header.

When your browser sends an HTTP request, this is what's actually on the wire (simplified):

   [ Ethernet hdr | IP hdr | TCP hdr | HTTP request ]
                    │         │         │
                    │         │         └── written by your browser
                    │         └──────────── written by your kernel's TCP stack
                    └────────────────────── written by your kernel's IP stack

Each layer's header has its own purpose. The IP header has source and destination IP addresses; routers read it to forward the packet. The TCP header has source and destination ports plus sequence numbers; the receiver's kernel uses it to deliver bytes in order to the right application.

IPsec inserts itself between the IP layer and whatever's above it. From the application's perspective, nothing changes — TCP still works the same. From the wire's perspective, there's now a new "IPsec header" between the IP header and the TCP header (or, in tunnel mode, a whole second IP header underneath everything). We'll see exactly how shortly.

2. Symmetric encryption (recap from Topic 3)

Two parties who share a secret key can encrypt + decrypt with that key using AES. Modern AES-GCM does encryption and integrity (a MAC) in one operation — it's an AEAD cipher (Authenticated Encryption with Associated Data). IPsec uses AES-GCM in modern deployments for the same reason TLS does: one primitive, both services.

3. Public-key crypto and Diffie-Hellman (recap from Topic 3)

Two parties who have never met can establish a shared secret over a public channel using Diffie-Hellman. The shared secret then becomes input to a key-derivation function that produces symmetric encryption keys. IPsec uses Diffie-Hellman the same way TLS and SSH do; the only difference is that IPsec wraps the DH exchange in its own protocol, called IKE (Internet Key Exchange).

4. The TCP/IP attacks from Topic 2

If you remember nothing else from Topic 2, remember this: plain IP, plain TCP, and plain ARP are unauthenticated. An attacker with access to the network path can:

  • Read every byte (eavesdropping).
  • Spoof the source IP address of packets they send.
  • Inject TCP RSTs (resets) to tear down connections.
  • Inject TCP data into existing connections by guessing sequence numbers (hijacking).
  • Poison ARP caches on a LAN, redirecting traffic through their own machine.

These attacks all exist because the protocols themselves provide no cryptographic protection. Each one assumes everyone on the network is honest. Upper-layer protocols like TLS and SSH protect specific applications against most of these attacks. IPsec is the network-layer answer that protects all traffic at once, transparently.

Wait, then why do we need TLS at all if IPsec exists? Two reasons. First, IPsec needs to be configured and supported on both endpoints, which historically meant kernel-level setup and IT involvement; TLS just needs an up-to-date browser and a certificate on the server. Second, IPsec usually terminates at gateways (VPN concentrators), not at the actual application server — so the traffic between the gateway and the server is no longer encrypted. End-to-end encryption between your browser and the server still requires TLS. We'll see exactly how this plays out in 6C.

🌍 The big picture: where Topic 6 fits

Topic 6 has three sub-topics that are all about protecting traffic at the IP layer or below the application layer:

6A · IPsec architecture + AH + ESP IPsec is a framework — not a single protocol — for adding cryptographic protection to IP packets. Its two main packet-protection protocols are AH (Authentication Header — integrity + auth, no encryption) and ESP (Encapsulating Security Payload — integrity + auth + encryption). Plus a key-management protocol called IKE that sets up the keys.
6B · Transport mode vs Tunnel mode IPsec can run in two modes: transport (insert AH/ESP between the existing IP header and its payload — original IP addresses still visible) or tunnel (wrap the entire original packet inside a new outer IP header — original IP addresses hidden, only outer addresses visible). Tunnel mode is what makes site-to-site VPNs possible.
6C · VPN A Virtual Private Network is an encrypted tunnel that makes a remote client appear "logically attached" to a private network. Two main protocol families: IPsec-based (using ESP in tunnel mode) and TLS-based (OpenVPN, Cisco AnyConnect, WireGuard). The choice is mostly about deployment context.
The unifying mental model
Every Topic 6 concept can be answered by asking three questions:
  1. Who's involved at each end of the secure connection? Two hosts? Or a host and a gateway? Or two gateways?
  2. What protection do they need? Just integrity (use AH)? Confidentiality too (use ESP)?
  3. Should the original IP addresses be visible or hidden? Visible → transport mode. Hidden → tunnel mode.

Hold this mental model as you read. You'll find every "why" question lands in one of these three buckets.

The running example for this whole document

To keep things concrete, we'll follow a single scenario throughout. Acme Corp has two offices:

  • Branch A — San Francisco. Internal network 10.1.0.0/24. Edge router/firewall has public IP 198.51.100.1.
  • Branch B — Berlin. Internal network 10.2.0.0/24. Edge router/firewall has public IP 203.0.113.5.

Inside Branch A, an employee named Alice (laptop at 10.1.0.42) needs to access a file server in Branch B (at 10.2.0.10). The two offices are connected by the public Internet, which means anything they send between them is up for grabs unless they protect it.

The IT teams configure the two edge routers to run IPsec between them, and any traffic from 10.1.0.0/24 destined for 10.2.0.0/24 (or the reverse) automatically gets wrapped in IPsec ESP tunnel-mode and shipped to the other side. From Alice's perspective, she's just connecting to a file server. The whole encrypted-tunnel machinery is invisible — that's the point.

The story: what problem does IPsec actually solve?

Picture the Internet in the late 1980s. The protocols (IP, TCP, ICMP) had been invented for a small, trusted research network. Everyone on the network knew everyone else; "security" meant locking the computer-room door. Then the Internet went commercial, the user base exploded, and suddenly the unprotected protocols were carrying credit cards and corporate secrets across an Internet full of strangers.

By the early 1990s, the IETF (the body that standardises Internet protocols) realised that every single Layer-3 attack from Topic 2 — IP spoofing, TCP reset, hijacking, ICMP redirect — exists because IP packets have no cryptographic protection. A packet's source address is whatever the sender wrote in it; nothing verifies the sender actually has that address. A packet's contents are whatever bytes the sender put in; nothing detects modification in transit. A passive observer can read every byte; an active attacker can forge or modify packets as they please.

Two reasonable responses to this problem:

  1. Fix it per-application. Add encryption to HTTP (resulting in HTTPS via TLS). Add encryption to remote login (SSH). Add encryption to email (SMTPS, IMAPS). This works, but every application has to be updated, and traffic-flow information (who's talking to whom, at what times) still leaks.
  2. Fix it at the IP layer once and for all. Add cryptographic protection directly to IP packets, so every application gets it for free without modification.

IPsec is approach (2). It's a set of standards, finalised across RFC 2401 (1998) and refined in RFC 4301 (2005), that adds confidentiality, integrity, authentication, and anti-replay to IP packets at the IP layer. The two big benefits, both spelled out on slide 370 of the lecture:

  • Transparent to applications. The kernel applies IPsec automatically. Browsers, email clients, custom apps — none of them need to be IPsec-aware. They just open ordinary TCP sockets, and the kernel quietly wraps the resulting IP packets in IPsec on the way out.
  • Can be deployed at gateways/firewalls. Because IPsec works on IP packets, you can run it at the edge of your network — protecting all traffic in or out — without touching the hosts inside. This is exactly the Branch A ↔ Branch B scenario from the big picture.
IPsec Internet Protocol Security. An IETF framework — not a single protocol — that adds cryptographic protection to IP packets at the network layer. It consists of several cooperating components: a key-management protocol (IKE), two protection protocols (AH and ESP), two operation modes (transport and tunnel), and a pair of databases that hold policy and state (SPD and SAD). All of these will be unpacked over the next few sections.

Now here's the part that trips beginners up: when people say "IPsec," they could mean any of four different things, depending on context.

  • The whole framework (the architecture as a system).
  • The AH protocol specifically (authentication only).
  • The ESP protocol specifically (encryption + authentication).
  • The IKE protocol for setting up keys.

When in doubt, treat "IPsec" as the framework, and call out which sub-component you mean. The exam will sometimes specifically ask "AH versus ESP" — a question that only makes sense if you remember they are two different protocols inside the IPsec framework.

Vocabulary, introduced one at a time

Don't try to memorise this all at once. Read each box and let the term register; we'll use these words constantly in the next sections.

Security Association (SA) An SA is the central concept in IPsec. Think of it as a one-way contract between two endpoints: "For traffic going from me to you, we'll use this encryption algorithm with this key, this MAC algorithm with that key, anti-replay enabled, mode = tunnel, lifetime = 8 hours."

Three things to remember:
  • An SA is simplex — one-way. Bidirectional communication needs two SAs (one for each direction).
  • An SA is uniquely identified by the triple (SPI, destination IP, AH-or-ESP). We'll define SPI in the next box.
  • An SA is the cryptographic context: it has the keys, the counters, the algorithms, the mode, the lifetime — everything needed to protect a packet.
Security Parameters Index (SPI) A 32-bit tag carried in every AH or ESP header. Its job is to tell the receiver: "I'm using the SA with this SPI." The receiver looks up the SA in its database, retrieves the keys, and processes the packet accordingly.

The SPI alone isn't globally unique — values 0 (reserved for local use) and 1–255 (reserved by IANA) are reserved, but otherwise different peers can pick the same SPI value independently. Uniqueness comes from the full triple (SPI + destination IP + AH/ESP). Two different SAs targeting the same destination IP can share an SPI only if they use different protocols (one AH, one ESP).
Security Association Database (SAD) A table inside the kernel listing every active SA. When a packet arrives, the kernel reads the SPI from the AH/ESP header, looks up the SA in the SAD, retrieves the keys, and decrypts/verifies. When a packet is being sent, the kernel finds the relevant SA, retrieves the keys, and encrypts/authenticates.

Each SAD entry holds: SPI · destination IP · protocol (AH/ESP) · cipher algorithm · cipher key · MAC algorithm · MAC key · sequence-number counter · anti-replay window state · mode (transport/tunnel) · lifetime.
Security Policy Database (SPD) A second table that decides whether a given packet should be IPsec-protected, and if so, with which SA. Each SPD entry has selectors (source IP, dest IP, protocol, ports) and an action: BYPASS (send in plaintext), DISCARD (drop), or PROTECT (use SA with such-and-such SPI).

The split is important: the SPD is policy, the SAD is state. The SPD says "should this be protected, and if so, by which SA?"; the SAD says "here are the actual keys and counters for that SA right now." On packet send, the kernel consults SPD first, then (if the action is PROTECT) the SAD.
IKE (Internet Key Exchange) The protocol that creates SAs. Before any IPsec-protected packets can flow, the two peers must run IKE to authenticate each other and negotiate the SA parameters (algorithms, keys, lifetime, mode). IKE is what runs Diffie-Hellman, what verifies certificates or pre-shared keys, what writes the resulting SAs into both peers' SADs.

IKE typically runs over UDP port 500. When NAT is involved, both IKE and the encrypted ESP packets get encapsulated in UDP port 4500 ("NAT-Traversal"). IKEv2 (RFC 7296) is the modern version; IKEv1 is older and being retired.
Authentication Header (AH) One of the two IPsec protection protocols. It's IP protocol number 51. AH provides:
  • Integrity — any tampering with the packet is detected.
  • Data-origin authentication — the packet came from someone who knows the SA's MAC key.
  • Optional anti-replay — replay attempts are detected.
AH does not encrypt. The payload remains readable on the wire. AH is rare in practice today (we'll see why); ESP-with-authentication delivers everything AH does plus encryption.
Encapsulating Security Payload (ESP) The other IPsec protection protocol — IP protocol number 50. ESP provides:
  • Encryption (confidentiality) of the payload.
  • Integrity + data-origin authentication via a MAC.
  • Optional anti-replay.
  • Limited traffic-flow confidentiality — but only in tunnel mode (we'll see why).
ESP is the workhorse of real-world IPsec. Almost every IPsec deployment uses ESP, not AH.
Integrity Check Value (ICV) The MAC tag at the end of an AH or ESP packet — typically HMAC-SHA-256 (classic) or GMAC (when AES-GCM is used as the AEAD cipher). The receiver recomputes the ICV from the packet and the SA's MAC key; if it matches the ICV in the packet, integrity holds.

AH's ICV covers immutable fields of the outer IP header plus the payload. ESP's ICV covers the ESP header and the (encrypted) payload, but not the outer IP header. This asymmetry is one of the biggest exam points in 6A.
Anti-replay window IPsec gives every packet within an SA a sequence number, increasing with each packet. The receiver tracks recent sequence numbers in a sliding window of width W (typically 32 or 64) and:
  • Accepts new sequence numbers above the highest seen so far.
  • Accepts numbers within the window only if they haven't already been seen.
  • Rejects anything below the window — too old to verify.
This bounds the receiver's memory while still detecting replays. We'll work through it visually in the anti-replay section.
Transport mode An IPsec mode where AH/ESP is inserted between the existing IP header and the payload. The original IP addresses remain visible. Used for end-to-end host-to-host protection. (We'll explore in 6B.)
Tunnel mode An IPsec mode where the entire original IP packet (header included) becomes the inner packet inside a new outer IP header. The original IP addresses are hidden inside the encrypted blob. Used for site-to-site VPNs. (We'll explore in 6B.)

So there are four moving parts:

  1. The SPD — the policy table. Decides if and how each packet gets protected.
  2. The SAD — the state table. Holds the live cryptographic context for each active SA.
  3. AH and ESP — the two protection protocols. Whichever the SA specifies is what gets applied to the packet.
  4. IKE — the key-management daemon. Creates and refreshes SAs by negotiating with peers.
Wait, why two databases instead of one? Because they answer different questions and change at different rates.

The SPD is set by an administrator: "all traffic from 10.1.0.0/24 to 10.2.0.0/24 should be IPsec-protected with our Branch-A-to-Branch-B SA." This rarely changes; it's policy.

The SAD is created and updated by IKE: "here's the actual current cipher key, here's the sequence counter, here's how many bytes have been transferred, here's when this SA expires." This changes constantly: every SA rekey, every packet sent, every SA expiry.

Separating policy from state is a clean design that lets administrators specify what they want without having to worry about the moving parts of how it gets implemented.

A packet's life through IPsec, beginning to end

Let's trace what happens when Alice's laptop in Branch A sends a single packet to the file server in Branch B. Assume IKE has already finished and two SAs exist (one for each direction). We'll focus on the outbound direction.

Phase 1 — Alice's application sends data Alice's file-sync app calls send() on a TCP socket. The destination is 10.2.0.10:445 (the file server). Alice's kernel TCP stack builds a TCP segment. Alice's kernel IP stack builds an IP packet with source 10.1.0.42 and destination 10.2.0.10. So far this is exactly normal networking — no IPsec involvement yet at Alice's laptop.
Phase 2 — Packet leaves Alice's laptop, arrives at the edge router The packet routes out of Alice's laptop, across the Branch A LAN, and arrives at the edge router (198.51.100.1 on its public side, 10.1.0.1 on its internal side). The router will be the IPsec endpoint, not Alice's laptop. This is a normal "gateway-to-gateway" topology.
Phase 3 — Edge router consults SPD The edge router's IPsec module sees the outbound packet:

  src=10.1.0.42 dst=10.2.0.10 proto=TCP dport=445

It searches the SPD with these selectors. Suppose the relevant rule is:
  • Selector: any traffic from 10.1.0.0/24 to 10.2.0.0/24.
  • Action: PROTECT with SA #42 (an ESP-tunnel-mode SA toward 203.0.113.5, our Branch B router).
The SPD says: "Yes, protect this packet with SA #42."
Phase 4 — Look up SA in SAD The router retrieves SA #42 from its SAD. Suppose the entry says:
  • SPI: 0xCAFE0042
  • Destination IP: 203.0.113.5
  • Protocol: ESP
  • Mode: tunnel
  • Cipher: AES-128-GCM, key = (16 bytes loaded from secure storage)
  • Sequence counter: currently 4087, increments on next use
  • Lifetime: expires at 12:34:56 UTC
The kernel grabs all this and prepares to apply ESP-tunnel-mode.
Phase 5 — Apply ESP-tunnel-mode encryption Because the SA mode is tunnel, the entire original IP packet (header included) becomes the inner packet that gets encrypted and wrapped. The ESP processing pipeline:
  1. Take the entire original packet bytes (IP header + TCP header + data).
  2. Append an ESP trailer (padding to align block size, padding-length byte, and the next-header byte = 4 meaning "what follows is an IP packet").
  3. Encrypt the whole inner-packet-plus-trailer bundle with AES-128-GCM, using the SA's cipher key + a fresh IV.
  4. Compute the ICV (integrity check value) over the ESP header + the encrypted body. With AES-GCM, this is produced as the GMAC tag during the same encryption pass.
  5. Prepend an ESP header with SPI = 0xCAFE0042 and sequence number = 4088.
  6. Increment the SA's sequence counter to 4088 in the SAD entry.
Phase 6 — Prepend a new outer IP header Because we're in tunnel mode, we now build a brand-new outer IP header: source = 198.51.100.1 (our public IP), destination = 203.0.113.5 (Branch B's public IP), protocol = 50 (ESP). We prepend this to the ESP-protected blob and send it out the public-facing interface.
Phase 7 — The packet crosses the public Internet Routers on the way only look at the outer IP header to forward the packet. They see:

  198.51.100.1 → 203.0.113.5 (proto = ESP)

Inside is opaque encrypted data. Anyone watching only learns "Acme's San Francisco gateway is talking to its Berlin gateway." They can't see Alice, can't see the file server, can't see what's being transferred.
Phase 8 — Branch B router receives, dispatches by SPI Branch B's edge router (203.0.113.5) receives the ESP-wrapped packet. It reads the outer IP header (proto = 50, that's ESP), then peeks at the ESP header's SPI: 0xCAFE0042.

It looks up its SAD for the matching incoming SA. Found: this is the inbound counterpart to Branch A's SA #42 — same algorithms, same keys, but in the opposite direction.
Phase 9 — Anti-replay check, then decrypt and verify The router reads the sequence number from the ESP header (4088). It checks against the SA's anti-replay window: is this number above the window's highest seen value (so it's new)? Or within the window but unseen (so it's still new)? Or below the window (replay — drop)?

Assuming the sequence is fresh, the router decrypts the ESP body with the SA key and verifies the GMAC tag. If verification fails, drop the packet and log. If it passes, the router has now recovered the original inner IP packet: source 10.1.0.42, destination 10.2.0.10, TCP segment, port 445.
Phase 10 — Deliver inner packet to internal network The router strips the outer IP header and ESP wrapping, leaving the original inner packet. Then it routes it onto the internal Branch B network as if it had been received from the local LAN. The file server at 10.2.0.10 sees a normal TCP segment from 10.1.0.42 and processes it normally. From the file server's perspective, no IPsec was involved at all.

Wait, the file server doesn't know any of this happened? Correct — and this is a key feature, not a bug. The file server runs on a regular Linux box on Branch B's internal network. It sees a TCP connection from 10.1.0.42 as if Alice were on the same LAN. The IPsec tunnel is invisible to it. This is what slide 370 means by "IPsec is transparent to applications" — it works in the kernel below the socket API, and (in the gateway-to-gateway case) doesn't even need to be present on the hosts.

AH versus ESP: same framework, different protections

Now that you've seen ESP in action, let's contrast it with AH. They're the two protection protocols inside IPsec; you have to know what each one does, and what it doesn't.

The fundamental difference, in one sentence

AH authenticates but never encrypts. ESP authenticates AND encrypts (and almost everyone uses it).

That's the headline. The interesting details are about what each protocol's MAC actually covers.

AH header in detail

   0                   1                   2                   3

The fields, in plain English:

  • Next Header (8 bits) — what protocol follows this AH header. Common values: 6 = TCP, 17 = UDP, 4 = IP-in-IP (this would mean tunnel-mode AH with an inner IP packet). The receiver uses this to decide what to hand the payload to after stripping AH.
  • Payload Len (8 bits) — length of AH itself in 32-bit words minus 2 (a quirky encoding from the RFC).
  • Reserved (16 bits) — set to zero, ignored.
  • SPI (32 bits) — identifies which SA to use, as we discussed.
  • Sequence Number (32 bits) — anti-replay counter, increments per packet.
  • ICV (variable, typically 12–32 bytes) — the MAC over the protected fields. With HMAC-SHA-256, it's typically truncated to 16 bytes.

What AH's ICV actually covers

This is the trickiest part of AH and a frequent exam target. AH's ICV is computed over:

  1. Immutable fields of the outer IP header — fields that don't change in transit. Source IP, destination IP, version, etc.
  2. Mutable-but-predictable fields — fields that can change, but whose final value is computable in advance. Example: destination IP under source routing.
  3. The AH header itself (with the ICV field zeroed during computation, since you can't include the value you're computing).
  4. The payload (TCP/UDP/whatever follows AH).

What's excluded (zeroed during ICV computation): the genuinely mutable, unpredictable fields of the IP header. The most important examples:

  • TTL (Time To Live) — decremented at every router hop. Including it would invalidate the ICV at every hop.
  • IP header checksum — recomputed at every hop because the TTL changes.
  • Type of Service (TOS) in some configurations — sometimes rewritten by routers for QoS.
Wait, why does AH bother covering the IP header at all? This is AH's entire reason for existing. ESP's MAC covers only the ESP header and the (encrypted) payload — not the outer IP header. So with ESP alone, an attacker could in principle modify the outer IP header (e.g., redirect the packet to a different address) and the receiver wouldn't catch it via cryptographic check (though normal routing would still get the packet to the destination listed in the modified header, which defeats the modification's purpose anyway).

AH's covering the outer IP header is the one service it provides that ESP doesn't. This is also why AH is incompatible with NAT: NAT rewrites the source IP, but AH's ICV treats the source IP as immutable, so the rewrite invalidates the ICV.

  • Padding is needed for block-cipher alignment (less of an issue with stream-style modes like GCM).

Common confusion: "An SA is a two-way connection." No — an SA is simplex (one-way). Anti-replay: how the receiver detects duplicates

Imagine an attacker captures one of your IPsec packets off the wire and replays it later. They can't read it (it's encrypted), they can't modify it (the ICV would fail), but can they at least retransmit it and get the receiver to process it twice? For example, replaying a packet that says "transfer $100 from A to B"?

The answer is no, because of the anti-replay service. Each SA has a per-direction sequence-number counter. The sender increments it for every outbound packet (1, 2, 3, …). The receiver tracks recent values in a sliding window and rejects anything it's already seen.

The sliding window, visually

Suppose the receiver's anti-replay window is W = 8 (small for the example; real deployments use 32 or 64). The receiver tracks two pieces of state: the highest sequence number seen so far (call it N), and a bitmap of width W telling which of the most recent W numbers have been accepted.

Walk through it. Suppose so far the receiver has accepted packets 12, 13, 14, 15, 16, 17, 18, 19. So N = 19, and the window covers numbers 12 through 19 (the most recent 8).

   accepted (✓) or not yet (✗)
                                         N (highest seen)
                                          │
         12   13   14   15   16   17   18  ▼19
   ──────┬────┬────┬────┬────┬────┬────┬────┬────────────► seq #
         │ ✓  │ ✓  │ ✓  │ ✓  │ ✓  │ ✓  │ ✓  │ ✓  │
         └────┴────┴────┴────┴────┴────┴────┴────┘
         ◄──────── window of width W=8 ─────────►
         lowest accepted = N − W + 1 = 12

Now four scenarios for the next packet that arrives:

Scenario A — A new, fresh packet arrives with sequence number 20 20 > N. Action: accept. Slide the window right by 1 (now covers 13–20). Update N to 20. Mark slot 20 as ✓. Marked it as 12 fell off the back.
Scenario B — A delayed (out-of-order but legitimate) packet arrives with sequence number 14 14 is within the window. Check the bitmap for slot 14: it's already ✓. Conclusion: this is a duplicate (or a replay). Action: discard, audit, do not deliver to the application.
Scenario C — A delayed packet arrives with sequence number 13, but the bitmap shows ✗ Wait, that can't happen in our example because we said all of 12–19 were accepted. But suppose at some earlier point the receiver hadn't yet seen 13 — say network reordering held it up. Then 13 is within the window and unseen. Action: accept, mark slot 13 as ✓, deliver to the application. The window doesn't slide because 13 isn't above N.
Scenario D — A very old packet arrives with sequence number 5 5 < (N − W + 1) = 12. It's below the window — too old to verify. We can't tell whether we already saw it or not, because we don't keep records that old. Action: discard, audit.

The sliding window gives O(1) memory (just N and the W-bit bitmap) while detecting all replays of any packet still inside the window, and erring on the side of safety for very old packets.

Wait, why discard the very old packet? What if it was just slow? Because the receiver has no way to know, with O(1) memory, whether it's already seen sequence number 5. Maybe it did and forgot; maybe it didn't. Discarding "very old" packets is a safe default — under normal network conditions, packets don't show up that late, so legitimate traffic is rarely affected. If your network is so badly reordered that 64 packets pass before a delayed one arrives, you have bigger problems than IPsec's anti-replay window.

What happens when sequence numbers run out

The sequence number is 32 bits — about 4 billion packets. At 1 Gbps with average 1 KB packets, you'd hit that limit in roughly 8 hours. The standard requires that, before the sender's counter wraps around, IKE establishes a fresh SA with new keys. The old SA is retired. This is one reason SAs have lifetimes.

(IPsec extensions support 64-bit "extended sequence numbers" for very-high-throughput use, but the standard 32-bit counter is the default.)

"Wait, but what about…?" — IPsec architecture questions

Why is IPsec called a "framework" instead of a protocol?

Because it's modular. AH is one piece. ESP is another. IKE is a third. Transport vs tunnel modes are independent dimensions. The cipher and MAC algorithms are pluggable (AES-GCM today, but the framework allows others). When you say "we're running IPsec," the next question is always "with which combination of these knobs?" That's why "IPsec framework" is more accurate than "IPsec protocol."

Where does AES-GCM fit into all this?

AES-GCM is one of the cipher algorithms IPsec ESP can use. When you negotiate ESP-with-AES-GCM, you get encryption AND integrity (the GMAC) in one pass — so the ESP "ICV" is produced as the GMAC tag during encryption, not by a separate HMAC computation. This is the same primitive TLS uses for its modern record-layer cipher suites and that WPA2/3 uses for link encryption. One mental model, several protocols.

Why is AH still in the standard if nobody uses it?

Two reasons. First, completeness — AH was in the original IPsec spec from 1995 and removing it would break the few legitimate deployments that use it. Second, AH is the only IPsec mechanism that authenticates the outer IP header. If you specifically need that property (paranoid scenarios where you want cryptographic proof the IP header wasn't tampered with at the IP layer), AH is your only option. In practice, almost no one needs that property strongly enough to fight NAT incompatibility, so AH stays as a niche tool.

How does IKE actually authenticate the peers?

Two main methods. Pre-shared keys (PSK) — both peers are configured with the same secret string, and they prove they know it during the IKE exchange. Common for small site-to-site VPNs. Certificates (X.509) — each peer has a certificate signed by a trusted CA; they verify each other's certificates and use the certificate keys to sign DH values, exactly like TLS. Common in enterprise deployments. Either way, the peer authentication happens during IKE; once IKE finishes, you have established SAs and the framework switches to using them.

Can a single computer run multiple IPsec connections simultaneously?

Yes. Each peer relationship is a pair of SAs (or more if multiple algorithms are used). They live in the SAD as separate entries, indexed by SPI. The kernel uses the SPD to figure out which SA applies to each outgoing packet. You can have hundreds of active SAs at once on a busy VPN concentrator — each connecting branch office or remote-access user contributes their own pair.

Q2 Without scrolling back: what triple uniquely identifies a Security Association? Why is each part of the triple necessary?

Q3 Alice's IPsec gateway is sending packets toward Bob's gateway. The administrator at Alice's site wants to add a new firewall rule: "all DNS traffic should not be IPsec-protected — let it go in plaintext." Which of the two IPsec databases does the administrator modify, and what entry do they add?
▶ Show answer

The administrator modifies the SPD (Security Policy Database), since this is a policy decision about whether to protect certain traffic.

The new entry:

  • Selector: protocol = UDP, destination port = 53 (DNS).
  • Action: BYPASS.

This entry is consulted by the IPsec module on each outbound packet. Packets matching the selector skip IPsec entirely and go out as plain IP. The SAD doesn't need any changes — that's where active SA state lives, and we're not creating or destroying any SAs.

Q4 An attacker captures an ESP-protected packet on the wire and replays it 30 minutes later. What happens at the receiver, and which security service prevents the replay from succeeding?
▶ Show answer

The attacker can resend the bytes; the receiver accepts the packet at IP, finds the matching SA via SPI, and starts processing it. But: the ESP header includes a sequence number. The receiver checks against its sliding anti-replay window. After 30 minutes of normal traffic, the window has moved far past the captured packet's sequence number — the captured number is now well below the window. The receiver discards the packet.

If the captured packet had been replayed almost immediately, when the sequence number was still inside the window, the receiver would still detect the replay because the bitmap shows that sequence number was already seen.

Either way, the anti-replay service (sliding-window sequence-number check) prevents the attack. Without it, the attacker would have a meaningful replay primitive.

Q5 A friend says: "I know IPsec uses AH and ESP. Can you tell me, in one sentence each, when I'd use one versus the other?"
▶ Show answer

Use ESP if you want any combination of encryption and authentication on a packet's payload. Almost every real-world deployment uses ESP; in modern usage with AES-GCM, ESP delivers confidentiality + integrity + authentication + anti-replay all in one.

Use AH if you specifically need to authenticate the outer IP header itself (a property ESP doesn't provide), and you don't care about confidentiality, and you don't have NAT in the path. This is a niche case — almost no one uses AH today.

If the question is "do I need confidentiality?" the answer is essentially always yes, and that means ESP.

Q6 A first-year student says: "I think I get it now — IPsec replaces TLS, since they both encrypt traffic." How would you correct them?
▶ Show answer

They overlap but solve different problems.

IPsec runs at Layer 3 (the IP layer), so it protects all packets between two endpoints regardless of application. It's typically deployed at gateways, terminating at the edge of a network — meaning between the gateway and the actual server, traffic is no longer IPsec-protected.

TLS runs at Layer 4½ (between TCP and the application), so it protects one application's traffic end-to-end, from the user's browser/client all the way to the actual server. But it requires the application to be TLS-aware.

In practice they're complementary, not redundant. A typical secure web request from a remote employee goes: TLS (browser to webserver, end-to-end) running inside an IPsec VPN tunnel (laptop to corporate gateway). TLS protects the application data through the gateway and beyond; IPsec hides the metadata (which servers are being accessed) on the public path between laptop and gateway.

6B Transport mode vs Tunnel mode 📄 pp. 384–388  ·  The instructor calls this out by name as exam material — pay attention

The story: why two modes?

When the IPsec designers sat down to spec how AH and ESP would actually wrap an IP packet, they faced a forked requirement. Two scenarios that both deserved IPsec:

  1. Two end hosts that both speak IPsec, talking directly to each other. They want their traffic encrypted, but they don't need to hide their own IP addresses from anyone — those addresses are how routers will deliver the packets. Example: your laptop with IPsec configured, talking to a server with IPsec configured, both on the public Internet.
  2. Two security gateways protecting whole networks behind them. The gateways want to encrypt the traffic of all the hosts behind them, while hiding the inner topology (which hosts are talking to which) from anyone watching the public Internet. Example: our running scenario — Branch A and Branch B routers, with the entire 10.x.x.x address space hidden inside.

The first scenario only needs the original IP packet's payload protected. The IP header can stay where it is — visible, routable. The second scenario needs the whole packet (header included) hidden inside something the public Internet thinks is just "gateway-to-gateway" traffic. One mode can't serve both. So IPsec has two:

  • Transport mode. Insert AH/ESP between the existing IP header and its payload. The original IP addresses remain visible.
  • Tunnel mode. Wrap the entire original IP packet (header + payload) inside a new outer IP header. The original IP addresses go inside the encrypted blob.
Transport mode An IPsec mode where AH/ESP is inserted between the existing IP header and the payload. Think of it as tagging an existing packet with cryptographic protection. The original IP header — including the source and destination addresses — stays visible on the wire. Used for end-to-end host-to-host protection.

Instructor's verbatim quote, lecture slide 386: "Transport mode: IP addresses are NOT encrypted."
Tunnel mode An IPsec mode where the entire original IP packet becomes the encrypted/authenticated payload of a new packet, with a new outer IP header in front. Think of it as wrapping the original packet inside a fresh envelope addressed gateway-to-gateway. With ESP, the inner IP header (including its addresses) is hidden inside the encrypted blob.

Instructor's verbatim quote, lecture slide 387: "Tunnel mode: inner IP IS encrypted."

An analogy: locked envelopes vs. locked diplomatic pouches

Imagine sending a confidential letter from Alice in San Francisco to Bob in Berlin.

Transport mode = locked letter inside a normal addressed envelope You take a regular envelope, write "From: Alice, 123 Mission St" and "To: Bob, Unter den Linden" on it, and mail it. Inside is your letter — locked in a tamper-evident pouch. Postmen along the way see who's sending to whom (the addresses are right on the envelope), but they can't read or modify the contents.
Tunnel mode = locked pouch inside an addressed envelope, but the inner pouch is itself another envelope with a different address You write your letter, put it in an envelope addressed "Alice → Bob," then put that whole envelope inside a locked diplomatic pouch, and put the pouch inside an outer envelope addressed "SF Consulate → Berlin Consulate." The public postmen see only consulate-to-consulate traffic. Only the two consulates know the inner letter is "Alice → Bob." Inside the consulate's locked pouch, even the inner addressing is hidden until decrypted at the receiving consulate.

The transport-mode envelope shows where the letter is going. The tunnel-mode envelope shows only the consulates. That's the metadata difference.

The four packet shapes: AH/ESP × transport/tunnel

There are four ways IPsec can wrap a packet, depending on which protection protocol (AH or ESP) and which mode (transport or tunnel) the SA specifies. We'll draw each one and walk through what's protected.

The original (unprotected) IP packet — our baseline

Before any IPsec, the packet on the wire looks like this:

The whole thing is integrity-protected by the ICV, but nothing is encrypted. The payload — TCP/UDP headers, application data, all of it — remains plaintext.

Use case: two hosts that want bit-level integrity protection on their traffic but don't need confidentiality. Rare in practice.

Shape 2 — AH in tunnel mode

  ┌────────────┬──────────┬────────────┬────────────────────────────┐

  │            │  ICV     │            │                             │
  └────────────┴──────────┴────────────┴────────────────────────────┘
   ◄── ICV covers entire inner IP packet + selected outer fields ──►
   ◄────────────── nothing encrypted ──────────────────────────────►

What you see on the wire: a new outer IP header (gateway-to-gateway), then AH, then the original packet (now treated as inner data). The whole inner packet is integrity-protected. The inner IP addresses are visible to anyone with a packet sniffer — AH never encrypts. So this mode hides nothing; it only authenticates.

Use case: almost none. AH-tunnel offers no advantage over AH-transport for typical use, and certainly less than ESP-tunnel.

Shape 3 — ESP in transport mode

  ┌────────────┬──────────┬────────────────────────────┬────────────┬────────┐
  │  IP hdr    │  ESP hdr │   TCP/UDP hdr + payload     │  ESP trail │  ICV   │

What you see on the wire: the original IP header is unchanged. After it: ESP header (SPI + sequence number, in plaintext so the receiver can read them). Then the encrypted blob: the original payload (TCP/UDP + data) plus an ESP trailer with padding and a "next-header" byte. Then the ICV (the MAC).

Source and destination IP addresses are visible. Port numbers and content are encrypted.

Use case: two hosts that want confidentiality of their direct communication. The classic example is two IPsec-aware servers in the same data centre.

Shape 4 — ESP in tunnel mode (the VPN workhorse)

Anyone watching sees only "Branch A gateway is talking to Branch B gateway." The fact that Alice's laptop is sending to a file server is hidden inside the encrypted blob.

Wait, why exactly does ESP-tunnel hide inner IPs but ESP-transport doesn't? Because of where the encryption boundary falls. In transport mode, the IP header is before the ESP header on the wire — so the IP header was added after ESP processing, meaning ESP couldn't have encrypted it. In tunnel mode, the inner IP header is inside the encrypted ESP body — so ESP encrypted it before the outer IP header was added.

Visually: in transport mode, the structure is "[IP][ESP][encrypted payload]." The IP header is outside the encryption boundary, by definition. In tunnel mode, the structure is "[outer IP][ESP][encrypted (inner IP + payload)]." The inner IP is inside the encryption boundary; the outer IP is outside.
Wait, why doesn't ESP also authenticate the outer IP header? Two reasons. First, technical: the outer IP header is added after ESP processes the packet, so when ESP computes the MAC, the outer header doesn't yet exist. To include it, ESP would need a complicated two-pass design. Second, practical: NAT happens in the wild. NAT rewrites the outer IP header. If ESP authenticated it, every NAT box would invalidate the MAC, and ESP would never traverse NAT — which would make ESP useless for most home and corporate deployments. So ESP just doesn't cover the outer header, and accepts the small risk of an attacker tampering with it (which doesn't really achieve much, since the packet still has to be routable).

AH covers the outer header precisely because that's its differentiator from ESP — and AH's NAT-incompatibility is the price.

"Wait, but what about…?" — mode questions

Why isn't there an "ESP-with-auth-and-also-IP-header-coverage" mode?

Because that's exactly what AH does, and the IPsec designers didn't want to bake AH-style outer-header coverage into ESP. The two protocols are intentionally separate. If you genuinely need outer-IP coverage plus encryption, you can run AH and ESP together (the standard allows nesting), though almost no one does — it's slow and complicated. In modern practice, ESP alone (with its standard authentication) is treated as enough.

Can the same SA support both transport and tunnel mode?

No — mode is fixed when the SA is created during IKE negotiation, and stored in the SAD entry. One SA, one mode. If you need both, you create two SAs.

Can a host (not a gateway) use tunnel mode?

Yes. The standard says tunnel mode is required when at least one endpoint is a security gateway acting on behalf of others, but it's allowed (just unusual) for two end hosts to use tunnel mode. Why might a host use tunnel mode? To hide its real IP address from public observers — for example, a remote-access VPN client connecting to a corporate concentrator. The concentrator is a gateway; the laptop is a host; the SA is tunnel-mode because the concentrator is a gateway.

What does the "Next Header" byte at the end of the ESP trailer tell the receiver?

It tells the receiver what protocol the encrypted contents represent — i.e., what to feed the decrypted bytes into next.

  • Next Header = 6: what's inside is a TCP segment. We're in transport mode (the TCP segment was the original payload).
  • Next Header = 17: UDP segment. Transport mode.
  • Next Header = 4: what's inside is an entire IP packet. We're in tunnel mode (the inner IP packet was the original).

So the same ESP processing code can transparently handle both modes — it decrypts, reads the trailer's next-header byte, and dispatches accordingly.

If tunnel mode hides the inner IPs, isn't transport mode strictly worse?

No, it's strictly cheaper. Transport mode adds only the AH/ESP header — say, 30 bytes — while tunnel mode adds an additional outer IP header on top, doubling the overhead. For two end hosts that already have routable public IPs and don't need to hide them, transport mode is the right choice. Tunnel mode is the right choice when you're protecting a network behind a gateway and want to hide the inner topology. Different tools for different deployments.

Q2 Of the four combinations of {AH, ESP} × {transport, tunnel}, exactly one provides "limited traffic-flow confidentiality." Which one, and why?
▶ Show answer

Only ESP in tunnel mode.

Reasoning by elimination:

  • AH never encrypts, in either mode. So both AH options leak inner addresses to anyone with a sniffer.
  • ESP transport leaves the IP header in front of ESP, so the addresses are visible. There is no inner IP header — the original is the only one — so there's nothing inner to hide.
  • ESP tunnel places the inner IP header inside the encrypted blob. Observers see only the outer (gateway-to-gateway) addresses. The original Alice→file-server addressing is hidden.

This is why every site-to-site VPN uses ESP in tunnel mode: it's the only combination that hides the inner topology.

Q3 An attacker captures an ESP-protected packet on the public Internet. The packet has source IP 198.51.100.1 and destination IP 203.0.113.5. What can the attacker conclude?
▶ Show answer

The attacker can conclude:

  • 198.51.100.1 is talking to 203.0.113.5 right now (this isn't hidden by IPsec — the outer header has to be in plaintext for the packet to be routable).
  • The protocol number is 50 (ESP).
  • The SPI value (visible at the start of ESP header).
  • The sequence number for this SA.
  • The packet size (in bytes).

What the attacker cannot determine:

  • Whether the SA is in transport or tunnel mode (the outer IP header looks the same in both cases — only decryption reveals what's inside).
  • If tunnel mode: the inner IP addresses, the ports, the content, the application protocol.
  • If transport mode: the ports, the content, the application protocol — but in this case the inner-IPs question doesn't apply because there's no inner IP header.

The attacker can do traffic analysis (correlating packet sizes and timings) but cannot read content.

Q4 A company is setting up a site-to-site VPN between two branch offices. The CTO asks: "Should we use AH transport, AH tunnel, ESP transport, or ESP tunnel?" Explain your recommendation with three reasons.
▶ Show answer

Recommended: ESP in tunnel mode. Three reasons:

  1. Gateway topology requires tunnel mode. Slide 387 says: "Tunnel mode is required when at least one endpoint is a security gateway." A site-to-site VPN puts gateways at both ends — they're protecting hosts behind them, so the SA endpoints (the gateways) are different from the actual traffic endpoints (the hosts). Tunnel mode is the right encapsulation.
  2. Confidentiality requirement rules out AH. Corporate traffic on the public Internet must be unreadable to outside observers. Only ESP encrypts. AH doesn't.
  3. Hiding internal addresses is valuable. The internal subnets (10.1.0.0/24 and 10.2.0.0/24) ideally should not be inferable from the public traffic. ESP-tunnel buries the inner addresses in the encrypted body. AH-tunnel does not (no encryption); ESP-transport doesn't apply (no inner header to hide).

Bonus: ESP in tunnel mode is also NAT-friendly (with NAT-T encapsulating ESP in UDP 4500), which AH is not. Site-to-site VPNs often have NAT in the path.

Q5 An exam stem says: "AH in tunnel mode encrypts the inner IP header." True or false, and explain.
▶ Show answer

False. AH never encrypts — in either mode. The clue is in the name: Authentication Header. AH only authenticates.

What AH-tunnel actually does: it wraps the original IP packet (header + payload) inside a new outer IP header, with an AH header in between. The ICV authenticates the entire inner packet and selected fields of the outer header. But every byte remains plaintext — the inner IP header is visible to anyone with a sniffer who captures the packet.

This is one of the most common exam-trap statements. If the stem mentions "AH" and "encrypt" together, it's almost certainly false.

The story: what is a VPN, really?

You've probably heard the word "VPN" thrown around. Maybe you've used one at work to access internal company resources from home. Maybe a friend recommended a "VPN service" for privacy on public Wi-Fi. Maybe you saw an ad selling "military-grade encryption" with a stylised globe. The term is overloaded enough that it's worth pinning down what a VPN actually is, mechanistically.

A VPN is, at its core, two simple ideas glued together:

  1. An encrypted tunnel. Some cryptographic protocol — usually IPsec or TLS — establishes an authenticated, encrypted channel between two points across the public Internet. We've seen exactly how this works: handshake, key exchange, symmetric record/packet encryption.
  2. Routing trickery. Traffic on one end of the tunnel is rerouted through it, so that it appears to originate from the other end of the tunnel. The remote-access user appears to be "on the corporate network." The branch office appears to be "directly connected" to the headquarters.

That's all. The "Virtual" in VPN means the network connectivity is logical rather than physical — there's no actual cable between your laptop and the office; the "Private" means it acts like a private network even though it crosses the public Internet.

VPN Virtual Private Network. An encrypted tunnel that makes a remote endpoint appear logically attached to a private network across an untrusted infrastructure (typically the Internet). Two main protocol families: IPsec-based (using ESP in tunnel mode, set up by IKE) and TLS-based (OpenVPN, Cisco AnyConnect, WireGuard).

Three deployment types

The literature talks about three VPN topologies, all of which we've already implicitly seen:

Site-to-site VPN — gateway-to-gateway Two networks (e.g., two branch offices) are connected by an encrypted tunnel between their edge routers. All hosts on both networks can transparently reach each other. Our running example (Branch A ↔ Branch B) is exactly this.
Remote-access VPN — host-to-gateway An individual user's device (laptop, phone) connects to a corporate VPN gateway over the Internet. After authentication, the device gets a virtual IP on the corporate network and can access internal resources. Used by remote employees connecting to their office network.
Extranet VPN — controlled cross-organisation Two distinct organisations connect specific subnets to allow controlled cross-access. Like site-to-site, but with restrictions on which traffic flows. Less common; mentioned for completeness.

What a VPN actually hides — and what it doesn't

This is the part beginners most often misunderstand. To frame it concretely, let's compare what a passive observer on the public Internet sees in three scenarios: no VPN, HTTPS only, and IPsec VPN. Suppose Alice (on a coffee-shop Wi-Fi) is browsing shop.example.com over HTTPS, perhaps over a VPN connection.

Scenario 1: No VPN, plain HTTP

What an eavesdropper at the coffee shop sees:

  src=Alice's_laptop_IP   dst=shop.example.com_IP    proto=TCP   dst_port=80
  HTTP request: "GET /products HTTP/1.1
                 Cookie: session=abc123
                 ..."
  HTTP response: "200 OK ... <html>[your shopping cart]</html>"

The eavesdropper learns: who Alice is talking to, what she's reading, her cookies, her shopping cart contents. Everything.

Scenario 2: HTTPS, but no VPN

Same coffee shop, same site, but now over HTTPS:

  src=Alice's_laptop_IP   dst=shop.example.com_IP    proto=TCP   dst_port=443
  TLS handshake (visible: SNI = "shop.example.com")
  Encrypted records (sizes visible, content not)

The eavesdropper now learns: Alice is visiting shop.example.com (the destination IP and SNI both leak that). They can guess from packet sizes that Alice is browsing — but they can't read what she's looking at, what her cookies are, or what's in her shopping cart.

Scenario 3: HTTPS plus IPsec VPN to corporate gateway

Now Alice is connected via VPN to her corporate VPN concentrator at vpn.acme.com. The HTTPS traffic to shop.example.com goes through the VPN tunnel:

  src=Alice's_laptop_IP   dst=vpn.acme.com_IP    proto=ESP
  Encrypted ESP blob (one giant tunnel-mode payload containing
  the entire IP-TCP-TLS-HTTP stack)

The eavesdropper at the coffee shop now learns: Alice is connected to the Acme corporate VPN. That's it. They can't tell whether she's browsing the internal corporate intranet, or shop.example.com, or doing video calls, or downloading software updates. The VPN tunnel hides all of that, because to the public-Internet observer, it all just looks like ESP packets to vpn.acme.com.

Wait, so what does the VPN add over HTTPS? Traffic-flow confidentiality. HTTPS hides what you say. The VPN additionally hides where you're going. With HTTPS alone, an observer knows Alice visits shop.example.com — even if they can't read the page contents. With the VPN tunnel adding to that, the observer sees only "Alice talked to vpn.acme.com" and has no idea what destinations she actually accessed inside.

Many people confuse this. HTTPS is end-to-end content protection. A VPN is metadata protection on the public path. They're complementary, not redundant — using both gives you the union of their protections.
Common confusion: "A VPN makes me anonymous." No, it doesn't. A VPN shifts trust from your ISP and Wi-Fi operator to the VPN provider. The VPN provider can see all your traffic — they're decrypting it before forwarding. They might log everything you do. They might be served with a subpoena. They might be hacked.

Anonymity tools (like Tor) work fundamentally differently — they bounce traffic through multiple independently operated relays, none of which knows both your identity and your destination. A consumer VPN service is just one company in the path that sees everything. If you want anonymity, use Tor.

A VPN is for: (a) reaching internal corporate resources from outside; (b) hiding your traffic from the local network's operator and your ISP; (c) appearing to be in a different geographic location. It is not for "becoming anonymous" or "hiding from the government."
Common confusion: "Once I connect to a VPN, all my traffic is encrypted end-to-end." No. The VPN's encryption protects only the path between your device and the VPN gateway. After the gateway decrypts your traffic, it goes out to the actual destination on the public Internet — unencrypted, unless that destination is using its own protocol (HTTPS, etc.).

So if you're using a VPN and visit http://insecure-site.com (plain HTTP), your traffic is encrypted from your laptop to the VPN gateway, but plaintext from the VPN gateway onward to insecure-site.com. The VPN provider sees your traffic in plaintext at the gateway. Anyone sniffing on the path between the VPN gateway and the destination also sees plaintext.

For end-to-end encryption between you and the destination, you still need TLS (or whatever the destination's own security layer is). The VPN protects only the segment that goes through the tunnel.

A complete IPsec VPN session, end to end

Let's go back to our Branch A ↔ Branch B running example and trace the entire lifecycle: from "no VPN configured" to "Alice's first packet flows" to "VPN tear-down."

Phase 1 — Configuration (out of band, by admin) The admins of Branch A and Branch B agree on:
  • Authentication method: pre-shared key "BranchAtoBranchB!2026" or X.509 certificates from a corporate CA.
  • IKEv2 parameters: AES-256-CBC for IKE encryption, SHA-256 for IKE integrity, DH group 14 (2048-bit) or 19 (Curve25519) for key exchange.
  • ESP parameters: AES-128-GCM for the data-protecting SAs, lifetime = 8 hours.
  • Traffic selector: "any traffic from 10.1.0.0/24 to 10.2.0.0/24" (and the reverse).
Each admin enters the configuration into their router's IPsec/IKE daemon. Nothing happens on the wire yet.
Phase 2 — IKE handshake (when the first packet needs to flow) Suppose at 9:00 AM, Alice opens a connection to the file server. Her packet routes to the Branch A edge router. The router's SPD says: "PROTECT with SA toward Branch B," but no SA exists yet. So the router starts IKE.

IKEv2 runs over UDP 500 (or 4500 if NAT is detected). It's a 4-message exchange (simplified):
  1. IKE_SA_INIT request (Branch A → Branch B): cookies + supported algorithms + DH public value + nonce.
  2. IKE_SA_INIT response (Branch B → Branch A): chosen algorithms + DH public value + nonce.
  3. IKE_AUTH request (Branch A → Branch B, encrypted with the IKE-derived keys): identity + signature (or PSK MAC) + child-SA proposal (the actual ESP SAs we want).
  4. IKE_AUTH response (Branch B → Branch A, encrypted): identity + signature + child-SA acceptance.
At the end of this exchange, both peers have:
  • An IKE SA for further IKE communication (e.g., to rekey or add more child SAs later).
  • A pair of ESP child SAs — one for outbound (Branch A → Branch B), one for inbound (Branch B → Branch A). These are written into both routers' SADs.
Phase 3 — Alice's traffic flows through the established SAs Now the SADs are populated. The router's outbound IPsec module finds the matching SA, applies ESP-tunnel-mode (per the 10-step walkthrough we did in 6A), and ships encrypted packets to Branch B's router. Branch B's router decrypts, strips the outer header and ESP wrapping, delivers the inner packet to the file server. File server replies; Branch B's router applies ESP using the inbound SA's counterpart; Branch A decrypts.

Both ends maintain anti-replay windows. Both keep counting bytes and packets toward the SA's lifetime threshold.
Phase 4 — Rekeying (after lifetime threshold or 2^32 packets) Before the SA expires (or before the sequence number wraps), IKE runs again — this time as a "CREATE_CHILD_SA" exchange — to establish replacement ESP SAs with fresh keys and a fresh DH-derived secret. The old SAs are kept active briefly to handle in-flight packets, then deleted. The transition is invisible to Alice.
Phase 5 — VPN tear-down Either side can initiate teardown by sending an IKE INFORMATIONAL exchange with a DELETE payload. Both SADs delete the relevant entries. Subsequent traffic matches no SA, so the SPD's PROTECT action kicks IKE off again to re-establish (assuming the policy is still active). If the admin wants to permanently end the VPN, they remove the SPD entries too.

Two protocol families: IPsec-based vs TLS-based

Slide 389 lists two main families of VPN protocols:

IPsec-based VPNs

Examples: strongSwan, Libreswan, the IPsec stacks built into Cisco/Juniper/Fortinet enterprise routers. All use IKE (typically IKEv2) for setup and ESP in tunnel mode for the data path.

Strengths:

  • Universal in enterprise gear. Every commercial-grade router supports it.
  • Kernel-level performance. The IPsec stack is in the kernel, so encryption is near line-rate.
  • Transparent to applications. Once the kernel is configured, applications don't need to know IPsec exists.
  • Real Layer-3 hiding. Tunnel-mode ESP genuinely hides which inner hosts are talking.

Weaknesses:

  • NAT can be tricky. Need NAT-T (encapsulating ESP in UDP 4500) to traverse NATs; some restrictive firewalls still block IKE/ESP entirely.
  • Configuration is complex. SAs, SPDs, certificates, policies — lots of moving parts.
  • Less common for individual users. Not what consumer VPN services typically deploy.

TLS-based VPNs

Examples: OpenVPN, Cisco AnyConnect, plus the modern WireGuard (which uses neither IPsec nor TLS but is grouped with TLS-based VPNs in slide 389).

How OpenVPN works mechanically: the client runs a user-space daemon. It establishes a TLS connection to the OpenVPN server, typically over UDP port 1194 (DTLS-style) or TCP port 443 for restrictive firewalls. The client kernel exposes a virtual interface called tun0. Routes are set up so that traffic to the corporate network (or, in "full-tunnel" mode, all traffic) is sent into tun0. The daemon reads packets from tun0, encapsulates them inside the TLS connection, and sends them to the server. The server's daemon decrypts and routes the inner packets onto its own network. Replies travel the same path in reverse.

  • Cross-platform. OpenVPN clients exist for every OS.

  • Slightly slower due to user-space overhead (less of an issue with modern hardware).
  • Requires client software. Not built into the OS the way IPsec is.

The protocol is small enough to fit in a few thousand lines of kernel code.

"Wait, but what about…?" — VPN questions

What does "split-tunnel" vs "full-tunnel" mean?

A full-tunnel VPN routes all the user's traffic through the corporate VPN — even traffic to public sites like Google or YouTube. Pro: simple policy, all traffic protected. Con: routes every byte through corporate infrastructure even for traffic that doesn't need protection.

A split-tunnel VPN routes only traffic destined for the corporate network through the tunnel; traffic to public sites goes directly out the user's local Internet connection. Pro: less load on corporate infrastructure, faster for general browsing. Con: more complex routing, and the VPN no longer provides traffic-flow confidentiality for public-site traffic (since it goes around the tunnel).

Why does my VPN occasionally drop or stutter?

Three common reasons. Rekeying: when an SA approaches its lifetime, IKE establishes a new one — usually invisible, but a brief glitch is possible if the transition is rough. Network changes: if your laptop hops between Wi-Fi and cellular, the source IP changes, and the VPN has to re-establish. NAT timeouts: NAT routers maintain mappings only as long as traffic flows; if the VPN goes idle for a few minutes, the NAT entry expires, and the next packet comes from a "different" IP/port pair, breaking the connection. Modern VPN clients send keepalive packets to prevent this.

Can a VPN protect me on public Wi-Fi?

Yes, and this is one of the genuinely useful applications. On public Wi-Fi, the access point operator can see all your traffic (whether they actively monitor or not). With a VPN, all the access point sees is encrypted ESP/UDP traffic to the VPN gateway. They can't tell what sites you're visiting, can't perform man-in-the-middle on HTTPS (since they don't have the VPN's keys), can't inject ads or track you. The protection extends only to the VPN gateway, but that's exactly the segment you don't trust.

Why do some streaming services block VPN traffic?

Because licensing deals are geographic ("show this content only to viewers in country X"). Some users tunnel their traffic through a VPN endpoint in another country to bypass the geo-restriction. Streaming services react by maintaining lists of known VPN provider IP ranges and blocking them. This is a cat-and-mouse game; the VPN provider keeps changing IPs, the streaming service keeps updating blocklists.

Where does the term "VPN concentrator" come from?

It's just an enterprise-marketing name for "VPN server that handles many simultaneous remote-access connections." Cisco called their dedicated VPN-server appliance a "concentrator" because it concentrates many remote-user tunnels into one device. The term stuck. Functionally it's the same as any other VPN gateway.

Q5 A friend asks: "I just signed up for a VPN service that promises 'military-grade encryption' and 'total anonymity online.' Is that legitimate?" How would you respond?

The encryption claim is fine — "military-grade" usually means AES-256, which is genuinely strong but is also what every modern VPN uses. It's a marketing phrase, not a meaningful differentiator.

The anonymity claim is the problem. A VPN doesn't make you anonymous; it shifts trust from your ISP to the VPN provider. The VPN provider can see all your traffic, log it, share it with law enforcement, sell metadata to advertisers, or get hacked and lose your data. There have been real cases of "no-logs" VPN providers turning over user data after court orders.

For genuine anonymity (multiple uncorrelated relays, no single party seeing both source and destination), the answer is Tor — not a VPN. For "hide my traffic from the local network and ISP," a reputable VPN is fine, but you're trusting the VPN provider as much as you previously trusted your ISP. Choose accordingly.

Each layer protects different things from different observers:

  • WPA2/3 protects only the wireless segment (your laptop to the access point). Past the AP, no protection.
  • IPsec protects packets across IP networks — typically gateway-to-gateway or host-to-gateway. Hides the inner topology in tunnel mode.
  • TLS protects an application's data end-to-end across all of the above layers. Hides content from anyone on the path, including the VPN gateway.
  • SSH is itself an application; protects shell sessions and tunnels end-to-end.

You can — and often should — stack them. Working from outside in: a remote employee's laptop connects via WPA2 to a coffee shop's Wi-Fi. From there, an IPsec VPN tunnels back to the corporate gateway. Inside that tunnel, the laptop's browser uses HTTPS (TLS) to reach an internal server. That's three layers of independent encryption, protecting against three different threat models. None of them is redundant; each addresses a different segment of the path.

Where Topic 6 connects to the rest of the course

The instructor's final-exam review notes flag specific "integration hooks" — places where Topic 6 material is likely to be combined with other topics in challenge questions. Understanding these isn't memorisation; it's seeing the cross-topic logic.

Hook 1 — IPsec as the "ultimate defence" against Topic 2 attacks Recall the Topic 2 attacks on TCP/IP: source-IP spoofing, TCP reset injection, session hijacking via guessed sequence numbers, ARP spoofing on a LAN. The instructor's review script literally calls IPsec the "ultimate way to prevent the transport layer attack."

Why does IPsec defeat all of those? Because the TCP segment is encrypted inside ESP. An attacker who wants to inject a forged TCP RST or a hijacking segment would need to construct a valid ESP packet — which requires the SA's keys. Without those keys, the attacker can produce only random bytes; the receiver's ICV check fails and the packet is dropped before TCP ever sees it.

ARP spoofing, IP spoofing, ICMP redirects — all the "manipulate-packets-on-the-LAN" attacks — fail similarly. Even if the attacker successfully redirects packets at L2, they still can't read or modify the encrypted ESP payload.

Likely exam stem: "Of the countermeasures you've learned for TCP attacks, which is the strongest, and why?" Answer: IPsec, because it encrypts the entire transport-layer payload, so attackers without the SA keys can't construct valid forged packets.
Hook 2 — Trust models compared across IPsec, TLS, DNSSEC, SSH All four protocols are public-key-based, but they anchor trust differently. Memorise this matrix:

  • IPsec ESP — most modern deployments use AES-GCM as the AEAD mode.
  • WPA3-Enterprise — uses GCMP (which is AES-GCM with an 802.11-specific framing).
  • SSH — modern OpenSSH supports aes128-gcm@openssh.com as a cipher.
Likely exam stem: "Name the cipher primitive that appears in TLS, IPsec, WPA3, and SSH. What services does it provide in a single pass?" Answer: AES-GCM. Provides confidentiality (via AES counter mode) and integrity (via GMAC) simultaneously.

📚 Glossary

Quick reference for every term introduced in this document. If you've forgotten what something means, look here first.

TermDefinition
AEADAuthenticated Encryption with Associated Data. A class of ciphers (AES-GCM, ChaCha20-Poly1305) that provide confidentiality and integrity in a single pass.
AES-GCMAdvanced Encryption Standard in Galois/Counter Mode. The modern default AEAD cipher in IPsec ESP, TLS 1.3, WPA3, and modern SSH. Provides encryption + integrity simultaneously.
AHAuthentication Header. IPsec protocol number 51. Provides integrity, data-origin authentication, and optional anti-replay — but never encrypts. Authenticates the immutable fields of the outer IP header in addition to the payload, which is its unique service vs ESP.
Anti-replay windowA sliding-window data structure (typically 32 or 64 entries wide) the receiver maintains per SA to detect replayed packets. Tracks the highest sequence number seen and a bitmap of recently accepted numbers.
AuthenticationThe process of verifying that a packet (or peer) is genuinely from who it claims to be. In IPsec, achieved via MAC (per packet) or peer authentication during IKE (per session).
CACertificate Authority. An entity that signs X.509 certificates. IPsec can authenticate peers using CA-signed certificates as an alternative to pre-shared keys.
ChaCha20-Poly1305A modern AEAD cipher used as an alternative to AES-GCM, especially in WireGuard and on hardware without AES acceleration. Provides confidentiality + integrity in one pass.
Cipher suiteA named bundle of algorithms used together — typically (key exchange, authentication, bulk cipher, MAC). IKE negotiates the analogous "transform set" for IPsec SAs.
ConfidentialityThe property that data cannot be read by unauthorised parties. In IPsec, provided only by ESP, never by AH.
Diffie-Hellman (DH)A cryptographic protocol for two parties to establish a shared secret over a public channel. The basis for IKE's key exchange.
(EC)DH / ECDHEElliptic-curve variant of Diffie-Hellman. Faster and uses smaller keys for equivalent security. Curve25519 is a popular modern choice (used in WireGuard, IKEv2 group 31).
ESPEncapsulating Security Payload. IPsec protocol number 50. The workhorse protection protocol — provides encryption, integrity, authentication, anti-replay, and (in tunnel mode) limited traffic-flow confidentiality.
Extranet VPNA VPN that connects two distinct organisations with controlled cross-access, exposing only specific subnets to each other.
Full-tunnel VPNA VPN configuration where all of the user's traffic — including traffic to public Internet sites — is routed through the corporate VPN tunnel.
Gateway / Security gatewayA network device (typically an edge router or firewall) that runs IPsec on behalf of the hosts behind it. The IPsec endpoint is the gateway, not the actual hosts. Site-to-site VPNs use gateways at both ends.
HMACHash-based Message Authentication Code. Used in IPsec for integrity when the SA isn't using AEAD (e.g., HMAC-SHA-256 with classic AES-CBC ESP).
ICVIntegrity Check Value. The MAC tag at the end of an AH or ESP packet. With AES-GCM ESP, this is the GMAC tag produced during encryption.
IKE / IKEv2Internet Key Exchange. The IPsec key-management protocol that authenticates peers, runs Diffie-Hellman, and establishes SAs. IKEv2 (RFC 7296) is the modern version. Runs over UDP port 500 (or 4500 with NAT-T).
Inbound SAAn SA used by the local peer to process incoming packets — i.e., to decrypt and verify packets received from the remote peer. The remote peer's outbound SA matches it directionally.
IntegrityThe property that data has not been modified in transit. Provided by both AH and ESP via the ICV (MAC).
IP protocol number 50The protocol field value in the outer IP header that identifies an ESP packet.
IP protocol number 51The protocol field value in the outer IP header that identifies an AH packet.
IPsecInternet Protocol Security. An IETF framework for adding cryptographic protection (confidentiality, integrity, authentication, anti-replay) to IP packets at the network layer. Comprises AH, ESP, IKE, SPD, SAD, plus transport and tunnel modes.
IV / NonceInitialisation Vector / Number-used-Once. A unique value combined with the encryption key to produce ciphertext. Must never repeat for the same key, especially for AES-GCM. In ESP, derived from the sequence number plus a per-SA salt.
Key derivationThe process of expanding a shared secret (e.g., from Diffie-Hellman) into multiple working keys (encryption key, MAC key, IV salt). Done after IKE_AUTH.
NAT-T (NAT Traversal)A technique that encapsulates ESP packets in UDP (port 4500) to traverse NAT routers, which don't handle bare ESP (IP protocol 50). Negotiated as part of IKE when NAT is detected on the path.
NonceSee IV. In IKE, also a random value exchanged during IKE_SA_INIT to prevent replay of the handshake itself.
OpenVPNA popular TLS-based VPN protocol. Runs in user space, uses a virtual tun0 interface to capture traffic, encapsulates packets in TLS over UDP 1194 or TCP 443.
Outbound SAAn SA used by the local peer to process outgoing packets — i.e., to encrypt and authenticate packets sent to the remote peer. The remote peer's inbound SA matches it directionally.
PSK (Pre-Shared Key)A symmetric secret string configured manually on both peers, used by IKE to authenticate them to each other. Common in small site-to-site deployments. Alternative to certificate-based authentication.
Public/private key pairA mathematically linked pair where the private key signs/decrypts and the public key verifies/encrypts. Used in IKE certificate-based authentication.
RekeyingThe process of replacing an SA's keys with fresh ones before the SA expires (or before the sequence number wraps). In IKEv2, done via the CREATE_CHILD_SA exchange.
Remote-access VPNA VPN topology where an individual user's device connects to a corporate VPN gateway over the Internet to gain access to internal resources. Host-to-gateway tunnel.
Replay attackAn attack where an adversary captures legitimate packets and resends them later, hoping to cause the receiver to process them again. Defeated in IPsec by anti-replay sequence numbers.
RSAA public-key cryptosystem. Can be used in IKE for peer authentication via X.509 certificates (where the certificate's signature is RSA-based).
SA (Security Association)The central concept in IPsec. A one-way (simplex) cryptographic context between two peers, holding the algorithm, keys, sequence counter, anti-replay window, mode, and lifetime needed to protect packets in one direction. Bidirectional communication needs two SAs. Identified by the triple (SPI, destination IP, AH/ESP).
SAD (Security Association Database)The kernel table holding all active SAs and their state (keys, counters, window). Looked up on every IPsec-protected packet — by SPI on inbound, by SA ID on outbound.
Sequence numberA 32-bit counter incremented per outbound packet within an SA. Used by the receiver's anti-replay window to detect duplicates.
Site-to-site VPNA VPN topology where two whole networks are connected by a tunnel between their edge routers. Gateway-to-gateway. Always uses tunnel mode.
SPD (Security Policy Database)The kernel table holding policy rules: which traffic should be IPsec-protected (PROTECT with SA X), which should bypass (BYPASS), which should be dropped (DISCARD). Consulted on every outbound packet before any IPsec processing.
SPI (Security Parameters Index)A 32-bit tag in every AH/ESP header. Tells the receiver which SA to use to process the packet. Unique only when paired with destination IP and protocol.
Split-tunnel VPNA VPN configuration where only traffic destined for the corporate network goes through the tunnel; traffic to public sites goes directly via the user's local Internet connection.
Symmetric encryptionEncryption where the same key is used for both encryption and decryption. AES is the standard symmetric cipher. Used for bulk data protection in IPsec ESP after IKE establishes shared keys.
TFC (Traffic-Flow Confidentiality)The property that an observer cannot infer who is talking to whom. IPsec's "limited traffic-flow confidentiality" service — provided only by ESP in tunnel mode — hides the inner IP addresses but not the outer gateway-to-gateway addresses.
TOFU (Trust On First Use)A trust model where a peer's public key is accepted on the first connection and pinned for future use. SSH's known_hosts uses this. Not used by IPsec, but worth contrasting with IPsec's PSK and certificate-based approaches.
Transport modeAn IPsec mode where AH/ESP is inserted between the existing IP header and its payload. Original IP addresses remain visible. Used for end-to-end host-to-host protection.
Tunnel modeAn IPsec mode where the entire original IP packet (header included) becomes the inner packet inside a new outer IP header. Original IP addresses hidden inside the encrypted blob (when ESP is used). Used for site-to-site VPNs and remote-access VPNs.
UDP port 500The default port for IKE/IKEv2 messages.
UDP port 4500The port used for IKE and ESP encapsulation when NAT-T is active.
VPNVirtual Private Network. An encrypted tunnel that makes a remote endpoint appear logically attached to a private network across an untrusted infrastructure. Two main protocol families: IPsec-based and TLS-based (with WireGuard as a modern third path).
VPN concentratorA dedicated network appliance that terminates many simultaneous VPN connections. Cisco's marketing term for what is functionally a VPN gateway optimised for high concurrent connection counts.
WireGuardA modern, minimal VPN protocol. Uses neither IPsec nor TLS — has its own protocol based on the Noise framework, with Curve25519 + ChaCha20-Poly1305. Runs in the kernel; very small codebase.
X.509The standard format for digital certificates. Used by IPsec when peers authenticate via certificates rather than pre-shared keys, identical to TLS's certificate format.
