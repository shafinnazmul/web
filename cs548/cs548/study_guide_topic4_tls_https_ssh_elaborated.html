The core question of Topic 4: How do two computers that have never met and share no secrets establish a private, tamper-proof, authenticated conversation across an Internet full of eavesdroppers and impostors? The three protocols in this topic — TLS, HTTPS, SSH — are three real-world answers to that question. Once you understand TLS, HTTPS is "TLS but for web traffic" and SSH is "TLS-flavoured but for remote login."

🧠 What you should already know (a 5-minute refresh)

This section reminds you of the four ideas from earlier topics that we'll lean on heavily. If any of them feel unfamiliar, pause and go back to Topic 3 before continuing — the rest of Topic 4 will not make sense without these.

1. Symmetric encryption (one shared key)

Alice and Bob share a single secret key — say, a 128-bit number. Alice scrambles her message with that key using an algorithm like AES; Bob unscrambles it with the same key. Anyone watching the wire sees only random-looking bytes. The hard part is: how did Alice and Bob get the same key in the first place, without ever meeting? That is precisely the problem TLS, SSH, and the rest are designed to solve.

2. Public-key cryptography (two-key pairs)

Each party has two keys: a public key they happily publish, and a private key they guard with their life. Two useful properties:

  • Encryption: Anyone can encrypt a message to you using your public key; only you can decrypt it with your private key.
  • Signatures: Only you can sign a message with your private key; anyone can verify the signature using your public key.

Public-key crypto is slow — too slow to encrypt every byte of a video stream. But it's perfect for one quick step: establishing a shared symmetric key. The pattern is "use slow public-key crypto once to set up a fast symmetric key, then use the symmetric key for the bulk of the data." Every protocol in Topic 4 follows this pattern.

3. Diffie-Hellman key exchange (the math trick)

This is the magic ingredient. Alice and Bob can shout numbers across a crowded room, in front of an eavesdropper, and end up agreeing on a shared secret that the eavesdropper cannot compute. The numbers they shout are not the secret — they are public values that, combined with each party's own private random number, produce the same secret on both sides. The math is modular exponentiation; the elliptic-curve variant (ECDH) is the modern fast version.

Ephemeral Diffie-Hellman (DHE / ECDHE) "Ephemeral" means "lasting only for one session." Alice and Bob each generate a fresh, throwaway DH key pair for this one handshake, agree on a secret, then destroy their throwaway keys. Even if their long-term keys are stolen tomorrow, today's secret cannot be recovered. This property is called forward secrecy, and it is the single biggest reason modern TLS uses ECDHE.

4. Hash functions and MACs

A hash function like SHA-256 takes any input and produces a fixed-length fingerprint (256 bits, in that case). Two key properties: it's infeasible to reverse, and it's infeasible to find two inputs with the same hash.

A MAC (Message Authentication Code) is a hash function with a key mixed in. Without the key, you cannot compute or verify the MAC. So if Alice sends Bob "transfer $100" with a MAC, and Bob recomputes the MAC using their shared key, he can detect any tampering — an attacker who flips a single bit will produce a MAC that doesn't match. The most common construction is HMAC-SHA-256 (HMAC = Hash-based MAC).

Wait, why both encryption and MAC? Encryption hides the contents — but does not detect tampering. An attacker can flip ciphertext bits at random; the receiver decrypts to garbage but doesn't know it's garbage. The MAC is what tells the receiver "yes, this is exactly what was sent." Modern protocols often use a single primitive — AES-GCM — that does both encryption and MAC in one pass; we'll see it in TLS, IPsec, and SSH.

🌍 The big picture: where these three protocols fit

Before we dive into TLS specifically, let's see why we need three protocols. They overlap a lot, but each has a purpose:

TLS — the reference architecture TLS is the general-purpose secure-session protocol. It does not care what application uses it. Web browsers use TLS for HTTPS. Email servers use TLS for SMTPS and IMAPS. Database clients use TLS to talk to databases. If you've ever seen a port number followed by "S" (HTTPS, IMAPS, SMTPS, LDAPS), it's TLS underneath.
HTTPS — TLS plus HTTP HTTPS is not a different protocol from HTTP. It is exactly HTTP, byte for byte, except it travels inside a TLS session instead of directly over TCP. Same GET requests, same status codes, same headers. The only difference: a TLS layer in the middle. Because TLS is so fundamental, "what is HTTPS" really just means "remind me what TLS does."
SSH — secure remote login (and more) SSH was invented before TLS solved every problem, for a specific use case: log into a remote server and type commands. It evolved into a much richer protocol that can also transfer files (SFTP/SCP) and tunnel arbitrary connections. SSH and TLS share the same conceptual foundations — public-key crypto + DH + symmetric record protection — but make different design choices. The most visible difference: TLS uses a global public-key infrastructure (CAs); SSH uses a local "trust on first use" model (known_hosts).
The unifying mental model
All three protocols follow the same blueprint:
  Step 1. Verify who the other side is (authentication via certificates, host keys, etc.).
  Step 2. Establish a fresh shared secret (Diffie-Hellman, run over the just-authenticated channel).
  Step 3. Derive symmetric keys from that secret.
  Step 4. Use the symmetric keys to encrypt + MAC the actual data.

Every difference between TLS, HTTPS, and SSH is a difference in how they execute one of those four steps. Hold this blueprint in mind and the three protocols stop feeling like three separate things to memorise.
4A TLS — Transport Layer Security 📄 Lectures 8 · pp. 207–233  ·  The cornerstone of the topic — read this section slowly

The story: what problem does TLS actually solve?

Imagine you want to buy a book online. You type shop.example.com into your browser and press Enter. Your browser opens a connection to that website's server somewhere on the Internet, and over that connection it sends the contents of your shopping cart, your name, your shipping address, and — most alarmingly — your credit card number.

Without any protection, here is what happens to those bytes on their way to the server:

  1. Your laptop sends a TCP segment to the server. It travels through your Wi-Fi router, your ISP, several backbone routers, the server's ISP, and finally to the server.
  2. Every device on that path can see every byte. TCP and IP add no protection — none. They were designed in the 1970s, when the Internet had a few hundred trusted research users and "security" meant locking the door of the computer room.
  3. Your credit card number sits in plaintext inside the TCP segment, readable by anyone running tcpdump on a coffee-shop Wi-Fi router or a compromised ISP node.

That's bad enough, but there's a worse problem. How do you even know shop.example.com resolves to the right server? The domain-name system (DNS) is itself unauthenticated. An attacker who can poison your DNS responses can make shop.example.com point to their server. Your browser will happily connect to the imposter, you'll happily type in your credit card, and the attacker will happily store it. From your browser's perspective, the connection looked completely normal.

So we need three things, all at once:

  1. Confidentiality. Eavesdroppers must not be able to read your bytes.
  2. Integrity. If someone tampers with bytes in transit, the receiver must detect it.
  3. Authentication. The server you're talking to must actually be shop.example.com — not an imposter.

TLS is a protocol that delivers all three, and it operates as a thin layer between TCP (which gives you a reliable byte stream) and the application (which sees an ordinary socket). Crucially, TLS does its work once per session, at the start, in something called a handshake. After the handshake, your bytes flow through TLS automatically — encrypted and integrity-protected — until the connection closes.

Handshake In networking, a "handshake" is the conversation two computers have at the start of a connection to agree on parameters and prove they are who they claim to be. TCP has a 3-way handshake (SYN, SYN-ACK, ACK). TLS has its own handshake on top of that, with around 7 messages. The handshake is the most complex part of TLS — and the part the exam will test most heavily.

Vocabulary, introduced one at a time

Don't try to memorise this section. Just read each box and let the term register. We'll use these words constantly in the next sections, and seeing them here once gives you a place to come back to.

TLS session vs. TLS connection These two words sound similar but mean different things in TLS, and the textbook is fussy about it.

A TLS session is the cached result of one full handshake — basically, the agreed-upon cipher suite plus a master secret key. It can live for a while (minutes to hours) and be reused.

A TLS connection is one active TCP stream that uses a session's keys to encrypt traffic. A single session can back many connections — for example, when your browser opens 5 parallel TCP streams to a web server, the first does a full handshake and creates a session; the other 4 reuse the same session's keys with a quick "session resumption" instead of repeating the expensive math.
Cipher suite A "cipher suite" is a named bundle that specifies four cryptographic choices the two endpoints must agree on. An example name is TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256, which decodes as:
  • ECDHE — how we'll establish the shared secret (Elliptic-Curve Diffie-Hellman, ephemeral)
  • RSA — what kind of certificate the server will use to authenticate itself
  • AES_128_GCM — what symmetric cipher will encrypt and MAC the data (AES with 128-bit keys, in Galois/Counter Mode)
  • SHA256 — what hash function will be used inside the key-derivation step

Every TLS handshake involves negotiating which cipher suite to use. The client offers a list (its preferences); the server picks one. We'll see exactly how in the walkthrough.
Certificate (X.509) A certificate is a small data file that says: "This public key belongs to this domain name. — Signed, a Certificate Authority you trust."

When you visit shop.example.com, the server sends you its certificate. The certificate contains shop.example.com's public key plus a signature from a Certificate Authority (CA), like Let's Encrypt or DigiCert, attesting that this public key really does belong to shop.example.com. Your browser ships with a built-in list of trusted CA root keys, so it can verify the CA's signature without ever having met the CA itself.

X.509 is the technical format of the certificate file. You don't need to know the byte layout — just remember: certificate = "this public key + this domain name, signed by a CA."
Nonce A nonce (rhymes with "ponce") is a random number used exactly once. In TLS, both the client and the server send a fresh 32-byte random nonce at the very start of the handshake. The nonces feed into the key-derivation function later, ensuring that even if two handshakes used the same shared secret somehow, they'd derive different session keys. This prevents replay attacks where an attacker records a handshake and tries to reuse it later.

Etymology: "nonce" is from "for the nonce," meaning "for this one occasion." Now you know.
Pre-master secret and master secret These are the two intermediate values in TLS's key derivation chain.

The pre-master secret is what the Diffie-Hellman exchange (or RSA key transport) actually produces — a 48-byte shared secret. It is never used directly to encrypt anything.

The master secret is derived from the pre-master plus the two nonces, by hashing them together. The master secret is then used to derive the actual session keys (one for client→server encryption, one for server→client encryption, plus MAC keys and IVs — six keys in total, in TLS 1.2).

Why two layers? Defence in depth: even if some structural flaw lets an attacker recover the master secret, they'd still need both nonces (visible only at handshake time) to recompute the session keys. And the pre-master itself is never reused or stored.
Forward secrecy This is the security property where, even if the server's long-term private key (the one matching its certificate) is stolen tomorrow, an attacker who recorded encrypted traffic today still cannot decrypt it.

TLS achieves forward secrecy when the cipher suite uses ephemeral Diffie-Hellman (DHE or ECDHE) for key exchange. The server's long-term key is used only to sign a fresh, throwaway DH public value; the actual shared secret is computed by both sides using throwaway DH keys that are wiped from memory after the handshake. Stealing the long-term key gives the attacker no way to reconstruct the throwaway keys.

The opposite is "RSA key transport," where the client encrypts the pre-master with the server's long-term public key. If the long-term private key leaks, every recorded session can be decrypted. TLS 1.3 removed RSA key transport for exactly this reason.
Record Once the handshake is done, all subsequent data flows in chunks called TLS records. Each record is up to 16 KB of application data, plus a header (saying "this is record type 23 — application data") and a MAC, all encrypted. The TLS Record Protocol is the post-handshake plumbing that fragments, MACs, and encrypts your data on the way out, and verifies and decrypts it on the way in.

The four "sub-protocols" inside TLS

TLS is internally organised as four little protocols, each identified by a one-byte content-type code in record headers. You will be asked to name them on the exam.

Sub-protocolContent typeWhat it does
Handshake 22 Negotiates the cipher suite, authenticates endpoints, and establishes the master secret. Runs only at the start (and during renegotiation).
Change Cipher Spec 20 A single one-byte message: "starting now, switch to using the new keys we just negotiated." It's a signal, not a negotiation.
Alert 21 Carries warnings and errors — bad certificate, handshake failure, or the graceful "I'm closing the connection" message called close_notify.
Record 23 Carries application data after the handshake. This is what HTTPS GET requests and responses travel as.

You can think of it like this: Handshake is the negotiation, Change Cipher Spec is the "switch flipped" signal, Alert is the intercom for problems, and Record is the everyday delivery channel.

One mnemonic: HARD — Handshake (22), Alert (21), Record (23), and the Data-Change-Spec (20). Whatever helps you remember the four-and-only-four.

The TLS 1.2 handshake, one message at a time

This is the most important section of the entire topic. We will walk through one complete TLS 1.2 handshake, message by message, with a concrete example. The protagonist is Alice's browser; the server is shop.example.com. Every value will be made up but plausible.

Before any TLS happens, Alice's browser must first open a TCP connection. So technically the first thing on the wire is the TCP 3-way handshake (SYN, SYN-ACK, ACK) on port 443. This isn't part of TLS — it's TCP — but exam questions sometimes count it as "step 1" of the overall HTTPS connection. Once TCP is up, TLS begins.

Step 1 — ClientHello (Alice → server) Alice's browser sends the very first TLS message. It contains four important fields:
  • Highest TLS version supported — say, "TLS 1.2."
  • RClient — Alice's 32-byte random nonce. The first 4 bytes are a timestamp (seconds since 1970); the remaining 28 bytes are random. Total: 32 bytes.
  • Session ID — empty, because this is a fresh connection.
  • Cipher suite list — Alice's browser sends an ordered list of the cipher suites it supports, like:
    TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 (preferred)
    TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
    TLS_RSA_WITH_AES_128_CBC_SHA (fallback)
    and a dozen more.
Alice's browser is essentially saying: "Hi! I speak these flavours of TLS, here's a fresh random number to make this session unique, please pick a flavour we both speak." 
WORKED EXAMPLE — ClientHello as bytes (simplified)

R_Client (32 bytes) =
  64 9b 2c 1f                                   ← 4-byte timestamp
  82 a3 7e 1d e5 c4 09 b8 12 3a 6f f0 d4 9c 1e 87
  4b 22 8d a1 6e 5b 31 09 fc 7d                 ← 28 random bytes

Cipher suites offered (in order of preference):
  0xc02f  TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
  0xc030  TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
  0x009c  TLS_RSA_WITH_AES_128_GCM_SHA256
  0x002f  TLS_RSA_WITH_AES_128_CBC_SHA
Wait, why a list of cipher suites? Because TLS has been around for 20+ years. Old servers might support only old ciphers; new servers want to use the newest, fastest, most secure ones. Letting the client list its options and the server pick is how the protocol evolves over time without breaking backward compatibility. The order in the list expresses the client's preference; the server picks the best one it also supports.
Step 2 — ServerHello (server → Alice) The server replies with its choices:
  • Chosen TLS version — "Let's use TLS 1.2."
  • RServer — the server's own 32-byte random nonce.
  • Session ID — a fresh ID it just made up; Alice's browser will cache this so it can reconnect quickly later.
  • Chosen cipher suite — exactly one, picked from Alice's list. Say it picks TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256.
The server is saying: "I'll take ECDHE-RSA-AES128-GCM-SHA256, here's my random number, and remember session ID 0xa3b1... so you can resume me later."
Step 3 — Certificate (server → Alice) The server sends its certificate chain. This is typically three certificates stacked one after the other:
  • Server's own certificate, saying "the public key 0x04AC... belongs to shop.example.com," signed by an intermediate CA.
  • Intermediate CA's certificate, saying "I, intermediate CA, am vouched for by Root CA X."
  • (Sometimes) the root CA's certificate, but Alice's browser already has this in its trust store.
Alice's browser walks the chain: it verifies the intermediate CA's signature on the server cert, then verifies the root CA's signature on the intermediate CA's cert, and finally checks that the root CA's certificate matches one in its built-in trust store. If any signature fails, or any cert is expired, or shop.example.com's name doesn't match the cert's Subject Alternative Name field, the browser shows that scary red "Your connection is not private" page.
Wait, what does the certificate actually prove at this point? By itself, almost nothing. The server has handed Alice a piece of paper saying "this public key belongs to shop.example.com." But anyone can copy that piece of paper from the real server and present it. The certificate only becomes meaningful in the next step, when the server uses the matching private key to sign something fresh. Alice will check that the signature verifies under the public key in the certificate. Only the legitimate server has the matching private key, so a successful signature means "the entity I'm talking to right now actually controls the private key for shop.example.com."

Think of the cert as a passport: handing it to me proves nothing on its own; me asking you to write your name right now and comparing it to the photo's signature line is what authenticates you.
Step 4 — ServerKeyExchange + ServerHelloDone (server → Alice) Because we negotiated ECDHE (ephemeral elliptic-curve Diffie-Hellman), the server now does its half of the DH exchange. It generates a fresh, throwaway EC private key — say, the integer d_S — and computes its public point Q_S = d_S × G, where G is a fixed generator point on the elliptic curve.

Then — and this is the crucial part — it signs the bundle (R_Client ‖ R_Server ‖ Q_S) using its long-term certificate private key (the one matching the public key in the certificate). It sends:
  • The DH parameters (the curve identifier, like secp256r1).
  • Q_S — its DH public point.
  • The signature.
Then it sends ServerHelloDone, a one-line message meaning "I'm done talking for now; your turn."
Wait, why sign the DH public value with the certificate key? This is the single most beautiful idea in TLS. Plain DH gives you a shared secret, but does not tell you who you shared it with — a man-in-the-middle could perform separate DH exchanges with each side and sit between them. By signing Q_S with the cert key, the server proves: "Yes, this DH value really came from the entity that controls shop.example.com's private key." The signature also covers both nonces, so the signature is bound to this specific handshake and cannot be replayed into a different one.

Two birds, one stone: certificate authentication and DH key agreement are tied together by this signature.
Step 5 — ClientKeyExchange + ChangeCipherSpec + Finished (Alice → server) Alice's browser now does several things in quick succession:
  1. Generate Alice's DH keypair. It picks a fresh throwaway integer d_C and computes Q_C = d_C × G. The ClientKeyExchange message contains Q_C.
  2. Compute the pre-master secret. Both sides now hold each other's DH public point. Alice computes shared = d_C × Q_S. The server computes shared = d_S × Q_C. Mathematics ensures both numbers are identical — that's Diffie-Hellman. Convert the shared point's x-coordinate to bytes — that's the 48-byte pre-master secret.
  3. Derive the master secret by hashing pre-master + R_Client + R_Server with the negotiated PRF (a key-derivation function based on SHA-256).
  4. Derive the six session keys from the master secret + nonces.
  5. Send ChangeCipherSpec. One byte saying "from my next message onwards, everything is encrypted with the new keys."
  6. Send Finished. This is the first message under the new encryption. Its content is a MAC computed over every handshake message Alice has sent and received so far. If anyone tampered with any earlier message, Alice's MAC will not match what the server computes, and the server will abort.

sequence-number reuse problems that would arise if both sides shared one key.
Step 6 — ChangeCipherSpec + Finished (server → Alice) The server mirrors what Alice did: it sends its ChangeCipherSpec byte, then its own Finished message — a MAC over every handshake message it has sent and received, computed with the new keys. Alice verifies the MAC against her own running record of the handshake. Mismatch = abort. Match = handshake complete.

What each step accomplished, in one sentence

  1. ClientHello. Tells the server what Alice can do; contributes Alice's nonce.
  2. ServerHello. Picks the cipher suite; contributes the server's nonce.
  3. Certificate. Lets Alice verify the server's claim to be shop.example.com.
  4. ServerKeyExchange + Done. Server contributes its DH public value, signed with its certificate key — binding identity to key exchange.
  5. ClientKeyExchange + CCS + Finished. Alice contributes her DH public value, both sides derive identical session keys, and Alice's Finished MAC proves the handshake wasn't tampered with.
  6. ChangeCipherSpec + Finished (server). Server's mirror of step 5; both sides confirm the handshake's integrity.
Common confusion: "Where's the encryption?" Beginners often expect the server to encrypt something with Alice's public key, or vice versa. That's not how modern TLS works. No long-term key encrypts any handshake data. The certificate's private key is used only to sign the ServerKeyExchange (proving identity); the ephemeral DH keys are used for the actual key agreement; and the symmetric session keys derived from the DH secret are what encrypts the application data afterwards. The certificate's role is identity, not encryption.

This is different from older textbook descriptions of "RSA key transport" TLS, where the client did encrypt a pre-master with the server's RSA public key. That mode is still allowed in TLS 1.2 but discouraged (no forward secrecy) and was removed entirely in TLS 1.3.

After the handshake: how data actually flows

The handshake is over. Both sides hold the same six session keys. Now Alice's browser wants to send GET /products HTTP/1.1\r\nHost: shop.example.com\r\n\r\n. What happens to those bytes?

They go through the TLS Record Protocol, which is just a fancy name for "the pipeline that wraps each chunk of application data in a protective envelope before handing it to TCP." There are five steps in the pipeline:

  1. Fragment. If the data is bigger than 16 KB, split it into 16 KB chunks. (HTTP requests are usually small enough that this doesn't kick in for a GET.)
  2. Compress. Optionally apply a compression algorithm. In practice, this step is almost always disabled because of the CRIME attack (compressing attacker-influenced data leaks length information). So in modern TLS, treat compression as a no-op.
  3. MAC. Compute HMAC(write_MAC_key, sequence_number ‖ record_header ‖ data). Append it to the data. The sequence number is implicit — it isn't sent on the wire; both sides count records and use the count as input to the MAC. This means a record dropped or reordered would produce a MAC mismatch.
  4. Encrypt. Encrypt (data ‖ MAC ‖ padding) using the negotiated symmetric cipher (e.g. AES-128-GCM) with the write_key and a fresh IV.
  5. Prepend record header. Add the TLS record header: type (1 byte = 23 for application data), version (2 bytes = 0x0303 for TLS 1.2), length (2 bytes). The header itself is not encrypted — TCP needs to see it.

The result is a TLS record, which gets handed to TCP and travels to the server.

                using client_write_key + fresh IV
                                                   ▼
              Prepend TLS Record Header:  type=23, version=0x0303, length
                                                   ▼
                                            Hand to TCP

The server receives the record from TCP, reads the header, decrypts the body, verifies the MAC against its own counter, and hands the recovered HTTP request to its application. The whole pipeline is transparent to the application — Alice's browser thinks it's talking to a normal TCP socket.

Wait, what does AES-GCM actually do? AES-GCM is an AEAD cipher — Authenticated Encryption with Associated Data. It does encryption and MAC in a single pass. So in steps 3 and 4 above, when GCM is used, you don't compute HMAC separately — GCM internally produces the integrity tag at the same time as the ciphertext. Modern TLS 1.3 only allows AEAD ciphers; the separate MAC + encryption pipeline is a legacy of pre-AEAD TLS 1.0/1.1 design.

Why is this important? Because AES-GCM is the same primitive used in IPsec ESP (Topic 6) and WPA2 (Topic 1). Once you understand AES-GCM here, you understand it everywhere.
Common confusion: "Why doesn't TLS encrypt the IP and TCP headers?" Because TLS lives above TCP, not below. By the time TLS gets the data, the IP and TCP headers haven't been added yet — and after TLS hands the encrypted record back to TCP, TCP adds its own header in plaintext. Result: the source IP, destination IP, source port, and destination port are all visible to anyone watching the wire. They have to be — routers and firewalls need them to deliver the packet.

This is why someone watching your Wi-Fi can tell you visited shop.example.com (the destination IP, and the SNI extension we'll mention shortly), even though they can't read your shopping cart. Hiding the metadata too is what IPsec and VPNs are for (Topic 6).

"Wait, but what about…?" — frequently confused TLS questions

What's the difference between TLS 1.2 and TLS 1.3?

TLS 1.3, released in 2018, simplified and modernised the protocol significantly:

  • 1 round trip instead of 2. TLS 1.2's full handshake takes about 4 messages (2 round trips) before application data can flow. TLS 1.3 squeezes this into 1 round trip by having the client speculatively send its DH public value in the first message. Sessions can even resume in 0-RTT (data sent before the server has replied) under specific conditions.
  • RSA key transport removed. TLS 1.3 supports only (EC)DHE-based key exchange, so forward secrecy is mandatory.
  • AEAD-only ciphers. The legacy CBC-mode + separate MAC suites are gone. Only AES-GCM and ChaCha20-Poly1305 remain.
  • Cipher suites split into two parts. Instead of one giant name like TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256, TLS 1.3 separately negotiates "the AEAD cipher" (e.g. TLS_AES_128_GCM_SHA256) and "the key exchange + auth" (e.g. ECDHE + RSA cert).

For exam purposes, the textbook focuses on TLS 1.2. The core ideas (handshake, record protocol, certificate authentication, DH key exchange) are identical; only the message ordering and cipher availability differ.

What does the server actually do with the certificate's private key?

The server uses its long-term private key in exactly one place during the handshake: to sign the ServerKeyExchange. That's it. The signature covers (R_Client ‖ R_Server ‖ DH_params ‖ Q_S), tying the server's identity to this specific handshake's DH value.

Notably, the long-term private key does not encrypt any traffic, does not appear in the master secret derivation, and is not needed to decrypt recorded sessions later. This is what gives ECDHE forward secrecy: even if the server's hard drive is seized tomorrow, recorded sessions from yesterday cannot be decrypted.

How does session resumption work?

The server in ServerHello sends a session ID. Alice's browser caches it. Next time Alice connects (within a few minutes to hours), she puts that session ID in her ClientHello. The server looks it up: "yes, I remember this session, here's the cached master secret." Both sides skip the certificate, key exchange, and signature steps — they just exchange new nonces, derive new session keys from the cached master secret + new nonces, and immediately do ChangeCipherSpec + Finished. This is much faster than a full handshake.

TLS 1.3 replaced session IDs with "session tickets" (the server gives the client an encrypted blob containing the session state, so the server doesn't have to store anything itself), but the principle is the same.

What is SNI and why does it matter?

SNI (Server Name Indication) is a small extension in the ClientHello where Alice's browser writes the hostname it intends to talk to: "Hi, I want to connect to shop.example.com." The server uses this to pick which certificate to present, because a single IP address often hosts hundreds of websites, each with its own certificate.

The catch: SNI in TLS 1.2 is in the ClientHello, which is sent before any encryption is set up. So a passive eavesdropper can read it and learn which website you're visiting (just not what you're doing there). TLS 1.3 introduces "Encrypted ClientHello" (ECH) to plug this leak, but adoption is partial.

Why doesn't the client also send a certificate?

It can — that's called "mutual TLS" (mTLS). It's optional in TLS and rarely used for normal web browsing because most websites authenticate users with a username/password sent inside the encrypted tunnel after the handshake. But mTLS is mandatory in some enterprise contexts (e.g. EAP-TLS for WPA2-Enterprise Wi-Fi, where both the user's laptop and the authentication server present X.509 certificates). When you see "client certificate required" in error messages, mTLS is what's being requested.

Q1 Why does the client send a list of cipher suites instead of just one?
▶ Show explanation

So that two endpoints with different supported cipher suites can find common ground. The client lists everything it knows, in preference order; the server picks the best one it also supports. This is how TLS evolves over decades — old clients can talk to new servers (and vice versa) by negotiating a mutually-supported suite. It also lets administrators retire weak algorithms by configuring servers to never pick them, even if old clients still offer them.

Q2 Suppose an attacker records all the bytes of an entire TLS 1.2 handshake using DHE, then steals the server's certificate private key a year later. Can the attacker now decrypt the session's traffic?
▶ Show explanation

No, and this is the entire point of forward secrecy. The certificate's private key was used only to sign the ServerKeyExchange — the actual shared secret was computed from the ephemeral DH values that both sides discarded after the handshake. Without those throwaway DH private keys (which the attacker doesn't have, since they were never sent on the wire), the pre-master secret is unrecoverable. No pre-master, no master secret, no session keys, no decryption.

If, however, the cipher suite had been a static-RSA one (TLS_RSA_WITH_AES_128_CBC_SHA), the answer would flip to "yes, completely" — because the client encrypted the pre-master directly under the server's long-term RSA public key, so anyone with the long-term private key can decrypt it. This is exactly why TLS 1.3 removed static RSA.

Q3 The server's Finished message arrives, but Alice's browser computes a different MAC than the one the server sent. What might have happened?
▶ Show explanation

Three possibilities:

  1. Tampering. Someone on the path modified one of the earlier handshake messages (most commonly, an attacker tried to downgrade the cipher suite by stripping strong options from the ClientHello). The MAC is over all prior handshake messages, so any modification anywhere causes a Finished mismatch.
  2. Bug or transient corruption. Less likely — TCP gives reliable delivery, but software bugs in either party's TLS stack could produce inconsistent records.
  3. Different cipher state. If somehow Alice and the server derived different keys (because of a software fault), the MAC will differ.

In all cases the correct response is to abort the handshake — never proceed with mismatched keys.

Q4 An eavesdropper at a coffee shop watches Alice's TLS connection. List two pieces of information the eavesdropper can still see, despite TLS encryption.
▶ Show explanation

Several, actually. Pick any two:

  • The destination IP address. It's in the IP header, which is below TLS and unencrypted.
  • The destination port (443 for HTTPS). Same reason — TCP header.
  • The hostname (SNI), if the connection is TLS 1.2. The browser puts it in the ClientHello before encryption begins.
  • Approximate sizes of requests and responses, by counting bytes. This can fingerprint specific pages on a known website.
  • Timing patterns. Burst of small encrypted records, then long pause, then a big record, suggests "page load → user reads → submit form."

What the eavesdropper cannot see: the URL path, query parameters, headers, body, cookies, passwords, or anything else inside the encrypted records.

Q5 In one paragraph, explain to a non-technical friend what TLS does and why HTTPS uses it.
▶ Show suggested answer

"When your browser opens a website starting with https://, it secretly does a quick negotiation with the website's server before any actual page loads. During that negotiation — called a TLS handshake — the server proves its identity by showing a certificate signed by a trusted authority, and both sides agree on a one-time secret encryption key using a clever math trick that even an eavesdropper watching the conversation can't unlock. After that, every byte you send (your password, your credit card, the page contents) is scrambled with that key so only the server can read it, and is also fingerprinted so the server can detect any tampering. TLS is the underlying mechanism; HTTPS is just normal web traffic running through TLS."

4B HTTPS — HTTP-over-TLS 📄 pp. 224–226, 233  ·  The shortest section — HTTPS is "TLS but used for web"

The story: HTTPS is just HTTP wearing a TLS coat

If you understood TLS, you've understood 95% of HTTPS. The remaining 5% is plumbing details.

Plain HTTP works like this: your browser opens a TCP connection to a web server on port 80, then sends a text request like:

  GET /products HTTP/1.1
  Host: shop.example.com
  User-Agent: Mozilla/5.0...
  Cookie: session_id=abc123

The server replies with text:

  HTTP/1.1 200 OK
  Content-Type: text/html
  Content-Length: 1234

  <html>...</html>

This is fine for, say, fetching a public weather forecast. It's a disaster for fetching anything private — login forms, banking, email, internal corporate apps. So in 1994 Netscape glued TLS (then SSL) underneath HTTP, used a different default port (443 instead of 80), and called it HTTPS. That's it. That is HTTPS in its entirety.

HTTPS HyperText Transfer Protocol Secure. Defined as: "HTTP, byte-for-byte unchanged, carried as the application-layer payload of a TLS session, running over TCP port 443." No new HTTP syntax. No new headers. No new status codes. Just HTTP, sent through a TLS-encrypted tunnel.

Notice what HTTPS is not:

  • It is not a new protocol with its own request format.
  • It is not "HTTP with extra encryption headers."
  • It is not implemented at the application level (the web server's HTTP code doesn't change at all).

It's the same HTTP, with TLS sitting between HTTP and TCP. Web server software handles this by accepting connections from a TLS library (like OpenSSL); the library does the TLS handshake and decryption, then hands the cleartext HTTP bytes to the web server's normal HTTP-parsing code. The web server doesn't know or care whether TLS was involved.

A complete HTTPS request, end to end

Let's trace exactly what happens when Alice types https://shop.example.com/products into her browser and presses Enter. We'll go through five phases.

Phase 1 — DNS resolution (not part of HTTPS, but happens first) Alice's browser asks her configured DNS resolver: "What's the IP address for shop.example.com?" The resolver replies with, say, 203.0.113.42. (Note: DNS itself is unauthenticated by default — DNSSEC, covered in Topic 8, is the fix. For now, assume DNS is honest.)
Phase 2 — TCP 3-way handshake on port 443 Alice's browser opens a TCP connection: SYN → SYN-ACK → ACK. After this, there's a reliable bidirectional byte stream between Alice's laptop and 203.0.113.42:443. Nothing is encrypted yet.
Phase 3 — TLS handshake (exactly as described in 4A) All seven messages of the TLS 1.2 handshake flow over the TCP connection: ClientHello → ServerHello → Certificate → ServerKeyExchange + HelloDone → ClientKeyExchange + CCS + Finished → ServerCCS + Finished. At the end, both sides hold the six session keys.

Importantly: during the Certificate-verification step, Alice's browser checks that the certificate's Subject Alternative Name field contains shop.example.com. If the server presented a certificate for a different domain, the browser aborts here.
Phase 4 — The actual HTTP exchange (now inside TLS records) Alice's browser builds the HTTP request as plain text: 

GET /products HTTP/1.1
Host: shop.example.com
User-Agent: Mozilla/5.0...
Accept: text/html
Cookie: session_id=abc123
This text is handed to the TLS library on Alice's machine. The library puts it through the Record Protocol pipeline (fragment, MAC, encrypt, prepend record header) and writes the resulting bytes to TCP. On the wire, an observer sees only an opaque encrypted blob.

The server's TLS library decrypts the record and hands the cleartext to the web server. The web server processes the request — looks up products, checks the session cookie, generates HTML — and writes the response as plain text. The server's TLS library encrypts the response into TLS records and sends them back. Alice's TLS library decrypts; her browser parses the HTTP and renders the page.
Phase 5 — Graceful close When Alice's browser is done with the connection (or the server decides to close it), the textbook prescribes a three-step shutdown:
  1. HTTP layer sends a final response (or request) including the header Connection: close.
  2. TLS layer sends a close_notify alert (an Alert-protocol record). This is encrypted and authenticated — only the legitimate peer with the session keys could have produced it.
  3. TCP layer sends FIN/ACK to tear down the underlying TCP connection.
The slide deck calls out an important security implication: if the TCP connection is closed (FIN or RST) without a preceding close_notify, the HTTPS client should treat this as a possible truncation attack — an attacker on the path may have cut the connection mid-response to mislead the client. The browser should warn the user.

Wait, why does the browser keep the connection open for multiple requests? Because TLS handshakes are expensive (DH math, certificate verification, etc.). Reopening a fresh TLS handshake for every CSS file, every image, every API call would slow page loads dramatically. The TLS handshake is paid once per page load, not per resource.

"Wait, but what about…?" — HTTPS questions

Does HTTPS hide which website I'm visiting?

It hides what you do on the website but not which website. The destination IP address is in the IP header, plaintext. The TLS SNI extension (in TLS 1.2) puts the hostname in the ClientHello, plaintext. So an eavesdropper learns "Alice visited shop.example.com" but not "Alice viewed /products/special-offers, added items costing $89, and used her Visa card."

If you also want to hide the destination, you need a VPN or Tor. The condensed reference table in this guide covers the relative protections.

What's the difference between HTTPS and "the site has a padlock"?

The padlock means: HTTPS is being used, the certificate validated, and your browser is satisfied with the cipher suite. It does not mean: the site is trustworthy, the operator is legitimate, your data won't be sold, or that there isn't malware on the page. A phishing site can have a perfectly valid certificate for shop-example-com.tk, get its padlock, and still steal your password. The padlock is a transport-layer guarantee, not an application-layer one.

Why is plain HTTP still allowed at all?

Because of the long tail of legacy systems and the operational cost of certificates. For decades, certificates were expensive and required manual renewal. The Let's Encrypt project (2015-) made them free and automatic, so most public web traffic is now HTTPS. Some private internal services still use HTTP because they're inside a trusted network. Modern browsers display warnings on plain-HTTP pages with password fields, encouraging migration.

Does HTTPS work over HTTP/2 or HTTP/3?

Yes, both. HTTP/2 still runs over TLS-over-TCP-port-443; it's a different binary framing of HTTP messages but the security layer is identical. HTTP/3 is more interesting: it runs over QUIC, a protocol that combines TCP-like reliability and TLS-1.3-like security into one layer, on top of UDP. QUIC integrates the TLS handshake into its own connection setup, removing some round trips. For the exam, the textbook focuses on HTTP/1.1 over TLS over TCP — the classic "HTTPS" scenario.

Why is HTTPS still vulnerable to certain attacks?

HTTPS is one layer in a stack. It can fail when:

  • The CA system is compromised. If a Certificate Authority is hacked (DigiNotar, 2011), an attacker can issue a valid certificate for any domain.
  • The user clicks past a warning. If a self-signed cert or an invalid one is presented, browsers warn — but users can override.
  • The application has bugs. SQL injection, XSS, broken authentication — none of these are TLS's problem; TLS just delivers the request faithfully.
  • The user's device is compromised. Malware can read decrypted bytes after they leave TLS.
  • DNS is poisoned and the user doesn't notice the mismatch. Less common with modern browsers and DNSSEC, but historically a vector.

Q4 A friend says "I don't trust HTTPS — what if the server certificate was forged?" How would you reassure them, and what residual risk is real?
▶ Show answer

Reassurance: Forging a certificate requires forging a CA signature, which would require either compromising a trusted CA's private key or breaking the cryptographic algorithm (RSA or ECDSA). Both are extremely hard. The CA system has had failures (notably DigiNotar in 2011), but they're rare enough that browsers' built-in trust stores catch them when noticed.

Residual risk: A nation-state attacker who controls a CA in some jurisdiction could issue a fake cert for a domain. This is real but uncommon. Modern defences include Certificate Transparency (logs of all issued certs, where domain owners can spot forgeries) and DNS-based Authentication of Named Entities (DANE), which lets a domain owner publish via DNSSEC the exact certificate it expects to use. For everyday use, HTTPS is solid; for high-assurance applications, additional layers help.

4C SSH — Secure Shell 📄 pp. 246–258  ·  A different architecture from TLS — pay attention to the differences

The story: how do system administrators log in remotely?

Imagine you're a system administrator in 1995. You manage a Unix server in another building. You need to log in, edit a configuration file, restart a service, and log out. The standard tools are telnet and rlogin, both of which transmit your username and password as plaintext. Anyone running a packet sniffer on the network can collect admin credentials all day long. This was an actual catastrophic problem.

In 1995, Tatu Ylönen wrote SSH (Secure Shell) as a drop-in replacement for telnet/rlogin/rsh that encrypted everything. It worked, it spread, and the protocol evolved into SSH-2 (the version we use today). Along the way it grew capabilities far beyond remote shell:

  • Remote shell (the original use case): type commands on a server as if you were sitting at it.
  • File transfer via SFTP and SCP, both of which run inside an SSH session.
  • Port forwarding / tunnelling: "I'm on a server I can SSH to; let me tunnel a connection through that server to reach a third machine that I can't reach directly." This is widely used to bypass firewalls and access internal services.
  • X11 forwarding: run a graphical Linux application on a remote server but display the window on your local machine.
  • Git operations over SSH (git@github.com:user/repo.git).

SSH is conceptually similar to TLS — both establish authenticated encrypted sessions using DH key exchange and symmetric encryption — but they make different design choices in two important places: how the server is authenticated, and what happens after authentication. The differences are exactly what the exam will test you on.

SSH Secure Shell. An application-layer protocol, typically over TCP port 22, that provides encrypted remote login and a multipurpose secure-tunnel framework. Unlike TLS, SSH was designed from the start to handle multiple parallel activities over a single secure connection — shell, file transfer, port forwards — and to authenticate the user separately from the server.

SSH's three internal layers, explained one at a time

SSH is internally organised into three sub-protocols stacked on top of one TCP connection. The slide deck calls these the SSH "three-layer architecture." This is the highest-frequency exam question in 4C — you must be able to name all three and say what each one does.

LayerNameWhat it does
Top Connection Protocol Multiplexes many independent channels (shell, SFTP, port-forwards, X11) over the one authenticated tunnel.
Middle User Authentication Protocol Authenticates the human user (or automated service) using password, public key, or hostbased methods.
Bottom Transport Layer Protocol Authenticates the server, runs Diffie-Hellman, derives session keys, encrypts everything above.

The order matters: the bottom layer runs first. Until it finishes, no encryption exists, and the upper layers cannot run. Once the Transport Layer is up, encryption protects everything that follows — including the User Authentication exchange. This is why "password authentication in SSH sends the password through the encrypted tunnel" — the password packet is encrypted because the Transport Layer Protocol already established encryption before User Auth ran.

Layer 1: SSH Transport Layer Protocol — what it actually does

This is the layer that mirrors TLS most closely. Its job: authenticate the server, perform DH key exchange, derive symmetric session keys, and encrypt+MAC everything from this point forward. The four sub-steps:

A — Protocol-version exchange and algorithm negotiation As soon as the TCP connection is established, both sides exchange ASCII banners like SSH-2.0-OpenSSH_8.4. Then they exchange lists of supported algorithms for key exchange (e.g. curve25519-sha256, diffie-hellman-group14-sha256), encryption (e.g. aes128-gcm@openssh.com), MAC (often integrated with AEAD), and compression. Each side picks the best algorithm both support. This is conceptually the same as TLS's cipher-suite negotiation; it's just expressed differently.
B — Server authentication via host key The server sends its long-term host public key. The client now has a problem: how do I know this is really the server I meant to connect to, not an imposter?

TLS solved this with a global CA hierarchy. SSH solves it differently: the client maintains a local file called ~/.ssh/known_hosts that maps each previously-visited hostname to its server's public key. The client looks up the server's hostname:
  • Match found, key matches → silently proceed.
  • First-time connection (host not in file) → prompt the user: "The authenticity of host shop.example.com (203.0.113.42) can't be established. ED25519 key fingerprint is SHA256:abc123… Are you sure you want to continue connecting (yes/no)?" If user says yes, the key is added to known_hosts. This is called Trust-On-First-Use (TOFU).
  • Match found, but key has CHANGED → loud warning: "@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
    @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
    It is also possible that a host key has just been changed."     The connection is refused unless the user manually fixes known_hosts. This is the on-wire signal of a man-in-the-middle attack (or a legitimate server-key rotation; the user must judge).
Wait, why not use CAs like TLS? Three reasons. First, SSH was invented before the CA system was ubiquitous, so it had to bootstrap differently. Second, SSH's typical use case is administrators connecting to servers they already know about — there isn't a "discoverability" problem the way there is on the public web. Third, SSH's designers explicitly didn't want the global single-point-of-failure that comes with CAs. If a CA is compromised in the TLS world, all certificates are forgeable. With SSH's TOFU model, a compromised CA can't fake your specific server because there's no CA to compromise.

The trade-off is that TOFU is weakest on the first connection. If an attacker is on-path the very first time you SSH to a new server, their fake key gets cached, and all future connections will silently accept it. The defence is to verify the fingerprint out-of-band the first time — for example, the server admin tells you over the phone: "the SHA256 fingerprint should be abc123..."
C — Diffie-Hellman key exchange Both sides run a fresh DH (typically Curve25519 or NIST P-256) to derive a shared secret. The server signs its DH public value with the host private key, the same way TLS does — this binds the server's authenticated identity to the DH exchange, defeating man-in-the-middle. Both sides then derive the symmetric session encryption + MAC keys from the DH shared secret.
D — Switch to encryption Both sides send a SSH_MSG_NEWKEYS message (the SSH analogue of TLS's ChangeCipherSpec). All subsequent SSH packets are encrypted and MAC-protected with the new keys.

And here's a specific detail the lecture slide highlights: SSH rekeys every 1 GB of data transferred or every 1 hour, whichever comes first. A long-running session triggers a fresh DH exchange and new keys at those thresholds, bounding the cryptanalytic exposure of any single key pair.
Host key Each SSH server has a long-term public/private key pair, generated when the SSH server software is first installed. The public key is what gets stored in clients' known_hosts files; the private key is used to sign DH values during the handshake. There's typically one key per supported algorithm (RSA, ECDSA, Ed25519), so /etc/ssh/ on the server contains files like ssh_host_ed25519_key and ssh_host_ed25519_key.pub.

Layer 2: SSH User Authentication Protocol — three flavours

The Transport Layer is up. Encryption is on. Now it's time to authenticate the user. SSH defines three methods; servers can support any combination, and clients try them in order until one works.

Method 1 — Password authentication The client sends a packet containing the user's password. Because the Transport Layer is already encrypting everything, the password travels as ciphertext on the wire — not as plaintext like in telnet. The server checks the password against /etc/shadow (or whatever authentication backend it uses) and replies success or failure.

This is the simplest method but has drawbacks: passwords can be brute-forced (mitigated by rate-limiting), can be guessed, and require the user to type them every time.
Method 2 — Public key authentication (preferred) The user has a public/private key pair stored locally — typically in ~/.ssh/id_ed25519 and ~/.ssh/id_ed25519.pub. The user copies their public key to the server, where the admin appends it to ~/.ssh/authorized_keys for that user account.

Authentication then works like this: the client sends a "USERAUTH_REQUEST" with method = publickey, including the user's public key. The server checks if that key is in the user's authorized_keys. If so, it sends a challenge — some session-bound bytes including the SSH session ID. The client signs the challenge with its private key. The server verifies the signature against the public key.

No password ever crosses the wire. No reusable secret is stored on the server (only the user's public key, which is harmless to leak). And the challenge is bound to this specific session, so a captured signature can't be replayed in a different session.
Method 3 — Host-based authentication The client signs a challenge using the client machine's private host key (not the user's key). The server checks that (a) the client host's public key is in /etc/ssh/ssh_known_hosts, and (b) the user is mentioned in ~/.shosts or /etc/hosts.equiv.

This is a "trust the source machine" model — useful in tightly-controlled environments where you trust everyone with login on certain hosts to act as themselves. Rare in modern practice; you'll see it on multiple-choice questions but probably not in real life.
Wait, so SSH password authentication isn't insecure? Correct — and this is a common confusion. The password is sent inside an encrypted, authenticated tunnel established by the Transport Layer. An eavesdropper sees only ciphertext. The reasons to prefer public-key auth aren't about wire-level security; they're about:
  • No reusable secret on the server side. A compromised server can't reveal users' passwords (because the server only has public keys).
  • Defence against weak passwords. A 4096-bit RSA key has no "weak password" failure mode.
  • Convenience for automation. Scripts can SSH-in without prompting for a password, using key-based auth + SSH agents.
But "SSH with password auth sends credentials in the clear" — this is a common wrong exam-trap statement. It's false.

Layer 3: SSH Connection Protocol — channels and tunnels

Now the user is authenticated. Time for the third layer, which is what makes SSH genuinely powerful beyond TLS. The Connection Protocol can multiplex many independent logical channels over the single authenticated tunnel. Four channel types are defined in the protocol:

Channel typePurposeReal-world use
session Run a shell or a single command on the server. The default — typing ssh user@host opens this.
x11 Forward graphical applications: server-side X clients display on the client's X server. ssh -X user@host firefox — Firefox runs on the server, displays on your laptop.
direct-tcpip Local port forwarding. Traffic to a port on your machine is tunnelled through SSH to a target the server can reach. ssh -L 8080:internal-db:5432 jump-host — connect to localhost:8080 on your laptop and reach the internal database via the jump host.
forwarded-tcpip Remote port forwarding. A port on the server is forwarded back to a target on the client side. ssh -R 9000:localhost:22 home-server — opens port 9000 on the home server that tunnels back to your laptop's SSH server. Useful for connecting back home from inside a NAT.

Multiplexing means: while you're typing in your shell, you can simultaneously have an SFTP transfer running, plus a tunnelled web connection, plus an X11 forward — all over the same TCP connection, the same encryption keys, with traffic for each interleaved by SSH frame headers identifying the channel ID. This is conceptually similar to HTTP/2's stream multiplexing, but built right into SSH's architecture from the start.

A complete SSH session, end to end

Concrete example: Alice wants to log into server.example.com as user alice to run some commands. She types:

$ ssh alice@server.example.com

Here's what happens, blow by blow.

Phase 1 — TCP connection Alice's SSH client opens a TCP connection to server.example.com:22. Standard TCP 3-way handshake.
Phase 2 — Banner exchange Both sides send ASCII banners over the (still unencrypted) TCP stream: 
Client → Server:  SSH-2.0-OpenSSH_9.6
Server → Client:  SSH-2.0-OpenSSH_9.6p1 Ubuntu-3
The banners advertise SSH protocol version 2 (the modern version) and the implementation versions.
Phase 3 — Algorithm negotiation Each side sends a SSH_MSG_KEXINIT packet listing supported algorithms for: key exchange, host-key signing, encryption (in each direction), MAC (in each direction), and compression (in each direction). They negotiate the best mutually-supported choice for each.
Phase 4 — Server authentication + DH The server sends its host public key. Alice's client looks it up in ~/.ssh/known_hosts:
  • If found and matches: silently proceed.
  • If not found: prompt Alice. "The authenticity of host server.example.com can't be established. ED25519 key fingerprint is SHA256:abc... Continue?" If yes, save to known_hosts.
  • If found but mismatch: scary warning, refuse to continue.
Then both sides run DH to derive a shared secret, with the server signing its DH public value using its host private key. Both derive symmetric session keys.
Phase 5 — Switch to encryption Each side sends SSH_MSG_NEWKEYS. From this point on, every SSH packet is encrypted and integrity-protected.
Phase 6 — User authentication Alice's client sends SSH_MSG_USERAUTH_REQUEST with username = "alice" and method = "publickey", offering Alice's public key from ~/.ssh/id_ed25519.pub.

Server checks: is alice's account valid? Is the offered public key in alice's ~/.ssh/authorized_keys? If yes, server sends SSH_MSG_USERAUTH_PK_OK: "yes, that key would work — sign this challenge to prove you have the private key."

Alice's client signs the challenge (a hash including the SSH session ID, to bind the signature to this specific session) with her private key, sends it back. Server verifies. Sends SSH_MSG_USERAUTH_SUCCESS.

If public-key auth had failed (key not in authorized_keys, or user changed their mind), the client would have fallen back to password authentication, prompting Alice to type her password.
Phase 7 — Open a shell channel Alice's client sends SSH_MSG_CHANNEL_OPEN with channel type = "session". Server replies SSH_MSG_CHANNEL_OPEN_CONFIRMATION.

Then the client sends SSH_MSG_CHANNEL_REQUEST with request type = "pty-req" (allocate a pseudo-terminal so the server-side shell can show progress bars and colours), then another SSH_MSG_CHANNEL_REQUEST with type = "shell" (start a login shell).

The server starts /bin/bash as user alice, attached to the pty. Bash prints a prompt. The prompt bytes flow back to Alice's terminal as a SSH_MSG_CHANNEL_DATA.
Phase 8 — Interactive use Every keystroke Alice types is wrapped in a SSH_MSG_CHANNEL_DATA for channel 0 and sent to the server. Every byte the shell prints comes back the same way. Alice can also open additional channels mid-session (e.g. start an SFTP transfer in another terminal — that opens a new channel within the same SSH connection).

Behind the scenes: every 1 GB of data or every 1 hour, both sides quietly run a fresh DH and switch to new keys.
Phase 9 — Close Alice types exit or presses Ctrl-D. Bash exits, the server sends SSH_MSG_CHANNEL_CLOSE. When all channels are closed, the connection's transport layer sends SSH_MSG_DISCONNECT and the underlying TCP connection is torn down.

Common confusion: "The TOFU model is just as secure as the CA model." Not quite. TOFU has a specific weak point: the first connection to a new host. If an attacker is on-path right at that moment, their fake key gets cached, and all future "verifications" silently match the fake key. The defence is to verify the fingerprint out-of-band the first time — a phone call, a printed datasheet, a secure messaging app.

TLS's CA model has the opposite failure mode: each individual connection is well-defended (certificates are checked every time), but the CA system itself is a global single point of failure. A compromised CA can issue a perfect-looking fake cert for any domain. Both models trade off in different directions; neither is strictly "better."

"Wait, but what about…?" — SSH questions

Why does SSH have two authentication phases?

One authenticates the server; the other authenticates the user. They're independent concerns. Server authentication says: "yes, this is server.example.com and not an imposter." User authentication says: "yes, the human (or service) connecting is who they claim to be." Both are needed, but they're separate problems with separate solutions.

TLS technically has both too, but only the server-auth half is mandatory; user auth is optional (mTLS) and rarely used in browser contexts. SSH treats both as first-class.

What's the difference between ssh-keygen and the host key on the server?

Both create public/private key pairs. The difference is what they're for.

ssh-keygen on a user's machine creates the user's key pair (e.g. id_ed25519) — used during User Authentication. The user's public key gets copied into ~/.ssh/authorized_keys on each server they want to log into.

The server's host key pair (e.g. ssh_host_ed25519_key in /etc/ssh/) is generated when SSH server software is first installed. It's used during Transport Layer authentication. Its public half goes into clients' ~/.ssh/known_hosts when they first connect.

Why does ssh -L 8080:internal-db:5432 jump.example.com work?

This is local port forwarding via direct-tcpip channel. Step by step: when you type that command, SSH opens a connection to jump.example.com (with all the usual handshake/auth/tunnel setup), then on your local machine starts listening on port 8080. Any time something connects to your localhost:8080, SSH opens a direct-tcpip channel inside the existing tunnel, with the channel parameters saying "destination = internal-db:5432". The traffic flows through SSH to jump.example.com, where the SSH server opens a new TCP connection to internal-db:5432 and proxies bytes between the two. From your perspective, localhost:8080 magically connects to internal-db:5432, even though only jump.example.com can route to internal-db.

Why is SSH still relevant when TLS exists?

Because SSH solves a different problem: interactive remote login + multipurpose tunnelling, with first-class user authentication and channel multiplexing. TLS is great for fixed protocols (HTTP, IMAP) where one side is a server and one is a client. SSH is great when you want a flexible authenticated tunnel that can host any number of activities. Both are still actively developed; both have their place.

Self-quiz — SSH

Q1 Name SSH's three sub-protocols and what each does, without scrolling back.
▶ Show answer

Transport Layer Protocol — server authentication via host key, DH key exchange, derives symmetric session keys, encrypts everything from this point forward, rekeys every 1 GB / 1 hour.

User Authentication Protocol — runs inside the encrypted tunnel; authenticates the user via password, public key, or hostbased methods.

Connection Protocol — multiplexes many independent logical channels (session, x11, direct-tcpip, forwarded-tcpip) over the single authenticated tunnel.

Q2 Alice connects to a server she's never used before. Her SSH client says: "The authenticity of host can't be established. Continue?" What is happening, and what is the safe response?
▶ Show answer

This is the Trust-On-First-Use prompt: the server's host key isn't yet in Alice's known_hosts file. The SSH client is asking her to verify out-of-band that the fingerprint matches what she expects.

Safe response: verify the fingerprint another way before saying yes. The server's admin can publish the SHA-256 fingerprint on a website (under HTTPS!), in a printed handout, or in a secure message. If Alice blindly types yes without verification, she's vulnerable to a man-in-the-middle who could be presenting their own key on the very first connection.

If she has no out-of-band channel, accepting on-trust is the practical choice — the same first-connection vulnerability exists in countless real-world SSH workflows. But the right answer on an exam is "verify the fingerprint out of band."

Q3 An attacker on the same Wi-Fi captures all bytes of Alice's SSH session as she logs in with password authentication. Can the attacker recover her password? Why or why not?
▶ Show answer

No. The Transport Layer Protocol established encryption before the User Authentication Protocol ran, so by the time Alice's password is sent, the SSH connection is already encrypted with session keys derived from the just-completed DH exchange. The attacker sees only ciphertext.

Public-key auth is preferred over password auth for other reasons (no reusable secret on the server, defence against weak passwords, automation friendliness), but "password sent in cleartext" is not one of those reasons. That's a common exam-trap claim — and it's false.

Q4 A user runs ssh -L 8080:database.internal:5432 jump.example.com. After connecting, they connect a database client to localhost:8080 and successfully query the internal database. Explain in 2-3 sentences how the bytes get there.
▶ Show answer

The SSH client establishes a normal SSH connection to jump.example.com (TCP, transport layer auth, user auth — all the usual setup). It then opens a local TCP listener on port 8080 of the user's machine. When the database client connects to localhost:8080, the SSH client opens a "direct-tcpip" channel inside the existing SSH tunnel with destination "database.internal:5432". The SSH server on jump.example.com sees the channel-open request, opens a new TCP connection to database.internal:5432, and starts proxying bytes between the channel and the new connection. From the database client's view it's connected to localhost:8080; from the database server's view it's connected from jump.example.com. The two are stitched together by SSH's channel multiplexing.

Q5 Compare TLS and SSH on three axes: (a) where they sit in the protocol stack, (b) how they authenticate the server, (c) how they handle multiple parallel activities.
▶ Show answer

(a) Stack position. TLS sits between TCP and the application — applications are TLS-aware via a library. SSH is itself an application: it runs over TCP, and its job is to host other things (shells, SFTP, channels) inside.

(b) Server authentication. TLS uses X.509 certificates signed by trusted CAs (a global PKI). SSH uses raw host public keys cached locally in ~/.ssh/known_hosts, with Trust-On-First-Use on the very first connection.

(c) Parallelism. A single TLS connection carries one logical activity (one HTTP request stream, say). HTTP/2 multiplexes many requests over one TLS stream, but that's an HTTP-layer feature. SSH bakes channel multiplexing right into the protocol — the Connection Protocol can run any number of channels (session + sftp + several port forwards) in parallel inside one authenticated tunnel.

🧩 Tying it all together

You've now seen all three protocols in detail. Let's pull back and see the unified pattern they share, and understand why each one made the design choices it did.

The blueprint, revisited

At the very beginning, we said all three protocols follow the same four-step blueprint:

  1. Verify who the other side is (server auth).
  2. Establish a fresh shared secret (Diffie-Hellman).
  3. Derive symmetric keys from that secret.
  4. Use the keys to encrypt and MAC the actual data.

Now you can fill in how each protocol implements each step:

StepTLSHTTPSSSH
1. Verify the server X.509 certificate signed by trusted CA Same as TLS (HTTPS = HTTP over TLS) Server host key matched against local known_hosts
2. Shared-secret exchange (EC)DHE in ServerKeyExchange + ClientKeyExchange Same (EC)DH in Transport Layer Protocol
3. Derive symmetric keys PMS → MS → 6 session keys via SHA-256 PRF Same Similar PRF construction, fewer named layers
4. Bulk encrypt + MAC TLS Record Protocol (AEAD: AES-GCM) Same — payload happens to be HTTP SSH Binary Packet (AEAD too in modern OpenSSH)

The shared cryptographic primitives are striking. AES-GCM appears in all three protocols' record/packet encryption. SHA-256 is the default hash and PRF basis. (EC)DHE is the default key exchange. The differences are essentially in where trust comes from (CAs vs known_hosts) and how the protocol is layered (TLS as a library, SSH as an application with three sub-protocols, HTTPS as TLS hosting HTTP).

Why each protocol made its design choices

TLS uses CAs because it has to scale to billions of unknown websites. Browsers cannot pre-trust every individual server; they need a delegated trust system, even with its known weaknesses. The CA system is a practical compromise.

SSH uses TOFU because it doesn't have to scale that way. System administrators connect to servers they already know about. There's no Google-of-SSH; there's your servers. Local trust on first use is a reasonable model when both sides know they're going to be talking to specific, named hosts.

HTTPS exists because TLS solved the right problem. Web traffic was the first big use case where lots of unknown clients needed to securely talk to lots of unknown servers. TLS, with the CA infrastructure, was the right tool. Slapping HTTP on top was the easy part — and that's all HTTPS really is.

SSH has channel multiplexing because remote login isn't enough. Once you have an authenticated tunnel, why not run any number of streams over it? File transfers, port forwards, X11. The Connection Protocol generalizes the original "one shell session" use case into a flexible tunnelling primitive.

Where these protocols connect to other course topics

This is the most likely source of "challenge questions" on the final exam — questions that combine Topic 4 with another topic.

Topic 1 (Wireless): EAP-TLS in WPA2-Enterprise WPA2-Enterprise uses 802.1X to authenticate Wi-Fi clients. One EAP method, EAP-TLS, runs a complete TLS handshake inside EAP packets. Both the client and the AAA server present X.509 certificates (mutual TLS). The TLS master secret then becomes input to the WPA2 4-way handshake's pairwise master key. So if you understand TLS and you understand WPA2 separately, you can answer "how does EAP-TLS work" by combining them.
Topic 6 (IPsec/VPN): TLS-based VPNs OpenVPN, Cisco AnyConnect, and similar tools tunnel arbitrary IP traffic through a TLS session. So a "TLS VPN" is a TLS connection running between a client and a VPN concentrator, with the application data being raw IP packets instead of HTTP. Understanding TLS gives you 90% of TLS-VPN.

The contrast with IPsec VPNs is informative: IPsec lives at Layer 3 and is transparent to applications; TLS-VPN lives at Layer 7 and requires a virtual network interface (tun0). Both achieve similar end results (encrypted tunnel with traffic-flow confidentiality) by very different means.
Topic 8 (DNS Security): Chains of trust TLS, DNSSEC, and SSH all use public-key cryptography but anchor trust differently. TLS roots in a global CA hierarchy. DNSSEC roots in the DNS root zone keys, signed in a hierarchical pattern matching the DNS namespace. SSH roots in known_hosts, locally. A common challenge question asks you to compare these three trust models and identify the failure modes unique to each.
Topic 2 (Common attacks): What TLS does and doesn't defend TLS prevents passive eavesdropping and active man-in-the-middle on the connection it protects. It does not prevent:
  • SQL injection or XSS — those are application-layer flaws.
  • DNS poisoning at the resolution stage — TLS catches the imposter via cert validation, but the resolution itself is unauthenticated unless DNSSEC is used.
  • Compromised endpoints — malware on Alice's laptop reads decrypted bytes after they leave TLS.
  • Side-channel attacks like traffic analysis — sizes and timings still leak.
This is "defence-in-depth" thinking: TLS solves one problem; you need other tools (firewalls, application-layer auth, secure coding, IDS) to solve the others.

Record (TLS)The unit of data flow in TLS post-handshake. Each record is up to 16 KB of application data + a header + a MAC, all encrypted. Carries content types 20-23.
Record protocol (TLS)One of TLS's four sub-protocols (content type 23). Fragments, MACs, and encrypts application data into records.
RekeyingPeriodically deriving fresh session keys to bound cryptanalytic exposure. SSH rekeys every 1 GB or 1 hour; TLS rekeys via session resumption or new connections.
Replay attackAn attacker captures legitimate messages and re-sends them later. Defeated by including nonces (in TLS handshakes) and sequence numbers (in TLS records and IPsec).
RSAA public-key algorithm based on the difficulty of factoring large numbers. Used in TLS for: certificates (signing), key transport (legacy), and signatures over key exchange (modern). Slowly being replaced by ECDSA / Ed25519 for new deployments.
Security Association (SA)An IPsec/Topic-6 concept that doesn't appear in TLS or SSH directly. Mentioned here only because it's the IPsec analogue of "TLS session" — a cached cryptographic context.
Server authenticationThe act of proving to the client that the server is the legitimate operator of the named service. TLS uses certificates; SSH uses host keys.
ServerHelloThe second message in a TLS handshake. Contains the chosen TLS version, the server's 32-byte nonce, the chosen cipher suite, and the chosen compression.
ServerKeyExchangeA TLS handshake message in which the server contributes its DH public value (when using DHE/ECDHE), signed with the certificate's private key.
Session (TLS)The cached result of one TLS handshake — cipher suite + master secret. Can back many connections via session resumption.
Session IDAn opaque identifier the server assigns to a TLS session. Clients can present it on later connections to request session resumption.
Session keysSymmetric keys derived from the master secret and used for one TLS session. In TLS 1.2 there are six (client write key, server write key, client write MAC, server write MAC, client write IV, server write IV).
SignatureA cryptographic operation: the holder of a private key produces a tag that anyone with the matching public key can verify. Proves both authenticity (it came from the holder) and integrity (the data wasn't modified).
SNIServer Name Indication. A TLS ClientHello extension where the client states the hostname it's trying to reach. Allows one IP to host many TLS-enabled sites. Plaintext in TLS 1.2 (privacy leak); ECH addresses this in TLS 1.3.
SSHSecure Shell. An application-layer protocol on TCP 22 for encrypted remote login, file transfer, and tunnelling. Three sub-protocols: Transport Layer, User Authentication, Connection.
Symmetric encryptionEncryption where both parties share the same key. Fast (AES is on the order of GB/sec). Contrasts with public-key (asymmetric) encryption, which is slow but doesn't require pre-shared keys.
TLSTransport Layer Security. A cryptographic protocol providing confidentiality, integrity, and authenticity for a TCP connection. The successor to SSL. Versions 1.2 (2008) and 1.3 (2018) are in active use.
TLS Record ProtocolThe post-handshake plumbing of TLS. Fragments, MACs, encrypts each record before handing to TCP.
Transport Layer Protocol (SSH)SSH's bottommost sub-protocol. Authenticates the server (via host key), runs DH, derives session keys, encrypts subsequent traffic, rekeys every 1 GB / 1 hour.
Truncation attackAn attacker tears down the TCP connection without letting close_notify travel, hoping to cut a response off mid-flow. Why TLS clients should warn the user when TCP closes without a prior close_notify.
Trust-On-First-Use (TOFU)SSH's trust model: on the first connection to a new server, the client presents the host key fingerprint to the user for confirmation. Once cached in known_hosts, future connections silently verify against the cache.
User Authentication Protocol (SSH)SSH's middle sub-protocol layer. Authenticates the user using one of three methods: password, public key, or hostbased. Runs inside the encrypted Transport Layer.
X.509The standard format for digital certificates. A binary format encoded in ASN.1 DER. Contains: subject (who the cert is for), issuer (which CA signed it), validity period, public key, extensions, and the CA's signature.
