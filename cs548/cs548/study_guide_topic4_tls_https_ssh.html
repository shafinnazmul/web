4A TLS — The Reference Secure Session Protocol 📄 Lectures 8 · pp. 207–233  ·  🔴 EXPLICIT ⭐

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 207–233

1.1 The Problem This Solves 📄 pp. 215, 228

TCP delivers bytes reliably between two endpoints, but anyone on the path can read or modify those bytes, and neither endpoint has cryptographic proof the other is who they claim to be. TLS (Transport Layer Security) fills that gap: it negotiates fresh session keys, authenticates the server (and optionally the client) via X.509 certificates, then encrypts and integrity-protects every subsequent byte until the session closes. 📄 pp. 215, 228

1.2 The Core Idea — Two Layers, Four Components 📄 pp. 215, 227, 238

TLS sits between TCP and the application and is split into two layers:

  • Upper (Handshake) layer — short-lived, responsible for negotiating crypto parameters and authenticating endpoints. Has three sub-protocols:
    • Handshake Protocol (content-type 22) — negotiates crypto for a "TLS session" that may be reused for many "TLS/TCP connections." 📄 pp. 215, 227, 229, 238
    • Change Cipher Spec Protocol (content-type 20) — one-byte message: "from the next record onward, use the newly negotiated keys." 📄 pp. 215, 227, 238
    • Alert Protocol (content-type 21) — conveys warnings/errors (bad certificate, close_notify, etc.). 📄 pp. 215, 227, 238
  • Lower (Record) layer:
    • Record Protocol (content-type 23) — carries all application data after handshake; provides confidentiality (symmetric encryption) and integrity (MAC). 📄 pp. 215, 227–228, 241

A "TLS session" is a cached negotiation (cipher suite + master secret); a "TLS connection" is one active TCP stream using that session. One expensive handshake → many cheap connections. 📄 pp. 215, 227

1.3 Analogy — Embassy Visa Interview 📄 pp. 229–232

Think of TLS as the security dance at an embassy:

  • ClientHello = you arriving and saying "Hi, I speak English/French/Mandarin; I prefer French."
  • ServerHello + Certificate = the visa officer replies "Let's use French," and hands you their government ID card signed by the national seal (CA).
  • You verify the ID against your trusted list of national seals (browser's root CA store).
  • Key exchange = you both mix a shared secret through the window using a math trick (Diffie-Hellman) that even a recorder outside the room cannot undo later.
  • Finished messages = you both repeat back everything said so far ("here's a cryptographic receipt of the entire conversation") so nobody tampered in the middle.
  • Record layer = now that you share a secret phrase, every later sentence is whispered through the phrase-book.

1.4 Technical Definition 📄 pp. 215, 228, 229

TLS Handshake Protocol allows a client and server to (a) authenticate each other, (b) negotiate encryption and MAC algorithms, and (c) negotiate cryptographic keys — via a phased series of TLV-encoded messages. TLS Record Protocol takes each outbound application-layer message, optionally compresses it, appends a keyed MAC, encrypts the result with a symmetric cipher (AES-128, AES-256, 3DES, or RC4-128) using the session's write keys/IV, and hands the record to TCP. 📄 pp. 228, 229, 241

Section 2 · How It Works — The TLS 1.2 Handshake, Step by Step

LECTURE PDF COVERAGE: pp. 217–233

The instructor's Wireshark walk-through of dropbox.com (📄 pp. 217–223) enumerates seven observable handshake steps. Below, each step gives its purpose (what security service it contributes), message content, and what it proves.

Step 1 · TCP 3-way handshake (background)
Purpose: Open the underlying TCP connection to port 443.
Content: SYN → SYN-ACK → ACK.
Proves: Nothing yet — no crypto. TLS only begins after TCP is established. 📄 p. 233
Step 2 · ClientHello (client → server)
Purpose: Initiate TLS, advertise capabilities, contribute client-side randomness.
Content: Highest TLS version supported · 32-byte nonce RClient (4-byte timestamp + 28 random bytes) · Session ID (empty if fresh, populated if resuming) · list of supported cipher suites · supported compression methods.
Proves: Nothing about identity; it's a proposal. The nonce binds the later key derivation to this specific session. 📄 pp. 225, 230, 240
Step 3 · ServerHello (server → client)
Purpose: Pick one cipher suite and contribute server-side randomness.
Content: Chosen TLS version · 32-byte nonce RServer · Session ID · chosen cipher suite (just one) · chosen compression.
Proves: Still nothing — server has not yet authenticated itself. 📄 pp. 221, 225, 230, 240
Step 4 · Certificate (server → client)
Purpose: Authenticate the server via public-key infrastructure.
Content: Server's X.509 certificate chain — server cert signed by intermediate CA, intermediate signed by root, root in browser's trust store.
Client checks: signature chain, expiry date, Common Name / SAN matches the hostname, not revoked (OCSP/CRL).
Proves: The server possesses the private key matching the cert — once it uses that key in the next step. 📄 pp. 220, 230
Step 5 · ServerKeyExchange + ServerHelloDone (server → client)
Purpose: Contribute the server side of an ephemeral Diffie-Hellman (if using (EC)DHE), and signal "done talking for now."
Content (DHE): DH parameters (p, g) + server's public DH value YS, signed with the server's long-term private key (from the certificate). Sending ServerHelloDone signals end of server's handshake contribution.
Proves: This is the step that proves the server owns the cert — the signature over (RC‖RS‖DH params) binds the handshake nonces to the cert's public key. Also enables forward secrecy: even if the long-term private key leaks later, the ephemeral DH secret is unrecoverable. 📄 pp. 222, 223, 230, 231
Step 6 · ClientKeyExchange + ChangeCipherSpec + Finished (client → server)
Purpose: Establish the shared secret; signal "from now on, encrypt"; prove integrity of the whole handshake.
Content:
  • ClientKeyExchange — client's DH public value YC (DHE/ECDHE case), or client sends a 48-byte pre-master secret encrypted with the server's public key (RSA case). Both sides now derive: pre-master → master secret (using RC, RS) → six session keys.
  • ChangeCipherSpec — one-byte message meaning "all records I send after this are encrypted/MACed with the new keys."
  • Finished — first record under the new keys. Contains MAC/HMAC of all previous handshake messages, verified by server. If someone tampered with any earlier message, Finished fails.
Proves: Integrity of the whole handshake (no downgrade). The Finished's MAC uses the freshly derived keys. 📄 pp. 223, 227, 230, 232
Step 7 · ChangeCipherSpec + Finished (server → client)
Purpose: Server's mirror of step 6: switch to new keys and prove handshake integrity from its side.
Content: ChangeCipherSpec byte, then Finished with server's MAC over the handshake.
Proves: The handshake is cryptographically complete. Both sides now hold identical session keys. HTTP requests and responses flow next as encrypted records. 📄 pp. 223, 227, 230

The Record Layer Pipeline (post-handshake) 📄 pp. 228, 241

Every application-layer message flows through this pipeline before hitting TCP:

                      ┌──────────────────────────────────────────┐
  Application data    │ "GET /file HTTP/1.1 …"                   │
                      └───────────────┬──────────────────────────┘
                                      ▼
        Fragment into ≤ 2^14 B chunks (16 384 bytes max)
                                      ▼
              (Optional compression — often null in practice)
                                      ▼
       Append MAC = HMAC( write_MAC_key , seq_num ‖ header ‖ data )
                                      ▼
       Encrypt  ( data ‖ MAC ‖ padding )   with AES-CBC or AES-GCM
                                      ▼
      Prepend TLS Record header (content-type 23, length, version)
                                      ▼
                              Hand to TCP

Services delivered by this pipeline: confidentiality via symmetric encryption, integrity via MAC, authenticity inherited from the handshake (only the legitimate server could have derived these keys). 📄 pp. 228, 241

Key Derivation Details 📄 p. 232

  • Pre-master secret — 48 bytes; established by the key exchange (RSA-encrypted under server's public key, OR derived from (EC)DHE).
  • Master secret = one-time 48-byte value derived by iterated hashing:
    MS = SHA('A' ‖ PMS ‖ R_C ‖ R_S) ‖ SHA('BB' ‖ PMS ‖ R_C ‖ R_S) ‖ SHA('CCC' ‖ PMS ‖ R_C ‖ R_S) (approx; actual TLS 1.2 uses PRF).
  • Six session keys then derived by hashing master secret with more labels: client-write-MAC-key, server-write-MAC-key, client-write-key, server-write-key, client-write-IV, server-write-IV.
  • Every key is direction-specific — client and server encrypt independently, removing sequence-number re-use issues.

Line-by-Line Handshake Annotation

  • Nonces (RC, RS) — exchanged in the clear; not secret. Their role is freshness — they guarantee each session derives unique keys even from the same pre-master. 📄 pp. 225, 232
  • Cipher suite list — client proposes, server chooses. A downgrade attacker could try to strip strong suites; the Finished MAC over the handshake defeats this. 📄 pp. 227, 230
  • Certificate — trust is anchored at the root CAs pre-installed in the browser. Compromise of a single CA (e.g. DigiNotar 2011) breaks the chain system-wide. 📄 pp. 104, 130, 230
  • ServerKeyExchange signature — covers (RC ‖ RS ‖ DH_params). Because both nonces are included, it cannot be replayed into a different session. 📄 pp. 222, 231
  • Finished — this is where handshake tampering becomes detectable; anyone who changed any earlier message breaks the MAC. 📄 pp. 223, 227, 230
  • Change Cipher Spec — architecturally separate because it activates a new state rather than carrying data. 📄 pp. 215, 227

3.2 Common Misconceptions & Traps

MYTH: "TLS encrypts the entire TCP segment including IP headers."

REALITY: TLS encrypts only the application-layer payload (the TLS record). IP and TCP headers remain in plaintext — that's why a network eavesdropper still sees who you connect to (SNI in early handshake, destination IP), just not what is said. 📄 pp. 227–228

EXAM TRAP: "HTTPS hides the URL you visit" is tricky — the domain is in the SNI (visible) but the path and query are encrypted.

MYTH: "The server picks a cipher suite from the client's list."

REALITY: Correct. The client proposes a list of cipher suites ordered by preference; the server makes the final choice. A misconfigured server that picks a weak suite is a common vulnerability. 📄 p. 230

EXAM TRAP: A distractor says "both sides independently choose" or "the client chooses" — false.

MYTH: "Anonymous Diffie-Hellman (ADH) key exchange is safe because it uses DH."

REALITY: ADH has no certificate, so the client cannot authenticate the server — trivially vulnerable to man-in-the-middle. Pure DH without authentication is exactly the "unauthenticated channel" from the crypto lectures. 📄 pp. 145, 231

EXAM TRAP: MCQ listing ADH as a secure option for production TLS.

MYTH: "The master secret is sent encrypted over the wire."

REALITY: The master secret is never transmitted. Both sides derive it independently from the pre-master secret plus the two nonces. Only the pre-master is communicated (RSA-encrypted under server's public key, or established via (EC)DHE). 📄 p. 232

EXAM TRAP: T/F question claiming the master secret appears on the wire.

MYTH: "ChangeCipherSpec is part of the Handshake Protocol."

REALITY: ChangeCipherSpec is its own TLS sub-protocol (content-type 20). It's architecturally separate because it activates the new cipher state rather than negotiating it. 📄 pp. 215, 227

EXAM TRAP: Multi-answer question "Which of the following are TLS sub-protocols?" — ChangeCipherSpec is one of four; missing it loses a point.

MYTH: "Certificate validation proves the server is trustworthy."

REALITY: It only proves the server holds the private key for the name on the certificate, and that the name was vouched for by a CA the browser trusts. It does not prove the server is not a phishing site, nor that the CA's vetting was rigorous. 📄 pp. 130, 230

EXAM TRAP: Link-building: the wireless-plus-TLS challenge question often hinges on what certificates do and don't guarantee.

3.3 Compare & Contrast — TLS Key Exchange Methods 📄 p. 231

Method How the pre-master is established Forward Secrecy? Authentication Status
RSA Client encrypts random 48-byte PMS with server's RSA public key. ❌ No — if server's private key later leaks, past recorded sessions are decryptable. Server authenticated via its RSA certificate. Removed in TLS 1.3; still in TLS 1.2 fallback.
Fixed DH Server's DH public value is fixed in its certificate; client generates fresh DH pair. ❌ No — server's DH key is long-term. Via certificate (fixed DH key). Rarely used; legacy.
DHE (Ephemeral DH) Both sides generate fresh DH keys per handshake; server signs its public value with long-term cert key. ✅ Yes. Via certificate (signing ephemeral DH). Standard in TLS 1.2; superseded by ECDHE.
ECDHE (Ephemeral EC DH) ⭐ Elliptic-curve variant of DHE — same forward secrecy with much smaller keys. ✅ Yes. Via certificate (signing ephemeral EC-point). Default in TLS 1.3, modern browsers.
Anonymous DH Plain DH, no certificate, no signature. ✅ Yes (but moot). ❌ None — MITM-vulnerable. Never use in production.

INTMCQ📄 pp. 225, 232

Q2. What is the primary purpose of the 32-byte random nonces (RClient, RServer) exchanged in ClientHello and ServerHello?

  1. They are the actual session keys used for encryption.
  2. They contribute to master-secret derivation so each session derives unique keys, preventing replay.
  3. They authenticate the server to the client.
  4. They are discarded immediately after the handshake.
▶ Show Answer

✅ Answer: B. The nonces provide freshness — they are inputs to the master-secret hash, so even identical pre-master secrets would yield different session keys. 📄 pp. 225, 232

  • Why A wrong: The nonces are inputs to key derivation, not keys themselves.
  • Why C wrong: Server authentication comes from the Certificate + ServerKeyExchange signature, not the nonce.
  • Why D wrong: The nonces feed the master secret, which lives for the whole session.
ADVMCQ📄 pp. 223, 227, 230

Q3. A downgrade attacker silently strips the strong cipher suites from the client's ClientHello so the server picks the weakest one. Which handshake message is designed to detect this attack?

  1. ChangeCipherSpec
  2. Alert close_notify
  3. Finished (contains MAC over all prior handshake messages)
  4. ClientKeyExchange
▶ Show Answer

✅ Answer: C. Each side's Finished message is a keyed MAC over every handshake message seen so far (in its unmodified form, as expected by that side). If an attacker altered any byte of ClientHello or ServerHello, the two sides' MACs disagree and the handshake aborts. 📄 pp. 223, 227

Exam tip: This is why the Finished message is said to give "integrity of the handshake" — a single sentence summary worth memorising.

ADVMCQ📄 pp. 231, 226

Q4. Why is ECDHE preferred over static RSA key transport in TLS 1.3?

  1. ECDHE uses shorter keys, making servers faster.
  2. ECDHE provides forward secrecy: compromising the server's long-term key does not decrypt past sessions.
  3. RSA cannot authenticate servers.
  4. ECDHE encrypts faster than AES.
▶ Show Answer

✅ Answer: B. With static RSA key transport, the pre-master is encrypted under the server's long-term RSA public key. If that private key ever leaks, a passive attacker who recorded past sessions can recover the pre-master → master secret → session keys. ECDHE uses per-session ephemeral DH; once the ephemeral values are discarded, the secret is irretrievable. 📄 pp. 226, 231

  • Why A partially true but not the main reason.
  • Why C wrong: RSA does authenticate (via signature/encryption).
  • Why D wrong: ECDHE is key exchange, not bulk encryption.

ADVSA📄 pp. 226, 231

Q10. Explain why TLS 1.3 deliberately removed RSA key transport and static-DH key exchange, keeping only (EC)DHE. Include the security property that motivated the change and one trade-off.

▶ Show Answer

📝 Model:

  • Motivation — forward secrecy. With RSA key transport, the pre-master secret is encrypted under the server's long-term RSA public key. An attacker who records ciphertext today and obtains the RSA private key later (via subpoena, breach, or algorithm break) can decrypt past sessions retroactively. 📄 p. 231
  • (EC)DHE fix. Both sides generate ephemeral DH keys per handshake; the long-term certificate key is only used to sign the ephemeral public value. Once the ephemeral secrets are zeroed after the session, the pre-master cannot be recomputed even with the long-term key. 📄 pp. 222, 231
  • Static DH removed for the same reason — its DH secret was long-term.
  • Trade-off: Every handshake now includes fresh DH computation on both sides (moderate CPU cost, mitigated by elliptic-curve arithmetic being fast). Simpler cipher-suite negotiation (no RSA-KT options) is actually a benefit. 📄 p. 226

🔑 Checklist: ☐ forward-secrecy property named ☐ RSA-KT vulnerability explained ☐ ECDHE mechanism stated ☐ trade-off mentioned.

Section 5 · 4A TLS Cheat Sheet

TLS is a two-layer, four-component protocol where the Handshake layer (Handshake/CCS/Alert sub-protocols) negotiates cipher suite, authenticates the server via X.509 certificate, and derives a master secret via (EC)DHE; the Record layer then fragments-MACs-encrypts all application data using direction-specific session keys.

THE 3 THINGS YOU MUST KNOW

  1. Four TLS sub-protocols — Handshake (22), Record (23), Change Cipher Spec (20), Alert (21). The first three define the handshake + data phases; the fourth carries warnings and close_notify. 📄 pp. 215, 227
  2. Seven-step TLS 1.2 handshake — TCP → ClientHello → ServerHello → Certificate → ServerKeyExchange + HelloDone → ClientKeyExchange + CCS + Finished → ServerCCS + Finished. Master secret derived from pre-master + RC + RS. 📄 pp. 217–232
  3. ECDHE > RSA key transport — ECDHE gives forward secrecy; TLS 1.3 removed RSA key transport entirely and keeps only (EC)DHE. 📄 pp. 226, 231

THE #1 MISTAKE STUDENTS MAKE

Confusing what TLS protects — TLS encrypts the record payload, not the IP or TCP headers. A network observer still sees destination IP and port. If that troubles you, reach for IPsec (transport/tunnel mode) at the next layer down. 📄 pp. 227–228

IF YOU REMEMBER NOTHING ELSE

"Hello → Hello → Cert → Sig → KX → Done. The Finished MAC signs the whole conversation."

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 224–226, 233

1.1 The Problem This Solves 📄 p. 233

Plain HTTP transmits URLs, cookies, form fields, and bodies in cleartext. Anyone on the network path (ISP, public Wi-Fi attacker, compromised router) can read them, inject scripts, or modify responses. HTTPS is the trivially-simple answer: run ordinary HTTP inside a TLS session. No change to the application semantics; every byte of HTTP is now confidential, integrity-protected, and the server is authenticated. 📄 p. 233

1.2 The Core Idea — Three-Layer Connection Init 📄 p. 233

A single HTTPS fetch cascades through three protocol layers, each depending on the one below it:

  1. TCP layer — 3-way handshake on port 443 establishes a reliable byte stream.
  2. TLS layer — full TLS handshake (see 4A) runs over that TCP stream, producing session keys.
  3. HTTP layer — actual GET/POST requests and responses flow, each one wrapped in an encrypted TLS record.

Slide 233 states it verbatim: "TCP 3-way handshake ⇒ TLS handshake ⇒ HTTP request(s)". That sequence is the single most likely HTTPS exam question.

1.3 Analogy — Talking on an Encrypted Phone 📄 p. 233

Imagine making a secret phone call:

  • TCP = dialling and hearing the ring tone until the other person picks up.
  • TLS = you both put on voice-scrambling headsets and agree on tonight's codebook.
  • HTTP = the actual conversation, now scrambled through the headsets.

Close-down mirrors the build-up: "okay, bye" (HTTP Connection: close) → "ending secure session" (TLS close_notify) → hang up (TCP FIN). If someone hangs up before saying "ending secure session," that's suspicious — could be a truncation attack. 📄 p. 233

1.4 Technical Definition 📄 p. 233

HTTPS is not a new protocol but a composition: HTTP semantics, carried as the application-layer payload of TLS records, flowing over TCP port 443. The HTTP client initiates by requesting a connection to the next-lower layer (TLS); the TLS layer in turn requests a TCP connection. All HTTP parsing logic is unchanged — the HTTP client and server see a plain socket; the TLS library transparently encrypts/decrypts in between. 📄 p. 233

Section 2 · How It Works — An HTTPS Connection, Beginning to End

LECTURE PDF COVERAGE: pp. 217–226, 233
Phase 1 · DNS Resolution (implied, not part of HTTPS itself)
What happens: Browser asks its resolver for gaia.cs.umass.edu, gets an A record with the IP.
Key exam note: If DNS is poisoned (Topic 2 DNS attacks), the client will connect to the attacker's IP — but TLS certificate validation in Phase 3 catches the impersonator because the cert's CN/SAN won't match. This is the classic TLS saves you when DNS lies integration point.
Phase 2 · TCP 3-way handshake on port 443
Messages: SYN → SYN-ACK → ACK.
Result: Reliable bidirectional byte stream between client:rand_port and server:443. Nothing is yet encrypted. 📄 p. 233
Phase 3 · TLS handshake (identical to 4A)
Messages: ClientHello → ServerHello → Certificate → ServerKeyExchange + HelloDone → ClientKeyExchange + CCS + Finished → ServerCCS + Finished.
Certificate verification: Browser checks that the cert's subject matches gaia.cs.umass.edu (SAN or CN), signature chain is valid up to a trusted root CA, and the cert is not expired/revoked.
Result: Both sides hold six session keys; encrypted tunnel is open. 📄 pp. 217–223
Phase 4 · Encrypted HTTP transaction(s)
Messages: HTTP GET / path → HTTP 200 OK + body. Every HTTP byte is a TLS record payload; the TLS Record Protocol encrypts each with AES-CBC/AES-GCM + MAC.
Observer sees: TCP segments to/from port 443, visible length pattern, but not the URL path, headers, cookies, or body.
Keep-alive: HTTP/1.1 by default keeps the TLS connection open for multiple requests. HTTP/2 multiplexes many requests over a single TLS connection. 📄 p. 233
Phase 5 · Graceful close
Messages: HTTP includes Connection: close in its last response → TLS sends close_notify alert (both sides) → TCP FIN/ACK exchange.
Security rule: If the TCP FIN/RST arrives without a preceding close_notify, the HTTPS client should raise a warning — possibly a truncation attack (attacker cut off the response early). 📄 p. 233

3.2 Common Misconceptions & Traps

MYTH: "HTTPS uses a different HTTP protocol from HTTP."

REALITY: HTTP syntax and semantics are identical — method verbs (GET/POST), headers, status codes, bodies. HTTPS only adds a TLS wrapper. A developer writing an HTTP server library doesn't change anything; the TLS library wraps the socket. 📄 p. 233

MYTH: "HTTPS protects the URL, including the domain name, from observers."

REALITY: HTTPS encrypts the path and query (those live inside HTTP), but the destination IP is visible (IP header, plaintext) and the domain appears in the TLS ClientHello's SNI extension (plaintext in TLS 1.2). An on-path observer knows you visited gaia.cs.umass.edu, just not which page. 📄 pp. 225, 227–228

EXAM TRAP: T/F "HTTPS hides which website you're visiting" — nuanced, but generally FALSE.

MYTH: "HTTPS runs over UDP for speed."

REALITY: HTTPS runs over TCP port 443. HTTP/3 (QUIC) does use UDP, but that's a separate protocol with its own crypto handshake derived from TLS 1.3 — not what the CS 448 slides cover. 📄 p. 233

MYTH: "A TCP RST always indicates a network problem."

REALITY: In the HTTPS context, a RST or unannounced FIN in the middle of a response may be a truncation attack — attacker cuts the response before sensitive data finishes, aiming to trick the client into processing an incomplete page. The slide tells the client to "issue some sort of security warning" in that case. 📄 p. 233

HTTPS is HTTP running inside a TLS session on TCP port 443: the same HTTP syntax you already know, with confidentiality + integrity + server authenticity bolted on by the lower TLS layer — set up as TCP handshake → TLS handshake → HTTP request and torn down in mirror order via close_notify + TCP FIN.

THE 3 THINGS YOU MUST KNOW

  1. Three-layer cascade — TCP 3-way handshake → TLS handshake → HTTP request/response. Port 443. 📄 p. 233
  2. Graceful close — HTTP "Connection: close" → TLS close_notify → TCP FIN. An unannounced TCP close = warn the user. 📄 p. 233
  3. What HTTPS hides and doesn't — Hides: path, query, headers, body. Visible: destination IP, SNI hostname (TLS 1.2), packet lengths/timings. 📄 pp. 225, 227

THE #1 MISTAKE STUDENTS MAKE

Treating HTTPS as a different HTTP protocol. It is literally the same HTTP — the only difference is the socket underneath is a TLS-wrapped TCP socket instead of a plain TCP socket. All HTTP library code is unchanged. 📄 p. 233

Section 1 · Concept Overview

LECTURE PDF COVERAGE: pp. 246–258

1.1 The Problem This Solves 📄 p. 246

Before SSH, remote administration tools like Telnet, rlogin, and rsh carried usernames, passwords, and typed commands across the network as plaintext. Anyone on the path could harvest admin credentials and shell sessions. SSH (1995) replaces those protocols with an authenticated, encrypted, integrity-protected tunnel — and on top of that, a flexible multiplexed-channel architecture that handles shell, file transfer, port forwarding, and X11 over the same connection. 📄 p. 246

1.2 The Core Idea — Three-Layer Architecture 📄 pp. 252–253, 257

SSH is organised as three sub-protocols stacked on top of a TCP connection to port 22:

  1. Transport Layer Protocol — server authentication, confidentiality, integrity, forward secrecy via Diffie-Hellman. Analogous to TLS's record + handshake layers. 📄 p. 252
  2. User Authentication Protocol — runs inside the already-encrypted tunnel to authenticate the user (distinct from server authentication). Supports password, public key, hostbased. 📄 p. 257
  3. Connection Protocol — provides multiple multiplexed logical channels inside the one authenticated tunnel: session, X11, forwarded-tcpip (remote port forwarding), direct-tcpip (local port forwarding). 📄 p. 253

This stacked design is what makes SSH a first-class "good example of challenge question" — the instructor flagged it explicitly — because each layer maps onto a TLS concept and so invites cross-protocol questions. 📄 p. 257

1.3 Analogy — A Secured Office Building 📄 pp. 252–253

  • Transport Layer Protocol = the guarded lobby. Identity of the building is verified (the building shows its ID — server public key). You exchange a one-time secret with the receptionist. From here, the whole building is sealed.
  • User Authentication Protocol = inside the sealed building, you present your badge to a turnstile (password in the tunnel, or public-key signature).
  • Connection Protocol = once past the turnstile, you can walk into multiple rooms simultaneously — the meeting room (shell session), the print room (SFTP), the conference line (port forward) — all inside the same building.

1.4 Technical Definition 📄 pp. 252, 257

SSH (Secure Shell) is an application-layer protocol (typically TCP port 22) providing encrypted remote login and other secure network services. The SSH Transport Layer Protocol authenticates the server using a long-term server public/private key pair and establishes fresh session keys via Diffie-Hellman, with keys rotated every 1 GB of data or every 1 hour. The User Authentication Protocol authenticates the user inside the encrypted transport, using password, public key, or hostbased methods. The Connection Protocol multiplexes independent logical channels (each with unique IDs) over the authenticated transport. 📄 pp. 252–253, 257

Section 2 · How It Works — An SSH Session End-to-End

LECTURE PDF COVERAGE: pp. 248–253, 256–257
Phase 1 · TCP connection to port 22
What happens: Client opens TCP to the server's port 22.
Exam note: Nothing yet is encrypted — but port 22 is the only required convention for SSH. 📄 p. 248
Phase 2 · SSH protocol-version exchange + algorithm negotiation
What happens: Each side sends an ASCII banner (SSH-2.0-OpenSSH_8.4). Both send lists of supported algorithms for key exchange, encryption, MAC, and compression. One is selected for each.
Analogy to TLS: This step corresponds to ClientHello/ServerHello in TLS.
Phase 3 · Server authentication + Diffie-Hellman key exchange (Transport Layer Protocol)
Messages:
  1. Server sends its long-term public host key.
  2. Client looks up the server's hostname in its local known_hosts file:
    • If match → proceed silently.
    • If first time → prompt user "The authenticity of host X can't be established, key fingerprint = SHA-256:…, continue?" (Trust-On-First-Use).
    • If mismatch → warn loudly: "@@@@@ REMOTE HOST IDENTIFICATION HAS CHANGED @@@@@" and refuse to connect unless the user overrides. This is the on-wire signal of a MITM attack.
  3. Both sides run a DH key exchange; the server signs its DH public value with the host private key to prove it holds the host key.
  4. Both derive fresh session encryption + MAC keys.
Proves: The server owns the host key whose fingerprint is in known_hosts, and both sides now share a secret unknown to eavesdroppers. Forward secrecy is achieved via ephemeral DH. 📄 p. 252
Phase 4 · User authentication (User Authentication Protocol) — inside the encrypted tunnel
Choose one of:
  • Password: client sends plaintext password encapsulated in an encrypted SSH packet → server verifies against /etc/shadow or PAM. 📄 p. 257
  • Public key: client sends its public key + a signature over a session-bound challenge, made with its private key. Server checks the key is in ~/.ssh/authorized_keys and signature verifies. No password ever crosses the wire. 📄 p. 257
  • Hostbased: client sends signature made with the client host's private key; server trusts users vouched for by that host. 📄 p. 257
Phase 5 · Multiplexed channels (Connection Protocol)
What happens: The authenticated tunnel now carries multiple logical channels simultaneously. Either side may open a channel; each has a unique numeric ID. Four channel types are defined:
  • session — a shell or single command execution.
  • x11 — X Window System forwarding (remote GUI).
  • forwarded-tcpip — remote port forwarding (server → client).
  • direct-tcpip — local port forwarding (client → server → destination).
All channels are multiplexed into the single encryption tunnel. 📄 p. 253
Phase 6 · Rekeying and close
Every 1 GB of data or 1 hour of session time, both sides rerun DH to derive new session keys, then switch over. 📄 p. 252 At end, either side sends a channel close; when all channels are closed, the transport layer tears down and TCP FIN follows.

Line-by-Line Annotation

  • No CA hierarchy in SSH — unlike TLS, SSH does not rely on global CAs. Trust is local (known_hosts) and established on first connect. This is a classic exam contrast. 📄 p. 252

3.2 Common Misconceptions & Traps

MYTH: "SSH uses X.509 certificates signed by CAs, like TLS."

REALITY: Classic SSH uses raw public keys stored locally in known_hosts. There is no global CA system; trust is established on first connection (Trust-On-First-Use). Enterprise SSH deployments can use SSH certificate authorities, but that is an extension, not the default. 📄 p. 252

EXAM TRAP: MCQ listing "server presents an X.509 certificate signed by a CA" as an SSH behaviour — wrong.

MYTH: "Password authentication in SSH is insecure because the password travels in plaintext."

REALITY: The password is sent inside the already-encrypted Transport Layer tunnel — so on the wire it is ciphertext. Public-key auth is preferred for other reasons (no reusable secret on the server, defence against offline attacks), but password over SSH is not equivalent to password over Telnet. 📄 p. 257

EXAM TRAP: T/F "SSH with password auth sends credentials in the clear" — FALSE.

MYTH: "TOFU (known_hosts) is as strong as TLS's CA model."

REALITY: TOFU is vulnerable on the first connection — if an attacker MITMs that initial connection, the attacker's key lands in known_hosts and subsequent sessions will silently accept it. TLS's PKI model avoids this by anchoring trust in pre-installed CA roots. Both are valid models with different attack surfaces. 📄 p. 252

EXAM TRAP: A question asking when SSH's trust model is weakest — answer is the first connection.

MYTH: "One SSH connection carries one service at a time (shell OR file transfer)."

REALITY: The Connection Protocol multiplexes multiple independent channels in parallel. A single SSH session can simultaneously carry an interactive shell, an SFTP transfer, and several port-forwards. 📄 p. 253

MYTH: "SSH runs over UDP for low latency."

REALITY: SSH requires reliable delivery and runs over TCP port 22. 📄 p. 248

Common Threads Across All Three Protocols

  • All three use (EC)DHE for forward secrecy. Modern TLS, SSH, and IPsec all rely on ephemeral Diffie-Hellman. The long-term keys are used only to sign (authenticate) the ephemeral values. 📄 pp. 231, 252
  • All three separate server authentication from record protection. A short handshake phase establishes identity + keys; a long data phase uses those keys. 📄 pp. 215, 252
  • All three tunnel application data. TLS tunnels HTTP (HTTPS); SSH tunnels shell/SFTP/forwarded ports; IPsec tunnels whole IP packets. 📄 pp. 233, 253, 373
  • All three have a "signal the switchover" step. TLS: ChangeCipherSpec. SSH: NEWKEYS. IPsec: IKE SA established. 📄 pp. 215, 252
